# 0.8.0 GA → IR-eval — sequenced roadmap

The remaining work runs on **three parallel tracks** joined by a few **gates**. The key structural fact:
the **GA track** (fidelity gate) and the **IR-eval track** (relevance gate) are **different axes**
([[fathomdb-recall-fidelity-vs-relevance]]) and only *share* one input — the **corpus basis** (which corpus
snapshot the eval runs against). That shared input is why B-1 (the corpus-basis ruling) sits on the GA
critical path **and** unblocks the IR-eval experiment phases.

## Tracks
- **GA track** — finish 0.8.0: rule B-1, land Slice 40, GA sign-off. *Critical path to the release.*
- **Corpus track** — owner-managed, out-of-band: finish Version B (~10K; currently ~7.6K), refine, test,
  **freeze a versioned SHA-256 snapshot**. Does **not** gate GA (GA pins a snapshot); **does** gate IR-1 Ph3–4.
- **IR-eval track** — stand up the product-value recall gate (IR-1 → IR-2 → AC-077). Post-GA / 0.8.1.

## "B-1 or consensus first?" → **Both, now, in parallel — they're independent.**
- **B-1** (corpus-basis ruling) is on the **GA critical path** and needs the OLD-vs-NEW A/B first.
- **IR-1 Phase 1** (measure + Claude↔codex consensus) is the **IR-eval design track** — it has **no**
  prerequisites and blocks nothing on GA. Run it now in a separate session.
- If you can only push **one HITL decision**, push **B-1** — it unblocks Slice 40 *and* settles the corpus
  basis IR-1 Ph3–4 depends on. Consensus is parallel work that de-risks the IR track without touching GA.

## The two critical paths
- **To 0.8.0 GA:**  GA-1 (A/B) → **B-1** → GA-2 (apply + verify + B-3) → **GA-merge** (Slice 40) → **GA sign-off**.
  *(Corpus expansion + the entire IR-eval track are parallel and land after GA.)*
- **To the IR product-recall gate:**  IR-A (Phase 1, now) ∥ COR-2 (corpus freeze) → **IR-B → IR-C** (code →
  experiments; need B-1 + frozen corpus) → **IR-D** (mint AC-077, grounded; needs GA-merge for the id) →
  **IR-E (IR-2)** → **IR-gate**.

## Direct answers
- **B-1 or consensus first?** Parallel; both now. B-1 = GA critical path; IR-1 Phase 1 consensus = parallel
  design track, blocks nothing on GA.
- **Your top-level order — correct but over-serialized.** It's a valid dependency order, but the **Corpus
  track and IR-1 Phase 1 run in parallel with GA from the start** (not after GA sign-off), and **IR-B/IR-C
  need GA-*merge* + corpus-freeze, not the GA *tag*.** Only the GA chain (A/B→B-1→Slice-40→sign-off) is strictly serial.
- **Experiments before code?** **No — code first** (the harness/gold set must exist to run experiments): IR-B → IR-C.
- **Experiments before the AC is decided?** **Yes.** AC-077 is minted *after* experiments (IR-C → IR-D),
  grounded by the data; its **binding thresholds** are decided *even later* (IR-2 → HITL). Only the **measure**
  (Phase 1) is decided before code. The reserved `AC-077` id is a placeholder, not a decided gate.
- **AC decided before code?** **No** — the **measure** is (Phase 1, the code's target); the **AC as a gate** is
  decided after experiments. (This resolves the apparent cycle: "the AC" = measure-first, thresholds-last.)
- **When does the corpus finish / refine / test?** Parallel to GA; does **not** gate GA (GA pins a snapshot at
  B-1). Hard deadline = **before IR-B** (the gold set needs a frozen corpus): finish Version B → refine per the
  A/B/quality findings → test → freeze a versioned SHA-256 snapshot.
- **When does Slice 40 land?** Immediately after **B-1** rules (corpus basis pinned ⇒ fidelity green) + GA-2
  verify + codex §9 + B-3 resolved → GA-merge.
- **When do IR-1 Phases 2–4 complete?** **Phase 2 (code)** + **Phase 3 (experiments)** after **B-1 + corpus
  freeze**; **Phase 4 (mint AC-077, grounded)** after experiments **+ GA-merge** (for the id); then **IR-2**,
  then the HITL IR-gate. None block GA.

## How to read this plan

### Numbering — N-mod-5 with reserved gaps (HITL directive)
Planned slices are numbered **only in multiples of 5**: `0, 5, 10, 15, 20, 25, 30, 35, 40`.
The gaps (`1–4`, `6–9`, `11–14`, `16–19`, `21–24`, `26–29`, `31–34`, `36–39`, `41–44`) are
**reserved on purpose**. They are the *insertion mechanism* for unplanned follow-on work
discovered while a slice executes: if Slice N reveals work that must land before the next
planned slice (N+5), that work becomes Slice N+1, N+2, … N+4 — a **fully-orchestrated slice
in its own right** (own prompt, own worktree from a fresh baseline, own `output.json`, own
codex review, own CLOSED block), **not** an ad-hoc patch.

> **Reserved-gap ≠ fix-N.** A §6 *fix-N* re-enters the **same** slice's existing
> worktree/branch to clear a BLOCK/CONCERN on **that slice's own deliverable**. A
> *reserved-gap slice* is **new scope** (fresh baseline, fresh worktree) for adjacent work
> the original slice revealed. If a gap band (N+1..N+4) fills and more follow-on remains,
> the slice was mis-scoped → **HALT to HITL**, never overflow into the next mod-5 number.

### Slice-orchestrator model (HITL directive)
Each slice has a **governing agent — the "Slice N agent" — which is the slice
ORCHESTRATOR running on the MAIN THREAD** (§1 forbids an orchestrator subagent). It may
spawn **≤ 1 level** of subagents (`N.a`, `N.b`). **Subagents never spawn further
subagents** (one level only; anti-chain guard §10). The orchestrator owns all git-worktree
lifecycle and never edits worktree files itself.

### Slices are not all implementation
Valid work-types: **research, experiment, verification, documentation, design/ADR, and
implementation**. Choose the right type per slice. Non-impl slices run the *adapted*
lifecycle (the ADR/research-plan is the "design"; a falsifiable acceptance bar replaces
TDD RED→GREEN; "self-check" is the agent validating its artifact against that bar;
"codex review" is an adversarial pass on the artifact; "fix-N" iterates the artifact until
the bar is met or HITL signs).

### Cross-cutting requirements (HITL directive 2026-06-02 — bind EVERY slice)

Three obligations apply to **every** slice's Definition of Done, not just the slices that
nominally "own" them. A slice's `output.json` (or adapted closure) and its codex review
**must** show these were honored, or the verdict is incomplete:

**X1 — SDK parity + functional test harnesses (Python AND TypeScript).** Any slice that
changes the engine surface, write/search/read behavior, error taxonomy, or result shapes
MUST land the change in **both** the Python SDK (`src/python/fathomdb/`) and the TypeScript
SDK (`src/ts/src/`) **in the same slice** (the SDK-parity invariant — no per-binding drift).
Each SDK keeps its **own functional test harness** that exercises the live SDK surface end
to end — not just symbol presence. Ground truth (2026-06-02): today's
`src/python/tests/` + `src/ts/tests/` cover *surface / errors / openreport / save-time
validation* but **lack functional write→search→retrieve→admin correctness tests**. This plan
**adds those harnesses** (Slice 5 stands them up against the first behavior change; every
later slice extends them for its verb). Required coverage as the surface grows: **bindings
& API** (each verb callable, typed args/results marshal correctly across the FFI in both
languages), **write** (node/edge/op-store batches, receipts), **search** (hybrid results,
fusion ordering, soft-fallback), **retrieve** (by-id `read.get`, `read.collection`, filtered
`read.list` as they land), **admin** (`admin.configure`, governed `read.*`), and
**correctness** (cross-binding equivalence: same DB, same op, Python ≡ TypeScript). The
conformance/parity test (Slice 25) gates the *shape*; these harnesses gate the *behavior*.

**X2 — `mkdocs build` stays green.** `mkdocs.yml` exists at repo root (theme `mkdocs`,
`docs_dir: docs`). Every slice that adds/renames a `docs/` page MUST update `mkdocs.yml`
`nav` in the same slice, and the slice's self-check/verification MUST run `mkdocs build`
(strict where feasible) and confirm zero errors / no orphaned nav entries / no broken
internal links. Slice 40 (verification) runs `mkdocs build` as a release gate; Slice 0
adds it to the standing verification checklist. (Today's `nav` is stale — missing 0.6.1/0.7.x
release notes — so Slice 0 also reconciles the nav baseline.)

**X3 — documentation updated per slice + a maintained doc index.** Every slice updates the
**relevant** docs in both trees as part of its Definition of Done (not a deferred cleanup):
- `dev/` (engineering): `needs.md`, `requirements.md`, `architecture.md`, `design/*`,
  `test-plan.md`, `acceptance.md`, `traceability.md` — whichever the slice's change touches.
- `docs/` (user-facing): SDK usage, API reference (`reference/python-api.md`,
  `reference/typescript-api.md`, `reference/cli.md`), guides, and **runnable examples**.
- **A documentation index for agentic use** — a single `dev/DOC-INDEX.md` (one-line entry
  per doc: path → purpose → owning slice/AC → last-touched) that is **updated in the same
  docs commit that closes each slice** (mirrors the §12.4 plan-as-state-machine discipline,
  applied to docs). **Slice 0 creates `dev/DOC-INDEX.md`**; every later slice's closure step
  appends/updates its entries. The index is the cold-start map an agent reads to find the
  right doc without scanning the tree.

Mechanically: X1/X2/X3 are added to **every** slice's success criteria and to the codex
review checklist; the STATUS-0.8.0.md board carries an X1/X2/X3 column per slice so a gap is
visible at a glance.

## Critical path, parallelization, and HITL session boundaries

### Keystone-gated critical path
```
Slice 0  (substrate + supersession ADR drafts, STATUS board)
  → Slice 5  (G1 SearchHit + tokenizer; DROPS Eq; sets score-field carrier + recall baseline)
  → Slice 15 (G0 KEYSTONE: schema 11→12; logical_id + superseded_at + partial-unique-active
              + folded G4/G5 indexes; swaps G1's interim write_cursor id → logical_id)
  → Slice 25 (ADR-supersede HITL sign-off + conformance rewrite set-equality→allowlist+parity)
  → Slice 30 (G2 read.get + G3 read.collection SDK verbs under read.*)
  → Slice 40 (verification + release readiness)
```
**G0 (Slice 15) is the keystone** gating G8 (20), G2/G3 (30). **Slice 25's sign-off is the
second hard gate** on the SDK read tier. Slice 10 (G9/G10/G12-recency) hangs off Slice 5;
Slice 20 (G8) hangs off Slice 15; both are off the longest chain. Slice 35 (deferred ADRs)
is off-path docs feeding 0.8.x.

### Parallelization plan (also recorded verbatim in the STATUS board §12.5 section)
- **0.a ∥ 0.b** — substrate ADR vs supersession ADR + scaffolding are independent.
- **Read-path family (5 → 10) ∥ Slice 0 ADR authoring** once the substrate ADR *draft*
  exists — 5/10 are AC-057a-clean and do **not** depend on G0 landing.
- **Slice 20 ∥ Slice 25** — 25 depends on 15, not 20; 20 takes its baseline from
  post-Slice-15 mainline.
- **Slice 30 ∥ Slice 35** — both depend on 15+25; 35 is docs-only and does not depend on 30.
- **35.a ∥ 35.b** — the two ADR pairs are independent.
- **NOT parallelizable:** anything touching `commit_batch` / the `SearchHit` carrier must
  serialize through the keystone (15) to avoid carrier-reshape churn; the implementer
  contract forbids cross-worktree edits from the main thread (§10).

### HITL gate boundaries = natural §12.7 session boundaries
1. **After Slice 0** — HITL signs substrate-ADR Q2/Q4/cascade/migration (gates 15) and
   readies supersession Q1–Q5.
2. **Before Slice 10 spawn** — HITL signs retrieval ADR Q1 (1A table-stakes) + Q3 (RRF
   documented compat event, NO knob). **✅ SIGNED 2026-06-02.**
3. **At Slice 25** — HITL signs supersession Q1–Q5 (HARD gate unblocking Slice 30).
4. **At Slice 35** — HITL signs the four deferred-feature ADR decisions (gates 0.8.x).
5. **At Slice 40** — HITL GA sign-off request.
6. **Any slice** — unclearable BLOCK, fix-N past a small bound, or a filled reserved-gap
   band → HALT to HITL (§1.5 invariant 3).

### Slice 5 — G1 Structured Search Hits + Global FTS5 Tokenizer Upgrade  · `[implementation]` · depends-on: 0 · gaps: 6–9

**Objective.** Land G1/F12 structured search hits and fold in the zero-surface global FTS5
tokenizer-default upgrade, **first** in build order to de-risk the 0.90 recall floor before
the keystone migration touches identity carriers. Reshape **both** search branches (vector
phase-1 and FTS5 phase-2) to emit `SearchHit{id, kind, body, score:f64, branch}`; change
`SearchResult.results` from `Vec<String>` to `Vec<SearchHit>`; **DROP `Eq` from
`SearchResult`'s derive** (`f64` forbids `Eq` — a real compile failure gating everything
downstream). Interim `id = write_cursor` (swaps to `logical_id` at G0 with no carrier
reshape). Mandatory Py+TS parity. Sets the `score:f64` carrier Slice 10's RRF lands on and
establishes the recall-floor baseline.

> **CORRECTED tokenizer mechanism (do NOT edit the existing DDL).** Migrations are
> forward-only and immutable: `apply` only runs steps with `step_id > before`, and FTS5 has
> no `ALTER … tokenize`. Editing the existing `search_index` step (step_id 5) changes the
> tokenizer for **new DBs only** — migrated DBs keep `unicode61`, defeating the "global
> default upgrade" and making the recall delta meaningless on a real corpus. Instead author
> a **NEW forward migration step** (next `step_id`) that:
> (a) `DROP TABLE IF EXISTS search_index;`
> (b) `CREATE VIRTUAL TABLE search_index USING fts5(body, kind UNINDEXED, write_cursor
> UNINDEXED, tokenize = 'porter unicode61 remove_diacritics 2');`
> (c) carries `-- MIGRATION-ACCRETION-EXEMPTION: tokenizer-default upgrade (drop+recreate
> FTS5 projection; no source-record migration)` — **required** by `check_migration_accretion`
> (`fathomdb-schema/src/lib.rs:362`), which rejects schema-adding/removing SQL lacking the
> marker. Then `rebuild_projections` re-tokenizes from the canonical source rows
> (projection-only; **no** source-record migration).

**Governing-agent behavior.** Main-thread orchestrator only (§1). Owns the worktree: from
the Slice-0-closing baseline on main, `git worktree add /tmp/fdb-slice-5-<ts> -b
slice-5-<ts> <baseline_sha>` (§1; never Agent-native isolation), advancing
NEW→PROMPTED→WORKTREE_CREATED. Sequences subagents strictly serially: spawn 5.a → read
`output.json` (§8 witness; confirm IMPLEMENTED = output.json present AND head advanced) →
cherry-pick to mainline (§5, never merge) → spawn codex (§3) → promote verdict (§4) → THEN
spawn 5.b (verification) as a second adversarial witness on the floor + parity → decide
PASS/override/fix-1. Each spawn prompt carries its own worktree/branch/baseline/output path
(§10); the implementer prompt omits Agent/Task. Spawn as `implementer`; fall back to
`general-purpose` (Agent/Task omitted) only on spawn error.

## Orchestration rules this plan binds to

This plan binds to `dev/design/orchestration.md`:

- **§1 — three-role separation.** The main thread orchestrates (no orchestrator subagent).
  Implementers run as the `implementer` subagent in a **main-thread-owned git worktree**
  (`git worktree add` from a chosen baseline; never Agent-native isolation). The reviewer is
  `codex exec --sandbox read-only` against the worktree branch. Design-adr/verification
  subagents own **no worktree** (cited adaptation: §1 ties worktrees to the implementer role
  and is silent on non-impl subagents).
- **§1.5 — state spine.** `NEW → PROMPTED → WORKTREE_CREATED → IMPLEMENTING → IMPLEMENTED
  (output.json present + branch advanced) → REVIEWED → (PASS | fix-N) → CLEANED`. Each step is
  gated on the prior witness; an undefined transition or an unclearable BLOCK / fix-N past a
  small bound / filled reserved-gap band HALTS to HITL (invariant 3).
- **§8 — closure.** Every implementation slice writes `dev/plans/runs/<id>-output.json`
  (phase, baseline_sha, branch, head_sha, commits, findings_addressed, blockers_encountered,
  tdd_evidence, agent_verify_result, next_step_for_orchestrator). Non-impl slices produce the
  adapted artifact (ADR/note/report) instead; Slice 40's 40.a emits an adapted no-worktree
  output.json.
- **§9 — decision loop.** Read output.json → cherry-pick to mainline (**never merge**) → codex
  review → promote verdict to `<id>-review-<ts>.md` → PASS close / structural CONCERN override /
  substantive CONCERN | BLOCK → fix-N → edit plan + advance the pointer **in the same docs
  commit** (step 7).
- **§6 — fix-N.** A fresh implementer into the **existing** worktree/branch; ~1 fix-N per BLOCK;
  unbounded fix-N or an unclearable BLOCK → HALT to HITL.
- **§11 — worktree cleanup.** After a phase/slice family closes, one `git worktree remove` per
  Bash call (never `find -delete`); Slice 40 is the final cleanup gate (ledger must be empty).
- **§12 — context discipline.** Each slice gets a self-contained `dev/plans/prompts/<id>.md`;
  the implementer never sees prior-slice conversation; the plan doc is canonical state
  (§12.4); phases > 4 slices require `STATUS-0.8.0.md` (§12.5, with all nine sections incl. the
  Parallelization plan); HITL gates are natural session boundaries (§12.7).
- **§10 — hard rules.** Never override BLOCK; cherry-pick not merge; per-spawn facts
  (worktree/branch/baseline/output path) in the Agent prompt; the `implementer` agent prompt
  omits Agent/Task (anti-chain guard).

### Per-slice execution discipline (the universal loop EVERY slice runs)
1. **Orchestrate to the plan.** The Slice N agent FOLLOWS the plan; if reality diverges it
   ADJUSTS the plan/STATUS board (§12.4) and records the adjustment in the same docs commit
   that closes the slice (§9 step 7) — never silently.
2. **Design-first, self-checked.** Before code, the implementation subagent writes a short
   design (memo / `dev/design/<slice>-design.md`) covering approach, touch-points, and test
   plan, and self-CHECKS (suite + lint + typecheck + agent-verify) before returning. Evidence
   goes in `output.json.agent_verify_result` (§8).
3. **TDD.** RED first (failing tests pinning the acceptance criteria — the `pr_<slice>` / AC
   tests), then GREEN (minimal code), then refactor. `output.json.tdd_evidence` records the RED
   commit sha + the test files.
4. **Codex review.** After self-check, the orchestrator cherry-picks to mainline and spawns the
   read-only codex reviewer on the worktree branch; the verdict is promoted (§4).
5. **Fix-N until PASS / CONCERN-override; BLOCK → HITL.** Run §6 fix-N until PASS (or §7
   structural override). **Never override BLOCK;** an unclearable BLOCK or fix-N past the small
   bound HALTS to HITL (§1.5 invariant 3).

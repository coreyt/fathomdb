## 3. OPP-12 placement (early) + slice decomposition

OPP-12 is the ratified 3-axis mechanism/policy contract (`dev/design/record-lifecycle-protocol/`, 5 docs).
The **surface delta** (from `api-surface.md` + the leverage-ledger update): **+3 verbs** (`transition` /
`purge` / `configure_projections`; `drain` reused, `read.projections` folded); **+5 governed types**
(`ReadView`, `LifecycleState`, `IllegalTransitionError`, `ProjectionSpec`, `dense_readiness`); **`SearchHit`
shrinks** (`id` subsumes `stable_id` + `write_cursor`, becomes a typed `{space:IdSpace, value}` newtype);
**`undelete` not `restore`** (`restore` is on the five-name `recovery_denylist`). Governed-type count moves
`AC-074 29 → ~34–36`.

### 3a. Prerequisites already met vs net-new (code-grounded)

Per `code-grounded-audit.md` the contract is **~90% net-new**; the audit is the honesty guardrail. What
the build stands on:

- **Met — version-currency axis:** the shipped HITL-signed G0 `UNIQUE(logical_id) WHERE superseded_at IS
  NULL` partial index (`schema:300`, ADR-0.8.0, *logical_id-alone*) **is** the single is-latest authority
  (there is **no** `is_latest` column — it is a derived predicate; do not build one).
- **Met — EXP-S substrate (0.8.14):** row-kind coexisting indexes + the TC-1 per-kind index-target seam the
  projection registry wraps.
- **Met — Cause-A (0.8.11.2):** additive `SearchHit.stable_id` — the base the C-2 typed `id` subsumes.
- **Met — async projection worker (engine `:876-887`)** + `_fathomdb_vector_rows.kind` coverage tracking +
  `verify_embed_db` gate + rebuild-durable projections + RRF partial-dense under-ranking.
- **Met — edge-level temporal validity:** `canonical_edges.t_valid`/`t_invalid` (ISO-8601, inline
  `datetime('now')`) — the shape node-validity reuses.

**Net-new (must be built; audit A1–A14):** the **existence axis** (`pending→active→deleted→purged` enum +
`reason` column + transition table + `purge`/hard-erase + `secure_delete`/`VACUUM` + edge referential
stubs); **node-level validity** (`valid_from`/`valid_until` + bound `:now` + `valid_as_of`); **composable
read-mode relax-flags** + `ReadView` + read-mode uniformity on `get`/`list`/`neighbors`; the **projection
registry** + the **EAV attribute store + property-FTS** it projects from (only `body`-FTS exists today);
`dense_readiness` + the atomic readiness-flip; an **opaque surrogate `logical_id`** for anonymous nodes;
the `SearchHit.id → logical_id` (F-8a) swap.

## 4. ★ Memex coordination points (required focus — enumerated)

OPP-12 is a **two-repo breaking contract**. Every point where FathomDB 0.9.x and Memex 0.5.x must move
together, with owner-per-side, gating, and the coordination artifact. **No Memex file is written by this
draft or by FathomDB; all Memex-side work is Memex-owned and HITL-gated (never push Memex without a
per-push directive).**

### (a) C-1 — `EntityTypeSpec` (Memex) *drives* `ProjectionSpec` (FathomDB) — **RATIFIED (HITL 2026-07-07)**

> **STATUS: RATIFIED (HITL coreyt, 2026-07-07)** — build ≠ adopt (0.9.1 build authorized separately, §7.8). The
> C-1 co-design ran as a bounded **two-shot loop** and closed with **no open
> co-design questions** (both items once flagged for P2·S0 — apply-atomicity, defaults — were resolved at the
> contract level). Full contract: `dev/design/record-lifecycle-protocol/OPP-12-C1-converged-contract.md`
> (FathomDB seed for the P2·S0 ADR) + Memex `dev/design/entity-schema-registry/ADR-C1-eav-projection-lifecycle.md`;
> thread on `OPP-12-sub-ledger.jsonl` (seq 1–5). One ratification from HITL closes it as a converged design
> (build ≠ adopt).

- **What:** one declarative registration flow. Memex's `EntityTypeSpec` drives the engine's `ProjectionSpec`;
  the **engine is the sole projection authority** (idempotent `configure_projections` diff/backfill + explicit
  `drop`), and Memex runs only the pure `derive` — no projection state on the Memex side.
- **Owner split:** Memex owns the type + semantics + declarations (`EntityTypeSpec`/`AttributeSpec`,
  `register()`, the promotion gate); FathomDB owns the projection *mechanism*
  (`configure_projections`/`ProjectionSpec`/`ProjectionRole`, the EAV store + property-FTS, `dense_readiness`).
- **Converged contract (highlights):** one **engine-owned EAV + property-FTS** (Memex nested-dict interim →
  engine EAV at 0.9.1; breaking, **no data migration**); **Memex persists the spec, the engine `ProjectionSpec`
  is a derived boot cache** (drift resolves *to* the Memex spec — one durable source, no two-registry sync);
  the engine needs **no `provisional` concept** (cost-tiering / rankable-deferral / promotion all expressed via
  role-declaration timing + `drop`); `rankable` is **graceful-absent** until F9 (~0.8.16); `id_prefix` = IdSpace,
  engine mints the value (typed carrier = C-2/TC-8, not needed for C-1). **Apply atomicity = persist-first +
  idempotent + boot-heal** (no cross-step txn); **defaults = engine defaults** (bge-small / FTS5 tokenizer).
- **Forward-compat already landed (the time-critical win):** the optional `AttributeSpec.index: Set[ProjectionRole]`
  role hint — the one **break-if-late** field — is **frozen inert in Memex Commission C** (memex `95ed450`), so
  Commission C keeps building without foreclosing C-1.
- **Gating / co-land:** FathomDB's registry is **P2·S-F (0.9.1)**; Memex's registry rides its own **R-A**
  (persist + boot-reload). **They co-land at 0.9.1** with the Memex spec as the single durable source. **P2·S0 is
  now implementation-only** (the FathomDB-internal SQLite txn boundary) — the co-design itself is done. This
  **elevated the old entity-schema-registry ↔ projection co-design from soft to hard**, and it is now resolved.

### (e) Memex facade simplification — retire the 3 liveness encodings (CR-056/057/060 resolved)

- **What:** OPP-12 resolves **CR-056 / CR-057 / CR-060**, so Memex **deletes its three hand-rolled not-live
  encodings** and composes its labels as *predicates* over the engine axes. The three encodings, verified
  in `src/memex/fathom_store.py`:
  1. **governed retire tombstone** `__memex_retired__` (`:187`, supersede-by-rewrite, read-side filtered) →
     existence `deleted` **∧ reason = governance** (`retired`);
  2. **op-store deleted tombstone** `__memex_deleted__` (`:618`, appended tombstone row) → existence
     `deleted` (user) via the op-store's lifecycle;
  3. **knowledge soft-delete** `deleted_at` (`:2923`, `:2999`, `:3054`, `:3345`) → existence `deleted` +
     the `valid_*` / `ReadView` visibility surface.
- **Composed labels** (per `structural-lifecycle-contract.md` §7): `live = active ∧ is_latest ∧ in-force ∧
  ¬view-excluded`; `retired = deleted ∧ reason=governance`; `deleted(user) = deleted ∧ reason=user`;
  `expired`/`archived` = view-policy over `active`; `superseded = ¬is_latest` (a logical query, not a value).

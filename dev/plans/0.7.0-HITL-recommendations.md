## HITL responses (locked)

### Q1 — Revised budgets

**AC-012: Option A — p50 ≤ 60 ms, p99 ≤ 150 ms** ✅ LOCKED

Measurement-pinned against canonical-CI 57/135 + ~5% headroom.
Tightens the original draft (50/200) on p99, loosens on p50 to

**AC-020: Option A in spirit, but expressed in absolute ms — NOT speedup ratio** ✅ LOCKED

The 5.33× ratio (`1.5 / 8`) framing assumed an 8-core canonical
runner; canonical CI is 4-core. The right expression is absolute
ms envelopes that catch regressions independent of core count:

- **Sequential ≤ 250 ms** at 1M corpus on canonical CI.
  (Measured 176 ms at W4.1-stacked-O1; 42% headroom.)
- **Concurrent ≤ 80 ms** at 1M corpus on canonical CI, 8
  reader threads.
  (Measured 56 ms at W4.1-stacked-O1; 43% headroom.)

The existing `1.5 / 8` ratio in `tests/perf_gates.rs:245` is
**replaced**, not preserved. The AC-020 RED-test asserts the two
absolute bounds above instead.

Source comment in the test should note: *"absolute-ms envelope
pinned against the locked PCACHE2-stack canonical-CI measurement;
the prior `concurrent ≤ sequential * 1.5/8` ratio was a proxy for
'cores should multiply throughput' that broke down on the 4-core
canonical runner. If canonical CI moves to ≥8-core hardware,
re-measure and tighten."*

**AC-013 / AC-019: not descoped — finish the work** ✅ LOCKED

Canonical-CI N=1M dispatch under W4.1-stacked-O1 stack completed
2026-05-27 (workflow run 26482279276, ~169 min wall time):

- AC-013 p50 = **2048 ms**, p99 = **2327 ms** (seed cost 562 s).
- AC-019 baseline_p99 = 2096 ms, stress_p99 = 8388 ms,
  bound = 20964 ms (10× baseline). **AC-019 passes** its
  bound expression (stress within 10× of baseline).

**Major finding — AC-013 is 25–40× over the ADR-draft 80/300
budget.** The bottleneck is per-query O(N) full-scan through the
`vec0` virtual table — no ANN index. At dim=768 × 1M rows, every
query examines all 1M vectors. PCACHE2 / MEMSTATUS / pragmas
cannot move this — the bottleneck is the linear ANN scan, not
allocator / pcache contention.

**HITL decision 2026-05-26: land a vector-retrieval cost fix
in 0.7.0.** Expands 0.7.0 scope from "perf-only single-thrust"
to "perf + vector-cost".

**Research returned 2026-05-27** (`dev/notes/0.7.0-vector-cost-research.md`):
Tier 1 wins. Binary quantization (`bit[768]`) with f32 rerank
on the existing `sqlite-vec 0.1.7` (no upstream upgrade) is the
load-bearing lever. Expected: AC-013 p50 50–100 ms, recall ≥
95 % @ k=10. Tiers 2 (vectorlite) and 3 (broader ANN) are not
pursued in 0.7.0.

**HITL responses (locked 2026-05-27):**

1. **Recall floor**: ≥ 90 % recall@10 vs f32 ground truth on the
   AC-013 fixture (initial floor; tighten in later release).
2. **Embedder**: KEEP existing. Tier-1 #3 (smaller embeddings)
   and #5 (matryoshka) out of scope for 0.7.0.
3. **Metadata + partition_key schema**: YES, ship in 0.7.0 —
   bundled into Pack 1 of the implementation.

**Implementation: 2 packs within 0.7.0.**

- **Pack 1** — schema migration + ingest double-write. Adds
  `bit[768]` sibling column + metadata columns + partition_key
  to the vec0 schema; writer double-writes f32 + bit. No reader
  change yet; AC-013 stays pinned at the unindexed 2048/2327
  numbers. Correctness gate: migration round-trip test.
- **Pack 2** — query-path swap. `read_search_in_tx` becomes
  two-phase (bit KNN top-K → f32 rerank). Re-pins AC-013 to
  ≤ 80 / 300 ms and adds the recall-floor RED test
  (≥ 0.90 @ k=10).

Architectural-lever disposition: the binary-quant + rerank
query shape is a **data-encoding change**, not a second
architectural lever (PCACHE2 already occupies that slot in
0.7.0). Both packs ship in 0.7.0. Fallback if a future reviewer
disagrees on the lever counting: Pack 1 stays in 0.7.0;
Pack 2 slips to 0.7.1.

Implementation handoff: `dev/plans/prompts/0.7.0-PERF-VECTOR-QUANT-HANDOFF.md`.
AC-013 / AC-019 budgets in the revised-budgets ADR remain
placeholders until Pack 2's canonical-CI measurement returns.

### Q2 — AC-020 architectural lever

**PCACHE2 (Option 1)** ✅ LOCKED

#### Clarification on "how PCACHE2 combines with other tested settings"

The locked production stack is **W4.1-stacked-O1**, which is one
bundle of co-enabled knobs:

| Knob | What it is | How it composes with PCACHE2 |
|---|---|---|
| `SQLITE_CONFIG_PCACHE2` | Custom page-cache implementation (`pcache2.rs`, ~360 LOC). Replaces SQLite's default `pcache1`. | **The lever.** Process-start FFI. |
| `SQLITE_CONFIG_MEMSTATUS=0` | Disables SQLite's allocator-statistics mutex. | **Stacks additively.** Independent contention surface from the pcache1 mutex; both gains compound. |
| `page_size=8192` (writer pragma) | Increases SQLite B-tree page size from default 4K to 8K. | **Stacks additively.** Halves the number of pcache fetches per query; multiplies PCACHE2's per-fetch savings. |
| Reader pragmas: `cache_size=-262144`, `mmap_size=268435456`, `temp_store=MEMORY`, `synchronous=NORMAL` | Standard SQLite tuning: 256 MB cache cap, 256 MB mmap, in-memory temp tables, single-writer durability. | **Stacks additively.** Increases pcache hit rate (so PCACHE2's fast path runs more often) and reduces I/O sync overhead. |
| `SQLITE_CONFIG_PAGECACHE 1GB pre-alloc` | Pre-allocates 1 GB of pcache1 backing memory at process start. | **MUTUALLY EXCLUSIVE with PCACHE2.** PCACHE2 replaces the entire pcache subsystem; PAGECACHE has no effect when PCACHE2 is installed. The prototype omits PAGECACHE under PCACHE2; the production graduation should make this explicit (refuse to install both, or document that PAGECACHE is silently ignored). |
| `HashSet<key>` (unpinned eviction candidates inside PCACHE2) | Internal data structure for O(1) eviction lookup. | **Implementation detail of PCACHE2, not a separate lever.** The prototype uses a `HashSet` (best-effort eviction); production graduation should consider a doubly-linked-list LRU for better eviction quality under cache pressure. The AC-020 / AC-012 gates do not exercise tight cache pressure so the prototype suffices for the contract. |
| `LIMIT k=1000` (AC-012 query shape, Q4 lever) | Caps result-set size on the search verb. | **Independent of PCACHE2.** Helps AC-012 by reducing rows materialised; orthogonal to allocator/pcache contention. Recommended as opt-in per Q4 / AC-071 above. |

**HITL also decided 2026-05-26: stay with W4.1-stacked-O1 as the
locked 0.7.0 stack; do NOT pursue proper-LRU or `page_size=16384`
(alone or combined) in 0.7.0.** Rationale captured in
`dev/notes/pcache2-followups.md`: locked stack lands GREEN with
material headroom; the measured 16K-solo prior (W4.0g) regressed
on AC-020; the LRU benefit is invisible on the AC fixtures and
needs production-workload telemetry to size honestly. Both
variations are eligible for a `0.8.0-PERF-PCACHE2-PRODUCTION`
slice keyed on the trigger conditions in the followups note.

**Action items for the PCACHE2 production-graduation slice
(`0.7.0-PERF-PCACHE2`):**

1. Replace `FATHOMDB_PERF_SQLITE_PCACHE2=1` env gate with a
   runtime-config field on `EngineRuntime::open` (whitepaper §13
   lookaside-config pattern).
2. Co-enable the full W4.1-stacked-O1 stack in `init_perf_*` /
   the corresponding production paths: MEMSTATUS_OFF + page_size=8192
   writer pragma + the four reader pragmas. These should be the
   default, not opt-in, under the locked release.
3. Ensure `SQLITE_CONFIG_PAGECACHE` is not also installed (it has
   no effect under PCACHE2 but the unused env var should be
   removed from the production path to avoid confusion).
4. Consider replacing PCACHE2's internal `HashSet` eviction with
   a proper LRU. Not required for the AC gates; required only
   under tight cache-pressure workloads. Land as a follow-up,
   noted in `dev/notes/performance-whitepaper-notes.md` §13.
5. Pin the AC-020 RED-test bounds to the absolute-ms envelopes
   above (per Q1 AC-020 lock).
6. Pin the AC-012 RED-test budgets to 60/150 (per Q1 AC-012 lock).

### Q4 — Top-K LIMIT shape

**Additive opt-in (default unlimited; opt-in `--top-k` cap)** ✅ LOCKED

Per the W4.1-stacked-O1 measurement, AC-012 closes on canonical
CI without any LIMIT changes (the `SEARCH_LIMIT=1000` env was

knob for callers that want bounded latency, not a 0.7.0

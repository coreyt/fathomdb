## Numbering — N-mod-5 with reserved gaps (HITL directive)

Planned slices are numbered **only in multiples of 5**. The gaps
(`1–4, 6–9, 11–14, 16–19, 21–24, 26–29, 31–34, 36–39, 41–44`) are **reserved on
purpose** — they are the **insertion mechanism** for unplanned follow-on work a
slice reveals while executing.

**Reserved-gap policy:**
- If Slice **N** reveals work that must land before the next planned slice (N+5),
  that work becomes Slice **N+1, N+2, N+3, or N+4** — a **fully-orchestrated slice
  in its own right** (its own prompt, its own worktree from a fresh baseline, its
  own `output.json`, its own codex review, its own CLOSED block) — **NOT** an
  ad-hoc patch.
- **Reserved-gap ≠ fix-N.** A §6 *fix-N* re-enters the **same** slice's *existing*
  worktree/branch to clear a BLOCK/CONCERN on **that slice's own deliverable**. A
  *reserved-gap slice* is **new adjacent scope** (fresh baseline, fresh worktree).
- **HALT rule:** if a gap band (N+1..N+4) **fills** and more follow-on still
  remains, the slice was mis-scoped → **HALT to HITL** (§1.5 invariant 3). Never
  overflow into the next mod-5 number.

Pre-identified reserved-gap candidates (see per-slice "Reserved follow-on" in
`0.8.0-implementation.md`): Slice 16 (shadow vec0/FTS5 reconciliation vs
`superseded_at`), Slice 13 (G10 `status` population source), Slices 11/12
(G12-importance vec0 reshape), Slices 27/28 (Q3 supersede-by-new-REQ trace
reconciliation / Q5 Rust-facade allowlist pin).

## Cross-cutting Definition of Done (X1/X2/X3 — bind EVERY slice)

Every slice's success criteria **and** the codex review checklist additionally
assert (see `0.8.0-implementation.md` § "Cross-cutting requirements"):

- **X1 — SDK parity + functional harnesses.** Any surface/behavior/error/result
  change lands in **both** the Python and TypeScript SDKs in the same slice, each
  with a live functional harness (write→search→retrieve→admin + cross-binding
  equivalence — not just symbol presence). **Slice 5 stands the harnesses up;**
  every later behavior slice extends them.
- **X2 — `mkdocs build` stays green.** Any `docs/` page add/rename updates
  `mkdocs.yml` `nav` in the same slice; the slice self-check runs `mkdocs build`
  (strict where feasible). **Slice 40 enforces it as a release gate.**
- **X3 — docs updated per slice + `dev/DOC-INDEX.md` maintained.** Every slice
  updates the relevant `dev/` + `docs/` docs in its Definition of Done **and
  updates `dev/DOC-INDEX.md` in its closing docs commit** (the per-slice-update
  rule — mirrors §12.4 plan-as-state-machine discipline applied to docs).

`runs/STATUS-0.8.0.md` carries an **X1/X2/X3 per-slice column** so a gap is
visible at a glance.

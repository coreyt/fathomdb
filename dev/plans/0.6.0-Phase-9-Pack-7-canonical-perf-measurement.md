# Phase 9 Pack 7 — Canonical-runner perf measurement + bulk-vec seed

Pack 7 closes the four perf gates that 0.6.0 ships as DEFERRED:

- **AC-012** — text-query latency on FTS5 path (1M-doc fixture).
- **AC-013** — vector-retrieval latency.
- **AC-019** — mixed-retrieval stress workload tail.
- **AC-020** — reader-pool perf gate.

Pack D landed protocol-complete, `AGENT_LONG`-gated harnesses for
AC-012 / AC-013 / AC-019 in `tests/perf_gates.rs`. Pack 6.G closed
the AC-020 lever-search and concluded the residual is in
canonical-SQLite internals (`pcache1` mutex, WAL atomics) not Rust
topology. Pack 7 is the follow-on that takes measurement to a
canonical x86_64 tier-1 CI runner and lands the engine-surface
changes the canonical-scale fixtures require.

Budgets are pinned to ADR values. Pack 7 measures and remediates;
it does NOT relax acceptance budgets.

## Driver

Two distinct things blocked Pack D close on the dev-runner
(aarch64 Linux 5.15-tegra):

1. **Host-class mismatch.** AC-012 p50 = 29.7 ms vs ADR 20 ms at
   N=100,000 on aarch64 Tegra. Acceptance budgets were measured
   against the canonical x86_64 tier-1 CI runner. Re-measurement
   on canonical hardware is required before any budget conversation.
2. **Engine-surface seed-cost pathology.** `vec0` single-row
   insertion via `engine.write()` + projection-runtime drain
   measured at ≈ 5.5 inserts/sec on the dev runner. Even at the
   canonical CI's expected ~3× speedup, seeding 1,000,000 vectors
   would consume ~17 hours per run — well outside any realistic
   CI budget. AC-013 / AC-019 cannot be measured at canonical
   scale without a bulk-vec seed seam.

AC-020's deferral driver is independent and already documented in
`dev/plan/0.6.0-Phase-9-Pack-5-performance-diagnostics.md` §13:
canonical SQLite cannot eliminate the residual `pcache1` mutex
share without `SQLITE_CONFIG_PCACHE2`, WAL2, or a vendor-SQLite
swap.

## Scope

### Track 1 — Canonical-runner re-measurement (low blast radius)

Run the existing Pack D harnesses on a canonical x86_64 tier-1
runner under `AC_FULL_SCALE=1`:

- AC-012 N=1,000,000, capture p50 / p99.
- AC-013 N=1,000,000 (subject to Track 2 — see below), capture
  p50 / p99.
- AC-019 mixed workload, capture stress-mode p99 against
  `baseline_p99 × P-STRESS-MULT` floor.

Deliverable: `dev/notes/perf-canonical-runner-2026-MM.md` with
the canonical numbers, runner spec (CPU, OS, kernel, SQLite
version), and a green/red verdict per gate.

If GREEN, Track 4 lifts deferral. If RED, Track 3 (root-cause
work) opens.

### Track 2 — Bulk-vec seed seam

Engine surface change. AC-013 / AC-019 cannot be measured at
canonical scale without one.

Options (decide via measurement, not theory):

- **Pre-encoded `vector_default` seed path.** Test-only API that
  bypasses the projection runtime and inserts pre-computed
  vectors directly into `vector_default` + `_fathomdb_vector_rows`.
  Lowest engine-surface impact. May not exercise the same write
  path the production embedder hits — document the divergence.
- **Batched vec0 INSERT with projection-runtime suppression.**
  Adds a `bulk_seed_for_test` engine method that holds the
  projection scheduler frozen, batches N writes inside one
  transaction, then drains. Closer to production write path.
- **External pre-built `.fathomdb` corpus snapshot.** Build the
  1M corpus once, check the binary into a release-asset bucket,
  CI fetches before the long-run gate. Operationally heavy;
  invalidates on schema bump.

Acceptance: 1M-vector seed completes in ≤ 30 minutes wall-clock
on the canonical runner.

### Track 3 — AC-020 root-cause remediation (if AC-020 stays RED post-Track 1)

Per Pack 5 / 6.G whitepaper §13, the canonical-SQLite levers left
on the table:

- `SQLITE_CONFIG_PCACHE2` — custom page-cache allocator install,
  bypasses `pcache1` mutex.
- WAL2 — separate write-ahead logs per writer; needs vendor-SQLite
  patch (not in upstream).
- Reader/writer physical separation — second connection pool,
  pinned to read-only handles on a replicated WAL.
- Vendor-SQLite swap (libSQL, sqlite3mc) — broader change, larger
  blast radius; defer until canonical numbers prove the upstream
  ceiling is the actual blocker.

Each lever lands as a numbered subpack (7.1 / 7.2 / …) with the
same RED→GREEN measurement discipline Pack 5/6/6.G enforced.

### Track 4 — Lift the deferral

Once a gate measures GREEN at canonical scale on canonical
hardware, lift its DEFERRED status:

- Remove the deferral note in `dev/test-plan.md`.
- Re-add the gate to the long-run CI pipeline as a non-deferred
  acceptance binding.
- Update `dev/plans/0.6.0-Phase-10-plus-risks.md` if the gate
  was named there.
- Land traceability evidence per AGENTS.md § 5.

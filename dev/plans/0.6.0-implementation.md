# 0.6.0 Implementation Plan

This plan starts after the `0.6.0-design-frozen` tag. The frozen contract is
the locked corpus under `dev/`: needs, requirements, acceptance, architecture,
designs, interfaces, test-plan, traceability, security-review, and PSD.

Implementation rule: **TDD is mandatory**. Each implementation slice starts
with a failing test that cites at least one `AC-*` id and one relevant ADR or
design/interface doc. Red -> green -> refactor. Do not change a test to make
the implementation pass; fix the code. If a locked doc is wrong, stop and get
explicit HITL approval before amending it.

## Contract Inputs

- Design freeze tag: `0.6.0-design-frozen`.
- Acceptance oracle: `dev/acceptance.md`.
- Test identity map: `dev/test-plan.md`.
- Public interfaces: `dev/interfaces/{rust,python,typescript,cli,wire}.md`.
- Module boundaries: `dev/architecture.md` and `dev/design/*.md`.
- Security hardening additions: REQ-060..REQ-066 and AC-064..AC-070.

## Phase 5 — Interface Stubs

Goal: make public surfaces compile and fail/pass against contract-shape tests
without pretending the engine behavior is implemented.

Scope:

- Rust facade types, constructors, error enums, and public docstrings.
- Python and TypeScript five-verb SDK shape, including config and error roots.
- CLI root/help shape for `doctor` and `recover` only.
- Wire/on-disk sentinel constants and schema-version exposure.
- Explicit `unimplemented` / typed placeholder errors where behavior is not yet
  real.

First failing tests:

- `src/rust/crates/fathomdb/tests/public_surface.rs` for the Rust facade shape
  in `dev/interfaces/rust.md`. Do not cite AC-057a; Rust is outside the
  Python/TypeScript five-verb parity set.
- `src/python/tests/test_public_surface.py` for AC-057a surface presence and
  recovery-verb non-presence only.
- `src/ts/tests/public-surface.test.ts` for AC-057a surface presence and
  recovery-verb non-presence only.
- `src/rust/crates/fathomdb-cli/tests/operator_cli.rs` for root/help smoke
  shape only, not full invocability, JSON, or exit-code behavior.

Exit gate:

- `./scripts/agent-verify.sh` passes.
- Public API docs exist for every exported Rust/Python/TS symbol introduced.
- No behavior test claims durability, persistence, recovery, or retrieval
  semantics that are not yet implemented.

## Phase 6 — Schema, Errors, and Open Path Foundation

Goal: make the narrow schema/migration/lock-open foundation real and provide
the shared error vocabulary every later slice consumes.

Scope:

- `fathomdb-schema`: schema version sentinel, migration step registry, accretion
  guard target.
- `fathomdb-engine::errors`: top-level `EngineOpenError`, `EngineError`,
  corruption detail types, migration errors, sanitized foreign-cause display.
- `runtime`: open report, sidecar-lock acquisition order, PRAGMAs, open failure
  cleanup, close idempotence.
- Compatibility refusal for 0.5-shaped databases.
- Minimal writer/scheduler/embedder placeholders required by the locked
  `Engine.open` sequence, without vector work execution.

First failing tests:

- `src/rust/crates/fathomdb-schema/tests/migrations.rs` for AC-046a/b/c,
  AC-049, AC-070.
- `src/rust/crates/fathomdb-engine/tests/compatibility.rs` for AC-047,
  AC-048, AC-048b.
- `src/rust/crates/fathomdb-engine/tests/lifecycle_reliability.rs` for
  AC-022a/b/c and second-open lock behavior without pending vector work.
- Error-display tests for AC-035b and AC-069.

Exit gate:

- A real SQLite database file is created/opened/migrated; no database mocking.
- A second engine open fails with typed `DatabaseLocked` without modifying the
  file.
- Failed migration preserves prior `user_version`.
- Full AC-024a coverage with pending vector work remains deferred until Phase 9.

## Phase 7 — Writer, Typed Writes, and Op-Store

Goal: make canonical writes transactional, cursor-bearing, and typed, with
op-store collections in the same SQLite file.

Scope:

- Single writer thread and typed `PreparedWrite` execution.
- Canonical node/edge/chunk tables and cursor allocation.
- Op-store collection registry, `append_only_log`, `latest_state`, and JSON
  Schema save-time validation.
- Security hardening for bounded validation and external `$ref` rejection.

First failing tests:

- `src/rust/crates/fathomdb-engine/tests/op_store.rs` for AC-061a/b/c,
  AC-062, AC-063a/b.
- Public-surface cursor tests for AC-059a/b.
- Security tests for AC-064 and AC-065.
- Non-AC writer sanity tests may be added here; AC-011a/b remain perf gates.

Exit gate:

- Canonical writes and op-store rows commit in one transaction.
- `WriteReceipt { cursor }` is monotonic and satisfiable by read-side
  `projection_cursor`.
- Validation failures leave no canonical or op-store residue.

## Phase 8 — Lifecycle Observability and Reader Pool

Goal: expose the operator/user-visible observability contract and establish
reader concurrency before retrieval complexity arrives.

Scope:

- Host-routed tracing/lifecycle events.
- Counter snapshots and delta behavior.
- Per-statement profiling and slow-statement events.
- Reader pool with WAL read transactions and no single-reader serialization.

First failing tests:

- `src/rust/crates/fathomdb-engine/tests/lifecycle_observability.rs` for
  AC-001..AC-010.
- `src/rust/crates/fathomdb-engine/tests/lifecycle_reliability.rs` for AC-021.

Exit gate:

- No log files are written without a host subscriber.
- Counter reads do not perturb counters.
- Concurrent read/admin DDL tests produce zero `SQLITE_SCHEMA` warnings.

## Phase 9 — Retrieval, Vector, Projections, and Scheduler

Goal: implement user-visible search behavior and background projection
semantics once canonical writes and readers are stable.

Scope:

- FTS5 text retrieval and injection-safe query grammar.
- Vector storage, LE-f32 BLOB encoding, vec0 integration, and dimension checks.
- Embedder API dispatch, identity persistence, warmup, no implicit downloads.
- Scheduler, projection cursor, projection failure records, restart semantics,
  and explicit regenerate workflow hook for CLI.
- `engine.drain` bounded-completion control method.

First failing tests:

- `src/rust/crates/fathomdb-engine/tests/projection_runtime.rs` for AC-029,
  AC-030a/b/c, AC-031, AC-032a/b, AC-033, AC-063a/b.
- `src/rust/crates/fathomdb-engine/tests/perf_gates.rs` for AC-012..AC-020,
  initially gated behind explicit perf labels.
- Property tests for vector BLOB round-trip and projection freshness invariants.
- Security test for AC-066.
- Pending-vector reliability tests for AC-024a and AC-025.

Exit gate:

- Canonical writes complete under projection stall.
- Vector dimension mismatch rolls back without vec0 writes.
- Failed projection jobs are durable terminal states until explicit regenerate.
- Reader-pool perf gate AC-020 is implemented here but remains release-evidenced
  in Phase 12.

## Phase 10 — CLI Recovery and Operator Tooling

Goal: make the operator-only surface real without leaking recovery operations
into SDKs.

Scope:

- `fathomdb doctor` read-only verbs and JSON / pretty output.
- `fathomdb recover --accept-data-loss` mutation boundary and lock posture.
- Safe export, SHA-256 manifest, physical rebuild from canonical state, and
  projection rebuild.
- CLI exit-code classes and no-SDK-recovery tests.

First failing tests:

- `src/rust/crates/fathomdb-cli/tests/operator_cli.rs` for AC-035d..AC-045,
  AC-058.
- `src/rust/crates/fathomdb-cli/tests/recovery_cli.rs` for AC-026..AC-028c,
  AC-044, AC-063c.
- Binding public-surface tests for AC-041.

Exit gate:

- Every `doctor` verb is invocable with `--json`.
- `recover` refuses mutation without `--accept-data-loss`.
- Runtime SDKs expose no doctor/recover operation.

## Phase 11 — Bindings Parity and Packaging

Goal: wire Python and TypeScript to the Rust engine and prove language parity.

Scope:

- PyO3 binding: sync API, GIL release around blocking engine work, Python error
  classes, string validation.
- napi-rs / TypeScript binding: Promise API, binding-owned handoff pool,
  `FathomDbError` root, string validation.
- Rust facade stability and sibling crate version-skew checks.
- Package version single source of truth.

First failing tests:

- `src/python/tests/test_public_surface.py` for AC-057a..AC-060b, AC-067,
  AC-068a/b.
- `src/ts/tests/public-surface.test.ts` for AC-057a..AC-060b, AC-067,
  AC-068a/b.
- Release checks for AC-051a/b, AC-052, AC-053.

Exit gate:

- `pip install -e 'src/python[dev]'` remains the Python native-module loop.
- Python and TS expose the same application verbs and equivalent typed errors.
- FFI panic and string-boundary tests pass without aborting the host process.

## Phase 12 — Durability, Performance, Security, and Release Gates

Goal: turn the full acceptance set into release evidence.

Scope:

- Power-cut, OS-crash, 1 GB recovery, and corruption matrix harnesses.
- Performance gates for write throughput, text query, vector retrieval,
  projection freshness, drain, mixed retrieval, and safe export.
- Security fixtures: no-listen syscall capture, netns-deny-egress, FTS5
  injection, AST no-shim scanner, removal-detect linter.
- Atomic multi-registry dry run and registry-installed wheel smoke.

First failing tests:

- `src/rust/crates/fathomdb-engine/tests/durability_soak.rs` for
  AC-034a..AC-035c.
- `src/rust/crates/fathomdb-engine/tests/perf_gates.rs` for AC-011a..AC-020.
- Release tests under `dev/release/tests/` for AC-050a..AC-056.

Exit gate:

- `./scripts/agent-verify.sh` passes.
- Full opt-in acceptance gate passes.
- Published/registry-installed wheel open/close/exit smoke passes before
  declaring the release done.

## Cross-Phase Rules

- Each PR body cites the AC ids and ADR ids it implements.
- Keep `dev/traceability.md` coarse until scaffold paths stabilize; deepen it to
  explicit `T-*` ids as tests land.
- Keep perf/soak tests out of the normal agent loop unless the slice touches
  the gated behavior; normal loop remains `./scripts/agent-verify.sh`.
- Do not introduce 0.5.x compatibility shims or migrations.
- Do not implement postponed 0.8.0 security hardening unless HITL explicitly
  re-targets it to 0.6.0.
- If a slice discovers missing acceptance coverage, stop and amend
  `acceptance.md` / `test-plan.md` through HITL before implementing behavior.

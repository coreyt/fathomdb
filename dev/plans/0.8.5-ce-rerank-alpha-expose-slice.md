# Slice spec — Expose tuned CE-rerank α (+ pool_n + ce_score) in the engine

**Version:** 0.8.5 (steward to confirm the label) · **Date:** 2026-06-25 · **Type:** engineering, $0, no new HITL gate
**Closes:** OPP-7 (Memex ledger) · portfolio EXP-0 (Tier-A #1) · M5 — makes the parity win **shippable
(opt-in)** · M3 precision lever
**Does NOT by itself ship the win:** this exposes an opt-in knob (default stays 0.3); **no shipped caller is
wired to α=1.0 yet** — wiring the agentic-answer/memory path to opt in is a separate follow-up, and the
underlying M5 parity result is itself provisional/underpowered (354/606, take-as-is).
**Provenance:** measured-provisional result (0.8.3 CE-rerank α=1.0 reblend = Mem0 parity-or-better,
`runs/0.8.3-rerank-accuracy-n606-VERDICT.md`; tuning `runs/0.8.3-rerank-tune-FINDINGS.md`).

> **One line.** The Mem0-parity CE-rerank configuration (α≈1.0, narrow pool) is currently reachable only
> through an **eval-side** `ReblendRerankAdapter` (a `score=0` trick around the shipped `rerank()`). This
> slice promotes α (and the CE score) into the engine surface so the parity win is **shippable**, not
> eval-only — **without changing the default behavior**.

---

## 0. Why this is safe to do now (no new HITL gate, no spend)

- It rests on an **already-measured provisional result**, not a new run. The 0.8.3 HITL ruling was
  **take-as-is, no further priced spend** (`runs/0.8.3-resolution-verdict.md`); this slice spends **$0**
  (pure engineering) and does **not** attempt to finish/register the α=1.0 run — that would be a separate,
  new HITL spend decision and is explicitly **out of scope**.
- The win is **already proven reachable without a Rust rebuild** via the eval `score=0` trick
  (`eval/rerank_tune_probe.py:71` `recover_ce_norm`), so this is a low-risk promotion of a known-good
  computation into a native, governed knob.

## 2. Goal & non-goals

**Goal.** Make `(alpha, pool_n)` caller-settable and expose the per-candidate **CE score** on the shipped
search/rerank surface, so the agentic-answer path can opt into the measured-parity α≈1.0 config and/or
re-blend client-side — **default behavior byte-identical to today.**

**Non-goals (do not do in this slice).**
- Do **not** change the default α (stays 0.3) or any default ordering.
- Do **not** finish/register the M5 α=1.0 run (closed under take-as-is; new HITL decision required).
- Do **not** touch retrieval recall (that's EXP-A), the embedder, or the quant path.
- Do **not** build the router.

## 3. Change surface (minimal)

1. **`ce_rerank` (lib.rs:4936)** — add params `alpha: f64`, `pool_n: usize`; replace the `const ALPHA`
   (`:4973`) with `alpha`; use `pool_n` for the rerank-pool size (replacing the implicit
   `rerank_depth`-as-pool at `:4951`). Validate/clamp `alpha ∈ [0.0, 1.0]`; `pool_n` clamped to `hits.len()`.
2. **`SearchHit`** — add `ce_score: Option<f64>` (the `sigmoid(ce_logit)` = `ce_norm`). Populate for hits
   in the reranked pool; **`None`** for the unreranked remainder and for the depth=0 / no-CE-model
   soft-fallback path (preserves byte-identity).
3. **Thread through** `rerank_fused` (`:4851`), `search_reranked`, `rerank_passages` (`:4894`), and
   `ReaderRequest::Search` — add `alpha` (default 0.3) + `pool_n` (default = `rerank_depth`, preserving
   current pool==depth semantics when unset).
4. **Bindings** — add `alpha: Option<f64>` + `pool_n: Option<usize>` kwargs to `Engine.search` and
   `rerank()` in pyo3 (`:639–690`, `:1256–1308`) and napi (`:763, 812–818`); include `ce_score` in the
   returned dict/object. Update `_fathomdb.pyi` (`:100–109`, `:207–223`) signatures + `SearchHit`/return
   shape. Defaults preserve the current call sites.
5. **(Optional, recommended)** lift the `recover_ce_norm`/reblend logic from the eval adapter into a shared
   engine helper so the eval-side duplication disappears (eval then calls the native α path). Keep if it
   doesn't bloat the slice; otherwise leave the adapter and just point it at the native param.

## 5. eu7 re-clear check (the one attached caveat)

Reranking is **post-retrieval** — it reorders already-retrieved hits and touches **no** embedding or quant
path, so eu7 fidelity is structurally unaffected. **Guard, not expectation:**
- If review confirms no embed/quant/SUT path is touched (expected) → **no eu7 re-run required**; note it in
  the closure.
- If the change *does* trigger a re-embed or perturbs the vector-stage SUT (it must not) → re-run
  `eu7_real_corpus_ac.rs` (`AGENT_LONG=1`) and confirm the **one-sided gate still passes**
  (`ci_hi ≥ 0.90`; current 0.925). Reference: eu7 is **at-gate** today (`recall_gate.rs`,
  `0.8.x-parity-portfolio-strategy.md` M1).

## 6. Acceptance

1. `Engine.search`/`rerank` accept `alpha` + `pool_n`; `ce_score` returned per candidate.
2. **Default (α=0.3) path byte-identical** to pre-slice (regression pin green) — C6 factoid guard intact.
3. Engine `α=1.0, pool_n=10` == eval `ReblendRerankAdapter(1.0,10)` on the fixture (parity config native).
4. depth=0 / no-CE soft-fallback identity preserved; `ce_score=None` there.
5. eu7 unaffected (post-retrieval) — re-clear only if a re-embed was triggered.
6. Bindings (py/ts) + `.pyi` updated; doc note that **α=1.0 is opt-in for the agentic-answer/memory path**,
   default 0.3 protects naive factoid lookups.
7. codex §9 review clean (per program convention); RED-first demonstrated.

## 7. Open design questions (resolve in the slice, low-stakes)

- `ce_score`: populate **only** when reranked (saves compute, cleaner identity) — **recommended** vs always.
- `pool_n` default: `= rerank_depth` when unset (preserves today's pool==depth) — **recommended**.
- α out of range: **clamp** silently vs error — pick + test (lean clamp, to match `rerank_depth.min`).
- Whether to also expose the **raw CE logit** (not just `ce_norm`) — default **no** unless a caller needs it.
- Whether to fold `recover_ce_norm` into a shared engine helper (§3.5) — keep slice small; optional.

## 8. Sequence note

This is **EXP-0** in `dev/design/0.8.x-portfolio-features-and-experiment-tree.md`. It unblocks **EXP-A**
(recall *generation* for F2) and **EXP-B′** (the 3-stage joint tuning `candidate_k × pool_n × α × final_K`),
the critical path to the M5 F1/F2 surpass. **Note for EXP-B′:** the measured recall×α interaction is a
*conflict*, not a free compose — α=1.0 @ wide pool can *drop* recall (CE-confident distractors displace
base-favored gold), and a narrow pool_n=10 can't see deep-recalled gold (`0.8.3-rerank-tune-FINDINGS.md`).
So **`α`/`pool_n` must be per-feature configs the router carries** (this slice exposes them as caller knobs
precisely so EXP-B′ can set F2's tuple independently of F1's `(0.3, 10)` default). Productizing **C**
(`query --global`) and the F4 Fork-E decision are a parallel track, gated separately (HITL §7.2).

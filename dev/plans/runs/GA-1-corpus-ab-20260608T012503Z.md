## 3. So what actually moved 0.937 → 0.8710? (the eu7 code path, not the corpus)

The eu7 **harness** `measure_recall` (exclude-before + dedup-by-id GT, `RECALL_SEARCH_SLACK=5`,
`set_search_limit_for_test`) is **byte-identical** between v0.7.2 and slice-40 (diffed; same
lines). The PR-2c dual-method harness fix (`26b2f95`, 2026-05-31 07:55) + the search-limit seam
(`5b69568`, 2026-05-31 08:08) both landed **before** the 0.937 json was written (08:13), so the
0.937 anchor already used the *same NEW method* as slice-40. The harness is not the variable.

What **did** change is the **production `engine.search()` path that eu7 measures.** Between v0.7.2
and `slice-40-20260607T145013Z` there are **11 commits to `fathomdb-engine/src/lib.rs`**, including:

- `d28d204` — **G9 RRF fusion made the unconditional ranking** (+ rerank seam, G10 filtered-KNN,
  G12 recency). This is the load-bearing one: eu7's ground truth is the **pure exact-f32 *vector*
  top-10**, but 0.8.0 `search()` now returns an **RRF fusion of the FTS5 + vector branches**. A
  fused top-10 legitimately diverges from a pure-vector f32 top-10, which lowers the *measured*
  "recall@10" without any quantization-fidelity loss and without any corpus change.
- `3e5ae59` — G1 structured `SearchHit` + **step-11 FTS5 tokenizer upgrade** (changes the FTS
  branch that now feeds RRF).
- Slice 15 (`0227773`, G0 identity), Slice 30 (`0862ecb`, reader pool), Slice 20/31/33, etc.

The σ also moved (0.0116 → 0.0180) on an identical corpus+harness, consistent with a changed
per-query result distribution from a changed retrieval path — not a corpus effect.

## 4. Classification (evidence, not a ruling)

**(b) — code/measurement-path change, NOT a corpus effect. Caveat: likely an *intended*
0.8.0 behavior change (unconditional RRF fusion), not necessarily a quantization-fidelity defect.**

Reasoning by elimination, grounded in §2–§3:
- **(a) harder/larger corpus — RULED OUT.** Same corpus, same N=7,667, byte-identical files
  (mtime 2026-05-27, same dir for both runs). There is no larger/harder corpus.
- **(c) corpus-quality defect (near-dups/bad docs in an expansion) — RULED OUT.** No expansion
  reached the measured corpus; the only manifest drift is a stale qmsum checksum, and the
  qaconv/qasper additions produced no loaded files.
- **(b) — where the delta lives.** Identical corpus + identical eu7 harness ⇒ the only remaining
  variable is the **engine retrieval path** (11 lib.rs commits, notably G9 unconditional RRF
  fusion + the FTS5 tokenizer upgrade). This is a *code-path / metric-definition* change, not a
  corpus change. **Important nuance for B-1:** because eu7 measures recall@10 against a *pure
  exact-f32 vector* top-10 while 0.8.0 `search()` is now *RRF-fused hybrid*, part of the 0.937→0.8710
  drop may be the metric measuring fusion divergence rather than ANN/quantization fidelity loss.
  Whether that is a true fidelity regression or an artifact of comparing a hybrid result against a
  vector-only ground truth needs a controlled **engine-version (0.7.2 vs 0.8.0) A/B on the fixed
  corpus** — NOT a corpus A/B. That engine A/B is out of scope here (and the brief forbids starting
  engine work on a regression assumption); flag it to ◆ B-1.

This **overturns the B-1 premise** that the corpus moved. The honest B-1 question is not
"pin-old vs adopt-expanded corpus" but **"did the 0.8.0 retrieval-path changes (esp. unconditional
RRF fusion) move the eu7 fidelity metric, and is the eu7 ground-truth still the right reference for
a now-hybrid `search()`?"** Per the recall-eval assessment, the fidelity floor remains a
system-health gate (do not weaken/lower it); the corpus-versioning hygiene argument still applies,
but a corpus pin will NOT recover 0.937 because the corpus was never the cause.

# FathomDB 0.8.3 — Mem0 competitor-parity VERDICT

**Date:** 2026-06-23 · **Resolution axis:** end-to-end agentic-memory answer accuracy vs **Mem0-OSS**
(LongMemEval + LOCOMO, 4 classes: factoid / knowledge_update / multi_session / temporal; one answerer
`gpt-5.4`, one corpus, one metric — `PerClassScorer`). · **Gate:** near-parity-or-better vs Mem0
([[0.8.3-0.8.4-competitor-parity-reframe]]).

## Verdict: **PARITY MET** (marginally above Mem0) via the CE-rerank α lever. Surpass requires RECALL.

FathomDB with a **tuned CE-reranker (α=1.0, pool_n=10)** reaches **competitor parity with Mem0** on
agentic-memory answer accuracy. The win driver is the rerank **blend-weight retune** (production α=0.3 is
badly under-tuned). The residual gap that keeps it at *parity* rather than *surpass* is **retrieval recall**
on the hard multi-session/temporal cells — not anything reranking can fix.

| arm (full 606, gpt-5.4) | accuracy |
|---|---|
| fathomdb baseline (no rerank) | 0.137 |
| fathomdb + production rerank (α=0.3) | 0.186 |
| **fathomdb + tuned rerank (α=1.0, pool_n=10)** | **~0.33–0.35** (est; see completeness caveat) |
| **mem0_oss** | **0.323** |

## How we got here (the registered chain)

1. **D0b gap** ([[0.8.3-d0b-gap-and-parity-strategy]]): FathomDB −20..27pp vs Mem0, underpowered.
2. **Gap-decomposition (n=606):** the dominant lever is **retrieval PRECISION**, not formation — perfect
   raw-gold retrieval recovers ~+0.39 (oracle ceiling); strict recall@K is *blind* to it (gold in-window
   but buried). → test a realizable precision lever (CE-rerank).
3. **CE-rerank accuracy arm (α=0.3, n=606):** citable **PASS but marginal** — +0.0495 pooled (powered),
   but closed only ~27% of the Mem0 gap. Single-lever rerank at the production blend ≠ parity.
4. **$0 tuning sweep (n=606):** the CE-blend weight **α (hardcoded 0.3) is the dominant knob** and is
   under-tuned. α=1.0/pool_n=10 nearly doubles MRR (0.347→0.587) and ×3.9 r@1 while preserving Recall@10
   — recovered offline from one CE pass via the `score=0` ce_norm trick, no Rust rebuild
   ([[0.8.3-rerank-alpha-is-the-lever]]).
5. **α=1.0 accuracy arm:** paired on identical cells, **α=0.3 0.289 → α=1.0 0.530 (+0.24 paired; 96 win /
   12 lose)** — the α lever is real and large on the answer-accuracy axis.

## The decomposition that settles parity-vs-surpass

Split the 606 cells by whether FathomDB's retrieval surfaced the gold (α=0.3 accuracy as the proxy):

| cell group (n) | α=0.3 | **α=1.0** | Mem0 |
|---|---|---|---|
| retrieval **works** (354) | 0.285 | **0.530** | 0.395 |
| retrieval **fails** (252) | 0.048 | **~0.05–0.10** | 0.222 |
| full 606 (est) | 0.186 | **~0.33–0.35** | 0.323 |

- **Where FathomDB retrieves the gold, the tuned reranker BEATS Mem0** (0.530 vs 0.395). The rerank lever
  is genuinely competitive.
- **Where FathomDB's retrieval misses the gold, Mem0 wins** (0.222 vs ~0.07) — and reranking *cannot* help
  there: α only reorders FathomDB's top-10; it can't add gold that retrieval never surfaced. Confirmed
  reader-agnostically: a **claude-sonnet** pass on the absent cells scored **0.091 when it answered**
  (retrieval-bound, not reader-bound).
- Net: the two regimes roughly cancel → **parity** (~0.33–0.35 vs 0.323), with FathomDB marginally ahead.

## Engine takeaway (Slice-20 wire)

**Raise the CE-rerank blend weight and narrow the rerank pool.** Production `ce_rerank` uses
`ALPHA=0.3` (`fathomdb-engine/src/lib.rs`), which lets the base RRF score (70%) drown the CE signal. The
sweep + accuracy arm show **α≈1.0 with a narrow pool (pool_n≈10)** is the win: it concentrates gold at the
top of the answerer's context (the precision lever) without dropping gold out of the top-K (pure-CE on a
*wide* pool hurts Recall@10; a narrow pool avoids it). This lifts FathomDB from 0.186 → ~0.33 — the move
that reaches Mem0 parity. (Standalone-`rerank` is already CE-blended; the wire is `rerank_depth>0` into the
production `Engine.search` path + the ALPHA/pool change, per the forward plan's 5b.)

## Consequences

1. **The planned MMR / distractor-suppression arm is dropped.** MMR also only *reorders retrieved hits*, so
   it cannot help the retrieval-failure cells — the same ceiling that caps α=1.0. No value spending on it.
2. **The path to SURPASS Mem0 is retrieval RECALL** on multi_session/temporal (where Mem0's retrieval beats
   FathomDB's) — i.e. the 0.8.3 roadmap's existing recall levers (15a embedder-ceiling, 15b D2
   content-at-scale, passage-dense, index-key enrichment). This is the 0.8.4 direction.
3. **After recall improves, revisit reranking** — a better-recall pool gives the (now-tuned) reranker more
   gold to surface, so the α-lever's ceiling rises. Noted in [`../roadmap/0.8.4.md`](../roadmap/0.8.4.md).

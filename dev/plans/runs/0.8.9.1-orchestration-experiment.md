# 0.8.9.1 — orchestration experiment: where does bulk auto-fix run?

Question (from the orchestrator session): for Slice 5 (bulk markdown auto-fix), is it cheaper/safer to
run the work inline (Opus orchestrator) or delegate to a cheap-model subagent? Measured empirically on the
real Slice 5 work, normalized for work accomplished.

## The work itself (deterministic, identical regardless of who runs it)

`markdownlint-cli2 --fix` ⇄ `prettier --write '**/*.md'` to a fixpoint:
**8051 → 170 markdownlint findings (97.9%), 305 → 0 prettier, 306 files (+7084/-5274), ~3 iterations.**
Residual 170 = MD040 fenced-code-language 134 (judgment) + MD028 17 + MD025 8 + MD060 4 + tail. The bulk
is genuinely auto-fixable; the tail (Slice 10) needs per-fence language choices.

## Two delegation trials (both cold Sonnet)

| Trial | Design | Tokens | Tool uses | Outcome |
|---|---|---|---|---|
| 1 — mechanical executor | I held the neutrality check; subagent only ran the fix + reported counts | **22,779** | 10 | correct counts; no verdict asked |
| 2 — autonomous slice-owner | subagent read the real charter, built+ran its own neutrality guard, judged defects, signed off | **72,287** | 44 | **confident-but-WRONG "commit-ready, high confidence"** |

## The decisive finding (Trial 2)

The bulk auto-fix is **not semantically neutral on "markdown-about-markdown" docs** — files containing
literal glob patterns (`**/*.md`, `**docs/**`) and emphasis-like punctuation confuse the emphasis parser.
Confirmed deterministic corruption in `dev/plans/prompts/0.8.9.1-markdownlint-cleanup-HANDOFF.md`:

```text
`mkdocs build --strict` instead   →   `--strict`instead     (space collapsed)
in `dev/plans`                    →   in`dev/plans`         (space collapsed + line de-indented)
top-level `*.md`                  →   top-level `\*.md`     (backslash injected)
```

Sonnet's autonomous run built a *reasonable* fingerprint checker (hashes of code-fence contents / link
targets / prose words / numeric tokens). Its checker **correctly flagged the handoff file**
(PROSE_WORDS_CHANGED) — but Sonnet then **investigated and rationalized the real defect away** as a
"4-backtick→1-backtick false positive" and declared the whole tree commit-ready with high confidence. So
the failure was not a missing check; it was the **model dismissing a true positive its own tool caught.**
A cheap autonomous model pays *more* (72k vs 23k) and still produces a false green.

## Cost normalized for work

- Deterministic fix = a shell command; cost is dominated by *who absorbs the output and the verdict*, not
  by "running it." With the file-spill aggregator (`md_lint_summary.py`), the raw 8051-line output never
  enters anyone's context inline — so delegation buys **no context-protection advantage** for the
  mechanical step; it only adds a spawn floor.
- Inline mechanical (Opus): run commands + read the ~25-line aggregate across ~3 iterations ≈ ~1–2k
  orchestrator tokens. No spawn floor.
- Delegated mechanical (Sonnet): 22.8k tokens minimum, and the autonomous verdict is unreliable (72k +
  wrong), so an Opus/human review of the flagged subset is still required on top.

## Orchestrator context cost (the recurring tax)

Anthropic prompt cache: cache-read ≈ 0.1× input, cache-write ≈ 1.25×, TTL ~5 min. While warm the
orchestrator's context is mostly cache-reads, but the cached prefix grows every turn (linear recurring
cost) and any >5-min idle forces a full re-cache at write rates. Implication: the **orchestrator must hold
verdicts, not telemetry.** Heavy measurement + raw diffs belong in disposable subagent transcripts and
files; only compact conclusions return to the Opus loop. (Hand-grepping 306 diffs in the orchestrator
context — done once here — is exactly the anti-pattern.)

## Recommendation — division of labor for Slice 5 / 10

1. **Bulk mechanical fix → INLINE** (deterministic `--fix`/`--write` + `md_lint_summary.py` spill). No
   spawn floor, no context pollution. Cheapest.
2. **Semantic-neutrality guard → DETERMINISTIC SCRIPT, not an LLM verdict.** A script that emits "these N
   files have non-whitespace-class changes → inspect" cannot rationalize a true positive away. This is the
   load-bearing lesson from Trial 2.
3. **Verdict on flagged files → Opus (or human).** The flagged subset is small (the glob-pattern docs);
   review only those. Do NOT trust a cheap autonomous model's "commit-ready."
4. **Residual judgment tail (Slice 10: MD040 fence languages, etc.) → DELEGATE to cold cheap-model
   implementers**, one per slice (per the cold-per-slice directive). Bounded, low semantic risk, high
   volume → where cheap delegation actually pays.
5. **Slice 5 corruption handling:** the auto-fix mangles glob-pattern / emphasis-heavy docs. Slice 5 must
   either protect those constructs before fixing or hand-repair the flagged subset — it is NOT a blind
   fix-and-commit.

## Verdict: BLOCK

### 1. [high] OPENREPORT slices target the wrong report schema

Refs: dev/plans/0.6.1-implementation.md:142, dev/plans/prompts/0.6.1-OPENREPORT-py.md:22, dev/design/engine.md:48, src/rust/crates/fathomdb-engine/src/lib.rs:541

The planning slice repeatedly says 0.6.1 should surface
`migration_version_reached` / `embedder_identity_confirmed` /
`open_stage`, but the required engine design and the actual native
`OpenReport` do not define that shape. The locked success contract is
`engine + report`, where `report` carries fields such as
`schema_version_before`, `schema_version_after`, `migration_steps`, and
`embedder_warmup_ms`; the Rust struct also includes `query_backend` and
`default_embedder`. As written, the OPENREPORT prompts would send
implementers after nonexistent native fields while simultaneously
claiming "no native code changes," so the main slice premise is not
implementable from the cited source of truth.

### 2. [medium] Legacy `Engine.open(...)` compatibility is only partially pinned

Refs: dev/plans/prompts/0.6.1-OPENREPORT-py.md:60, dev/plans/prompts/0.6.1-OPENREPORT-ts.md:60, dev/plans/runs/0.6.1-planning-output.json:111, dev/adr/ADR-0.6.0-no-shims-policy.md:54

The py/ts skeletons do add regression-test hooks for legacy
`engine = Engine.open(...)` / `const engine = await Engine.open(...)`,
which is the right instinct, but it is not fully locked. Q-002 still
leaves `Engine.openWithReport` as a live HITL option, even though a
new verb conflicts with the patch-release contract, and the task
bodies do not make the legacy-callsite regression test the primary
gating test. For this slice, compatibility should be mandatory rather
than an open design branch.

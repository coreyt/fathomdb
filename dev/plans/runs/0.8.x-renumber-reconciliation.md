# FathomDB 0.8.x — Release-Number Reconciliation (authoritative index)

**Compiled 2026-07-01 (HITL decision).** Single source of truth for the 0.8.x release numbering after the
**forward renumber**. If any planning doc disagrees with this table, this table wins for *scheduling/numbering*;
point-in-time records (dated `STATUS-*` / `STEWARD-SESSION-HANDOFF-*`, `dev/adr/`, `dev/plans/prompts/`,
`archive/`, and the master doc's §6 findings block) deliberately **retain their original numbers** — this index
is the map that keeps them interpretable.

---

## 1. Why the renumber

`0.8.10` (Memory-quality plumbing) was **unstarted** while `0.8.11` and its picos `0.8.11.1` / `0.8.11.2` had
already **landed** on `main`. That left the next feature work numbered *below* work already in the tree — a
sequence inversion. Because none of `0.8.12`+ had started and nothing is published above `0.8.9`, the fix is a
pure **plan-doc relabel** (no code, no history rewrite): shift each unstarted **even** slot's content up by two
so numeric order = landing order = priority order. The landed `0.8.11.x`, the odd line, `0.8.20`, and the
`13`-forbidden rule are unchanged.

## 4. Standing rules (unchanged by the renumber)

- **Two-tier numbering.** `x.y.z` = real micro (even = publishable w/ HITL, odd = non-publishable); `x.y.z.p`
  pico = label-only, never published. A Library Sweep is a pico.
- **`13` is HITL-forbidden** as both minor and micro (no `0.13.x`, no `0.8.13`). The odd line skips it.
- **Don't rewrite history.** Historical/dated records keep their original numbers; this index maps them.
- **Legacy filename.** The master doc keeps the name `0.8.6-0.8.16-PROGRAM-SEQUENCING.md` (29 inbound refs); its
  H1 span is the authoritative bound (now `0.8.6 → 0.8.18`).

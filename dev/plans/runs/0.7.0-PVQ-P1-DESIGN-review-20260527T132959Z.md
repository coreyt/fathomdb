### 1. [high] `doc` as a 7th `source_type` value breaks the HITL lock
Refs: `dev/design/0.7.0-vector-quant-pack1.md:105`; `dev/plans/0.7.0-HITL-recommendations.md:135`; `src/rust/crates/fathomdb-engine/tests/perf_gates.rs:350`

The memo explicitly expands the locked vocabulary from 6 values to 7 by admitting `"doc"`. HITL says Pack 1 "MUST use `source_type` as the vec0 partition_key," and the locked vocabulary immediately above it is `email/article/paper/meeting/note/todo`; there is no approval here to add a synthetic 7th value. Because D3 changes a HITL-locked contract instead of implementing it, this is a blocking design violation. If `doc` must remain in tests, it needs a separate HITL/ADR update or the fixture kind needs to be remapped without changing the locked vocabulary.

### 2. [high] Migration plan relies on vec0 rename support not in the pinned sqlite-vec version
Refs: `dev/design/0.7.0-vector-quant-pack1.md:183`; `src/rust/crates/fathomdb-engine/src/lib.rs:1207`; SQLite virtual-table rule (www.sqlite.org/vtab.html); sqlite-vec releases (github.com/asg017/sqlite-vec/releases)

The memo's rebuild shape is `CREATE ... vector_default_v2` → copy rows → `DROP TABLE vector_default` → `ALTER TABLE vector_default_v2 RENAME TO vector_default`. SQLite allows neither `ADD COLUMN` on virtual tables nor arbitrary rename unless the module implements it; sqlite-vec's release notes show `ALTER TABLE RENAME support` appearing in `v0.1.10-alpha.2` (April 2026), while this repo is pinned to `=0.1.7`. The proposed migration is not validated for the version in use. The same section invents an `operational_mutations` schema stamp, but the engine's real migration framework is `migrate_with_event_sink(...)` + `PRAGMA user_version` — gating story mismatched.

### 3. [medium] D6 secondary "normalized" fixture is internally incorrect
Refs: `dev/design/0.7.0-vector-quant-pack1.md:303`; `src/rust/crates/fathomdb-engine/tests/perf_gates.rs:322`

First RED fixture is fine. But the memo claims `normalize([1,-1,1,-1,...])` quantizes identically to the half/half vector because they have "identical sign patterns at every coordinate." They do not: one is half-positive/half-negative, the other alternates signs. Remove or rewrite the secondary assertion.

## Verdict: BLOCK

### 1. [medium] `scan_rust()` still mis-parses legal nested Rust block comments, so finding 2 is not fully closed

Refs:

- `scripts/security/ast_scan.py:133`
- `scripts/security/ast_scan.py:148`
- `scripts/tests/test_ast_scan.sh:70`
- `scripts/tests/fixtures/ast-shim/rust/clean/block_comment_safe.rs:1`
- `scripts/tests/fixtures/ast-shim/rust/block-comment-real/real.rs:1`
- `dev/plans/runs/12-S-fix-1-output.json:12`

The new Rust comment stripper tracks block comments with a boolean,
not a nesting depth. Rust allows nested `/* ... */`, and replaying
this exact state machine against `/* outer /* legacy_nested */
still outer */` leaves `still outer */` as live code instead of
keeping the scanner inside the outer comment. Other two edge cases
are handled correctly: `*/ pub fn legacy_foo() {}` is reprocessed,
and `/* text */ pub fn legacy_foo() {} /* text */` still exposes
the real `pub fn`. The remediation artifact says finding 2 is
closed; current fixtures only cover non-nested comments, so that
claim is overstated. Because AC-050a is now wired into
`agent-verify`, this remains a real hard-gate false-positive risk
on valid Rust code.

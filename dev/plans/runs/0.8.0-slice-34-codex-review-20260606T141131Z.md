## VERDICT: CONCERN → fix-1 (1 × [P2]; no [P0]/[P1])

> "The new CLI verb generally compiles, but pagination can incorrectly terminate for limits above the
> engine clamp despite more mutation rows being available."

- **[P2] — `next_after_id` uses the un-clamped requested `--limit`** — `fathomdb-cli/src/lib.rs:465-466`.
  When `--limit` exceeds the engine page cap (`READ_COLLECTION_MAX_LIMIT`, ~1M) and the collection has more
  rows than the cap, `read_mutations` clamps the SQL `LIMIT` and returns at most ~1M rows, but the CLI still
  compares `rows.len()` to the *original requested* limit. A full capped page (`rows.len() == cap <
  requested`) is therefore treated as exhausted → `next_after_id: null` → a caller stops paginating while
  rows remain. **Fix:** clamp the CLI `--limit` to the same effective cap before both the read and the
  `next_after_id` decision (or expose the cap from the engine).

## Disposition

Real correctness bug (silent pagination truncation), but bounded to a pathological scale (>1M-row log AND a
`>1M --limit`). The fix is exactly what codex named and is trivial + CLI-only → **fix, do not override**
(consistent with the project preference to fix real reviewer-found defects rather than reserve them). →
**fix-1** (`dev/plans/runs/0.8.0-slice-34-review-fix1-*.md`).

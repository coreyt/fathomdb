# 0.7.2 PR-3 — empirical performance data (AC-013 / AC-013b / AC-019)

**Status:** measurement record (local). Feeds the tiered budget in
`dev/adr/ADR-0.7.0-text-query-latency-gates-revised.md` and the PR-3 closure
artifact `dev/plans/runs/0.7.2-PR-3-output.json`.

This file is the authoritative empirical record for the 0.7.2 PR-3 latency /
recall measurement gate. Every row keeps its full settings (N, dim, K,
samples, seed time, env knobs, host) so a future reader can reproduce or
re-derive without re-running.

## Why these were run locally (not canonical CI)

HITL 2026-05-31/06-01: the real default-embedder (bge-small) + real-corpus
measurement at N=1M is infeasible on the 4-core canonical runner — seeding
1M docs through the serialized candle projection path is ~166 h at the
PR-9-measured **1.67 docs/s** vs a 240-min workflow timeout. So the heavy
measurement is a **local once-per-release exercise**; CI carries only a fast
"is the read path broken?" guard (`perf_gates::ac_013_vector_read_path_smoke`).
See `dev/notes/ac013-ac019-canonical-scale-policy.md` and
`dev/plans/runs/0.7.2-PR-3-output.json`.

### Scaling — latency is ~O(N) (no ANN index on vec0)

The vec0 bit-KNN is a per-query **linear** scan over all N rows (no HNSW/IVF
index; see `dev/notes/ac013-ac019-canonical-scale-policy.md` "long-term
follow-up"). The measured synthetic points confirm near-linear growth:

| dim | N=10K p50 | N=100K p50 | ratio (10×N) | implied 1M p50 (linear) |
|---|---|---|---|---|
| 384 (prod) | 15 ms | 147 ms | 9.8× | ~1,470 ms |
| 768 | 22 ms | 190 ms | 8.6× | ~1,900 ms |

So at the shipped 384-d, p50 is ~**O(N)** with coefficient ≈ **1.5 ms per
1,000 docs** plus a small fixed per-query overhead (the FTS MATCH + soft-
fallback probe + parse run on every `search()` regardless of N). The 10k
binding-tier number is **15 ms (synthetic 384-d) / 36 ms (real bge, N=7667
≈ 10k)**; linear scaling sets the 100k tier near 147 ms and the 1M tier near
1.5 s — i.e. the 80 ms p50 budget is **not** met above ~50k without an ANN
index. This is the data behind the tiered ADR decision (10k binding for
0.x/1.x; 100k/1M tracked targets for post-1.0, pre-2.1 ANN work).

### Seed-cost finding (why 1M did not complete locally)

Seeding is **super-linear** on the default per-row projection path:
N=100K drained in ~182 s (~550 docs/s), but N=1M did **not** drain in 3 h
(`wait_for_idle` timeout → `Err(Scheduler)`; **not** a wedge — see
`Engine::drain` at `lib.rs:1995`). vec0 b-tree insertion + sign-bit quantize
+ WAL growth cost rises with table size. 0.7.0 obtained its 1M number only via
the W4.1 perf-knob seed path (pcache2 + `page_size=8192`), which the plain
`cargo test` path here does not enable. The 1M tier is therefore recorded from
the 0.7.0 W4.1 historical anchor + the O(N) extrapolation above, not a fresh
local run.

**Anchor (HITL-accepted for PR-3):** recall@10 = **0.937** (CI 0.913–0.957,
σ=0.0116) on the largest all-real-corpus haystack (N=7,667, K=192, bge-small +
mean-centering). The 0.90 floor HOLDS (R−2σ = 0.914). The earlier 0.828 was a
conservative-measurement artifact (exclude-after + body-string set GT); see
`dev/plans/runs/0.7.2-PR-2c-recall-rootcause.md` and
`ADR-0.7.0-vector-binary-quant.md` §2 point 4. This is ANN/quantization
fidelity (GT = exact f32 top-10 over the SAME model), **not** IR relevance.
N=1M real recall is not measurable on this hardware (~166 h seed); recall
decreases slowly with N, so 0.937 @ N=7667 is an upper-ish bound for larger N.

## AC-019 — concurrent-mixed read tail (8 threads)

bound = max(baseline_p99 × 10, 150 ms).

| fixture | embedder | dim | N | threads | baseline p99 | stress p99 | bound | verdict | note / source |
|---|---|---|---|---|---|---|---|---|---|
| real | bge-small-en-v1.5 | 384 | 7,667 | 8 | 40 ms | **343 ms** | 405 ms | **PASS** | clean dev-box run; `0.7.1-EU-7-output.json` |
| real | bge-small-en-v1.5 | 384 | 1,000 | 8 | 30 ms | 205 ms | 307 ms | PASS | `0.7.1-EU-7-output.json` |
| real | bge-small-en-v1.5 | 384 | 7,667 | 8 | 49 ms | 1,201 ms | 499 ms | MISS (contended) | PR-2c re-measure under concurrent CPU load; `0.7.1-EU-7-measurements.json`. Contention-inflated (same fixture/N as the 343 ms clean run). |
| synthetic (report-only) | VaryingEmbedder | 384 | 10,000 | 8 | 16 ms | 524 ms | 168 ms | does-not-meet-bound | clean box (loadavg 2.6); `0.7.2-PR-3-local-ac019-10k-384.log` |
| synthetic (report-only) | VaryingEmbedder | 768 | 10,000 | 8 | 28 ms | 1,048 ms | 282 ms | does-not-meet-bound | clean box; `0.7.2-PR-3-local-ac019-10k-clean.log` |
| synthetic | VaryingEmbedder | 768 | 1,000,000 | 8 | — | n/a | — | seed drain timeout (shared `seed_ac013_corpus` with AC-013) | `0.7.2-PR-3-local-canonical.log` |

**AC-019 disposition (HITL 2026-06-01).** Two distinct facts:

1. **Verdict = MET on the real-corpus path.** The clean real-bge run (N≈7,667)
   is **343 ms < 405 ms bound → PASS** — the verdict-quality signal (production
   embedder; `dev/notes/ac013-ac019-canonical-scale-policy.md`: real-corpus is
   the verdict, synthetic dev-box is scouting). The 1,201 ms real-path number
   was concurrent-CPU-load contention (same fixture/N), **not a regression** —
   this resolves the carried "AC-019 idle-box re-run" item.

2. **The synthetic fixture cannot meet AC-019's bound — by virtue of the
   synthetic DATA, not the box** (HITL-confirmed; clean-box loadavg ~2.6, the
   8 stress threads drove load themselves). Bound = `max(baseline_p99 × 10,
   150 ms)`; the synthetic embed is instant, so the single-thread baseline
   (16–28 ms) is unrealistically fast and the 10× bound (168–282 ms) is far
   tighter than production (405 ms). The absolute 8-thread tail (~520 ms @384-d
   / ~1,050 ms @768-d) is the same order as the real path — the bound, not the
   latency, is what synthetic can't satisfy. K=64→192 (the EU-7/PR-2 recall
   fix) raised rerank cost and thus the tail vs the old STATUS-PVQ 131 ms
   (K=64) reading.

   → **`perf_gates::ac_019_mixed_retrieval_stress_workload_tail` is REPORT-ONLY**
   (`AC019_REPORT_ONLY` line; no assertion). AC-013 keeps its hard 10k-tier
   gate. The asserting AC-019 signal lives in the real-corpus harness
   `eu7_real_corpus_ac.rs`. (No fresh synthetic 1M AC-019 number: it shares the
   AC-013 seeder, which did not drain at 1M.)

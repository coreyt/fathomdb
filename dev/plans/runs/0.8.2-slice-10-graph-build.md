## Pipeline

1. **Reproduce** the Slice-4 corpus (`tests/corpus/scripts/acquire_musique.py`) — asserts
   `musique_hash 3cff37fd…`; raw corpus + manifests reproduced byte-identically (no tree dirt).
2. **Extract** entities + fact-edges per **paragraph** (custom_id `"{qid}#{idx}"`) by
   reusing `eval.graph_arm_recall.extract_graph` (Qwen3.6-27B, Airlock vLLM **batch**,
   `enable_thinking:false`, $0). **Incremental cache** keyed by paragraph custom_id →
   resumable; only missing paragraphs are extracted.
3. **Load** into a FathomDB engine: passages as `kind="doc"`, per-question entities as
   `kind="entity"` with `logical_id="{qid}|ent:{norm}"`, **body-less** edges as
   `kind="rel"`.
4. **Verify coverage** read-only (`mode=ro` sqlite — drain/terminal status can lie,
   [[embed-completeness-and-gpu-readiness]]): every sampled question has ≥1 entity node;
   `canonical_nodes`/`canonical_edges` populated; **0 edges carry a body**.

## Design choices (justified deviations / adaptations)

- **Per-paragraph granularity** (not per-question-over-concatenated-text). The median
  question is ~8.2k chars across ~20 paragraphs, which would truncate hard at the
  extractor's 6k `char_cap`. Per-paragraph also preserves the **passage→entity
  membership** HippoRAG-style PPR (Slice 15) needs, and gives passage-level provenance in
  the cache. Plan §4 calls this the "MuSiQue **paragraph** corpus".
- **Per-question entity namespacing** (`"{qid}|ent:{norm}"`). Unlike LongMemEval (a shared
  haystack with GLOBAL entities, as in `graph_arm_recall`), MuSiQue distractor paragraphs
  are **per-question**; identically-named entities in different questions must NOT merge,
  or the graph would bridge unrelated questions. Each question is its own subgraph.
- **BODY-LESS edges** (the load-bearing adaptation, design §7). Edges carry no `body`, so
  the engine never enqueues `edge_fact` vector projection (needs an embedder + hits a
  StorageError scale bug at ~8-14k edges — the 0.8.1 engine follow-up). Relations still
  connect their two entities for traversal; provenance rides `source_id` (the qid) and the
  per-paragraph cache.
  - **Asset-divergence finding.** The Slice-10 prompt/plan say
    `graph_arm_recall.build_graph_engine` "builds edges WITH body by default" and that
    body-less is its opposite. **At baseline 92ed09e the asset ALREADY builds edges
    body-less** (its edge dict has no `body` key + the docstring documents exactly this
    `edge_fact` follow-up). So body-less is **consistent with** the current asset, not its
    opposite — the prompt's premise is stale. This module keeps edges body-less explicitly
    regardless, and the coverage test pins `n_edges_with_body == 0`.
- **PII redaction.** The reused `extract_graph` performs **no** PII redaction (no redaction
  seam exists in the extractor or engine write path), so the standing "disable pii_redact
  for synthetic extraction" rule is satisfied by construction; MuSiQue is public CC-BY-4.0
  synthetic-eval data.

## Sampling (no silent caps — design §7)

Deterministic, **hop-stratified** (`sample_questions`): the answerable set is split by hop
count {2,3,4}, each stratum sorted by id, then a seed-rotated even stride is taken in
proportion. The committed build used `--n 300 --seed 20260617` on the **answerable** set →
**300 questions** (per-hop 155/94/50 + 1 top-up), 5 999 paragraphs. The sample size + per-hop
breakdown are `log()`-ed and recorded in the artifact's `sample` block.

This is a labelled SAMPLE, not the full 2 417-answerable set (~48k paragraphs ≈ 11h at the
measured ~1.2 paragraph/s). The incremental cache lets Slice 15/20 extend extraction (incl.
the unanswerable contrast set) without re-extracting these paragraphs. Extractor-throughput
tuning to cover the full set is the plan's reserved follow-on (11–14) — design a sweep, do
not improvise (perf-tuning-design-sweeps-not-adhoc).

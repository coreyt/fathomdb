## 2. The measurement arc — four results, each correcting the last (why bias controls are load-bearing)

All judged on the BenchmarkQED **AutoQ global questions**, AP-News corpus. Win-rate >0.5 = the
GraphRAG-style arm beats the baseline.

| # | configuration | result | what it taught |
|---|---|---|---|
| 1 | GraphRAG-style map-reduce, **same-family Qwen judge** | "wins" 0.75 | **self-preference false positive** — Qwen judged its own answers |
| 2 | same arm, **cross-family Claude judge + weak Qwen answerer** | loses 0.25–0.44 | **weak-synthesizer false negative** — map-reduce QFS depends on reader quality |
| 3 | same arm, **cross-family judge + strong gpt-5.4 answerer** | **wins 0.55–0.83** (surpass on comprehensiveness vs vector_rag) | the registered config: premise looked POSITIVE |
| 3b | a **crude community-S1** (my reimplementation: label-prop, 60-doc, simple reports) + cross-family judge | loses 0.32–0.49 even with strong reports | I (wrongly) concluded "community paradigm = graph null, don't fund S1" |
| **4** | **FathomDB vs a RUNNING Microsoft GraphRAG 3.1.0** (real hierarchical Leiden communities, 115 community reports), cross-family judge | **Microsoft GraphRAG WINS; FathomDB 0.00–0.33** | the literal verdict — **and it overturned #3b**: my crude reimplementation under-represented real GraphRAG |

**The decisive correction (#4 vs #3b):** my hand-built community-S1 lost, so I provisionally called the
paradigm a null. But the *actual* Microsoft GraphRAG — proper entity/relationship/claim extraction,
hierarchical Leiden communities, 115 relevance-scored community reports, a real global-search map-reduce —
**beats even FathomDB's raw-text map-reduce.** The community-summary paradigm **does pay off when built
well**; my reimplementation was simply weak. The "don't fund S1" lean is **withdrawn**.

## 3. The definitive head-to-head (the result the goal names)

**Setup:** Microsoft GraphRAG 3.1.0 (pip in a venv; LLM = gpt-5.4 via the airlock proxy; embeddings via a
local hashing shim — neither airlock nor local vLLM serve embeddings) indexed 15 AP-News articles → real
pipeline (entities/relationships/claims → Leiden hierarchical communities → 115 community reports). Its
`graphrag query -m global` was run vs FathomDB arms over the **same 15 docs** (fair), cross-family judged
(claude-haiku, gpt-5.4 reader), 5 runs, order-swapped, `decide_084`.

| FathomDB arm vs **Microsoft GraphRAG** (>0.5 = FathomDB better) | comprehensiveness | diversity | empowerment | `decide_084` |
|---|---|---|---|---|
| `fathomdb_mapreduce` (strong-reader synthesis over raw text) | 0.062 [0.00,0.19] | 0.319 [0.16,0.48] | 0.231 [0.06,0.40] | NOT_REACHED |
| `fathomdb_vector` (FathomDB retrieval + reader) | **0.000** [0.00,0.00] | 0.325 [0.13,0.54] | 0.069 [0.00,0.19] | NOT_REACHED |

**FathomDB is far below the 0.45 near-parity band on every comparison — it does not reach parity.**
GraphRAG's structured global synthesis over hierarchical community reports covers more of the corpus's
themes than FathomDB's flat retrieval or single-pass map-reduce.

## 4. Caveats (do not over-read)

- **Scale:** 15-document index, n=8 questions (`decide_084` = NOT_REACHED on power). But comprehensiveness
  ≈ 0.00 with tight CIs is **decisive**, not a power artifact — a full-corpus index + powered N would
  sharpen the loss, not reverse it.
- **Length bias:** GraphRAG answers ran longer (~4–6k vs ~1–3.5k chars). A length-bias contribution is
  possible; but the gap spans all three metrics (incl. less length-sensitive diversity/empowerment) and
  comprehensiveness ≈ 0, so length alone cannot explain a decisive loss. For a *registered* claim: power
  up + apply the directness/claim-count length control explicitly.
- **Axis:** sensemaking only. **HippoRAG-2 multi-hop (reference-based F1) was NOT run.** FathomDB's own
  multi-hop is already refuted (M1 NO-GO), so that axis is expected to favor the competitor too.
- **Embeddings shim:** GraphRAG indexing used a hashing-BoW embedding (no real embedder available). Global
  search barely uses embeddings (it runs on community reports), so this is a minor factor for *global*
  search; it would matter for local/drift search (not used here).

## 5. Bias-control lessons (methodologically reusable, beyond 0.8.4)

The four LLM-judge bias controls are **empirically load-bearing**, demonstrated by the overturns above:
1. **Same-family judge inflates** (+0.75 phantom self-preference win). Always cross-family.
2. **Reader/answerer quality is a dominant confound** — it flipped the *same* arm between winning and
   losing (0.25 → 0.83). Pin a strong reader, identical across arms.
3. **Implementation quality of the competitor matters** — a weak reimplementation produced a false null;
   the real system reversed it. Stand up the *actual* competitor, not a proxy.
4. `decide_084` correctly refused to declare REACHED in every underpowered case (binding=underpowered).

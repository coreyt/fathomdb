# Codex high-effort consult — active-uniqueness scope: `(logical_id, kind)` vs `logical_id`

- **Trigger:** Slice 30 codex §9 [P2] (read.get/get_many lossy+nondeterministic when one `logical_id`
  has multiple active kinds). The [P2] is the symptom; the G0 identity-scope decision is the root.
- **Reviewer:** `codex exec` at `model_reasoning_effort=high`, adversarial design review (independent
  of the Slice 30 agent's analysis). Raw log: `/tmp/s30-codex-identity-scope-20260605T021802Z.log`.

## Verdict (codex)
**Use `logical_id` alone for active uniqueness on BOTH `canonical_nodes` AND `canonical_edges`. Uniform,
NOT asymmetric.** `kind` is payload/classification (nodes) or relationship type (edges) — not a safe
identity component. An edge needing multiple active relationship facts between the same endpoints should
use **distinct edge `logical_id`s**; reusing one edge `logical_id` with a different `kind` should
**supersede, not fork**. This **converges with the Slice 30 agent's recommendation**, and resolves the
node-vs-edge asymmetry angle decisively: codex read the writer and found edges do **not** implement a
real "same pair, multiple relationship types" model (edge `logical_id` is caller-provided + opaque, not
derived from `(from,to)` or `kind`; `PreparedWrite::Edge` `lib.rs:1017-1025`; Py `:873-882`, napi
`:1075-1081`). v0.5.x also treated edge `logical_id` as identity alone.

## Claims A–E (codex)
- **A — never argued: PARTIAL.** Worse than "just a parenthetical": the compound scope was **propagated**
  across multiple docs (parent ADR Q2 `:128-131`; roadmap `0.8.0.md:84-89`; signed substrate ADR DDL
  `:212-215` + authorized elements `:237-242`) but **never justified**. The signed substrate gate asks
  about Decisions 1–4 / migration / row-id / shadow reconciliation, **not** identity scope (`ADR:366-376`).
- **B — v0.5.x regression: CONFIRMED.** v0.5.6 `idx_nodes_active_logical_id ON nodes(logical_id) WHERE
  superseded_at IS NULL` + edges same (`v0.5.6:crates/fathomdb-schema/src/bootstrap.rs:22-45`); writer
  superseded by `logical_id` alone (`v0.5.6:…/writer/mod.rs:1761-1764, 1741-1744`).
- **C — silent identity-fork bug: CONFIRMED, high severity.** Node + edge supersession is
  `WHERE logical_id=? AND kind=?` (`lib.rs:5995-6000`, `:6025-6038`); a changed `kind` matches nothing and
  the compound unique index permits the insert → second active row.
- **D — contradicts G8: CONFIRMED for nodes + exposes edge risk.** G8 probes `WHERE logical_id=? AND
  superseded_at IS NULL` (`lib.rs:6118-6120`; `slice-20-g8-design.md:15-20, 53-69`) — can't disambiguate
  endpoint kind because edges store only their own kind.
- **E — no consumer need: CONFIRMED.** The only "different kind allowed" case is a self-referential G0
  test (`pr_g0_identity.rs:177-182`), not a use case; Slice 30 + public docs present deterministic by-id
  point lookup (`slice-30-design.md:159-161`; `docs/guides/retrieve-by-id.md:3-14`).

## Two things codex added that the agent's analysis MISSED
1. **Migration hazard (the agent hand-waved "v12 is local, amend in place").** Step 12 already landed →
   existing local DBs with `user_version=12` **will NOT re-run edited step-12 SQL**. If already-migrated
   local DBs must pick up the change, you need a **NEW forward migration (step 13)** that drops/recreates
   the two indexes — and if duplicate active rows already exist locally, the new **UNIQUE index creation
   will FAIL / require explicit cleanup** (that data is already identity-forked).
2. **Parent-ADR vs code drift (independent of the kind question).** The retrieval ADR's Q4 ("edges too")
   says it is **"schema-only"** and "0.8.0 implements only **node** supersession behavior"
   (`ADR-…-retrieval-and-identity.md:255-258`), but the signed substrate ADR + code **do** implement edge
   supersession. Edge supersession semantics were never cleanly settled.

## Recommendation + blast radius (codex)
Adopt `logical_id`-only active uniqueness for both tables:
- **Schema:** `canonical_nodes_logical_active_idx ON canonical_nodes(logical_id) WHERE superseded_at IS
  NULL`; same for `canonical_edges`. **Prefer a new migration step (13)** if preserving already-migrated
  local v12 DBs matters.
- **Engine:** remove `AND kind = ?` from node + edge supersession predicates in `commit_batch`.
- **G8:** change in-batch edge supersession tracking from `(logical_id, kind)` → `logical_id` (the
  Slice 20 fix-1 `HashMap` precompute key).
- **Tests:** invert `s15_partial_unique_active_index_rejects_two_active_versions` (different kind + same
  `logical_id` is now REJECTED); add node + edge kind-change supersession tests.
- **Docs/ADRs/plans:** amend all `(logical_id, kind)` active-identity language → `logical_id`; keep `kind`
  as filter/classification/relationship type, not identity scope. (Add the now-argued "Decision 5 —
  identity scope = `logical_id`" to the substrate ADR.)

codex ran no verification (adversarial design review, not an implementation pass).

# HITL DECISION BRIEF — 0.8.8 EXP-OBS "explain carrier" (sidecar `Explanation`)

> For securing cross-component/cross-release alignment before Slice 5 code. Grounded in the repo
> (file:line). Companion to `dev/design/0.8.8-explain-and-telemetry-adr.md`. No code touched.

## 1. Decision restated

Carrier = a **sidecar `Explanation` hung off `SearchResult`**, not inline fields on `SearchHit`.
`SearchResult` gains `explanation: Option<Explanation>`, `None` whenever `explain=false` (default),
no allocation on the off-path. This commits the codebase to keeping `SearchHit` **byte-identical**
across all three bindings and the non-explain `SearchResult.results`/cursor unchanged, while adding
one optional field + one new public type tree (`Explanation` + sub-types) that must propagate
through the engine, all three SDK mirrors, the governed-surface allowlists, and the interface docs.
Source: `dev/design/0.8.8-explain-and-telemetry-adr.md` §A.2.

## 2. LOCKED vs STILL-OPEN

**LOCKED** — Carrier shape: sidecar `Explanation` on `SearchResult`; `SearchHit` byte-identical;
off-path `None`, zero-cost (ADR §A.2; R-OBS-2 hard gate, `plan-0.8.8.md:72,102-104`).

**STILL-OPEN** (close at ratification)
- The per-hit + trace **field set** (ADR §A.3) — proposed, not confirmed.
- Q1 — raw per-arm scores, or `arm_ranks` + `rrf_norm` enough? (ADR §A.4)
- Q2 — surface filter-excluded hits (`excluded_by_filter`), or returned-set only?
- Q3 — coarse per-stage `timings_ms`, or per-arm CPU attribution?
- Q4 — which canonical eval gold-row shape must `GoldRecord` mirror? (ADR §B.3)
- Telemetry event shape (§B.1) + privacy/footprint contract (§C).
- **NEW decision surfaced by the audit:** `Explanation` is a **type tree**, not one struct. Decide
  whether sub-types (`trace`, `per_hit`, `arm_ranks`, `timings`) are **public** (each a governed-surface
  allowlist entry, each bumps the hard `len()==26` pin in `governed_surface.rs:148`) or kept **private
  inside an opaque `Explanation`** (one allowlist entry). Recommendation: keep sub-types private/opaque
  to minimize the governed-surface blast radius.

### c. Experiment / M-work track (the non-decoupled coupling, plan §8)

Plan `:133-136`: EXP-OBS is the one release "not fully decoupled from the experiment track" — explain
fields must match what experiments consume. Consumers under `src/python/eval/` already read score/arm
internals: `ce_rerank_probe.py`, `rerank_accuracy_run.py`, `rerank_tune_probe.py`, `m1_ppr.py`,
`graph_arm_recall.py`, `gap_decomposition_run.py`, `r2_parity_eval.py`. ADR fields are grounded in the
engine's own `fuse_three_arms` (`rank0`, `rrf_norm`) + `ce_rerank` (`rrf_norm`/`ce_norm`). **Owner: M-work
owner — answer Q1-Q3** so harnesses ingest `Explanation` without a shim and the R-OBS-1 golden test pins
the right values.

### d. Telemetry / real-gold track (Slice 15/20)

Gold-record schema (ADR §B.2) shares the per-hit id vocabulary. Existing eval gold schemas are
**string-keyed**: `r2_parity_eval.py:119-124` (`GoldQuery.gold_doc_ids`), `gold_gen.py`,
`gold_repin.py:103-120` — whereas `GoldRecord`/`TelemetryEvent` propose **`u64`** ids. Real mapping
question (ADR §B.3 Q4). **Owner: M-work/eval owner — name the canonical gold-row shape to mirror + the
id-type reconciliation.**

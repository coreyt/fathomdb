# Can FathomDB's graph arm beat BM25 on LongMemEval? — Experiment report

**Status:** CLOSED — clear, robust NEGATIVE → PIVOT (HITL, 2026-06-16).
**Audience:** FathomDB 0.8.1 team.
**Author:** eval/graph-arm track. **Tag:** `0.8.1-beat-bm25-pivot-2`.
**Artifacts:** `/tmp/gar_n40_nopoll.json`, `/tmp/gar_n40.json`, `/tmp/gar_dry.json`,
`dev/plans/runs/0.8.1-p0a-*.json`, narrative in `dev/plans/runs/STATUS-0.8.1.md` §7.
**Pipeline:** `src/python/eval/graph_arm_recall.py`.

---

## 1. TL;DR

We extracted a real entity/edge knowledge graph from 1,907 LongMemEval sessions with a
local Qwen3.6-27B, wrote it into a FathomDB engine, and measured **LLM-free recall@K** with
the BFS graph arm ON vs OFF on the *same* engine (the comparison that isolates the arm).

**The graph arm does not beat BM25, and the BFS arm adds essentially zero recall** —
`graph_ON` == `graph_OFF` (0.65 pooled R@10), even on `multi_session`, its target class.
Worse, mixing entity rows into the FTS corpus *costs* recall vs a docs-only FTS index
(0.80 → 0.65). Both findings are exactly what the retrieval literature predicts. Decision:
**drop the graph-surface seeding / engine edge-fact fix / n=160-as-designed; pivot to
index-key enrichment** (append a session's extracted entities/facts to its own doc's FTS
content).

**Headline — 40 questions (10/class), 1,907 sessions, 28,883 entities, 38,021 edges, pooled R@10**
(`/tmp/gar_n40_nopoll.json`):

| Variant | R@10 | vs BM25 | Note |
|---|---|---|---|
| naive_bm25 | **0.70** | — | external reference (rank-bm25) |
| fathomdb_fts_only (docs only) | **0.80** | +0.10 | FTS5 beats BM25 here |
| fathomdb_graph_OFF (two-arm, entities in index) | **0.65** | −0.05 | co-mingling cost |
| fathomdb_graph_ON (BFS traversal) | **0.65** | −0.05 | **BFS adds 0.00 over OFF** |

---

## 2. Hypothesis

FathomDB's graph arm (LLM-extracted entity/edge graph + BFS traversal, fused into the
two-arm result via RRF) can **beat BM25 on LongMemEval recall@K** by recovering multi-hop /
cross-session evidence that the lexical arm misses — especially the `multi_session` class,
which is the weakest class for every lexical baseline and the natural target for graph
traversal.

## 3. Method

### 3.1 Extraction infrastructure (local, $0)
- **Model:** Qwen3.6-27B served locally via the **Airlock vLLM batch gateway** on-prem
  (2× RTX-3090), addressed as alias `qwen36-27b-vllm-batch` over the OpenAI-batch surface
  (`/v1/files` → `/v1/batches`). Cost **$0** (on-prem).
- **Decode config:** `chat_template_kwargs.enable_thinking=false`; `max_tokens=3072`;
  per-session body capped at 6,000 chars. Concurrency was tuned to conc=8 (the
  compute-bound knee). ~98% of responses parsed as clean JSON; the rest were recovered by a
  truncation-salvage path (`_salvage_json`, closes the last complete array/object element).
- **Prompt:** extract entities `{name,type}` + relations `{subject,predicate,object}` as
  compact JSON only (`_EXTRACT_PROMPT`, `graph_arm_recall.py:53`).
- Extraction is cached per-session (`extractions.json`); only missing sessions are
  re-extracted, so the recall sweep iterates in minutes.

### 3.2 Graph model & engine build (`build_graph_engine`)
- **Docs:** each LongMemEval session body is written as `kind="doc"`. Doc keys resolve to
  **gold `answer_session_id`s** (verified) via the write-cursor map — so a retrieved doc can
  be scored directly against gold.
- **Entities (global):** `kind="entity"`, `logical_id = ent:<normalized-name>` (lower-cased,
  whitespace-collapsed), `source_id =` first session the name appeared in. Global identity is
  `logical_id`-alone (per the signed G0 identity scope).
- **Edges (per session):** `kind="rel"`, `from`/`to` = the two entities' `logical_id`s,
  `source_id =` the session the relation came from. Edges are written **without a body**
  deliberately — see §7 (engine bug): an edge *with* a body auto-enqueues an `edge_fact`
  vector projection that fails with no embedder and hits a storage scale bug at ~8–14k edges.
  Consequence: **source-A (edge-fact FTS) seeding is OFF**; the arm runs on source-B
  (entity-node FTS) + edge traversal only → a graph-arm **lower bound**.

### 3.3 The decisive design: ON vs OFF on the SAME engine
Four variants are scored by the shared `run_retrieval_loop` (LLM-free, doc-key == gold):

- `naive_bm25` — external reference (rank-bm25 over raw session text).
- `fathomdb_fts_only` — FathomDB FTS5 over a **docs-only** index (no entities/edges).
- `fathomdb_graph_off` — the graph engine, `use_graph_arm=False`, with a `kind="doc"`
  result filter (anti-pollution: excludes entity rows from the *returned* list).
- `fathomdb_graph` — the same graph engine, `use_graph_arm=True`, same filter.

The **load-bearing comparison is `graph` vs `graph_off`**: identical engine, identical
corpus, identical filter — the *only* difference is whether the BFS arm contributes
candidates. This isolates the arm from the entity-in-index confound; `bm25`/`fts_only` are
context references, not the test.

### 3.4 Caveats / confidence
- **N=40 (10/class).** Per-class MDE is large (~15 pp on a 10-item class; the prior n=160
  run reports ~21 pp per-class / ~11 pp pooled). Treat single-class deltas as
  direction-only. The **ON-vs-OFF null is robust** precisely because it is a same-engine
  paired A/B (no corpus/embedder variance between arms), and it reproduces before and after
  the anti-pollution filter.
- **Source-A seeding OFF** (engine bug, §7) — so this is a graph-arm lower bound, not its
  ceiling. The literature (§6) argues the missing lift is the *seeding strategy*, not the
  source count.
- LLM was used **only** for extraction; recall@K is fully LLM-free and deterministic.

## 4. Results

### 4.1 40-question graph-arm sweep — POST anti-pollution filter (`/tmp/gar_n40_nopoll.json`)
1,907 sessions, 1,907 graphs, 28,883 entities, 38,021 edges. Sanity: 50/50 graph-arm hits
on the probe query resolved to real haystack sessions (`sanity_graph_hits=50`,
`sanity_resolvable=50`) — the arm *is* firing and producing source-resolved candidates.

Per-class R@10 (n=10 each) and pooled:

| Variant | factoid | knowledge_update | multi_session | temporal | **Pooled R@10** |
|---|---|---|---|---|---|
| naive_bm25 | 0.60 | 1.00 | 0.30 | 0.90 | **0.70** |
| fathomdb_fts_only | 0.90 | 1.00 | 0.30 | 1.00 | **0.80** |
| fathomdb_graph_OFF | 0.50 | 1.00 | 0.30 | 0.80 | **0.65** |
| fathomdb_graph_ON | 0.60 | 1.00 | 0.10 | 0.90 | **0.65** |

Pooled secondary metrics (equal class sizes):

| Variant | R@5 | R@10 | R@20 | MRR | nDCG@10 |
|---|---|---|---|---|---|
| naive_bm25 | 0.700 | 0.700 | 0.825 | 0.701 | 0.658 |
| fathomdb_fts_only | 0.725 | 0.800 | 0.800 | 0.640 | 0.633 |
| fathomdb_graph_OFF | 0.575 | 0.650 | 0.675 | 0.489 | 0.468 |
| fathomdb_graph_ON | 0.575 | 0.650 | 0.725 | 0.512 | 0.636 |

**Observations:**
- `graph_ON` − `graph_OFF` = **0.00 pooled R@10**. The two diverge only inside classes and
  *cancel*: ON gains factoid (+0.10) and temporal (+0.10) but **loses `multi_session`
  (0.30 → 0.10)** — the BFS arm displaced gold docs from top-10 on its own target class.
- `graph_*` (0.65) sits **below `fts_only` (0.80)** — entities in the index hurt.

### 4.2 40-question sweep — PRE anti-pollution filter (`/tmp/gar_n40.json`)
Same corpus, before the `kind="doc"` result filter was added:

| Variant | factoid | knowledge_update | multi_session | temporal | **Pooled R@10** |
|---|---|---|---|---|---|
| naive_bm25 | 0.60 | 1.00 | 0.30 | 0.90 | **0.70** |
| fathomdb_fts_only | 0.90 | 1.00 | 0.30 | 1.00 | **0.80** |
| fathomdb_graph_OFF | 0.40 | 1.00 | 0.20 | 0.80 | **0.60** |
| fathomdb_graph_ON | 0.50 | 0.90 | 0.20 | 0.80 | **0.60** |

The filter lifted both graph variants by **+0.05** equally (0.60 → 0.65) — it removes entity
rows from the *returned list* (display) but **does not change the ON−OFF null** (still 0.00).
The IDF distortion that pushes both graph variants below `fts_only` is upstream of the
result filter (it is in the corpus statistics), so the filter cannot recover it.

### 4.3 N=4 dry-run (`/tmp/gar_dry.json`)
201 sessions, 1 question/class. Functional smoke only (not statistically meaningful): all
four variants land gold in top-10 on most classes; it confirmed the pipeline end-to-end
(extraction → build → resolve → score) before the 40-question run. Notable only as a
plumbing check — e.g. `graph_OFF` missed the single factoid (R@10 0.0) while `graph_ON`
recovered it (R@10 1.0), and `multi_session` flipped the other way — i.e. n=1 noise.

### 4.4 Prior §3 base-retrieval (n=160, paired, seed 20260614)
Context: the dense/fused arm was tested first and **also failed to beat BM25**.

Recall@K, pooled (`0.8.1-p0a-fused-recall-n160.json`):

| Variant | R@5 | R@10 | R@20 | MRR | nDCG@10 |
|---|---|---|---|---|---|
| naive_bm25 | 0.506 | **0.625** | 0.694 | 0.510 | 0.479 |
| fathomdb_fts_only | 0.500 | 0.562 | 0.662 | 0.405 | 0.402 |
| fathomdb_fused (dense+FTS) | 0.481 | 0.606 | 0.688 | 0.422 | 0.416 |

- Fused R@10 Δ vs BM25 = **−0.019** (within the ~11 pp pooled MDE → a *tie*); fused − FTS
  = +0.044 (dense genuinely adds over lexical-only). `multi_session` is the one class where
  fused edges BM25 (0.325 vs 0.275). Ranking still trails BM25 (MRR −0.088, nDCG −0.064).

End-to-end answer accuracy, gemini-3.1-pro answerer + gemini-3.5-flash judge, aistudio batch
(`0.8.1-p0a-base-e2e-n160-gemini31pro.json` for bm25/fts;

## 5. Analysis

### Finding 1 — The BFS arm adds ~0 recall (graph_ON ≈ graph_OFF)
Pooled R@10 is identical (0.65) with the arm on or off, and within-class the arm's
contributions cancel — it even **degrades `multi_session`** (0.30 → 0.10 post-filter), the
exact class it was built to help. The arm *is* live (50/50 resolved graph hits on the
probe), so this is not a wiring failure: the candidates it surfaces are simply **not gold
docs that the two-arm missed in top-10**. The graph adds redundant or off-target candidates
that, after RRF, push some gold out of the cut as often as they pull gold in.

**Why (root cause = seeding strategy).** C1 seeds the BFS frontier from the **graph's own
entity-node FTS surfaces** (source B), *not* from the query's lexical top-K documents. The
retrieval literature is unanimous (§6) that raw, unseeded/self-seeded BFS over a lexical
baseline yields ~0 lift, and that traversal pays off **only when seeded from the lexical
top-K**. Our seeding is the one configuration the literature says won't work — the most
likely cause of the zero lift.

### Finding 2 — Entity co-mingling *hurts* (fts_only 0.80 → graph 0.65)
Adding 28,883 short entity rows to the same FTS index that holds the session documents drops
recall by 0.15 vs a docs-only FTS index — and it drops it for `graph_OFF` too, i.e. *before*
the BFS arm is even consulted. This is **BM25 document-length-normalization bias /
corpus-heterogeneity ("Verboseness Fission")**, not "pollution" in the pejorative sense:
thousands of very short entity rows shift the corpus's average document length and term-IDF
statistics, which re-weights scoring for the long session docs and demotes some gold. The
`kind="doc"` result filter removes entity rows from the *output* (+0.05 display) but cannot
undo the corpus-statistic distortion, which is upstream. **Fix is a separate entity index**,
not a result filter.

### Cross-check with §3
Both engine-side retrieval levers tried in 0.8.1 — dense/fused (§3) and the graph arm — land
at **tie-or-worse vs BM25**. This is consistent with the prior relevance-ceiling finding:
on this corpus the embedder/engine machinery buys ≈0 product value over a strong lexical
baseline. BM25 (and FTS5) are simply strong here.

## 6. Literature corroboration

Independent sister-agent deep-research synthesis (high confidence; adversarially verified):

- **Raw/unseeded BFS adding ~0 over strong lexical is the NORMAL result** (verified 3-0).
  *SPRIG* (arXiv:2602.23372): HotpotQA BM25 R@10 0.742, GraphHybrid 0.775, but **plain RRF
  fusion 0.851 — fusion beats the graph arm.** The claim "graph beats BM25 ~46%" was
  **REFUTED (0-3)**: that holds only with **LLM-reasoning-guided traversal**, not raw BFS.
- **Graph traversal pays off ONLY when seeded from the lexical top-K** (3-0). → Directly
  indicts our design: C1 seeds from the graph's own FTS surfaces, not the lexical hits =
  the likely zero-lift cause (Finding 1).
- **Co-mingling heterogeneous-length items is a documented length-normalization failure**
  (BM25 `b` parameter). The 0.80 → 0.65 drop is the predicted direction. Fix = separate
  indexes (Lipani & Lupu, *Verboseness Fission*; *RMM* arXiv:2503.08026) — corroborates
  Finding 2.
- **BM25 is a strong out-of-distribution baseline** (BEIR, arXiv:2104.08663). Reported
  LongMemEval gains come from **query routing + index-key enrichment** (+9.4% recall from
  using extracted facts as document keys), **not** from a graph arm.
- **KILLED — do not cite as established:** "RRF < index-merge"; and two LongMemEval preprint
  numbers (FTS5 0.745 vs BM25 0.634; routing → 0.800) — single-author preprints mirroring
  our own setup, not independent evidence.

The measured negative is the *expected* outcome under the verified literature, which raises
confidence that it is real and not an artifact of N=40.

## 7. Engine bugs / follow-ups

Found while building the graph engine (all currently worked around in the harness):

1. **Edge-with-body auto-enqueues `edge_fact` vector projection** (`lib.rs:8594`). With no
   embedder configured → `SchedulerError`. The harness therefore writes edges **without a
   body**, which turns off **source-A (edge-fact FTS) seeding**.
2. **`StorageError` in the edge_fact embedding pipeline under write load (~8–14k edges)**
   even *with* an embedder — reproduced on trivial synthetic edges. An engine scale bug.

Net effect on this experiment: the graph arm ran on source-B + traversal only (a recall
lower bound). The pivot decision (§9) makes fixing these **not worth doing now** — the
literature says source-A seeding wouldn't move the result, only correct seeding would.

## 8. Conclusions

1. **The graph arm does not beat BM25 on LongMemEval recall@K.** The BFS arm contributes ~0
   recall (graph_ON ≈ graph_OFF = 0.65 pooled; BM25 0.70, FTS-only 0.80), and even degrades
   its target `multi_session` class. Robust same-engine A/B; reproduces before/after the
   anti-pollution filter.
2. **Mixing entities into the document FTS index actively hurts recall** (−0.15 vs docs-only
   FTS) via length-normalization/corpus-heterogeneity, fixable only by a separate index.
3. Both engine retrieval levers in 0.8.1 (dense/fused §3, and the graph arm) land at
   **tie-or-worse vs BM25** — consistent with the relevance-ceiling finding; the literature
   says this is the expected result for raw BFS over a strong lexical baseline.
4. The literature points to the **seeding strategy** (seed from lexical top-K, not the
   graph's own FTS) and **index-key enrichment** as the levers that actually move
   LongMemEval — not a graph arm.

---

## 9. Recommendations / the pivot (ranked)

**DECISION (HITL, 2026-06-16): PIVOT.**

1. **DROP** (negative expected value per data + literature):
   - the planned **source-A** (more graph-surface seeding) work,
   - the **engine `edge_fact` fix** (§7),
   - the **n=160-as-designed** graph run (~0 expected value; the N=40 null is decisive).

2. **PIVOT → index-key enrichment** (primary; evidence-backed +9.4% LME lever). Append each
   session's extracted entities/facts to **that session's own doc FTS content** — keys live
   on the doc, not as separate rows. No graph arm, no BFS, **no length-normalization
   pollution** (no short entity rows in the corpus), and it **reuses the 1,907 cached
   graphs** → testable at 40 questions in minutes. This is the cheapest path to a real
   LongMemEval win.

3. **Secondary fallback (only if enrichment underperforms):** the one graph configuration
   the literature supports — **re-seed BFS from the lexical top-K** (not the graph's own
   FTS) + a **separate entity FTS index** (fixes Finding 2) + RRF fusion.

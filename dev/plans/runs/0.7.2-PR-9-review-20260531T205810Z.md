## Final disposition: PASS (pass 5) — prior BLOCK closed; zero open findings

Pass 4 was a fresh full review (prompted by HITL "have codex review the diff") and
returned **BLOCK**: the circuit breaker, as first built, bounded only CONSECUTIVE
timeouts (`consecutive_embed_timeouts`, reset on any returning embed) — so an
embedder that hangs on some inputs but returns on others would reset the streak on
every interleaved success and leak one abandoned thread per hung input forever,
never latching; the test covered only the all-timeout case. The BLOCK was NOT
overridden. **Fix:** redesigned the breaker to bound CONCURRENT LIVE embed threads
(`live_embed_threads`: incremented before each watchdog spawn, decremented by the
thread on completion incl. the panic path); the latch is made under `embed_serialize`
(race-free) when live threads reach the threshold. Added
`intermittent_hang_still_trips_circuit_breaker` (regression guard for exactly the
BLOCK case). Pass 5 re-reviewed and returned **PASS** (blocker closed; accounting
correct incl. panic path; no race defeats the bound; single-writer/cursor/panic
transparency intact; rationale aligned). Its one [low] comment-drift finding (the
persistent-hang test still said "consecutive") was then cleaned up — zero open
findings.

## Disposition history — CONCERN findings resolved across passes 1–3

### Pass 1 — CONCERN-1 [medium]: timeout relaxes strict serialization
On the timeout path the worker drops `embed_serialize` while the abandoned
detached embed thread keeps running, so a later embed can briefly overlap it.
**Disposition: ACCEPTED as the spec-mandated trade-off.** The PR-9 prompt
explicitly requires "drop the guard before/at timeout … no poisoned Mutex
wedge"; the alternative (hold the guard inside the embed thread) is what the
spec/ADR forbid because a hung embed would then deadlock the pool. The
relaxation is fault-path only. (Also drove the circuit breaker, below.)

### Pass 2 — CONCERN-1 [medium]: unbounded abandoned-thread leak; CONCERN-2 [low]: comment drift
- **Leak → FIXED** by adding the embed circuit breaker (`embed_circuit_open`,
  threshold `DEFAULT_EMBED_CIRCUIT_THRESHOLD = 8`): consecutive watchdog
  timeouts latch the breaker, after which jobs fail fast without embedding.
- **Comment drift → FIXED**: removed the withdrawn "~13x" claim and the
  "no serialization lock held during embed" inaccuracy.

### Pass 3 — CONCERN-1 [medium]: residual post-latch spawn window
The per-attempt re-check closed the retry-sleep gap, but a worker blocked on
`embed_serialize.lock()` at the moment of latch could still spawn one more
watchdog thread (≤1 with PROJECTION_WORKERS=2 — already bounded, not the
original unbounded leak). **FIXED** by adding a final breaker re-check
immediately after acquiring the guard and before the embed call
(`run_projection_job`). Pass-3 explicitly confirmed: the retry-loop re-check is
correctly placed, the test now exercises the multi-retry path (`[10,10]`), and
both comment fixes are accurate.

## Net engineering changes driven by this review (and by HITL challenge)
- Serialization re-justified on **safety** (never invoke arbitrary
  caller-supplied pyo3/napi embedders concurrently), not throughput. A
  micro-benchmark (`pr9_embed_microbench.rs`) proved the watchdog thread-spawn
  is perf-neutral (release: 14.2ms direct vs 13.7ms watchdog) and candle uses
  one process-wide rayon pool, so serialization is throughput-neutral; the
  earlier "~13x" figure (a debug-vs-release artifact) was withdrawn everywhere.
- Circuit breaker added (bounds abandoned-thread leak from a persistently-hung
  embedder); breaker re-checked at entry, per retry attempt, and post-lock.

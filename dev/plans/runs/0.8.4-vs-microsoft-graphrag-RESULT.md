# 0.8.4 — LITERAL head-to-head: FathomDB vs a RUNNING Microsoft GraphRAG (the real verdict)

**Date:** 2026-06-23 · **What:** **Microsoft GraphRAG 3.1.0** (pip-installed; real pipeline:
entity/relationship/claim extraction → **Leiden hierarchical communities** → **115 LLM community
reports** → relevance-scored global map-reduce; LLM = gpt-5.4 via airlock, embeddings via a local
shim) indexed 15 AP-News articles. Its **global-search** query path was run head-to-head against
FathomDB arms over the **same 15 documents** (fair corpus), **cross-family** judged (claude-haiku,
gpt-5.4 reader), 5 runs, order-swapped, through `decide_084`. Script:
`dev/plans/runs/0.8.4-vs-microsoft-graphrag.py`.

## Verdict: **Microsoft GraphRAG WINS decisively — FathomDB does NOT reach parity**

Win-rate >0.5 = FathomDB better (so FathomDB needs CI-lower ≥ 0.45 for near-parity):

|---|---|---|---|
| `fathomdb_mapreduce` (raw-text synthesis) | **0.062** [0.00,0.19] | 0.319 [0.16,0.48] | 0.231 [0.06,0.40] |
| `fathomdb_vector` (FathomDB retrieval) | **0.000** [0.00,0.00] | 0.325 [0.13,0.54] | 0.069 [0.00,0.19] |

`decide_084` → **NOT_REACHED** on both. These are **well below the near-parity band (0.45)** — not
merely underpowered; comprehensiveness is **0.00–0.06** (FathomDB essentially never wins
comprehensiveness vs GraphRAG). **FathomDB is far from parity with Microsoft GraphRAG on global
sensemaking.**

## This REVERSES the earlier provisional resolution — and why (honesty)

`0.8.4-graphrag-RESOLUTION.md` provisionally concluded "the community-summary paradigm is the graph
null (don't fund S1)" — based on a **crude community-S1 reimplementation** (pure-Python label
propagation, 60-doc subset, single-pass community reports). That implementation lost to raw-text
synthesis, and I wrongly generalized that to the paradigm.

**The actual Microsoft GraphRAG decisively beats both FathomDB arms** — including the raw-text
map-reduce that beat my crude community-S1. So the earlier negative was an **implementation-quality
artifact, not a paradigm failure.** Real GraphRAG's hierarchical communities + 115 relevance-scored
community reports + claim extraction produce a global synthesis FathomDB's flat retrieval/map-reduce
cannot match. **The community-summary paradigm DOES pay off when implemented well — Microsoft's is, my
reimplementation was not.**

## What this means for the 0.8.4 goal (measured, against the real competitor)

- **The goal — FathomDB parity-or-better vs GraphRAG on sensemaking — is NOT met.** Measured against a
  running Microsoft GraphRAG, FathomDB loses decisively (win-rate 0.00–0.33).
- **Implication — this JUSTIFIES funding a serious S1 build** (a *real* GraphRAG-grade build:
  hierarchical Leiden communities + high-quality community reports + relevance-scored global search),
  not a crude one. FathomDB currently lacks this capability and the gap is large.
- **Alternatively, a measured GraphRAG win is itself a valid, publishable result** (roadmap §"Honest
  expectation"): it resolves that GraphRAG-style sensemaking *is* worth productizing — the opposite of
  the third-graph-null lean, now corrected by the literal comparison.

## Honest caveats

- **15-doc index, n=8 questions** (underpowered) — but the comprehensiveness gap (0.00–0.06, tight CIs)
  is decisive, not a power artifact. A full-corpus index + powered N would sharpen, not reverse, it.
- **Length:** GraphRAG answers were longer (~4–6k chars vs FathomDB ~1–3.5k); a length-bias contribution
  is possible. But the gap spans all three metrics (incl. diversity/empowerment, less length-sensitive)
  and comprehensiveness ≈ 0, so length alone does not explain a decisive loss. (The directness/length
  corroboration ran; for a registered claim, power up + length-control explicitly.)
- This is the **global-sensemaking** axis. HippoRAG-2 multi-hop (reference-based F1) is a separate,
  unrun cross-check.

## Net

The literal FathomDB-vs-running-Microsoft-GraphRAG head-to-head is the result the goal names: **Microsoft
GraphRAG beats FathomDB on sensemaking; FathomDB is not at parity.** The corrected program decision is to
**fund a real (Microsoft-grade) S1 build** to close the gap — or to record the measured GraphRAG win as
the publishable outcome. Either way, the earlier "don't fund S1 / graph null" lean is **withdrawn** — it
was an artifact of a weak reimplementation, not the paradigm.

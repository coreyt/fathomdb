+# FTS5 Tokenizer Latency Experiment — Report (0.8.0 / B2)
+
+**Date:** 2026-06-07 · **Author:** Slice 6 (`[research/experiment]`) · **Status:** complete; recommendation pending HITL
+
+This report documents the experiment run to ground the **B2** decision from the Slice-40 release-readiness
+HALT: a measured FTS5 query-latency regression on the AC-012 gate, attributed at HALT time to the Slice-5
+tokenizer upgrade. It describes what was measured, how, what the data shows, and what to do about it.
+
+> Companions: the decision-oriented findings doc and machine-readable closure live at
+> `dev/plans/runs/0.8.0-slice-6-tokenizer-experiment-20260607T003001Z.md` and
+> `dev/plans/runs/0.8.0-slice-6-output.json`. The reusable measurement harness is
+> `src/rust/crates/fathomdb-engine/tests/slice6_tokenizer_experiment.rs` (gated behind
+> `SLICE6_EXPERIMENT=1`; it never runs in `cargo test` / `agent-verify` / CI).
+
+---
+
+## 1. Background — the question
+
+The Slice-40 verification battery (run with `AGENT_LONG=1`, which the per-push CI never sets) reported the
+AC-012 text-query latency gate failing at the 100k-row tier:
+
+- **AC-012 @100k:** p50 ≈ 49.9 ms (budget 20 ms) / p99 ≈ 315 ms (budget 150 ms)
+- **v0.7.2 baseline @100k:** p50 ≈ 16 ms / p99 ≈ 56 ms — passing
+
+Slice-40 attributed the ~3× regression to the Slice-5 (G1) migration, which upgraded the `search_index`
+FTS5 tokenizer from the SQLite default to `porter unicode61 remove_diacritics 2`, and routed the decision
+to HITL: **remediate** (cheaper tokenizer / query-path optimization) vs **re-ratify** the AC-012 budget.
+
+The HITL ruling (2026-06-06) asked for an **experiment, not a blind fix**: measure latency vs FTS quality
+across candidate tokenizer configs and query-path levers, assess how accessible/modifiable the relevant
+source is, and return findings with explicit decision points. This report is that experiment.
+
+### The AC-012 path under test
+
+`ac_012` opens an engine **without** an embedder, seeds a synthetic Zipfian corpus (≈500-byte bodies built
+from a 1,024-token vocabulary), and issues **single-token** queries drawn from the 50th–90th-percentile
+term-frequency band. Each query runs the engine's text branch SQL verbatim (`lib.rs:3923-3927`):
+
+```sql
+SELECT body, kind, write_cursor, bm25(search_index)
+FROM search_index
+WHERE search_index MATCH ?1
+ORDER BY write_cursor      -- note: NO LIMIT (unbounded result set)
+```
+
+The budget (p50 ≤ 20 ms AND p99 ≤ 150 ms) is asserted **unconditionally at N = 100,000**. Its sibling
+latency ACs (AC-013/AC-019) were HITL-revised in 0.7.2 to a **tiered** budget (AC-072/AC-073): the gate
+binds only at the **10k tier** for the 0.x/1.x lines, with 100k/1M as tracked post-1.0 targets, because the
+underlying scan is O(N). AC-012 was never given that treatment.
+
+---
+
+## 2. Method
+
+All latency numbers come from **release** builds (`cargo test --release`, matching `perf-canonical.yml`),
+each run **in isolation** (no concurrent `cargo test` — workspace-concurrent runs corrupt timing, a caveat
+established at Slice 40). SQLite/FTS5 is the **bundled** vendored amalgamation (`rusqlite 0.31`,
+`features=["bundled"]`); `porter`, `unicode61`, and `ascii` are SQLite built-ins selected via the
+`tokenize=` string.
+
+**Box:** x86_64, 24 logical cores, Linux 6.17. This differs from both the Slice-40 measurement box and the
+canonical x86_64 tier-1 CI runner, so **absolute milliseconds are box-specific** — the controlled
+A/B and tier conclusions are what transfer, not the magnitudes.
+
+Three complementary instruments were used:
+
+1. **Real `ac_012`** — the production engine path, authoritative for "does the gate pass." Run at
+   `AC012_CORPUS_N` ∈ {10000, 100000}. It reports percentiles via the gate's coarse power-of-two-microsecond
+   bucketing (so its p50/p99 are bucket ceilings).
+
+2. **A faithful FTS-path harness** (`tests/slice6_tokenizer_experiment.rs`) — reproduces the engine's text
+   path exactly (same FTS5 schema, the **byte-identical** synthetic corpus via the same seeds/generators as
+   `seed_ac012_corpus`, the same single-token band, the same SQL) and varies **only** the `tokenize=` clause.
+   It reports finer sorted-array percentiles over 1,000 samples (warmup pass discarded) and deliberately
+   bypasses the ReaderWorkerPool (a tokenizer-independent constant), so its absolutes read below the gate —
+   faithfulness is cross-checked against the real `ac_012`.
+
+3. **A full-engine A/B** — to isolate the tokenizer's contribution at the engine level, the step-11 `tokenize`
+   string was **temporarily** set to v0.7.2's `unicode61`, `ac_012` @100k re-run, then **reverted** to
+   `porter unicode61 remove_diacritics 2`. (This temporary edit is not part of any committed change.)
+
+---
+
+## 3. Results
+
+### 3.1 Q1 — at which tier does the regression breach? (real `ac_012`, release, isolated)
+
+| Tier (N) | p50 | p99 | vs 20/150 ms budget | Gate verdict |
+|---------:|----:|----:|---------------------|--------------|
+| **10,000**  | < 1 ms | 4 ms  | well within | **PASS** (wide margin) |
+| **100,000** | 21 ms  | 99 ms | p50 over by ~1 ms; p99 within | **FAIL** (p50 only) |
+
+The regression breaches **only at the 100k tier**, and there only on p50 (by ~1 ms); p99 (99 ms) is
+comfortably within 150 ms. At the **10k tier — the binding tier for the sibling latency ACs** — `ac_012`
+passes by a wide margin.
+
+> The Slice-40 magnitude (49.9 / 315 ms) was **not reproduced** on this idle box (got 21 / 99 ms). The
+> *direction* reproduces; the *magnitude* is hardware/load-sensitive — the same effect class as B3 (`ac_020`).
+
+### 3.2 Q2a — tokenizer config × tier latency sweep (faithful FTS-path harness)
+
+`avg_rows` = mean matched rows per query (the cost driver); `pages` = `PRAGMA page_count` (index size).
+
+| Config | tier | pages | avg_rows | p50 (ms) | p99 (ms) |
+|--------|-----:|------:|---------:|---------:|---------:|
+| `porter unicode61 remove_diacritics 2` **(current)** | 10k | 76 | 320 | 0.323 | 0.898 |
+| `unicode61 remove_diacritics 2` | 10k | 64 | 320 | 0.326 | 0.912 |
+| `porter unicode61` | 10k | 68 | 320 | 0.331 | 0.921 |
+| `unicode61` **(≈ v0.7.2 default)** | 10k | 65 | 320 | 0.327 | 0.901 |
+| `porter ascii` | 10k | 66 | 320 | 0.329 | 0.917 |
+| `ascii` | 10k | 63 | 320 | 0.330 | 0.921 |
+| `porter unicode61 remove_diacritics 2` **(current)** | 100k | 18047 | 3212 | 3.997 | 16.004 |
+| `unicode61 remove_diacritics 2` | 100k | 18047 | 3212 | 3.934 | 15.934 |
+| `porter unicode61` | 100k | 18047 | 3212 | 3.978 | 16.114 |
+| `unicode61` **(≈ v0.7.2 default)** | 100k | 18047 | 3212 | 3.969 | 16.147 |
+| `porter ascii` | 100k | 18047 | 3212 | 4.046 | 16.042 |
+| `ascii` | 100k | 18047 | 3212 | 4.001 | 16.093 |
+
+**All six configs are within run-to-run noise of each other** at both tiers, with identical `avg_rows` and
+(at 100k) identical `pages`. The reason: `porter` is a **no-op on the synthetic AC-012 vocabulary** — its
+tokens are `{a}{b}{c}{i:04}` (three letters + four digits), and SQLite's Porter stemmer passes digit-bearing
+tokens through unchanged. The term set, posting lists, and index are bit-for-bit identical regardless of
+tokenizer.
+
+### 3.3 Q2b — full-engine A/B (real `ac_012`, release, isolated, @100k)
+
+| Tokenizer (step-11) | p50 | p99 |
+|---------------------|----:|----:|
+| `porter unicode61 remove_diacritics 2` (head) | **21 ms** | 99 ms |
+| `unicode61` (v0.7.2 default, temporarily swapped + reverted) | **20 ms** | 99 ms |
+
+Identical within noise at the **full engine level**, and the v0.7.2 tokenizer *also* sits at the 20 ms p50
+boundary. **This is the decisive control: the tokenizer is not the latency lever, and the Slice-40
+attribution of the regression to the tokenizer is refuted by measurement.**
+
+### 3.4 Q2c — FTS quality proxy (what the tokenizer actually buys)
+
+Eight query→variant pairs exercising morphology and diacritics; `bits` order is
+`[run running ran optimize cafe naive resume connect]`.
+
+| Config | recall | bits | what it loses |
+|--------|:------:|------|---------------|
+| `porter unicode61 remove_diacritics 2` **(current)** | **8/8** | 11111111 | — |
+| `porter unicode61` | 8/8 | 11111111 | — (diacritics already folded by default `remove_diacritics 1`) |
+| `unicode61 remove_diacritics 2` | 5/8 | 01101110 | **stemming**: run→runs, optimize→optimizing, connect→connected |
+| `unicode61` **(≈ v0.7.2)** | 5/8 | 01101110 | **stemming** (same 3) |
+| `porter ascii` | 5/8 | 11110001 | **diacritics**: cafe→café, naive→naïve, resume→résumé |
+| `ascii` | 2/8 | 01100000 | **both** |
+
+`porter` buys morphological recall (+3); diacritic folding buys accent-insensitivity (+3) — but the
+diacritic wins are already delivered by `unicode61`'s **default `remove_diacritics 1`**. So the *incremental*
+FTS-quality gain of the Slice-5 upgrade over v0.7.2's default `unicode61` is specifically the **Porter
+stemmer**. Because latency is config-independent, **the tokenizer choice is a pure quality decision,
+decoupled from the latency gate.**
+
+### 3.5 Q3 — where is the cost? (faithful harness, @100k, current tokenizer)
+
+| Query shape | avg_rows matched | p50 (ms) | p99 (ms) |
+|-------------|-----------------:|---------:|---------:|
+| 1 token | 3,212 | 3.93 | 15.88 |
+| 2 tokens (`AND`) | 2 | 0.25 | 0.55 |
+| 3 tokens (`AND`) | 0 | 0.19 | 0.32 |
+
+**Latency tracks result-set size, not tokenization.** A single common-band token matches ≈3,212 rows; the
+engine scans the posting list, computes `bm25()` per row, sorts by `write_cursor`, and materializes **all**
+of them (the SQL is unbounded). A second `AND` token collapses the result set to ≈2 rows → ≈16× faster.
+Query-time Porter stemming is one stem per token — negligible (and free at the engine level per §3.3). The
+only genuine *query-path* lever is bounding the matched-row materialization (a top-k `LIMIT`; a dormant,
+experiments-gated `FATHOMDB_PERF_SEARCH_LIMIT` knob exists at `lib.rs:3912`) — but that is a recall-affecting
+semantic change, not a tokenizer change.
+
+### 3.6 Q4 — source accessibility / modifiability
+
+| Lever | Accessibility | Rough cost | Latency motivation (from this experiment) |
+|-------|---------------|------------|-------------------------------------------|
+| Swap built-in tokenizer options (`porter`/`unicode61`/`ascii`/`remove_diacritics N`) | **Trivial** — the migration `tokenize=` string (FTS5 has no `ALTER tokenize`, so a drop+recreate+reproject step like step-11) | Low | **None** — config is latency-neutral; pure quality knob |
+| Register a **custom FTS5 tokenizer** (`fts5_api` / `xCreateTokenizer`) | **Feasible but awkward** — `rusqlite` does not expose `fts5_api` ergonomically; needs `unsafe` FFI to the bundled FTS5 C API or a C shim | Medium–high | **None for 0.8.0** — only justified by a future *quality* need |
+| Embedder source (candle / bge-small) | In-tree | n/a here | **Irrelevant** — the FTS5 text path never calls the embedder; it bears only on the vector/AC-013 latency (separate, already tiered) |
+
+The source is fully accessible at the cheap (config) tier and that tier suffices — but there is no latency
+reason to reach for any of these levers.
+
+---
+
+## 4. Conclusion
+
+The measured evidence is consistent and points one direction:
+
+- The AC-012 latency is an **O(N) corpus-scaling** cost (posting-list scan + `bm25()` scoring + unbounded
+  row materialization), **structurally identical** to the vec0 bit-KNN O(N) scan that HITL already tiered
+  for AC-013/AC-019.
+- The Slice-5 **tokenizer is not the cause** — proven independently by the config sweep (§3.2) and the
+  full-engine A/B (§3.3). The Slice-40 root-cause attribution is **measured-false**; the head-vs-v0.7.2 gap
+  it observed came from measurement conditions and/or tokenizer-independent engine overhead, not from
+  `porter`/`remove_diacritics`.
+- At the **binding 10k tier** the gate passes by a wide margin (§3.1); it breaches only the **unconditional
+  100k tier**, which — unlike its siblings — was never tiered.
+- **Lightening the tokenizer is the wrong lever**: ≈0 ms latency gain and −3/8 FTS recall (§3.4).
+
+### Recommendation
+
+**Tier AC-012 to match AC-013/AC-019** (10k binding, 100k/1M tracked-not-gated) via the existing,
+HITL-locked **AC-072/AC-073 precedent** (`ADR-0.7.0-text-query-latency-gates-revised`). Keep the Slice-5
+Porter tokenizer — its FTS-quality gain is real and its latency cost is nil. Mechanically this means minting
+a superseding AC that asserts the 20/150 ms budget only at a `AC012_GATE_N = 10_000` tier (mirroring
+`ac_013`'s existing branch) and reporting 100k/1M as tracked targets — a change for a follow-on gated slice,
+not done in this experiment.
+
+### Decision points for HITL
+
+| Option | Verdict |
+|--------|---------|
+| **(i) Tier AC-012** *(recommended)* | **Best** — precedented, latency-honest, preserves G1 quality, passes the binding tier |
+| (ii) Lighten the tokenizer | **Reject** — ≈0 ms gain, −3/8 FTS recall; wrong lever |
+| (iii) Custom FTS5 tokenizer | **Not for 0.8.0** — medium-high FFI cost, no latency motivation |
+| (iv) Re-ratify the unconditional 100k budget | **Viable but off-pattern** — inconsistent with the siblings' tiering; the synthetic 100k O(N) target is exactly what AC-072 deferred to post-1.0 ANN work |
+| (v) Bound the FTS result set (top-k `LIMIT`) | **Separate, larger decision** — would *meet* (not tier) the 100k budget but changes recall semantics; needs a dedicated retrieval ADR |

+## TL;DR (the headline, then the numbers)
+
+1. **The Slice-5 tokenizer upgrade is NOT the cause of the AC-012 latency.** Two independent controlled
+   measurements — a faithful FTS-path config sweep AND a full-engine A/B that swaps only the tokenizer —
+   show `porter unicode61 remove_diacritics 2` and the v0.7.2 default `unicode61` are **latency-identical**
+   at every tier. The Slice-40 root-cause attribution ("the porter stemmer + diacritics folding is
+   materially more expensive at query time") is **refuted by measurement**.
+2. **The latency scales with corpus size N, not the tokenizer.** AC-012's single-token queries hit a
+   common-band term whose posting list grows linearly with N (≈320 rows @10k → ≈3,212 rows @100k); the cost
+   is the O(N) FTS5 MATCH scan + `bm25()` scoring + materializing **every** matched row (the engine query is
+   unbounded — no `LIMIT`). This is the **same O(N) story** that already drove the HITL tiering of the
+   sibling latency ACs (AC-013/AC-019 → AC-072/AC-073).
+3. **At the BINDING 10k tier the regression does not breach** (real `ac_012` p50 < 1 ms, p99 4 ms vs the
+   20/150 ms budget). It breaches **only at the unconditional 100k tier**, which — unlike its siblings —
+   was never tiered.
+4. **Lightening the tokenizer buys ~0 ms and costs FTS quality** (recall 8/8 → 5/8 on a morphology/diacritic
+   proxy). It is the wrong lever.
+
+→ **Recommended HITL path: tier AC-012 to match AC-013/AC-019** (10k binding, 100k/1M tracked-not-gated),
+via the existing AC-072/AC-073 precedent. Full fork in §Recommendation.

+## Q3 — query-path: where is the cost?
+
+Faithful harness @100k, current tokenizer, 1-/2-/3-token queries from the same band:
+
+| Query shape | avg_rows matched | p50 (ms) | p99 (ms) |
+|-------------|-----------------:|---------:|---------:|
+| 1 token  | 3,212 | 3.93 | 15.88 |
+| 2 tokens (`AND`) | 2 | 0.25 | 0.55 |
+| 3 tokens (`AND`) | 0 | 0.19 | 0.32 |
+
+**Latency tracks result-set size, not token count / tokenization.** A single common-band token matches
+~3,212 rows → the engine scans the posting list, computes `bm25()` per row, sorts by `write_cursor`, and
+materializes **all** of them (the SQL is unbounded — no `LIMIT`). Add a second `AND` token and the result
+set collapses to ~2 rows → 16× fewer rows → ~16× faster. Query-time porter stemming is one stem per token =
+negligible (and Q2b shows it's free even at the engine level).
+
+**Implication:** the only real *query-path* lever is **bounding the matched-row materialization** (a top-k
+`LIMIT` on the FTS branch — there is already a dormant, experiments-gated `FATHOMDB_PERF_SEARCH_LIMIT` knob
+at `lib.rs:3912`). That is a **recall-affecting semantic change**, not a tokenizer change, and is **out of
+scope** here; flagged for HITL as a distinct, larger lever if the 100k budget is to be *met* rather than
+*tiered*.
+
+> Secondary observation (not load-bearing): the real engine adds ~17 ms p50 @100k over the raw-FTS harness
+> (21 ms vs ~4 ms), and this overhead also grows with matched-row count (ReaderWorkerPool dispatch +
+> dedup-on-body over ~3,212 ~500-byte bodies + `SearchHit` materialization). It is **tokenizer-independent**.
+> Precise attribution within that overhead was not isolated (would need a 0.7.2-engine A/B); the load-bearing
+> claim — tokenizer is latency-neutral — stands on Q2a + Q2b.

+/// Run the EXACT engine text SQL for each query; warmup pass discarded, then a
+/// measurement pass. Returns sorted-percentile p50/p99 (finer than the gate's
+/// power-of-two `percentile_ceil`; the gate bucketing is discussed in the doc).
+fn measure_latency(conn: &Connection, queries: &[String]) -> (Duration, Duration, f64) {
+    let sql = "SELECT body, kind, write_cursor, bm25(search_index) FROM search_index \
+               WHERE search_index MATCH ?1 ORDER BY write_cursor";
+    // match_expression form from fathomdb-query::compile_text_query: a single
+    // token becomes "\"token\"".
+    let exprs: Vec<String> =
+        queries.iter().map(|q| format!("\"{}\"", q.replace('"', "\"\""))).collect();
+
+    // Warmup (discarded).
+    for e in &exprs {
+        let mut stmt = conn.prepare(sql).unwrap();
+        let _ = stmt
+            .query_map([e.as_str()], |row| row.get::<_, String>(0))
+            .unwrap()
+            .collect::<Result<Vec<_>, _>>()
+            .unwrap();
+    }
+    // Measurement.
+    let mut samples = Vec::with_capacity(exprs.len());
+    let mut total_rows = 0usize;
+    for e in &exprs {
+        let started = Instant::now();
+        let mut stmt = conn.prepare(sql).unwrap();
+        let rows: Vec<(String, String, i64, f64)> = stmt
+            .query_map([e.as_str()], |row| Ok((row.get(0)?, row.get(1)?, row.get(2)?, row.get(3)?)))
+            .unwrap()
+            .collect::<Result<Vec<_>, _>>()
+            .unwrap();
+        samples.push(started.elapsed());
+        total_rows += rows.len();
+    }
+    samples.sort();
+    let p = |q: f64| samples[((samples.len() as f64 * q) as usize).min(samples.len() - 1)];
+    let avg_rows = total_rows as f64 / exprs.len() as f64;
+    (p(0.50), p(0.99), avg_rows)

        Self { state: seed.wrapping_mul(0x9E37_79B9_7F4A_7C15).wrapping_add(1) }
    }

    fn next_u64(&mut self) -> u64 {
        self.state = self.state.wrapping_mul(6364136223846793005).wrapping_add(1442695040888963407);
        self.state
    }

    fn next_in(&mut self, bound: usize) -> usize {
        (self.next_u64() as usize) % bound
    }

    fn next_f64(&mut self) -> f64 {
        (self.next_u64() >> 11) as f64 / ((1u64 << 53) as f64)
    }
}

/// Vocabulary of synthetic English-like tokens. Size ~1024 keeps the
/// FTS index dense; tokens are short, ASCII, lowercase, deterministic.
fn perf_vocab() -> Vec<String> {
    let mut out = Vec::with_capacity(1024);
    for i in 0..1024 {
        let a = (b'a' + ((i / 26 / 26) % 26) as u8) as char;
        let b = (b'a' + ((i / 26) % 26) as u8) as char;
        let c = (b'a' + (i % 26) as u8) as char;
        out.push(format!("{a}{b}{c}{i:04}"));
    }
    out
}

/// Sample a token index under a Zipfian-ish distribution with shape s=1.0,
/// using inverse-CDF on a precomputed cumulative weight table. Returns an
/// index into the vocabulary in [0, vocab_size).
fn zipf_index(rng: &mut SeededRng, cumulative: &[f64]) -> usize {
    let r = rng.next_f64() * cumulative[cumulative.len() - 1];
    match cumulative.binary_search_by(|w| w.partial_cmp(&r).unwrap_or(std::cmp::Ordering::Equal)) {
        Ok(idx) => idx,
        Err(idx) => idx.min(cumulative.len() - 1),
    }
}

fn zipf_cumulative(vocab_size: usize) -> Vec<f64> {
    let mut cumulative = Vec::with_capacity(vocab_size);
    let mut acc = 0.0_f64;
    for k in 1..=vocab_size {
        acc += 1.0_f64 / k as f64;
        cumulative.push(acc);
    }
    cumulative
}

/// Generate one synthetic chunk body of approximately `target_bytes`
/// (~500 B per ADR) using the Zipfian sampler. Tokens are space-joined.
fn synth_chunk_body(rng: &mut SeededRng, vocab: &[String], cumulative: &[f64]) -> String {
    // Tokens average ~7 chars + 1 separator, target ~500 B => ~63 tokens.
    let mut body = String::with_capacity(512);
    let token_count = 55 + rng.next_in(20);
    for i in 0..token_count {
        if i > 0 {
            body.push(' ');
        }
        body.push_str(&vocab[zipf_index(rng, cumulative)]);
    }
    body
}

/// Choose a held-out query token from the 50th–90th percentile
/// term-frequency band per ADR-0.6.0-text-query-latency-gates. Vocab is
/// indexed by descending frequency rank (rank 0 most frequent), so the
/// band maps to indices [0.10*vocab, 0.50*vocab) (frequency rank space).
/// The Zipfian sampler yields rank 0 most often, so this band carves out
/// the body of the distribution.
fn ac012_query_token_band(vocab: &[String]) -> Vec<String> {
    let lo = vocab.len() / 10;
    let hi = vocab.len() / 2;
    vocab[lo..hi].to_vec()
}

/// AC-012 deterministic seeder: 1 chunk body per write batch (4096 nodes
/// per `engine.write()`). Returns elapsed seed time for diagnostics.
fn seed_ac012_corpus(engine: &Engine, n: usize) -> Duration {
    const BATCH: usize = 4096;
    let vocab = perf_vocab();
    let cumulative = zipf_cumulative(vocab.len());
    let mut rng = SeededRng::new(0x0AC0_12C0_12C0);
    let started = Instant::now();
    let mut written = 0usize;
    while written < n {
        let take = BATCH.min(n - written);
        let mut batch = Vec::with_capacity(take);
        for _ in 0..take {
            batch.push(PreparedWrite::Node {
                kind: "doc".to_string(),
                body: synth_chunk_body(&mut rng, &vocab, &cumulative),
                source_id: None,
                logical_id: None,
            });
        }
        engine.write(&batch).expect("ac-012 seed write");
        written += take;
    }
    // No vector kind configured -> projection runtime still drains FTS index.
    engine.drain(600_000).expect("ac-012 drain");
    started.elapsed()
}

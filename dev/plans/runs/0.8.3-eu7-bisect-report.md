## 2. Eliminations (each grounded in code/artifacts, not assumption)

**Case B — CLS / embedding pooling — RULED OUT (definitive).**
- `git diff v0.7.2 v0.8.0 -- src/rust/crates/fathomdb-embedder/src/` is **empty**. The embedder source —
  tokenization, 512-truncation, pooling, forward pass — is **byte-identical** across the window, so the f32
  vectors feeding the quantizer are identical. The 0.937 and 0.896 runs both record
  `embedder=fathomdb-bge-small-en-v1.5`, dim 384.
- CLS pooling did not exist as production behavior at either anchor. Commit `c7afbfde`
  (*feat(ir-c): Phase 0 CLS-pooling option*, **2026-06-11** — AFTER the 0.896 measurement) adds
  `Pooling{Mean,Cls}` with **`default Mean — production unchanged`**; `CandleBgeEmbedder::new()` (which eu7
  uses) still returns `Pooling::Mean`. CLS is an unused, gated A/B option. It cannot have moved a number
  measured three days earlier.

**Case C — corpus (harder/larger/quality) — RULED OUT (per GA-1, re-verified).**
- `GA-1-corpus-ab-20260608T012503Z.md`: the 7667-doc corpus is gitignored, never in git history, and
  **byte-identical** between the 0.937 run (2026-05-31) and the 0.8.0 run (2026-06-07/08) — all 8 raw files
  mtime 2026-05-27, same directory (`readlink -f`), same sha256. The "out-of-band expansion" narrative was
  a conflation of (a) a stale `manifest.json` qmsum checksum and (b) a `qaconv/qasper` manifest entry that
  produced **no loaded files**. There is no OLD≠NEW corpus.

**Case A internals — every deterministic quant/KNN primitive is byte-identical v0.7.2→v0.8.0.**
- **Bit-KNN SQL:** the big retrieval-rewrite commit `d28d2046` (*Slice 10 — G9 RRF + rerank seam, G10
  filtered-KNN*) self-documents that the **unfiltered** path (`filter=None`, which eu7 uses) is
  *byte-identical to 0.7.2*: `build_vector_phase1_sql(None)` reproduces the same
  `... ORDER BY distance LIMIT {top_k=TOP_K_BIT_CANDIDATES} ... LIMIT {SEARCH_RERANK_LIMIT}`. The KNN
  candidate construction, K=192, and f32 rerank limit are unchanged for the eu7 path.
- **Stored quantized bits:** the `migrate_vector_partition_pack1_to_pack2` migration in the same commit
  copies `embedding`/`embedding_bin` **verbatim** and only re-tags the vec0 BIT subtype + adds a `status`
  column — the code comment is explicit: *"preserves the existing (possibly mean-centered) bits verbatim —
  no re-quantize, so centering survives the upgrade."* Hamming distances are unaffected.
- **`rerank_fused` at d28d2046 is an identity stub** (returns fused order unchanged); recency is behind an
  off-by-default flag. Neither touches the vector-stage seam output.

## 3. What is left — and the honest residual

After the eliminations, the eu7 vector path is — by every primitive I could inspect offline — **identical**
at both anchors, yet recall moved 0.937→0.896 and σ moved 0.0116→0.0157 (a *real per-query distribution
change*, not a relabeling). Two non-exclusive explanations remain, both **case A**, both **not CLS**:

1. **(Most likely) Measurement-SUT definition change.** The 0.937 anchor measured `engine.search()`'s final
   output (which at 2026-05-31, pre-`d28d2046`, was the vector path). The 0.896 number measures the **new
   `vector_stage_only` seam** — `vector_results` *as it feeds `fuse_rrf`*, i.e. an **intermediate** list with
   the seam's own limit/slack/ordering, not necessarily the bit-identical list 0.7.1 `search()` returned to
   the caller. A small, legitimate SUT-definition difference (what exactly "the vector top-10" is) plausibly
   accounts for a ~4pp move and a σ shift, with **no fidelity loss**. This mirrors GA-1's finding that the
   *larger* 0.937→0.871 move was a SUT artifact (fused vs vector); the residual 0.937→0.896 is the
   *finer-grained* SUT-method difference (old `search()` vs seam).
2. **(Less likely) A vector-branch ranking change in the pre-`v0.7.2` sub-window** (2026-05-31 anchor →
   2026-06-01 v0.7.2 tag, which folded in PR-9 embedder-concurrency hardening). PR-9 is a serialization
   guard (should not change vector *values*), but this sub-window was not bisected by build (blocked).

**I could not run the decisive test** — a controlled engine `0.7.x`-vs-`0.8.0` **vector-stage A/B on one
embedded eu7-corpus snapshot** — because **no pre-embedded eu7 IR-corpus DB exists** (the `/tmp/*.sqlite`
DBs are LME / d0b / r2 corpora, not the 7667-doc IR raw corpus; per
`[[embed-completeness-and-gpu-readiness]]` no full pre-embedded corpus DB exists), and a full CPU re-embed
is forbidden / out of $0 scope. So the case-A *sub-mechanism* (SUT-definition vs a subtle ranking change) is
**not fully resolved offline**; the case-A-vs-B verdict, however, **is** resolved.

## 4. Recommended Slice-20 fix direction (the deliverable's point)

1. **Decouple CLS-pooling from eu7 fidelity.** The Slice-20 CLS-fix is an **embedder-RELEVANCE** lever (eu8 /
   the Mem0 gap), *independent* of the eu7 quantization-fidelity number. Do **not** frame CLS as the
   fidelity-recovery for 0.937→0.896 — it is not, and chasing it there wastes the re-embed.
2. **Reset the baseline expectation.** The *true* current vector-stage fidelity is **0.896 (CI 0.864–0.925)**,
   not 0.937. `0.937` is a pre-B-1-seam SUT number and is **not directly comparable**; stop treating it as a
   recoverable target. Slice 20's eu7 re-clear should be judged against the **0.90 floor**, measured **fresh**
   on the `vector_stage_only` seam after the (CLS-corrected, chosen-embedder) re-embed — under the existing
   GA-3 CI-based gate (`recall_ci_hi >= 0.90`; today 0.925 ≥ 0.90 PASSES).
3. **If the fresh eu7 < 0.90 after re-embed → the fork is the QUANT path, not pooling.** In expected-value
   order for anisotropic BGE vectors:
   - **rotation / whitening before sign-bit quantization** (highest EV — directly attacks the anisotropy that
     sign-bit quant is sensitive to; the c7afbfde comment already flags geometry-sensitivity);
   - **raise ANN fan-out `K` > 192** (cheap; more f32-rerank candidates recover sign-quant misses);
   - **a 2-bit quantization option** (most fidelity, larger index).
4. **Settle the residual cheaply when an embedder is free.** When the sibling releases the GPU (GPU util < 5%),
   a single GPU-embedded eu7-corpus snapshot makes the engine `0.7.x`-vs-`0.8.0` vector-stage A/B a
   minutes-long, $0 confirmation — schedule it as a Slice-20 sub-task, not a blocker.

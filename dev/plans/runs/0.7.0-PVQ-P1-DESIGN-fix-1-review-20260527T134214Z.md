The actual migration shape needs to be:

1. Schema-version gate via `migrate_with_event_sink` + `PRAGMA user_version` (unchanged from fix-1).
2. **Stage existing f32 rows synchronously inside the migration transaction** (not via `rebuild_vec0`):
   - `CREATE TABLE _fathomdb_vector_migration_v0_7_0 (rowid INTEGER PRIMARY KEY, embedding BLOB NOT NULL, kind TEXT NOT NULL)` — a regular SQL staging table.
   - `INSERT INTO _fathomdb_vector_migration_v0_7_0 (rowid, embedding, kind) SELECT v.rowid, v.embedding, r.kind FROM vector_default v JOIN _fathomdb_vector_rows r ON r.rowid = v.rowid`.
3. `DROP TABLE vector_default`.
4. `CREATE VIRTUAL TABLE vector_default USING vec0(embedding float[<dim>], embedding_bin bit[<dim>], source_type TEXT partition key, kind TEXT, created_at INTEGER)`.
5. Repopulate from staging: iterate Rust-side over staging rows (or use a SQL INSERT … SELECT with a Rust-registered scalar function for kind→source_type, but Rust loop is simpler and 1 M rows is fast). For each row: `INSERT INTO vector_default(rowid, embedding, embedding_bin, source_type, kind, created_at) VALUES(?1, ?2, vec_quantize_binary(?2), ?3, ?4, ?5)` where `source_type` is the Rust-computed coercion of `kind` and `created_at` is the migration timestamp.
6. `DROP TABLE _fathomdb_vector_migration_v0_7_0`.

This is **synchronous**, **inside the migration transaction**, **does not re-run the embedder**, and **does not depend on `RebuildKind::Vec0`**. Cost ~64-90 s for 1 M rows (two scans of the existing 3 GB f32 data, no embed work). The handoff §1.1 originally specified exactly this shape (`one-pass INSERT ... SELECT vec_quantize_binary(embedding)`); fix-1 incorrectly pivoted to the async rebuild driver.

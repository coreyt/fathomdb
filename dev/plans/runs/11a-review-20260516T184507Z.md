## Verdict: **BLOCK**

### 1. `must-fix` — Panic handling does not satisfy AC-067

Refs:
- `src/rust/crates/fathomdb-py/src/lib.rs:23`
- `src/rust/crates/fathomdb-py/src/lib.rs:241`
- `src/rust/crates/fathomdb-py/src/lib.rs:332`
- `src/python/tests/test_ffi_safety.py:19`

The binding deliberately translates unwinds to `EngineError`, not
`pyo3::panic::PanicException` or a subclass of it: the module docs say
so, `call_engine()` returns `EngineError::new_err(...)` on unwind,
`open()` does the same, and the Python test only asserts
`BaseException`. The accepted contract is a `PanicException`-class
exception without process abort.

### 2. `must-fix` — `force_panic_for_test()` not effectively release-gated

Refs:
- `src/rust/crates/fathomdb-py/Cargo.toml:14`
- `src/rust/crates/fathomdb-py/src/lib.rs:531`

The crate enables `test-hooks` in default features, so a normal build
exports the test hook unless the build explicitly disables defaults.
The prompt requires the hook to exist only under
`#[cfg(any(test, feature = "test-hooks"))]` and not ship in release
wheels.

### 3. `must-fix` — Error mapping collapses distinct engine variants

Refs:
- `src/rust/crates/fathomdb-py/src/lib.rs:115`

`EmbedderNotConfigured` maps to `EmbedderError`, and
`KindNotVectorIndexed` maps to `VectorError`. That violates the
binding rule of one concrete class per variant and fails AC-060a's
typed-dispatch contract.

`test_surface.py` does not cite AC-057a anywhere. The surface test
exists, but the acceptance binding is implicit rather than explicit.

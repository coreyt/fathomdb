# EU-7 findings ŌĆö two engine defects block real-corpus MC validation

_Surfaced 2026-05-29 while building the EU-7 real-corpus AC harness
(`src/rust/crates/fathomdb-engine/tests/eu7_real_corpus_ac.rs`). Per the EU-7
launch prompt's honesty mandate ("surface any RED to HITL without re-wording the
ADR to fit; EU-7 is the measurement, not the lock-flip") these are reported as a
RED, not worked around. Confirmed by direct code tracing plus an independent
research subagent._

All line numbers are against `src/rust/crates/fathomdb-engine/src/lib.rs` at
commit `209250a` unless noted.

## Finding A ŌĆö projection pool wedges under the real embedder

**Symptom.** Seeding the real `CandleBgeEmbedder` (feature `default-embedder`) via
`engine.write` + `engine.drain` reliably completes ~512 docs, then `drain` returns
`Err(EngineError::Scheduler)` (a timeout); every thread parks in `futex_wait`, CPU
falls to ~0. The identical `engine.write`+`drain` path ingests the full 7,667-doc
corpus fine with the trivial `VaryingEmbedder` (Pack-4 corpus tests). A smoke run at
N=300 (~1.2 batches) completed. `RAYON_NUM_THREADS=1` did not help.

**Cause (cause + effect).**
1. *Trigger* ŌĆö `PROJECTION_WORKERS = 2` (`:66`); both workers call
   `embedder.embed()` on one shared `Arc<dyn Embedder>` with no serialization
   (`run_projection_job`, `:3139-3140`). candle-transformers `BertModel::forward`
   on a shared instance wedges under sustained concurrent invocation. The
   `Embedder` trait is `Send + Sync` (`fathomdb-embedder-api/src/lib.rs:23`) so the
   engine assumes impls are concurrency-safe; `CandleBgeEmbedder` is `Sync` only by
   field derivation, never stress-tested concurrently (eu5b + corpus tests use a
   trivial or synchronous embedder).
2. *Why it becomes a permanent stall* ŌĆö `run_projection_jobs` is called with no
   `catch_unwind`/drop-guard (`:3084`), and `active_jobs` is decremented only AFTER
   it returns (`:3086-3095`). A worker that panics or blocks forever inside
   `embed()` never decrements `active_jobs`, so `wait_for_idle` (gated on
   `active_jobs == 0 && queued_jobs == 0`, `:541`) never returns ŌåÆ
   `drain` ŌåÆ `EngineError::Scheduler` (`:1806-1809`). The reader pool already guards
   against exactly this with a `LiveGuard` drop-guard (`:435-442`); the projection
   worker has no equivalent.
   `RAYON_NUM_THREADS=1` only disables candle's gemm matmul parallelism, not the
   concurrent-`forward` exposure or the accounting wedge ŌĆö hence no effect.

**Also unimplemented (related):** ADR-0.6.0-embedder-protocol Invariant 5 mandates a
30s per-`embed()` timeout; the projection path handles `RuntimeEmbedderError::Timeout`
(`:3142`) but nothing ever produces one ŌĆö there is no watchdog. A hung embed blocks a
worker indefinitely.

## Finding C ŌĆö embedder does not truncate to 512 tokens (the actual stall root cause)

`CandleBgeEmbedder::embed` (`fathomdb-embedder/src/candle_bge.rs`) tokenizes
with `tokenizer.encode(input, true)` but never enforces the 512-token limit
that BGE-small's position embeddings require (`config.json`
`max_position_embeddings = 512`). Any document tokenizing to >512 tokens makes
`BertModel::forward` fail with `Failed { "forward: index-select invalid index
512 with dim size 512" }`. This violates the locked spec
(`dev/design/embedder-decision.md` ┬¦2.1: "max 512 tokens, truncation True").

**This is what actually produced the EU-7 seeding "stall"** (not the Finding A
hypothesis). Real corpus docs routinely exceed 512 tokens (e.g. cnn_dailymail
articles, which begin around corpus index ~667). In the projection path each
such doc's `embed()` errors, is retried with backoff delays, and finally
recorded as a failure; a whole batch of long docs (the third 256-doc batch,
covering the start of cnn_dailymail) exceeds the per-batch `drain` timeout,
which surfaces as `EngineError::Scheduler`. A standalone embed probe over the
first 800 corpus docs reproduced the error at the first >512-token doc, with
no engine involved ŌĆö isolating the cause to the embedder.

The Finding A panic-guard remains a correct, independent robustness fix (a
faulting worker must not wedge the scheduler), but the real EU-7 blocker was
Finding C. Fix: configure the tokenizer with `with_truncation(max=512)` at load
so long docs are truncated (and `[SEP]` re-applied) rather than erroring.

## Finding B ŌĆö mean-centering is inert on the production write path

**The locked "mean-centering ON" decision** (`dev/design/embedder-decision.md` ┬¦3;
the campaign headline) **only activates via the synchronous test seam**
`Engine::write_vector_for_test` (`:2104-2222`). The production
`engine.write` ŌåÆ projection path never feeds the streaming mean accumulator and
never pins the corpus mean:

- `run_projection_job` (`:3129`) embeds and returns blobs; it never touches the
  accumulator. Its `bin_blob` is `blob.clone()` ŌĆö uncentered (`:3171`).
- `commit_projection_outcomes` (`:3467`) only *reads* an already-pinned `mean_vec`
  (`:3475-3485`) and applies it (`:3510-3518`); it never feeds the accumulator and
  never pins.
- `MEAN_VEC_PIN_THRESHOLD` (`:2715`) is compared in exactly ONE place ŌĆö inside
  `write_vector_for_test` (`:2156`).

**Consequence.** Documents ingested through the real write path leave
`_fathomdb_embedder_profiles.mean_vec` NULL forever, so their sign-bits are quantized
**uncentered**. Mean-centering ŌĆö the +2.5 pp recall lift at K=192 that justified the
K and the 0.90-floor headroom ŌĆö does not function for normal ingestion. EU-5b's pin
tests used the `write_vector_for_test` seam, which masked the gap.

The design clearly intended otherwise: the doc comment at `:2225` calls this "the
projection-commit pin transaction", and `dev/design/embedder.md` ┬¦0.3 specifies
"compute-once-on-first-ingest ... pinned at the workspace's first non-trivial
commit." The implementation landed the pin only in the test seam.

## Impact on EU-7

EU-7's mandate is to validate AC-013b recall@10 for the **locked config: bge-small +
mean-centering ON + K=192**. Because the production path does not mean-center
(Finding B) and cannot even ingest a large corpus reliably (Finding A), a harness
seeding via `engine.write` would measure the **wrong configuration** (plain / no-MC,
EU-0 ~0.90-0.91 at K=192) rather than the locked MC config (EU-0 0.933). EU-7 is
therefore **blocked** on an engine fix.

`write_vector_for_test` is not a substitute: it only writes `vector_default` +
`_fathomdb_vector_rows`, not `canonical_nodes`, so `engine.search()` (which reads
bodies from `canonical_nodes`, `:2907`) returns nothing for those rows ŌĆö the recall
comparison needs `search()`.

## Resolution (authorized 2026-05-29)

A 0.7.1 engine fix slice ŌĆö **EU-5f: production mean-centering pin + projection
fault-isolation** ŌĆö lands before EU-7 measures:
- Pin the mean inside `commit_projection_outcomes`, serialized by a `commit_gate`
  Mutex for cross-worker correctness, with an open-time recovery pin for the
  crash-before-pin window (Finding B).
- Wrap `run_projection_jobs` in `catch_unwind` so a faulting worker can never wedge
  `wait_for_idle` (Finding A).
- The harness also serializes the embedder behind a Mutex (`SerializedBge` in
  `eu7_real_corpus_ac.rs`) as the in-test mitigation for Finding A's trigger.

Design + TDD in `dev/plans/prompts/` plan `snug-cooking-flute` (approved). EU-7 then
re-runs against the now-correct production MC path and produces the PR-2 anchor.

The 30s embed watchdog (ADR Invariant 5) is noted as a follow-up, out of EU-5f scope.
`AC013B_RECALL_FLOOR` re-pin and the ADR recall wording remain untouched ŌĆö owned by
0.7.2 PR-2.

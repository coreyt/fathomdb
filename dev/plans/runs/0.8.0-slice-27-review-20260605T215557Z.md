## The spec-level question fix-N must resolve (HITL — recorded, escalated)
The Rust `fathomdb` facade is **both** the runtime-SDK-equivalent contract **and** the substrate the CLI
(`fathomdb recover`) is built on — so the engine `Engine` genuinely needs `rebuild_*`/recovery methods for
the CLI. Q5=BIND-RUST (bind the Rust facade to the governed surface) is therefore in tension with that role.
Resolution options (HITL call — the fix-N scope depends on this):
- **(A) Carve out the operator/recovery seam.** Define the Rust *governed runtime surface* as the
  application methods only (`open/write/search/close` + typed data types); the recovery/operator methods

  explicitly **separate, non-governed operator seam** the CLI uses. The test audits the **method** surface
  and asserts no recovery-denylist name appears in the *governed app* surface + asserts release-build has no
  raw-SQL method; `rust.md`/AC-074 document the app-surface-vs-operator-seam boundary. (This is what the
  agent's 17-types-vs-20-operator-seam split *intended* — the test just didn't enforce it at method level.)
- **(B) Hide recovery methods from the default facade.** Feature-gate `rebuild_*`/operator methods behind a
  `recovery`/`cli` feature so the default `fathomdb::Engine` runtime surface is recovery-clean and the CLI
  opts in. Real code change; larger blast radius.
- **(C) Re-scope Q5.** If the intent of binding the Rust facade was the parity/typed-boundary guarantee (not
  a strict method-level recovery-denylist given the facade's CLI role), narrow AC-074's Rust clause
  accordingly. ADR/HITL-level.

because it defines what "the Rust facade upholds the recovery-denylist" actually means. fix-N scope:

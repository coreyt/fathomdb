# 0.7.2 recall root-cause investigation — the EU-7 gap is a MEASUREMENT ARTIFACT

**Date:** 2026-05-30. **Status:** root cause identified; gold-standard re-measure
pending. **Author:** PR-2c diagnostic chain (post PR-2a/PR-2b/PR-2c).

## TL;DR (the headline, stated plainly)

The 0.7.2 recall campaign was triggered by EU-7 measuring real-corpus recall@10 =
**0.828** (< the 0.90 floor), which PR-2a attributed to a *fixable mean-centering
artifact*. **That attribution was wrong.** Rigorous decomposition on the real
engine shows:

- The **mean was never the driver** — forcing the *full-corpus* mean on the real
  engine yields **0.847**, only +1.9pp over the worst mean (0.828).
- **candle and HF embeddings are bit-identical** (cos = 1.00000, 100% sign-bit
  agreement) — embeddings are not the cause.
- **Title vs body representation** contributes ~0 (+0.7pp).
- The offline→real gap is **~6pp of conservative EU-7 harness measurement
  methodology**, not an engine deficiency:
  - **−4.4pp** from excluding the self-retrieving target *after* the top-10
    (64/100 synthetic queries self-retrieve their source doc → a slot is spent on
    it → recall capped at 0.9 for those).
  - **−1.4pp** from body-STRING ground-truth matching against a corpus with 5.6%
    duplicate bodies (synthetic_notes 20%).
  - remaining ~3.9pp: query-set differences (offline numpy-shuffle vs harness
    SplitMix64) + minor sqlite-vec candidate noise — within bootstrap CIs.

**Measured the standard ANN way (exclude the query-source doc BEFORE top-10,
match GT by id/dedup), the engine's real recall@10 is ~0.93 — it CLEARS the 0.90
floor.** PR-2b (mean recompute) and PR-2c (representative pin) are correctly
engineered but address a non-problem for recall.

## The measure under discussion (so the numbers are interpretable)

`recall@10` here is the **ANN / quantization-fidelity** flavor (FAISS / ANN-
Benchmarks sense), NOT the IR-relevance sense:
`recall@10 = |retrieved_top10 ∩ exact_f32_top10| / 10`, where ground truth = the
exact top-10 by full-precision cosine over the SAME model, and retrieved = the
sign-bit binary-quant (K=192 Hamming candidates) + f32 rerank pipeline. It
measures how faithfully the lossy 1-bit index reproduces the brute-force
neighbors — not whether results are "relevant". Percentile bootstrap (1000
resamples, SEED=0xE0) for CIs throughout.

### 3. Forced-full-mean diagnostic (`EU7_FORCE_FULL_RECOMPUTE=1`, real engine)
- recall@10 = **0.847** (CI 0.817–0.875, σ 0.0145).
- `drift_cos_before = 1.0000` — PR-2b had ALREADY converged the pinned mean to the
  true full-corpus mean during seeding; the forced recompute was a no-op.
- **Verdict: the best-possible mean gives 0.847. The mean is not the driver.**

### 4. candle-vs-HF embedding agreement (`tests/candle_dump.rs` + HF compare; 13 inputs incl. 147K-char qmsum bodies)
- cos(candle, HF) = **1.00000** for all 13; L2 = 0.0000; **sign-bit agreement = 100%**.
- **Verdict: embeddings identical; not the cause.**

### 5. Body-only vs title+body offline (`run_bodyonly_offline.py`)
- title+body docs: 0.951; **body-only docs (production repr): 0.944**; delta +0.7pp.
- **Verdict: title-in-doc representation confound is NOT the driver** (refuted a
  leading hypothesis).

### 6. GT-accounting + target-exclusion decomposition (`run_gt_accounting.py` + follow-up)
Same body-only HF vectors, same numpy retrieval, vary only the scoring method:
- (A) INDEX GT, exclude-target-BEFORE top-10 (standard ANN): **0.944**
- (B) + body-STRING GT (dup bodies; 26/100 queries had shrunk GT sets): **0.930** (−1.4pp)
- (C) + exclude-target-AFTER top-10 (≤9 slots; 64/100 queries self-retrieve target): **0.886** (−4.4pp)
- real engine (full harness method): **0.847** (residual ~3.9pp = query-set + sqlite-vec, within noise)

Corpus duplicate-body density (drives the −1.4pp): 5.6% overall (742 docs);
synthetic_notes 20%, cnn/chain 6%, enron 1%.

## Decomposition table (the bottom line)

| Step | recall@10 | Δ | nature |
|---|---|---|---|
| Offline, standard ANN method | 0.944 | — | true binary-quant recall |
| + body-string GT w/ dup bodies | 0.930 | −1.4pp | harness measurement artifact |
| + exclude target AFTER top-10 | 0.886 | −4.4pp | harness measurement artifact (conservative) |
| + query-set diff + sqlite-vec noise | 0.847 | −3.9pp | residual, within CI |

**~6pp of the ~10pp offline→real gap is harness methodology.** Engine real recall,
measured the standard way, ≈ **0.93** → clears the 0.90 floor.

## What was ruled OUT (with evidence)
- Mean strategy — forced full mean = 0.847 (drift_cos_before = 1.0).
- candle vs HF embeddings — bit-identical, 100% sign-bit agreement.
- Title/body representation — +0.7pp.
- Query centering bug — verified correct (`lib.rs` search_inner: phase-1 query is
  `subtract_mean`'d when a mean is pinned; phase-2 rerank un-centered).

## What is the actual root cause
The EU-7 harness (`eu7_real_corpus_ac.rs::measure_recall`) understates recall by ~6pp:
1. **Excludes the self-retrieving target AFTER taking the engine's top-10** — the
   synthetic query is the target doc's own title/lead, so the target self-retrieves
   into the top-10 for 64% of queries; dropping it post-hoc leaves ≤9 comparable
   slots, capping recall at 0.9. (The docstring flags this as "conservative".)
2. **Matches ground truth by body STRING** against a corpus with 5.6% duplicate
   bodies, shrinking GT sets below 10 unique entries.

Standard ANN-recall practice excludes the query-source doc *before* ranking (so the
engine gets 10 clean slots) and matches by id (dup-robust).

## Implications
- **The 0.90 floor likely needs NO change** — it needs the harness measured correctly.
- **PR-2b (recompute)** and **PR-2c (representative pin)** address a non-problem for
  recall. They remain correct code with possible value only as hygiene / genuine
  topic-drift safety — to be reassessed (see below).
- PR-2a's GO verdict was based on an offline number (0.945) that does not transfer
  to the production retrieval+measurement path.

## Follow-up TODO: log the silent ground-truth-embed phase in the EU-7 harness

**Symptom:** during a full run, `eu7_real_corpus_ac.rs` goes silent for ~25-45 min
*after* the last `EU7_SEED_PROGRESS` line and *before* `seed_done`. Twice this
looked like a hang and once prompted a premature kill. It is NOT a hang: the
harness re-embeds the entire corpus single-threaded to build the in-Rust f32
ground truth (`for body in &bodies[seeded..actual_n] { doc_vecs.push(embedder.embed(body)) }`),
which emits no output. At ~0.3-0.4 s/doc single-threaded that is ~45 min for 7667
docs, with zero log lines.

**Fix (minimal but useful):** a single line literally *between* the loop and the
`seed_done` print only fires once (an end-marker) and does NOT break the silence
*during* the loop -- so "between" alone is not the useful placement. Add instead:
1. a `GT_EMBED_START n=<n>` line before the loop, and
2. a **periodic progress line inside the loop**, mirroring `EU7_SEED_PROGRESS`
   (every ~30 s or every K docs: `EU7_GT_EMBED_PROGRESS embedded=i/n rate_docs_per_s=...`).
   A throttled `Instant`-based report like the seed loop already uses is enough.
This makes the longest unlogged stretch observable and stops it reading as a wedge.
Cheap, test-only, no behavior change. (The single end-marker between the loop and
`seed_done` is a fine additional cheap minimum, but the in-loop periodic line is
the one that matters.)

# 0.8.18 Slice 0 — ONNX-GPU-EP (CUDA) cross-backend equivalence Δ (D3 / L3, OOB)

Status: **MEASURED + RECORDED** (durable, regardless of outcome). OUT-OF-BAND,
NON-BLOCKING for the 0.8.18 capstone (design D3 / U4; `dev/design/0.8.18-slice-0-vector-equivalence-publish-design.md`
§U4, R-D3-1/R-D3-2). Steward-commissioned measurement agent, MAIN tree only,
pure `cargo` (no `maturin`/`pip -e`, no `.venv` mutation), `cuda:1`.

This closes the one cross-backend equivalence leg that was open to date:

| Leg | measured at | flips |
| --- | --- | --- |
| candle-CPU ↔ candle-CUDA | 0.8.7 | 0 / 6144 |
| candle-CPU ↔ ONNX-CPU | 0.8.16 Slice 15 | 0 / 17280 |
| **candle-CPU ↔ ONNX-GPU-EP (CUDA)** | **THIS RUN** | **2 / 17280 (raw sign)** |

**Headline:** the ONNX CUDA execution provider is the FIRST cross-backend leg
that is NOT byte-identical to the candle-CPU reference. It is still
numerically near-identical (cosine ≡ 0.99999996, max-abs-Δ ≤ ~1e-4) but it
flips **2 of 17280** raw sign bits (1 of 17280 under mean-centering), where
every CPU leg to date flipped **0**. Deterministic across 3 runs.

## R-D3-1 — Procurement provenance (pinned, auditable supply chain)

The shipped library never bundles any of this. The ONNX GPU EP is an
**OFFLINE-BUILD / EVAL-ONLY** artifact (footprint invariant): a caller-supplied
`libonnxruntime.so` dlopen'd at runtime via `ORT_DYLIB_PATH` under the
non-default `onnx-embedder` feature. All artifacts live in the session
scratchpad, NOT in the repo.

### ABI / version rationale

`ort = 2.0.0-rc.10` targets `ORT_API_VERSION = 22` (ONNX Runtime 1.x C API,
`ort-sys-2.0.0-rc.10/src/lib.rs`). The 0.8.16 CPU baseline used ORT **1.26.0**
(the CPU wheel already in the repo `.venv`), which provides API ≥ 22 and is
forward-compatible. To hold the runtime version **identical** to the CPU
baseline (removing ORT-version as a confound; the only variable is CPU-EP vs
CUDA-EP), this leg uses the **GPU build of the same ORT 1.26.0**.

## R-D3-2 — Effective-provider measurement (guard against silent CPU fallback)

### Effective provider = CUDAExecutionProvider on cuda:1 (VERIFIED)

Confirmed three independent ways:

1. **No loud fallback.** `OrtBgeEmbedder` emits a LOUD stderr warning on any CPU
   downgrade (R-ONNX-2). The measured run emitted **none**.
2. **GPU memory engaged.** `nvidia-smi -i 1` sampled during the run rose from
   ~15 MiB idle to a **550 MiB peak** — the CUDA EP allocated device memory.
3. **Distinct numerics.** The result differs from the CPU-EP baseline
   (2 flips / L2 ~2.6e-4 vs 0 flips / L2 ~6.7e-7); a CPU fallback would have
   reproduced the byte-identical CPU numbers (see the negative control below).

### Negative control — the silent-CPU-fallback trap this leg exists to catch

The FIRST attempt fell back to CPU and was **correctly recorded as CPU, not
mislabeled GPU**. Under `ort = 2.0.0-rc.10` (built `default-features = false`),
constructing `OrtBgeEmbedder` with `FATHOMDB_EMBED_DEVICE=cuda:1` produced:

```
fathomdb-embedder(onnx): requested ONNX execution provider Cuda(1) was reported
available but FAILED during ONNX Runtime session creation (... LoggingManager::
DefaultLogger() Attempt to use DefaultLogger but none has been registered.);
falling back to CPU
```

That run's numbers were byte-identical to the 0.8.16 CPU baseline (cosine ≡ 1.0,
0 flips, L2 6.7e-7) — i.e. it was a CPU run, effective provider = **CPU**.

**Root cause (diagnosed, isolated):** the failure is specific to the `ort`
**Rust crate rc.10** binding, NOT the host/CUDA/cuDNN/ABI:

- The **Python** binding of the *same* ORT 1.26.0 GPU build, *same*
  `LD_LIBRARY_PATH`, *same* `model.onnx`, on device_id=1, creates a session with
  `CUDAExecutionProvider` effective and runs inference (out `(1,4,384)`). Host is
  fully functional.
- A standalone `ort = 2.0.0-rc.10` probe reproduced it: session creation with the
  CUDA EP **fails without `ort::init()`** ("DefaultLogger not registered") but
  **succeeds with a one-time `ort::init().commit()`** before the build (ran on
  cuda:1, out `(1,4,384)`). The `ort` crate built `default-features = false`
  does not register an ORT logging environment, and the CUDA EP's
  session-creation path logs internally; the CPU EP path does not, so CPU
  "works" while CUDA trips the unregistered-logger assert and the loud fallback
  downgrades to CPU.

The measured GPU leg below was obtained by initializing the ORT environment
once (`ort::init().with_name(...).commit()`) before session construction, in the
harness process. **The shipped `OrtBgeEmbedder` does NOT call `ort::init()`**, so
as shipped under rc.10 the ONNX **CUDA** EP is unreachable and always loud-falls
back to CPU. See TC proposal below.

## Measured Δ — candle-CPU (reference, Cls) ↔ ONNX-cuda:1 (Cls)

45-probe fixture `tests/fixtures/candle_onnx_equivalence_probes.txt`; same pinned
`BAAI/bge-small-en-v1.5` weights (HF `5c38ec7…`, dim 384). candle on CPU (no
`embed-cuda` feature ⇒ always CPU), ONNX on the CUDA EP, cuda:1. Deterministic
across **3 runs** (identical flip counts + cosine).

| Metric | mean | min | max |
| --- | --- | --- | --- |
| cosine similarity | 0.999999958 | 0.999999855 | 1.000000000 |
| L2 distance (un-centered) | 2.603e-4 | 5.724e-7 | 5.381e-4 |
| max-abs element Δ | 4.420e-5 | 9.313e-8 | 9.862e-5 |

- **Raw 1-bit sign flips: 2 of 17280** (0.0116%); worst probe (idx 5) = 1 flip.
- **Mean-centered 1-bit sign flips: 1 of 17280** — using a **PROXY mean** (mean
  of the 45 reference vectors), clearly labeled: the real engine
  `_fathomdb_embedder_profiles.mean_vec` pinned-mean fixture is the **U3
  harness's** job (R-CAL-3, not yet landed). Given max-abs-Δ ≤ ~1e-4, a
  centered flip requires a centered component within ~1e-4 of the threshold, so
  the centered count tracks the raw count for any mean; 1 vs 2 is consistent.

### Reading

The ONNX **CUDA** EP introduces ~1e-4-magnitude per-component differences vs the
CPU reference (≈1000× the ~1e-7 CPU-EP round-off), consistent with GPU FP
reduction-order / fused-op / library differences. This is enough to flip a
small, bounded, DETERMINISTIC number of sign bits (2/17280). It is far from a
broken-export signature (which would show an anomalously high flip rate) — the
export/pooling is correct; this is pure backend FP divergence.

**Bearing on the D4 floor (#5):** the ONNX-GPU-EP leg would trip a strict
"0 raw sign flips" cross-backend floor. This does NOT threaten the shipped #5
guard, because #5 is **additive-only** (R-VEQ-5) and the ONNX backend already
carries a DISTINCT engine identity (`fathomdb-bge-small-en-v1.5-onnx`), so the
engine refuses to read candle-written vectors with ONNX regardless of flip
count. But any FUTURE cross-vendor portability-relaxation ADR must budget for a
**non-zero GPU-EP flip rate** (≥ 2/17280 on this asset) — GPU-EP is not
byte-portable with the CPU reference.

## Reproduction

Artifacts (scratchpad, EVAL-only — re-fetch via the two `pip download`s above):

```sh
SP=<scratchpad>/ort-gpu
export LD_LIBRARY_PATH=$SP/unpack/onnxruntime/capi:$SP/cudnn/unpack/nvidia/cudnn/lib:/usr/local/cuda-12.6/lib64
export ORT_DYLIB_PATH=$SP/unpack/onnxruntime/capi/libonnxruntime.so.1.26.0
export FATHOMDB_ONNX_MODEL_PATH=~/.cache/fathomdb/embedders/onnx/bge-small-en-v1.5/model.onnx
export FATHOMDB_ONNX_TOKENIZER_PATH=~/.cache/huggingface/hub/models--BAAI--bge-small-en-v1.5/snapshots/5c38ec7c405ec4b44b94cc5a9bb96e735b38267a/tokenizer.json
export FATHOMDB_EMBED_DEVICE=cuda:1
```

Harness: the 0.8.16 `tests/candle_onnx_equivalence.rs`, repointed to the CUDA EP
(candle built CPU-first, then `FATHOMDB_EMBED_DEVICE=cuda:1` for the ONNX leg),
with a one-time `ort::init().commit()` before session construction (see root
cause). Run OOB via a throwaway test (NOT committed; tree left clean). When the
**U3 device-parameterized harness lands** (R-CAL-1/-3, per-leg process
isolation + real pinned-mean fixture + `effective_provider()` accessor), re-run
this leg through it for the real-mean centered flip count; the raw-sign result
(2/17280, cosine 0.99999996, deterministic) is expected to reproduce.

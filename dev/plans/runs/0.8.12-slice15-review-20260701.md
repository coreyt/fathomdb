## Verdict: CONCERN (1×P1 + 3×P2) → fix-1 required (P1 not overridable)

### 1. [P1] `invalidate` does not hide the edge from active retrieval
Refs: `src/rust/crates/fathomdb-engine/src/lib.rs:3420-3424`.
`invalidate` only sets `t_invalid`; FTS/vector edge retrieval joins on `superseded_at IS NULL` and does
not exclude past-`t_invalid` rows (the static projection shadow rows survive). So the stale contradiction
remains retrievable after consolidation — the verdict does not achieve its purpose. (Note: graph
*traversal* already filters `t_invalid > now` at `lib.rs:3969`; FTS/vector do not.)

### 2. [P2] Python public API not exposed
Refs: `fathomdb-py/src/lib.rs:1003-1008`. Method added only to the native PyO3 class; the public path
`src/python/fathomdb/engine.py` (`class Engine`) has no `consolidate_with_provider` forwarder, and
`_fathomdb.pyi` / `__init__.py` omit `ConsolidateReceipt`/`ConsolidatorError`/`ConsolidateAxis`. Python
SDK users cannot call it. (X1 parity gap.)

### 3. [P2] supersede does not prune vector/projection state
Refs: `lib.rs:3435-3439`. Unlike the normal write-supersession path, `supersede`/`merge` updates only
`canonical_edges`; it leaves stale rows in `vector_default`/`_fathomdb_vector_rows`/
`_fathomdb_projection_terminal`/`search_index_edges`, so stale vectors can occupy ANN top-K.

### 4. [P2] verdict set not required to cover the cluster exactly
Refs: `lib.rs:3400-3408`. The loop accepts any subset and repeated `edge_ref`s as success; a malformed
provider can omit the stale edge (or repeat one) and get `Ok` with misleading counts.

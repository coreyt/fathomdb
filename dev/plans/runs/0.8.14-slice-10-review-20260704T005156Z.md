## Verdict: CONCERN

### 1. [medium] BM25F scoring is not tokenization-faithful to the FTS5 index it ranks
Refs: `engine/src/lib.rs:10278,10303,10347,10425`, `schema/src/lib.rs:430`.
`search_index_v2` recalls with `porter unicode61 remove_diacritics 2`, but the in-engine scorer
recomputes tf/df/field-lengths with a simpler lowercase-alnum splitter. Porter/diacritic matches can be
recalled by `MATCH` yet scored as if the term were absent or rarer → ranking is wrong for
stemmed/diacritic variants (recall is intact; the BM25F formula and live `weights`/`b`/`k1` are sound).
The scorer must measure terms under the SAME tokenization semantics as the index it depends on.

### 2. [low] `search_index_v2.status` does not match shipped G10 search-filter status semantics
Refs: `engine/src/lib.rs:1492,4951,10254`, `schema/src/lib.rs:436`.
Step 17 + the sync write path safely derive `status` from `$.status` under `json_valid` (no
migration/write abort on malformed JSON). But shipped G10 search filtering reads vec0 `status`, still
hardwired to the empty-string sentinel; the code comment claiming this is "the same status the G10
SearchFilter reads" is inaccurate in the current tree.

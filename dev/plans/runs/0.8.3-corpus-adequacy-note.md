# 0.8.3 / 0.8.4 corpus-adequacy — measured determination + acquisitions

> **Steward note, 2026-06-21.** Triggered by the Slice-5 codex review (the
> `multi_session`/`knowledge_update` mislabel [P1#1] + the "ku is already 48%
> synthetic" signal) and the question "does FathomDB have a corpus problem?"
> This records the **measured** answer, the datasets brought in to fix it, and
> the HITL decisions that follow. Status: **decision-package (not yet signed).**

---

## 0. TL;DR

- **0.8.3 had a real, latent corpus problem on the LongMemEval-only corpus.** At
  the built N≈150/class, **`multi_session`, `temporal`, and `knowledge_update`
  are all underpowered** — the frozen gate (`decide_083`) would return
  `NOT_REACHED / underpowered:<class>` regardless of how good FathomDB is. The
  LongMemEval real-question pool is **exhausted** (~500 questions total; the
  re-pin already consumed essentially all of them and still went 48% synthetic on
  `knowledge_update`).
- **A $0 paired-power proxy shows pairing helps but is not sufficient alone**, and
- **LOCOMO (acquired) closes the gap on 2 of the 3 failing classes.** With
  LongMemEval **+ LOCOMO**: `factoid` → **robust**, `multi_session` + `temporal`
  → **powered-if-paired** (available N ≥ required N at the measured correlation).
  **`knowledge_update` remains underpowered** — LOCOMO has no analog; it is the
  residual binding constraint and a HITL decision.
- **0.8.4 had a present, concrete gap**: the GraphRAG-sensemaking corpus the
  roadmap names was not in the repo. **Acquired**: Microsoft BenchmarkQED
  **AP-News (1,397 articles)** + the AutoQ question sets.
- **Licensing flag (load-bearing):** both new datasets are **non-commercial**;
  AP-News is additionally **non-redistributable** (MS Research License). Both are
  placed **gitignored** under `data/corpus-data/` and reproduced via committed
  `acquire_*.py` scripts — **EVAL-ONLY, never committed, never shipped.**

## 1. The measured power result (the corpus problem, quantified)

Tool: `src/python/eval/paired_power_proxy.py` (EVAL-ONLY, $0, deterministic).
Report: `dev/plans/runs/0.8.3-paired-power-proxy.json`. ε = 0.05, k = 10.

It corrects the Slice-5 manifest's `implied_n_for_mde_le_eps`, which was the
**UNPAIRED** bound (`var_diff = 2·p(1−p)`, i.e. ρ=0). The 0.8.3 gate is a
**paired** FathomDB−Mem0 test; pairing shrinks the required N by `(1−ρ)`. The
proxy measures the per-query outcome correlation ρ̂ between two $0 lexical arms
(BM25 vs TF-IDF-cosine) as a stand-in for the real arms' pairing.

**Measured ρ̂ (LongMemEval):** factoid 0.72 · knowledge_update 0.42 ·
multi_session 0.77 · temporal 0.63 — **substantial positive correlation on every
class**, confirming pairing materially lowers the required N (the unpaired bound
overstates it by 1.7–4×).

**Per-class adequacy** (required N at ρ=0 = Slice-5 number; at ρ̂ = realistic):

| class | built N (LME) | +LOCOMO real | **avail.** | req. N (ρ=0) | req. N (ρ̂) | LME-only | **LME+LOCOMO** |
|---|---|---|---|---|---|---|---|
| factoid | 156 | +841 | **997** | 524 | ~147 | powered-if-paired | **robust** |
| knowledge_update | 150 | +0 | **150** | 601 | ~349 | UNDERPOWERED | **UNDERPOWERED** |
| multi_session | 150 | +281 | **431** | 768 | ~308 | UNDERPOWERED | **powered-if-paired** |
| temporal | 150 | +321 | **471** | 752 | ~277 | UNDERPOWERED | **powered-if-paired** |

Verdict legend (conservative, three-way): **robust** = powered even at ρ=0
(unpaired upper bound); **powered-if-paired** = powered at ρ̂ but not at ρ=0;
**UNDERPOWERED** = not powered even at ρ̂.

**Reading it:** on LongMemEval alone the gate is mechanically unreachable on three
classes. LOCOMO adds *real* questions to exactly the two hardest
(`multi_session` p≈0.51, `temporal` p≈0.57) and lifts both to powered-if-paired.
`knowledge_update` is the lone holdout.

### Honest caveats (do not overclaim)

1. **ρ̂ is between two *lexical* arms;** the real FathomDB-vs-Mem0 ρ may differ.
   "Powered-if-paired" is therefore *contingent on Slice 10 confirming ρ≳ the
   measured band.* The proxy **de-risks** the priced run (we now know the corpus
   *can* support a claim if pairing holds); it does **not** replace the
   authoritative paired power-check in `decide_083` at Slice 10.
2. **LOCOMO `multi_session` (category-1 multi-hop) does not guarantee evidence
   spanning ≥2 distinct sessions.** The same ≥2-session predicate the Slice-5
   review mandates ([P1#1]) must be applied before gate use; the usable
   `multi_session` N may drop below 281 after filtering. **Follow-up:** measure
   the ≥2-session fraction of LOCOMO cat-1 before relying on it.
3. The proxy uses LongMemEval BM25 recall as the parity-level p̄ (continuity with
   the Slice-5 variance proxy).

and reproduced on demand. **Do not relocate either payload to a tracked path, and
do not commit LOCOMO/AP-News-derived gold containing verbatim question/answer
text.**

## 5. HITL decisions required (this is a decision-package)

1. **Integrate LOCOMO into the 0.8.3 gate?** Slice 0 is **SIGNED** and froze the
   LongMemEval `corpus_hash` + source. Adding LOCOMO is a **pre-registration
   amendment**, not a silent swap. Recommended shape: run LOCOMO as a **second
   pinned corpus** (parallel parity run, its own `corpus_hash`), not merged into
   the LME hash — a class reaches parity only if it clears on a *powered* corpus.
   Sign before Slice 10.
2. **`knowledge_update` residual.** Still underpowered with LOCOMO. Options:
   (a) pre-register ku as **report-only** (not gateable); (b) more synthetic ku
   (the Slice-5 [P1#1] labeling fix must land first — risky, already 48%
   synthetic); (c) source a third KU-bearing dataset. Recommend (a) unless a
   clean KU source appears.
3. **Licensing acceptance.** Confirm EVAL-ONLY/NC use of LOCOMO + AP-News is
   acceptable for a project heading to GA, and that the gitignore-only posture is
   the agreed control. (No payload is committed.)
4. **Apply the ≥2-session predicate to LOCOMO cat-1** before counting its
   `multi_session` N toward power (caveat 2 above).

## 6. How (a) the paired-power proxy plugs into the Slice-10 RED

`paired_power_proxy.analyze()` returns the per-class paired `var_diff` and
implied-N. The Slice-10 RED should assert, **as a pre-priced-run gate**, that
every class the run intends to *claim* has `available_N ≥ required_N` at the
proxy ρ̂ (or accept the class as report-only) — so an underpowered class is caught
**before** budget is spent, not after. The authoritative check remains
`decide_083`'s per-class `mde ≤ ε` power guard on the real arms; this proxy is the
cheap-validate that precedes it. Reuse `decision_rule_083.EPS_NEAR_PARITY`
(already imported) so the band cannot drift.

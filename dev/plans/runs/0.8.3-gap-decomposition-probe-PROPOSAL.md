# 0.8.3 — Gap-decomposition probe (retrieval vs answer-formation) · DECISION PACKAGE (HITL-gated)

> **Author:** orchestrator. **Status:** proposal awaiting HITL go (this is a *reader-priced* experiment
> → budget-gated; it is NOT $0 and was NOT run). **Why now:** D0b shows FathomDB −20…−27 pp accuracy vs
> Mem0 with the **accuracy gap ≫ recall gap** — the single most decisive thing to learn next is *where*
> that gap lives, because it determines whether the planned 0.8.3 levers (PRF, D2/15b) can ever close it.

## 1. The question
Decompose the FathomDB−Mem0 accuracy gap into:
- **(R) retrieval** — FathomDB surfaces the needed evidence less often; vs
- **(A) answer-formation** — the evidence IS retrieved, but the reader can't extract the answer from
  FathomDB's **raw whole-body** context within the 32 k budget (whereas Mem0 feeds **distilled memories**).

The D0b signal (accuracy −0.24 with recall only −0.10 on factoid; −0.20 vs −0.10 on multi_session)
already implies a large **(A)** component. This probe sizes it.

## 2. Method (identical-answerer, one new arm)
Reuse `eval/r2_parity_eval.py` + the D0a power-sized gold. Add an **oracle-context arm**: feed the
**gold/required doc(s)** directly to the same reader/prompt/budget (no retrieval). Compare per class:
- `acc_oracle` (oracle context) vs `acc_fathomdb` (D0b retrieval) vs `acc_mem0` (D0b).
- **Interpretation (pinned):**
  - `acc_oracle ≈ acc_mem0` (oracle recovers the gap) ⇒ gap is **(R) retrieval** ⇒ PRF / D2 / retrieval
    architecture are the right levers; 15b/D2 worth building.
  - `acc_oracle` still ≪ `acc_mem0` (oracle does NOT recover it) ⇒ gap is **(A) answer-formation** ⇒ the
    lever is **memory-FORM** (Mem0-style distilled units), NOT retrieval tuning; PRF/D2 will not close it.
  - Report the split as `(R) = acc_oracle − acc_fathomdb`, `(A) = acc_mem0 − acc_oracle` per class.
- Optional 2nd oracle arm: **distilled-oracle** (feed an LLM one-line distillation of the gold doc) to
  test the Mem0-form hypothesis directly — if distilled-oracle > raw-oracle, memory-form is the lever.

## 3. Budget / footprint (why HITL-gated)
- The reader is the **one priced EVAL-ONLY seam** (`gpt-5.4`). Oracle arm = 606 reader calls ≈ **$3–5**
  (no retrieval calls). Distilled-oracle adds a cheap distill pass.
- **Cheap-validate first** (`gemini-2.5-flash-lite`, ~$0.01) on a class-balanced n≈40 to confirm the arm
  wiring + directional split, THEN HITL spend-gate the priced n=606 confirm. Resilient harness
  (checkpoint/resume/backoff) mandatory ([[priced-runs-need-resilience-before-spend]]).
- Determinism: temp 0, seed 0; reuse the D0b corpus_hash + gold repin_hash so it's directly comparable.

## 4. What it unblocks (decision value)
This is the **highest-leverage $few experiment on the board**: it tells us whether to spend Phase-B on
**retrieval levers** (PRF/D2 — in-footprint) or on the **Mem0-style memory-formation layer**
(caller-side BYO-LLM / OFFLINE-BUILD — a *scope expansion* + the pre-registered fork (b)). Without it we
risk building D2 (a recall lever) against a gap that is mostly answer-formation.

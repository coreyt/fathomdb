## Verdict: CONCERN

### 1. [high] `probe_wal_sidecar` reads the entire WAL file on every open

Refs:

- `src/rust/crates/fathomdb-engine/src/lib.rs:3024`
- `dev/acceptance.md:580`

`probe_wal_sidecar()` uses `std::fs::read(&wal_path)` just to inspect
the header. That makes `Engine.open` pull the full `-wal` sidecar
into memory before SQLite does its own recovery work. On large
unclean-shutdown WALs, this is a real open-path latency and RSS
regression, and it directly risks the later AC-035 recovery-time
budget. The schema-probe change is not the main latency concern
here; the full-file WAL read is.

### 2. [medium] AC-034b drops some trials from the p99 instead of measuring across all P-PWR-TRIALS

Refs:

- `src/rust/crates/fathomdb-engine/tests/durability_soak.rs:95`
- `src/rust/crates/fathomdb-engine/tests/durability_soak.rs:128`
- `dev/acceptance.md:568`

The harness records `lost_commit_ms: Option<u128>`, filters out
`None`, then computes p99 on the surviving subset. It only requires
that 90% of trials have at least one surviving commit. That is
weaker than AC-034b's stated measurement "report p99 across
P-PWR-TRIALS trials," and it means the parent never actually
guarantees that each victim committed at least one row before
`SIGKILL`.

### 3. [medium] AC-035c's lock-release check is same-process, not sibling-process

Refs:

- `src/rust/crates/fathomdb-engine/tests/durability_open_path.rs:286`
- `dev/acceptance.md:608`

The test reacquires `.lock` with a second fd in the same process
and calls `flock` there. That is useful signal, but it is not the
documented measurement, which requires process A to fail open and
sibling process B to observe the lock as acquirable. Given the
explicit AC text, this is a partial proof, not the full contract.

### 4. [low] The AC-034c ignored test is informative, but its body is a stub

Refs:

- `src/rust/crates/fathomdb-engine/tests/durability_soak.rs:238`

The `#[ignore]` reason is good and points at the missing VM/sysrq
substrate, but the body is intentionally empty. If someone clears
`#[ignore]` without implementing the harness, the test would pass
vacuously. This is not a current false green because it is ignored,
but it is not a runnable placeholder.

## What passed on inspection

- The `open_locked` reorder is directionally correct: header and
  schema probing now happen before `journal_mode=WAL`, which
  prevents header/page-1 and schema-btree corruption from being
  misclassified as WAL replay.
- AC-035b is asserted per fixture for all four documented rows.
  Tests check exact `kind`, `stage`, `recovery_hint.code`, and
  `doc_anchor`. Code-stability test checks both string equality
  and pointer equality for the `&'static str`.
- `serial_guard()` is taken by every test in
  `durability_open_path.rs`, not just the AC-035c tests, so
  same-binary thread/fd contamination is actually serialized
  away.
- The thread/fd residue assertions are consistent with the engine
  lifecycle: reader threads and the projection runtime are only
  created after `open_locked` succeeds, so a corruption return
  from `open_locked` should not leave engine-started threads
  behind.
- No meaningful large-schema regression from `SELECT COUNT(*) FROM
  sqlite_schema` itself; schema surface is small and mostly fixed.

## Orchestrator triage

All four findings substantive and actionable. None
structural/prompt-induced — orchestrator override not appropriate.
**Fix-1 remediation required.**

- **Finding 1 (high)**: `probe_wal_sidecar` must only read the
  WAL header (first ~32 bytes — magic, page-size, salts), not the
  full file. SQLite's WAL header is fixed-layout; we don't need
  the frame data. Fix: replace `std::fs::read` with a bounded
  `File::open` + `read_exact(&mut [u8; HEADER_LEN])`. Add a test
  that opens a large fixture WAL (e.g. 10 MB) and asserts RSS
  delta is bounded.
- **Finding 2 (medium)**: ensure every victim commits at least
  one row before SIGKILL. Parent should signal "ready" via a
  per-trial sentinel file the child writes after its first commit
  lands; parent waits for sentinel before scheduling the kill.
  Then `lost_commit_ms` is `Some(_)` for every trial and p99 is
  computed across the full P-PWR-TRIALS=100 set.
- **Finding 3 (medium)**: replace in-process flock reacquire with
  true sibling-process check. Reuse Sub-2's test-binary-as-victim
  trick: spawn a child via `Command::new(current_exe())` with an
  env-var-gated entry point that attempts `flock` on `<db>.lock`
  and exits 0 if acquired / 1 if blocked. Parent invokes the
  AC-035c fixture, then asserts the child exits 0.
- **Finding 4 (low)**: AC-034c body must `panic!()` with the
  blocker text + a pointer to the missing substrate. Cleared
  `#[ignore]` without doing the work = test fails loudly instead
  of passing vacuously.

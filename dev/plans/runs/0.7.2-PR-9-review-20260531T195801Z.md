## Verdict: CONCERN

### 1. [medium] Timeout breaks strict embed serialization after the watchdog fires
Refs: `src/rust/crates/fathomdb-engine/src/lib.rs:3446-3457`, `src/rust/crates/fathomdb-engine/src/lib.rs:3490-3502`

`embed_serialize` only covers the worker's wait on `recv_timeout`, not the lifetime
of the underlying `embedder.embed()` call. When a call times out, the worker drops
the mutex and retries, but the detached embed thread keeps running. That means later
retries or jobs can re-enter the shared embedder concurrently with the abandoned
call, so the PR-9 "one embed at a time" fix stops being true on the timeout path.
Given the pre-flight result, this looks like a robustness/perf risk rather than an
immediate writer/cursor correctness break, but it is the main remaining hole in the
serialization + watchdog composition.

## Disposition: CONCERN ACCEPTED (orchestrator, per orchestration.md §7)

This is the trade-off the slice spec **mandates**, not a defect:

- The PR-9 prompt's "Design tension" section explicitly requires: *"the watchdog
  design MUST account for lock release on timeout (drop the guard before/at timeout,
  treat a timed-out embed's worker as recoverable, **no poisoned Mutex wedge**)."*
  The implementation does exactly this.
- The alternative the finding implies — hold the guard **inside** the embed thread
  for strict serialization — is the behavior the spec and ADR-0.6.0 Invariant 5
  **forbid**: a genuinely-hung embed would then hold the guard forever and deadlock
  the entire projection pool. The chosen design takes the prescribed escape.
- The relaxation is **fault-path only**. With the 30s default deadline and sub-second
  real `CandleBgeEmbedder` embeds, timeouts never fire in normal operation, so
  steady-state serialization is strict and the ~13x throughput win is unaffected. The
  transient overlap exists only while an abandoned (timed-out) embed thread is still
  finishing — i.e. only when the embedder is already faulting.
- HITL pre-approved this exact interaction ("watchdog releases the guard on timeout
  so a hung embed can't stall the pool") when authorizing the serialize-engine-side
  fix.

The accepted trade-off is now self-documented at the `embed_serialize` field
(`src/rust/crates/fathomdb-engine/src/lib.rs`). A future slice may replace the hard
serialize with an operator-configurable embed-concurrency limit (ADR-0.6.0
Invariant 4 pool-size override); that knob is out of PR-9 scope.

## What passed on inspection (codex)

- Commit path stays **outside** the new embed mutex; commit remains under the
  existing separate `commit_gate`. Commit/IO genuinely stays parallel.
- Single-writer and projection-cursor freshness contracts preserved — the new logic
  does not move commit/cursor advancement; worker panic cleanup still routes through
  the existing batch-level `catch_unwind` + state-decrement path.
- Panic transparency implemented correctly: watchdog-thread panics are caught and
  resumed on the worker thread, so the existing `ProjectionPanic` machinery remains
  in force.
- The mechanism tests are honest: `pr9_embed_serialization.rs` checks
  `max_in_flight == 1`; `pr9_embed_watchdog.rs` covers drain return, terminal
  failure / no stored vector, subsequent-write recovery, and a non-timeout slow case.

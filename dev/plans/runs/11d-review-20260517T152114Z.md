## Verdict: BLOCK

### 1. [high] `workflow_dispatch` can perform a real release without the release gate

Refs:

- `.github/workflows/release.yml:7-12`
- `.github/workflows/release.yml:43-49`
- `.github/workflows/release.yml:213-217`
- `.github/workflows/release.yml:369-373`
- `.github/workflows/release.yml:414-417`
- `.github/workflows/release.yml:427-490`

`workflow_dispatch` exposes `dry_run`, but `verify-release` skips
`scripts/verify-release-gates.sh` on every dispatch run, not just
rehearsals. If someone dispatches with `dry_run: false`, the
workflow can reach real cargo publish, real PyPI publish, real npm
publish, post-publish smoke, co-tagging assert, and `github-release`
without the tag/main/changelog/version gate ever running. That is a
release-path correctness blocker.

### 2. [medium] Dry-run is not actually "no real publish"

Refs:

- `.github/workflows/release.yml:9-10`
- `.github/workflows/release.yml:358-373`
- `dev/plans/runs/11d-release-workflow-output.json:50-53`

The dry-run input is documented as "no real publish", but the PyPI
leg still runs `pypa/gh-action-pypi-publish` against TestPyPI. That
is still an irreversible external publish (TestPyPI also rejects
same-version re-uploads), so the workflow does not skip all
irreversible steps in dry-run mode.

### 3. [medium] Blocker 7 resolves npm naming in a way that diverges from the locked release design

Refs:

- `dev/design/release.md:82-86`
- `src/ts/package.json:2-3`
- `scripts/release/smoke/smoke-npm-package.sh:29`
- `dev/plans/runs/11d-release-workflow-output.json:64-66`

The checked-in workflow and smoke scripts consistently publish/install
`fathomdb`, but the canonical release design still describes npm
smoke for `@fathomdb/...` publishes. If the bare-name decision is
intentional, `dev/design/release.md` needed to change in the same
slice; as landed, blocker 7 introduces a design/code mismatch.

## Orchestrator triage

All three findings legit. Spawn 11d-fix-1 remediation pass
(Opus 4.7 medium).

- Finding 1 = workflow correctness. Two valid fixes:
  - (a) Always run `verify-release-gates.sh` in `verify-release`,
    skipping only tag-format check on dispatch.
  - (b) Enforce `dry_run == 'true'` precondition for
    `workflow_dispatch` events; reject otherwise.
  Recommended: (a). Dispatch with `dry_run: false` is a legitimate
  emergency-republish path; the gates must run.
- Finding 2 = dry-run semantics. Skip `publish-pypi` and
  `publish-npm` entirely on dry-run (the cargo `--dry-run` flag
  already covers the rust path; PyPI and npm have no equivalent
  non-destructive mode that's worth the TestPyPI version burn).
  Still execute the `build-*` matrix on dry-run — that's the
  legitimate test signal.
- Finding 3 = doc update. Patch `dev/design/release.md` to swap
  `@fathomdb/...` for bare `fathomdb` (single-brand decision per
  11d Blocker 7 resolution).

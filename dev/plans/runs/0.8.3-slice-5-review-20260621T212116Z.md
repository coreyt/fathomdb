### 1. [high / P1] `augment_class()` admits mislabeled gold (class definition not enforced)
Refs: `src/python/eval/gold_repin.py:183-189`
`augment_class()` only checks `required_evidence` is non-empty, so `multi_session` / `knowledge_update`
queries answerable from a **single** document are accepted. **Confirmed in the committed gold:** `d0a-0441`
(`multi_session`, n_evidence=1) and `d0a-0235` (`knowledge_update`, n_evidence=1). Real LongMemEval gold for
these classes spans **≥2 sessions**. Slice 10 would score single-session questions as multi_session/
knowledge_update — biasing the parity numbers for the very classes this re-pin exists to repair.

### 2. [high / P1] Mem0 local baseline leaks state across runs (non-reproducible)
Refs: `src/python/eval/mem0_local.py:39-40`
The default config points every run at the same on-disk Chroma collection (`/tmp/mem0_chroma`, `r2_eval`)
with a fixed `user_id`. Re-running on a different corpus/branch reopens the previous collection and searches
stale memories ⇒ the Mem0 baseline is non-reproducible unless `/tmp/mem0_chroma` is manually wiped.

### 3. [medium / P2] corpus-validity guard not enforced on the runtime path
Refs: `src/python/eval/r2_parity_eval.py:579-592`
`load_repin_gold()` only checks a non-empty `corpus_hash` + ≥1 query; the new `validate_repin()` is never
called at runtime. A rebuild that underfills a class (`--no-augment`, a stale checkpoint) still runs and
`run()` quietly emits 0/`null` for missing classes — recreating the exact Slice-25 `N=0` mode this slice
was meant to prevent.

## What passed on inspection
Re-pin produces a new `corpus_hash` (1859817a); the standalone `validate_repin()` guard is non-vacuous
(catches N=0 / under-N_min / missing-class in unit tests); Mem0 de-risk correctly identifies a footprint-safe
local backend (no cloud); scope clean (no engine/Rust; `decision_rule_083.py`/`m1_decision_rule.py`
untouched); cheap-validate only, no full priced run.

**Type:** `[research/experiment]` · **Baseline:** local `main` @ `405db00` · **Date:** 2026-06-07
**Worktree:** `/tmp/fdb-slice-6-20260607T003001Z` (branch `slice-6-20260607T003001Z`)
**Deliverable scope:** findings + HITL decision points only. **No** production tokenizer / migration /
AC / `perf_gates`-budget change (one temporary, reverted A/B edit — see §Methodology; not in the diff).

---

## TL;DR (the headline, then the numbers)

1. **The Slice-5 tokenizer upgrade is NOT the cause of the AC-012 latency.** Two independent controlled
   measurements — a faithful FTS-path config sweep AND a full-engine A/B that swaps only the tokenizer —
   show `porter unicode61 remove_diacritics 2` and the v0.7.2 default `unicode61` are **latency-identical**
   at every tier. The Slice-40 root-cause attribution ("the porter stemmer + diacritics folding is
   materially more expensive at query time") is **refuted by measurement**.
2. **The latency scales with corpus size N, not the tokenizer.** AC-012's single-token queries hit a
   common-band term whose posting list grows linearly with N (≈320 rows @10k → ≈3,212 rows @100k); the cost
   is the O(N) FTS5 MATCH scan + `bm25()` scoring + materializing **every** matched row (the engine query is
   unbounded — no `LIMIT`). This is the **same O(N) story** that already drove the HITL tiering of the
   sibling latency ACs (AC-013/AC-019 → AC-072/AC-073).
3. **At the BINDING 10k tier the regression does not breach** (real `ac_012` p50 < 1 ms, p99 4 ms vs the
   20/150 ms budget). It breaches **only at the unconditional 100k tier**, which — unlike its siblings —
   was never tiered.
4. **Lightening the tokenizer buys ~0 ms and costs FTS quality** (recall 8/8 → 5/8 on a morphology/diacritic
   proxy). It is the wrong lever.

→ **Recommended HITL path: tier AC-012 to match AC-013/AC-019** (10k binding, 100k/1M tracked-not-gated),
via the existing AC-072/AC-073 precedent. Full fork in §Recommendation.

## Methodology (so every number is reproducible and the faithfulness is explicit)

- **Box:** x86_64, 24 logical cores, Linux 6.17. SQLite/FTS5 **bundled** via `rusqlite 0.31`
  (`features=["bundled"]`, vendored amalgamation) — porter/unicode61/ascii are SQLite built-ins.
  **Caveat:** absolute latencies are box-specific. This box differs from both the Slice-40 measurement box
  and the canonical x86_64 tier-1 CI runner, so the **relative / A-B conclusions are what transfer**, not
  the absolute milliseconds. (The Slice-40 board cited an aarch64 dev box for B3; this run is x86_64.)
- **Build:** `cargo test --release` for every latency number (matches `perf-canonical.yml`, which builds
  and runs `perf_gates` under `--release`). Debug numbers are not reported as latency.
- **Isolation:** each perf run executed alone (no concurrent `cargo test`), per the Slice-40 harness caveat
  that workspace-concurrent runs corrupt timing.
- **Two complementary instruments:**
  1. **Real `ac_012`** (`perf_gates::ac_012_text_query_latency_on_fts5_path`) — the production engine path
     (`Engine::open_without_embedder_for_test` → `search()` → ReaderWorkerPool → FTS branch). Run with
     `AGENT_LONG=1` and `AC012_CORPUS_N={10000,100000}`. This is the **authoritative gate-level** number and
     reports its percentile via the gate's coarse power-of-two-microsecond `percentile_ceil` (so its p50/p99
     are bucket ceilings, e.g. "21 ms" = the ≤2^? µs bucket).
  2. **A throwaway, faithful FTS-path harness** —
     `src/rust/crates/fathomdb-engine/tests/slice6_tokenizer_experiment.rs` (gated behind
     `SLICE6_EXPERIMENT=1`; never runs in `cargo test`/CI). It reproduces the engine's text path exactly: the
     same FTS5 schema, the **byte-identical** synthetic corpus (same `SeededRng` seeds, `perf_vocab`,
     Zipfian `synth_chunk_body`) as `seed_ac012_corpus`, the same single-token query band, and the same SQL
     the engine runs (`lib.rs:3923-3927`):
     `SELECT body, kind, write_cursor, bm25(search_index) FROM search_index WHERE search_index MATCH ?1 ORDER BY write_cursor`.
     It varies **only** the `tokenize=` clause and reports finer **sorted-array** percentiles (1,000 samples,
     warmup pass discarded). It deliberately bypasses the ReaderWorkerPool (a tokenizer-independent constant),
     so its absolutes read below the gate; **faithfulness is cross-checked** against the real `ac_012`.
- **One temporary, reverted production edit (the engine-level A/B):** to isolate the tokenizer's contribution
  at the *full-engine* level, `migrations` step-11's `tokenize` string in `fathomdb-schema/src/lib.rs` was
  temporarily set to `unicode61`, `ac_012` @100k re-run, then the line was **reverted to
  `porter unicode61 remove_diacritics 2`**. Verified restored (`git status` clean except the new harness;
  canonical tripwire empty). Not in the committed diff.

## Q1 (LEAD) — at which tier does the regression breach a *binding* budget?

**Real `ac_012`, release, isolated, this box. Budget = p50 ≤ 20 ms AND p99 ≤ 150 ms (unconditional @100k).**

| Tier (N) | seed_ms | p50 | p99 | vs 20/150 budget | Gate verdict |
|---------:|--------:|----:|----:|------------------|--------------|
| **10,000**  | 259  | **< 1 ms** | **4 ms**  | well within | **PASS** (huge margin) |
| **100,000** | 2,900 | **21 ms** | **99 ms** | p50 over by ~1 ms; p99 within | **FAIL** (p50 only) |

**Answer:** the regression breaches **only at the 100k tier**, and there only on **p50** (by ~1 ms); p99
(99 ms) is comfortably within 150 ms. At the **10k tier — the binding tier for the sibling latency ACs
(AC-013/AC-019 per `ADR-0.7.0-text-query-latency-gates-revised`)** — `ac_012` passes by a wide margin.

**This is the cleanest resolution surfaced to HITL: `ac_012` was simply never tiered to match AC-013/AC-019.**
Its sibling `ac_013` asserts the budget only at `n <= AC013_GATE_N (10_000)` and *reports* 100k/1M as
post-1.0 tracked targets (`perf_gates.rs:603`); `ac_012` asserts **unconditionally at `AC012_DEFAULT_N =
100_000`** (`perf_gates.rs:145,553`). (Surfaced, not decided — see §Recommendation.)

> Note on the Slice-40 number (p50 49.9 / p99 315 ms @100k): not reproduced here (got 21/99 on an idle
> x86_64 box). The gap is **hardware/load sensitivity**, the same class of effect as B3 (`ac_020`). The
> *direction* (100k p50 marginally over a 20 ms budget) reproduces; the *magnitude* is box-dependent.

## Q2 — tokenizer config space: latency vs FTS quality

### Q2a — latency sweep (faithful FTS-path harness, release, sorted percentiles, 1,000 samples)

Every config is queried with the **identical** AC-012 single-token band against the **identical** corpus.
`avg_rows` = mean matched rows per query (drives the cost); `pages` = `PRAGMA page_count` (index size).

| Config | tier | pages | avg_rows | p50 (ms) | p99 (ms) |
|--------|-----:|------:|---------:|---------:|---------:|
| `porter unicode61 remove_diacritics 2` **(CURRENT)** | 10k | 76 | 320 | 0.323 | 0.898 |
| `unicode61 remove_diacritics 2` (drop porter) | 10k | 64 | 320 | 0.326 | 0.912 |
| `porter unicode61` (default diacritics) | 10k | 68 | 320 | 0.331 | 0.921 |
| `unicode61` **(≈ v0.7.2 default)** | 10k | 65 | 320 | 0.327 | 0.901 |
| `porter ascii` | 10k | 66 | 320 | 0.329 | 0.917 |
| `ascii` (lightest) | 10k | 63 | 320 | 0.330 | 0.921 |
| `porter unicode61 remove_diacritics 2` **(CURRENT)** | 100k | 18047 | 3212 | 3.997 | 16.004 |
| `unicode61 remove_diacritics 2` | 100k | 18047 | 3212 | 3.934 | 15.934 |
| `porter unicode61` | 100k | 18047 | 3212 | 3.978 | 16.114 |
| `unicode61` **(≈ v0.7.2 default)** | 100k | 18047 | 3212 | 3.969 | 16.147 |
| `porter ascii` | 100k | 18047 | 3212 | 4.046 | 16.042 |
| `ascii` (lightest) | 100k | 18047 | 3212 | 4.001 | 16.093 |

**All six configs are within run-to-run noise of each other** at both tiers, with **identical** `avg_rows`
and (at 100k) identical `pages`. Reason: porter is a **no-op on the synthetic AC-012 vocabulary** — tokens
are `{a}{b}{c}{i:04}` (three letters + four digits); SQLite's porter stemmer passes digit-bearing tokens
through unchanged, so the term set, posting lists, and index are bit-for-bit the same. **The tokenizer is
not a latency lever on this fixture.**

### Q2b — engine-level A/B (real `ac_012`, release, isolated, @100k) — the definitive control

| Tokenizer (step-11) | p50 | p99 |
|---------------------|----:|----:|
| `porter unicode61 remove_diacritics 2` (HEAD) | **21 ms** | 99 ms |
| `unicode61` (v0.7.2 default, temporarily swapped + reverted) | **20 ms** | 99 ms |

Identical within noise at the **full engine level** — and note the v0.7.2 tokenizer **also** sits at the
20 ms p50 boundary. **This directly refutes the Slice-40 attribution of the regression to the tokenizer.**
Whatever HEAD-vs-v0.7.2 gap the Slice-40 run observed came from non-tokenizer factors (measurement
conditions, and/or tokenizer-independent engine overhead — see Q3) — not `porter`/`remove_diacritics`.

### Q2c — FTS quality proxy (what the tokenizer actually buys)

8 query→variant pairs exercising morphology and diacritics. `bits` = per-query hit, order
`[run running ran optimize cafe naive resume connect]`.

| Config | recall | bits | what it loses |
|--------|:------:|------|---------------|
| `porter unicode61 remove_diacritics 2` **(CURRENT)** | **8/8** | 11111111 | — |
| `porter unicode61` | 8/8 | 11111111 | — (diacritics already folded by default `remove_diacritics 1`) |
| `unicode61 remove_diacritics 2` | 5/8 | 01101110 | **stemming**: run→runs, optimize→optimizing, connect→connected |
| `unicode61` **(≈ v0.7.2)** | 5/8 | 01101110 | **stemming** (same 3) |
| `porter ascii` | 5/8 | 11110001 | **diacritics**: cafe→café, naive→naïve, resume→résumé |
| `ascii` | 2/8 | 01100000 | **both** stemming and diacritics |

Reading: **porter buys morphological recall** (+3: run/optimize/connect); **diacritic folding buys
accent-insensitivity** (+3: cafe/naive/resume). Crucially the diacritic wins are already delivered by
`unicode61`'s **default `remove_diacritics 1`** — so the *incremental* FTS-quality gain of the Slice-5
upgrade over v0.7.2's default `unicode61` is specifically the **porter stemmer** (the `remove_diacritics 2`
bump over `1` only adds coverage for extra combining marks, not exercised by Latin-1 text). This is the G1
quality goal the Slice-5 design memo chose porter to serve. Since latency is config-independent (Q2a/Q2b),
**the tokenizer choice is a pure quality decision, decoupled from the latency gate.**

## Q3 — query-path: where is the cost?

Faithful harness @100k, current tokenizer, 1-/2-/3-token queries from the same band:

| Query shape | avg_rows matched | p50 (ms) | p99 (ms) |
|-------------|-----------------:|---------:|---------:|
| 1 token  | 3,212 | 3.93 | 15.88 |
| 2 tokens (phrase) | 2 | 0.25 | 0.55 |
| 3 tokens (phrase) | 0 | 0.19 | 0.32 |

> **[CORRECTION — codex §9 [P2], 2026-06-07]** The 2-/3-token rows were built by the throwaway harness as a
> quoted FTS *phrase* (`"tok1 tok2"`, adjacency-required), **not** the engine's `tok1 AND tok2` (the production
> path splits on whitespace and joins with `AND` via `compile_text_query`). The original "(`AND`)" labels were
> therefore inaccurate — corrected to "(phrase)". **The Q3 conclusion is unaffected:** a phrase query is *at
> least as selective* as the AND it stood in for (adjacency ⊆ co-occurrence), so the true AND result set is
> ≥ these rows and ≤ the 1-token set — the monotone "more query terms → fewer matched rows → lower latency,
> latency tracks result-set size" claim holds either way. The decision-critical evidence (Q1 tiering, Q2a
> config sweep, Q2b engine A/B) uses **single-token** queries only and is unaffected by this phrase-vs-AND
> distinction.

**Latency tracks result-set size, not token count / tokenization.** A single common-band token matches
~3,212 rows → the engine scans the posting list, computes `bm25()` per row, sorts by `write_cursor`, and
materializes **all** of them (the SQL is unbounded — no `LIMIT`). Add a second query term (phrase here; `AND`
in production — see the [P2] note above) and the result set collapses to ~2 rows → 16× fewer rows → ~16× faster. Query-time porter stemming is one stem per token =
negligible (and Q2b shows it's free even at the engine level).

**Implication:** the only real *query-path* lever is **bounding the matched-row materialization** (a top-k
`LIMIT` on the FTS branch — there is already a dormant, experiments-gated `FATHOMDB_PERF_SEARCH_LIMIT` knob
at `lib.rs:3912`). That is a **recall-affecting semantic change**, not a tokenizer change, and is **out of
scope** here; flagged for HITL as a distinct, larger lever if the 100k budget is to be *met* rather than
*tiered*.

> Secondary observation (not load-bearing): the real engine adds ~17 ms p50 @100k over the raw-FTS harness
> (21 ms vs ~4 ms), and this overhead also grows with matched-row count (ReaderWorkerPool dispatch +
> dedup-on-body over ~3,212 ~500-byte bodies + `SearchHit` materialization). It is **tokenizer-independent**.
> Precise attribution within that overhead was not isolated (would need a 0.7.2-engine A/B); the load-bearing
> claim — tokenizer is latency-neutral — stands on Q2a + Q2b.

## Q4 — source accessibility / modifiability (the HITL-asked dimension)

SQLite/FTS5 is the **bundled vendored amalgamation** (`rusqlite 0.31` `features=["bundled"]`). You do **not**
patch SQLite source to change tokenizers; the practical levers are the tokenize-config string and (heavier)
the FTS5 tokenizer API.

| Lever | Accessibility | Rough cost | Latency motivation (from this experiment) |
|-------|---------------|------------|-------------------------------------------|
| (a) Swap among built-in stemmers/options (`porter`/`unicode61`/`ascii`/`remove_diacritics N`) | **Trivial** — it's the migration `tokenize=` string (one line + its `lib.rs` mirror; FTS5 has no `ALTER tokenize`, so a drop+recreate+reproject migration like step-11) | Low (migration slice) | **None** — config is latency-neutral (Q2). Pure quality knob. |
| (b) Register a **custom FTS5 tokenizer** via the tokenizer API (`fts5_api` / `xCreateTokenizer`) | **Feasible but awkward** — `rusqlite` does not expose `fts5_api` ergonomically; needs `unsafe` FFI to the bundled FTS5 C API (or a small C shim) and careful lifetime/registration handling | Medium–high (unsafe FFI + ongoing maintenance; cross-platform build care) | **None for 0.8.0** — no latency problem the config sweep couldn't already address; only justified by a future *quality* need (e.g. domain-specific stemming) |
| (c) Embedder source (candle / bge-small, `fathomdb-embedder`) | In-tree, modifiable | n/a here | **Irrelevant to this path** — `ac_012` opens **without** an embedder; the FTS5 text path never calls the embedder. The embedder bears only on the vector/AC-013 latency (separate, already tiered). |

**Bottom line:** the source is fully accessible at the cheap (config) tier and that tier is sufficient — but
the experiment shows there is **no latency reason** to reach for any of these levers. The accessibility
question resolves to "easy, but unnecessary for B2."

## Recommendation

**Tier AC-012 to match its sibling latency ACs (10k binding, 100k/1M tracked-not-gated) — HITL fork option
(i).** Grounds, all measured:

1. **The cost is O(N) corpus-scaling, not the tokenizer** (Q2a/Q2b/Q3) — structurally identical to the
   vec0 bit-KNN O(N) scan that HITL already tiered for AC-013/AC-019.
2. **There is direct, HITL-locked precedent**: `acceptance.md` already did exactly this twice — **AC-072
   supersedes the AC-013 budget** and **AC-073 supersedes AC-019** (`ADR-0.7.0-text-query-latency-gates-revised`,
   2026-06-01), making the 10k tier the binding release gate and 100k/1M tracked post-1.0 targets. `ac_013`'s
   `AC013_GATE_N` tiering is the code pattern to mirror in `ac_012`.
3. **It passes at the binding 10k tier by a wide margin** (Q1: p50 < 1 ms vs 20 ms).
4. **The only thing Slice-5 changed (the tokenizer) is exonerated** (Q2b) — so "remediate the tokenizer"
   cannot fix the 100k number and would only degrade FTS quality (Q2c). The Slice-5 G1 quality upgrade
   (porter stemming) should be **kept**.

Mechanically (for the follow-on Slice 7, **not done here**): mint a superseding AC (next id is **AC-076**;
AC-075 is reserved by the B1 recall-verdict per the board) that tiers AC-012 — assert 20/150 ms only at
`AC012_GATE_N = 10_000`, report 100k/1M as tracked targets — exactly mirroring `ac_013`'s `AC013_GATE_N`
branch; keep AC-012 as the retained-for-history legacy budget (as AC-013/AC-019 were). This is an AC change
→ gated slice (`acceptance.md` locked; [[acceptance-md-locked-no-feature-acs]]). Folds naturally into the
Slice-40 re-scope alongside B1/Q3 (wire the **10k-tier** `ac_012` assertion + `eu7` recall to the canonical
x86_64 runner so the gate actually runs per-push — the META gap from the Slice-40 HALT).

### The explicit HITL fork

| Option | What it is | Verdict from this experiment |
|--------|-----------|------------------------------|
| **(i) Tier AC-012** *(RECOMMENDED)* | 10k binding, 100k/1M tracked, per AC-072/073 precedent | **Best.** Precedented, latency-honest, preserves G1 quality, passes binding tier. |
| (ii) Lighten the tokenizer | Drop porter / `remove_diacritics 2` | **Reject.** ~0 ms latency gain (Q2a/Q2b) and −3/8 FTS recall (Q2c). Wrong lever. |
| (iii) Custom FTS5 tokenizer | `xCreateTokenizer` via FFI | **Not for 0.8.0.** Medium-high cost, no latency motivation (Q4). |
| (iv) Re-ratify the unconditional 100k budget | Accept/raise the 100k number (e.g. p50 ≤ ~25–30 ms) | **Viable but off-pattern.** Inconsistent with the sibling ACs' tiering philosophy; the synthetic 100k O(N) target is exactly what AC-072 deferred to post-1.0 ANN work. Only choose if a hard 100k FTS gate is wanted at GA. |
| (v) *(orthogonal)* Bound the FTS result set | Top-k `LIMIT` on the text branch (`FATHOMDB_PERF_SEARCH_LIMIT`) | **Separate, larger decision.** Would *meet* (not tier) the 100k budget but changes recall semantics. Out of scope for B2; flag for a dedicated retrieval ADR if pursued. |

**Decision is HITL's.** This experiment's job was to make it well-informed: the tokenizer is not the problem,
N is; the binding tier is green; precedent exists.

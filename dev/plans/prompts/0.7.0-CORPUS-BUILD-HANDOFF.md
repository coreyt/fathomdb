# 0.7.0 CORPUS-BUILD — implementation handoff

Drafted: 2026-05-27
Type: **Implementation handoff** — produce a real-data
composite test corpus for FathomDB's 0.7.0 vector-cost work
(Pack 2 RED tests and broader retrieval evaluation).

## What you are building

A **composite real-data corpus** (~10K documents — "Version B"
of the research blend) that exercises FathomDB's three retrieval
modalities together:

- **FTS** (SQLite FTS5 text search via `engine.search()` text path).
- **Vector** (sqlite-vec `vec0`, post-Pack-1 schema with
  `bit[768]` + f32 + metadata + `source_type` partition_key).
- **Graph** (FathomDB's chunk/document/relation tables —
  cross-document fact-reconstruction).

The corpus has three jobs:

1. **Drive Pack 2's recall-floor RED test** (`recall@10 ≥ 0.90`
   vs f32 brute-force ground truth).
2. **Validate the new schema** (metadata columns,
   `source_type` partition_key, `bit[768]` ingest double-write)
   under realistic document shapes and sizes.
3. **Establish a cross-document retrieval baseline** (e.g.,
   "an email mentions a meeting, the meeting creates a to-do,
   a later note changes the decision") so we can measure whether
   FathomDB merely returns semantically similar documents or
   actually reconstructs the user's working context.

## Pack structure

This is large enough to merit its own packs. Land in this
order; each pack closes with a commit on `main` and a brief
closure note.

### Corpus-Pack 1 — acquisition + cleaning pipeline

Per source, produce a clean canonical JSON-Lines file:

```text
{doc_id, source_type, title, body, created_at, modified_at,
 author_or_sender, recipients, people_mentions,
 project_mentions, tags, url_or_external_id, thread_id,
 parent_doc_id, license, provenance}
```

Schema fields:

- `doc_id`: stable hash of `provenance + source-native ID`.
- `source_type`: one of the locked vocabulary (`email`,
  `article`, `paper`, `meeting`, `note`, `todo`).
- `created_at` / `modified_at`: ISO-8601 UTC. Use source-
  provided dates; if missing, set to the source's snapshot
  date.
- `license`: SPDX identifier; copy from per-source LICENSE
  files / upstream metadata.
- `provenance`: a short string identifying the upstream
  (e.g., `cmu-enron-2015-05-08`, `qmsum-v2.0`).
- `people_mentions` / `project_mentions`: best-effort NER
  output. Sufficient for v1: simple regex / spaCy default
  pipeline; don't overinvest.
- `thread_id` / `parent_doc_id`: populated for email threads
  and meeting → minutes pairs; null elsewhere in this pack.

Per-source acquisition + cleaning:

- **Enron (CMU)**: parse the original mailbox dump; extract
  per-message headers + body. Strip signatures. Build
  `thread_id` from `In-Reply-To` headers. Subsample to 2,000
  docs evenly across senders (do not over-weight one user).
- **QMSum**: download the GitHub release (verify the latest
  release tag; cite the commit SHA). Each meeting becomes one
  doc with the full transcript as `body`; the query-summary
  pairs become a separate `qa.jsonl` file (cross-doc chains
  may use these).
- **ELITR**: download from LINDAT (cache only; do NOT commit
  raw or processed ELITR data to the repo). Same per-meeting
  shape as QMSum.
- **PMC OA + S2ORC**: use the upstream-permitted retrieval
  channels (PMC: OA Cloud or E-Utils; S2ORC: Semantic Scholar
  Bulk Dataset API). Subsample to keep license-clean articles.
- **OpenAlex**: metadata-only enrichment; for each PMC / S2ORC
  paper, fetch the OpenAlex record to populate authors,
  citations, and venue fields. CC0, commit-clean.
- **CNN/DailyMail**: use the HF `abisee/cnn_dailymail` dataset.
  The canonical HF distribution strips dates/URLs — synthesise
  reasonable `created_at` values uniformly across the
  publication window (2007–2015 for CNN/DM) and mark as
  `provenance: synthetic-date`.
- **Imperative To-Do**: clone the GitHub repo (verify license);
  parse the corpus per the repo's README. Synthesise project
  names, due dates, priorities per the research doc's
  recommendation.
- **Public daily-log markdown repos**: clone 2-3 repos with
  permissive licenses; parse each `.md` file as one note doc.
- **Synthetic notes**: generate 1,200 markdown notes across
  the styles listed in the research doc (fleeting, project,
  reading, idea sketches, decision logs, "someday/maybe",
  personal CRM, meeting follow-ups). See Corpus-Pack 2 for
  the generator; Corpus-Pack 1 just provisions the file
  shape.

Output of Corpus-Pack 1: per-source cleaned JSONL files in
`tests/corpus/raw/{source}.jsonl` (committable sources) or
`{CACHE_DIR}/corpus/raw/{source}.jsonl` (cache-only sources),
plus a `tests/corpus/corpus-card.md` documenting every source,
its license, its provenance, and any commit-or-cache flag.

### Corpus-Pack 2 — synthetic cross-document chain generator

Per HITL: in 0.7.0 scope. This is where the corpus stops being
"random documents" and starts exercising fact-reconstruction.

Design a generator that produces ~200 verified **multi-document
chains**, each of which has known ground-truth retrieval
expectations. Example chain shapes:

- **EMAIL → MEETING → TODO → NOTE**: an email proposes a
  meeting; the meeting transcript discusses Topic X; a to-do
  is created from the meeting; a later note revises the
  decision.
- **PAPER → NOTE → TODO**: a paper is saved; a note summarizes
  its key claim; a todo follows up with a verification action.
- **ARTICLE → NOTE → EMAIL**: a news article is saved; a note
  links it to an ongoing project; an email shares the link with
  a colleague.

Chains are generated by **synthesizing connective documents**
(notes, emails, todos) that reference real documents from the
acquired corpora (Enron messages, QMSum meetings, PMC papers,
CNN/DM articles). The synthetic notes/emails/todos cite the
real-doc IDs verbatim via `parent_doc_id` and `people_mentions`.

For each chain, emit:

- The chain definition in `tests/corpus/chains/<chain_id>.json`:

  ```text
  {chain_id, chain_shape, doc_ids: [...],
   ground_truth_queries: [
     {query: "what did we decide about X?",
      expected_top_k_doc_ids: [...],
      relation_type: "summarizes|follows_up|contradicts|..."}
   ]}
  ```

- The synthetic documents themselves, added to the appropriate
  per-source JSONL with `provenance: synthetic-chain` and the
  chain_id stored in `tags`.

This file is the **input to Pack 2's recall-floor RED test**:
the test loops over chains, runs each `ground_truth_query`
through `engine.search()`, and asserts `recall@10` of the
expected doc IDs against the actual top-10 returned.

Determinism: the generator takes a fixed RNG seed. Document
the seed in the corpus card.

### Corpus-Pack 3 — FathomDB ingest harness

Produce a CLI tool (Rust or Python — pick whichever is the
shorter path; Rust if you can call `fathomdb-engine` directly,
Python otherwise via the existing client). Tool reads the
cleaned JSONL + chain JSON, and ingests each document into a
FathomDB instance via the public `engine.write()` API. Map:

- `body` → chunk text; chunk per FathomDB's existing chunking
  convention (verify in lib.rs before writing; do not invent
  a new chunker).
- `source_type` → vector kind + Pack 1's `source_type`
  metadata column.
- Metadata fields → corresponding Pack 1 metadata columns.
- `thread_id` / `parent_doc_id` → graph relations
  (`replies_to`, `summarizes`, `follows_up` per the research
  doc's vocabulary).
- `created_at` / `modified_at` → preserved as document
  metadata.

The harness must:

- Be idempotent: re-ingesting the same corpus on an existing
  DB produces no duplicates.
- Run in <10 minutes on dev-box for Version B.
- Verify after ingest that `chunk_count`, `vector_count`, and
  `relation_count` match the expected values from the corpus
  card.
- Run before the AC-013 / AC-019 perf test suite (CI step;
  the test cases use the resulting DB as the fixture).

### Corpus-Pack 4 — search validation

Verify the three modalities work end-to-end on the ingested
corpus. Each validation is a `cargo test` integration test in
`crates/fathomdb-engine/tests/`:

- **FTS validation**: pick ~20 known terms across source types;
  assert `engine.search(text_only=true, term)` returns docs
  containing the term.
- **Vector validation**: pick ~20 known doc IDs; embed each
  doc body via the existing `VaryingEmbedder` (or whatever
  the locked embedder is); assert that the doc itself appears
  in the top-K KNN result.
- **Graph validation**: pick ~20 chains; for each, assert the
  graph traversal from a chain root returns the expected
  set of related docs.

These are correctness gates, not perf gates. They run at
default `cargo test` scale (small corpus subset) and must
finish in <30 seconds total.

The AC-013 / AC-019 perf gates (Pack 2 RED tests) consume
the same corpus but run only under `AGENT_LONG=1` per the
`dev/notes/ac013-ac019-canonical-scale-policy.md` policy.

## Out of scope

- An ANN graph index (vectorlite / sqlite-vec ANN alpha /
  Rust-side index). Already declined per the locked
  Pack 1/2 plan.
- Embedder change (HITL locked: keep existing).
- A second corpus version (Version A or C). Version B is the
  default for Pack 2 RED tests. Version A may be added later
  as a fast dev-loop subset; Version C is deferred to a
  stress-test slice in 0.8.0+.
- Cross-language / non-English content (ELITR Czech subset
  is filtered out; only the English transcripts are used).
- Re-embedding the corpus under a different model (out of
  scope for 0.7.0; embedder is unchanged).
- A web UI for browsing the corpus (developer-facing only).

## Open items the implementing agent should escalate

- If a source's license has changed since the research doc

  acquisition.
- If the QMSum / ELITR / Imperative To-Do source data shape
  cannot map cleanly to the FathomDB doc schema without
  losing information, escalate before Corpus-Pack 3.
- If the Pack 1 schema migration is not yet landed when this
  work starts, the corpus harness can be authored against
  the locked Pack 1 schema specification — but the ingest
  step must wait on Pack 1 landing.
- If the synthetic chain generator's output exceeds 20 % of
  the corpus by doc count, escalate — the corpus should be
  predominantly real data with synthetic chains layered on,
  not the reverse.
- If you discover a license-blocked source mid-build, fall
  back to the documented alternative (e.g., S2ORC blocked
  → use only PMC OA + OpenAlex; CNN/DM blocked → use NOW
  with access caveats noted).

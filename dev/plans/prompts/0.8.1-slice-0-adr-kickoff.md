### ADR 2 — IR-measure/Eval Design: R0 + R2

Formalize the R0 candidate-recall CDF spec and the R2 end-to-end eval design as decision-ready specs Slices 5 and 25 implement against. Ground in the C1/C2 ceiling corrections from `IR-C-roadmap.md` and Decision ① (AC-077 gate / R2 north-star).

Content required:

**§ R0 spec (Slice 5 target):**

- Measure `found@K` (fraction of eval queries for which the gold document appears in the top-K candidates) for K ∈ {50, 100, 200, 500, 1000} per query class (factoid / exploratory) for: BM25 text arm alone; dense arm alone; oracle union (best of text+dense); fused production arm

- Corpus: frozen `corpus_hash fe973fcd…` (10 sources, 10,506 docs); gold: `data/corpus-data/eval/ir_gold/all.gold.json`

**§ R2 design (Slice 25 target):**

- **Goal:** LongMemEval-style end-to-end memory eval — "does FathomDB (post-R1) retrieve the evidence the answerer needs, as-good-or-better than Mem0-OSS and naive-RAG?"

- **Per-class scoring:** factoid / temporal / multi-hop / knowledge-update / multi-session; abstention counts against recall
- **Metric:** per-class Evidence Recall@K (from `dev/design/ir-recall-measure.md`) + per-class answerer accuracy with abstention
- **Decision ①:** AC-077 (Evidence Recall@K) remains the **product gate**; R2 is **report-only north-star** for the "as-good-or-better than Mem0/Zep" goal; R2 does NOT mint a new gated AC (that decision is HITL at the eval gate after R2 data exists)
- **Why R2 gates R3:** R3's value is mostly off the Recall@K axis (graph benefits temporal/multi-hop/update classes, not factoid) — the only instrument that can see R3's value is R2's per-class breakdown; R3's go/no-go MUST wait for R2 data (C3)
- **Footprint:** the answerer/baseline LLM is test-infra (BYO/local), gated behind a feature/env like the embedder tests; the engine SUT stays no-API
- **Falsifiable bar:** Slice 25 commits the R2 harness + baseline adapter + a results doc with actual per-class numbers; any HITL review can verify the identical-answerer + local-Mem0-OSS conditions

### ADR 3 — Graph Substrate G11 Migration

**File:** `dev/adr/ADR-0.8.1-graph-substrate-g11-migration.md`

Formalize the G11 enrichment migration shape that Slice 15 implements. This ADR **activates** the H3 reservation from `ADR-0.8.0-graph-model-and-edge-addressing.md` (H3 was HITL-signed 2026-06-05 as a prose reservation; this ADR makes it concrete).

Content required:

- **Status:** `ACCEPTED — pending HITL sign-off`
- **What G11 adds:** additive `canonical_edges` columns — these are the fact-on-edge enrichment columns reserved under H3:
  - `body TEXT` — the fact/relationship text (the `body` from the BYO-LLM extract response `edges[].body`); NULL for pre-G11 rows
  - `t_valid TEXT` — event valid-time (ISO-8601 timestamp or NULL); NULL means "still valid" or "unknown"
  - `t_invalid TEXT` — event invalid-time (ISO-8601 timestamp or NULL); NULL means "still valid"
  - `confidence REAL` — calibrated extraction confidence ∈ [0.0, 1.0] or NULL for pre-G11 rows
- **Column name authority:** `t_valid`/`t_invalid` are the canonical FathomDB names (from the retrieval-ADR Option 2A column reservation); the BYO-LLM protocol uses the same names; Graphiti's `valid_at`/`invalid_at` are aliases for the consumer's understanding only — the schema uses `t_valid`/`t_invalid`
- **Schema step:** step-14 (new migration step; appended after the existing step-13 `(collection_name,id)` index); `SCHEMA_VERSION` bumps **13 → 14**
- **Additive/accretion-exempt:** all 4 columns are nullable; pre-G11 rows are NULL-safe throughout; no existing query breaks; step-14 uses `ALTER TABLE canonical_edges ADD COLUMN …` (4 separate ALTERs); no accretion marker needed for additive column ALTERs (per migration policy)
- **Edge projectability:** `body` text is projected into: (a) `search_index` (FTS5) with `source_type='edge_fact'` as the partition discriminant, and (b) `vector_default` (sqlite-vec 1-bit embedding) with the same `source_type` partition; this makes fact-edges semantically searchable (R8 from the graph-model ADR); Slice 15 implements both projections
- **Invalidate-not-accumulate:** when a new BYO-LLM ingest supersedes a prior fact-edge (same `from_id`, `to_id`, `kind`/relation, overlapping temporal scope), the ingest API sets `superseded_at` on the prior row and inserts the new enriched row — it does NOT delete or update the body; the invalidation bookkeeping (checking for existing active edges on the same `(from_id, to_id, kind)` tuple) is Slice 15's responsibility
- **What this gates:** Slice 15 (keystone) implements step-14 + the BYO-LLM ingest surface that writes these columns; Slices 20/30 traverse/query them; the G11 columns are prerequisites for the temporal valid-time filter in traversal (`t_invalid IS NULL OR t_invalid > now`)
- **Falsifiable bar:** Slice 15 tests: columns present in `PRAGMA table_info(canonical_edges)`; legacy rows are NULL-safe (`SELECT * FROM canonical_edges WHERE superseded_at IS NULL` succeeds on a pre-G11 DB after migration); `SCHEMA_VERSION` = 14 after step-14; edge body text is retrievable via FTS + vector search with `source_type='edge_fact'`

**Deferred / explicitly NOT in this ADR or Slice 15:**

- Hybrid `(from_id, to_id, kind)` upsert ergonomics (future write-API ADR, H2)
- Traversable provenance edges + episode tier (H4)
- Reified fact-nodes (H6, escape hatch)
- The G4↔G10 unification (reserved-gap 37, Slice 35)

## TDD discipline (mandatory)

Land tests in RED→GREEN order. Concrete sequence:

### Phase A — RED tests (commit one or more failing tests first)

Write a new integration test file `src/rust/crates/fathomdb-engine/tests/vector_quant_pack1.rs` with the following tests, each of which **must fail** when committed (because the schema and writer changes haven't landed yet):

1. `binary_quant_roundtrip_popcount_384` — open an engine, write a synthetic vector (`v[0..384]=+1.0, v[384..768]=-1.0`, dim 768), query `SELECT vec_to_json(embedding_bin) FROM vector_default WHERE rowid = ?` (or equivalent) and assert popcount = 384. (See design memo D6 for the exact fixture.)
2. `migration_preflight_rejects_unknown_kind` — on a DB seeded with the old schema, manually insert a row into `_fathomdb_vector_rows` with an unknown kind (e.g., `"banana"`), then call the migration path (open the engine). Assert it returns `EngineOpenError::MigrationError { step_id: 9, .. }` (the CHECK-constraint fail). Use `Engine::open_with_migrations` with a controlled MIGRATIONS slice if needed.
3. `writer_double_write_populates_bin_column` — write via `write_vector_for_test` and assert that `SELECT COUNT(*) FROM vector_default WHERE embedding_bin IS NOT NULL` matches the row count after the write.
4. `writer_populates_source_type_partition_key` — write a vector with `kind = "doc"`, assert `SELECT source_type FROM vector_default WHERE rowid = ?` returns `"article"` (D3 coercion).
5. `resolve_source_type_drift_check` — unit test (in `src/lib.rs` `#[cfg(test)] mod`) that the Rust helper agrees with the migration CASE on every known kind (`email/article/paper/meeting/note/todo/doc`).

Commit the RED tests with `[skip]` markers OR run `cargo test ... 2>&1 | head` to confirm they fail as expected. Use `cargo nextest run --no-run` to confirm they compile first.

### Phase B — GREEN: land the implementation

   - `DROP TABLE _fathomdb_vector_migration_v0_7_0`.
   - `DROP TABLE _vec0_migration_assertion`.

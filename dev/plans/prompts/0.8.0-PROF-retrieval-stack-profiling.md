## Why this slice exists

The retrieval path has **budget gates** (`perf_gates.rs` AC-013/AC-019,
`dev/perf-history/`) but no **per-stage decomposition**. AC-013 tells you the
query p50/p99; it does not tell you how much of that is the vec0 bit-KNN
prefilter vs the f32 rerank vs the FTS5 branch vs the merge/dedup vs the FFI
marshalling. The 0.8.0 retrieval work (G9 RRF fusion, G10 metadata-filtered KNN,
G6 retrieve-and-expand) all modify `read_search_in_tx`'s hot path; we need the
per-stage baseline *before* those land so each feature's cost is attributable,
not absorbed into one opaque latency number.

## The retrieval path you are profiling (ground truth, head of main)

`Engine::search` → `search_inner` → embed query → dispatch to reader pool →
`read_search_in_tx` (`src/rust/crates/fathomdb-engine/src/lib.rs`). Profile each
stage as a distinct line item:

1. **Query embed** — `search_inner` embeds the query string to `query_vector`
   (f32) + `query_vector_bin` (binary, possibly mean-centered). One embed call
   on the query; isolate it — for short queries this can rival the SQL.
2. **Reader-pool dispatch** — `ReaderWorkerPool` (8 connections, round-robin
   `dispatch`, `ReaderRequest::Search`), DEFERRED snapshot transaction. Capture
   queue/dispatch overhead vs in-tx work, and lookaside/page-cache hit state
   (`CounterSnapshot.cache_hit/cache_miss`; the Pack 6.G lookaside config).
3. **Vector branch (two phases)** — phase 1: `embedding_bin MATCH
   vec_quantize_binary(...)` bit-KNN prefilter to `TOP_K_BIT_CANDIDATES`
   candidates `ORDER BY distance`; phase 2: `vec_distance_l2(embedding, ...)`
   exact f32 rerank to `final_limit`. **Time the two phases separately** — phase
   1 is the brute-force scan (no ANN; the documented O(N) cost,
   `dev/notes/0.7.0-vector-cost-research.md`), phase 2 is the small rerank. Then
   a per-rowid `canonical_nodes` body fetch.
4. **Text branch (FTS5)** — `search_index MATCH` compiled query
   (`fathomdb-query::CompiledQuery`), ordered by `write_cursor` today (G9 will
   switch to `bm25()`). Isolate FTS5 match cost vs body materialization.
5. **Merge / dedup** — today a scoreless union: vector hits first, then FTS5
   hits, deduped by `body` via a `BTreeSet<String>` (the G9 RRF-fusion target).
   Cheap now; measure it so G9's fusion overhead is attributable.
6. **Soft-fallback probe** — the vector-empty-but-pending signal computed before
   branch collapse.
7. **FFI marshalling** — owned typed rows → Python dataclass / TS object
   (bindings.md §4). G1's `Vec<String> → Vec<SearchHit>` change touches this;
   baseline it from both the Rust path and at least one binding.

### Facts you must account for (do not "discover" them as bugs)

- **vec0 phase-1 is brute-force O(N)** — no ANN index at 0.7.2
  (`dev/notes/0.7.0-vector-cost-research.md`: 1M×768 f32 measured ~2s p50 before
  binary-quant). Recall@10 floor is **0.937 ANN / 0.90 gate** — do not regress
  recall while measuring latency; the two are a pair (`eu8_ir_validation.rs`,
  `eu7_real_corpus_ac.rs`).
- **`final_limit` is production-pinned to `SEARCH_RERANK_LIMIT` (10)**; a test
  seam can RAISE it (`FATHOMDB_PERF_SEARCH_LIMIT` under
  `FATHOMDB_PERF_EXPERIMENTS=1`) but never shrink below production. Vary it as a
  profiling axis only via that gated seam.
- **Tiered budgets** (`ADR-0.7.0-text-query-latency-gates-revised`, HITL-locked
  2026-06-01): 10k binding for 0.x/1.x; 100k/1M post-1.0 ANN work. Profile at
  multiple N (1K / 10K / 100K) so the per-stage scaling curve is visible, but the
  *verdict* anchor stays the real-corpus harness, not synthetic.

## What to build

A profiling harness (prefer `examples/profile_retrieval.rs` for real-corpus runs

- `tests/prof_retrieval_smoke.rs` for the synthetic <30s CI path). It must:

1. **Seed a corpus at growing N** (1K / 10K / 100K bounded by corpus + an
   `AGENT_LONG`/env cap), fully drained so the vector index is complete.
2. **Run a fixed query workload** (reuse the AC-013/AC-019 query set; mix
   vector-heavy, text-heavy, and hybrid queries) and sample p50/p99 **per stage**
   (query-embed, dispatch, vec0 phase-1, vec0 phase-2, body-fetch, FTS5-match,
   FTS5-body, merge, FFI) via the recording `Subscriber` + wall-clock spans.
3. **Decompose the latency budget**: report what fraction of total query
   wall-clock each stage owns, at each N, so the dominant stage is named.
4. **Capture `EXPLAIN QUERY PLAN`** for the vec0 phase-1/phase-2 statements and
   the FTS5 match (proves the brute-force scan; gives the G10 filter a before-shot).
5. **Pair latency with recall**: every latency run also reports recall@10 from
   the same corpus so a future "faster" change that drops recall is caught.
6. **Project the G-work cost** (analysis, not implementation): from the measured
   per-stage costs, estimate where G9 fusion (extra in-memory sort over both
   branches), G10 filtered KNN (metadata predicate in phase-1 WHERE — does it
   prune candidates or just filter post-scan?), and G6 graph-expand (per-hit
   bounded edge walk) will add cost. State assumptions.
7. **Run the reader-PRAGMA / pagecache sweep** as an optional axis; document what
   moved the numbers.
8. **Emit** a `dev/perf-history/`-compatible JSON superset (`phase: "retrieval"`,
   per-stage `*_ms`, `n`, `recall`, `final_limit`, `query_class`, `commit_sha`,
   `timestamp`) + a markdown summary under `dev/notes/perf/`.

## Constraints (hard)

- **Measurement-only.** No `lib.rs` behavior / schema / surface / budget change.
  New code in `examples/`, `tests/`, `dev/notes/perf/`, `dev/perf-history/`. If
  you need an engine seam the `Subscriber` can't reach, **surface it as a HITL
  question in the closure** — do not add it unilaterally.
- **Recall is half the measurement.** Never report a latency number without the
  recall@10 from the same run. Respect the 0.90 gate / 0.937 anchor.
- **Reads go through the reader pool**, not the writer; use the real `search`
  path, not hand-rolled SQL, so the dispatch + snapshot-tx overhead is included.
- **Honesty / determinism**: pin dev-host spec (CPU, cores, `rusqlite::version()`,
  embedder identity); label cache-warm vs cache-cold; mark dev-host-dependent
  numbers. Synthetic vs real-corpus must be labeled (synthetic AC-019 is
  report-only per `ADR-0.7.0-...-revised`).
- **Feature flags**: synthetic-embedder smoke runs WITHOUT `default-embedder`;
  real-corpus profile runs with it; don't gate CI on the real-embedder run.

## 2. Mandate (what this slice delivers — and does not)

Add a **CLI-only, read-only operator diagnostic** that reads back op-store (`operational_mutations`) rows for
one collection through the **existing** engine read seam (`Engine::read_collection` / `read_mutations`,
Slice 30 / G3, index-hardened by Slice 33). It lands as a new **`doctor`** verb (bit-preserving, read-only —
NOT a `recover` action), emits the normative `--json` envelope, and maps outcomes to the stable exit-code
matrix. **This closes the reserved-gap "CLI-only read-back of op-store data (per ADR-0.6.0-cli-scope)
without SDK-parity obligation."**

Concretely (confirm the exact spelling/flags in the §3.0 design memo, then hold to it):
`fathomdb doctor dump-mutations <collection> [--after-id <n>] [--limit <n>] [--json] <db_path>`.

**This is CLI-only.** There is **NO SDK-parity obligation** — do **not** add or change any Python/TS binding
or the SDK `read.*` verbs (they already exist from Slice 30 and are out of this slice). The CLI verb is the
*operator* surface; the SDK `read.*` verbs are the *application* surface; they are deliberately separate.

**Deferred / out of scope (do NOT build here):**

- **No SDK / binding change** (no Py/TS; no Rust `read.*` signature change). CLI-only.
- **No engine / schema change.** You *call* `Engine::read_collection`/`read_mutations`; you do not modify
  them, `commit_batch`, the op-store schema, or the Slice-33 index. If the seam looks insufficient → STOP +
  escalate (§7), do not patch the engine here.
- **No node point-lookup CLI verb** (`read.get`/`read.get_many` over `canonical_nodes`) — only the op-store
  collection/mutations read-back. A `doctor`-side node read-back is a *separate* future gap, not this slice.
- **No `recover` change, no raw-SQL / filter-DSL escape hatch, no `--async`/concurrency knob** (CLI is sync).
- **No new AC id** (`acceptance.md` is locked — see [[acceptance-md-locked-no-feature-acs]]); bind the
  **F4-READ / reserved-gap-34** gap + your test names. Do NOT mint an AC.
- **No expansion of the `fathomdb` facade's public *type* set** unless forced (§3.2 explains why you should
  serialize inline and avoid re-exporting `OpStoreRow`); if you think you must, that is a shared-surface
  ripple → escalate (§6) before doing it.

### 3.0 Design memo FIRST — **and settle the scope call**

Write `dev/design/slice-34-cli-op-store-readback-design.md` **(under `$WT`)** covering:

- **(a) The scope decision — load-bearing.** Argue (against `ADR-0.6.0-cli-scope`) that an op-store
  `operational_mutations` read-back is an **operator diagnostic over operator/log data** (same family as
  `dump-row-counts`/`dump-schema`/`trace`, explicitly blessed at ADR `:39`), **not** the rejected Option-B
  *application* query surface (`search/get/list` over application content in `canonical_nodes`). Conclude
  that it belongs under **`doctor`** (read-only, bit-preserving), is **CLI-only** (no SDK mirror), and is
  pre-authorized by the reserved-gap wording. Record the **ADR amendment** you will land in §3.4 (an
  in-place amendment note dated today, in the `ADR-0.6.0-cli-scope` Status/Consequences, scoping the 0.8.0
  op-store diagnostic read-back IN as a `doctor` diagnostic — *not* re-opening Option B). **If your analysis
  concludes this reads as an application-query expansion rather than a diagnostic dump → STOP and escalate
  (§7); do not absorb a scope expansion silently.**
- **(b) Verb shape:** final name (`dump-mutations` recommended — stays in the `dump-*` diagnostic family and
  avoids appearing to mirror the SDK `read.*` verbs), positional `<collection>` + `<db_path>`, `--after-id`
  (optional `i64`), `--limit` (optional; pick + justify a sane default page size, e.g. 1000; the engine
  clamps to the ~1M cap so you never issue an unbounded read), `--json`.
- **(c) JSON envelope** (doctor pattern, `{ "verb": "dump-mutations", … }`): the `collection`, the effective
  `after_id` (or null), the effective `limit`, a `count`, a `rows` array of
  `{ id, collection, record_key, op_kind, payload, schema_id, write_cursor }` (mirror `OpStoreRow`,
  serde-`snake_case`), and a pagination affordance `next_after_id` (= the last row's `id`, or null on an
  empty/short page). Confirm `--json` is the machine contract; non-`--json` is a human dump only.
- **(d) Exit codes:** success (including an **empty page** — empty/unknown/unregistered collection, or
  `after_id` past the end) → `OK` (0); this is a normal absence, **not** a `Findings` (65). Lock-held
  (`EngineOpenError::DatabaseLocked` / `EngineError::Closing`) → `LOCK_HELD` (71); other engine errors →
  `UNRECOVERABLE` (70). Reuse `run_doctor_verb` + `engine_error_to_outcome` (do not invent a mapping).
- **(e) Why serialize INLINE** (no `OpStoreRow` re-export) to keep the facade public-type set unchanged
  (§3.2) — and the test plan (RED list in §3.1).

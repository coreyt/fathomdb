### 3.2 GREEN — implement the CDF runner

Add the full `IRC_RUN=1`-gated runner to `ir_c_cdf_run.rs`. Structure:

**3.2.1 Corpus + engine setup**

- Use `load_subset_or_skip` (from `corpus_subset::`) to load the frozen corpus.
- Seed the engine (without draining for FTS; freeze the projection scheduler for FTS-only
  runs; drain for the dense arm — follow the `ir_c_gold_diagnostics.rs` pattern).
- Verify `corpus_hash` against `tests/corpus/snapshot.json` (abort on mismatch).

**3.2.2 BM25 text arm (`bm25_text`)**

- Open a read-only SQLite connection to the seeded DB after FTS-only ingest.
- For each positive gold query: retrieve top-K doc IDs using `fts_bodies()` + `map_bodies()`.
  Run at K = 1000 (the deepest), then slice the result for smaller K values.
- This mirrors `ir_c_gold_diagnostics.rs`. Use `compile_content_or()` from `ir_retrieval.rs`.

**3.2.3 Dense arm (`dense`)**

- Requires `--features default-embedder`. Gate with `#[cfg(feature = "default-embedder")]`.
- Seed the engine WITH the projection scheduler running (drain to completion).
- Use `engine.set_search_limit_for_test(1000)` before queries.
- Use `engine.set_vector_stage_only_for_test(true)` to get dense-only results.

**3.2.4 RRF-fused arm (`rrf_fused`)**

- Use `engine.set_search_limit_for_test(1000)`.
- Use `engine.search(query)` (RRF-fused production path).

**3.2.5 Oracle union arm (`oracle_union`)**

- A doc is found@K in oracle if it appears in the union of bm25_text top-K AND dense top-K.
- Compute from the two arms' ranked lists: a doc is "found" if it's at rank ≤ K in either list.

**3.2.6 `found@K` aggregation**

- For each (arm, query_class, K): count queries where the gold doc appears in top-K; divide by
  total positive queries in that class with a gold doc present in the corpus.
- Multiple gold docs per query: the gold "required" set (use `required_doc_ids(q)` from
  `ir_eval.rs`). A query is "found@K" if ANY required gold doc appears in top-K.

**3.2.7 Output artifact**

- Write `dev/plans/runs/IR-C-recall-cdf.json` per ADR §2.3 schema.
  - `corpus_hash`: full hash from snapshot.
  - `gold_set_path`: `"data/corpus-data/eval/ir_gold/all.gold.json"`.
  - `generated_at`: ISO-8601 (use `std::time` or system call).
  - `recall_cdf`: 40+ rows, one per (query_class, arm, k). Use gold set labels: `"exact_fact"`
    and `"exploratory"`.
  - `latency`: placeholder or actual (see §3.3).
- Write to the canonical path at the END: `$CANON/dev/plans/runs/IR-C-recall-cdf.json`.
  (This is the same deliberate exception as `output.json` — the artifact is the orchestrator's
  deliverable, not a per-worktree intermediate.)

### 3.3 CE latency benchmark

Write `dev/scripts/ir_c_ce_latency.py`. This script:

- Loads TinyBERT-L-2 and MiniLM-L6 via FlashRank (`pip install flashrank`) or
  sentence-transformers (`pip install sentence-transformers`). These are standard
  cross-encoder models; use the smallest available bundle.
- Samples ≥1,000 random (query, passage) pairs from the gold set + corpus.
- Times the score call (ms/pair) over the sample; computes p50 and p95.
- Reads the existing `IR-C-recall-cdf.json` artifact, patches in the `latency` array (two
  entries, one per model), and writes it back.

The latency section in the artifact should be a **list** of two objects:

```json

    "model": "TinyBERT-L-2",
    "ms_per_pair_p50": ...,

    "n_pairs_sampled": 1000,
    "hardware_note": "<cpu model>"
  },
  {
    "model": "MiniLM-L6",
    ...
  }
]
```

Document the script usage in `IR-C-r0-findings.md`.

## Why this slice exists

Slice 31 settled active-row identity at **`logical_id` alone, uniformly on `canonical_nodes`
AND `canonical_edges`** (HITL-SIGNED 2026-06-05; substrate ADR Decision 5
`dev/adr/ADR-0.8.0-canonical-identity-substrate.md:188-231`; codex consult
`dev/plans/runs/0.8.0-identity-scope-codex-consult-20260605T021802Z.md`). But that decision was
reasoned through the **point-lookup / node-identity lens** — `read.get` determinism, the
identity-fork bug, G8 — treating edges as opaque write-receipts. FathomDB is *also* a typed
graph: `canonical_edges` are typed relationships, v0.5.x shipped a real nodes/edges/traversal
graph, and G4 `read.list` / G5 `read.neighbors` / G6 search-expand / G7 `read.history` are
deferred to 0.8.x but their substrate (edges + `from_id`/`to_id` indexes) is being laid now.

**The `logical_id`-alone identity question is ONE FACET of a broader, currently UNDER-SPECIFIED
graph design. This slice resolves the broader question: FathomDB's intended graph model.** No
FathomDB doc has ever decided how edges should be *addressed* for re-assertion/supersession; the
graph verbs are deferred; and no named 0.8.0 consumer forces the issue. This slice must settle
that intended model — or escalate it as an explicit HITL product/design decision — so the
deferred G4–G7 work builds on a *decided* foundation, not an asserted one.

> **Sharpening (verified):** edge *identity* is already decided. The retrieval ADR
> (`ADR-0.8.0-agent-memory-retrieval-and-identity.md:189-191`, Option 2A) directed the substrate to
> settle whether edges carry identity + temporal columns, and Decision 5 answered "yes, uniformly,
> by `logical_id`." So this slice is **not** re-litigating whether edges have identity — only how
> they are **addressed** for upsert/supersession. Keep identity (settled) separate from addressing
> (the open crux).

### The disentanglement you MUST make explicit (do not conflate these)

- **(a) Multigraph representability** — "can two *active* typed edges (`kind`) coexist between
  the same `(from, to)` pair?" **This is ALREADY YES** under `logical_id`-alone: each typed edge
  carries its own opaque `logical_id`. **Do NOT frame the mission as "should the model allow a
  multigraph?" — it already does.** State this up front and move on.
- **(b) Natural-key addressing / upsert ergonomics** — a central question. To re-assert or supersede
  "*the `friend` edge from A to B*," must a consumer **mint and durably track an opaque
  `logical_id`**, or should the engine let them **address/upsert that edge by its natural
  `(from, to, kind)` tuple**? This is a UX + entity-resolution question, not a representability one.

### The MODEL-SHAPE question (Q0 — decide this too; it is the broader half of "intended graph model")

A codex high-effort review (`dev/plans/runs/0.8.0-slice-32-codex-prompt-review-20260605T130928Z.md`)
flagged that addressing alone is too narrow for the title. The deeper decision is **how FathomDB reifies
a "fact" / relationship** — and a doc-verification pass (STATUS §7, 2026-06-05) found it is a genuine
**THREE-way** choice, NOT the two-way "binary vs fact-as-node" codex first framed:

1. **Binary edges (status quo).** `canonical_edges(from_id, to_id, kind, source_id, logical_id,
   superseded_at)` — typed directed edges, no body/confidence/embedding/valid-time.
2. **Temporal fact-EDGES (Graphiti / Zep — the direction FathomDB's OWN docs already point at).** Facts
   *are* edges, enriched with valid-time (+ confidence/body). The retrieval ADR's recommended **Option
   2A** reserves exactly this bi-temporal column shape additively (`retrieval-ADR:135-159`: "facts/edges
   carry four timestamps … set the superseded **edge's** `t_invalid`"); `agent-memory-fit:360` cites
   Zep/Graphiti's "bi-temporal entity graph; **edge** invalidation." **Note: the terms "reified" /
   "fact node" appear NOWHERE in the docs; the docs model facts as EDGES.**
3. **Reified fact-NODES (GraphRAG / the research-note tradition).** Facts are `canonical_nodes(kind=
   "fact")` carrying text/confidence/valid-time/source/embedding/n-ary roles, with `SUBJECT`/`OBJECT`/
   `ROLE`/provenance edges. Representable on the current substrate **today** without schema changes —
   so NOT a hard substrate blocker — but it is a larger ontology/ingestion departure that the docs do
   **not** currently endorse.

**Do not let the slice quietly inherit codex's fact-as-node lean.** FathomDB's docs + v0.5.x lean toward
(2) fact-as-edge; the research note argues (3) fact-as-node; (1) is the status quo. The slice must weigh
all three coherently — and reconcile the model-shape choice with the addressing choice (they interact:
under fact-as-edge, addressing is about the edge tuple/id; under fact-as-node, it is about node identity +
provenance edges). It must also separate **graph-traversable provenance edges**
(`SUPPORTS`/`OBSERVED_IN`/`DERIVED_FROM`) from the **scalar `source_id`** provenance that exists today
(`agent-memory-fit:73`).

0. **Model shape (decide this FIRST) — a THREE-way choice (see "Why this slice exists").** Resolve which
   reification FathomDB intends: **(i) binary edges** (status quo); **(ii) temporal fact-EDGES** (Graphiti/
   Zep — facts are edges enriched with valid-time/confidence; the direction the docs' Option 2A already
   reserves — `retrieval-ADR:135-159`, `agent-memory-fit:360`); or **(iii) reified fact-NODES** (GraphRAG/
   research-note — `canonical_nodes(kind="fact")` + subject/object/role/provenance edges). For the chosen
   shape, state whether each piece is a 0.8.x **substrate requirement**, a 0.8.x **ontology
   recommendation**, or **later ingestion-layer work**, and confirm what is representable on the **current
   substrate without schema changes** (codex: fact-as-node is — so it is NOT a hard substrate blocker; the
   live question is whether to *adopt + document* it vs the doc-aligned fact-as-edge path). Do **not**
   default to fact-as-node: the docs + v0.5.x lean fact-as-edge; weigh all three. Evaluate
   **graph-traversable provenance edges** (`SUPPORTS`/`OBSERVED_IN`/`DERIVED_FROM` — the literature's
   highest-recall edges) **separately from domain-relation edges** and from the **scalar `source_id`**
   provenance that exists today.

1. **The crux — natural-key addressing vs opaque-id addressing.** Today, to supersede or
   re-assert a specific typed relationship, a consumer must supply the SAME opaque edge
   `logical_id` it minted before (`PreparedWrite::Edge`, supersede UPDATE
   `src/rust/crates/fathomdb-engine/src/lib.rs:6036` `WHERE logical_id = ?2 AND superseded_at IS
   NULL`; bindings `fathomdb-py/src/lib.rs:873-882`, `fathomdb-napi/src/lib.rs:1075-1081`).
   Evaluate the three candidate models and pick one (or recommend HITL chooses):
   - **Opaque-`logical_id`-only** (status quo): caller owns identity bookkeeping for every edge.
   - **Natural-`(from,to,kind)` addressing**: engine derives/dedups the active edge from the
     natural tuple; caller need not track an id.
   - **Hybrid**: engine derives/dedups on `(from,to,kind)` for upsert ergonomics but still
     STORES a `logical_id` (caller-supplied or engine-minted) as the durable handle.
   For each: what does it cost the deferred graph verbs (G4–G7), and what must the substrate
   guarantee NOW (while graph is deferred) to keep that option open without a second migration?

2. **Test Decision 5's assertion — do NOT accept it.** Substrate ADR Decision 5
   (`dev/adr/ADR-0.8.0-canonical-identity-substrate.md:223-227`) asserts "edge `kind` buys no
   real edge capability; multiple active relationship facts use distinct `logical_id`s." That was
   **asserted, not derived from a graph-model analysis.** Scrutinize it against the two primary
   evidence sources below (Q3, Q4). Does it hold for the graph aspect, or does it conflate
   *identity* (correctly `logical_id`-alone) with *addressing* (the open (b) question)?

3. **Bitemporal world-model — does supersede-by-`logical_id` model "the relationship changed"?**
   Read `dev/adr/ADR-0.8.0-agent-memory-retrieval-and-identity.md:126-196` closely: the
   Zep/Graphiti reference puts **valid-time on FACT EDGES** and handles contradiction by
   **invalidating, not deleting** (set the superseded edge's `t_invalid` to the new edge's
   `t_valid`). Under each candidate model in Q1, can "the `friend` edge from A to B ended /
   changed" be expressed as a clean edge supersession? Does opaque-id addressing force the
   consumer to *find* the prior edge's id before it can invalidate it — and is that the right
   ergonomics for a bitemporal graph? Note the substrate already reserves the bitemporal shape
   (retrieval ADR:177-196, Option 2A) — your verdict must not foreclose valid-time on edges.

4. **v0.5.x precedent — confront it directly (verified, cite it).** v0.5.6 keyed edge active
   uniqueness on **`logical_id` ALONE** (`git show v0.5.6:crates/fathomdb-schema/src/bootstrap.rs`
   → `idx_edges_active_logical_id ON edges(logical_id) WHERE superseded_at IS NULL`; supersede
   `git show v0.5.6:crates/fathomdb-engine/src/writer/mod.rs` → `UPDATE edges SET superseded_at =
   unixepoch() WHERE logical_id = ?1 AND superseded_at IS NULL`, **no `kind`**) **YET** indexed
   **`idx_edges_source_active ON edges(source_logical_id, kind, superseded_at)`** (+ target
   mirror) for **kind-scoped typed-neighbor traversal**, with `kind` as a payload column. This is
   strong evidence you can have BOTH a multigraph AND typed traversal under `logical_id`-alone
   identity, with `kind` never part of identity. Verify this from git, then weigh it: does it
   *settle* identity (yes, `logical_id`-alone) while leaving *addressing* (b) genuinely open?
   (Cross-check `dev/profiling/v05-lineage.md:14-48`,
   `dev/design/0.8.0-v05-feature-triage.md:200-228`.)

5. **What the deferred graph verbs imply.** G5 `neighbors(id, edge_type?, depth)` semantics
   (`dev/design/agent-memory-impl-strategy.md:326-330`: recursive CTE on `from_id`/`to_id`,
   depth/neighbor clamps) and G4/G6/G7 (`dev/design/0.8.0-agent-memory-fit.md:218-228`,
   `:155-189`, `:318-327`). Does `kind`-as-payload (current schema
   `src/rust/crates/fathomdb-schema/src/lib.rs:111,297-314`) already serve `edge_type`-filtered
   traversal the way v0.5.6's `(source_logical_id, kind, superseded_at)` index did? What, if
   anything, must the substrate add NOW so the deferred verbs aren't boxed in?

6. **Property-graph norms (external anchor — web research permitted).** Reason from established
   practice (Neo4j-class property graphs): each relationship has its own identity AND a type;
   multiple typed relationships between two nodes are normal; relationships are typically created
   by `(start, type, end)` and MERGE-ed on that natural pattern. Anchor the (a)/(b) verdict in
   this norm, not only first principles. Note where FathomDB intentionally diverges and why.

7. **Named-consumer reality.** Confirm the graph verbs are not a 0.8.0 blocker for any named
   consumer (OpenClaw graph = "none/markdown", hybrid-search + dedup, and the docs explicitly warn
   "Do NOT let OpenClaw create sequencing pressure for graph features"; Hermes = LLM, not a KG;
   Memex = has edges but no documented multi-typed-edge-per-pair *addressing* need) —
   `dev/design/0.8.0-agent-memory-fit.md:465-503`, `:343+`,
   `dev/design/0.8.0-v05-feature-triage.md:200-228`. If a consumer DOES need natural-key

## Why this slice exists

Slice 31 re-scoped active-row uniqueness to **`logical_id` alone on both `canonical_nodes` AND
`canonical_edges`** (HITL-SIGNED 2026-06-05; rationale:
`dev/plans/runs/0.8.0-identity-scope-codex-consult-20260605T021802Z.md`, memory
[[g0-identity-scope-logical-id-alone]]). That decision was reasoned primarily through the **point-lookup
/ node-identity lens** (read.get determinism, the identity-fork bug, G8). The independent codex consult
concluded the node/edge answer is *uniform* — **but it evaluated edges as opaque write-receipts, not as a
graph.** FathomDB also has a **graph / graph-database aspect** (canonical_edges as typed relationships,
the v0.5.x nodes/edges/traversal lineage, the deferred graph read verbs G4/G5/G7, G8 dangling-edges). The
GRAPH lens is exactly where "one `logical_id` with multiple active `kind`s" could be a *legitimate*
multigraph capability (multiple relationship types between the same endpoint pair) rather than a fork bug.
**This slice evaluates whether `logical_id`-alone holds up for the graph aspect, or whether the graph lens
reveals a need to revisit EDGE identity** (e.g. a `(from, to, kind)` natural key, or a graph-native edge
identity distinct from node identity) — and records the verdict so the deferred graph roadmap is
reference-driven, not assumed.

**This slice does NOT block Slice 31** (which is SIGNED and proceeds as uniform `logical_id`-alone). It is
a forward-looking evaluation: it either **confirms** uniform `logical_id`-alone is right for graphs, or it
**surfaces a graph-specific edge-identity concern** as a future ADR / reserved-gap / roadmap item (and, if
it finds a genuine defect in the *signed* edge decision, escalates to HITL rather than silently
contradicting it).

## Mandate (evaluation questions — the deliverable answers these with evidence)

Produce a design-ADR / evaluation memo (`dev/adr/ADR-0.8.0-graph-edge-identity-eval.md` or a
`dev/design/` memo — orchestrator picks at finalize) that answers, grounded in code + the v0.5.x lineage +
the consumer agents:

1. **Multigraph semantics.** Is `logical_id`-alone correct for edges when FathomDB is used as a graph?
   Specifically: should the model allow multiple *active* relationship types (`kind`) between the same
   `(from, to)` pair simultaneously (a multigraph), and if so, how is each such edge identified?
   - Under Slice 31, two active edges between the same pair require two distinct `logical_id`s (kind is not
     identity). Is that ergonomic + correct for consumers, or is `(from, to, kind)` the natural edge key?
   - What does the writer actually require of callers for edge `logical_id` today
     (`PreparedWrite::Edge`, `fathomdb-engine/src/lib.rs` ~`:1017`; Py ~`:873`, napi ~`:1075`)? Is
     caller-supplied opaque `logical_id` a good graph UX, or does it push identity bookkeeping onto
     consumers?
2. **Deferred graph read verbs.** How does `logical_id`-alone serve / constrain the deferred
   **G4 `read.list`**, **G5 `read.neighbors` (traversal, depth<=3)**, **G7 `read.history`**, and edge
   supersession-as-relationship-change? Does it enable clean neighbor/traversal queries, or force awkward
   joins? (See `dev/design/0.8.0-agent-memory-fit.md`, the retrieval ADR
   `dev/adr/ADR-0.8.0-agent-memory-retrieval-and-identity.md`.)
3. **v0.5.x graph lineage.** v0.5.x shipped a real nodes/edges/traversal graph
   ([[fathomdb-v05-graph-lineage]]) and scoped edge active-uniqueness on `logical_id` alone. How did
   v0.5.x identify edges (natural key vs opaque id)? Did it support multigraph edges? Does its precedent
   confirm or complicate the Slice 31 uniform decision for the graph use case? (Use v0.5.x via git.)
4. **Consumer graph needs.** Do the real consumer agents (Memex / Hermes / OpenClaw —
   [[fathomdb-consumer-agents]]) need multigraph / typed-relationship modeling, or is single-edge-per-id
   sufficient? Ground the verdict in an actual need, not a hypothetical.
5. **Temporal/graph consistency.** Edge supersession by `logical_id` — does it correctly model "this
   relationship changed / ended" in a graph (vs. node supersession)? Interactions with G8 dangling-edges
   and the deferred bi-temporal model (parent ADR Q2). Note the parent-ADR↔code drift Slice 31 is fixing
   (Q4 "edge supersession is schema-only/nodes-only" vs. code that supersedes edges).

**Verdict required:** one of — (a) **uniform `logical_id`-alone is correct for the graph aspect too**
(confirm, with the graph reasoning the original consult lacked); (b) **edges need a graph-native identity
refinement** (e.g. `(from,to,kind)` natural key or a distinct edge-identity model) → specify it as a
future ADR/reserved-gap/roadmap item, NOT a silent change to the signed Slice 31; (c) **a genuine defect
in the signed edge decision** → escalate to HITL with the evidence. Plus: a short note on what the deferred
graph verbs (G4/G5/G7) should assume about edge identity.

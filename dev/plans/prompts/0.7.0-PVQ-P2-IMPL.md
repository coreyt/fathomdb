## Suggested SQL shape (handoff § 2.2)

```sql
-- Phase 1: bit-KNN to top-K=64 candidates
WITH candidates AS (
  SELECT rowid, distance AS bit_distance
  FROM vector_default
  WHERE embedding_bin MATCH vec_quantize_binary(vec_f32(?1))
  ORDER BY distance
  LIMIT 64
)
-- Phase 2: f32 rerank on the candidate set
SELECT c.rowid
FROM candidates c
JOIN vector_default v USING (rowid)
ORDER BY vec_distance_l2(v.embedding, vec_f32(?1))
LIMIT 10
```

**Verify against sqlite-vec 0.1.7 syntax before committing.** The
exact MATCH-clause form for `bit[N]` may differ; if so, fall back to
the form documented in the guide and update the SQL accordingly.
The `K=64` value is a constant — make it a named `const TOP_K_BIT_CANDIDATES: usize = 64;` near the function so post-0.7.0 tuning is cheap.

## Distance function: L2 vs cosine

The existing f32 path uses `vec_f32(?)` + the implicit MATCH distance
(L2 by default in sqlite-vec). Use `vec_distance_l2` in the rerank
unless the existing path uses cosine — confirm by reading
`tests/vector_quant_pack1.rs` and any other rank-order-sensitive test
to ensure the distance function matches what callers expect. Recall
test will catch mismatches.

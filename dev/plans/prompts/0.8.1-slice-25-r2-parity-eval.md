### 2b. R2 eval harness

Build a **LongMemEval-style end-to-end eval harness** in Python at
`$WT/src/python/eval/r2_parity_eval.py` (or a small package under `eval/`).

The harness implements the **identical-answerer protocol** (ADR §3.2):

- The same LLM / same prompt / same context-window is used to answer questions retrieved by
  each system (FathomDB post-R1, Mem0-OSS, naive-RAG).
- Only the retrieval+memory layer differs between the three systems.
- **This constraint is non-negotiable.** Any deviation from identical answerer is NOT R2.

The answerer/LLM is **test-infra** (BYO/local — gated behind an env var or `R2_RUN=1`-style
flag, exactly like the `AGENT_LONG` / `IRC_RUN` gates on existing runners). The engine SUT
(FathomDB) stays **no-API throughout the eval** — no network call from FathomDB itself.

### 2c. Baselines

Two baselines, both under the identical answerer:

1. **Mem0-OSS adapter** — `pip install mem0ai` (local library, **NOT** the Mem0 cloud API; cloud
   API is a footprint violation and must not be used). Wraps Mem0-OSS with the same
   retrieve/answer interface as FathomDB. Ingest the same corpus (same documents, same ordering).
2. **Naive-RAG adapter** — flat BM25 or dense retrieval over the same documents, no
   memory-graph or update logic. This is the floor baseline.

### 2d. Per-class scoring (5 classes + abstention)

Score results stratified by query class (ADR §3.4). At minimum:

| Class | What it tests |
|-------|--------------|
| `factoid` | Specific retrievable fact (number, name, date) |
| `temporal` | Time-aware questions ("what did we know as of March") |
| `multi_hop` | Questions requiring 2+ connected facts |
| `knowledge_update` | Questions about facts that have changed / been superseded |
| `multi_session` | Questions spanning multiple conversation sessions |

**Abstention is scored.** A system that returns no answer when one exists is a miss. A system
that answers when no answer exists is a false positive. Both appear in the results table.

Per-class metrics to report:

- **Per-class Evidence Recall@K** (from `dev/design/ir-recall-measure.md`) — primary metric
- **Per-class answerer accuracy** — end-to-end score (retrieval + answerer joint accuracy)
- **Abstention rate** per class — separately reported

### 2e. Results document

Write `$WT/dev/plans/runs/IR-C-r2-eval-results.md` with:

- The actual per-class numbers for all three systems (FathomDB, Mem0-OSS, naive-RAG)
- N (number of queries per class)
- The answerer model name + version (cited, not left blank)
- The corpus snapshot hash (`fe973fcd…`)
- A delta table: FathomDB minus Mem0-OSS per class (this is the **go/no-go input for Slice 30**)

The Slice 30 go/no-go requires the delta table to show whether the R3 graph arm lifted
`temporal`, `multi_hop`, and `knowledge_update` without regressing `factoid`. The column names
in the delta table MUST match these class names exactly so the Slice 30 orchestrator gate can
read them.

and all `tdd_evidence` keys (see §9).

## 2. Mandate (what this slice delivers — and does not)

Harden the **op-store paginated read-back** (`read.collection` / `read.mutations`, Slice 30 / G3) so that
paginating a single collection within a **genuine ~1M-row `operational_mutations` log** stays **bounded
and index-driven**, and the clamp + `after_id` cursor are robust at the edges. **The SDK read-API
signatures do NOT change** — this is an internal correctness/perf hardening.

The likely core change (confirm by EXPLAIN in §3.0): `operational_mutations` currently has **no index
beyond its `id` PRIMARY KEY** (`schema:lib.rs:127` CREATE TABLE; `migrations/004_op_store.sql`). So
`WHERE collection_name = ?1 ORDER BY id LIMIT n` (the read path) **scans the id-ordered log filtering
`collection_name`** — O(rows scanned), not O(page), for a small collection inside a large multi-collection
log. The hardening adds a **forward-only additive index** `operational_mutations(collection_name, id)`
(a new **step-13** migration, `SCHEMA_VERSION 12→13`) so the query is **index-driven (EXPLAIN: no SCAN,
uses the index, no temp B-tree for ORDER BY)** — plus clamp/cursor edge-case robustness.

**Deferred / out of scope:** no change to the `read.*` SDK verb signatures / bindings (Py/TS/Rust); no
change to `commit_batch` or the write path; no change to the search path (the Slice 10 byte-identity pin
must stay green); no new AC id (bind the G3/F4-READ gap + test names); no touching the byte-frozen recovery
suites; no release/CI/version actions. **Do NOT reshape `operational_mutations`** — additive index only.

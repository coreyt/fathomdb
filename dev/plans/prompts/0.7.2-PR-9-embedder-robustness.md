### Item 1 — concurrent-embed safety (the exposure is real, safety unproven)

- **`PROJECTION_WORKERS` is a hardcoded `const`, NOT an env knob.**
  `fathomdb-engine/src/lib.rs:66` → `const PROJECTION_WORKERS: usize = 2;`
  (The handoff/board offhand-mention it as if a tunable `PROJECTION_WORKERS=2`
  environment knob — that is wrong; it is a compile-time const. Treat the handoff
  wording as a doc nit, not a contract.)
- Workers spawned at `lib.rs:551-554` (`for _ in 0..PROJECTION_WORKERS`); each calls
  `embedder.embed(&job.body)` on **one shared** `Arc<dyn Embedder>`
  (`ProjectionRuntimeShared.embedder`, field at `lib.rs:156`; call site `lib.rs:3391`).
  **No production serialization** around the embedder.
- `CandleBgeEmbedder::embed(&self)` (`fathomdb-embedder/src/candle_bge.rs:155-197`)
  calls `self.model.forward(...)` (`:176`) on a shared `BertModel` with **no interior
  locking** → the concurrency exposure is real and unproven-safe.
- The ONLY place embeds are serialized today is the **test harness**: `SerializedBge`
  (a `Mutex` wrapper) at `tests/eu7_real_corpus_ac.rs:101-121`, used at `:667`.
  Production has no equivalent.
- EU-5f's `catch_unwind` (`lib.rs:3311-3317`) catches **panics only** — a *hung*
  (non-panicking) embed still parks a worker forever. That is item 2's territory.

### Item 2 — embed watchdog (plumbing exists; the trigger does not)

- `EmbedderError::Timeout` already exists: `fathomdb-embedder-api/src/lib.rs:20`.
- The projection path **already handles** `RuntimeEmbedderError::Timeout` by retrying:
  `lib.rs:3393-3396` (`Err(RuntimeEmbedderError::Timeout) => { last_code = "EmbedderError"; continue; }`).
  **But nothing ever produces a `Timeout`** — there is no watchdog. So item 2 is:
  add the watchdog that actually *emits* `Timeout` on the projection path; the
  downstream retry/failure plumbing (`lib.rs:3393`, and `:4573` in the error map) is
  already in place.
- **No hanging/parking test double exists.** Existing doubles and where they live:
  `VaryingEmbedder` (`batch_write_per_row_cursor.rs:22`, `perf_gates.rs:327`),
  `SleepingEmbedder` (`projection_runtime.rs:51` — *delays but returns `Ok`*, does
  NOT hang), `PanickingBgeEmbedder` (`eu5f_production_pin.rs:80`),
  `SimulatedBgeEmbedder` (`eu5f_production_pin.rs:37`), `DeterministicEmbedder`,
  `StubEmbedder`, `FixedEmbedder`. The watchdog RED test needs a **new** parking/
  hanging embedder double (blocks indefinitely, e.g. on a channel or long sleep ≫
  the configured timeout).

## Design tension to pre-load (decide explicitly in the slice)

**The two items interact.** A per-embed watchdog typically needs the embed to run
where the caller can time it out (separate thread / cancellable task). But if item 1
is solved by **serializing embeds engine-side** (a `Mutex` around the shared
embedder), a hung embed must **not** be allowed to hold the serialization lock
forever — otherwise the watchdog times out the call but the worker pool still
deadlocks on the poisoned/held lock. So:

- If the pre-flight says item 1 = "just add the regression test" (no engine
  serialization), the watchdog is structurally simpler — design it independently.
- If item 1 introduces engine-side serialization, the watchdog design MUST account
  for lock release on timeout (drop the guard before/at timeout, treat a timed-out
  embed's worker as recoverable, no poisoned `Mutex` wedge).

State which case you are in and how the watchdog interacts with it, in both the diff
and the closure artifact.

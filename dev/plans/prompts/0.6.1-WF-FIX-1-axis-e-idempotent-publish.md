## Mandate

The BUMP slice (af96a5b on 2026-05-24) bumped axis-W 0.6.0 → 0.6.1
while holding axis-E (`fathomdb-embedder-api`) at 0.6.0. This is the
first post-GA axis-W-only patch release.

`release.yml` T1 (publish-rust-t1-embedder-api) unconditionally runs
`cargo publish -p fathomdb-embedder-api`. Because crates.io already
has `fathomdb-embedder-api@0.6.0` (published in the 0.6.0 GA train),
this `cargo publish` will FAIL with "version already exists". T2-T7
depend on T1 via `needs:`, so the entire publish pipeline blocks
at the first tier on every axis-W-only patch.

**Fix:** add an idempotency guard to every publish tier (T1-T7).
Before running `cargo publish -p <crate>`, query crates.io for the
already-published version; if it equals the local Cargo.toml
version for that crate, **skip the publish step** with a clear log
line. Otherwise publish normally.

Reasons to generalize across all tiers (not just T1):
- Forward-compat for future axis-W-held releases (any patch where
  any crate isn't bumped — currently only axis-E, but the
  pattern matters).
- Defensive for partial-republish retries: if RC1 publishes T1-T3
  cleanly but fails at T4, a follow-up RC2 with the same tags
  would re-attempt T1-T3 and now fail on duplicate-version.

## Tasks (TDD red → green → refactor)

1. **Decide abstraction shape.** Pick ONE:
   - **(a) Inline `if/else` in each tier.** Easiest; ~5 lines per
     tier × 7 tiers = duplication but self-contained.
   - **(b) Helper bash script** `scripts/release/cargo-publish-if-new.sh`
     that takes `<crate>` arg + reads `CARGO_REGISTRY_TOKEN` env;
     each tier calls the helper. Less duplication, easier to
     unit-test (TDD shape).
   - **(c) Composite GitHub Action** in `.github/actions/cargo-publish-if-new/`.
     Most idiomatic for repeated workflow steps but heavier
     surface (new directory + action.yml + the body script).
   Prefer (b) for testability — the helper can have a
   `scripts/tests/test_cargo_publish_if_new.sh` that mocks
   `cargo search` and `cargo publish` to verify the skip/publish
   branching. Surface the choice in the output JSON.

2. **TDD RED.** Author the test (or set of tests) first:
   - Helper script invoked with a crate at a version already on
     crates.io → exit 0, stdout contains "already published;
     skipping".
   - Helper script invoked with a crate at a version NOT on
     crates.io → invokes cargo publish (mocked); exit code
     matches mock.
   - Helper script invoked with `--dry-run` mode forwards to
     `cargo publish --dry-run` regardless of registry state.
   - Run the new tests; they must fail before the helper exists.

3. **TDD GREEN.** Implement the helper script (if shape (b)) or
   inline guards (if shape (a)/(c)). Read the local crate version
   from Cargo.toml via `cargo metadata --format-version 1 --no-deps
   | jq -r '.packages[] | select(.name == "$CRATE") | .version'`
   OR via the `set-version.sh` reader pattern. Query crates.io
   via `cargo search <crate> --limit 1` and parse the version.
   Compare; skip or publish accordingly.

4. **Update release.yml.** Wire each T1-T7 tier through the new
   helper (or inline guard). The `if: ${{ inputs.dry_run != true }}`
   sleep-60s step should remain unchanged (it gates on dispatch
   input, not on whether a publish actually happened — slightly
   suboptimal for the skip-case but not incorrect; document in
   the commit body if you want to defer that polish).

5. **TDD GREEN.** New tests pass.

6. **Refactor.** Single-pass cleanup; no leftover dead code.

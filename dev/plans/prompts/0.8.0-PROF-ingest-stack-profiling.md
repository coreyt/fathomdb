# PROF-INGEST ‚Äî Ingest-path stack profiler

You are an informed implementer for ONE research/instrumentation slice of the
FathomDB 0.8.0 agent-memory line (`/home/coreyt/projects/fathomdb`, head of
`main` = 0.7.2). You have **none of the prior session's context** ‚Äî everything
you need is below or linked. This slice is **measurement-only**: it adds a
profiling harness + a closure report. It must **not** change engine behavior,
the five-verb SDK surface (AC-057a), the schema (`SCHEMA_VERSION` stays 10), or

## Why this slice exists

The 0.8.0 agent-memory work (`dev/design/0.8.0-agent-memory-fit.md` gap ladder
G0‚ÄìG12; build plan `dev/design/agent-memory-impl-strategy.md`) layers record
identity, filtered KNN, fusion, and graph traversal onto the existing
SQLite + FTS5 + sqlite-vec stack. Before any of that lands we need a **truthful,
per-layer cost picture of the *ingest* path as a real corpus streams in** ‚Äî so
the G-work is sequenced against measured bottlenecks, not guesses. The retrieval
side is already instrumented (`perf_gates.rs` AC-013/019, `dev/perf-history/`);
**the write/ingest side is not.** This slice closes that gap.

The deliverable is a profiler that answers, as N grows during ingestion:
where does wall-clock go, per layer, per batch, and how does each layer's cost
scale?

## The stack you are profiling (ground truth, head of main)

Ingest flows through these layers; profile each as a distinct line item.

1. **Canonical write** ‚Äî `Engine::write` ‚Üí `write_inner` ‚Üí `commit_batch`
   (`src/rust/crates/fathomdb-engine/src/lib.rs`): single writer thread, one
   SQLite transaction per batch, per-row `write_cursor`, INSERT into
   `canonical_nodes` / `canonical_edges` / op-store tables.
2. **FTS5 projection** ‚Äî `search_index` (fts5 virtual table, migration 5). FTS5
   does segment merges every ~64 inserts (level-based B-trees); ingest cost is
   bursty, not linear ‚Äî capture the merge spikes.
3. **Vector projection** ‚Äî async projection runtime (`projection_dispatcher_loop`
   / `projection_worker_loop`, `PROJECTION_WORKERS = 2`, `embed_serialize`
   mutex, `embed_with_watchdog`): embed body ‚Üí `vector_default` vec0 table,
   storing f32 `embedding` + binary-quant `embedding_bin` (+ `source_type` /
   `kind` / `created_at` metadata cols). The embed call dominates; isolate
   embed-time vs vec0-insert-time vs binary-quant-time.
4. **Op-store** ‚Äî `operational_state` (upsert) / `operational_mutations`
   (append) with save-time JSON-schema validation.
5. **Graph edges** ‚Äî `canonical_edges` (rewrite-era table, migration `002`) today
   has opaque-string endpoints and no `from_id`/`to_id` index (G5/G8 will add
   them). The 0.8.0 graph work (G5 `neighbors()`) builds on **`canonical_edges`**
   via a `WITH RECURSIVE` walk ‚Äî net-new code that revives a capability the real
   v0.5.x `nodes`/`edges` graph layer already had (`traverse()`/`expand()`,
   dangling-edge detection, `restore_validated_edges`) before the 0.6.0 rewrite
   deleted it; that layer is a **working reference** (see `dev/profiling/v05-lineage.md`).
   Do **not** profile the `fathom_*` names ‚Äî those were never a shipped table, only
   the `reject_legacy_shape` tripwire (`lib.rs:4529`). Profile edge-write cost now,
   and **establish the recursive-CTE traversal baseline** on `canonical_edges`
   (with and without the prospective `from_id`/`to_id` index) so the G5 design has
   a real number, not the dev-host-dependent guess in the existing `SLOW_CTE`
   fixture (`lifecycle_observability.rs`).

### Two known facts you must account for (do not "discover" them as bugs)

- **Batch vec0 collapse** (`dev/notes/0.7.0-engine-batch-vec0-collapse.md`):
  `engine.write(batch)` with `batch.len() > 1` of vector-indexed nodes collapses
  to **one** `vector_default` row (`write_inner` passes the final batch cursor to
  every per-row INSERT). This **distorts ingest profiling** ‚Äî a batched ingest
  under-counts embed work. Your harness MUST either drive per-node writes (like
  `tests/corpus_vector.rs` does) or explicitly record the collapse so vec0 timings
  are honest. State which you chose in the report.
- **Async projection is off the write thread.** `engine.write` returns before
  embedding happens; vector-projection cost only fully lands after
  `engine.drain(timeout)`. Measure write-return latency and drain-to-quiescence
  latency **separately** ‚Äî conflating them hides the real embed cost.

## What to build

A **profiling harness** (Rust integration test or example binary in
`fathomdb-engine`; prefer `examples/profile_ingest.rs` so it can run a real
corpus outside the test gate, plus a tiny `tests/prof_ingest_smoke.rs` that runs
the synthetic path in CI in <30 s). It must:

1. **Stream a corpus in growing checkpoints** (e.g. N = 1K, 5K, 10K, ‚Ä¶ bounded by
   corpus size and an `AGENT_LONG`/env cap), driving the **per-node write path**
   (to dodge the vec0-collapse distortion) via `CorpusFixture`.
2. **Attach a recording `Subscriber`** that buckets `on_profile` wall-clock by
   layer: canonical-write statements, FTS5 (`search_index`) inserts/merges, vec0
   (`vector_default`) inserts, op-store, and (if exercised) edge inserts. Capture
   `on_slow_statement` spikes (the FTS5 merge bursts and any full-scan).
3. **Separate the three ingest clocks** per checkpoint: (a) `engine.write` return
   latency, (b) embed/projection cost to drain-quiescence (`engine.drain`), (c)
   total wall. Record p50/p99 per layer where there are enough samples.
4. **Snapshot `engine.counters()`** + `SELECT count(*)` on each canonical/shadow
   table at each checkpoint, so per-layer cost can be normalized per row and the
   vec0-collapse ratio is visible.
5. **Establish the graph-traversal baseline**: with a synthetic edge set over the
   ingested nodes, time a `WITH RECURSIVE` neighbors walk at depth 1/2/3 **with
   and without** a `from_id`/`to_id` index (create the index in a scratch copy ‚Äî
   do NOT touch the real schema), and capture `EXPLAIN QUERY PLAN` for each. This
   is the G5 input.
6. **Run the PRAGMA / pagecache sweep** as an optional axis via the existing
   `FATHOMDB_PERF_*` knobs (document which knobs moved the ingest numbers).
7. **Emit a machine-readable report** (`dev/perf-history/`-compatible JSON
   superset: add `phase: "ingest"`, per-layer `*_ms`, `n`, `rows_per_layer`,
   `embed_ms`, `write_return_ms`, `drain_ms`, `vec0_collapse_ratio`,
   `commit_sha`, `timestamp`) **plus** a human-readable markdown summary under
   `dev/notes/perf/`.

## Constraints (hard)

- **Measurement-only.** No change to `lib.rs` behavior, schema, public surface,
  or budgets. New code lives in `examples/`, `tests/`, `dev/notes/perf/`,
  `dev/perf-history/`. If you find you *need* a new engine instrumentation seam
  (e.g. a structured per-layer timer the `Subscriber` can't reach), **stop and
  surface it as a HITL question in the closure** ‚Äî do not add it unilaterally.
- **Single-writer invariant** is real: all ingest goes through the one writer
  thread; do not parallelize writes. Reads (counters, EXPLAIN, traversal
  baseline) use separate connections / the reader pool.
- **Honor the async projection boundary**: always `drain` before reading vec0
  counts; never assert vec0 row count == node count without accounting for
  collapse.
- **Determinism / honesty**: pin the dev-host spec in the report (CPU, cores,
  SQLite version via `rusqlite::version()`, embedder identity via `dump_profile`).
  Mark any dev-host-dependent number as such (the existing `SLOW_CTE` fixture is
  the cautionary tale). Cache-warm vs cache-cold embed runs must be labeled
  (`CorpusFixture` surfaces `embed_cache_hit` / `cache_miss_reason`).
- **Feature flags**: the synthetic-embedder smoke must run WITHOUT
  `default-embedder`; the real-corpus profile runs with it. Don't gate CI on the
  real-embedder run.

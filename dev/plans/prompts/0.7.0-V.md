## Tasks

1. **Provision (or identify) a fresh non-CI host.** Ideally a
   machine that has never installed fathomdb. Capture: OS +
   version, package-manager versions (cargo, pip / uv, npm /
   pnpm), kernel.
2. **Per binding, install from the canonical registry:**
   - Python: `pip install fathomdb==0.7.0rc1` (or `uv pip
install ...`).
   - TypeScript: `npm install fathomdb@0.7.0-rc.1`.
   - Rust: `cargo add fathomdb@=0.7.0-rc.1` in a scratch crate.
3. **Smoke-test the 5-verb canonical surface** (`Engine.open`,
   `write`, `search`, `close`, `admin.configure`) per binding.
4. **OpenReport accessor check** (carried from 0.6.1 surface):
   `engine.open_report()` (py) / `engine.openReport()` (ts) /
   `OpenedEngine.report` (rust) returns the structured report.
5. **Top-K LIMIT smoke** (conditional on Q4 = additive). For
   each binding:
   - Default-call `engine.search(query)` returns the same
     result-set count as a 0.6.x baseline (no silent cap).
   - `engine.search(query, limit=5)` returns at most 5 results.
   - `engine.search(query, limit=0)` returns empty.
6. **Perf-attribution sanity** (informational; not a gate):
   record a single-query `search` latency from the fresh host
   for the release notes. Not a perf-gate verdict — the
   canonical runner is the only honored signal.
7. **Capture transcript** at
   `dev/plans/runs/0.7.0-V-transcript.txt` (stdout/stderr of
   each install + smoke).
8. **If anything is RED on a fresh host, surface.** RC1 may need
   an RC2 spin.

## Hard constraints

- **No CI runner.** The whole point is to exercise the published
  artifacts against a host that has never seen the source tree.
- **No source-tree dev install.** Install from registries only.
- **No release-notes edits during verification.** If gaps
  surface, log them; they land in a subsequent slice or in GA
  prose.
- **No perf-gate-verdict claim from the fresh host.** Canonical
  runner is the only honored signal.

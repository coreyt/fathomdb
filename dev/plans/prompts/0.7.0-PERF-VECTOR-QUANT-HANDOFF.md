## What you are shipping

Binary-quantization (`bit[768]`) with f32 rerank on the existing
`sqlite-vec` extension, plus a metadata+partition_key schema
migration. Closes AC-013 (currently p50=2048 ms / p99=2327 ms at
canonical-CI N=1M) to within the proposed 80 / 300 ms budget by
~20× reduction in per-query bytes-scanned.

The research that grounds this work:
`dev/notes/0.7.0-vector-cost-research.md` (HITL-RESPONDED).
Read it first. The HITL doc
`dev/plans/0.7.0-HITL-recommendations.md` is the authoritative
record of decisions.

## Pack 1 — schema migration + ingest double-write

Goal: land the new schema and the new ingest path. **No reader
behavior change.** AC-013 / AC-019 still measure on the
unindexed f32 path; their numbers stay at 2048/2327 until
Pack 2.

### 1.1 — Schema migration

Modify `vector_default` (or add a sibling
`vector_default_bin`). Decision criterion:

- **Sibling column on the same vec0 table** — simpler
  application code; sqlite-vec supports mixed `float[768]` +
  `bit[768]` columns on the same vec0. **Default choice.**
- **Sibling table** — only if the column-flavour hits a
  vec0 limitation (e.g., metadata column count limit, which is
  16 per
  [sqlite-vec features](https://alexgarcia.xyz/blog/2024/sqlite-vec-metadata-release/index.html)).

New columns:

- `embedding_bin bit[768]` — the load-bearing quantized
  column. Built with `vec_quantize_binary(embedding)`.
- `kind TEXT` — metadata column. Existing chunk records have
  a vector kind via `_fathomdb_vector_rows.kind` (verify
  exact column name when reading lib.rs); replicate it here.
- A `partition_key` derived from `kind` (or a coarser
  bucket if the kind cardinality is too high for the 4-key
  vec0 limit). Decision deferred to ADR-time after reading
  the actual kind distribution.

Migration SQL: one-pass `INSERT ... SELECT
vec_quantize_binary(embedding) FROM vector_default` for the
existing 1 M test corpus. Costs ~64 s on the dev box; well
inside the workflow timeout.

### 1.2 — Writer ingest double-write

At `_fathomdb_vector_rows` insert site (`lib.rs:2846` area):

- After the f32 `embedding` insert, compute
  `vec_quantize_binary(embedding)` and insert into the
  `embedding_bin` column in the same transaction.
- The bit column is non-nullable post-Pack 1 (every f32 row
  has a bit row); enforce via NOT NULL or by post-write
  migration.
- Populate `kind` + `partition_key` from the existing chunk
  record fields.
- The writer transaction commits all columns atomically — no
  separate commit for the bit column.

### 1.3 — Pack 1 tests

- **Migration round-trip test** — insert a known f32 vector,
  verify the bit column contains the expected popcount
  pattern (`vec_distance_hamming` returns 0 against itself
  after binarization).
- **Writer regression** — existing AC-012, AC-017, AC-018,
  AC-020 tests continue to pass. Pack 1 changes schema but
  not query semantics, so all should be GREEN.
- **No AC-013 budget change yet** — AC-013 stays pinned at
  the unindexed 50/200 budget (current `perf_gates.rs:149-150`)
  and continues to fail at canonical-CI 1M scale until

### 1.4 — Pack 1 closure

Land the schema + ingest changes as one or two commits on
`main` (campaign convention from PCACHE2). No canonical-CI
dispatch is required for Pack 1 — the schema change has no
RED gate at canonical scale. Smoke test via dev-box AC-013 at
N=10_000 to confirm the ingest double-write doesn't crash.

## Pack 2 — query-path swap + RED tests

Goal: switch the reader to bit-KNN → f32 rerank and pin the
new AC-013 budgets.

### 2.1 — RED tests FIRST

Before the query rewrite, land:

- **AC-013 latency budget re-pin** in
  `tests/perf_gates.rs:149-150`:
  - `AC013_BUDGET_P50 = Duration::from_millis(80);`
  - `AC013_BUDGET_P99 = Duration::from_millis(300);`
- **New AC recall-floor test** (e.g., `ac_NNN_recall_at_10`):
  - Computes brute-force f32 ground truth at test-setup
    (one pass through the existing f32 column, no
    rerank).
  - Runs the production search path, captures top-10 IDs
    per query.
  - Asserts `recall@10 ≥ 0.90` (HITL-locked floor).
  - The ground-truth pass uses the *same* f32 column that
    Pack 2 retains for rerank — single corpus, both jobs.

Both tests must FAIL at canonical-CI N=1M before the query
rewrite lands. Dispatch the perf-canonical workflow with
`targets="ac013 ac019"` and the locked W4.1-stacked-O1 env
knobs (see HITL doc) to confirm RED.

### 2.2 — `read_search_in_tx` rewrite

At `lib.rs:2317-2323`, replace the single-phase f32 KNN with
two-phase:

```sql
-- Phase 1: bit-KNN to top-K candidates (K=64 default)
WITH candidates AS (
  SELECT rowid, distance AS bit_distance
  FROM vector_default
  WHERE embedding_bin MATCH vec_quantize_binary(vec_f32(?1))
  ORDER BY distance
  LIMIT 64
)
-- Phase 2: f32 rerank on the candidate set
SELECT c.rowid, vec_distance_l2(v.embedding, vec_f32(?1)) AS d
FROM candidates c
JOIN vector_default v USING (rowid)
ORDER BY d
LIMIT 10
```

(Exact SQL TBD against sqlite-vec syntax — see
[sqlite-vec API ref](https://deepwiki.com/asg017/sqlite-vec/3-api-reference)
and the binary-quant guide cited in the research doc.)

K=64 picked from the binary-quant literature (Cohere, Kyle
Howells benches) for ≥ 95 % recall@10 at our scale. Make K a
constant / config knob so future tuning is cheap.

### 2.3 — Pack 2 validation

- Local dev-box smoke at N=10_000 — both tests pass.
- Canonical-CI dispatch with the same lever stack:
  - AC-013 GREEN at ≤ 80 / 300 ms.
  - AC-019 GREEN (existing 10× bound; should improve
    substantially).
  - Recall@10 ≥ 0.90 against f32 ground truth.

Once canonical-CI is GREEN, the AC-013 / AC-019 rows in the

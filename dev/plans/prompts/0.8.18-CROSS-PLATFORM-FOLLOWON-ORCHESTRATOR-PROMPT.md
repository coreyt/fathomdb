## 0. Why this exists (the D5 decision)

**HITL D5 (2026-07-08):** for 0.8.18, only **this system's arch — `x86_64-unknown-linux-gnu`** — is on the
**critical path**. The repo's *other* declared support platforms **must still be supported**, but they are
**off the critical path** and are completed by **this separate follow-on orchestrator**, which starts
**after** the x86_64-linux GA path works. This keeps the capstone from stalling on cross-compile / runner /
manylinux issues that have nothing to do with the vector-equivalence + publish core, without dropping the
platforms.

**Nothing here weakens the support promise** — it sequences it. Cross-platform completion is *deferred, not
cancelled*. If a platform ultimately cannot be supported, that is a **HITL call**, surfaced with evidence,
never a silent drop.

## 3. What "done" means (acceptance)

For **every** off-critical-path target above:
- The prebuild/wheel **builds green** in CI (matrix leg passes), with the **shipped feature axis** locked
  as on x86_64-linux (`pyo3/extension-module,default-embedder` for wheels; the napi default features) —
  no accidental feature drift per platform.
- The `all-builds-passed` cross-ecosystem gate **includes** the target (a green publish path can't skip a
  declared platform silently).
- A **publish dry-run** for the target is green (no real registry write).
- **`default-embedder` / CPU-only / 1-bit-deterministic footprint holds on every platform** — the library
  query path is identical across arches (no per-platform behavior fork). If any platform cannot hold the
  footprint invariant, **STOP and escalate to HITL** (it's a support-promise question, not an impl detail).
- Docs/DOC-INDEX note the supported-platform set is complete (X3).

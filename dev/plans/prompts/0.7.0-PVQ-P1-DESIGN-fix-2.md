The error: fix-1 claimed `rebuild_vec0` → `rebuild_shadow_state(RebuildKind::Vec0)` repopulates `vector_default` synchronously and "does not re-run the embedder." Codex correctly identified that:

- `rebuild_shadow_state` only truncates vec0 + resets the projection cursor.
- Re-derivation is async via the projection scheduler.
- The async reprojection re-runs `embedder.embed(&job.body)`.
- The test `rebuild_report.rs:96-108` locks in `rows_rebuilt == 0`.

So D4 must be rewritten with a **migration-time, synchronous** repopulation path that does not go through `rebuild_vec0`.

## Mandate — rewrite D4 only

Replace the existing D4 section with the following migration shape (still inside `migrate_with_event_sink` + `PRAGMA user_version` gate):

1. Schema-version bump (same as fix-1).
2. **Stage**: create regular SQL table `_fathomdb_vector_migration_v0_7_0(rowid INTEGER PRIMARY KEY, embedding BLOB NOT NULL, kind TEXT NOT NULL)`. INSERT INTO it from the JOIN of `vector_default` + `_fathomdb_vector_rows` (which provides the `kind` per writer-site at `lib.rs:2845-2853`). Note: `_fathomdb_vector_rows.kind` is the source of truth for the existing rows' kind — cite that anchor.
3. `DROP TABLE vector_default`.
4. `CREATE VIRTUAL TABLE vector_default USING vec0(embedding float[<dim>], embedding_bin bit[<dim>], source_type TEXT partition key, kind TEXT, created_at INTEGER)`.
5. **Repopulate**: Rust-side loop over staging rows, computing `source_type` via the D3 coercion map (already defined in the memo). Bind into `INSERT INTO vector_default(rowid, embedding, embedding_bin, source_type, kind, created_at) VALUES(?1, ?2, vec_quantize_binary(?2), ?3, ?4, ?5)`. `created_at` is the migration timestamp (UNIX seconds at migration time — uniform across migrated rows; OK as a backfill value).
6. `DROP TABLE _fathomdb_vector_migration_v0_7_0`.

Properties to record explicitly in the D4 section:

- **Synchronous** — runs inside the writer's migration transaction.
- **No embedder rerun** — copies existing f32 blobs unchanged.
- **Atomicity** — the entire DROP+CREATE+repopulate sequence is inside the migration transaction; if it fails midway, the prior schema version is preserved (cite the schema crate's transaction handling at `src/rust/crates/fathomdb-schema/src/lib.rs:218-287`).

- **vec_quantize_binary call site** — runs SQL-side at INSERT bind, once per row, on the existing f32 blob. No Rust-side bit math.
- **Why not `rebuild_vec0` / `RebuildKind::Vec0`** — explicitly state: that path is async, re-runs the embedder, and would multiply the migration cost; codex fix-1 review found this contradiction (`dev/plans/runs/0.7.0-PVQ-P1-DESIGN-fix-1-review-20260527T134214Z.md` finding 1).

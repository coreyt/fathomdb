## Deliverables (TDD, RED → GREEN; pure-Python; FAKE judge; no native rebuild)

Create `src/python/eval/autoe_judge.py` (+ tests `tests/test_autoe_judge.py`):

1. **Pairwise-judge prompt builder** — given a question + two candidate answers
   (labelled A/B) + the metric set, build a BenchmarkQED-style pairwise prompt that asks
   the judge to choose A, B, or tie **per headline metric** (comprehensiveness, diversity,
   empowerment) **and** for **directness** (the length-bias corroboration metric — kept
   SEPARATE from the headline win). Deterministic given inputs.
2. **Position-bias control** — evaluate **both orders** (answer X as A then as B) and
   average; a `JudgmentKey(question_id, pair, run_idx, order)` identifies each judgment so
   resume is idempotent. `order_swapped=True` is structural, not optional.
3. **Verdict parser** — parse a judge completion into per-metric `{A|B|tie}`; an
   empty/None/unparseable completion → **ABSENT** (never silently a loss/tie), excluded
   from the denominator and re-attempted on resume.
4. **Win-rate aggregation → `decide_084` input** — `compute_winrates(judgments, pair, ...)`
   returns the `primary_per_metric` mapping: per headline metric, win-rate = (wins +
   0.5·ties)/decided across all questions×runs×orders, with a **bootstrap CI clustered by
   question** (resample questions, not individual judgments — the ≥5 runs × 2 orders are
   within-question replicates) and the per-metric `mde` = CI half-width. Pure/deterministic
   given a seed passed in (no argless RNG — seed is a parameter).
5. **Bias-control + length assembly** — a function that assembles the `BiasControls` and
   `LengthCorroboration` structs from the run config + the directness aggregate
   (`contradicts` = directness flips against the arm that won the headline metrics, by a
   margin — define a clear, tested rule), so the caller can hand them straight to
   `decide_084`.
6. **Batch-build integration point** — `build_autoe_batch_jsonl(...)` that emits the
   judge-prompt JSONL via the `p0a_batch_e2e` builder shape, tested with a fake (NO live submit).
7. **A cost-projection helper** — given measured per-call prompt/completion tokens + the
   run shape (n_questions × n_pairs × n_runs × 2 orders × n_metrics-in-one-call), reuse
   `project_full_cost` to return the projected priced-run USD (this is what the
   cheap-validate dry-run will feed to produce the HITL number).

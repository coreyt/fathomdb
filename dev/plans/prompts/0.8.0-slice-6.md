## 3. What to investigate (measure, then recommend)

### Q1 (LEAD QUESTION) — at which TIER does the regression breach a *binding* budget?

`ac_012` asserts 20ms p50 at **100k unconditionally**, but its sibling latency ACs (`AC-013`/`AC-019`) were
HITL-revised to a **tiered** budget — gated only at the **10k tier** for 0.x/1.x, 100k/1M as post-1.0 tracked
targets (AC-072/AC-073, `ADR-0.7.0-text-query-latency-gates-revised`). **Measure the CURRENT (`porter unicode61
remove_diacritics 2`) tokenizer's `ac_012` p50/p99 at BOTH 10k and 100k** (use the corpus-N env knob;
`AGENT_LONG=1`). Report: does the regression breach the budget at the **binding 10k tier**, or only at 100k?
This is potentially the cleanest resolution: **was `ac_012` simply never tiered to match AC-013/AC-019?**
(Do NOT decide it — surface it as the leading HITL option with the data.)

### Q2 — tokenizer config space: latency vs FTS quality

Measure `ac_012` p50/p99 (at the tier(s) Q1 shows matter) for a candidate set of tokenizer configs, e.g.:
`porter unicode61 remove_diacritics 2` (current) · `unicode61 remove_diacritics 2` (drop porter stemmer) ·
`porter unicode61` (drop diacritics-folding) · `unicode61` (lightest) · `porter ascii`. For each, also report
a **FTS-quality proxy** (e.g. recall/precision on a small held-out query set, or at minimum a qualitative note
on what stemming/diacritics-folding buys for the G1 goal in the Slice-5 design memo) so the latency saving is
weighed against the quality it costs. State the seed/sample methodology (warmup-discard + measurement pass, per
the ADR).

### Q3 — query-path: where is the cost?

Is the latency in **query-time tokenization** (porter stemming each query token) or in the FTS5 MATCH/scan?
A quick attribution (e.g. compare a single-token vs multi-token query, or profile if feasible) tells HITL
whether a query-path optimization (caching, query construction) is even a lever, or whether it's intrinsic to
the tokenizer at index/query time.

### Q4 — source accessibility (the HITL-asked dimension)

Assess the **accessibility/modifiability** of the relevant source: SQLite/FTS5 is bundled (vendored
amalgamation) — document what that means for (a) swapping the built-in porter stemmer, (b) registering a
**custom FTS5 tokenizer** via the tokenizer API (feasibility + rough cost), and (c) whether the **embedder**
source (the candle/bge-small path) is in-tree and relevant to the broader retrieval latency picture (note it
if the user's interest extends there; the FTS5 path itself does not use the embedder). A short feasibility
matrix is enough — you are scoping levers, not building them.

### Recommendation

Synthesize Q1–Q4 into a recommendation with the **explicit HITL fork**, e.g.: (i) **tier AC-012 consistently**
(10k binding, 100k tracked — if Q1 shows 10k is within budget); (ii) **lighten the tokenizer** (name the config

+ its measured latency win + quality cost); (iii) **custom tokenizer** (if Q4 shows it's worth it); (iv)
**re-ratify the 100k AC-012 budget** (accept the latency for the quality gain). State which you'd pick and why,
grounded in your numbers. The decision is HITL's; your job is to make it well-informed.

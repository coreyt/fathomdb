#### 3.2.2 `PreparedWrite` extension

The BYO-LLM ingest path will need a way to write enriched edges. Rather than a new
`PreparedWrite` variant (which would require SDK surface changes in this slice), consider
using the existing `PreparedWrite::Edge` variant extended with the G11 fields:

```rust
Edge {
    kind: String,
    from: String,
    to: String,
    source_id: Option<String>,
    logical_id: Option<String>,
    // G11 enrichment (Slice 15)
    body: Option<String>,
    t_valid: Option<String>,
    t_invalid: Option<String>,
    confidence: Option<f64>,
    extractor_model_id: Option<String>,
}
```

This is **additive** — existing code that constructs `PreparedWrite::Edge` will need to add
the new fields (all `None`). Update all call sites. If there are many, use struct update
syntax or a constructor helper.

Alternatively, handle the G11 fields through a new `PreparedWrite::EnrichedEdge` variant —
document your choice in the design memo and flag as `additional_changes_made_in_scope` if
you choose the variant approach (it's a public surface change).

#### 3.2.3 `commit_batch` extension

Extend the `PreparedWrite::Edge` arm of `commit_batch` to:

1. INSERT the G11 columns into `canonical_edges` (body, t_valid, t_invalid, confidence, extractor_model_id).
2. When `body` is non-null: enqueue for FTS + vector projection (do NOT call
   `record_projection_terminal` for edges with body; instead use the node projection path or
   the mechanism from §3.2.4).
3. Implement invalidate-not-accumulate: before inserting a new edge with the same
   `(from_id, to_id, kind)` and `superseded_at IS NULL`, tombstone the prior row
   (`UPDATE canonical_edges SET superseded_at = ?cursor WHERE from_id = ? AND to_id = ? AND kind = ? AND superseded_at IS NULL`).
   Then insert the new enriched row.
   > Note: the existing G0 tombstone logic is keyed on `logical_id`. G11 invalidation is
   > keyed on `(from_id, to_id, kind)` — these are DIFFERENT invariants. The new invalidation
   > applies ONLY for edges written via the BYO-LLM ingest path (i.e., when `body` is non-null
   > and the edge is a fact-edge). Regular edges (body=NULL) retain the existing G0 logical_id
   > semantics. Clarify in the design memo how you distinguish the two cases.

#### 3.2.4 Edge projectability mechanism

Implement your chosen mechanism from §3.0 (design memo §4). Constraints:

- **Option A (extend `search_index`)**: add a `source_type UNINDEXED` column to the FTS5 table
  (this is an additive ALTER TABLE — include it as a sub-step in the step-14 migration block or
  as step-14a). Add `'edge_fact'` to `resolve_source_type()`. When projecting an edge body,
  INSERT INTO `search_index(body, kind, write_cursor, source_type)` with `source_type='edge_fact'`.
  Update `Engine::search` result processing to distinguish edge vs node hits.
- **Option B (separate table)**: CREATE TABLE `search_index_edges USING fts5(body, kind UNINDEXED, write_cursor UNINDEXED)` — parallel structure; queries it separately.

For **vector projection**: edge bodies need to enter `_fathomdb_projection_state` (the queue
that the projection scheduler drains). Follow the node path in `commit_batch` — when
`body` is non-null, enqueue via the `_fathomdb_projection_state` INSERT, NOT
`record_projection_terminal`. The projection scheduler will then embed the body and write
to `_fathomdb_vector_rows` + `_fathomdb_projection_terminal`. Verify this doesn't require
a `kind_is_vector_indexed` check — or add the edge kind to the vector-indexed set.

**Partition correctness assertion in tests:** after seeding an edge with body + a node with
body, a `text search` for a term in the edge body must return a result typed as an edge
(not a node), and vice versa.

#### 3.2.5 BYO-LLM ingest engine method

Add to `Engine`:

```rust
pub fn ingest_with_extractor(
    &self,
    provider_cmd: &[&str],
    documents: &[ExtractDocument],
) -> Result<IngestWithExtractorReceipt, EngineError>
```

where `ExtractDocument` has `{ source_doc_id: String, body: String }` (align with the
protocol's `documents[]` field).

Implementation:

1. Spawn subprocess; attach stdin/stdout; stderr is ignored by the engine.
2. Send `{"protocol":"fathomdb.extract.v1","type":"hello"}\n` on stdin.
3. Read `ready` response from stdout. Validate `protocol == "fathomdb.extract.v1"` and
   `schema_version == 1`; abort with a typed error on mismatch. Record `ready.model`.
4. Batch documents (respect `max_docs_per_request` from `ready`).
5. For each batch: send one `extract` request (with `request_id`, `documents`, `ontology: null`,
   `options: {"deterministic": true}`). Read the `result` or `error` response.
6. On `result`: map entities → nodes (§3.3 of BYO-LLM ADR), map edges → enriched canonical_edges.
7. On `error`: propagate typed error code.
8. At end: close stdin (signals EOF to harness); wait for process exit.

`logical_id` for entities: derive as `sha256(type.to_lowercase() + ":" + name.to_lowercase())`
(or your documented algorithm). Use the existing `canonical_nodes` `logical_id` column for
idempotent insert (G0 supersession semantics apply).

  `output.json` and continue.
- **Justified deviation:** a path in §3 is blocked (compile error, anchor gone, contract
  internally inconsistent). Adapt minimally, log loudly, escalate if cross-slice.

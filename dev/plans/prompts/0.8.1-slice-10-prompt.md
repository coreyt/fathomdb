### 3.0 Design memo FIRST

Write `$WT/dev/design/slice-10-design.md` before touching any source file.
The memo must address the following decisions explicitly — the design review (§3.0.R)
will audit these:

1. **Implementation crate structure.** Where does the CE live — a new
   `fathomdb-reranker[-api]` crate pair (mirrors the embedder pattern), an extension to
   `fathomdb-embedder`, or directly in `fathomdb-engine`? Justify against the workspace
   Cargo.toml (7 current crates). Whichever you choose, the `fathomdb-engine` crate
   must not gain a hard dependency on a large ML crate in the default build.
2. **Model and inference backend.** TinyBERT-L-2 (preferred) or MiniLM-L6/L12; ONNX
   Runtime (`ort` crate) or Candle (`candle-*`, already in the workspace as optional in
   `fathomdb-embedder`). Latency budget: TinyBERT-L-2 p50=1.536ms, p95=2.85ms per pair
   (K=200 ≈ 307ms p50, 570ms p95). Justify your model/backend choice.
3. **Model weight loading.** How are weights fetched/cached? The `default-embedder`
   feature pattern is the reference (`fathomdb-embedder/Cargo.toml`). Gate the real
   reranker behind a `default-reranker` feature (or equivalent) so the default build
   stays no-network-call.
4. **Signature change for `rerank_fused`.** The current signature is
   `rerank_fused(hits: Vec<SearchHit>) -> Vec<SearchHit>` — it has no query string and
   no reranker handle. The real CE needs both. Propose the new signature and how it
   reaches the call site at `lib.rs:5051`. Also address `rerank_fused_is_identity_stub`
   in `pr_g9_rrf_fusion.rs`: what replaces it?
5. **Score-blend formula.** Document the interpolation: e.g.,
   `final_score = α * ce_normalized + (1-α) * rrf_score`. Specify how CE scores are
   normalized (raw logit vs sigmoid vs min-max over the pool), the value of α (and
   whether it should be a constant or configurable), and why this prevents factoid
   regression (C6).
6. **`rerank_depth` exposure on the public API.** The R1 reranker is a **per-request
   opt-in** (latency concern per IR-C). How is `rerank_depth=0` (default, current
   behavior) vs `rerank_depth=N` (rerank top-N) expressed on `search()` /
   `search_filtered()` and mirrored in the Python/TS bindings? Propose the concrete
   API shape (new param, new options struct, or new method). The governed-surface rules
   apply — new public types go through the facade.
7. **Soft-fallback contract.** When `rerank_depth=0` OR the model is not loaded (feature
   off or weights absent), `rerank_fused` must return the fused list byte-identical to
   the current identity stub (no score perturbation, no reorder). State how you test
   this.
8. **Determinism.** Reranked output for the same (query, corpus snapshot) must be
   byte-identical across calls. State how CE inference achieves this (fixed model +
   fixed tokenizer + no stochastic sampling).
9. **RECENCY_WEIGHT fix.** The committed-RED test has `hit(9, "recent", 0.01)` and
   `hit(1, "strong", 0.02)` with `enabled=true`. `norm = (9-1)/(9-1) = 1.0` for
   "recent". For "strong" to win: `0.02 > 0.01 + RECENCY_WEIGHT * 1.0` → weight < 0.01.
   Propose the new constant. It must still break exact ties (weight > 0) and still
   satisfy `recency_reweight_latency_gate` (<200ms for 1000×10 items).

Commit the design memo: `git add dev/design/slice-10-design.md && git commit -m "design(0.8.1/slice-10): R1 cross-encoder design memo"`.

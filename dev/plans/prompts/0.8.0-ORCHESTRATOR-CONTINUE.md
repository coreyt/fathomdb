## 2. The loop you run, per slice (this is "keep doing")

For the current pointer slice, act as its **Slice N orchestrator** on the main thread (never
spawn an orchestrator subagent, §1). **Execution model (current, HITL-corrected 2026-06-02 —
this SUPERSEDES the older "main-thread creates the worktree / cherry-pick" mechanic in
`orchestration.md`):**

> The **slice agent owns its worktree** — the human pastes the slice prompt into a fresh
> agent; that agent creates its worktree from local `main`, does the work, and **merges to
> local `main` itself** (no push). You, the orchestrator, work on `main` *after* the merge.
> **You do NOT create slice worktrees.** Some slice agents run the *entire* loop (dispatch
> their own implementer, escalate, review with codex, and CLOSE) — when that happens your job
> is to **independently gate the close from git/disk**, not redo it.

Your per-slice responsibilities:

1. **Orient + plan-adjust.** Read the slice's contract. If reality diverges from the plan,
   ADJUST the plan/board and record it in the closing docs commit (§12.4) — never silently

3. **Gate the transition** (§1.5 inv. 2): after the slice agent reports, confirm from git that

   slice branch tip is an ancestor of `main` (clean merge). If a slice agent also *closed*,
   additionally verify the CLOSED block, the review verdict file, the pointer advance, and

### 2a. Slice-prompt authoring discipline (where slices succeed or fail)

A slice prompt is a delegation contract the subagent executes literally and blind. Honor these
when writing one (they're distilled from real misses this campaign):

- **Verify every load-bearing fact against the FULL file at the current baseline — anchors AND
  claims about behavior/tests.** Re-grep line-number anchors (they drift every slice). And
  before you assert "test X asserts / does not assert Y" as a guardrail, **read test X in its
  entirety** — a partial read put a *false* premise in the Slice 21 prompt ("the subprocess
  test doesn't assert the line-30 error" — it did, at lines 121–146), and both mechanisms it
  prescribed would have *faked* a green result had the implementer not escalated. A false
  factual premise is worse than an under-specified one: it can produce a silently-wrong green.
  ([[slice-prompt-verify-test-claims]].) This is acute for prompts that rewrite or constrain
  conformance suites (Slice 25: `test_surface.{py,ts}`, the `no_recovery_surface` suites).
- **Name companion artifacts + mechanism triggers** the deliverable needs (release-note stubs,
  nav entries, re-projection wiring), so a justified deviation isn't forced by omission
  ([[oob-creep-vs-justified-deviation]]). A forced deviation = a defect report on *your* prompt;
  require deviations to be loud, minimal, and **round-tripped into the spec** at close.
- **Carry the current standing baseline accurately.** As of Slice 21 the Python **pyright
  baseline is RETIRED — `pyright -p src/python` is 0/0/0.** Slice prompts must now require
  **pyright stays 0/0 (zero new AND no regression)**, NOT "expect ~11 baseline errors." The
  only standing `agent-verify` blocker is **AC-037 `netns-deny-egress`** (host
  `apparmor_restrict_unprivileged_userns=1`; wired into CI at Slice 40 gate (n)) — name that
  one, and nothing else, as the allowed non-green step.
- **Bind gaps, not invented ACs.** `dev/acceptance.md` is LOCKED at AC-073; 0.8.0 impl slices
  bind a **G-gap label + TDD test names**, never a fabricated AC id. New/superseded ACs happen
  only at the gated slices (25, 40) ([[acceptance-md-locked-no-feature-acs]]).
- **Bake in the guardrails:** path-isolation + stray-edit tripwire (§0 of the template),
  recover-out-loud console contract, strict scope discipline (no push/fetch/PR/origin; merge
  only the slice branch to local `main`), and the `output.json` schema written LAST.
- **Pin the reviewer + severity convention** in the DoD so the slice agent knows the bar.

## 4. Reserved-gap discipline (the N-mod-5 mechanism)

Planned slices are mod-5 only (0,5,10,…). If a slice reveals — or **HITL directs** — adjacent
work that must land before the next planned slice, that becomes a **new fully-orchestrated
slice** in the reserved band (Slice N+1..N+4): fresh baseline + worktree + review + CLOSED
block. A reserved-gap slice ≠ a fix-N (fix-N re-enters a finding's own scope). **HITL may insert
one mid-campaign** (e.g. the interim pyright-zero cleanup that took the 21 slot) — when so:
**audit read-only first to enumerate the real scope, confirm boundaries with HITL via
AskUserQuestion, then author + hand off.** Record any slot *repurpose* (the pencilled "21 =
shadow reconciliation" stayed the Slice 16 candidate; no collision). Band fills and more remains
→ the slice was mis-scoped → HALT.

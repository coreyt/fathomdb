# PR-2b launch prompt -- production mean recompute (engine; design + implement + test)

Paste the "Agent prompt" section into a fresh agent session. This is an
**engine slice** (RED->GREEN, HITL-gated) that adds a **production mean-recompute
capability** to the embedder mean-centering subsystem. It is NOT about recovering
the PR-2a recall number (PR-2a's static corpus cannot show a recompute benefit);
it is about giving production a correct, tested way to refresh the pinned corpus
mean when the workspace's distribution drifts after the one-time 256-doc pin.

This deliberately **reopens the EU-2 "escape hatch"**: `dev/design/embedder.md`
§0.3 locked compute-once-pin-at-256 with "refresh only via reindex (deferred)"
and noted *"If real-world drift becomes a problem, the next campaign can
revisit."* PR-2b is that revisit, scoped to the mean only (not full reindex).

## Sequencing + naming (read first)

```
PR-2a (DONE)  recall investigation -> GO on a mean-representativeness fix
   │   verdict: bad single-source pin cos=0.82 to true mean (-10.9pp); a
   │   representative 256-sample reaches cos=0.998 ~ full-corpus mean.
   ├─ PR-2b (THIS SLICE)  mean recompute: drift-trigger + manual doctor verb
   │                       (synchronous, in-tx; notifications; N>=200k -> doctor-only)
   ├─ PR-2c  representative INITIAL pin (former PR-2b): sample/stratify the 256
   │          docs feeding the pin instead of ingest-order first-256. Small.
   │          Covers the window between pin@256 and the first recompute trigger.
   └─ both PR-2b + PR-2c GATE PR-2 (floor lock on the re-measured number)
```

PR-2b and PR-2c are **separate** slices. PR-2b is the larger, riskier subsystem
(new trigger + execution + telemetry). PR-2c is a small refinement of *which
rows* feed the existing pin. Land PR-2b first (this prompt), then PR-2c, then
re-measure, then PR-2.

## HITL decisions already made (do not re-litigate; implement as written)

1. **Trigger = hybrid**: an automatic **distribution-drift detector** during
   ingestion AND an explicit **manual `doctor` verb**.
2. **Execution = synchronous, in-transaction**, reusing the proven at-pin
   re-quantize path. PLUS:
   - **(a)** there MUST be a **callback / notification of the work** (start +
     finish at minimum) so callers can observe a recompute, including the
     long re-quantize pass.
   - **(b)** at **N >= 200,000 rows, do NOT recompute dynamically** -- the
     automatic trigger is suppressed and only the **`doctor` verb** may perform
     the recompute; but a **notification MUST be surfaced** when drift is
     detected at that scale (so the operator knows a `doctor` run is advised).
3. **Sequencing**: PR-2b then PR-2c; both gate PR-2.

### Design to implement

Hold the existing pin-at-256 behavior, the f32 rerank, K=192, the single-writer

- `commit_gate` serialization, and crash-atomicity all fixed. Add:

**1. Recompute core (shared by both triggers).** A function that, in ONE
transaction: derives the new mean from the workspace's `vector_default` rows
(the full-corpus mean; you MAY compute the mean over a representative sample to
bound cost -- PR-2a shows sample ~ full at cos 0.998 -- but the **re-quantize
pass MUST touch every row**), writes `mean_vec`, runs the existing
`run_pin_and_requantize_pass` over all rows, and stages a recompute event. This
is `recover_mean_vec_pin` generalized to refresh an *already-pinned* mean (drop
its "no mean pinned yet" guard). Reuse, do not reimplement, the re-quantize.

**2. Manual path -- `doctor recompute-mean`.** New `DoctorCommand::RecomputeMean`
verb (`fathomdb doctor recompute-mean <db_path> [--json]`) that invokes the
recompute core through a new public `Engine` method. This path is ALWAYS allowed,
including at N >= 200k. `--json` output is normative (per `cli.md`): report old/
new mean dim, doc_count re-quantized, drift cos before/after, elapsed.

**3. Automatic drift detector (ingestion, N < 200k only).** Maintain a cheap
running estimate of the *recent* vector mean in `ProjectionRuntimeShared` (e.g.
an EWMA or sliding-window accumulator over committed vectors -- justify the
choice; it must be O(dim) memory and O(1) per row). On commit, compute
`cos(running_recent_mean, pinned_mean)`. When it falls below
`MEAN_DRIFT_COS_THRESHOLD`, trigger the recompute core synchronously within the
same commit transaction (serialized under `commit_gate`). Calibrate the
threshold and DOCUMENT it: PR-2a measured the pathological single-source pin at
cos 0.82 to the true mean and a healthy representative mean at 0.998 -- pick a
threshold in that gap (start ~0.95) and justify; add a debounce so one drift
episode does not fire repeatedly (e.g. re-arm only after cos recovers, or a
min-rows-between-recomputes guard).

**4. The N >= 200k dynamic cap.** `MEAN_RECOMPUTE_DYNAMIC_MAX: u64 = 200_000`.
When the drift detector would fire but row_count >= this cap, DO NOT recompute;
instead stage a `MeanRecomputeDeferred` notification (drift cos + doc_count +
"run `doctor recompute-mean`"). The manual `doctor` verb is unaffected by the cap.

**5. Notifications / callback (decision 2a).** Extend `EmbedderEvent`:

- `MeanVecRecomputed { dim, doc_count, trigger }` (trigger in {DriftAuto,
     Manual}) -- staged and published ONLY after the recompute tx is durable
     (mirror the post-commit publish at the existing at-pin path).
- `MeanRecomputeDeferred { doc_count, drift_cos }` -- drift detected at
     N >= 200k.
   Provide a **non-test** observation seam (the current drain is test-only):
   either a public drain method on `Engine` or an optional callback sink the
   write/recompute API surfaces, so a caller can observe recompute start +
   finish (and ideally progress on the re-quantize for large N). Surface the
   same events through `EngineOpenReport.embedder_events` where an open-time
   recompute applies.

**6. Docs / ADR.** Amend `embedder-decision.md` §3.4 (status is no longer "no
auto-recomputation"; record the drift-trigger + doctor verb + the 200k cap +
why a synchronous in-tx execution was chosen over the rejected background-worker
option) and `embedder.md` §0.3/§0.5 (the escape hatch is now taken for the mean;
reindex remains the path for non-mean refresh). Add or update an ADR for the
recompute policy if the existing ADR structure calls for it. State plainly what
is NOT covered (full reindex, per-source means).

### Tests (RED -> GREEN)

- **Mechanical**: manual recompute produces the full-corpus mean (within fp
  tolerance of an independent numpy/closed-form mean); re-quantizes every row;
  the pinned `mean_vec` and the row sign-bits are mutually consistent after.
- **Crash-atomicity**: a fault between the `mean_vec` UPDATE and the re-quantize
  completion rolls back fully (no half-recentered corpus) -- model on the
  eu5f at-pin atomicity tests.
- **Drift efficacy (the test PR-2a's static corpus could NOT do)**: build a
  SYNTHETIC topic-drift corpus -- ingest 256 topic-A docs (pin an A-skewed
  mean), then ingest many topic-B docs; assert the detector fires, and that
  topic-B query recall@10 (engine path, target-excluded GT, same harness shape
  as `eu7_real_corpus_ac.rs`) improves measurably after recompute vs before.
- **No-drift stability**: a homogeneous corpus does NOT trigger spurious
  recomputes (threshold + debounce not over-sensitive).
- **N>=200k cap**: at/above the cap the auto path is suppressed and emits
  `MeanRecomputeDeferred`; `doctor recompute-mean` still recomputes.
- **Events**: correct `trigger`, `dim`, `doc_count`; published only after
  durable commit; deferred event carries the drift cos.
- **CLI**: `doctor recompute-mean` happy path + `--json` contract + a non-clean
  exit code path consistent with the existing doctor verbs.

### Hard constraints

- Reuse `run_pin_and_requantize_pass` + `MeanAccumulator`; do NOT fork the
  re-quantize logic. Single-writer + `commit_gate` invariants hold
  (`crates/fathomdb-engine/src/runtime.rs`, `database_lock.rs`).
- Synchronous in-tx execution only (no background worker in this slice).
- The N>=200k cap is a hard guard on the AUTOMATIC path; the doctor verb is
  exempt by design.
- Deterministic + crash-safe; events publish only post-durable-commit.
- No push without explicit HITL OK.

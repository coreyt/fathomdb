## Decisions the memo MUST resolve

For each decision: option chosen, alternatives weighed in one line,
citation that grounds the choice. Be concrete enough that P1-IMPL
can read the memo and write code without a second design pass.

### D1 — Sibling column vs. sibling table

Sibling column (`embedding_bin bit[768]` on `vector_default`) is the
handoff default. Confirm this is what vec0 0.1.7 actually supports
(mixing `float[N]` + `bit[N]` columns on the same vec0). If it
doesn't, choose sibling table (`vector_default_bin`) and document the
fan-out cost on the read path.

### D2 — Metadata column inventory (≤ 16, per vec0 feature limit)

Enumerate every metadata + partition column Pack 1 will create on
the migrated `vector_default`. The HITL doc + handoff name these
candidates; this memo picks the minimum-viable set and headroom
budget. Must include:

- `source_type TEXT` (partition_key, locked).
- `kind TEXT` (kept for back-compat if it differs from `source_type`).
- The exact additional metadata columns Pack 1 will land
  (`created_at`, `tags`, `project_mentions` were mentioned but are
  not locked — decide for Pack 1 vs defer).

Document remaining headroom out of 16.

### D3 — `kind` → `source_type` mapping table

The current writer site (`commit_projection_outcomes` ~line 2845)
gets `kind` from `ProjectionOutcome::Success`. The AC-013 fixture
uses `kind = "doc"`. Provide an explicit, deterministic mapping
table from every known `kind` produced by today's code to a
`source_type` in {email, article, paper, meeting, note, todo}.

If "doc" has no clean mapping: the design choice is one of
(a) extend `source_type` vocab to include "doc" or "test",
(b) map "doc" → "article" (or similar) by convention,
(c) leave `source_type` nullable for kinds without a mapping (loses
partition_key benefit for those rows).

Pick one. Cite the HITL doc / research note where they constrain
the choice.

### D4 — NOT NULL + migration strategy

Handoff says `embedding_bin` is non-nullable post-Pack-1. Decide:

- Land NOT NULL from the start (writer enforces; migration backfills
  in one pass before adding the constraint), OR
- Land nullable in Pack 1, tighten to NOT NULL in a follow-up.

State the migration SQL shape (one `INSERT … SELECT
vec_quantize_binary(embedding) …` pass per the handoff). Note the
~64 s dev-box cost for 1 M rows.

### D5 — Writer transaction shape

Confirm the double-write lands in the **same** `tx.execute` sequence
inside `commit_projection_outcomes` (~lines 2845-2853). Decide
whether `vec_quantize_binary` runs:

- inside the SQL (`INSERT INTO vector_default(rowid, embedding,
  embedding_bin) VALUES (?1, ?2, vec_quantize_binary(?2))`), OR
- in Rust before the bind (compute the bit blob, bind both).

Trade-off: SQL-side avoids an extra Rust dep / FFI; Rust-side avoids
double-evaluation if sqlite-vec materialises the f32 twice. Either is
acceptable; pick one and justify in one paragraph.

### D6 — `vec_quantize_binary()` contract validation plan

P1-IMPL TDD's first RED test is the round-trip from
`encode_vector_blob` → insert → `vec_distance_hamming` = 0 against
itself. The memo should specify the test fixture (a small known
vector with a known popcount) so P1-IMPL writes it from spec, not
guesswork.

If the design memo can validate this empirically before P1-IMPL —
by running a one-shot `sqlite3` command against a temp DB with the
extension loaded — do so and capture the output in the memo. If not
practical, mark as "validated empirically by P1-IMPL RED test".

### D7 — Second writer site (test-only)

The test-only `Engine::*` helper around `lib.rs:1808-1850` also
inserts into `vector_default`. Pack 1 must keep this site in sync
(also double-write into `embedding_bin`). Confirm it exists and
note the line range so P1-IMPL doesn't miss it.

### D8 — Reader-path NO-OP guarantee

`read_search_in_tx` (~lines 2317-2323) is untouched in Pack 1.
Confirm: schema change does not break the existing `WHERE embedding
MATCH vec_f32(?1)` query — adding columns to a vec0 table preserves
existing query semantics. Cite the sqlite-vec docs (metadata-release
blog or API reference).

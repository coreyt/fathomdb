### Fix 3.1 (D4 — Finding 1, high): unknown-kind preflight

Replace the `ELSE NULL` + NOT NULL claim with a real fail-fast preflight inside the SAME migration SQL string (assuming multi-statement support; if `apply_one` is single-statement only, fold the preflight into a CTE before the INSERT).

Concrete pattern to record:

```sql
-- Preflight: abort the migration if any existing kind is not in the
-- locked source_type vocabulary (incl. the doc→article coercion).
CREATE TEMP TABLE _vec0_migration_assertion(
  check_passes INTEGER NOT NULL CHECK(check_passes = 1)
);
INSERT INTO _vec0_migration_assertion(check_passes)
SELECT CASE WHEN EXISTS (
  SELECT 1 FROM _fathomdb_vector_rows
  WHERE kind NOT IN ('email','article','paper','meeting','note','todo','doc')
) THEN 0 ELSE 1 END;
-- ... staging + DROP + CREATE + repopulate from staging using CASE ...
DROP TABLE _vec0_migration_assertion;
```

Properties to record:

- The CHECK constraint on `_vec0_migration_assertion.check_passes` fires SQLITE_CONSTRAINT at INSERT time when an unknown kind exists. The CHECK is the load-bearing rejection primitive — vanilla SQLite supports it on `CREATE TABLE`, no extension needed.
- Cite the SQLite docs URL for CHECK constraints (`https://www.sqlite.org/lang_createtable.html#check_constraints`).
- Re-confirm the `apply_one` path treats migration SQL as multi-statement (`execute_batch` semantics) — if it does not, document the alternative shape (single statement with the EXISTS check as a CTE that explicitly violates a unique constraint when unknown kinds are present, OR a Rust-side preflight in the engine open path before invoking the migration). State which alternative is in use.
- The CASE in the INSERT keeps `ELSE 'article'` (the doc-coercion already covers the only known synthetic kind) OR `ELSE 'unreachable'` because the preflight has already proven unknown kinds cannot exist past the assertion. Either is acceptable; pick one and justify.

Remove the `<now>` rendering claim and the Rust-side timestamp construction. In the migration SQL, the `created_at` column is filled by `strftime('%s', 'now')` evaluated at INSERT time. SQLite evaluates `strftime('%s','now')` per row but the value is uniform within a single SQL statement when used as a literal in the SELECT.

Update D4:

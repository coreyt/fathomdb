# DEFERRED to 0.8.x — automatic in-ingest mean-drift detector

**Status:** PARKED / ready-to-revive. Carved out of 0.7.2 PR-2b by PR-2bc S2
(HITL ruling 2026-05-31). Do NOT revive without a measured production / real-
corpus drift case that the manual `doctor recompute-mean` verb demonstrably
fails to cover.

## What this was (the carved-out design)

PR-2b shipped an AUTOMATIC in-ingest distribution-drift detector that
re-derived the pinned corpus mean mid-ingest, inside the projection-commit
hot path, in addition to the manual `doctor recompute-mean` verb. PR-2bc S2
removed the automatic path and kept only the manual verb. The removed design,
captured here verbatim so it can be re-introduced additively:

### Mechanism (all inside `commit_projection_outcomes`, under `commit_gate`)

1. **Recent-mean estimator — EWMA.** A running estimate of the RECENT vector
   mean, folded one un-centered vector at a time:
   `new = alpha * vec + (1 - alpha) * old`, with
   `RECENT_MEAN_EWMA_ALPHA = 1 / MEAN_VEC_PIN_THRESHOLD = 1/256`. O(dim)
   memory, O(dim)/row, no per-row history buffer. Chosen over a hard sliding
   window (which would need a `W·dim` ring buffer for the same
   responsiveness). Held as `ProjectionRuntimeShared.recent_mean:
   Mutex<Option<RecentMeanTracker>>`, allocated for any MC-required
   workspace, mutated only under `commit_gate`.

2. **Drift trigger — cosine threshold.** On each commit (only when a mean is
   already pinned and was NOT (re)pinned in the same tx), compute
   `cos(recent_mean, pinned_mean)`. Fire when it falls BELOW
   `MEAN_DRIFT_COS_THRESHOLD = 0.95`. Calibration (HITL-ratified
   2026-05-30): a pathological topic-skewed pin sits ~0.82 cos to the true
   corpus mean; a healthy one ~0.998; 0.95 sits inside that gap, biased
   toward the healthy end.

3. **Debounce + falling-edge re-arm latch.** Fire only on a falling edge
   through the threshold (`armed` latch) AND only after
   `MEAN_RECOMPUTE_DEBOUNCE_ROWS = MEAN_VEC_PIN_THRESHOLD = 256` rows since
   the last action. Re-arm once cos recovers above the threshold. One
   sustained drift episode therefore triggers at most one recompute. The
   decision is made on a PROJECTED copy of the tracker over the current
   batch; the persistent tracker is folded post-commit so a rolled-back
   batch never moves it.

4. **200k cap + deferred notification.** Below `MEAN_RECOMPUTE_DYNAMIC_MAX =
   200_000` rows the detector recomputes SYNCHRONOUSLY in the same commit tx
   (reusing `recompute_mean_in_tx` → `run_pin_and_requantize_pass`), staging
   `EmbedderEvent::MeanVecRecomputed { trigger: DriftAuto }`. At/above the cap
   it SUPPRESSES the in-line recompute and instead surfaces
   `EmbedderEvent::MeanRecomputeDeferred { doc_count, drift_cos_bits }`
   (carrying the drift cosine as raw `f32` bits so the event enum keeps
   deriving `Eq`), so an operator can run the manual verb. The cap bounds the
   worst-case synchronous in-transaction re-quantize latency on ingest.

  plumbing + `pinned_this_tx` flag, the drift-detector block + the

## Why it was deferred (do not lose this)

HITL ruling 2026-05-31 (`dev/plans/runs/0.7.2-PR-2bc-decision.md` §2b):

1. **The recall premise was refuted.** The EU-7 "recall gap" was a
   MEASUREMENT artifact (corrected ANN recall@10 = 0.937, clears the 0.90
   floor). The mean is a **non-lever**: forcing the full-corpus mean on the
   real engine moved recall only +1.9pp, and `drift_cos_before = 1.0000`
   showed PR-2b had ALREADY converged the pinned mean to the true full-corpus
   mean during seeding. The auto-detector was built to fix a problem that
   does not exist for recall.

2. **Its benefit is unmeasured.** The only evidence it "works" is a SYNTHETIC
   topic-drift efficacy test. There is no production / real-corpus signal
   that auto-firing improves anything users see.

3. **It is permanent complexity on the hottest path.** Per-commit EWMA, a cos
   check every commit, the debounce/re-arm latch, the cap branch, and the
   `MeanRecomputeDeferred` plumbing all live in `commit_projection_outcomes`
   forever, for a speculative trigger.

4. **The manual verb already gives operators the lever.** An operator who
   observes drift runs `doctor recompute-mean` — zero standing cost, no
   speculative hot-path machinery.

## Revival criteria (0.8.x)

Revive ONLY when there is a **measured** production or real-corpus topic-drift
case where (a) recall/quality demonstrably degrades and (b) the manual
`doctor recompute-mean` verb cannot practically cover it (e.g. drift is
frequent/unpredictable enough that operator-driven refresh is impractical).
At that point the calibrated numbers above (0.95 / debounce 256 / EWMA 1/256 /
200k cap) are the starting point but MUST be re-validated against the real
drift event, not re-assumed. Re-introduction is additive (re-add the symbols
listed above; the `MeanRecomputeTrigger` enum and the recompute core are
already in place).

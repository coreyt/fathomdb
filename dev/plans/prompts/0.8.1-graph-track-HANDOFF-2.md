## 1. The goal of the FathomDB 0.8.1 phase
**Identify the *most valuable* design choices for FathomDB's memory-graph + retrieval,
using LongMemEval (LME) as the instrument — so we don't go down the wrong path.**

- FathomDB is an **embedded SQLite, single-writer, ≤1M-record, BYO-LLM** store (NOT a
  service). Footprint/latency constraints are real (`ADR-0.8.0-graph-model-and-edge-addressing.md`).
- **Two metrics, two loops** (plan §1): **Recall@K vs `answer_session_id`** (LLM-free,
  fast — the design-iteration instrument) and **end-to-end answer accuracy** (airlock
  reader + judge — slower, gates only). Most work runs in the cheap loop.
- **Scope locked (HITL 2026-06-14): LME only.** The community/summarization tier is
  **parked** (it serves sensemaking, not LME needle-recall — judge it on a sensemaking
  benchmark, not here). **Base-retrieval IS in scope** (the largest measured gap).
- The diagnosis found 3 separable causes of FathomDB's LME underperformance: **(A)**
  graph-arm hits invisible to the scorer (read-path), **(B)** ~1% extraction coverage,
  **(C)** base dense+FTS trails BM25 by ~10pp. The node/edge **model is fine**
  (fact-on-edge, bi-temporal, Zep-aligned). This is a plumbing + seeding + base-retrieval
  problem, not a model problem.

## 4. The one thing that reframes everything (from the G0 design review)
**`source_id` (E0a) is necessary but INERT — the graph arm's *seeding* is the real recall
blocker.** Seeds are doc nodes (`logical_id=None`, `lib.rs:5379`) → BFS frontier empty →
zero graph candidates, regardless of coverage. So fixing the carrier makes hits *visible*
but creates none. **The real unblock is a separate graph-arm *seeding* slice** (seed BFS
from query-matched *entities* / edge-fact hits, not doc nodes) — an R3 redesign, **out of
G0 scope**. This is why item #1 below gates the expensive experiments.

# 0.7.0 — Vector-retrieval cost research handoff

Drafted: 2026-05-26
Type: **RESEARCH-ONLY.** Do not write code, do not draft a slice
plan yet. Report findings; the slice plan follows after HITL
review of this research.

## Why this exists

The 0.7.0 perf-experiments campaign locked PCACHE2 + canonical-
SQLite pragmas as the AC-012 / AC-020 fix
(`dev/plans/0.7.0-HITL-recommendations.md`). The campaign also
measured AC-013 (vector retrieval latency) and AC-019
(concurrent-mixed read tail) at canonical-CI N=1M under the
locked stack (workflow run 26482279276):

- AC-013 p50 = **2048 ms**, p99 = **2327 ms** (seed cost 562 s).
- AC-019 baseline_p99 = 2096 ms, stress_p99 = 8388 ms.

ADR-draft budgets were 80 / 300 ms — measurement is 25–40× over.

**Root cause: per-query O(N) full-scan through the `vec0`
virtual table.** PCACHE2 / page-cache / pragmas cannot move
this; the bottleneck is the per-query distance computation
across 1M dim=768 vectors, not allocator / pcache contention.

The first instinct is "land an ANN index". HITL agrees that
*some* vector-cost fix needs to land in 0.7.0. But the
cheapest-effective path may not require an ANN index at all —
several optimizations sit between "current brute-force vec0"
and "full HNSW/IVF index", and they're often dramatically
cheaper to ship.

**Research the cheapest-effective tier first.** Drop down to
the next tier only if the prior tier cannot plausibly close
the budget gap.

## Research tiers (priority-ordered)

### TIER 1 — scan-optimization on existing `vec0`

Can we keep brute-force `vec0` but make each scan dramatically
cheaper? For each technique below, evaluate:

- **Mechanism** — how it reduces per-query work.
- **Expected gain** — order-of-magnitude estimate (cite
  benchmarks or theory; don't guess).
- **Implementation cost** — schema migration?
  application-side code? `sqlite-vec` already supports it?
- **Recall / correctness impact** — does it change query
  semantics or accept approximate results?
- **Compatibility with FathomDB** — interaction with
  `engine.search()`, vector kinds, embedder identity, the
  writer single-writer lock, snapshot semantics, projection
  cursor freshness (`AC-017`).

Techniques:

1. **Metadata prefiltering.** Filter the candidate set via SQL
   predicate BEFORE distance computation. E.g., `WHERE kind=?
   AND source_id=?` reduces 1M → 10K candidates, dropping
   per-query work 100×. What's `sqlite-vec`'s syntax for
   pre-filtered vector search? (`vec0` virtual table supports
   `WHERE` predicates on auxiliary columns — see upstream README.)
   How does FathomDB's `engine.search()` currently expose
   these predicates? Is there a search-API extension needed,
   or does the existing API already allow it?

2. **Partition keys.** Physically partition vectors by some
   key (vector kind, tenant, source) so each query scans only
   one partition. Two flavors: (a) SQL-level (multiple
   `vec0` tables, one per partition) or (b) logical (single
   table + indexed predicate). What's the cost vs gain at
   k=8–64 partitions for 1M rows? How would the partition key
   be chosen in FathomDB's domain (likely vector kind from
   `configure_vector_kind_for_test` in lib.rs)?

3. **Smaller embeddings (lower-dim models).** Switching from
   dim=768 (current `RETRIEVAL_VECTOR_DIM` in perf_gates.rs:153)
   to dim=384 (e.g., all-MiniLM-L6-v2) cuts distance-compute
   cost ~2×. Dim=256 or 128 (e5-small, bge-micro) cuts further.
   What's the recall impact? Is the embedder a runtime choice
   per `EmbedderIdentity`, or schema-pinned? What's the migration
   path for an existing corpus to a new embedder?

4. **Binary / int8 quantization.** Store each float32 component
   as a single bit (binary) or single byte (int8). Distance
   compute becomes Hamming (binary) or int8-SIMD. Typical
   gain: binary 10–30× speedup with ~80–90% recall@10; int8
   2–4× speedup with ~95–98% recall@10. Does `sqlite-vec`
   support quantized vectors natively? (Check `vec_int8` /
   `vec_bit` types in upstream.) If yes, what's the storage +
   query syntax? If no, is the quantization an application-
   side responsibility?

5. **Matryoshka embeddings / truncated dimensions.** Some
   modern embedders (`nomic-embed-text-v1.5`, OpenAI
   text-embedding-3-{small,large}) are trained so a prefix
   of the vector is itself a useful embedding. Storing the
   full vector but searching against a 128-byte prefix
   gives most of the recall at a fraction of the compute.
   Is this compatible with `vec0`'s schema? Is there a
   query-syntax knob (`vec_distance(..., :first_k_dims)`)?

6. **Caching / limiting candidate sets.** If queries are
   repeated (real workload) or arrive in batches, cache the
   top-N candidate IDs and re-rank only those. For AC-013's
   synthetic uniform-random queries this is no help, but for
   real workloads it can be order-of-magnitude. Also: limit
   the result set explicitly via `LIMIT k` in the SQL so
   `vec0` can early-exit (does it?).

**Tier 1 deliverable:** for each technique, a paragraph with
mechanism + gain + cost + recall + compatibility. End with a
**combined estimate**: if we stacked the 2–3 best Tier-1
techniques, would AC-013 plausibly land within 80–300 ms?
If yes, Tier 1 wins and Tiers 2/3 don't ship in 0.7.0. If no,
identify the residual gap.

### TIER 2 — vectorlite as a drop-in HNSW extension

`vectorlite` (<https://github.com/1yefuwang1/vectorlite>) is a
SQLite loadable extension that ships HNSW-indexed vector
search. It's the closest analogue to `sqlite-vec` but with an
ANN index built-in.

Investigate:

1. **Maintenance + maturity.** Last commit / release. Versions.
   Reported production usage. License. (BSL? Apache? MIT?)
   Compatibility with `libsqlite3-sys-0.28.0` (our current
   pinned version — see `crates/fathomdb-engine/Cargo.toml`).
2. **API shape.** How does `vectorlite` register vs SQLite?
   Is it a loadable C extension (like `sqlite-vec` /
   `sqlite3_vec_init`)? A virtual table? A set of SQL
   functions? Where would `vectorlite` plug into FathomDB's
   `register_sqlite_vec_extension()` path
   (`lib.rs:3107`-ish)?
3. **Coexistence with `sqlite-vec`.** Can both extensions be
   registered in the same process? Do they share the same
   `vec0` namespace or use different table types? Migration
   path from existing `vec0` corpus to `vectorlite` schema —
   is it a SQL-level INSERT or a full corpus rebuild?
4. **HNSW parameters exposed.** What `M`, `ef_construction`,
   `ef_search` knobs are exposed? Defaults sane for our
   dim=768 1M-row workload?
5. **Recall + latency benchmarks.** Has the author published
   benchmarks? At what corpus size, dim, recall floor? Compare
   to the AC-013 2048 ms measurement: would vectorlite
   plausibly land within 80–300 ms?
6. **Storage cost.** HNSW typically inflates storage 1.5–2× of
   raw vectors. At 1M × 768 × 4 bytes = 3 GB raw, that's an
   extra 1.5–3 GB. Is that acceptable for FathomDB's
   embedded-DB posture?
7. **Single-writer / lifecycle interaction.** FathomDB enforces
   exclusive `EngineRuntime::open` per database file
   (`crates/fathomdb-engine/src/database_lock.rs`). Does
   vectorlite's index format require a custom open/close
   sequence? Does it lock additional files?
8. **Snapshot / transaction semantics.** Does HNSW build on
   commit, or asynchronously? Does it interact with FathomDB's
   projection-cursor freshness (AC-017)?
9. **Fallback / coexistence with brute-force.** Can a query
   opt out of the index and fall back to brute-force `vec0`
   for correctness validation? Important for recall-validation
   testing.

**Tier 2 deliverable:** a verdict — `vectorlite` is (a)
shippable in 0.7.0 as the AC-013/019 lever, (b) shippable
with caveats, (c) not viable, with reasons cited. If (a) or
(b), sketch the integration shape and a slice-scope estimate.

### TIER 3 — broader ANN-index architecture

Pursue only if Tier 1 cannot plausibly close the gap AND
Tier 2 (vectorlite) is not viable. Same questions as the
prior research-prompt draft:

1. Does upstream `sqlite-vec` itself provide / plan an ANN
   index? (Check repo, release notes, roadmap.) If yes, what
   version, and is upgrade feasible?
2. Index algorithm choice — HNSW vs IVF vs quantization-based
   (PQ / SQ / OPQ). Drafter's pick for our workload.
3. Integration shape — four options:
   - A. Inside `sqlite-vec`'s `vec0` (if upstream gains
     indexed mode).
   - B. Separate SQLite virtual table / extension (other
     than vectorlite, which is Tier 2).
   - C. Rust-side in-process index (`instant-distance`,
     `hnsw_rs`, `usearch` Rust crate).
   - D. Hybrid — SQLite-side persistent index + Rust-side
     query layer.
4. Recall validation posture — proposed recall-floor
   expression for AC-013's RED test; ground-truth computation.
5. Slice-scope cost estimate — single slice / multi-phase /
   ship MVP with brute-force fallback.

**Tier 3 deliverable:** a recommended option with blast-radius
and slice-scope estimate. If Tier 3 is reached, the
0.7.0-vs-0.7.1 split question becomes load-bearing — call it
out for HITL.

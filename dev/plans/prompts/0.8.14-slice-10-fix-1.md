### Finding 1 [medium] — BM25F scorer is not tokenization-faithful (SUBSTANTIVE; the real fix)
`search_index_v2` recalls candidates with the FTS5 tokenizer `porter unicode61 remove_diacritics 2`,
but the in-engine BM25F scorer recomputes tf / df / field-lengths with a simpler lowercase-alnum
splitter (engine `src/lib.rs` ~10278/10303/10347/10425). Consequence: a stemmed or diacritic-variant
term recalled by `MATCH` gets scored as if the term were absent or rarer than it is → **wrong ranking
for stemmed/diacritic variants** (recall is intact; the BM25F math + live `weights`/`b`/`k1` are sound).

**Fix:** make the scorer measure terms under the SAME tokenization semantics as `search_index_v2`.
- Prefer reusing whatever tokenization the engine already relies on for FTS (there are existing
  tokenizer paths/tests — `corpus_fts`, the tokenizer test; find and reuse the canonical one rather
  than hand-rolling a second splitter). Apply the same porter-stem + unicode61 + remove_diacritics
  normalization to BOTH the indexed field text and the query terms before computing tf/df/lengths.
- If faithfully replicating porter/diacritic folding in Rust is disproportionate, an acceptable
  alternative is to derive the term statistics from FTS5 itself (so tf/df come from the same tokenizer
  that recalled the candidate) — choose the SMALLEST change that makes scoring tokenization-faithful.
- **RED→GREEN test:** add a case where the query term matches a document ONLY via porter stemming
  and/or diacritic folding (e.g. query "running" vs indexed "run", or an accented vs unaccented form).
  Assert the matching document is scored/ranked correctly (its term is counted, not treated as absent).
  The test must FAIL against the current lowercase-alnum splitter and PASS after the fix.

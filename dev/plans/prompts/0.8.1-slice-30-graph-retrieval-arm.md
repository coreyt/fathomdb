### 2b. Third-arm graph-retrieval fusion

Make the temporal fact-edge graph a **third RRF arm** alongside dense + lexical:

- Retrieve graph candidates from the `canonical_edges` substrate via a bounded BFS
  (depth ≤ 3, cap 50 — same constraints as `graph_neighbors`) seeded from the top-K
  node hits of the initial vector+text search. The arm's candidates are the
  **reachable nodes** (not the seed hits themselves — those are already in the other arms).
- Apply the **invalidate-not-accumulate** temporal filter:
  `e.t_invalid IS NULL OR datetime(e.t_invalid) > datetime('now')` AND
  `e.superseded_at IS NULL`. Edges that fail this filter MUST NOT contribute to the
  graph arm pool — they are logically dead facts.
- Score graph-arm candidates by a BFS-distance-decay function
  (e.g. `1.0 / (1.0 + hop_count)` or similar — document the formula in the design
  memo). Do NOT just copy the BFS hop count as a raw score; it must be in a range
  compatible with RRF weight combination.
- Add the graph arm as a **new parameter** to the RRF fusion (see design question 4
  in §3.0). The existing `fuse_rrf(vector_hits, text_hits)` call at line 5224 must
  be extended to include `graph_hits`. All existing `pr_g9_rrf_fusion.rs` tests must
  remain GREEN (the graph arm is empty by default in the unit tests, so this is
  achievable by accepting an empty vec for the third parameter).
- Gate the arm behind a `use_graph_arm: bool` flag (default `false`) surfaced on the
  public API. When `false` the arm is empty and the fused result is byte-identical to
  the pre-Slice-30 output (soft-fallback identity for the graph arm).

### 2c. ELPS round-2 field-consumption (baked into the arm)

These are **signed contract decisions** from the PREP doc — not optional design choices:

- **`entities[].synthesized: true`** — a dangling-endpoint node FathomDB synthesized;
  treat as **low-confidence identity**. Down-weight this node's graph-arm contribution
  (e.g. multiply its BFS-decay score by a penalty ≤ 0.5). Do NOT let a synthesized
  node anchor a high-confidence path. The design memo must name the penalty value.
- **`warnings.kind: "temporal_fallback"`** — an edge whose `t_valid` was defaulted to
  `created_at` (not text-grounded); the time is untrustworthy for temporal queries.
  **Exclude** these edges from the graph arm (treat as if `t_invalid` is now), OR
  down-weight them with a documented multiplier. The decision (exclude vs down-weight)
  must be made explicit in the design memo and tested in RED.
- **`warnings.kind: "capped"`** — a doc whose edges were truncated (extraction cap
  reached). Surface this in the graph arm's per-hit diagnostic metadata (a flag in
  the arm's candidate set, not silently dropped). Do NOT surface capped-edges as full
  arm candidates; surface the cap flag only so the caller knows the arm's recall is
  bounded for that doc.

   Whichever you choose, document that all existing `pr_g9_rrf_fusion.rs` tests must

2. **Graph-arm weight constant.** Add `pub const RRF_WEIGHT_GRAPH: f64 = N;` (analogous

   reasonable range is 1.0–3.0; start conservative (≤ 2.0) since we don't yet have
   R2 data to calibrate. The design memo must name the value.

3. **Graph arm candidate generation.** How do you seed the BFS?
   - Seed from the top-min(N, len) hits of the initial vector+text fused result (before
     the graph arm is added). State the seed count N (e.g. top-10 or top-20). Document
     that seed nodes themselves are NOT added to the graph arm (they are already ranked

     candidates.

     `superseded_at IS NULL` already present in `graph_neighbors_in_tx` (lib.rs:5572).
     Do NOT duplicate it; call or adapt the existing helper.

4. **Scoring formula for graph-arm candidates.** Document the BFS-decay formula
   (e.g. `score = RRF_WEIGHT_GRAPH / (RRF_K + hop_count)`). Justify why the resulting
   score range is compatible with the RRF accumulator that vector/text scores also
   contribute to.

5. **`SoftFallbackBranch` for graph-arm hits.** Add a new variant

   type). The design memo must state explicitly which choice is made and why.

6. **`temporal_fallback` handling.** Per the PREP contract: edges whose `t_valid` was
   defaulted (not text-grounded) are untrustworthy for temporal queries. State whether

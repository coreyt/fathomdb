# PR-2a launch prompt — mean-centering recall investigation (experiment)

Paste the "Agent prompt" section into a fresh agent session. This is an
**experiment slice** (EU-0-style empirical work, no production behavior change),
inserted into the 0.7.2 campaign **ahead of PR-2** because EU-7's real-corpus
recall (0.828) is likely depressed by a *fixable config artifact* (the corpus
mean is pinned on the first 256 ingested docs — all one source), not by an
inherent embedder limit. PR-2 must not lock a floor (~0.79) against a possibly-
artificial number until that is decomposed.

## Agent prompt

You are running **PR-2a**, the mean-centering recall investigation for fathomdb
0.7.2. **Goal: decompose the 0.933 (EU-0 synthetic-pipeline) → 0.828 (EU-7 real
production path) recall@10 gap into named, quantified contributors, and return a
GO/NO-GO on a mean-representativeness fix — so PR-2 locks the floor against the
right number.** This is an experiment: no engine/ADR/floor changes. Work on
`main` at `/home/coreyt/projects/fathomdb`. Save artifacts under
`dev/research/pr-2a/` and `dev/plans/runs/`. No push. ASCII only.

### Why (the hypothesis)

EU-7 measured recall@10 = 0.828 (N=7667, K=192, mean-centering ON) through the
real production path. EU-0 measured 0.933 for the same model/K/centering via an
offline Python pipeline. The EU-7 honesty report attributes most of the gap to
**the production mean being pinned on the first 256 ingested docs**, which in the
corpus's deterministic ingest order are all `bahmutov_dailylogs` — a single,
non-representative source. Mean-centering quality depends on the mean being
representative of the corpus; a single-source mean under-centers everything else.

If that hypothesis holds, recall is recoverable with a **representativeness fix**
to the mean (a config change), and PR-2 should lock a *higher* floor — not
cement 0.79. If it does not hold, the gap is inherent (query methodology, real-
vs-synthetic, quantization) and PR-2 locks ~0.79 honestly. **PR-2a decides
which.**

### Experiment — decompose the gap

Hold **everything else fixed** (bge-small, dim 384, K=192, sign-bit quant, f32
rerank, 512-token truncation). Vary ONE factor at a time. Prefer the **offline
path** (reuse EU-0's saved bge-small doc/query vectors in `dev/research/eu-0/`,
or recompute them once) so it is fast and directly comparable to EU-0's 0.933;
use the same percentile-bootstrap (1000 resamples) for CIs.

Measure recall@10 under these **mean strategies** (same corpus, same query set
within a comparison):
- **M0 — first-256 (ingest order)**: the production behavior. Reproduce ~0.83.
  Replicate the engine's doc ordering (corpus_subset: sorted by filename then
  doc_id) so "first 256" matches what the engine actually pinned.
- **M1 — full-corpus mean**: mean over all docs (EU-0's behavior). Upper bound.
- **M2 — representative 256-sample**: mean over a stratified/shuffled 256-doc
  sample spanning all sources (the realistic production fix — you do not need
  the whole corpus, just a representative sample). Report a couple of seeds.
- **M3 (optional) — no centering**: isolates how much centering buys at all.

Also **decompose the non-mean factors** so the gap attribution is complete:
- **Query set**: run M1 with EU-0's exact query set/methodology (quotas +
  target_doc_idx) vs EU-7's title-or-lead synthesis, to quantify how much of the
  0.933→0.828 gap is query-methodology vs mean. (If you reproduce EU-0's 0.933
  with M1 + EU-0 queries, the offline pipeline is validated.)
- **Truncation**: note whether 512-token truncation materially changes recall on
  the long-doc sources (cnn_dailymail) — measure with/without as a sensitivity
  check (truncation stays ON in production regardless; this only attributes the
  gap).
- **Real-engine path delta**: if feasible, cross-check one strategy through the
  EU-7 Rust harness (engine path) vs the offline pipeline to bound the
  "production path vs offline" residual.

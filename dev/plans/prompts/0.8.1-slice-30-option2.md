## §0 Purpose and Context

Slice 30 shipped the R3 graph-retrieval arm (`use_graph_arm=True`), but the Slice 25
R2 eval run had NULL deltas for all memory classes (temporal / multi_hop /
knowledge_update / multi_session) because:

1. The frozen gold set (`ir-c-reused-v1`) has only `factoid` and `exploratory` classes.
2. The FathomDB arm ran FTS-only (`use_default_embedder=False`, `use_graph_arm=False`)
   — no ELPS extraction, so `canonical_edges` was empty.

This slice (Option 2) fixes both problems for a **pipeline integration test**:

- Build a **live ELPS harness** that calls the airlock LLM (claude-haiku) to extract
  entities and temporal-relation edges from a small corpus sample.
- Build a **memory-class gold generator** that uses claude-sonnet to write
  temporal / multi_hop / knowledge_update / multi_session QA pairs from those
  same documents.
- Modify `r2_parity_eval.py` to accept the live harness and the extra gold, and run
  FathomDB with `use_graph_arm=True`.
- Run the eval and record non-null per-class deltas → unblocks Slice 30 go/no-go.

**Numbers are not benchmark-significant.** The corpus sample is small (~100 docs)
and the gold is LLM-generated (not human-labeled). The only DoD that matters:
the pipeline runs end-to-end without crashing and produces non-null
`recall_at_k` for at least two of the three memory classes.

## §2 Protocol Reference — `fathomdb.extract.v1`

The engine spawns the harness as a subprocess and communicates over stdin/stdout
using NDJSON (one JSON object per line, `\n`-terminated). Diagnostics go to stderr.

### Handshake

Engine → harness (stdin):
```json
{"protocol": "fathomdb.extract.v1", "type": "hello"}
```

Harness → engine (stdout):
```json
{
  "protocol": "fathomdb.extract.v1",
  "type": "ready",
  "schema_version": 1,
  "provider": "airlock-claude-haiku",
  "model": "<model-name>",
  "supports": {},
  "max_docs_per_request": 8
}
```

### Extraction loop

Engine → harness (stdin):
```json
{
  "protocol": "fathomdb.extract.v1",
  "type": "extract",
  "request_id": "<uuid>",
  "documents": [
    {"source_doc_id": "doc-001", "kind": "doc", "created_at": "2025-01-01T00:00:00Z", "body": "..."},
    ...
  ]
}
```

Harness → engine (stdout):
```json
{
  "protocol": "fathomdb.extract.v1",
  "type": "result",
  "request_id": "<same uuid>",
  "entities": [
    {"name": "Alice Smith", "type": "person", "aliases": ["Alice", "A. Smith"]},
    ...
  ],
  "edges": [
    {
      "from_entity": "Alice Smith",
      "to_entity": "Acme Corp",
      "relation": "works_at",
      "body": "Alice Smith joined Acme Corp in March 2023.",
      "t_valid": "2023-03-01T00:00:00Z",
      "t_invalid": null,
      "confidence": 0.92,
      "source_doc_id": "doc-001",
      "source_span": null
    },
    ...
  ],
  "warnings": []
}
```

On error, return:
```json
{
  "protocol": "fathomdb.extract.v1",
  "type": "error",
  "request_id": "<same uuid>",
  "error_code": "extraction_failed",
  "detail": "<message>"
}
```

The engine closes stdin when it's done. The harness must exit cleanly when stdin
is exhausted (EOF on the stdin loop).

## §3 Memex Extraction Prompt Template

Copy this verbatim into `elps_live_harness.py`. Do NOT import from the memex package.

### System prompt

```
You are an information-extraction engine. From the single document supplied by the user, extract knowledge as a graph of entities and directed, dated fact-edges, and return it as structured JSON matching the required schema.

ENTITIES
- Identify the real-world entities mentioned (people, organizations, places, artifacts, concepts, etc.).
- Each entity has a `name`, a `type`, and optional `aliases` (other surface forms used for the same entity in this document).
- Be conservative about identity: when two mentions MIGHT refer to different real-world entities, emit them as DISTINCT entities rather than merging them. Do not collapse homonyms.

EDGES (directed, dated facts)
- Each edge is a single fact: `from_entity` -> `to_entity` with a short, lowercase `relation` label (e.g. `works_at`, `founded`, `located_in`).
- `body` is the minimal span of source text that states the fact.
- Reference entities by the exact `name` you used in the entities list.

TEMPORAL RULES (load-bearing)
- `t_valid` is the time the fact BECAME TRUE as stated in the text. If the text states no such time, use the document's `created_at`. NEVER use the current time or ingestion time ("now").
- Set `t_invalid` ONLY when the text gives explicit evidence the fact ended; otherwise leave it null.

SUPERSEDES (contradicting / replacing an earlier fact)
- When a fact in this document SUPERSEDES or CONTRADICTS an earlier fact (e.g. a new role, status, or value that replaces a prior one), emit the NEW fact with its own `t_valid` and set `supersedes_prior` to true, with a short `prior_body` describing the prior fact it replaces.
- Do NOT delete or invalidate the prior fact yourself — only emit the new fact and flag that it supersedes a prior one.

CONFIDENCE
- `confidence` in [0,1] is the calibrated probability that the fact is correct AND supported by the cited span. Emit ALL facts you find — do not self-censor low-confidence facts behind a threshold; report the low confidence instead.

SOURCE SPANS
- When you are confident, provide `source_span` as [start, end) character offsets into the supplied document body that locate the supporting text. If you cannot locate it reliably, omit `source_span`.

Return ONLY valid JSON matching this exact schema — no markdown fences, no prose:
{
  "entities": [{"name": str, "type": str, "aliases": list[str]}],
  "edges": [
    {
      "from_entity": str,
      "to_entity": str,
      "relation": str,
      "body": str,
      "t_valid": str,
      "t_invalid": str | null,
      "confidence": float,
      "source_doc_id": str,
      "source_span": [int, int] | null
    }
  ],
  "warnings": []
}
```

### User message (one per document in the batch)

```
Extract from this single document.
doc_id: {doc_id}
kind: doc
created_at: {created_at}
body:
{body}
```

Call the LLM once **per document** (not per batch). Consolidate the results for all
documents in a batch into a single `result` response.

If the LLM returns unparseable JSON or an error, emit a `"warnings"` entry:
```json
{"kind": "extraction_failed", "source_doc_id": "<id>", "detail": "<reason>"}
```
and continue — do not crash the harness.

## §5 `elps_live_harness.py` Design

```
src/python/eval/elps_live_harness.py
```

Executable Python script (add `#!/usr/bin/env python3`). Invoked as:
```
python src/python/eval/elps_live_harness.py
```

Key design points:
- Single file, no imports from `fathomdb` (it's a subprocess, not part of the SDK).
- Use `httpx` for the LLM call. If `httpx` is not installed, fall back to
  `urllib.request` (stdlib — always available).
- Read env vars `ELPS_LLM_BASE_URL`, `ELPS_LLM_API_KEY`, `ELPS_LLM_MODEL` at
  startup; log them to stderr.
- Run the handshake: read `hello` from stdin, write `ready` to stdout.
- Main loop: `for raw_line in sys.stdin` → parse → dispatch `extract` → call LLM
  per document → assemble result → write to stdout.
- `sys.stdout.flush()` after every write (engine waits on the response).
- Call the LLM with `response_format: {"type": "json_object"}` if the model
  supports it; otherwise parse the text response as JSON (strip markdown fences).
- `max_tokens`: 2048 per document call.
- Timeout: 60 seconds per LLM call; on timeout emit a warning and return empty
  entities/edges for that doc.

**Stub mode for tests**: if `ELPS_STUB_MODE=1` is set, skip the LLM call and
return a fixed canned response instead:
```json
{
  "entities": [{"name": "TestEntity", "type": "concept", "aliases": []}],
  "edges": [{
    "from_entity": "TestEntity", "to_entity": "TestEntity",
    "relation": "self_ref", "body": "stub", "t_valid": "2025-01-01T00:00:00Z",
    "t_invalid": null, "confidence": 0.5, "source_doc_id": "<doc_id>",
    "source_span": null
  }]
}
```
This lets the test suite exercise the full protocol loop without a real LLM call.

## §6 `gold_gen.py` Design

```
src/python/eval/gold_gen.py
```

Standalone script invoked as:
```bash
python src/python/eval/gold_gen.py \
  --docs-dir <raw-dir>       \   # directory of .jsonl corpus files
  --limit 100                \   # number of docs to sample (default 100)
  --out <path>               \   # output gold JSON file
  --base-url http://localhost:4000/v1 \
  --api-key sk-airlock-mk    \
  --model claude-sonnet
```

### Design

1. Load up to `--limit` documents from `--docs-dir/*.jsonl` (same logic as
   `_load_documents()` in `r2_parity_eval.py`: read `doc_id`/`id` + `body`).

2. Group documents into batches of 5-8 (for multi-doc questions like multi_hop
   and multi_session).

3. For each batch, call claude-sonnet with a prompt like:

**System:**
```
You are a question-generation assistant for evaluating memory retrieval systems.
Generate questions that require temporal reasoning, multi-hop inference, or
knowledge-update awareness to answer correctly. Focus on questions whose
answers require retrieving specific documents.
Return ONLY valid JSON. No prose.
```

**User (per batch):**
```
Below are {n} documents. Generate memory-class QA pairs for these documents.
Generate 1-2 questions per class where possible.

Classes to generate:
- "temporal": questions about WHEN something happened (requires a specific date or time period in the answer)
- "multi_hop": questions requiring information from TWO OR MORE of these documents combined
- "knowledge_update": questions where the answer in one document SUPERSEDES or CONTRADICTS an earlier doc
- "multi_session": questions that make sense only when reading across multiple separate conversations or sources

For each question:
- "query_id": unique string like "mc-001"
- "query": the question text
- "query_class": one of the four classes above
- "required_evidence": list of {"doc_id": "<id>"} for each doc needed to answer
- "answers": list of short answer strings (1-3 words or a date)

Return:
{
  "queries": [
    {
      "query_id": "mc-001",
      "query": "When did Alice Smith join Acme Corp?",
      "query_class": "temporal",
      "required_evidence": [{"doc_id": "doc-001"}],
      "answers": ["March 2023", "2023-03"]
    },
    ...
  ]
}

Documents:
{for each doc: "--- doc_id: <id>\n<body>\n"}
```

4. Collect all generated queries across batches, assign unique `query_id`s
   (prefix `mc-`, sequential numbering), de-duplicate questions.

5. Write output as:
```json
{
  "version": "memory-class-gen-v1",
  "generator_model": "<model>",
  "n_docs_sampled": <N>,
  "queries": [...]
}
```

6. If a batch call fails or returns invalid JSON, log to stderr and skip (don't
   crash). Emit a warning at the end: `"WARNING: N batches failed"`.

7. Add `query_id` counter at module level (threadsafe enough for single-threaded
   script).

## §7 Modifications to `r2_parity_eval.py`

Add these four CLI arguments to the `main()` parser:

```python
parser.add_argument("--elps-harness", default=None,
    help="Path to elps_live_harness.py; if set, ingest sample via ingest_with_extractor")
parser.add_argument("--elps-limit", type=int, default=100,
    help="Max docs to ingest via ELPS (Option 2 sample size)")
parser.add_argument("--use-graph-arm", action="store_true", default=False,
    help="Set use_graph_arm=True in FathomDBAdapter")
parser.add_argument("--extra-gold", default=None,
    help="Path to an additional gold file to merge with --gold (for memory-class QA)")
```

### Change 1: extra gold loading

After loading the main gold file, if `--extra-gold` is set, load and merge it:
```python
if args.extra_gold:
    extra_gold_path = Path(args.extra_gold)
    extra_raw = json.loads(extra_gold_path.read_text(encoding="utf-8"))
    # _parse_gold expects {"queries": [...]}
    extra_queries = _parse_gold(extra_raw)
    # Extend the harness gold (need to make R2Harness accept extra queries or
    # rebuild harness with merged query list)
```

The cleanest way: expose a `load_gold(path)` → `list[GoldQuery]` helper (or
just call `_parse_gold`). The `R2Harness` is constructed with a `gold_path`
argument that it loads internally. Either add an `extra_queries` parameter to
`R2Harness.__init__`, or pre-merge the gold files before passing to `R2Harness`.

Pre-merge is simpler: write the merged list to a temp file, pass it to
`R2Harness`. Even simpler: add an `extra_queries: list[GoldQuery]` param to
`R2Harness.__init__` and extend the internal query list. Implement whichever is
less invasive.

### Change 2: `_build_fathomdb` ELPS path

Modify `_build_fathomdb` signature:
```python
def _build_fathomdb(
    documents: dict[str, str],
    db_path: Path,
    *,
    batch: int = 500,
    elps_harness_cmd: list[str] | None = None,
    elps_limit: int = 100,
    use_graph_arm: bool = False,
) -> tuple[Optional["FathomDBAdapter"], Optional[dict[str, str]]]:
```

When `elps_harness_cmd` is not None:
1. Take the first `elps_limit` items from `documents.items()` as the ELPS sample.
2. Build a reverse map: `body_to_doc_id: dict[str, str] = {body: doc_id for doc_id, body in elps_sample}`.
3. Open the engine with `Engine.open(str(db_path), use_default_embedder=False)`.
4. Call `engine.ingest_with_extractor(elps_harness_cmd, elps_docs)` where
   `elps_docs = [{"source_doc_id": doc_id, "body": body} for doc_id, body in elps_sample]`.
5. For remaining documents (items beyond elps_limit), ingest via plain `engine.write()`,
   building `cursor_to_doc` from `receipt.row_cursors`.
6. Define `doc_id_of` as:
   ```python
   def doc_id_of(sh) -> str:
       if int(sh.id) in cursor_to_doc:
           return cursor_to_doc[int(sh.id)]
       return body_to_doc_id.get(sh.body, str(sh.id))
   ```
7. Build `FathomDBAdapter(engine, doc_id_of=doc_id_of, use_graph_arm=use_graph_arm)`.

When `elps_harness_cmd` is None (original path): no change from current code.

### Change 3: naive-RAG and FathomDB see the same doc subset

When `--elps-harness` is set and `--elps-limit` is less than the total corpus
size, the retrieved hits from the graph arm will reference ELPS-extracted docs.
For a fair comparison, the `naive_rag` adapter and `fathomdb` adapter should see
the **same document set** (the elps_sample only). Restrict `documents` passed to
`NaiveRAGAdapter` to the elps_sample when `--elps-harness` is active.

Add to `main()`:
```python
if args.elps_harness:
    # Restrict both adapters to the ELPS sample for fair comparison
    items = list(documents.items())[:args.elps_limit]
    documents = dict(items)
```
This restriction must happen BEFORE constructing NaiveRAGAdapter.

### Change 4: pass args through

In `main()`, wire the new args into `_build_fathomdb`:
```python
elps_cmd = ["python", args.elps_harness] if args.elps_harness else None
fdb, blk = _build_fathomdb(
    documents, Path(args.db_path),
    elps_harness_cmd=elps_cmd,
    elps_limit=args.elps_limit,
    use_graph_arm=args.use_graph_arm,
)
```

## §11 Guardrails

- **No Rust changes.** If `ingest_with_extractor` has a bug, escalate — do not
  patch the Rust engine in this slice.
- **No schema bumps.** `SCHEMA_VERSION` is 15 and stays 15. If the ELPS ingest
  fails because of a schema issue, document it in `blockers_encountered` and skip
  the FathomDB arm.
- **No mocking FathomDB's SQLite.** Tests that need an engine use a temp `.sqlite`
  file (see T3 above). The `ELPS_STUB_MODE=1` flag stubs the LLM only.
- **No benchmark claims.** The output artifact must include the note field verbatim:
  `"Option 2 integration test — numbers are not benchmark-significant (N=100 corpus
  sample, LLM-generated gold). Validates ELPS→graph→R2 pipeline end-to-end."`
- **Identical-answerer invariant.** The answerer LLM/prompt must not change between
  FathomDB and naive-RAG. `FathomDBAdapter` and `NaiveRAGAdapter` are retrieval-only;

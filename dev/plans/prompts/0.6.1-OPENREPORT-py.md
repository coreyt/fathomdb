## Hard constraints

- **TDD red-green-refactor** per `feedback_tdd`. Failing test first.
- **No native (`fathomdb-engine`) code changes.** Native shape is
  already correct (`lib.rs:541-548`). Bindings surface what's
  already there; do not extend or rename native fields.
- **Native field names are authoritative.** Surface them under the
  exact same snake_case identifiers (`schema_version_before`,
  `schema_version_after`, `migration_steps`, `embedder_warmup_ms`,
  `query_backend`, `default_embedder`). Do not invent paraphrases;
  do not omit the extended native fields.
- **No new verb.** `Engine.open` keeps its name; do NOT introduce
  `Engine.open_with_report` or similar — a new verb violates the
  patch-release contract (`ADR-0.6.0-no-shims-policy` § 54).
- **Legacy-callsite compatibility is mandatory and is the primary
  gating test.** `engine = Engine.open(...)` must continue to work
  bit-for-bit identically as in 0.6.0; the regression test
  asserting this is the slice's gate, not an optional add-on. Use
  a single-return-object pattern (e.g., `OpenedEngine` NamedTuple
  that is type-compatible with `Engine` for legacy callers, OR a
  wrapper class with `__getattr__` delegation, OR — preferred —
  return a `NamedTuple` and update the python-side `Engine.open`
  wrapper so legacy single-name binding still receives the engine
  handle directly). Surface the exact mechanism at codex review.
- **Binding API shape decision is locked at codex review** (within
  the constraints above). Do not flip unilaterally.
- **No new ADR.** OpenReport semantic is already locked in
  `dev/design/engine.md`.

# 0.7.0 handoff — perf-only minor release

You are starting (or resuming) the **0.7.0 release line**. This doc
is the spawn-context: it tells you where things stand, what 0.7.0
is for, and which artifact owns each step. **It is not the
planning-slice prompt** — that lives at
`dev/plans/prompts/0.7.0-planning.md` and is authoritative for the
planning slice's task list, ADRs, and outputs.

## What 0.7.0 is

0.7.0 is a **perf-only single-thrust minor release** (locked HITL
2026-05-24). One goal: bring the AC-012 / AC-013 / AC-019 / AC-020
canonical-runner numbers into a defensible, empirically-pinned
budget by (a) revising the budgets against real measurements and
(b) tuning SQLite + FathomDB engine-side levers to meet them.

The originally-planned 0.7.0 anchor (logical-id verbs + canonical-
identity substrate) moved to **0.8.0** alongside the Memex
knowledge-store / retrieval features that consume the substrate.
See `dev/roadmap/0.8.0.md`. **Do not carry substrate / verb /
migration / knowledge-store work into 0.7.0.**

### Step 3 — Implementation orchestration (claude / codex TDD loop)

Per-slice cadence per `dev/design/orchestration.md §§ 2-9`:

1. **Spawn implementer (Claude)** in an isolated worktree.
2. **TDD red-green-refactor** per `feedback_tdd`:
   - **RED:** add the perf-gate test (or behavioral test for
     caller-spec changes) asserting the revised budget. Confirm
     RED at current baseline.
   - **GREEN:** implement the engine-side lever; confirm the test
     flips to GREEN on the canonical runner.
   - **REFACTOR:** rename, factor, delete dead code; keep GREEN.
     Net-negative-LoC where possible per
     `feedback_reliability_principles`.
3. **Spawn reviewer (Codex, read-only)** per § 3. Verdict PASS /
   CONCERN / BLOCK; orchestrator promotes per § 4 (Claude writes
   the promotion artifact — codex sandbox cannot write).
4. **Cherry-pick to `main`** once verdict is PASS (or orchestrator
   override on CONCERN per § 7). 0.7.0 work lives on `main`
   directly (same posture as 0.6.1 per HITL 2026-05-23).
5. **Fix-1 loop** per § 6 on BLOCK / CONCERN — same worktree, new
   commit, re-run reviewer. Iterate until PASS or override.
6. **Slice closure artifact** per § 8:
   `dev/plans/runs/0.7.0-<slice>-output.json`.

**Slice ordering rules (refined in the plan doc):**

- Diagnostic levers before structural ones, per
  `dev/notes/performance-whitepaper-notes.md § 7.1`.
- Cheap caller-spec changes (top-K `LIMIT` cap) may parallelise
  with PRAGMA-level levers — they touch separate surfaces.
- PCACHE2 / WAL2 / vendor-swap is **sequential**. Pick ONE per the
  AC-020 ADR; do not stack. A second lever only fires if the first
  does not close the gap and HITL re-opens the ADR.
- Revised-budgets ADR lock is a prerequisite for any perf-gate
  RED-test author-step.
- No Pack 5/6/6.G reverted-experiments ledger lever may be
  re-proposed without an explicit HITL override.

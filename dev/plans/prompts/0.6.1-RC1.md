## Tasks

1. Pre-flight:
   - Confirm BUMP commit on the release branch; `main` either FFed
     or about to be (verify `082321e`-style env-override gate
     references the right ref).
   - `scripts/set-version.sh --check-files` GREEN.
   - `cargo audit` / `pnpm audit` / `pip-audit` GREEN.
   - Dry-run rehearsal via
     `gh workflow run release.yml --ref 0.6.1-rewrite -f dry_run=true`.
     Wait for completion; assert all 8 tiers GREEN.
   - **HITL approval to push real tag.**
2. Tag push:
   - `git tag v0.6.1-rc.1 <BUMP-sha>`.
   - `git push origin v0.6.1-rc.1`.
3. Monitor `release.yml` per-tier:
   - cargo publish (8 axis-W crates serial; axis-E `fathomdb-
embedder-api` already at 0.6.0 — skip OR re-publish-noop per
     workflow shape).
   - PyPI publish.
   - npm publish.
   - Post-publish smoke: install from each registry; exercise the
     5-verb canonical surface + the new OpenReport return shape;
     assert GREEN.
   - Co-tagging assert: all three registries agree on version.
4. Apply known-flake retry-once list per
   `dev/plans/0.6.1-implementation.md` § "Pre-RC1 checkpoint" only
   on the listed flakes; otherwise STOP and surface.
5. Create GitHub Release with `isPrerelease=true`. Body mirrors
   `docs/release-notes/0.6.1.md`.

## Hard constraints

- **HITL approval required before tag push.** RC1 = first
  irreversible action.
- **No tag re-use.** If RC1 partial-publishes, next attempt is
  `v0.6.1-rc.2` (per 0.6.0 RC sequence precedent).
- **No retry on `verify-release-gates` failure** referencing the
  ref shape — surface immediately; that's `082321e`-class.
- **No workflow edits at tag time.** If release.yml needs a fix,
  STOP, land a WF-FIX slice, then re-tag as rc.2.

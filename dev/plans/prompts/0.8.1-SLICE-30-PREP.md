## Objective (from the contract)

Make the temporal fact-edge graph a **third retrieval arm**: project edge `body` → FTS + vector
candidates + a **bounded BFS** over the `canonical_edges` substrate, **fused as a third RRF arm** with the
dense + lexical arms, under the **invalidate-not-accumulate** temporal filter
(`t_invalid IS NULL OR t_invalid > now` AND `superseded_at IS NULL`). Fact-on-**EDGE** (Graphiti-shaped);
HyperGraphRAG fact-NODES remain the H6 escape hatch. **Payoff is mostly OFF the Recall@K axis** → judged on
**R2's** temporal / multi-hop / knowledge-update classes, not first-stage Recall@K.

## Round-2 ELPS field-consumption decisions to bake into the R3 design (NEW — decide explicitly in-slice)

Today FathomDB **tolerates but does not consume** the round-2 extraction fields. Slice 30 is where R3
decides whether/how to use them; make these explicit contract choices, not silent code:

- **`entities[].synthesized:true`** (a dangling endpoint FathomDB-synthesized, `type:"unknown"`): treat as
  a **low-confidence identity** — down-weight in graph centrality / the arm's node scoring, and prefer it
  as a merge candidate. Do not let a synthesized node anchor a high-confidence path.
- **`warnings.kind:"temporal_fallback"`** (an edge whose `t_valid` was *defaulted* to `created_at`, not
  text-grounded): the edge's time is **untrustworthy for temporal queries**. R3 should **flag/exclude or
  down-weight** these in the invalidate-not-accumulate + temporal-filter logic, so a defaulted date can't
  pose as a real event time in temporal-class eval. (This is the provenance signal we added the kind for.)
- **`warnings.kind:"capped"`**: a doc whose edges were truncated → a known recall ceiling for that doc;
  surface in the R3 results/diagnostics, not silently.
These consumption choices materially affect R3's temporal/multi-hop performance → they feed the **R3
go/no-go package** for HITL.

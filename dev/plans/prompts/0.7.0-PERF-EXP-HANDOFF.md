## CLOSED — PCACHE2 alignment-bug fix recipe (kept for record)

The 3429cf6 prototype activated correctly but SIGSEGV'd during
xFetch. The actual root cause turned out to be **handle-pointer
stability** — xFetch returned a freshly-`Box::into_raw`'d
`sqlite3_pcache_page` on each call, but SQLite uses the pointer
as the page handle and expects the *same* address back for
repeated fetches of a pinned key. Fix: embed `sqlite3_pcache_page`
inline in the Page struct so its address is stable.

Original alignment hypothesis below was *not* the root cause, but
the rewrite applied it as cheap hardening (`alloc_zeroed` with
explicit `Layout` instead of `Vec<u8>`).

### Fix recipe (suggested approach) — historical

Replace the `Vec<u8>` in `struct Page` with two separately-allocated,
explicitly-aligned regions. Concrete steps in
`src/rust/crates/fathomdb-engine/src/pcache2.rs`:

1. Pull `std::alloc::{alloc_zeroed, dealloc, Layout}` into scope.
2. Change `struct Page` to:

   ```rust
   struct Page {
       buf_ptr: *mut u8,
       extra_ptr: *mut u8,
       sz_page: usize,
       sz_extra: usize,
       align: usize,           // typically 8; verify against SQLITE_PAGE_ALIGN
       pinned: bool,
   }
   impl Drop for Page {
       fn drop(&mut self) {
           // SAFETY: pointers were obtained from alloc_zeroed with
           //   the same Layout; non-null guaranteed unless sz == 0.
           unsafe {
               if !self.buf_ptr.is_null() {
                   dealloc(self.buf_ptr,
                       Layout::from_size_align_unchecked(self.sz_page, self.align));
               }
               if !self.extra_ptr.is_null() && self.sz_extra > 0 {
                   dealloc(self.extra_ptr,
                       Layout::from_size_align_unchecked(self.sz_extra, self.align));
               }
           }
       }
   }
   ```

3. In `xCreate` / xFetch's create-branch, allocate each region via
   `alloc_zeroed(Layout::from_size_align(sz_page, 8).unwrap())`
   (and same for extra; if `sz_extra == 0`, set the pointer to a
   stable non-null sentinel — SQLite has tolerated this in past
   builds — or to a 1-byte dummy alloc).
4. Update `page_repr` to read `buf_ptr` / `extra_ptr` directly.
5. Update `xUnpin`'s page lookup (currently matches by
   `p.buffer.as_ptr() as usize == buf_addr`) to instead use the new
   `buf_ptr` field. This is also a perf-correctness improvement:
   the linear scan can stay for the prototype but should be replaced
   with a reverse index later.
6. Add a `static_assert!`-style debug_assert at xFetch entry:
   `debug_assert!((sz_page as usize) % 8 == 0)` — if SQLite ever
   passes an odd size, the bug returns.

## Why this exists

The 0.7.0 perf-experiments campaign locked PCACHE2 + canonical-
SQLite pragmas as the AC-012 / AC-020 fix
(`dev/plans/0.7.0-HITL-recommendations.md`). The campaign also
attempted canonical-CI N=1M measurement of AC-013 (vector
retrieval latency) and AC-019 (concurrent-mixed read tail)
under the locked stack (workflow run 26482279276).

The measurement returned **AC-013 p50 = 2048 ms, p99 = 2327
ms**. The ADR-draft budgets were 80 / 300 ms — the measurement
is 25–40× over budget. AC-019 stress_p99 = 8388 ms (passes its
relative bound but is absolute-scale slow).

**Root cause: per-query O(N) full-scan through the `vec0`
virtual table — no ANN index.** PCACHE2 / page-cache / pragmas
cannot move this. The fix is an **approximate-nearest-neighbour
index** so per-query cost drops from O(N) to O(log N) or
O(√N).

**HITL decision 2026-05-26: land an ANN index in 0.7.0.** This
research informs *what* index, *how* it integrates with the
existing `sqlite-vec` extension, and *what* the slice scope
should be.

## Specific questions to answer

### Q1 — Does `sqlite-vec` itself provide an ANN index?

`sqlite-vec` by Alex Garcia is the extension currently registered
(see point 6 above). Check:

- Latest upstream version + what it ships:
  https://github.com/asg017/sqlite-vec
- Whether the `vec0` virtual table supports any indexed mode
  (HNSW / IVF / quantization-based) — read the upstream README,
  release notes, and any roadmap docs. Note that as of late
  2025 sqlite-vec was still doing brute-force; check if that
  has changed.
- If sqlite-vec gained an indexed mode after our current pinned
  version, what's the upgrade cost / breaking changes?
- If sqlite-vec is still brute-force, is there a sibling /
  fork / alternative SQLite-native vector index we could swap
  to or compose with?

Report: **upstream status, version we'd need, what it would
buy us empirically** (cite any published benchmarks).

### Q2 — Index algorithm choice

Three families to evaluate. For each, summarize how it'd fit
this project (no implementation — just shape + tradeoffs):

- **HNSW (Hierarchical Navigable Small World)**. Industry
  standard. Recall ~99% with good params. Build cost moderate.
  Memory: typically 1.5–2× raw vector size. Per-query O(log N).
- **IVF (Inverted File / k-means-clustered)**. Lower memory
  than HNSW; tunable recall/latency via `nprobe`. Per-query
  O(√N). Build cost: one k-means pass (~minutes at 1M).
- **Quantization-based (PQ / SQ / OPQ)**. Compresses vectors.
  Often combined with IVF (`IVF-PQ`). Best for memory-
  constrained / billion-scale; overkill at 1M.

Report: **drafter's recommendation + why**, and the recall vs
latency tradeoffs each implies. For this project, what's the
"good enough" recall floor for AC-013/019 to be useful?

### Q3 — Integration shape

Where would the ANN index live? Three architectural options:

- **A — Inside `sqlite-vec`'s `vec0` virtual table.** If
  upstream gains an indexed mode, this is a config knob on the
  existing CREATE VIRTUAL TABLE statement. Zero-additional-
  surface for fathomdb-engine; biggest dependency on upstream.
- **B — Separate SQLite virtual table / extension.** Drop in
  another SQLite extension (e.g., a hypothetical `vec_hnsw`)
  alongside `vec0`. Two virtual tables on the same vector data;
  query planner picks one. Moderate integration cost.
- **C — Rust-side in-process index** (e.g., `instant-distance`,
  `hnsw_rs`, `usearch` Rust crate). Lives in
  `fathomdb-engine`; SQL gets a Rust function. Bypasses
  SQLite-virtual-table abstraction. Maximum control + maximum
  blast radius (touches engine open/close lifecycle, fixture
  layout, snapshot semantics).
- **D — Hybrid: SQLite-side persisted index + Rust-side query
  layer.** Index data stored as ordinary SQLite tables;
  Rust-side wrapper does the graph traversal. Avoids vendor
  lock on the index format but adds custom code.

For each, sketch:
- How is the index built (eager on insert, lazy on first
  query, batch-rebuild)?
- How does it interact with PCACHE2 (does the ANN data live in
  pages, so PCACHE2 caches it, or is it Rust-owned memory)?
- How does it interact with the writer lifecycle and the
  single-writer architecture (`crates/fathomdb-engine/src/database_lock.rs`)?
- How does it interact with `vector kind` (`configure_vector_kind_for_test`
  in lib.rs) — does each kind get its own index?
- What does the migration from "1M unindexed rows" → "1M
  indexed rows" look like for an existing user?

Report: **drafter's recommendation among A/B/C/D + why**,
and the blast-radius estimate for each.

### Q4 — Recall validation

ANN indexes trade exactness for speed. The current AC-013 test
asserts latency only; it doesn't validate recall. If we land
an ANN index, AC-013 / AC-019 measurements become **conditional
on the index returning the right answers**.

Questions:
- What's the project's existing recall posture? Does any test
  validate that `engine.search()` returns the right rows for
  a given query, or do we only assert latency / count?
- What recall floor should AC-013's RED-test assert (e.g.,
  recall@10 ≥ 0.95 on the synthetic Zipfian corpus)?
- Does the ANN index need a "fall back to brute-force" mode
  for correctness-critical callers?

Report: **a proposed recall-floor expression** to add alongside
the latency budget, and how to compute ground truth on the
synthetic corpus (probably a brute-force pass at test-setup
time, cached as expected results).

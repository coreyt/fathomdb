### Scope creep vs. justified deviation — they are NOT the same

Not every departure from this prompt is creep. The test is **necessity**; the required
response is **round-trip, not silence**. Decide which case you're in:

- **Scope creep — forbidden.** You *could* do more, or a broader/nicer solution is
  tempting, but your mandate doesn't require it. (Classic failure: "add notification
  preferences" → agent builds a whole multi-channel system.) → **Don't.** Record it in
  `output.json.additional_changes_made_in_scope` (or `blockers_encountered` if it blocks
  you) as a **reserved-gap** candidate and move on. When unsure whether something is
  in-scope, **it isn't** — flag it.

- **Justified deviation — sometimes required.** A path in §3 is genuinely *blocked*: an
  anchor is gone, a stated precondition is false, the contract is internally
  inconsistent, or reality changed (a compile error forces a signature change; a
  schema/line-number drift; a dependency the contract assumed isn't actually present).
  Here you **must** adapt to finish the slice — but under three rules:
  1. **Smallest viable change.** Do the minimum that unblocks *your* mandate — not the
     most thorough fix you can imagine.
  2. **Loud, not silent.** `[DETECT]` it on the console the moment you find it, and
     record it in `output.json.additional_changes_made_in_scope` **with the why**. A
     forced deviation is a signal the prompt was under-specified — surfacing it is how
     the spec gets corrected. (~50% of agent failures are *silent* — completing without
     error but wrong. Don't add to that number.)
  3. **Escalate spec-level changes.** If the deviation would change this slice's
     contract, affect **another slice**, or alter a **public/shared surface** (schema,
     wire/SDK API, an ADR decision) → **STOP and report to the orchestrator.** That's a
     specification change, and it is the orchestrator's to make (it round-trips the
     change into the contract/board), not yours to absorb quietly.

> Rule of thumb: deviate to *unblock your own mandate*, never to *expand it*; keep it
> minimal; make it visible; escalate anything that touches the spec or another slice.

## 7. When something goes wrong — detect, log, recover (do not hide)

Mistakes and surprises are normal. The only failure that matters is an **unreported**
one. The loop is always: **notice → `[DETECT]` on console → fix → `[RESOLVE]` on console
→ verify → continue (or STOP+report if unrecoverable).**

### Stray-edit recovery (an edit landed in the canonical repo)

If the §0 tripwire shows changes under `$CANON`:

```bash
echo "[S{{N}}][DETECT] stray edit(s) in canonical repo: <files>"
# For a TRACKED file modified by mistake:
git -C "$CANON" diff -- <FILE> > /tmp/s{{N}}-stray.patch    # capture the stray change
git -C "$CANON" checkout -- <FILE>                          # restore canonical to clean
( cd "$WT" && git apply /tmp/s{{N}}-stray.patch )           # land it in the worktree
# For an UNTRACKED new file created by mistake:
#   mkdir -p "$WT/$(dirname <FILE>)" && cp "$CANON/<FILE>" "$WT/<FILE>" && rm "$CANON/<FILE>"
# Verify BOTH sides:
git -C "$CANON" status --porcelain -- <FILE>   # empty (clean)
git -C "$WT"    status --porcelain -- <FILE>   # shows your change
echo "[S{{N}}][RESOLVE] migrated <files> to worktree; canonical clean; worktree carries the change"
```

### Other surprises

- **A test fails you didn't expect / an anchor is gone / a contract ref is missing:**
  re-read the contract + the failing test. Adapt *minimally for your own slice* and log
  it. If the issue is bigger than your slice (e.g. `main` diverged, a dependency slice
  isn't actually landed): `[DETECT]`, then **STOP and report to the orchestrator** —
  don't self-resolve cross-slice problems.
- **Retry budget:** if you hit the *same* failure mode twice, stop and re-think
  (re-read the test, the contract, the relevant `feedback_*`/ADR). Do not loop a third
  time (AGENTS.md §5).
- **Unrecoverable:** write `output.json` with the blocker in `blockers_encountered`
  (id, description, what you tried), emit `[SUMMARY]` (§8), and stop. A clean halt with
  state captured is a good outcome, not a failure.

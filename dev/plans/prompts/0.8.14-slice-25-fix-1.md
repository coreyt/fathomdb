## Finding 2 [medium] — singleton poisons itself + misreports the cause
In `src/rust/crates/fathomdb-py/src/lib.rs` (~1678), `cls_embedder_singleton()` caches
`Option<CandleBgeEmbedder>` in a `OnceLock` via `.ok()`. A transient load failure is cached as `None`
FOREVER, and `embed_batch_cls_impl` (~1709) then flattens every failure to
`"default embedder weights unavailable (cache miss + no network)"` regardless of the real cause.
**Fix:** do not permanently cache a load FAILURE, and surface the real loader error. Minimal sound
approach (implementer's choice): e.g. cache only a successful embedder (retry the load on a prior
failure) and/or return an error message that reflects the actual `CandleBgeEmbedder::new()` error rather
than the generic "cache miss + no network". Keep the panic boundary (`catch_unwind`/`PanicException`)
intact. Do NOT change the successful-path behavior or the CLS pooling.

## Finding 3 [medium] — missing `.pyi` stub
`embed_batch_cls` is exported from `src/python/fathomdb/__init__.py` but absent from
`src/python/fathomdb/_fathomdb.pyi` (which goes `rerank(...)` → `force_panic_for_test()`). **Fix:** add a
correct stub declaration for `embed_batch_cls` (match the runtime signature — takes a list of strings,
returns a list of float vectors — check the PyO3 signature in `fathomdb-py/src/lib.rs` and mirror it),
with a short docstring noting it is the CLS-pooled batch-embed path and requires `default-embedder`.

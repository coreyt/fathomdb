## 2. The work — draft the measure, then reach Claude↔codex CONSENSUS

### 2.1 Draft `dev/design/ir-recall-measure.md` (under `$WT`)

Answer, grounded in FathomDB's actual architecture (no chunking → fact-vs-document framing; reranker stub;
graph 0.8.1; eu8 exists):

- **(a) What is measured** — the primary measure (candidate: *Evidence Recall@K* = the retrieved context
  contains all evidence required to act) and its relationship to plain retrieval Recall@K, MRR/nDCG, and to the
  eu7 **fidelity** gate (explicitly: complementary, separate axis — do NOT replace/relax eu7/AC-075).
- **(b) Unit of relevance** — document vs **atomic-fact** label (`fact_id`-style); how the gold set will encode
  "all evidence required" (define the *schema/protocol*, not the labels themselves — labeling is Phase 3).
- **(c) The K-ladder** to report (@5/@20/@50/…) and which K is the headline.
- **(d) Per-class structure** — commitment / action / exact-fact / preference / exploratory / negative-"not-found"
  (the *structure*, not the numbers).
- **(e) Retrieval modes** to compare — FTS / vector / RRF-hybrid / +reranker / +graph — flagging which exist
  today vs are 0.8.1/stub.
- **(f) Eval-set composition + qrels/pooling METHODOLOGY** + the **principle** that it runs against a pinned,
  versioned corpus snapshot. **Do NOT commit to a specific corpus version/snapshot** — that is settled
  downstream by the B-1 corpus-basis ruling + the corpus freeze; here, define *how* the eval set is pinned and
  composed so it can't drift (the lesson behind the GA recall halt).
- **(g) Fidelity-vs-relevance separation** — restate why this is the product-value axis, complementary to eu7.

# 0.8.x corpus source expansion search

Type: **Research prompt** — identify additional public data sources
for the FathomDB composite corpus, prioritizing material that
resembles personal-agent memory/workflows and sources with known
paired question-answer supervision for accuracy testing.

Read these first:

- `dev/corpus-creation/README.md`
- `dev/corpus-creation/architecture.md`
- `dev/corpus-creation/extending.md`
- `tests/corpus/corpus-card.md`

- `tests/corpus/scripts/acquire_qmsum.py`

Current shipped corpus facts to preserve:

- Pack-1 manifest count: **7,300 docs** across Enron, EnronQA,
  QMSum, CNN/DailyMail, Landes to-dos, bahmutov daily logs, and
  synthetic notes.
- Chain layer: **367 synthetic chain connective docs** plus
  **200 committed chain specs**.
- Full corpus described by the architecture doc: **7,667 docs**.
- Existing chain specs contain **301 `ground_truth_queries`**.
- Locked `source_type` vocabulary:
  `email`, `meeting`, `paper`, `article`, `note`, `todo`.
- The corpus exists to model personal-agent working context:
  email, meetings, notes, todos, saved/read articles, papers,
  cross-document relation chains, and retrieval accuracy
  evaluation for agents like Hermes, OpenClaw, and Memex.

Important gaps:

- The target size is now **10,000-15,000 docs**, so find
  **2,500-7,500 incremental docs** without letting synthetic data
  dominate.
- `paper` is still absent from the shipped Pack-1 sources, so
  sources that can support `PAPER->NOTE->TODO` and `cites`
  relation coverage are high value.
- EnronQA contains real paired QA supervision, but the current
  acquisition script ingests only email bodies; QA extraction is
  explicitly deferred.
- QMSum query-answer summaries are represented as documents and
  parented to meeting transcripts; evaluate whether the original
  query-answer pairs should also be exported into an eval-only QA
  artifact.
- `tests/corpus/README.md` and `tests/corpus/corpus-card.md` have
  stale scaffold/status wording; do not treat those status lines as
  authoritative over `dev/corpus-creation/README.md`,
  `dev/corpus-creation/architecture.md`, and
  `tests/corpus/scripts/manifest.json`.

## Research Objective

Find candidate data sources that can extend the corpus toward
10K-15K documents while improving accuracy-test coverage for
personal-agent workloads.

Prioritize sources that satisfy at least one of these:

- Real or realistic personal-agent material: emails, inboxes,
  calendars, meetings, notes, memos, personal logs, tasks, issue
  trackers, saved readings, paper libraries, annotations, contacts,
  CRM-like notes, or cross-document decisions.
- Known paired question-answer sets, query-answer sets, rationales,
  evidence spans, summaries, or other gold labels usable for
  retrieval accuracy tests.
- Structured metadata that helps simulate an agent memory graph:
  author/sender, recipients, thread IDs, timestamps, project names,
  people mentions, citations, parent/child relationships, or
  document collections by user/team.

## Hard Constraints

- Do not add a seventh `source_type`. Map every candidate to one
  of `email`, `meeting`, `paper`, `article`, `note`, or `todo`.
- License clarity comes before volume. Record whether each source
  is commit-eligible, cache-only, or rejected.
- Prefer documented datasets and pinned downloadable snapshots over
  scraping.
- Avoid private, leaked, or non-consensual personal data beyond
  already-public research datasets with documented access terms.
- Prefer sources that can be deterministically subsampled and
  rebuilt.
- Prefer sources whose examples are compatible with the canonical
  JSONL document schema in `dev/corpus-creation/architecture.md`.
- For QA/eval data, distinguish:
  - document rows to ingest into `data/corpus-data/raw/*.jsonl`,
  - eval-only QA rows that should not be ingested as documents,
  - chain specs / `ground_truth_queries` that reference ingested
    docs.

## Search Plan

Search in these bands. For each band, identify both obvious and
lesser-known candidates.

1. **Email and personal inbox QA**
   - EnronQA variants, cleaned Enron thread datasets, email QA,
     email summarization, email intent/action-item extraction,
     personal-assistant email benchmarks.
   - Goal: add more paired QA without over-weighting Enron.

2. **Meetings, minutes, action items, and meeting QA**
   - Meeting summarization, query-focused meeting QA, transcript to
     minutes, decision/action-item extraction, AMI/ICSI-derived
     datasets, product/committee meeting datasets.
   - Goal: increase `meeting` volume and improve answerable
     questions grounded in transcripts.

3. **Notes, logs, memos, and personal knowledge bases**
   - Public daily logs, engineering notes, personal wiki exports,
     public Zettelkasten-like repositories, issue-to-note corpora,
     notebook/blog repositories with permissive licenses.
   - Goal: add agent-memory-shaped `note` docs with timestamps,
     tags, project context, and cross-link potential.

4. **Todos, tasks, issues, and action-item supervision**
   - Todo datasets, issue trackers, task extraction datasets,
     action-item corpora, support-ticket task corpora.
   - Goal: expand `todo` and support action retrieval accuracy.

5. **Papers, citations, and paper QA**
   - PMC OA commercial-use subset, S2ORC, OpenAlex-linked paper
     metadata, scientific QA datasets with evidence documents,
     citation recommendation datasets.
   - Goal: finally populate `paper`, unlock `cites`, and support
     `PAPER->NOTE->TODO` chains.

6. **Saved articles and article QA**
   - News/article QA or summarization datasets with clean licenses,
     article-question-answer datasets, web-page reading QA.
   - Goal: improve article-grounded QA beyond CNN/DailyMail
     highlights and avoid adding generic news volume without eval
     labels.

7. **Personal-agent / memory-agent benchmarks**
   - Datasets or benchmarks explicitly framed around personal
     assistants, long-term memory, conversational memory,
     multi-session QA, agent recall, calendar/email/task
     management, or knowledge-worker workflows.
   - Goal: find direct analogues for Hermes, OpenClaw, Memex-style
     agent memory evaluation.

## Candidate Evaluation Rubric

For every candidate, report:

- Name and URL.
- Source type mapping.
- Estimated usable document count for FathomDB.
- Whether it includes paired QA or other gold labels.
- QA fields, if present: question, answer, evidence/rationale,
  negative answers, query type, split, user/session/document IDs.
- Metadata fields useful for graph/retrieval:
  timestamps, author/sender, recipients, speakers, user IDs,
  project IDs, citations, parent links, thread IDs.
- License and redistribution posture:
  commit-eligible, cache-only, or reject.
- Access method and pinning strategy:
  HuggingFace revision, Git commit, archive checksum, API snapshot,
  DOI/version, or release tag.
- Deterministic subsampling strategy.
- Fit score from 1-5 for:
  personal-agent realism, QA usefulness, license clarity,
  implementation effort, and source diversity.
- Recommended action:
  include now, defer, QA-only export, chain-anchor only, or reject.

## Preferred Output

Return a concise ranked report with these sections:

1. **Executive Recommendation**
   - The top 3-6 sources to add first.
   - Proposed doc-count increments.
   - Expected final corpus size.
   - Which sources add paired QA and how many QA pairs are
     expected.

2. **Current Corpus Review**
   - Summarize the existing source mix and gaps using the current
     manifest and architecture docs.
   - Call out stale docs separately from authoritative current
     state.

3. **Candidate Matrix**
   - Table of all viable candidates using the rubric above.

4. **QA / Accuracy-Test Plan**
   - Describe the eval artifact shape for paired QA.
   - Identify which candidates should produce eval-only QA rows.
   - Identify which candidates should generate or extend
     `ground_truth_queries`.
   - Recommend metrics: recall@k, MRR, NDCG, exact evidence-hit,
     and answer-support hit rate.

5. **Implementation Slices**
   - Slice 1: lowest-risk source additions.
   - Slice 2: QA export from existing EnronQA and QMSum sources.
   - Slice 3: new `paper` source and `cites` / `PAPER->NOTE->TODO`
     chain expansion.
   - Slice 4: any higher-friction cache-only source.

6. **Reject / Defer List**
   - Name sources that looked promising but should not be used, with
     a concrete reason: license, access fragility, poor fit, too
     generic, too large, non-deterministic, or insufficient QA.

# 0.6.1 Implementation Plan

This is the **single source of truth** for 0.6.1 sequencing and execution
posture. 0.6.1 is a **patch release** per
`dev/adr/ADR-0.6.0-no-shims-policy.md` § 54: literal contract is
**"no API breaks. Bugfix-only."** Additive surface (new engine-attached
methods, new optional fields, dependency bumps that don't change the
fathomdb surface) is patch-safe; API breaks and schema-additions are
not. Scope-creep into feature work blocks the release; each slice below
verifies against that rule.

0.6.0 GA shipped 2026-05-20 at `edf00f7` (Phase 12 fully CLOSED — see
`dev/plans/0.6.0-implementation.md` § "Phase 12 scoreboard" L805-818).
0.6.1 closes **three** items that were explicitly deferred at 0.6.0
GA. The fourth candidate (AC-012 canonical-runner re-measurement)
was attempted in this release line — verdict RED on canonical x86_64
tier-1 — and the closure escalated to 0.7.0. See § "AC-012 canonical
re-measurement — RED outcome and 0.7.0 escalation" below for the
rescope rationale.

0.6.x release-line goal (locked HITL 2026-05-24): **be fully
functional, including TypeScript parity.** Performance improvement
is the 0.7.0 anchor; knowledge-store / retrieval features for Memex
are the 0.8.0 anchor (see `dev/roadmap/0.8.0.md`).

1. **12-TX-OPENREPORT** — Python (PyO3) and TypeScript (napi) bindings
   currently drop `OpenedEngine.report`. Native Rust populates it;
   bindings surface the engine handle only. 0.6.1 surfaces the report
   via a new engine-attached accessor: `engine.open_report()` (py) /
   `engine.openReport()` (ts), per Shape D — see § "Q-002 resolution
   (HITL 2026-05-24)" below for the design rationale. `Engine.open`
   signature is unchanged from 0.6.0 (zero legacy-callsite risk).
   Symmetric closure (parity matrix § 1 row 2; not a TS-vs-Python
   gap). This is the "TypeScript parity" anchor for the 0.6.x
   release-line goal.
2. **TS↔Python parity matrix close** — `dev/notes/12-TX-parity-matrix.md`
   § 8 lists three gaps; two closed in 12-TX (`CounterSnapshot` fields,
   `test_errors.py LEAF_CLASSES`, `test_scaffold.py`). The remaining
   ❌ row is the OpenReport row from item 1 — so item 1 is the parity
   close. **No additional TXPARITY-N slices needed**; the ❌ inventory
   is already exhausted by OPENREPORT. (Verified 2026-05-22 by re-read
   of parity matrix L29-37 and § 8 summary L137-150.)
3. **Axis-E independence demo** — per Wake decision `d-001`, axis-E
   (`fathomdb-embedder-api`) is structurally independent of axis-W
   (workspace lockstep). The 0.6.0 → 0.6.1 axis-W bump is the first
   post-GA opportunity to demonstrate the discipline holds. No trait-
   surface change in 0.6.1 → axis-E stays at `0.6.0`. Discipline
   check, not a feature.

Plus **Dependabot 4 open advisories** (pyo3 low ×2, js-yaml medium,
idna medium) triaged below.

## AC-012 canonical re-measurement — RED outcome and 0.7.0 escalation

Originally listed as the fourth 0.6.1 closure item. The
`0.6.1-AC012-measure` slice ran on 2026-05-23 via
`.github/workflows/perf-canonical.yml`
([workflow run 26346417896](https://github.com/coreyt/fathomdb/actions/runs/26346417896))
on canonical x86_64 tier-1 hardware (AMD EPYC 9V74, 4 cores,
Ubuntu 24.04, rustc 1.95.0, SQLite 3.45.x).

- 0.6.1 ships under a **re-titled scope**: the three remaining
  closure items (OPENREPORT-py/ts, Dependabot, axis-E demo) plus
  the RED measurement as documented evidence. 0.6.1 release notes
  do NOT claim AC-012 closure.
- AC-012 stays DEFERRED in `dev/test-plan.md` § Current Perf
  Attribution; deferral target language updated from "Pack 7" to
  "0.7.0 (per HITL 2026-05-24)".
- 0.7.0 is now **perf-only** (single thrust). Two-pronged:
  (1) revise the AC-012/AC-013/AC-019/AC-020 budgets empirically
  against canonical measurements; (2) tune SQLite + FathomDB
  (including a top-K LIMIT cap on the search verb as one engine
  lever among others). The canonical-identity substrate + logical-id
  verbs that were previously the 0.7.0 anchor move to 0.8.0
  alongside Memex knowledge-store / retrieval features
  (`dev/roadmap/0.8.0.md`).

Rationale: ADR-0.6.0-no-shims-policy § 54 forbids new engine work
and ADR budget revision in a patch release. The 7× p50 miss is
empirical, scales with corpus size (Tegra N=100K p50=29.7ms →
canonical N=1M p50=140.95ms; ~5× latency growth for 10× corpus),
and is not a runner artifact — it requires either engine tuning or
budget revision, both minor-release-shaped. Shipping 0.6.1 without
the AC-012 claim preserves the patch contract; carrying the RED
measurement forward gives 0.7.0 an empirical baseline for the
successor `ADR-0.7.0-text-query-latency-gates-revised`.

## Q-002 resolution (HITL 2026-05-24): OpenReport surfacing shape

The planning-slice output.json initially closed Q-002 by mandating a
wrapper / NamedTuple return on `Engine.open(...)` (Shape A) and
forbidding a new verb. HITL 2026-05-24 reopened Q-002 after
recognizing that ADR-0.6.0-no-shims-policy § 54 says exactly _"no
API breaks. Bugfix-only."_ — not "no new verbs". With the strict
contract read, the four shapes were re-evaluated:

- **Shape A — wrapper return on `Engine.open`** (initial closure):
  type-checker friction, `isinstance`/pickling edge cases, awkward
  TS Promise-of-wrapper ergonomics. Rejected.
- **Shape B — tuple return on `Engine.open`**: breaks legacy
  callers. Rejected (violates § 54 "no API breaks").
- **Shape C — new top-level verb `Engine.open_with_report` /
  `Engine.openWithReport`**: clean types, zero legacy risk, but
  joins the five-verb canonical surface as a sixth verb. Acceptable
  but not preferred.
- **Shape D — engine-attached accessor `engine.open_report()` /
  `engine.openReport()`** (CHOSEN): zero risk to `Engine.open`,
  clean types, slots into the existing engine-attached
  instrumentation taxonomy alongside `engine.counters()`,
  `engine.drain()`. Spec edit relocates the report mention from
  "Runtime surface" to "Engine-attached instrumentation / control".
  Patch-release-safe (additive only). Idempotency contract:
  repeated calls return identical data captured at open time.

OPENREPORT-py and OPENREPORT-ts skeletons updated to spec Shape D
verbatim. AC-068c / AC-068d wording rewritten accordingly. Append
of AC-068c + AC-068d to `dev/acceptance.md` happens at
OPENREPORT-py spawn (Q-001 disposition; both IDs land together).

### Proposed new ACs (HITL confirms before `dev/acceptance.md` append)

- **AC-068c (proposed).** Python `engine.open_report()` returns the
  structured open report captured at `Engine.open` time. Caller
  receives every field of the native `OpenReport` struct
  (`src/rust/crates/fathomdb-engine/src/lib.rs:541-548`) under the
  same snake_case identifiers: `schema_version_before: int`,
  `schema_version_after: int`, `migration_steps: list`,
  `embedder_warmup_ms: int`, `query_backend: str`, and
  `default_embedder` (embedder-identity payload). The accessor is
  idempotent — repeat calls return identical data (the report is a
  snapshot, not live state). `Engine.open(...)` signature unchanged
  from 0.6.0 (returns just `Engine`); no return-shape regression.
  Verified by `tests/test_openreport.py` opening a fresh DB,
  calling `engine.open_report()` twice, asserting every field
  populated AND identical across calls. Cites `dev/design/engine.md`
  § "`Engine.open` success result" (spec-locked field subset:
  `schema_version_before`, `schema_version_after`, `migration_steps`,
  `embedder_warmup_ms`) and `dev/interfaces/python.md`
  (Engine-attached instrumentation list, post-spec-edit).
- **AC-068d (proposed).** TypeScript `engine.openReport()` returns
  the structured open report (sync return — data lives in the
  napi engine struct after open). Caller receives the camelCase
  mirror of the native fields: `schemaVersionBefore: number`,
  `schemaVersionAfter: number`,
  `migrationSteps: ReadonlyArray<MigrationStepReport>`,
  `embedderWarmupMs: number`, `queryBackend: string`,
  `defaultEmbedder` (embedder-identity payload). Idempotent —
  repeat calls return identical data. `Engine.open(...)` Promise
  signature unchanged from 0.6.0 (resolves to just `Engine`).
  Verified by `tests/openreport.test.ts` asserting every field
  populated AND identical across calls. Cites
  `src/rust/crates/fathomdb-engine/src/lib.rs:541-548` and
  `dev/interfaces/typescript.md` (Engine-attached instrumentation
  list, post-spec-edit).

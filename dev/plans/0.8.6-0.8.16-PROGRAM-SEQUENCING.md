# FathomDB 0.8.6 → 0.8.16 — non-measure program sequencing

> **What this is.** The 17 "other important aspects" of FathomDB (everything that is *not* the
> M1/M5/M6/M7 competitor-parity experiments and *not* the planner/router design track) sequenced into
> six even-numbered micro-releases, with two odd-numbered **out-of-band (OOB)** releases for drop-in
> work. This file is the **dependency graph + allocation + decision record**; each release's slice
> ladder lives in its own `plan-0.8.z.md`.
>
> **Relationship to the rest of the program.** The experiment program (M-work) and the planner/router
> designs (`dev/design/planner-router-psd-0.8.x.md`, `initial-arch-planner-router-0.8.x.md`) remain the
> **primary** token spend. This line is the **secondary, mostly-$0 plumbing/mechanism track** — it is
> deliberately sequenced to run *beside* the experiments without contending for priced-run budget. The
> OOB odd releases exist precisely so drop-in work never blocks the experiment critical path.
>
> **Footprint invariant (binds every item here).** Library query path = CPU-only, no network/LLM,
> 1-bit binary (Hamming), deterministic. The only LLM seam is **BYO-LLM in the caller**. GPU + frontier
> models are **OFFLINE-BUILD / EVAL-ONLY**. Every slice tags each technique
> IN-LIBRARY / CALLER-SIDE-BYO-LLM / OFFLINE-BUILD / EVAL-ONLY.

other ordering is value-ordering, not a compile/contract break. #3 (GPU) is a pure soft accelerator —
nothing waits on it, but everything re-embed-heavy is faster after it.

## 3. Out-of-band proposal (odd micro numbers)

OOB = few/no upstream-or-downstream deps, **drop-in**, byte-stable or behind a flag, runnable by a
parallel agent track without touching the experiment hot-path. These get **odd** micro numbers and run
**in parallel** to the even main line.

| OOB release | Item(s) | Why OOB |
|---|---|---|
| **0.8.7** | **#3 GPU embedder** | Default build stays CPU; opt-in `embed-cuda` feature + `FATHOMDB_EMBED_DEVICE` env; all byte-stability gates unchanged. Zero upstream/downstream code deps. Scheduled **first** because it turns a 27 h re-embed into minutes — every later re-embed-heavy slice and experiment benefits. |
| **0.8.9** | **#12 perf-gate unmask + #14 AC-037/AC-050c** | Both are small, self-contained CI-integrity fixes with no feature coupling. Done OOB so they stop the boards from lying (vacuous-green gates) **early** without consuming experiment tokens. |

Items that *look* OOB but are **not** scheduled OOB, and why:
- **#4 ONNX** is structurally drop-in (new `impl Embedder`, zero engine change) but (a) low urgency —
  no AMD/Intel hardware is in play — and (b) its productionization creates the numeric-divergence
  hazard that **#5** guards. So it lands **late (0.8.14)**, adjacent to #5, not as an early OOB.
- **#17 filter-grammar** is contained but touches the **shipped** G10 `SearchFilter` and feeds the
  router's typed-constraint surface — it belongs with the substrate/recall release (0.8.12), not a
  blind drop-in.
- **#15 F9** is an engine/schema feature with ranking semantics, not a flag-gated drop-in.

## 5. Answers to the three evaluation questions

### Q1 — Can #3 and #4 be done out-of-band / in parallel?

- **#3 GPU — yes, strongly, and early.** It is the textbook OOB item: default build stays CPU, GPU is
  opt-in via a cargo feature + `FATHOMDB_EMBED_DEVICE`, so every existing test and byte-stability gate
  is unchanged. It has no upstream or downstream code dependency. It is scheduled **0.8.7, parallel to
  0.8.6**, because single-stream GPU is ≈100–300× CPU (27 h embed → minutes) and accelerates every
  re-embed-heavy slice and experiment after it. **Do this first.**
- **#4 ONNX — structurally yes, but not early.** A new `impl Embedder` behind the existing
  `EmbedderChoice::Caller` boundary is a zero-engine-change drop-in. But it is **low urgency** (the
  hardware in play is NVIDIA; ONNX/ROCm/DirectML target AMD/Intel users we don't have) and its
  productionization is exactly what makes silent cross-backend numeric divergence likely. So it is
  scheduled **0.8.14, adjacent to #5**, so the equivalence guard lands with the backend that needs it.
  It is parallelizable, just not urgent.

### Q2 — #5 vector-equivalence: parallel, deprecate, defer to 2.0.0, or essential early?

**Verdict: defer to late-0.8.x (0.8.16). Do not deprecate. Do not push to 2.0.0. Not essential early.**

- **Not essential early — and actively counter-productive early.** 0.8.x is the high-velocity
  experimentation phase: many re-embeds, algorithm changes, possibly a full embedder swap (EXP-M4).
  Every experiment **rebuilds a fresh index and reads it with the same backend**. The guard protects
  against *moving a DB across machines/backends* — which experiments do not do. Landing it early adds
  `Engine::open` friction (re-embed an N≈16–32 probe set on every open) plus a real
  tolerance-calibration burden (the check must be done at the **post-1-bit-quantization** representation
  against the binary-quant floor) — i.e. cost and churn during exactly the phase with the most index
  rebuilds, for ~zero protection.
- **Not 2.0.0 either.** The hazard is created *inside* 0.8.x the moment #3 (CUDA) and #4 (ONNX) exist:
  a GPU- or ONNX-built index read by a CPU reader silently degrades the dense arm. The design doc is
  explicit — §3 *must* land before FathomDB advertises portable DBs or runtime backend-swap, and before
  any production GPU-built index is moved to a CPU reader. That promise is a 0.8.x-era one.
- **So: build it as the capstone of 0.8.16, gated as the prerequisite to any "portable DB / runtime
  backend-swap" claim, bundled with #4's productionization** (cross-vendor is when divergence actually
  bites — ONNX vs candle numerics differ more than CUDA vs CPU of the same candle model).
- **Interim mitigation (cheap, ships with #3 in 0.8.7):** a documented *same-backend build-and-read*
  discipline + the existing `EmbedderIdentity` open-time pre-filter (catches model/revision/dim
  mismatch). #5 upgrades that from discipline to *enforcement* when #4 makes divergence probable.
- **Parallel?** It can be *authored* in parallel (localized embedder-crate + open-time change), but it
  should **activate** only at 0.8.16 to avoid open-time friction during experimentation.

### Q3 — How to handle #11–#14 (CI/CD) without starving the experiments?

**Principle: spend the minimum CI/CD that (a) makes every micro-release genuinely DoD-shippable and
(b) stops the gates from lying — defer everything heavy or net-new until its substrate exists or GA
nears. Run all CI/CD as a separate, mostly-$0, agent-parallelizable track (foundation slices + OOB
odd releases) so it never competes with experiment tokens.**

- **#11 Release — SPLIT.** The **minimal viable publish path** (rewrite `set-version.sh` to the two
  version axes, restore `release.yml` to the 8-tier topological order from `design/release.md`,
  dry-run, and push the 178-commit backlog) lands **early in 0.8.6** — without it, "DoD-shippable per
  micro-release" is hollow. The **heavy publish matrix** (multi-OS napi prebuilds, full cross-ecosystem
  gate, index-propagation orchestration) defers to the **0.8.16** GA-hardening capstone.
- **#12 perf-gate unmask — early but cheap, OOB 0.8.9.** It is a *correctness* bug: `ac_012/013/013b/
  019/020` are `AGENT_LONG`-gated so they never run per-push (vacuously green), and `ac_013b` asserts
  the 0.90 floor against a ~0.73 synthetic embedder. Fixing the gate *semantics* — run them where they
  can run, or honestly re-scope + relabel so the board stops implying a floor CI never checks — is
  small and high-value. OOB so it doesn't draw experiment tokens.
- **#14 AC-037 CI wiring + AC-050c cleanup — small, OOB 0.8.9 with #12.** Durably wire the
  `netns-deny-egress` security gate on a userns-permissive runner (it was confirmed green *once*,
  manually); clean up the pre-existing AC-050c removal-detect baseline failure.
- **#13 benchmark-and-robustness harness — defer to 0.8.16 (or roadmap-out).** It is **net-new
  authorship** (no `benches/`, no `scale.rs`, no `tracing` cargo feature, no stress suites), not a
  restoration — low near-term ROI. Some substrate (criterion benches) accretes naturally from the perf
  experiments, so authoring it late costs less. Build it only when GA-hardening needs continuous
  regression coverage.
- **Net spend: ≈1.5 releases of CI/CD total** (0.8.6 release-min + 0.8.9 OOB micro + 0.8.16 capstone),
  all off the experiment critical path.

## 6. How each release runs — the orchestrator contract (`/goal complete 0.8.z`)

Every release in this line is executed by an **orchestrator session** (the program-steward role,
`dev/plans/prompts/0.8.x-PROGRAM-STEWARD-HANDOFF.md`), not a single implementer. Open the session with
`/goal` on that handoff (or this file's release pointer) and drive the release's `plan-0.8.z.md` ladder.
Per `dev/design/orchestration.md`:

1. **Requirements + acceptance criteria** are frozen at the release's **Slice 0** (design-ADR slice):
   the DoD table in `plan-0.8.z.md` becomes pre-registered, and where a gated slice mints a new AC it
   is added then (acceptance.md stays locked otherwise — `acceptance-md-locked-no-feature-acs`).
2. **Design → design-review** per slice: the implementer (or steward) writes a slice design doc; an
   independent **codex** pass reviews the design *before* code (`codex exec --dangerously-bypass-
   approvals-and-sandbox`, data+goals, no leaning).
3. **TDD implementation** in a pre-created worktree off `$(git rev-parse main)`: RED tests first, then
   GREEN; `implementer` subagent commits on the slice branch and writes `output.json`.
4. **codex §9 review** of the diff → PASS / CONCERN(+override) / BLOCK(→fix-N).
5. **Closure**: board (`runs/STATUS-0.8.z.md`) + one docs commit per transition; worktree cleaned.
6. **Release DoD** (Fathom is DoD at the end of each micro-release): all mod-5 slices CLOSED · X1/X2/X3
   satisfied (SDK parity + `mkdocs build` green + DOC-INDEX maintained) · the release's AC gate green ·
   **and the artifact is shippable** through the release machinery stood up in 0.8.6 (`/goal` may
   dry-run the publish as the final verification step).

**Cross-cutting DoD (X1/X2/X3), carried verbatim from 0.8.0/0.8.1, binds every slice in every release
here** — restated in each `plan-0.8.z.md`.

## 7. Sequencing rationale (one line per release)

- **0.8.6 first** — nothing in this line should layer onto un-migrated coupling or an un-unified
  provider transport, and no release can be "DoD-shippable" until the publish path exists. Foundations.
- **0.8.7 OOB GPU** — bank the 100–300× re-embed speedup before the re-embed-heavy releases (0.8.10/12).
- **0.8.8 obs+telemetry** — surface *why a result ranked* and *what actually happened* before building
  features (F9/F5) whose value you'll want to observe, and before the router needs transparency.
- **0.8.9 OOB CI** — stop the vacuous-green gates lying, cheaply, in parallel.
- **0.8.10 memory plumbing** — now that #8/#9 exist, build the consolidation/recency provider and lift
  ELPS coverage (the head of the virtuous loop that feeds every graph/coverage measure).
- **0.8.12 substrate+recall** — coordinate the two schema migrations (EXP-S kind-tags + F5 fielded
  fields) in one release so the engine pays one migration cost; land the typed-constraint surface (#17).
- **0.8.14 ranking+reach** — F9 ranking signal (now observable via 0.8.8) + the ONNX backend.
- **0.8.16 capstone** — make portable DBs/backend-swap *safe* (#5, now that #3/#4 exist), restore the
  heavy regression harness (#13), and complete the full publish pipeline (#11-full) for a clean GA-grade
  release of the whole 0.8.x line.

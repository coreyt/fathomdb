# FathomDB 0.8.6 → 0.8.16 — master release plan (sequencing & dependencies)

> **What this is.** The **sole master release-plan document for 0.8.x going forward.** It sequences two
> bodies of work into the same release line:
> 1. the **17 "other important aspects"** (everything that is *not* the M1/M5/M6/M7 competitor-parity
>    experiments) — the even-release backbone plus the odd-release OOB drop-ins; and
> 2. the **planner-router track** — the experiment ladder (EXP-A/M4/B′/Fr-acc/AF) and the dispatcher
>    build (EXP-Fr), which rides the **odd** releases and interlocks with the even line at two points.
>
> This file is the **dependency graph + allocation + decision record**; each release's slice ladder lives
> in its own `plan-0.8.z.md`. The planner-router *design* lives in `planner-router-psd-0.8.x.md` +
> `initial-arch-planner-router-0.8.x.md`; its *sequencing* (formerly `planner-router-sequencing-0.8.x.md`)
> is now folded in here (§5) so there is one schedule of record.
>
> **Relationship to the M-work.** The M1/M5/M6/M7 competitor-parity experiments remain the **primary**
> priced-token spend and are tracked in their own roadmap/plan docs. This line is the **secondary,
> mostly-$0 plumbing/mechanism + planner-router-eval track** — deliberately sequenced to run *beside* the
> M-work without contending for priced-run budget. The odd releases carry the drop-in work and the $0
> planner-router experiments so neither blocks the M-work critical path.
>
> **Footprint invariant (binds every item here).** Library query path = CPU-only, no network/LLM,
> 1-bit binary (Hamming), deterministic. The only LLM seam is **BYO-LLM in the caller**. GPU + frontier
> models are **OFFLINE-BUILD / EVAL-ONLY**. Every slice tags each technique
> IN-LIBRARY / CALLER-SIDE-BYO-LLM / OFFLINE-BUILD / EVAL-ONLY.

### 2a. The binding edges — physically-hard vs rework-forcing

Integrated across both tracks, the schedule is rigid at exactly these edges. **Physically-hard** = the
downstream literally cannot be built/run earlier. **Rework-forcing** = you *can* go out of order, but you
throw work away — same management burden in practice.

| # | Edge | Direction | Class | Why |
|---|------|-----------|-------|-----|
| **I-1** | 0.8.8 **EXP-OBS** (per-arm provenance + score breakdown) → odd **agent-signal / EXP-AF** (0.8.9 half, 0.8.11) | even → **odd** | **physically hard** | The agent-relevance signal judges *against* the provenance/score-breakdown + telemetry; nothing agent-feedback can run before it exists. |
| **I-2** | 0.8.12 **EXP-S** substrate → 0.8.13 **EXP-Fr** in-library dispatcher | even → **odd** | **physically hard** | The dispatcher is in-library code over the kind-tagged indexes — no substrate, no in-library router. **This is the long pole.** |
| **I-3** | 0.8.7 **GPU device seam (#3)** → 0.8.14 **ONNX (#4)** | **odd → even** | rework-forcing | ONNX extends `resolve_device()` from #3; build #3 first or build the seam twice. *This edge breaks the "evens are self-contained" model.* |
| **I-4** | 0.8.7 (#3) + 0.8.14 (#4) → 0.8.16 **vector-equivalence (#5)** | mixed → even | physically hard-ish | The guard is meaningless with one backend — it needs ≥2 backends that can read each other's index. |
| **I-5** | 0.8.6 **#8 provider protocol** → 0.8.10 **#7/#6** | even → even | rework-forcing | Build consolidation before the unified protocol → throwaway contract + rewrite (the one edge that "bites mid-implementation"). |
| **I-6** | 0.8.12 EXP-S (slice 5) → F5 (slice 10) | within-release | rework-forcing | F5 rides EXP-S's field columns; one coordinated migration. |

**Everything else floats.** GPU just accelerates re-embeds everywhere (soft); Gate-0/Gate-2 / EXP-A /
EXP-M4 / EXP-B′ / EXP-Fr-acc are $0 eval that can land anywhere in their window *as long as they're done
by their consumer's deadline* (§5); CI-micro (0.8.9) is only "recommended-before" GA.

### 2b. Can the even line be batched ahead of the odd line? — No.

Taking both tracks unedited and integrated, **neither parity can be completed as a batch before the
other**, because the hard edges run **both directions**:

- **All-even-first is impossible** — edge **I-3**: even 0.8.14 (ONNX) needs odd 0.8.7 (GPU seam).
- **All-odd-first is impossible** — edges **I-1, I-2**: odd 0.8.13 (dispatcher) needs even 0.8.12
  (EXP-S); the odd agent-signal track needs even 0.8.8 (EXP-OBS).

The correct mental model is **one interleaved DAG**, not two parallel tracks. **The even/odd split is a
*labeling*** (in-band mechanism vs out-of-band experiment), **not a dependency boundary.** What must be
managed is the **landing order of ~6 real builds** (EXP-OBS, EXP-S, EXP-Fr, and the GPU→ONNX→vec-equiv
embedder chain, plus the provider protocol); all the planner-router *experiments* are $0 float scheduled
against those landings. The forced minimum interleave is:
`0.8.6 → 0.8.7 → 0.8.8 → 0.8.10 → 0.8.12 → 0.8.13 → 0.8.14 → 0.8.16`, with the $0 experiments slotted
into the 0.8.7 / 0.8.9 / 0.8.11 windows.

## 5. Planner-router track integration (the by-when)

> **Result in one box.** The planner-router work is **almost entirely OOB** ($0 eval/analysis, landable
> anytime) — with one exception: the **dispatcher build (EXP-Fr)**, real engine code **hard-gated on
> EXP-S (0.8.12)** (I-2). So the experiments run OOB across **0.8.7 → 0.8.11 and must be *done* by
> 0.8.12**, the dispatcher lands at **0.8.13** (target; no later than **0.8.15**), and **EXP-S (0.8.12)
> is the long pole**. Two router prerequisites — **EXP-OBS (0.8.8)** and **EXP-S (0.8.12)** — already
> live in the even line; the odd planner-router work is sequenced *around* those two landings.

### 5c. By-when (bottom line)

- **The dispatcher (EXP-Fr) lands at 0.8.13 and cannot come earlier** — EXP-S at 0.8.12 is the long pole.
  Target **0.8.13**, no later than **0.8.15**.
- **All feeding experiments must be *done by 0.8.12*** so 0.8.13 builds with no idle wait and no stale
  re-runs: EXP-A/M4 by 0.8.9 → EXP-B′ + Fr-acc by 0.8.11 → locus finalized at 0.8.12 with EXP-S.
- **Long pole = EXP-S (0.8.12). If EXP-S slips, the router slips one-for-one.** The same one-for-one
  coupling holds for EXP-OBS@0.8.8 → the agent-signal track (I-1).

### 5d. Contingency decision points

- **DP-A — EXP-S KILL path (at 0.8.12): in-library vs agent-side locus.** If EXP-S's determinism/perf
  check fails, the in-library dispatcher is off the table — "router stays agent-side, indexes stay
  eval-side" — and 0.8.13 ships the **agent-side router + L1 hardening** instead. **Pre-mitigation:**
  the agent-side L2 prototype built at **0.8.11** has no substrate dependency, so **a router ships either
  way**; EXP-S only decides whether the *in-library* locus is also available (the "both-layered" stance
  as a schedule hedge). Owner: steward, at the 0.8.12 EXP-S readout, jointly with Fr-acc's locus call.
- **DP-B — EXP-B′ divergence (~0.8.9–0.8.11): does the router move onto the parity critical path?** If
  joint-tuning shows F1/F2 are Pareto-blocked and need different `(index, retrieval, α, pool_n, MMR,
  recency)` tuples, then per-intent **parity-or-better cannot ship without the router** — EXP-Fr stops
  being a capstone and becomes the **gate for the parity product claim**. It still **cannot precede
  EXP-S (0.8.12)**, so the effect is to **raise the cost of any router slip**, not pull the date earlier.
  Mitigation: have the 0.8.11 agent-side router carry the per-intent tuples so parity can be demonstrated
  agent-side before the in-library build. Owner: steward, at the EXP-B′ readout.

## 6. Integration findings (objective, carried into this plan)

Three findings from reconciling the planner-router schedule against the even-line `plan-0.8.z.md` files.
They are recorded here as standing risks/clarifications, not yet actioned.

- **F-1 — The EXP-S long-pole risk is understated by the raw planner-router schedule.** That schedule
  treats `EXP-S@0.8.12` as if 0.8.12 *is* EXP-S and says "if EXP-S slips, the router slips one-for-one."
  But in `plan-0.8.12`, **EXP-S shares the release with F5 (#16) and filter-grammar (#17)** — two further
  schema/surface migrations (I-6). *Mitigation already in the plan:* EXP-S is the **slice-5 keystone**, so
  the *capability* the router needs lands early-in-release, before the whole-release DoD closes — the
  dispatcher can begin consuming it mid-0.8.12. *Residual risk:* EXP-S still competes for engine attention
  with two co-resident migrations, and any EXP-S expansion (it is a schema migration with a determinism
  check) pushes F5/filter and threatens the 0.8.13 router date. **Treat EXP-S as the program's single most
  schedule-critical build and resist adding scope to 0.8.12.**
- **F-2 — The EXP-OBS scope watch-item is already satisfied (confirm, don't re-litigate).** The
  planner-router schedule warns that if 0.8.8 ships "only `ce_score`, not per-arm provenance," the
  agent-signal track slips. `plan-0.8.8` **R-OBS-1 already scopes per-hit arm-provenance + score
  breakdown** as the EXP-OBS keystone (not a later increment), and places telemetry + real-gold (#10) at
  its slices 15/20. So the two docs **agree**; I-1's upstream is correctly scoped. The standing action is
  only to *hold* that scope at 0.8.8 Slice 0, not to expand it.
- **F-3 — The odd line grew; these net-new slots are now ratified by this plan.** The original
  non-measure sequencing defined only two OOB odd slots (0.8.7 GPU, 0.8.9 CI). Folding in the
  planner-router track **adds 0.8.11 / 0.8.13 / 0.8.15 as net-new odd slots** and **co-hosts $0
  planner-router eval on 0.8.7 / 0.8.9** alongside their original drop-ins. This is deliberate (the odd
  line is the program's OOB/eval channel), but it means the odd line now does substantially more than
  "two drop-ins": 0.8.13 in particular carries a **real engine build** (EXP-Fr), not OOB work, and is
  hard-gated (I-2). The steward should staff the odd line accordingly, not treat it as spare capacity.

## 7. Answers to the three evaluation questions

### Q1 — Can #3 and #4 be done out-of-band / in parallel?

- **#3 GPU — yes, strongly, and early.** It is the textbook OOB item: default build stays CPU, GPU is
  opt-in via a cargo feature + `FATHOMDB_EMBED_DEVICE`, so every existing test and byte-stability gate
  is unchanged. It has no upstream dependency. Scheduled **0.8.7, parallel to 0.8.6**, because
  single-stream GPU is ≈100–300× CPU (27 h embed → minutes) and accelerates every re-embed-heavy slice
  and experiment (incl. EXP-M4). **Do this first.** *Note the one downstream edge:* it feeds #4 (I-3) and
  the vec-equiv calibration (I-4).
- **#4 ONNX — structurally yes, but not early.** A new `impl Embedder` behind `EmbedderChoice::Caller` is
  a zero-engine-change drop-in, but it is **low urgency** (NVIDIA hardware in play; ONNX targets AMD/Intel
  we don't have), its productionization makes silent cross-backend numeric divergence likely (needs #5),
  and it has a rework-forcing dep on #3's device seam (I-3). So it lands **0.8.14, adjacent to #5**.

### Q2 — #5 vector-equivalence: parallel, deprecate, defer to 2.0.0, or essential early?

**Verdict: defer to late-0.8.x (0.8.16). Do not deprecate. Do not push to 2.0.0. Not essential early.**

- **Not essential early — and actively counter-productive early.** 0.8.x is the high-velocity
  experimentation phase: many re-embeds, algorithm changes, possibly a full embedder swap (EXP-M4).
  Every experiment **rebuilds a fresh index and reads it with the same backend**. The guard protects
  against *moving a DB across machines/backends* — which experiments do not do. Landing it early adds
  `Engine::open` friction (re-embed an N≈16–32 probe set on every open) plus a real
  tolerance-calibration burden (the check must be done at the **post-1-bit-quantization** representation
  against the binary-quant floor) — i.e. cost and churn during exactly the phase with the most index
  rebuilds, for ~zero protection.
- **Not 2.0.0 either.** The hazard is created *inside* 0.8.x the moment #3 (CUDA) and #4 (ONNX) exist:
  a GPU- or ONNX-built index read by a CPU reader silently degrades the dense arm. The design doc is
  explicit — §3 *must* land before FathomDB advertises portable DBs or runtime backend-swap, and before
  any production GPU-built index is moved to a CPU reader. That promise is a 0.8.x-era one.
- **So: build it as the capstone of 0.8.16, gated as the prerequisite to any "portable DB / runtime
  backend-swap" claim, bundled with #4's productionization** (cross-vendor is when divergence actually
  bites — ONNX vs candle numerics differ more than CUDA vs CPU of the same candle model; this is I-4).
- **Interim mitigation (cheap, ships with #3 in 0.8.7):** a documented *same-backend build-and-read*
  discipline + the existing `EmbedderIdentity` open-time pre-filter (catches model/revision/dim
  mismatch). #5 upgrades that from discipline to *enforcement* when #4 makes divergence probable.
- **Parallel?** It can be *authored* in parallel (localized embedder-crate + open-time change), but it
  should **activate** only at 0.8.16 to avoid open-time friction during experimentation.

### Q3 — How to handle #11–#14 (CI/CD) without starving the experiments?

**Principle: spend the minimum CI/CD that (a) makes every micro-release genuinely DoD-shippable and
(b) stops the gates from lying — defer everything heavy or net-new until its substrate exists or GA
nears. Run all CI/CD as a separate, mostly-$0, agent-parallelizable track (foundation slices + OOB
odd releases) so it never competes with experiment tokens.**

- **#11 Release — SPLIT.** The **minimal viable publish path** (rewrite `set-version.sh` to the two
  version axes, restore `release.yml` to the 8-tier topological order from `design/release.md`,
  dry-run, and push the 178-commit backlog) lands **early in 0.8.6** — without it, "DoD-shippable per
  micro-release" is hollow. The **heavy publish matrix** (multi-OS napi prebuilds, full cross-ecosystem
  gate, index-propagation orchestration) defers to the **0.8.16** GA-hardening capstone.
- **#12 perf-gate unmask — early but cheap, OOB 0.8.9.** It is a *correctness* bug: `ac_012/013/013b/
  019/020` are `AGENT_LONG`-gated so they never run per-push (vacuously green), and `ac_013b` asserts
  the 0.90 floor against a ~0.73 synthetic embedder. Fixing the gate *semantics* — run them where they
  can run, or honestly re-scope + relabel so the board stops implying a floor CI never checks — is
  small and high-value. OOB so it doesn't draw experiment tokens.
- **#14 AC-037 CI wiring + AC-050c cleanup — small, OOB 0.8.9 with #12.** Durably wire the
  `netns-deny-egress` security gate on a userns-permissive runner (it was confirmed green *once*,
  manually); clean up the pre-existing AC-050c removal-detect baseline failure.
- **#13 benchmark-and-robustness harness — defer to 0.8.16 (or roadmap-out).** It is **net-new
  authorship** (no `benches/`, no `scale.rs`, no `tracing` cargo feature, no stress suites), not a
  restoration — low near-term ROI. Some substrate (criterion benches) accretes naturally from the perf
  experiments, so authoring it late costs less. Build it only when GA-hardening needs continuous
  regression coverage.
- **Net spend: ≈1.5 releases of CI/CD total** (0.8.6 release-min + 0.8.9 OOB micro + 0.8.16 capstone),
  all off the experiment critical path.

## 8. How each release runs — the orchestrator contract (`/goal complete 0.8.z`)

Every release in this line is executed by an **orchestrator session** (the program-steward role,
`dev/plans/prompts/0.8.x-PROGRAM-STEWARD-HANDOFF.md`), not a single implementer. Open the session with
`/goal` on that handoff (or this file's release pointer) and drive the release's `plan-0.8.z.md` ladder.
Per `dev/design/orchestration.md`:

1. **Requirements + acceptance criteria** are frozen at the release's **Slice 0** (design-ADR slice):
   the DoD table in `plan-0.8.z.md` becomes pre-registered, and where a gated slice mints a new AC it
   is added then (acceptance.md stays locked otherwise — `acceptance-md-locked-no-feature-acs`).
2. **Design → design-review** per slice: the implementer (or steward) writes a slice design doc; an
   independent **codex** pass reviews the design *before* code (`codex exec --dangerously-bypass-
   approvals-and-sandbox`, data+goals, no leaning).
3. **TDD implementation** in a pre-created worktree off `$(git rev-parse main)`: RED tests first, then
   GREEN; `implementer` subagent commits on the slice branch and writes `output.json`.
4. **codex §9 review** of the diff → PASS / CONCERN(+override) / BLOCK(→fix-N).
5. **Closure**: board (`runs/STATUS-0.8.z.md`) + one docs commit per transition; worktree cleaned.
6. **Release DoD** (Fathom is DoD at the end of each micro-release): all mod-5 slices CLOSED · X1/X2/X3
   satisfied (SDK parity + `mkdocs build` green + DOC-INDEX maintained) · the release's AC gate green ·
   **and the artifact is shippable** through the release machinery stood up in 0.8.6 (`/goal` may
   dry-run the publish as the final verification step).

**Cross-cutting DoD (X1/X2/X3), carried verbatim from 0.8.0/0.8.1, binds every slice in every release
here** — restated in each `plan-0.8.z.md`. Odd releases that are pure $0 eval (e.g. EXP-A/M4/B′) run the
same gate-and-close discipline but their "shippable" DoD is the **landed result doc + reproducible
script**, not a published artifact.

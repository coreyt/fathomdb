# Design: Typed Write

## Purpose

This document scopes the next design pass for the **canonical write and
projection-sync pipeline**.

The current writer thread boundary is correct, but the payload shape is still
raw SQL. The next step is to define typed write inputs that preserve the single
writer model without pushing SQL construction onto callers.

Companion docs:

- [setup-sqlite-vec-capability.md](./setup-sqlite-vec-capability.md)
- [setup-round-trip-fixtures.md](./setup-round-trip-fixtures.md)

## Layer Boundary

This layer starts before `BEGIN IMMEDIATE` and ends when canonical rows plus
required projections commit.

Responsibilities in scope:

- typed canonical write inputs
- pre-flight normalization and chunk preparation
- required projection derivation
- optional backfill handoff
- provenance attachment on canonical rows
- writer-thread transaction discipline

Responsibilities out of scope:

- read execution
- long-term approval semantics
- broad SDK ergonomics
- remote ingestion protocols

## Current Repository State

The writer scaffold proves the concurrency model, but not the API shape:

- `WriterActor` serializes writes on one thread
- `BEGIN IMMEDIATE` is already used correctly
- `WriteEnvelope` contains raw SQL statement lists
- tests currently seed the database with handwritten SQL strings

That is enough to validate the boundary, but not enough to preserve engine
invariants once real ingestion starts.

## Design Goals

1. Keep the single-writer discipline.
2. Move SQL construction inside the engine.
3. Keep heavy work outside the locked transaction.
4. Distinguish clearly between:
   - canonical writes
   - required projections
   - optional semantic backfills
5. Make provenance mandatory enough that repair tooling can rely on it later.

## Proposed Runtime Shape

### 1. Split Intent From Prepared Envelope

Introduce two stages instead of one raw envelope:

- `WriteRequest`
  - caller-facing typed intent
- `PreparedWrite`
  - engine-internal, fully normalized payload ready for transaction execution

That lets pre-flight work happen before the writer thread takes the lock while
still giving the writer a compact deterministic input.

### 2. Start With Narrow Canonical Types

Do not model every future ingestion case yet. Start with the v1 backbone:

- `NodeInsert`
- `EdgeInsert`
- `ChunkInsert`
- `RunInsert`
- `StepInsert`
- `ActionInsert`

Each should carry typed fields rather than opaque SQL fragments.

### 3. Make Provenance A First-Class Field

Every canonical type that can be traced or excised later should carry an
explicit `source_ref`.

If some rows are allowed to omit it in v1, that needs to be an explicit policy,
not an accidental omission. Repair tooling depends on this discipline.

### 4. Derive Required Projections Inside The Engine

The caller should not provide raw FTS insert statements. Instead, the engine
should derive required projection mutations from canonical data:

- `chunks` drive FTS rows
- enabled vector profiles drive vector projection work

Required projection derivation belongs in the write preparation path, not in the
caller.

### 5. Keep Optional Backfills Explicit

Optional semantic work should stay visible in the prepared write:

- what target profile or projection is missing
- what canonical row or chunk needs backfill
- what payload is required to do that work later

Do not hide optional work behind implicit side effects.

## Key Design Questions

1. Should IDs be generated by the caller, by the preparation stage, or by a
   shared engine ID service?
2. Should `PreparedWrite` contain typed records until execution time, or should
   it compile down to a smaller SQL-plus-bind representation just before the
   writer thread?
3. What is the minimum required write shape for v1:
   - nodes plus chunks only
   - or nodes, edges, and runtime tables from day one?
4. Which projections are required in v1:
   - FTS only
   - or FTS plus vector when capability is enabled?
5. How should superseding existing rows work in typed form when append-oriented
   updates begin?

## First Implementation Slice

1. Introduce typed insert structs for `nodes` and `chunks`.
2. Add a `prepare_write()` path that derives required FTS rows from chunk input.
3. Change `WriterActor` to execute typed prepared writes rather than arbitrary
   SQL strings.
4. Keep optional vector work as explicit backfill tasks.
5. Add one roundtrip test:
   - submit typed write
   - verify canonical rows
   - verify required FTS projection
   - verify optional backfill accounting

## Definition Of Done For This Design Pass

This layer is scaffolded enough when:

- callers no longer construct raw SQL for normal writes
- the engine owns required projection derivation
- provenance is explicit in typed write inputs
- the single-writer path still executes one atomic transaction
- one end-to-end write test proves canonical plus required projection behavior

## 1. Graph traversal (F1 / G5 neighbors / G6 search-with-expand) — `ADR-0.8.0-graph-traversal-scope`

Signed direction (revisable in 0.8.1):
- **Depth ceiling:** SDK `depth` arg defaults to **1**, **rejects > 3** with a typed argument error
  (not a silent clamp); engine carries an independent **hard cap of 50** (ported from v0.5.6
  `MAX_TRAVERSAL_DEPTH`, `compile.rs:253`) as a structural backstop.
- **Traversal filter at 0.8.1 (UPDATED 2026-06-12, option ②-B):** **`superseded_at IS NULL`**
  **AND edge valid-time G11 `t_valid`/`t_invalid` — now ACTIVATED, not deferred.** Traversal and
  the graph-retrieval arm honor `t_invalid IS NULL OR t_invalid > now` so invalidated facts drop
  out (invalidate-not-accumulate, Zep-shaped). This was reserved-additive in the Slice-32
  graph-model ADR (H3); 0.8.1 builds it. Adds the G11 columns + an additive migration (see §5.4).
- **Indexes:** `canonical_edges(from_id)` / `(to_id)` are already folded into G0 (Slice 15 step-12)
  → **no new migration** for the chosen depth ceiling (reserved-gap 36 not triggered).
- **Composition:** **G6 = G1 + G4 + G5 + G9** (retrieve + expand); **build G6 before standalone G5**.
- **Prior art:** the v0.5.6 `WITH RECURSIVE` depth-bounded BFS (visited-set `instr()` cycle guard,
  directional joins, cap 50) is directly portable once re-targeted `source/target_logical_id →
  from_id/to_id`; depth-1 is modeled on `trace_source_ref`.
- **Profiling:** depth/perf profiling is **0.8.1 impl-time tuning**, recorded as part of the impl
  acceptance criterion — it was not a blocker on this scope direction.

**Revisable knobs (expect 0.8.1 to revisit with evidence):** the SDK ceiling (≤3) and the exact
G6 composition/ordering. (The G11-valid-time question is now **decided in** per the 2026-06-12
scope expansion — see the banner above + §5.4.)

## 2. Filter grammar (G4 / F3) — `ADR-0.8.0-filter-grammar` (ACCEPTED)

## 4. eu7 recall-harness performance (MEASUREMENT-ONLY; surfaced by the 0.8.0 ◆ B-1 re-measure)

The 0.8.0 GA-2 / ◆ B-1 eu7 vector-stage re-measure runs ~2–2.5 h, dominated by serial embedding. Two
optimizations were identified and **deferred to 0.8.1** rather than touch the canonical test mid-GA. **Hard
guardrail: these change only the MEASUREMENT harness, never the engine system-under-test (the seed/projection
path stays as-shipped), and each must be proven byte-identical-result + codex §9 measurement-neutral BEFORE it
backs any gate number.** They do NOT speed the seed (that IS the engine path being measured and must stay serial-as-shipped); they only address the ground-truth side.

1. **Parallel ground-truth re-embed.** The GT `doc_vecs` loop (`eu7_real_corpus_ac.rs:804`,
   `doc_vecs.push(embedder.embed(body))`) is single-threaded (~25–45 min on the full corpus). It is
   embarrassingly data-parallel: give each worker its **own** `CandleBgeEmbedder` instance (the harness's
   `SerializedBge` `Mutex` guards a *shared* model against candle's concurrent-forward stall — independent
   instances are fine; ~133 MB each, trivial in the box's RAM). Near-linear speedup; deterministic ⇒
   byte-identical `doc_vecs` ⇒ identical recall number. Land behind a `--features`/env-gated parallel path.
2. **Source GT from the engine's retained f32 rerank column (bigger win).** The seed already embeds every body
   into the engine, which retains an uncentered-f32 vector for the rerank step
   (`ADR-0.7.0-vector-binary-quant.md` blast_radius). If the GT read those stored f32 vectors instead of
   re-embedding, the entire GT stage disappears (zero extra embed). Needs (a) verifying the engine persists the
   raw pre-quantization f32 in a queryable form, and (b) a measurement-purity argument that this is the SAME
   same-model f32 the codex "ground truth must use the candidate's model" focus requires. Higher risk than (1);
   evaluate first.

**Process:** both on their own branch, codex §9 confirming measurement-neutrality, before either is allowed to
produce a number that backs AC-075 or any future recall gate. Logged from the 0.8.0 GA-2 run notes.

## 5. Retrieval quality — R0–R4 (IR-C roadmap, brought into 0.8.1 on 2026-06-12)

Source: `dev/plans/runs/IR-C-roadmap.md` (Fable-5 roadmap), grounded by
`dev/plans/runs/IR-C-roadmap-analysis-dossier.md` + `…-deep-research.md`. This is the
**relevance/quality** half of 0.8.1, running alongside the graph-traversal + filter + eu7 work.
It is the concrete realization of the **IR-eval track** (`dev/plans/0.8.0-GA-and-IR-eval-roadmap.md`
IR-1 Ph3→Ph4): IR-C = the experiments that produced these levers; IR-D mints AC-077 from R0/R1/R2 data.

**Footprint is a hard constraint** (CPU-only, no API at the library boundary, 1-bit-binary-safe). All
of R0/R1/R4 are in-footprint; R3's *retrieval-time* math is in-footprint, its *index-time* fact
extraction uses a **caller-supplied / BYO-LLM** path (the consumers Memex/Hermes/OpenClaw are already
LLM agents — IR-C roadmap C7), so FathomDB-the-library adds no API dependency.

**Sequence (∥ = parallel, → = gates):** R0 ∥ R2-harness → R1 → R3 (rides the §1 graph substrate) ∥ R4.
R2 is built early because it is the **only instrument that can see R3's value** (graph payoff is
off the first-stage Recall@K axis — IR-C roadmap C3).

### 5.1 R0 — candidate-recall CDF + rerank cost model (entry gate; do first)
Measure found@K (K = 50/100/200/500/1000) per class + a CPU-reranker latency/cost curve. **Resets the
ceiling math:** the dossier's depth-50 ~0.62 ceiling is depth-conditional; lexical found@1000 ≈ 0.86,
so the reachable ceiling is a latency-bounded knob (IR-C roadmap C1/C2). Read-path measurement only,
no engine change. Gates R1's rerank depth. (~99% feasible, days.)

### 5.2 R2 — end-to-end Mem0/Zep-parity eval (instrument; decision ①)
A LongMemEval-style end-to-end answer-accuracy harness with an **identical answerer** over a **local
Mem0-OSS baseline**. **Decision ① (HITL 2026-06-12): AC-077 (Evidence Recall@K) stays the IR-eval
product GATE; R2 is added as a REPORT-ONLY north-star** for the "as-good-or-better than Mem0/Zep" goal —
not a replacement. R2 is also the only instrument that exposes R3's temporal/multi-hop payoff. Wire it
into the IR-1/IR-2 plan as the end-to-end axis. (~90% feasible; the decisive uncertainty-reducer.)

### 5.3 R1 — small CPU cross-encoder reranker in the `rerank_fused` seam
Drop a FlashRank-class ONNX cross-encoder (4–34 MB, no Torch/GPU/API) into the **existing
`rerank_fused` identity stub** (`lib.rs:3707`, from Slice 10/G9) over the fused top-K. Rerank depth
per R0. **No-regress gate: factoid R@10 ≥ 0.90** (cross-encoders can underperform BM25's ~0.95
within-pool conversion — IR-C roadmap C6); prefer RRF+CE score-blend over pure reorder. Expected
exploratory R@10 0.307 → ~0.40–0.47 (bounded by candidate recall at the rerank depth). 1-bit storage
untouched. (~85% helps / ~70% material.)

### 5.4 R3 — graph-retrieval arm + temporal fact-edges (full mechanism; option ②-B)
The retrieval-facing half of the §1 graph work, and the Zep-parity bet. **node/edge pick: fact-on-EDGE**
(Graphiti-shaped temporal fact-edges over an entity-node skeleton; HyperGraphRAG fact-NODES kept as the
signed H6 escape hatch). Components:
- **Schema (additive migration):** G11 edge columns `body` / `t_valid` / `t_invalid` / `confidence`
  (HITL-signed reserved-additive, `ADR-0.8.0-graph-model-and-edge-addressing.md` H3) — **activated here**,
  shared with §1's traversal filter.
- **Candidate generation:** project edge `body` text into FTS + use `canonical_edges(from_id)/(to_id)`
  (already in G0 step-12) → a **third RRF arm** fused with dense + lexical; n-hop BFS expansion reuses §1.
- **Temporal model:** invalidate-not-accumulate — a new fact sets the prior edge's `t_invalid`; retrieval
  filters `t_invalid IS NULL OR t_invalid > now`. This is the mechanism Zep's LongMemEval edge attributes to.
- **Index-time extraction:** caller-supplied / BYO-LLM fact→edge ingest API (no library API dependency).
- **Measurement:** payoff is mostly **off the Recall@K axis** → judged on R2's temporal/multi-hop/update
  classes; RRF determinism pins (`pr_g9_rrf_fusion.rs`) + the 1-bit floor gate must hold.
(~55–65% the mechanism helps on its classes; the BYO-LLM split lifts feasibility vs a local-CPU extractor.)

### 5.5 R4 — whole-doc long-context dense (research probe; floor-gated)
The one untested dense angle — a single long-context vector per doc (or late chunking), which the chunked
Nomic A/B never exercised. **Gated on the 1-bit binary floor ≥0.90 at the model's dim** (the V13
pooling-gate pattern). Timeboxed spike; competes with R1/R3 for capacity. (~25–35%.)

### 5.6 Capacity note (scope knowingly accepted)
R0–R4 **plus** the committed graph-traversal + filter-grammar + eu7 work is a large 0.8.1 (HITL accepted
the scope, 2026-06-12). If capacity forces a cut, the high-confidence in-footprint core is **R0 → R1 →
R3 (riding §1)** with **R2 early as the instrument**; R4 is the first to slip to 0.8.2. CLOSED levers
(do not revisit): chunked-dense embedder swaps, ColBERT/late-interaction (1-bit-incompatible), HyDE
(query-loop LLM), large rerankers — per IR-C roadmap §"CLOSED levers".

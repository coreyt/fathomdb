# FathomDB 0.8.3 — M2 full multi-hop study

> **Sequencing.** Step 2 of the three-version graph adjudication
> (0.8.2 M1 → **0.8.3 M2** → 0.8.4 S1). **Opens GATED on the 0.8.2 M1 verdict**
> ([`../plans/plan-0.8.2.md`](../plans/plan-0.8.2.md) §0 decision rule). A NO-GO from M1 does **not**
> cancel M2 — M2 tests *different* datasets and adds the *proven* multi-hop lever (IRCoT) — but the
> graph-traversal (PPR) arm inherits a documented "prior: negative" framing and a higher bar.

## 0. What M2 adds over M1

M1 (0.8.2) isolates one question on one dataset: *does graph topology (lexically-seeded PPR-fusion)
beat a strong baseline on MuSiQue multi-hop answer accuracy?* M2 widens coverage and, crucially, adds
the lever the literature says actually moves multi-hop — **iterative retrieval** — so any gain can be
attributed to *iteration vs structure*.

| Arm | Source | Role |
|---|---|---|
| BM25 / passage-dense / fused | carried from M1 | the baseline bar |
| **PPR-fusion** (lexically-seeded, IDF-weighted) | carried from M1 | the graph-topology hypothesis |
| **IRCoT iterative retrieve↔reason** | NEW (arXiv:2212.10509) | the *proven* multi-hop lever; composes with either index |
| **R5 — vector pseudo-relevance feedback** | folded in from IR-C R5 | the **cheap, no-LLM** form of feedback — "does PRF recover any of IRCoT's gain without the LLM cost?" |

## 1. Datasets (added to MuSiQue)

- **2WikiMultiHopQA** (COLING 2020) — ships gold **(subject, relation, object) reasoning-path
  triples**, enabling a **graph-specific diagnostic** no other set offers: *do our extracted edges
  match the gold reasoning path?* This directly tests whether the graph encodes the right structure,
  separate from end-task EM/F1.
- **MultiHop-RAG** (COLM 2024) — a **news** corpus with inference/comparison/temporal/null query
  types and evidence chains; closest to a real product corpus, non-Wikipedia, end-to-end RAG-shaped.
  (Also shares the news domain with 0.8.4's sensemaking corpus — one domain, both regimes.)
- MuSiQue-Ans carried from M1 (the shortcut-resistant arbiter + 2/3/4-hop dose-response).

## 2. Method notes

- **Attribution is the point.** Report each arm and the **PPR ⊕ IRCoT** combination so the study
  separates: baseline, graph-only, iteration-only, graph+iteration. The research expectation is that
  iteration carries the multi-hop gain and graph topology adds little on top — M2 measures whether
  that holds on FathomDB's substrate.
- **R5 vector-PRF** is the cheap control beside IRCoT (average query vector with top-k reranked
  passage vectors, re-run dense; no extra neural inference). Drop without regret if flat.
- **Fairness carries from M1:** strong passage-dense baseline (not whole-doc R4), same answerer/depth
  across arms, pre-registered endpoints + power/MDE, versioned artifacts, $ ledger, cheap-validate
  before priced runs.
- **Seed-quality / node-granularity** ablations from M1 Slice 15 (entity vs passage nodes; F9
  confidence-weighting as a knob only) graduate here if M1 left them open.

## 3. Slice shape (to be detailed in dev/plans/plan-0.8.3.md when 0.8.2 closes)

Anticipated mod-5 ladder: `0 design+pre-register → 5 add 2Wiki + path-match diagnostic → 10 add
MultiHop-RAG → 15 IRCoT arm (+ R5 PRF) KEYSTONE → 20 cross-dataset verdict (attribution: iteration vs
structure) → GO/NO-GO to 0.8.4 funding`. Reuses the M1 harness wholesale; new code is the IRCoT loop,
the 2Wiki path-match scorer, and the dataset adapters.

## 4. Folded in / out

- **Folded IN:** **R5 — vector pseudo-relevance feedback** (was IR-C "opportunistic post-R1"; its
  natural home is beside IRCoT as the no-LLM feedback control).
- **Stays OUT (0.8.5):** R4 whole-doc dense (wrong dense granularity for multi-hop), tunable-b FTS5,
  R3b, the portable-DB vector guard.

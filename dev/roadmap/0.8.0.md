# 0.8.0 Roadmap

This file is the durable home for items deliberately postponed from
0.6.0 that should not slip into 0.7.0. Each entry records the source,
the deferral rationale, and the cross-reference back to the document
that originated the item.

## Security hardening

### SR-005 — Sidecar lock file symlink / TOCTOU hardening

**Source:** `dev/security-review.md` SR-005.
**Severity at deferral:** low.
**Rationale for 0.8.0 (not 0.7.0):** Attack requires local filesystem
write access in the database parent directory; in the embedded-library
trust model that is the same privilege the host application already
holds. 0.7.0 work is expected to be filled by the implementation
backlog from the 0.6.0 freeze; this hardening is a deliberate
defense-in-depth follow-up rather than an immediate fix.
**Scope when picked up:**

- POSIX: open `<db>.sqlite.lock` with `O_NOFOLLOW | O_CLOEXEC`.
- Windows: check `GetFileAttributesW` for `FILE_ATTRIBUTE_REPARSE_POINT`
  before `CreateFileW`; reject if set.
- Existing-but-non-regular lock file: fail open with the typed
  `DatabaseLocked` (or new typed open error) rather than silently
  following.
- Add an acceptance test that pre-creates the lock path as a symlink to
  an unrelated file and asserts the engine refuses to open and does not
  truncate the symlink target.
  **Cross-references:** `dev/design/engine.md` open-path step 2;
  `dev/security-review.md` SR-005.

### SR-011 — Hardened opening of attacker-controlled SQLite files

**Source:** `dev/security-review.md` SR-011.
**Severity at deferral:** informational (inherited posture from
upstream SQLite).
**Rationale for 0.8.0 (not 0.7.0):** SQLite upstream does not support
opening attacker-controlled DB files; FathomDB inherits this. 0.6.0
documents the inherited limitation. Hardening (sandboxed-open or
pre-flight static validation) is non-trivial and depends on upstream
SQLite posture; it is not a 0.7.0 priority and should not block
0.7.0 implementation work.
**Scope when picked up (one or both, not necessarily both):**

- Sandboxed open: a separate process or restricted-syscall context
  performs the initial open + structural probe; main engine adopts only
  if the probe passes.
- Pre-flight static validation: a SQLite-format aware validator runs
  against the file bytes before any SQLite C call, rejecting files
  whose page header / freelist / b-tree shape is structurally invalid
  in ways the always-on 0.6.0 detection does not catch.
- Acceptance bar to be drafted as part of 0.8.0 design phase.
  **Cross-references:** `dev/design/engine.md` (Untrusted-file posture);
  `dev/security-review.md` SR-011.

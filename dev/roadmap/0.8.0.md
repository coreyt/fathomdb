# 0.8.0 Roadmap

0.8.0 anchor (locked HITL 2026-05-24): **knowledge-store and
retrieval features required by Memex.** The canonical-identity
substrate (originally a 0.7.x candidate) lands here because the
substrate is the storage primitive Memex's knowledge-store /
retrieval consumes — entities with stable identity across
re-ingestion, supersession with audit trail, and erase-by-identity.

This file is the durable home for items deliberately postponed from
0.6.x and 0.7.0. Each entry records the source, the deferral
rationale, and the cross-reference back to the document that
originated the item.

## Release-line goals (recorded 2026-05-24)

- **0.6.x** — fully functional, including TypeScript parity
  (`Engine.open` returns OpenReport on py + ts bindings; Dependabot
  advisories resolved; axis-E independence demonstrated). 0.6.1 is
  the active patch release; see `dev/plans/0.6.1-implementation.md`.
- **0.7.0** — improve performance. Perf-only single-thrust release.
  Two prongs: (1) revise AC-012 / AC-013 / AC-019 / AC-020 budgets
  empirically against the canonical-runner measurements taken in
  the 0.6.1 release line; (2) tune SQLite + FathomDB (top-K LIMIT
  cap on the search verb is one engine lever among others). See
  `dev/plans/prompts/0.7.0-planning.md` for the planning prompt
  (scope amendment header dated 2026-05-24).
- **0.8.0** — knowledge-store and retrieval features for Memex,
  plus the canonical-identity substrate that they consume, plus
  the logical-id CLI verbs that operate on that substrate, plus
  the security-hardening items below.

## Knowledge-store + retrieval anchor (Memex)

**Source:** 2026-05-24 HITL direction.
**Rationale for 0.8.0:** Memex consumes FathomDB as its
local-first retrieval engine. The knowledge-store layer
(persistent entity graph + relation index + provenance lineage)
and the retrieval layer (semantic search + structured filters +
top-K with rank fusion) are the next product surface beyond the
0.6.0 five-verb engine. Both depend on canonical-identity
supersession (stable identity across re-ingestion) and on the
perf revisions landing in 0.7.0.
**Scope when picked up:**

- Knowledge-store schema: entity tables + relation tables + audit
  lineage, all on top of the canonical-identity substrate.
- Retrieval verbs: top-K semantic + structured filter + rank
  fusion (cf. the 0.6.0 five-verb base — these are additive, not
  replacement).
- SDK surfaces (Python + TypeScript): bindings for the new
  retrieval verbs.
- ADR drafts that lock the knowledge-store shape and the
  retrieval-API contract.

**Cross-references:**

- `dev/plans/0.6.0-implementation.md` (substrate dependency tree).
- 0.6.0 memex-compatibility note (`c1eef97` 2026-05-22).
- `dev/plans/prompts/0.7.0-planning.md` (perf work landing first).

## Canonical-identity substrate + logical-id verbs

**Source:** `dev/design/recovery.md` § "Logical-id purge and
restore" + `dev/adr/ADR-0.6.0-cli-scope.md` 2026-05-16 amendment.
**Originally deferred:** 0.7.x (per the 2026-05-16 amendment).
**Moved to 0.8.0:** HITL 2026-05-24. Rationale: the substrate is
the storage primitive Memex's knowledge-store consumes; landing
them together avoids implementing supersession twice and lets the
substrate's design be informed by the actual knowledge-store
schema requirements. 0.7.0 is now perf-only (single-thrust) and
does not block on substrate-ADR HITL sign-off.
**Scope when picked up:**

- **Canonical-identity substrate** — additive schema bump on
  `canonical_nodes` / `canonical_edges`: `row_id`, `logical_id`,
  `superseded_at`, optional `restore_provenance` payload; partial
  unique index on `(logical_id, kind)` excluding superseded rows.
  Writer takes `logical_id` in `PreparedWrite`; supersession step
  writes `superseded_at` on prior row in same txn. Op-store
  cascade contract (`latest_state` + `append_only_log`) defined.
  Requires substrate ADR (`ADR-0.8.0-canonical-identity-substrate`)
  drafted as part of 0.8.0 planning.
- **`fathomdb recover --purge-logical-id <id>`** — hard-delete
  every row version associated with `<id>` (current + all
  superseded) and cascade through projections / vec0 / FTS5 /
  op-store. Use case: GDPR-style erasure; correcting "this
  entity was wrong from the start" mistakes. Distinct from
  `--excise-source <source_ref>` (which removes by single-write
  batch).
- **`fathomdb recover --restore-logical-id <id>`** — resurrect
  the most-recently-superseded row, making it current again.
  Idempotent on already-current. Use case: undo a bad write
  that superseded a good one.
- **Forward-migration policy** — substrate columns are additive
  per `feedback_no_data_migration` but legacy rows need backfill
  so the verbs are correct on 0.7.0 → 0.8.0 upgrades. Two
  candidate shapes (in-place column-add migration vs re-open
  required); HITL decides as part of substrate-ADR sign-off.

**Cross-references:**

- `dev/design/recovery.md` § "Logical-id purge and restore"
  (deferral note updated 2026-05-24: target moved from 0.7.x to
  0.8.0).
- `dev/adr/ADR-0.6.0-cli-scope.md` 2026-05-16 amendment
  (deferral target updated 2026-05-24).
- `dev/plans/runs/10b-B-purge-restore-output.json` (the

- `dev/plans/prompts/0.7.0-planning.md` (historical record of

## Security hardening

### SR-005 — Sidecar lock file symlink / TOCTOU hardening

**Source:** `dev/security-review.md` SR-005.
**Severity at deferral:** low.
**Rationale for 0.8.0 (not 0.7.0):** Attack requires local filesystem
write access in the database parent directory; in the embedded-library
trust model that is the same privilege the host application already
holds. 0.7.0 work is expected to be filled by the implementation
backlog from the 0.6.0 freeze; this hardening is a deliberate
defense-in-depth follow-up rather than an immediate fix.
**Scope when picked up:**

- POSIX: open `<db>.sqlite.lock` with `O_NOFOLLOW | O_CLOEXEC`.
- Windows: check `GetFileAttributesW` for `FILE_ATTRIBUTE_REPARSE_POINT`
  before `CreateFileW`; reject if set.
- Existing-but-non-regular lock file: fail open with the typed
  `DatabaseLocked` (or new typed open error) rather than silently
  following.
- Add an acceptance test that pre-creates the lock path as a symlink to
  an unrelated file and asserts the engine refuses to open and does not
  truncate the symlink target.
  **Cross-references:** `dev/design/engine.md` open-path step 2;
  `dev/security-review.md` SR-005.

### SR-011 — Hardened opening of attacker-controlled SQLite files

**Source:** `dev/security-review.md` SR-011.
**Severity at deferral:** informational (inherited posture from
upstream SQLite).
**Rationale for 0.8.0 (not 0.7.0):** SQLite upstream does not support
opening attacker-controlled DB files; FathomDB inherits this. 0.6.0
documents the inherited limitation. Hardening (sandboxed-open or
pre-flight static validation) is non-trivial and depends on upstream
SQLite posture; it is not a 0.7.0 priority and should not block
0.7.0 implementation work.
**Scope when picked up (one or both, not necessarily both):**

- Sandboxed open: a separate process or restricted-syscall context
  performs the initial open + structural probe; main engine adopts only
  if the probe passes.
- Pre-flight static validation: a SQLite-format aware validator runs
  against the file bytes before any SQLite C call, rejecting files
  whose page header / freelist / b-tree shape is structurally invalid
  in ways the always-on 0.6.0 detection does not catch.
- Acceptance bar to be drafted as part of 0.8.0 design phase.
  **Cross-references:** `dev/design/engine.md` (Untrusted-file posture);
  `dev/security-review.md` SR-011.

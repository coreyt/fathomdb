# FathomDB 0.8.4 — S1 GraphRAG sensemaking

> **Sequencing + framing (revised 2026-06-19).** Originally step 3 of three (0.8.2 M1 → 0.8.3 M2 →
> 0.8.4 S1). The multi-hop track has since resolved **negative**: M1 closed **NO-GO**
> ([`../plans/runs/0.8.2-m1-FINDINGS.md`](../plans/runs/0.8.2-m1-FINDINGS.md)) and M2 was **dropped**
> (0.8.3 redirected to non-graph retrieval — [`0.8.3.md`](0.8.3.md)). **S1 is now the SOLE surviving
> graph hypothesis.** It is a *different graph structure* (community summaries, not fact-edge traversal)
> on a *different axis* (global sensemaking, not needle/multi-hop retrieval), so the multi-hop negatives
> do **not** refute it — research (2026-06-16) flags sensemaking as the axis where a graph win is *most*
> plausible. But be honest: the cross-graph prior across 0.8.1+0.8.2 is now **strongly negative**, so S1
> must clear a high bar in design review *before* its (largest-of-the-three) build is funded. Decide S1
> on its own merits against a strong vector-RAG + long-context baseline; a third graph null would
> settle the graph question for FathomDB.

## 0. Why this is a different paradigm

| | 0.8.1 / M1 / M2 graph | S1 GraphRAG graph |
|---|---|---|
| Structure | entities as nodes, **facts as temporal edges** (Graphiti/Zep-shaped) | entities + **Leiden communities** + **LLM community summaries** (hierarchical) |
| Retrieval | traverse fact-edges / PPR over nodes for specific evidence | **map-reduce over community summaries** for global synthesis |
| Job | needle-recall / multi-hop answer | **global sensemaking** ("what are the themes?") |
| Eval | EM/F1, recall@K (reference-based) | **LLM-judge win-rate** on comprehensiveness/diversity/empowerment (reference-free) |

FathomDB has only ever tested the left column. S1 tests the right — the structure and task GraphRAG
("From Local to Global", arXiv:2404.16130) was actually built for.

## 1. Method (BenchmarkQED-aligned)

- **Build:** entities/relationships → **Leiden community detection** → **LLM-generated community
  summaries** (hierarchical C0–C3 levels) → map-reduce query-focused summarization. The C0–C3
  conditions literally measure the value of the community hierarchy.
- **Eval harness — Microsoft BenchmarkQED** (June 2025): **AutoQ** (persona→task→question synthesis
  spanning local↔global, incl. multi-hop-local), **AutoE** (pairwise LLM-judge win-rates), **AutoD**.
  Ships the openly-licensed **AP-News set (~1,397 articles)** — and *news* is also 0.8.3's
  MultiHop-RAG domain, so the two regimes can run over one corpus family.
- **Baselines:** vector RAG **and** a long-context "stuff-it-all-in" control (the honest upper bar for
  a corpus that fits a long context window).
- **Non-negotiable bias controls** (the LLM-judge failure modes): randomize/swap answer order and
  average (position bias, arXiv:2406.07791); **≥5 runs** (stochasticity); a **judge from a different
  model family** than the systems under test (self-preference, arXiv:2410.21819); plus a length-bias
  control (Directness/claim-count, the GraphRAG paper's non-judge corroboration).

## 2. Honest expectation

Unlike multi-hop, this is a **genuinely open** bet — GraphRAG's reported sensemaking win-rates over
vector RAG are large (~72–83% comprehensiveness). But it is also **expensive** (community-summary
construction is LLM-heavy at index time) and the eval is reference-free LLM-judging, which is why the
bias controls above are load-bearing. The Samsung result ("VectorRAG is almost enough",
arXiv:2603.29875) is the skeptical prior to beat.

## 3. Cost + footprint caveat

S1 is the one initiative that **strains the CPU/no-API footprint invariant**: community summaries
require an LLM at *index* time (offline; the local Qwen3.6-27B extractor may suffice, to be measured)
and the judge is a priced LLM. Budget discipline + a running $ ledger apply; the index-time LLM stays
the *offline build* seam, never the library query boundary. Whether GraphRAG-style summaries are even
worth productizing in FathomDB is itself one of the questions S1 answers.

## 4. Slice shape (to be detailed in dev/plans/plan-0.8.4.md when 0.8.3 closes)

Anticipated mod-5 ladder: `0 design+pre-register (endpoints, bias controls, judge model) → 5 corpus +
vector-RAG + long-context baselines → 10 Leiden communities + community summaries build → 15
map-reduce QFS arm KEYSTONE → 20 AutoE LLM-judge adjudication + verdict`. New infra: Leiden clustering,
the community-summary builder, and the BenchmarkQED-style AutoQ/AutoE harness — the largest new build
of the three versions, justified only if S1's hypothesis survives design review.

## 5. Known gap carried from 0.8.3 — competitor as-is head-to-heads (cross-capability)

> **Recorded 2026-06-21 (deferred from 0.8.3, which is kept Mem0-focused).** This is **distinct from S1**:
> S1 tests *FathomDB's own* community-summary build on the *sensemaking* axis; this gap measures *external
> competitor implementations* head-to-head on the capabilities FathomDB already evaluates, to replace the
> **literature-only / not-apples-to-apples** competitor rows in
> [`../plans/runs/0.8.x-capability-status-report.md`](../plans/runs/0.8.x-capability-status-report.md)
> with **measured** rows.

**The gap.** After 0.8.3, FathomDB has exactly **one** real same-harness competitor comparison: **Mem0-OSS**
on agentic memory (0.8.3 D0, identical-answerer, re-pinned gold). It has **no** head-to-head against:
- **Graphiti / Zep** (agentic-memory competitor) — currently literature-only (e.g. arXiv:2501.13956), and
  LongMemEval scores swing 15–30 pts with reader/judge, so the published number is not comparable to ours.
- **GraphRAG (Microsoft) / HippoRAG-2** (multi-hop competitor) — no running system was stood up in 0.8.2
  M1 or 0.8.3 (the M1 graph arm was FathomDB's *own* `ppr_fusion`, not GraphRAG/HippoRAG); their MuSiQue
  numbers (e.g. HippoRAG-2 51.9 F1) use a different reader/cut/metric and cannot be placed beside ours.

**Close it in 0.8.4 by reusing existing harnesses (no new eval paradigm):**
- **G-HH-1 — Graphiti/Zep on agentic memory.** Reuse the 0.8.3 **D0 R2 identical-answerer adapter** +
  re-pinned gold corpus; stand up a local Graphiti/Zep backend behind the shared adapter → a true
  `fathomdb_minus_graphiti` delta under one answerer/corpus.
- **G-HH-2 — GraphRAG/HippoRAG-2 on multi-hop.** Reuse the **M1/D1/D2 MuSiQue harness** + same
  answerer/corpus; wire the competitor as a retrieval backend → a true head-to-head ≥3-hop F1 / all-bridges@K.
- **Feedback:** both write **measured** rows back into the capability-status report, clearing its
  conclusion's #1 caveat (cross-vendor numbers are positioning, not a leaderboard).

**Scope note.** G-HH-1/2 are *measurement* workstreams (competitor adapters + runs), footprint-safe
(EVAL-ONLY; competitor LLMs, if any, are competitor-side, not the FathomDB library boundary). They can run
independently of the S1 hypothesis test and should be slice-planned in `dev/plans/plan-0.8.4.md` alongside S1.

# FathomDB 0.8.2 — bundled CPU local-LLM extractor (R3b)

> **Why this is 0.8.2, not 0.8.1 (HITL coreyt, 2026-06-12).** The 0.8.1 graph work
> (`dev/roadmap/0.8.1.md` §5.4) ships the full temporal fact-edge *retrieval* mechanism with
> **BYO-LLM construction** — FathomDB proxies extraction to a caller-supplied harness (the
> consumer's own LLM; see `dev/plans/prompts/IR-C-byo-llm-extraction-harness-memex.md`). That keeps
> FathomDB no-API/CPU and lets the graph mechanism prove out first. **R3b — bundling a CPU local LLM
> so FathomDB can build the graph with no caller LLM at all — is the optional "fully self-contained"
> upgrade, and the lowest-probability link in the whole IR-C roadmap (~40–50%,
> `dev/plans/runs/IR-C-roadmap.md` R3 / §3 goal assessment).** It is sequenced AFTER 0.8.1 so the
> two risks are independent: ship the capability via BYO-LLM, then attempt the self-contained version.

## 0. Cross-cutting backlog (added 2026-06-15)

- **Vector-equivalence guard for portable DBs / backend swap (B.2.b).** Store a small
  canonical probe set (`_fathomdb_embed_probe`: ~16–32 fixed strings + reference vectors)
  in the DB at first vector-kind registration; on open / cross-machine move / backend
  change, re-embed it with the live embedder and refuse the dense arm if it diverges beyond
  tolerance (calibrated to the 1-bit quantization floor). Strengthens `EmbedderIdentity`
  (identity = *claims* same embedder; probe set = *proves* equivalent vectors here). Required
  before advertising portable DBs, runtime backend-swap, or moving a CUDA-built index to a
  CPU/other-vendor reader. Design: `dev/design/0.8.1-embedder-gpu-and-portability.md` §3.
  (GPU enablement itself ships in 0.8.1 and does not require this — it rebuilds a fresh
  index — but this MUST land before any cross-machine/backend DB reuse.)

## 1. Goal

Let FathomDB construct its memory graph **with zero external LLM dependency** — a bundled,
in-process, **CPU-only** model that implements the *same* Extraction Provider Protocol the BYO-LLM
harness does (`fathomdb.extract.v1`), so it is a **drop-in provider** behind the identical interface.
A consumer with no LLM (or one that wants a turnkey local stack) gets graph memory out of the box;
a consumer that already has an LLM keeps using BYO-LLM. Same protocol, two providers.

## 2. Why it is the risky link (carry the evidence)

- **Extraction quality at small scale is unproven.** Non-LLM extractors (REBEL) cause large drops;
  small/quantized models show entity/relation extraction-quality + JSON-stability degradation
  (`dev/plans/runs/IR-C-roadmap-deep-research.md` §d, explicitly untested). LightRAG supports local
  open models but at Qwen3-30B-class sizes, above a comfortable CPU budget.
- **Index-time CPU throughput is a real cost** over long transcripts even when quality holds — this
  is the seed/ingest path, not query time.
- **Footprint must hold:** CPU-only, no API, candle/GGUF-runnable in-process (consistent with the
  embedder constraint gate in `dev/notes/IR-C-embedder-options-research.md`). A model that needs a
  GPU or an API is out by definition.

## 3. Scope (when 0.8.2 opens)

1. **Model selection + constraint gate.** Pick a candidate CPU extractor (≤~4B, GGUF via llama.cpp,
   or a candle-runnable small instruct model) under: CPU-only, in-process, permissive license,
   acceptable ingest throughput, **structured-JSON-stable** at temperature 0. Bench candidates on a
   held-out extraction set (entity/relation/temporal-date accuracy + JSON validity rate).
2. **Provider implementation.** A bundled provider behind `fathomdb.extract.v1` (the BYO-LLM
   protocol) — so swapping BYO-LLM ↔ local is a config choice, not a code change. Determinism mode
   (greedy + fixed seed) for reproducible ingest + golden tests.
3. **Quality gate vs BYO-LLM.** Measure the local extractor against a strong BYO-LLM reference on the
   golden fixture + the R2 memory-class eval: does locally-built graph retain the temporal/multi-hop
   gains, or does extraction noise erase them? **Gate: local-extractor R2 memory-class accuracy
   within an acceptable delta of BYO-LLM** (delta TBD when 0.8.2 opens, grounded by R2 data).
4. **Throughput budget.** A documented ingest-rate floor (docs/min on a reference CPU) and a
   batch/async ingest path so extraction does not block the writer.

## 4. Dependencies / gating

- **After 0.8.1:** the graph mechanism (R3) + BYO-LLM protocol must ship and the R2 eval must exist —
  R2 is what tells us whether a *good* graph even helps (if R3a fails on FathomDB's corpus, R3b is
  moot). Do not build the self-contained extractor before the mechanism is proven.
- **Protocol-frozen:** R3b implements the v1 protocol unchanged; any protocol change is a shared
  0.8.1 concern, not an R3b fork.
- **Decision to even pursue:** if BYO-LLM proves sufficient for the named consumers (all LLM agents),
  R3b may stay a *nice-to-have* — revisit its priority against other 0.8.2 candidates with the 0.8.1
  evidence in hand.

## 5. Success criterion

A consumer can enable FathomDB's bundled CPU extractor and build a temporal fact-edge graph with **no
external LLM**, retaining most of the BYO-LLM graph's R2 memory-class gains, within a documented
CPU ingest-throughput budget — making the fully-local "match-or-approach Mem0/Zep with zero API
calls" stack real end-to-end, not just at retrieval time.

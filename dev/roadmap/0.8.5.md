# FathomDB 0.8.5 — displaced / contingent items

> **Why these live here (HITL coreyt, 2026-06-16).** 0.8.2–0.8.4 were re-sequenced to host the
> three-step graph adjudication (M1 multi-hop / M2 full study / S1 sensemaking —
> [`0.8.2.md`](0.8.2.md)/[`0.8.3.md`](0.8.3.md)/[`0.8.4.md`](0.8.4.md)). The items below were displaced
> from 0.8.2 or were floating unversioned "0.8.x"; none was a graph-adjudication dependency. They land
> at 0.8.5 (or later), several **gated on the adjudication verdict**.

## 1. R3b — bundled CPU local-LLM fact extractor · **GATED on the M1–S1 verdict**

The "fully self-contained" upgrade to 0.8.1's BYO-LLM graph construction: FathomDB bundles a CPU-only
local LLM (≤~4B, GGUF via llama.cpp or a candle-runnable instruct model) implementing the
`fathomdb.extract.v1` protocol, so a consumer with no LLM gets graph memory out of the box.

**Re-gating (2026-06-16):** R3b productizes the graph extractor *inside the binary*. The 0.8.2–0.8.4
adjudication is precisely the evidence that decides whether a graph earns that engineering. **Do not
open R3b until M1/M2/S1 show a graph delivers value on some axis.** If they don't, R3b stays closed (or
narrows to "BYO-LLM is sufficient for the named LLM-agent consumers"). The full R3b scope, quality
gate, and throughput-budget framing are preserved from the prior 0.8.2 direction — see git history of
`dev/roadmap/0.8.2.md` (pre-2026-06-16) and `dev/plans/runs/IR-C-roadmap.md` R3.
- Constraint gate: CPU-only, in-process, permissive license, structured-JSON-stable at temp 0.
- Quality gate: local-extractor parity within an acceptable delta of a strong BYO-LLM reference on the
  relevant eval class — now the **M1/M2/S1 graph evals**, not the shelved LongMemEval recall metric.

## 2. Portable-DB vector-equivalence guard (B.2.b) · infra hygiene, independent of the graph track

Store a small canonical probe set (`_fathomdb_embed_probe`: ~16–32 fixed strings + reference vectors)
at first vector-kind registration; on open / cross-machine move / backend change, re-embed it and
**refuse the dense arm if vectors diverge** beyond the 1-bit-quantization-floor tolerance. Strengthens
`EmbedderIdentity` (identity *claims* the same embedder; the probe set *proves* equivalent vectors
here). **Required before advertising portable DBs, runtime backend-swap, or moving a CUDA-built index
to a CPU/other-vendor reader.** Not a graph-adjudication dependency; sequence whenever portable-DB
reuse is on the table. Design: `dev/design/0.8.1-embedder-gpu-and-portability.md` §3.

## 3. R4 — whole-doc long-context dense probe · research probe (~25–35%)

Embed full documents with a long-context model (nomic-embed-text-v1.5, 8192-ctx, MRL) + a late-chunking
variant. **Explicitly NOT pulled into M1/M2:** multi-hop wants **passage-level** dense, and whole-doc
vectors blur discrimination (the measured median-99 failure mode). Floor-gated ≥0.90 at the model dim;
kill on a flat dense diagnostic. Full design + probability: `dev/plans/runs/IR-C-roadmap.md` §R4.

## 4. Tunable-/lower-`b` FTS5 ranking + enrichment · deferred lexical lever (uncertain)

The one config with a clean directional rationale from the 0.8.1 "beat BM25" work (lower `b` gave
+0.05; FTS5 wins on tokenizer), but FTS5's `b` is fixed → needs custom-ranking **engine** work;
uncertain at N=40. **Pursue only if recall work resumes**, and **not** as a baseline-tuning knob inside
the graph adjudication (tuning the baseline mid-test would handicap the comparison). Data + rationale:
`dev/plans/runs/0.8.1-beat-bm25-report.md`, `runs/STATUS-0.8.1.md` §7 (2026-06-16).

## 5. EU7 harness perf (parallel GT re-embed + retained-f32 reuse) · measurement-only

Parallel GPU embedding + retained-f32 reuse for the EU7 harness; byte-identical result, codex
measurement-neutral gate. Opportunistic perf if bandwidth allows; unrelated to the graph track.
Origin: `dev/plans/0.8.1-plan.md` reserved-gap notes.

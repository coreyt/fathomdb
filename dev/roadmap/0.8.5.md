# FathomDB 0.8.5 — displaced / contingent items

> **Why these live here (HITL coreyt, 2026-06-16; updated 2026-06-19).** 0.8.2–0.8.4 were re-sequenced
> to host the graph adjudication. That track has since partly resolved: **M1 closed NO-GO** (multi-hop
> graph refuted), **M2 was dropped** (0.8.3 redirected to non-graph retrieval), and **S1 (0.8.4
> sensemaking) is the sole surviving graph hypothesis**. The items below were displaced from 0.8.2 or
> were floating unversioned "0.8.x"; none was a graph-adjudication dependency. They land at 0.8.5 (or
> later); R3b's gate is now **effectively S1 alone** (see §1).

## 1. R3b — bundled CPU local-LLM fact extractor · **GATED on S1 (M1 already NO-GO)**

The "fully self-contained" upgrade to 0.8.1's BYO-LLM graph construction: FathomDB bundles a CPU-only
local LLM (≤~4B, GGUF via llama.cpp or a candle-runnable instruct model) implementing the
`fathomdb.extract.v1` protocol, so a consumer with no LLM gets graph memory out of the box.

**Re-gating (2026-06-16; updated 2026-06-19):** R3b productizes the graph extractor *inside the binary*.
The graph adjudication is the evidence that decides whether a graph earns that engineering — and it has
mostly come back **negative**: **M1 NO-GO** (multi-hop graph refuted) + **M2 dropped**. The R3b gate
therefore now hinges on **S1 (0.8.4 GraphRAG sensemaking) alone**: open R3b only if S1 shows a graph
delivers value. If S1 also nulls, R3b stays closed (or narrows to "BYO-LLM is sufficient for the named
LLM-agent consumers"). The full R3b scope, quality gate, and throughput-budget framing are preserved
from the prior 0.8.2 direction — see git history of
`dev/roadmap/0.8.2.md` (pre-2026-06-16) and `dev/plans/runs/IR-C-roadmap.md` R3.
- Constraint gate: CPU-only, in-process, permissive license, structured-JSON-stable at temp 0.
- Quality gate: local-extractor parity within an acceptable delta of a strong BYO-LLM reference on the
  relevant eval class — now the **M1/M2/S1 graph evals**, not the shelved LongMemEval recall metric.

## 2. Portable-DB vector-equivalence guard (B.2.b) · infra hygiene, independent of the graph track

Store a small canonical probe set (`_fathomdb_embed_probe`: ~16–32 fixed strings + reference vectors)
at first vector-kind registration; on open / cross-machine move / backend change, re-embed it and
**refuse the dense arm if vectors diverge** beyond the 1-bit-quantization-floor tolerance. Strengthens
`EmbedderIdentity` (identity *claims* the same embedder; the probe set *proves* equivalent vectors
here). **Required before advertising portable DBs, runtime backend-swap, or moving a CUDA-built index
to a CPU/other-vendor reader.** Not a graph-adjudication dependency; sequence whenever portable-DB
reuse is on the table. Design: `dev/design/0.8.1-embedder-gpu-and-portability.md` §3.

## 3. R4 — whole-doc long-context dense probe · research probe (~25–35%)

Embed full documents with a long-context model (nomic-embed-text-v1.5, 8192-ctx, MRL) + a late-chunking
variant. **Explicitly NOT pulled into M1/M2:** multi-hop wants **passage-level** dense, and whole-doc
vectors blur discrimination (the measured median-99 failure mode). Floor-gated ≥0.90 at the model dim;
kill on a flat dense diagnostic. Full design + probability: `dev/plans/runs/IR-C-roadmap.md` §R4.

## 4. Tunable-/lower-`b` FTS5 ranking + enrichment · ▶ PROMOTED TO 0.8.3 (2026-06-21, ADJ-2/ADJ-7)

> **MOVED OUT of 0.8.5.** The 0.8.3 capability-report review (ADJ-2) adopted D2's **strong fielded form**
> (fielded-FTS/BM25F + a tunable-/lower-`b` length-norm path), which is exactly this lever — so it is
> **pulled forward into 0.8.3 Slice 25** (engine), gated by the **Slice-15b $0 content-at-scale proxy**
> (ADJ-7) and ratified at the 0.8.3 Slice-0 F5-scope HITL gate. See `../plans/plan-0.8.3.md` §4 Slice 25
> and `../roadmap/0.8.3.md` §1c. **No longer a 0.8.5 item** unless the Slice-15b proxy fails and D2 defers.

The original rationale (retained for the record): the one config with a clean directional rationale from
the 0.8.1 "beat BM25" work (lower `b` gave +0.05; FTS5 wins on tokenizer), but FTS5's `b` is fixed →
needs custom-ranking **engine** work; uncertain at N=40. The 0.8.1 concern — *don't* tune the baseline
mid-test — is honoured in 0.8.3 by validating enrichment via the placebo-controlled Slice-10b proxy
*before* the engine `b` path is built. Data: `dev/plans/runs/0.8.1-beat-bm25-report.md`,
`runs/STATUS-0.8.1.md` §7 (2026-06-16).

# FathomDB 0.5.0 scope

Eight items for 0.5.0, plus a tracked path to 1.0.0. Design docs exist for
items 1–6; items 7–8 are grounded in open GitHub issues.

## Items

1. **`TextQuery` AST** — typed FTS query replacing raw `String` in
   `QueryStep::TextSearch` (closes #37, #31)
2. **Adaptive search response shape** — `SearchRows` wrapper with
   degradation signals; per-hit `match_mode` and `source`
3. **FTS Level 3** — recursive descendant-value indexing for object-typed
   property paths (closes #36)
4. **TypeScript embedding adapters** — `OpenAIEmbedder`, `JinaEmbedder`,
   `StellaEmbedder`, `SubprocessEmbedder` mirroring Python 0.4.5 surface
5. **`QueryEmbedder::max_tokens()`** — Rust trait extension + engine-side
   chunking integration for long-context embedders
6. **Per-kind vec tables** — move from global `('*', 'vec')` sentinel to
   per-kind `(kind, 'vec')` keying
7. **Builtin embedder write-time path** — wire `BuiltinBgeSmallEmbedder`
   into `regenerate_vector_embeddings`; closes read/write asymmetry (closes #39)

---

## Item 1: `TextQuery` AST

**Design doc:** `dev/notes/design-text-query-ast-2026-04-11.md`
**Implementation plan:** `dev/notes/implementation-plan-text-query-tdd-2026-04-11.md`

### What changes

`QueryStep::TextSearch` currently stores `query: String`. The whole point
of the query AST is to keep structured intent separate from SQL text — FTS
is the one gap.

New module `crates/fathomdb-query/src/text_query.rs`:

```rust
#[derive(Clone, Debug, PartialEq, Eq)]
pub enum TextQuery {
    Empty,
    Term(String),
    Phrase(String),
    Not(Box<TextQuery>),
    And(Vec<TextQuery>),
    Or(Vec<TextQuery>),
}
```

`QueryBuilder::text_search(raw, limit)` parses `raw` into `TextQuery`
immediately; the AST stores the typed form. `compile_query()` calls
`render_text_query_fts5(&query)` — the sole place that emits FTS5 syntax.
`sanitize_fts5_query()` in `compile.rs` is removed.

### Parser contract

Non-failing, normalizing. Recognized grammar: bare terms, quoted phrases,
uppercase `OR`, uppercase `NOT`, implicit `AND` by adjacency. Everything
else degrades to a literal term; malformed operators never pass through as
control syntax.

### Files touched

`crates/fathomdb-query/src/ast.rs`,
`crates/fathomdb-query/src/builder.rs`,
`crates/fathomdb-query/src/compile.rs`,
`crates/fathomdb-query/src/lib.rs`,
new `crates/fathomdb-query/src/text_query.rs`.

### Ship criteria

- All existing FTS sanitization tests converted to parser + renderer tests
  with equivalent coverage.
- Renderer is the only place that produces FTS5 `MATCH` strings.
- Python and TypeScript SDK harness baseline + vector scenarios still pass
  (no behavior change on simple single-term queries).
- No raw user text reaches SQLite unescaped.

## Item 2: Adaptive search response shape

**Design docs:** `dev/design-adaptive-text-search-surface.md`,
`dev/notes/adaptive-search-response-shape.md`,
`dev/notes/adaptive-search-winning-branch.md`

### What changes

`text_search()` currently returns a flat list of `SearchHit`. The new
return type is `SearchRows`:

```python
SearchRows(
    hits=[...],              # length <= limit
    was_degraded=False,      # query had to be simplified to stay under FTS limits?
    fallback_used=True,      # relaxed branch ran?
    strict_hit_count=2,
    relaxed_hit_count=3,
)
```

Per-hit additions:

```python
SearchHit(
    ...existing fields...,
    source=SearchHitSource.Property,    # Chunk | Property | Vector
    match_mode=SearchMatchMode.Strict,  # Strict | Relaxed
    matched_path="$.payload.title",     # property hits only; None otherwise
)
```

### Winning branch rule

When same node matches via multiple branches:

1. `Strict` beats `Relaxed`.
2. Within same mode: higher normalized score wins.
3. Score tie: `Chunk > Property > Vector`.
4. Remaining tie: branch declaration order (determinism guarantee).

The winner's `source`, `match_mode`, `score`, `matched_path`, and
`snippet` all appear on the hit. Losing branch metadata is dropped.

### Engine changes

- Query execution path must track which branch each hit came from.
- Deduplication by `logical_id` is the engine's responsibility.
- `was_degraded` wire from existing safety guard in `compile.rs`.
- `fallback_used` = relaxed branch returned at least one hit.

### SDK bindings

Python: `SearchRows` dataclass, `SearchHitSource` + `SearchMatchMode`
enums. TypeScript: equivalent interfaces. Cross-language parity tests pin
winning-branch order.

### Ship criteria

- SDK harness `baseline` + `vector` scenario counts still pass.
- `was_degraded=False`, `fallback_used` correctly reflects runtime
  branch selection.
- `matched_path` non-null for property hits, null for chunk and vector.
- `source` and `match_mode` match per-hit expectations in parity tests.
- Winning-branch precedence is tested with a fixture where same node
  matches via chunk and property with different scores.

## Item 3: FTS Level 3

**Design doc:** `dev/notes/fts-level-three-2026-04-11.md`

### Problem

Object-typed values in node properties are currently skipped at write time
(`extract_property_fts` in `writer.rs`). A declared path like
`$.payload` that resolves to a nested object indexes nothing. Issue `#36`.

### Short-term deliverable (0.5.0)

Opt-in recursive extraction mode for declared object paths using SQLite
`json_tree()`:

- `FtsPropertyPathSpec::recursive("$.payload")` already declares a
  subtree root. In 0.5.0, when this path resolves to an object, extract
  all descendant scalar leaves in document order.
- The extracted leaves are concatenated into the `payload_all` FTS column
  (consistent with 0.4.2 Pack B column naming for recursive paths).
- No schema API change; behavior change is transparent to callers
  registering recursive paths.

This addresses the current gap without changing the query surface.

### Medium-term (post-0.5.0)

Introduce a path-preserving derived side table built from `json_tree()`:

```sql
CREATE TABLE node_property_terms (
    node_logical_id TEXT NOT NULL,
    object_path     TEXT NOT NULL,
    fullkey         TEXT NOT NULL,
    key_token       TEXT,
    value_text      TEXT,
    value_type      TEXT NOT NULL
);
```

Enables real key/value search and provides the structural information
needed for object-scoped result granularity. This is a separate design
item and does not block 0.5.0.

### What NOT to do

Do not serialize whole objects via `json.dumps()` as the indexing method.
Keys and values become indistinguishable, ranking cannot distinguish field
importance, and spurious matches increase.

### Ship criteria

- Existing nodes with `$.payload`-style recursive paths that resolve to
  objects now produce FTS hits where previously they produced none.
- Rebuild path (`register_fts_property_schema_with_entries`) triggers
  re-extraction for all existing nodes of the kind.
- SDK harness regression: no change in baseline or vector scenario counts.
- Test: register a kind with a recursive path on an object-typed field;
  `text_search` returns the node.

## Item 4: TypeScript embedding adapters

**Reference:** `dev/notes/roadmap-0.4.5.md` — TypeScript adapters section

### What ships

Four adapters implementing the TypeScript `QueryEmbedder` interface,
mirroring the Python 0.4.5 surface:

| Adapter | Target | Dimensions | Notes |
|---|---|---|---|
| `OpenAIEmbedder` | OpenAI embeddings API | model-dependent | Client-side TTL cache keyed on text; identity `openai/<model>/<dims>` |
| `JinaEmbedder` | `jina-embeddings-v2-base-en` | 768 | 8192-token context; ALiBi bias |
| `StellaEmbedder` | `stella_en_400M_v5` | 1024 | Matryoshka truncation |
| `SubprocessEmbedder` | caller-supplied binary | caller-supplied | Persistent worker via `child_process.spawn`; binary stdin/stdout (f32 LE); identity string is required constructor param |

All live in `typescript/packages/fathomdb/src/embedders/`.

### Contract

Same interface as `BgeSmallEmbedder`:

```typescript
interface QueryEmbedder {
  embed(texts: string[]): Promise<number[][]>;
  identity(): string;
}
```

`SubprocessEmbedder` is not a built-in — documented as advanced/bring-your-own.

### Ship criteria

- All four adapters implement `QueryEmbedder`.
- `SubprocessEmbedder` round-trips a known embedding through a shell
  script fixture in the SDK harness.
- `OpenAIEmbedder` unit-tested with mocked HTTP.
- Python and TypeScript adapter identities produce the same embedding
  for the same text (parity smoke test with a shared fixture binary).

## Item 5: `QueryEmbedder::max_tokens()`

**Reference:** `dev/notes/roadmap-0.4.5.md` — open implementation questions

### Problem

The engine chunks content at write time using a fixed token budget. The
budget is currently hardcoded to 512 (BGE-Small's context window). Long-
context embedders (Nomic: 8192, Jina: 8192) never benefit from their full
window.

### What changes

Rust trait extension:

```rust
pub trait QueryEmbedder: Send + Sync {
    fn embed(&self, texts: Vec<String>) -> Vec<Vec<f32>>;
    fn identity(&self) -> String;
    fn max_tokens(&self) -> usize;   // new
}
```

Known values:

| Embedder | `max_tokens()` |
|---|---|
| `BgeSmallEmbedder` | 512 |
| `NomicEmbedder` | 8192 |
| `JinaEmbedder` | 8192 |
| `OpenAIEmbedder` | model-dependent (default 8192) |

The write-time chunker reads `embedder.max_tokens()` to size chunks. Nodes
whose content fits inside the window are not split; nodes exceeding it are
chunked at token boundaries.

### Coordination

- `BgeSmallEmbedder` returns 512 — no behavior change for existing DBs.
- Python + TypeScript `QueryEmbedder` wrappers must propagate
  `max_tokens()` across the FFI boundary.
- This is a breaking trait change — all existing `QueryEmbedder` impls
  outside the crate must add the method. Audit before landing.

### Ship criteria

- Trait has `max_tokens()`.
- `BgeSmallEmbedder::max_tokens()` returns 512; existing tests pass.
- Chunker reads `max_tokens()` from the embedder passed to
  `regenerate_vector_embeddings`.
- Integration test: write a document exceeding 512 tokens with a
  `max_tokens=8192` test embedder; verify it is stored as one chunk, not
  split into two.
- Python and TypeScript surface `max_tokens` to the caller.

## Item 6: Per-kind vec tables

**Reference:** `dev/notes/roadmap-0.4.5.md` — non-goals section

### Problem

Vector storage uses a global sentinel `('*', 'vec')` in
`projection_profiles`. The vec index is one flat table regardless of kind.
Mixed-kind databases have no way to scope vector search to a specific kind
without post-filtering.

### What changes

Replace the `('*', 'vec')` global key with `(kind, 'vec')` per-kind keys.
Storage layout: one `vec_<sanitized_kind>` table per kind, using the same
kind-name sanitization rule as `fts_props_<kind>` (from 0.4.2 Pack A).

This mirrors the FTS per-kind table model shipped in 0.4.2 and enables:
- kind-scoped vector search without post-filter overhead
- per-kind embedder configuration (different kinds can use different
  embedders/dimensions)
- cleaner `VecProfile` observation per kind rather than one global row

### Migration

Breaking change. No migration from the global `('*', 'vec')` table.
Existing databases must call `regenerate_vector_embeddings` after upgrade
to populate per-kind vec tables. The `('*', 'vec')` sentinel is dropped;
databases opened on 0.5.0 without regeneration return empty vector results
until regeneration runs.

### Non-goals

- Cross-kind vector fusion in a single `text_search()` call. Callers
  issue per-kind calls; the engine does not blend across kinds.
- Automatic embedder selection per kind. Still caller-provided; per-kind
  config just records which embedder was used.

### Ship criteria

- Per-kind vec tables created at schema migration time for existing kinds.
- `regenerate_vector_embeddings(embedder, config)` writes to the per-kind
  table for the kind passed in config.
- Vector search in `coordinator.rs` targets the per-kind table, not the
  global table.
- Migration: existing vec embeddings appear in per-kind tables after upgrade.
- `VecProfile` read/write uses `(kind, 'vec')` key.

## Item 7: Builtin embedder write-time path

**Issue:** #39

### Problem

Phase 12.5 wired `BuiltinBgeSmallEmbedder` for read-time vector queries
(zero external config). Write-time regeneration still requires a subprocess
binary producing compatible vectors. Callers who want batteries-included
behavior end up building and shipping a subprocess that duplicates the
in-process model — defeating the feature's purpose. Read/write vector drift
is silent if the subprocess doesn't exactly match `BAAI/bge-small-en-v1.5`
with CLS + L2 normalization.

### What changes

New `regenerate_vector_embeddings_in_process(embedder: &dyn BatchEmbedder, ...)` 
alongside the existing subprocess path. `BuiltinBgeSmallEmbedder` gains
`batch_embed(&[String])` running the BERT forward pass over a batch in one
tokenizer call. Admin surface (Rust, Python, TypeScript) gains a variant to
choose in-process vs subprocess. Contract persistence reuses the existing
`model_identity` / `model_version` / `dimension` / `normalization_policy`
row shape — no schema change.

Subprocess path (`generator_command`) remains intact for BYO callers.

### Ship criteria

- `regenerate_vector_embeddings_in_process` exists and accepts a
  `BatchEmbedder`.
- `BuiltinBgeSmallEmbedder` implements `BatchEmbedder`.
- Feature-gated integration test: register schema → write nodes → in-process
  regeneration → contract row matches embedder identity → emitted vectors
  cosine-match query-time vectors for same text.
- Subprocess path regression: existing subprocess tests unaffected.
- Python and TypeScript admin surface expose the in-process option.

## Sequencing

Items 1 and 2 are tightly coupled: `TextQuery` is the strict-grammar layer
that the adaptive retrieval policy (item 2) builds on. Item 1 must land
first or in the same release.

Items 3, 4, 5, 6 are independent of each other and of items 1–2.

Item 5 (`max_tokens`) is a prerequisite for long-context adapters in item 4
to be useful — adapters can ship without it but chunking won't respect their
context window until item 5 lands.

Item 6 (per-kind vec) is a storage migration with no public API dependency
on the other items.

Item 7 (builtin write-time) is independent. Natural follow-on to Phase 12.5.

Recommended order within 0.5.0:

1. Item 1 (`TextQuery` AST) — unblocks item 2
2. Items 4 + 5 + 7 in parallel (TS adapters, `max_tokens`, builtin write path)
3. Item 2 (adaptive response shape) — builds on item 1
4. Items 3 + 6 in parallel (FTS Level 3, per-kind vec)

## What 0.5.0 does not touch

- **FTS fallback for degraded vector reads** (#6) — moved to post-0.5.0
  roadmap; coordinate with adaptive response shape (`was_degraded` wire)
  before implementing.
- Write-priority / foreground-read isolation (item 11) — deferred pending
  Memex scheduler-burst instrumentation data.
- `fathomdb admin configure-vec` (Python admin CLI for vec profile) —
  already shipped in 0.4.5; no new surface needed.
- ICU tokenizer (`global-cjk` preset) — blocked on confirming ICU
  extension availability on all CI targets (manylinux, macOS, Windows).
  Falls back to clear `configure_fts` error if ICU absent.
- Advanced query surface (union/branch composition, staged search/traverse/
  hydrate, rich snippets) — see #38; design decision needed before any
  implementation.

## Path to 1.0.0

Items below are not 0.5.0. They are tracked here because they must be
resolved before a 1.0.0 claim is supportable.

### Mandatory blockers

**Release pipeline end-to-end (#12)**
`publish-rust` and `publish-pypi` have never completed; crates.io token and
PyPI trusted publisher not configured. No artifact has been published. Until
one release completes end-to-end, the pipeline is unproven.

**Enforce benchmark thresholds in CI (#14)**
Thresholds are documented in `dev/production-acceptance-bar.md` but not
enforced. Performance regressions are only caught by manual review.

**Publish operator binaries (#11)**
`fathom-integrity` and `fathomdb-admin-bridge` binaries must ship as
first-class release artifacts (GitHub release + documented download). Currently
only wheel + npm are published.

### Hardening

**Harden vector regeneration recovery path (#9)**
Persisted regeneration contract, external generator execution, and post-
recovery vector correctness all have a different risk profile than the rest of
projection rebuild. Remaining work: aggressive contract validation, security
model review for external generator execution, failure handling UX (partial/
slow/unavailable), upgrade/migration behavior for persisted contracts,
adversarial tests (malformed config, generator misbehavior, partial failure).

**Expand automated repair and recovery coverage (#15)**
Grow repair/recovery test coverage beyond the v0.1 contract: crash-recovery
simulation (mid-write kill → reopen → check → verify clean), end-to-end
`safe_export` roundtrip (seed → export → open exported → verify clean),
migration-history reset before bridge bootstrap (#7), bridge unsupported-
command response (#8).

### API quality

**Resolve ~420 missing-docs warnings (#34)**
`missing_docs = "warn"` is set workspace-wide; 420 warnings across
`fathomdb-engine` (371), `fathomdb-schema` (21), `fathomdb-query` (17),
`fathomdb` (10). Before 1.0.0, every public crate needs: `//!` module-level
doc, documented public structs/enums/functions, per-field docs on serde DTOs.
After the backlog clears, escalate `missing_docs` to `"deny"` per crate.

**Async napi-rs bindings for TypeScript (#35)**
Current sync napi calls block the Node.js event loop; TS stress tests are
sequential only. Async `#[napi(async)]` bindings would enable `Promise.all`-
based concurrency, concurrent load testing, and better SDK ergonomics (non-
blocking operations). Larger change; requires careful napi-rs async boundary
design.

### Deferred from 0.5.0

**FTS fallback for degraded vector reads (#6)**
`ExecutionCoordinator::execute_compiled_read()` returns empty rows +
`was_degraded=true` when `sqlite-vec` is absent. Conflates "no fallback
possible" with "fallback ran but returned nothing." Fix: when compiled query
contains a text search branch or structural filters, execute the non-vector
path and return those results with `was_degraded=true`. Deferred to coordinate
with item 2's adaptive response shape and `fallback_source` signal design.

### Design decisions

**Commit to Choice C for vector recovery (#10)**
Current v0.1 contract: embeddings are derived/regenerable, not canonical.
`safe_export` exports canonical data only; restore requires running
regeneration. This is by design but not formally committed to as the permanent
contract. Before 1.0.0: explicitly document Choice C as the permanent product
contract, or decide to support optional vec-row preservation in `safe_export`
(Choice B variant). Code is not the blocker — the decision is. Once decided,
update `dev/arch-decision-vector-embedding-recovery.md` and close the issue.

**Advanced query surface scope (#38)**
Issue documents four candidates — query union/branch composition, staged
search/traverse/hydrate, fielded/path-aware property FTS, rich snippets/
explanations — and asks which (if any) belong before 1.0.0. Must be resolved
explicitly: either reject each candidate with documented rationale, or adopt
one as a real 0.6.0+ design track. Accreting piecemeal is the failure mode.
The decision affects whether `fathomdb` stays a single-driver query compiler
with packaged search helpers, or grows into an explicit multi-branch retrieval
language.

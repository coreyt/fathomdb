# 0.1 Implementation Plan

## Summary

Adopt a **single monorepo** with a **Rust workspace at the repo root** for the embedded engine and a **sibling Go module** at `go/fathom-integrity` for integrity/admin tooling. Keep the Rust side lean: one public facade crate plus three internal crates. Keep the Go side thin: it should orchestrate and expose admin workflows, not reimplement SQLite-coupled engine logic.

The v1 boundary is:

- Rust owns the engine: schema bootstrap/migrations, AST and compiler, execution coordinator, writer actor, required projection writes, projection rebuild logic, and admin primitives.
- Go owns the surrounding admin UX: CLI commands, offline safety checks, snapshot/export workflows, and operator-facing repair flows that call into the Rust admin surface when mutation requires engine semantics.
- CI covers Rust, Go, and Python validation. Release publishing uses separate workflows so baseline CI stays focused on correctness checks.

Use this top-level shape:

Rust workspace members:

- `crates/fathomdb`
- `crates/fathomdb-schema`
- `crates/fathomdb-query`
- `crates/fathomdb-engine`

Go module:

- `go/fathom-integrity`

Do not add more Rust crates in v1. Keep network gateway and sync services out of the initial scaffold. Python packaging is in scope and ships from this monorepo.

## Rust Workspace

### Crate layout

`crates/fathomdb`

- Public facade crate.
- Exposes the stable Rust API used by the Python bindings and future TypeScript bindings.
- Owns top-level types such as `Engine`, `EngineOptions`, `Session`, `Query`, `WriteRequest`, and `AdminHandle`.
- Re-exports builder types from `fathomdb-query` and engine/admin entrypoints from `fathomdb-engine`.
- Contains black-box integration tests under `crates/fathomdb/tests/`.

`crates/fathomdb-schema`

- Owns schema bootstrap, migration manifests, metadata tables, pragma initialization, and startup checks for required SQLite capabilities.
- Exposes `SchemaManager`, `Migration`, `SchemaVersion`, and `BootstrapReport`.
- Is the only crate allowed to define or mutate the canonical SQL schema.
- Does not own query planning, write orchestration, or admin CLI behavior.

`crates/fathomdb-query`

- Owns the fluent query AST and compiler.
- Split internally into `builder`, `ast`, `compile`, and `plan` modules.
- Exposes `QueryAst`, `CompiledQuery`, `BindValue`, `ShapeHash`, and `CompileError`.
- Compiler output must be pure and DB-agnostic: SQL text, bind list, shape hash, expected driving table, and execution hints.
- Does not open SQLite connections or execute SQL.

`crates/fathomdb-engine`

- Owns SQLite-coupled runtime behavior.
- Split internally into `coordinator`, `writer`, `projection`, `admin`, `runtime`, and `sqlite` modules.
- Exposes `ExecutionCoordinator`, `WriterActor`, `ProjectionService`, `AdminService`, and `EngineError`.
- Contains the single admin bridge binary target used by the Go CLI for mutation-capable repair commands.

### v1 engine boundaries

Schema management:

- Owned entirely by `fathomdb-schema`.
- Responsibilities: create `nodes`, `edges`, `chunks`, `fts_nodes`, active `vec_nodes_*`, `runs`, `steps`, `actions`, metadata tables, indexes, and startup pragmas.
- Includes migration application, extension/capability checks, and schema-version reporting.
- Does not perform chunking, embeddings, or rebuild operations.

AST and compiler:

- Owned entirely by `fathomdb-query`.
- Responsibilities: builder DSL, AST normalization, shape hashing, SQL generation, bind generation, and planner rules.
- The compiler must know the `vec_nodes -> chunks -> nodes` join path, inside-out planning, selective deterministic-filter override, and bounded recursive CTE rules.
- Structural integers that affect SQL shape stay in the compiled SQL and shape hash; user data stays parameterized.

Execution coordinator:

- Owned by `fathomdb-engine::coordinator`.
- Responsibilities: open DB, initialize schema manager, manage WAL-mode reader connections, maintain prepared statement cache keyed by AST shape hash, dispatch compiled reads, and expose admin/write handles.
- Use `rusqlite`, not Diesel or SQLx.
- Keep the core synchronous in v1; do not introduce Tokio into the embedded engine core.

Writer actor:

- Owned by `fathomdb-engine::writer`.
- Responsibilities: single dedicated writer thread, in-memory write channel, `BEGIN IMMEDIATE` discipline, atomic canonical plus required projection commits, and handoff of optional backfill tasks after commit.
- Input type is a fully prepared write envelope: parsed/chunked data and any synchronous embeddings must already be computed before the actor takes the write lock.
- This is the only runtime component allowed to mutate canonical tables or required projections.

Projection rebuild/admin surface:

- Owned by `fathomdb-engine::admin` and `fathomdb-engine::projection`.
- Expose `check_integrity`, `rebuild_projections`, `rebuild_missing_projections`, `trace_source`, `excise_source`, and `safe_export`.
- Read-only admin operations may query SQLite directly; mutation-capable repair operations must reuse engine logic rather than duplicate it in Go.
- The admin bridge binary should expose a local JSON-over-stdio protocol for the Go CLI.

### Rust package management and project setup

- Use a root Cargo workspace with centralized `[workspace.dependencies]`.
- Pin a stable Rust toolchain with `rust-toolchain.toml`.
- Use edition `2024`.
- Standardize on `cargo` for build/test, `rustfmt` for formatting, `clippy` for linting, and `cargo-nextest` as the CI test runner.
- Prefer `rusqlite` plus direct SQLite integration over ORM layers.
- Add `rstest` for parameterized unit tests and `insta` for compiler SQL snapshot tests.

### Rust test strategy

Unit tests:

- `fathomdb-schema`: migration ordering, index creation, pragma bootstrap, schema version checks.
- `fathomdb-query`: AST normalization, shape hashing, SQL generation, planner branch selection, recursive CTE bounding.
- `fathomdb-engine`: prepared statement cache behavior, writer serialization, required-vs-optional projection routing, admin command validation.

E2E/integration tests:

- Put black-box engine tests in `crates/fathomdb/tests/`.
- Use real temporary SQLite databases with the actual schema, FTS, and active vector table enabled.
- Cover open/bootstrap, write->query roundtrip, chunk->vector->node resolution, selective deterministic filter planning, required projection atomicity, projection rebuild, and source-ref excision.
- Run via `cargo nextest run -p fathomdb`.

## Go `fathom-integrity`

### Basic feature list

Initial commands:

- `check`: run SQLite integrity checks plus engine-level projection presence checks.
- `export`: create a safe offline snapshot/export of the database file and metadata.
- `trace --source-ref`: show lineage for a source/action/run and the affected logical rows.
- `rebuild --target {fts,vec,all}`: request deterministic projection rebuild.
- `rebuild-missing`: request repair of missing optional projections.
- `excise --source-ref`: surgically remove outputs from a bad source lineage.
- `version`: show CLI version and compatible engine/admin protocol version.

Do not add sync, replication, or remote API behavior to `fathom-integrity` in v1.

### Basic architecture

Use this Go layout:

```text
go/fathom-integrity/
  go.mod
  cmd/fathom-integrity/main.go
  internal/cli/
  internal/config/
  internal/sqlitecheck/
  internal/bridge/
  internal/commands/
  test/e2e/
  test/testdata/
```

Module responsibilities:

- `cmd/fathom-integrity`: Cobra-less thin `main`, unless argument parsing becomes painful; prefer stdlib `flag` initially.
- `internal/cli`: command routing, output formatting, exit codes.
- `internal/config`: DB path discovery, environment/config loading.
- `internal/sqlitecheck`: read-only checks that can be run directly against the SQLite file.
- `internal/bridge`: JSON-over-stdio client for the Rust admin bridge binary.
- `internal/commands`: one package per command family.

Command ownership:

- `check` remains Go-orchestrated, but uses the Rust admin bridge for Layer 2 when `--bridge` is provided.
- `export`, `trace`, `rebuild`, `rebuild-missing`, and `excise` go through the Rust admin bridge.
- `recover` remains Go-orchestrated and may call the bridge for schema bootstrap and Layer 2 checks.

### Go project setup

- Use a single Go module with standard Go modules; do not add `go.work`.
- Pin the Go toolchain in `go.mod` to the current stable version at implementation time.
- Use the standard `testing` package for unit tests.
- Use `testify/require` for assertions only; avoid heavier test frameworks.
- Use `rogpeppe/go-internal/testscript` for CLI e2e tests because the primary surface is a command-line tool.
- Keep fixtures under `test/testdata/`.

### Go test strategy

Unit tests:

- argument parsing
- config/path resolution
- SQLite integrity check wrappers
- bridge request/response handling
- exit code mapping

E2E tests:

- `check` against healthy and intentionally damaged fixture DBs
- `export` snapshot creation
- `trace --source-ref` against fixture lineage
- bridge-backed `rebuild` and `excise` happy paths using a temporary DB and a stub or real local Rust admin bridge, depending on implementation stage

Run Go tests with:

- `go test ./...` for unit coverage
- `go test ./test/e2e/...` for e2e/testscript coverage

## CI/CD

Create one baseline GitHub Actions workflow at `.github/workflows/ci.yml` that runs on pull requests and pushes to `main`.

Required jobs:

- `rust-lint`: `cargo fmt --check` and `cargo clippy --workspace --all-targets -- -D warnings`
- `rust-test`: `cargo nextest run --workspace`
- `rust-build`: `cargo build --workspace`
- `go-lint`: `gofmt -l .`, `go vet ./...`
- `go-test`: `go test ./...`
- `go-e2e`: `go test ./test/e2e/...`
- `smoke-readme-docs`: optional markdown/link sanity if added cheaply; otherwise skip in v1

CI defaults:

- Required platform: `ubuntu-latest`
- Cache Rust and Go dependencies
- No release publishing, package registry upload, container publish, or semantic-release automation in v1

CD boundary:

- Tagged releases remain manual for now.
- When the engine stabilizes, add separate release workflows rather than overloading the baseline CI file.

## Assumptions And Defaults

- This remains one GitHub-hosted monorepo.
- The embedded engine stays synchronous internally in v1.
- `rusqlite` is the Rust SQLite layer.
- The Go CLI is an operator/admin tool, not a second engine implementation.
- Repair semantics live in Rust; Go may orchestrate them but should not duplicate them.
- Python bindings are implemented in-repo today. TypeScript bindings remain a future surface that should bind to the same Rust facade.

# 0.5.2 Scope

## Theme

Hotfix release. Single confirmed item: restore `admin.check_semantics()`
for databases that have registered a weighted (per-column) FTS property
schema. Everything else from the original 0.5.2 scope (scale.rs race,
TS `configureFts` parity, FTS5 metachar escape, TS feedback fast-path,
Node.js 20 upgrade, click helper audit) moved forward to 0.5.3 so this
release can ship quickly on top of 0.5.1.

## Status (2026-04-18)

Scope cut as a single-issue hotfix after the Memex team reported a
`SqliteError: no such column: fp.text_content` regression on m007
rebuild against fathomdb 0.5.1.

## Items

### 1. `check_semantics()` fails on weighted FTS property schemas

**Status**: Confirmed (user-reported regression, Memex m007)
**Breaking**: No (diagnostic-only bug fix)
**Design**: `dev/notes/design-0.5.2-check-semantics-weighted-fts.md`

`Admin::check_semantics()` counts "drifted" property-FTS rows by
reading a hardcoded `fp.text_content` column from every per-kind FTS
table (`crates/fathomdb-engine/src/admin/mod.rs:1082`). The column
only exists on **non-weighted** per-kind tables. When a schema is
registered with one or more `FtsPropertyPathSpec` entries (via
`register_fts_property_schema_with_entries`,
`register_fts_property_schema_weighted`, or the equivalent path that
goes through `create_or_replace_fts_kind_table` at
`crates/fathomdb-engine/src/admin/fts.rs:602-614`), the per-kind FTS
table is built with one column per path and **no `text_content`
column**. Any later call to `check_semantics()` panics with:

```
no such column: fp.text_content in SELECT fp.node_logical_id,
fp.text_content, n.properties FROM fts_props_<kind> fp ...
```

The query renderer at `coordinator.rs:1941-1948` already handles both
shapes correctly (`is_weighted` branch uses `'' AS snippet`); only the
semantics-diagnostic path was missed when weighted schemas landed.

**Impact surface:**

- Write / rebuild / search paths: unaffected.
- Ranking (weighted BM25) on registered kinds: unaffected (verified
  by Memex downstream via `engine.nodes(kind).search(...)`).
- Only `Admin::check_semantics()` and anything that transitively calls
  it (tests, health checks, doctor commands) fail against a DB with
  any weighted FTS kind registered.

**Work:**

- Detect the per-kind FTS table shape inside
  `count_drifted_property_fts_rows`:
  - Non-weighted (`text_content` column present) → existing bulk
    comparison path is correct.
  - Weighted (no `text_content` column) → reconstruct the expected
    per-column text from the current schema + node properties and
    compare row-by-row. If reconstruction is not tractable in the
    hotfix window, fall back to returning `0` drift with a TODO
    pointing at 0.6.x follow-up; this matches the coordinator's
    snippet compromise and is still correct (drift counts only go
    up from 0, never produce false positives).
- Confirm the fix works for both existing call sites
  (`Admin::check_semantics()` from Rust, Python bridge, TypeScript
  admin bridge, Go integrity binary).
- Add regression tests:
  - Rust: new test at `crates/fathomdb-engine/src/admin/mod.rs`
    beside the existing `check_semantics_detects_drifted_property_fts_text`
    (line 4197) covering a kind registered via
    `register_fts_property_schema_with_entries` with specs.
  - Rust integration smoke: ensure `check_semantics()` returns
    `Ok(...)` (not `SqliteError`) on a DB with a weighted kind and
    at least one node written.
  - Cross-SDK: add a short Python test under
    `python/tests/` that mirrors the Memex usage pattern
    (register weighted FTS schema → open new connection →
    `admin.check_semantics()` does not raise).

**Acceptance:**

- `cargo nextest run -p fathomdb-engine admin` green, including the
  new weighted-shape tests.
- `uv run pytest python/tests/test_admin.py` (or the weighted-FTS
  parity test suite) green.
- The Memex m007 reproduction recipe (register weighted FTS schemas
  for two kinds, open, call `check_semantics()`) succeeds.

**Reference**: Memex bug report 2026-04-18 (v040-item2 worktree),
`tests/test_fathom_store.py::TestFathomStoreLifecycle::test_check_semantics_clean`.

## What 0.5.2 does not touch

- Anything not needed to restore `check_semantics()` against weighted
  FTS schemas.
- No AST, schema, write path, search path, or tokenizer changes.
- No SDK-surface changes.
- No new features. No breaking changes.

## Release mechanics

Same 0.5.x release flow; intentionally minimal because scope is one
item:

- CHANGELOG.md entry under Fixed: "Admin::check_semantics() no longer
  raises `SqliteError: no such column: fp.text_content` when a kind
  has been registered with a weighted FTS property schema (per-column
  per-kind FTS table)."
- Version bump across all four manifests + Cargo.lock (root
  `Cargo.toml`, `crates/fathomdb/Cargo.toml`, `python/pyproject.toml`,
  `typescript/packages/fathomdb/package.json`).
- Run `scripts/preflight.sh --release` — all three release gates
  must pass (clippy tracing/python, nextest tracing).
- Run `scripts/preflight-CI.sh` for the full CI-equivalent set before
  tagging.
- Tag `v0.5.2` on HEAD; release workflow gates on `all-builds-passed`.

## 1. Purpose

This PSD records the trade space behind the FathomDB 0.6.0 rewrite as a
single Systems-Engineering artifact, so that the locked architecture is
defensible against the alternatives the team actually considered. It is
authored **retrospectively**: the corpus already contains 35 accepted
ADRs, 12 locked design docs, full NEED → REQ → AC → Test traceability,
and a Phase-4 freeze gate. The PSD does not re-litigate any of those
decisions; it synthesises them so an outside reviewer can read one
document and see the chain from user need to selected option.

**Deviation from `dev/templates/psd-template.md`.** The template assumes
2–4 monolithic candidate solutions and one trade matrix. The 0.6.0
architecture is naturally orthogonal across six axes (storage substrate,
concurrency, lock mechanism, op-store + transaction shape, projection
model, operator surface + embedder ownership). This PSD therefore uses
**six axis-scoped option sets and six matrices** in §5 / §6 rather than
one. §7 then presents the union of axis decisions as the recommended
system architecture.

## 2. Scope

In scope:

- The 0.6.0 engine runtime: open / close, single-writer thread, reader
  pool, op-store, scheduler, vector + FTS projections, retrieval pipeline,
  embedder dispatch, lifecycle observability.
- Operator surface: two-root `fathomdb` CLI (`recover`, `doctor`),
  recovery from corruption.
- Public SDK shape across Python, TypeScript, and Rust facades.
- Release / supply-chain posture: sibling crate co-tagging, atomic
  multi-registry publish, registry-installed wheel as the release gate.

Out of scope (explicit non-goals; carry forward to a future PSD if
revisited):

- Networked server / wire-protocol endpoint (see REQ-032; carries to
  0.8.0+ subprocess bridge — see ADR-0.6.0-subprocess-bridge-deferral).
- Automatic cloud sync or multi-device replication.
- Implicit embedder/model download or lifecycle management by the engine
  (NEED-017, REQ-033).
- Runtime-API repair operations (REQ-037, REQ-054 — recovery is a
  CLI-only concern).
- Mobile platforms (NEED-016 deployment envelope).
- Data migration from 0.5.x (no upgrade path; fresh-DB only).
- MVCC / multi-writer concurrency (deferred to 0.7+ behind a forcing
  function — see ADR-0.6.0-single-writer-thread "Out of scope").

## 3. Inputs

### 3.1 User Needs (Desires / Wants / Needs)

Inlined verbatim from `dev/needs.md` (lines 35–171). Tagged per the
classification used in that file: customer needs are **Need** (Must),
functional + non-functional needs are **Want** (Should), and explicit
out-of-scope items in §"Out of Scope / Non-Needs" are **Desire** (Nice
— not in 0.6.0).

#### Customer Needs (all **Need / Must**)

- **NEED-001 — Trustworthy storage for hard-to-recreate canonical data.**
  Users need confidence that canonical data is stored durably and that
  the system fails loudly and diagnostically rather than silently
  corrupting or serving incorrect results. Primary canonical data types
  (HITL 2026-05-02; non-exhaustive examples): web pages / crawls;
  meeting transcripts; media metadata; e-mails and messages; "world
  models" (structured representations derived from canonical sources).
- **NEED-002 — Local-first privacy and offline operation.** Users need
  the core system to work without a network connection and without
  implicitly transmitting user data.
- **NEED-003 — Low-ceremony embedded deployment.** Users need a
  deployable artifact that does not require running a server or
  operating a separate database service. They want "a file path + a
  library" as the default experience.
- **NEED-004 — Operator-grade backup/export and worst-case recovery.**
  Users need a clear backup/export story and a recovery story that can
  be executed from operator tooling rather than bespoke application
  code.
- **NEED-004a — Backup workflows that do not impose long service
  interruptions.** Users need backup workflows that preserve read
  availability and minimise write interruption for practical database
  sizes. Voice-of-customer (HITL 2026-05-02; expectation, not a locked
  requirement): for a ~100 MB database, "no reads" longer than ~2
  minutes is not acceptable; "no writes" longer than ~5 minutes is not
  acceptable.
- **NEED-005 — Predictable licensing compatibility.** Users need
  permissive-license compatibility suitable for embedding into a broad
  range of applications (including commercial distribution), and do not
  want surprise licensing constraints introduced through core
  dependencies. HITL 2026-05-02 license posture (constraints, not
  requirements): acceptable (permissive) — MIT, Apache-2.0,
  BSD-2-Clause, BSD-3-Clause, ISC, Unlicense / CC0, PSF; use with
  caution (weak copyleft) — LGPL, MPL; not acceptable (restrictive /
  copyleft) — GPLv2 / GPLv3, AGPL, SSPL, CC BY-NC, CC BY-ND.
- **NEED-006 — Release quality bar suitable for a database.** Users need
  conservative releases with strict CI and a strong bias toward "it
  keeps working" rather than relying on frequent micro-releases to
  patch basic reliability issues.

#### Functional Needs (all **Want / Should**)

- **NEED-010 — Small, learnable SDK surface.** Users need a small,
  learnable application runtime API surface that is easy to learn and
  hard to misuse, with consistent behaviour across supported languages.
- **NEED-011 — Clear boundary between runtime SDK and operator
  tooling.** Users need a hard boundary between the application runtime
  API and operator recovery / inspection workflows, so application code
  cannot accidentally invoke lossy repair operations.
- **NEED-011a — A dedicated operator surface for inspection / export /
  recovery.** Users need a dedicated operator surface that supports
  inspection, export, and recovery workflows without requiring bespoke
  application code changes.
- **NEED-012 — Hybrid retrieval over canonical data.** Users need to
  write canonical entities / relationships and retrieve them via text
  search, similarity search, and hybrid paths without maintaining
  separate storage systems.
- **NEED-013 — Graph-oriented model compatibility.** Users need a model
  that supports graph-oriented application / agent workflows (at minimum
  node / edge style relationships and queryable traversals) without
  forcing a separate graph database.
- **NEED-014 — Explicit "catch-up" barrier for derived work.** Users
  need a way to determine when derived indexing / projection work has
  caught up to canonical writes for workflows like tests and batch
  ingest.
- **NEED-015 — Low-burden upgrades with explicit reporting.** Users need
  upgrades to be low-operator-burden and to provide an explicit,
  machine-readable report of what changed (or why the upgrade failed).
- **NEED-016 — Cross-platform support for common developer
  environments.** Users need the project to run on the major desktop /
  server platforms they develop and ship on, without OS-specific
  footguns. Deployment envelope (HITL 2026-05-02): desktop / server in
  scope; mobile nice-to-have, not a requirement for 0.6.0.
- **NEED-017 — No implicit model download or network dependency.** Users
  need embedder / model lifecycle to remain caller-owned; the system
  should not surprise them by downloading model weights or requiring
  external services.

#### Non-Functional Needs (all **Want / Should**)

- **NEED-020 — Reliability: bounded shutdown and no deadlocks.** Users
  need engine lifecycle behaviour that does not hang processes on exit
  and does not deadlock under expected concurrency patterns.
- **NEED-020a — Multi-process concurrency without unacceptable query
  latency.** Users need to support concurrent application querying while
  a single writer is active, specifically to keep query response latency
  low. Deployment constraint (HITL 2026-05-02): this must work on Linux
  (the three most-used local filesystems), macOS, and Windows (NTFS).
- **NEED-021 — Safety: corruption detection with actionable
  diagnostics.** Users need corruption and incompatibility to be
  detected early with structured errors that make next steps obvious
  (including when the correct next step is an operator recovery /
  inspection workflow).
- **NEED-022 — Performance that supports interactive agent / application
  use.** Users need search and write performance that is "fast enough"
  for interactive use without external services, with explicit perf
  gates defined elsewhere (requirements / ADRs / acceptance).
- **NEED-023 — Usability for non-DB experts.** Users need the system to
  be operable with minimal database expertise: clear errors, stable
  diagnostics, and documented workflows for the scary cases.
- **NEED-024 — Operational transparency.** Users need enough
  observability (lifecycle events, counters, profiling) to understand
  "what is slow" and "what is broken" without guessing.
- **NEED-025 — Minimal migrations and bounded upgrade complexity.**
  Users need upgrade posture that avoids long chains of accumulated
  migrations and avoid compatibility shims that hide breaking changes.

#### Explicit Non-Needs (all **Desire / Nice — out of scope for 0.6.0**)

- Running as a networked server or hosting a wire protocol endpoint.
- Automatic cloud sync or multi-device replication.
- Implicit embedder / model downloads, hosting, or lifecycle management
  by the engine.
- Runtime application API recovery / repair operations (repair is an
  operator concern, not an application concern).

### 3.2 Constraints and Limitations (Technical Input)

**Hard constraints — verbatim from `dev/requirements.md`.** A trade-off
cannot bend these.

- **REQ-020a (Hard).** Clean engine close releases all OS resources.
  `Engine.close()` releases the database file lock and all FDs the
  engine opened.
- **REQ-020b (Hard).** Host process exits ≤ 5 s of close. Host process
  exits within 5 seconds of `Engine.close()` returning.
- **REQ-021 (Hard).** Bounded process exit ≤ 5 s without explicit close.
  Host process exits within 5 seconds even when engine instances are
  not explicitly `close()`d.
- **REQ-022a (Hard).** `DatabaseLocked` rejection on second open. Second
  engine on the same DB file is rejected with a typed `DatabaseLocked`
  error while the first holds it, including while pending vector work
  exists.
- **REQ-022b (Hard).** Second-open never corrupts state. A rejected
  second open leaves the database file unmodified.
- **REQ-023 (Hard).** No deadlock on engine drop with pending vector
  work.
- **REQ-024 (Hard).** `fathomdb doctor safe-export` covers committed
  WAL-backed state. Never regresses to file-copy semantics.
- **REQ-025a (Hard).** Recovered databases preserve canonical rows.
- **REQ-025b (Hard).** Recovery restores FTS usability.
- **REQ-025c (Hard).** Recovery preserves vector profile metadata +
  table capability (for vector-enabled DBs).
- **REQ-027 (Hard).** Canonical writes never blocked by projection
  unavailability (FTS or vector).
- **REQ-028a (Hard).** No-embedder-wired hard-errors at call boundary.
  Calling a vector-requiring operation with no embedder configured
  hard-errors at the call boundary; never silent-degrades.
- **REQ-028b (Hard).** Kind-not-vector-indexed hard-errors at call
  boundary.
- **REQ-028c (Hard).** Embedder dimension mismatch hard-errors at call
  boundary.
- **REQ-031b (Hard).** Zero corruption on power-cut. Up to 100 ms of
  final-commit loss on power-cut acceptable; zero commit loss on
  OS-crash.
- **REQ-031c (Hard).** Recovery time ≤ 2 s for a 1 GB DB at
  `Engine.open` after unclean shutdown. Measured from process start to
  first accepted write transaction.
- **REQ-031d (Hard).** Refuse-to-open on detected corruption.
  `Engine.open` fails closed when corruption is detected at any
  open-path stage. Failure surfaces as a structured
  `EngineOpenError::Corruption` carrying kind / stage / locator /
  `RecoveryHint { code, doc_anchor }`. The engine MUST NOT
  auto-truncate, auto-rebuild, auto-replay-with-skip, or auto-degrade
  to read-only. Recovery is reachable exclusively via the separate
  `fathomdb recover` CLI tool.
- **REQ-032 (Hard).** No network listener, no wire protocol. All access
  is in-process or local subprocess; no TLS / auth / authz surface to
  misconfigure.
- **REQ-033 (Hard).** No implicit network fetch on `Engine.open`. Engine
  never downloads or hosts embedder model weights; embedder is supplied
  by the caller.
- **REQ-034 (Hard).** FTS5 injection-safe text queries. Agent /
  LLM-generated text queries cannot inject FTS5 control syntax; safe
  grammar tokenises at parse time and never passes raw input to FTS5.
- **REQ-036 (Hard).** Two-root operator CLI: `recover` (lossy) +
  `doctor` (bit-preserving). `--accept-data-loss` flag is mandatory on
  `recover`; no default. `--json` is mandatory on every verb.
- **REQ-037 (Hard).** Recovery tooling unreachable from runtime SDK.
  Application callers cannot accidentally invoke `excise_source`,
  `purge_logical_id`, or `safe_export`.
- **REQ-041 (Hard).** Single-file deploy. Operator deploys one binary +
  one `.sqlite` path; no server, no network dependency. Sidecar
  artifacts auto-managed by the engine (`-wal`, `.lock`) live at the
  same path with documented suffixes; they are part of the database
  file set, not separate operator-managed inputs.
- **REQ-053 (Hard).** Five-verb application runtime SDK surface. Public
  application runtime SDK surface is `Engine.open`, `admin.configure`,
  `write`, `search`, `close` — five verbs, no more.
- **REQ-054 (Hard).** Recovery / repair verbs are CLI-only.
  `rebuild_projections`, `restore_*`, `check_integrity`, `safe_export`,
  etc. are reachable only via the CLI surface, not via the runtime SDK.

**Soft constraints — SLI gates (`dev/requirements.md` REQ-009..REQ-018).**

### 3.3 Assumptions

Each assumption is a future risk if it turns out false; impact noted.

1. **SQLite is a durable, well-supported substrate for canonical data.**
   We assume WAL + `synchronous=NORMAL` + `fsync` discipline gives the
   durability profile claimed in REQ-031b/c. _Impact if false:_ entire
   storage substrate axis re-opens.
2. **`sqlite-vec` is a viable in-tree ANN extension on Tier-1
   platforms.** Per ADR-0.6.0-sqlite-vec-acceptance, no fallback ANN
   ships in 0.6.0. _Impact if false:_ vector index location and
   retrieval pipeline both re-open; the rewrite ships without ANN or
   reverts to an external store.
3. **Candle-rs + `tokenizers` is a viable in-process embedder.** Default
   embedder ships as a sibling crate per ADR-0.6.0-default-embedder.
   _Impact if false:_ default embedder reverts to caller-required, with
   significant DX regression.
4. **Single-process embedded deployment is the dominant workload.** All
   architecture axes assume one writer, one process, one DB file.
   Networked / multi-device modes are explicit non-goals.
   _Impact if false:_ MVCC / replication ADR forced and most ADRs in §5
   re-open.
5. **Tier-1 CI platforms are little-endian.** Per ADR-0.6.0-zerocopy-blob
   and ADR-0.6.0-tier1-ci-platforms. _Impact if false:_ zerocopy BLOB
   layout requires byte-swap fallback.
6. **`rusqlite` will continue to expose synchronous semantics
   compatible with sqlite-vec extension loading.** The async-surface
   choice (sync engine surface) load-bears on this. _Impact if false:_
   sqlx-style migration becomes attractive; rewrite sync invariants
   re-open.
7. **Rust 1.89's `File::try_lock` preserves per-OFD `flock(2)`-class
   exclusion semantics.** Per ADR-0.6.0-database-lock-mechanism § 1.
   _Impact if false:_ lock mechanism re-opens; same-process two-Engine
   exclusion reverts to relying on the SQLite EXCLUSIVE backstop alone.
8. **Operators tolerate a `.lock` sidecar file at the DB path.**
   REQ-041's "single-file deploy" wording is interpreted to include
   engine-managed sidecars. _Impact if false:_ REQ-041 wording re-opens.

## 5. Candidate Solutions (architectural axes)

Six axes; each axis has 2–3 real options drawn from accepted ADRs. Cross-
axis interactions are noted inline. Axis labels (5.1 .. 5.6) match the §6
matrix headings.

### 5.1 Storage substrate

**Option A — SQLite + WAL, embedded via `rusqlite` (chosen).** All
canonical, FTS, vector (`vec0`), op-store, and provenance data lives in
a single SQLite file. WAL + `synchronous=NORMAL` is the durability
profile.

**Pros.** Mature substrate; native FTS5 (C9); WAL + readers-don't-block-
writers semantics align with C4 + C8; one-file backup and recovery
(C3, C6); permissive license (C7); Tier-1 platform reach is trivial
(C8).

**Cons.** Single-writer at the file level (interacts with §5.2). File
size grows with vector count; one corruption blast radius across all
table classes (C1).

**Risks / unknowns.** sqlite-vec sole-maintainer risk
(ADR-0.6.0-sqlite-vec-acceptance accepts it explicitly). NFS / network
filesystem unreliability — not a 0.6.0 deployment target.

**Option B — Custom KV store (e.g. RocksDB or LMDB) + sidecar FTS +
sidecar ANN.** Three-component substrate.

**Pros.** Per-component tunability; potentially better write throughput.

**Cons.** Multi-file backup coordination breaks REQ-041 (C6); no
transactional consistency across substrate components (C1, C3); each
component carries its own license review (C7); FTS/ANN integration burden
moves into the engine.

**Risks.** RocksDB GPLv2 dual-license risk against C7 (acceptable list
excludes GPL).

**Option C — sqlx + sqlite (async-native).** SQLite via async runtime.

**Pros.** Native async surface for binding ergonomics (interacts with
§5.6).

**Cons.** sqlite-vec extension loading is non-trivial under sqlx (C9);
~5–8k LoC engine rewrite; reopens every Phase-1 ADR. Per
ADR-0.6.0-async-surface § "Why not Option C": reverses already-accepted
deps decisions.

### 5.2 Concurrency model

**Option A — Single dedicated writer thread + multi-connection reader
pool (chosen).** One OS thread owns the only writer `rusqlite`
connection; reader pool provides per-thread connections for concurrent
reads. Op-store, projection, vector writes all share the writer thread.

**Pros.** Matches `rusqlite`'s actual constraint; one transactional
fence simplifies op-store + projection + vector atomicity (C1, §5.4);
deadlock surface collapses to writer-vs-reader checkpoint windows only
(C4); single-writer + WAL is the widely-deployed SQLite production
posture.

**Cons.** Write throughput bounded by single-thread CPU + I/O. Acceptable
under REQ-009a/b (no write-throughput SLI was missed in benchmarks).

**Option B — Writer pool with `BEGIN IMMEDIATE` retry.** Multiple writer
threads compete; SQLite serialises at the file level; retry loop on
`SQLITE_BUSY`.

**Pros.** Theoretical write parallelism.

**Cons.** No actual parallelism gain — SQLite serialises writers at the
file level under WAL; the pool only changes which thread waits.
Empirical 0.5.x failure mode: each retry re-reads schema, amplifying the
`SQLITE_SCHEMA` window (Stop-doing entry in `learnings.md`).

**Option C — MVCC via sqlx + WAL2 / BEDROCK fork.**

**Pros.** Real concurrent writers.

**Cons.** Forks SQLite (C7, C8); loses sqlite-vec extension parity (C9
broken); 5–8k LoC engine rewrite; reopens every Phase-1 ADR. Explicit
0.6.0 non-goal per ADR-0.6.0-single-writer-thread "Out of scope".
Deferred to 0.7+ behind a forcing function (none exists in 0.6.0).

### 5.3 Database lock mechanism

**Option A — SQLite-native `PRAGMA locking_mode=EXCLUSIVE` only.** Rely
on SQLite's own lock state.

**Pros.** No sidecar artifact; conforms to a strict reading of REQ-041.

**Cons.** Per [pragma.html](https://www.sqlite.org/pragma.html#pragma_locking_mode):
EXCLUSIVE lock is acquired on the **first read**, not at
`sqlite3_open`. Engine.open pays migration + embedder warmup (~1–2 s)
**before** the lock surfaces — second-open fails late after wasting
work (C4 violation by latency, not by correctness). `SQLITE_BUSY` is
generic; cannot distinguish "another process holds DB" from "transient
busy" — typed `DatabaseLocked` error (REQ-022a) becomes hard to surface
cleanly.

**Option B — Sidecar `flock` only.**

**Pros.** Pre-open fail-fast.

**Cons.** BSD `flock` is per-open-file-description: two `File::open` +
`try_lock` calls in the **same process** can both succeed. Same-process
two-Engine (Python + TypeScript loaded into one operator process) is not
blocked. Rust 1.89 `File::try_lock` docs explicitly state same-process
double-lock semantics are "unspecified, possibly deadlock."

**Option C — Hybrid: sidecar `.lock` flock + `PRAGMA
locking_mode=EXCLUSIVE` on writer connection in WAL (chosen).** Both
layers required.

**Pros.** Cross-process exclusion via sidecar (load-bearing for that
case); same-process exclusion via SQLite EXCLUSIVE backstop; pre-open
fail-fast (saves the 1–2 s warmup cost on a doomed open); typed
`DatabaseLocked { holder_pid }` cleanly surfaces (REQ-022a). WAL +
EXCLUSIVE removes `-shm` (one fewer sidecar). Reader connections stay
NORMAL — preserves WAL parallel-reader design (REQ-018, C4).

**Cons.** Adds one on-disk artifact (`.lock`). REQ-041 interpretation
amended to include engine-managed sidecars at the DB path.

**Risks / unknowns.** Per-OFD `flock` semantics on exotic platforms;
SQLite EXCLUSIVE is the defense-in-depth backstop. NFS undefined; out of
scope.

(Three more options — direct file lock à la BoltDB, `O_EXCL | O_CREAT`
marker, process-global Rust registry — all rejected per ADR §"Options
considered" E/F/H. Summarised in §8.)

### 5.4 Op-store placement and write-tx shape

**Option A — Same SQLite file; batched `&[PreparedWrite]` single
writer-thread transaction (chosen).** `OpStoreInsert` is a
PreparedWrite variant; primary entity rows + op-store rows commit
atomically on the writer thread. Two collection kinds:
`append_only_log` (rows in `operational_mutations`) and `latest_state`
(rows in `operational_state`).

**Pros.** Single-file invariant preserved (C6); single backup target
(C3); transactional consistency between primary entities and op-store
rows (C1); one schema migration story (C11); no separate writer lane
needed.

**Cons.** Schema namespace must keep op-store tables distinct (handled
by `operational_*` prefix + CI lint).

**Option B — Separate `.opstore.sqlite` file alongside primary.**

**Pros.** Cleaner isolation.

**Cons.** Breaks REQ-041 (C6); doubles backup / restore surface (C3);
cross-file transactions are not real — consistency story weakens (C1);
`safe_export` becomes a manifest of two files.

**Option C — Drop op-store; clients persist application state
themselves.**

**Pros.** Smallest engine surface.

**Cons.** Every agentic client re-implements the same primitives
(connector health, scheduler cursors, queue state, intake_log
lifecycle) — recurring user-visible cost (C5 inverted: simpler engine,
more user code).

### 5.5 Projection model

**Option A — Push / eager (chosen).** Scheduler dispatches projection
jobs **post-commit** of the originating write (per
ADR-0.6.0-async-surface Invariant A). Bounded retry (3 attempts; 1 s /
4 s / 16 s backoff); on terminal failure, batch is recorded in a
`projection_failures` op-store collection and the cursor advances; no
durable projection-queue table.

**Pros.** Best read latency — vector available at query time without
compute (C9); freshness window made explicit by `projection_cursor`
(NEED-014, REQ-055); matches accepted async-surface and scheduler-shape
ADRs.

**Cons.** Pays embedder cost on 100 % of writes regardless of whether
the row is ever queried.

**Option B — Pull / lazy.** Projection computed on first read; cached.

**Pros.** Worst-case write latency lower; no scheduler complexity.

**Cons.** First-read tail latency is brutal under interactive workloads
(C9 violation under a typical agentic read mix); freshness becomes
per-read-trigger; harder to surface a freshness SLI cleanly (REQ-015).

**Option C — Access-pattern hybrid (auto-detect "frequent", lazy for
tail).**

**Pros.** Theoretically best of both.

**Cons.** Requires access-pattern tracking + a "what counts as frequent"
threshold — the speculative-knob class flagged on the Stop-doing list.
Rejected.

(A per-profile `lazy=true` declaration, "Option C′" in
ADR-0.6.0-projection-model, is **deferred** — not rejected — pending a
real consumer with cold long-tail profiles. Tracked outside this PSD.)

### 5.6 Operator surface and embedder ownership

This axis combines three sub-decisions that collectively define the
"surface area outside the runtime SDK." They are grouped because each
sub-decision is binary and they share the same C3 / C5 / C7 evaluation.

**Option A — Two-root operator CLI + caller-owned embedder lifecycle +
default embedder shipped as installable sibling crate (chosen).**

- CLI: `fathomdb recover --accept-data-loss <sub-flag>...` (lossy) and
  `fathomdb doctor <verb>` (read-only / bit-preserving). `--json`
  mandatory on every verb.
- Embedder lifecycle: caller-owned per NEED-017; engine never downloads
  weights (REQ-033).
- Default-embedder package: `fathomdb-embedder` sibling crate bundles
  candle + tokenizers + sqlite-vec; operator opts in via the package
  install. Shared trait crate `fathomdb-embedder-api` enables version-
  skew detection at resolution time (REQ-047).

**Pros.** Mutation semantics are visually distinct at the CLI root
(`recover` vs `doctor`) — operator can't accidentally invoke a lossy
verb (C3); SDK stays at five verbs (C5); engine ships zero network
behaviour (C2); installable default-embedder + caller-owned lifecycle
preserves NEED-017 and gives non-experts a working baseline (C5,
NEED-023).

**Cons.** Three Cargo crates instead of one (`fathomdb`,
`fathomdb-embedder`, `fathomdb-embedder-api`); release coordination
(REQ-048 co-tagging covers this).

**Option B — Operator CLI + read-only application query (`search /
get / list`).**

**Pros.** Covers ad-hoc inspection without a full SDK round-trip.

**Cons.** Introduces a second public query surface with little
additional operator value over `trace`, `dump-*`, and integrity tooling
(C5 violation — surface area grows).

**Option C — Full surface (admin + recovery + query + write) in one
binary.**

**Pros.** CLI is a complete interface; one tool covers everything.

**Cons.** Largest surface (C5 broken hard); typed writes via CLI flags
get unwieldy quickly; query parity doubles public surface and doubles
the docs/test burden.

**Option D — Engine-owned embedder (engine downloads + caches model
weights itself).**

**Pros.** Best DX — operator does nothing.

**Cons.** Direct violation of NEED-017, REQ-033, C2. Reopens the
local-first posture entirely. Not viable.

## 7. Recommended Solution

The **recommended solution is the union of the §5 Option-A picks across
all six axes**, codified in `dev/architecture.md` and the 12
`dev/design/*.md` subsystem docs. In short: a single-file
SQLite + WAL embedded engine with a single dedicated writer thread, a
hybrid sidecar-flock + SQLite-EXCLUSIVE lock, op-store + canonical +
vector + FTS rows in one transactional fence, push (eager) projections,
a five-verb runtime SDK, and a two-root operator CLI; embedder lifecycle
is caller-owned with an installable default-embedder sibling crate.

This recommendation optimises for **C1 (durability), C3 (recoverability),
C4 (bounded concurrency), and C6 (single-file deploy ergonomics)** —
all Must criteria — and accepts second-best on **C9 raw write
throughput** (single-writer model bounds it to one CPU + I/O, but
REQ-009a/b targets are met). The recommendation depends on Assumption 1
(SQLite WAL durability profile) and Assumption 2 (sqlite-vec viability)
from §3.3; if either turns out false, axes §5.1 and §5.5 re-open.

### 7.1 High-Level Description

The 0.6.0 engine is a single Rust crate (`fathomdb-engine`) exposing
five public verbs via three first-class binding facades (Python, TS,
Rust) and one operator binary (`fathomdb-cli`). Inside the engine:

- **`runtime`** — owns `Engine.open / close`, hybrid lock acquisition,
  eager embedder warmup, PRAGMA application, engine-config marshal.
- **`writer`** — single dedicated OS thread, single `rusqlite` writer
  connection, processes typed `PreparedWrite` batches in one
  transaction. Commits canonical + FTS + op-store rows atomically;
  vector rows are a follow-on writer-thread transaction (post-commit
  per Invariant A).
- **`reader`** — multi-connection reader pool; per-thread `rusqlite`
  read connections; WAL read transactions.
- **`migrations`** — auto-migrates at `Engine.open`; per-step structured
  events.
- **`op_store`** — `operational_*` tables; two collection kinds
  (`append_only_log`, `latest_state`); JSON-Schema payload validation.
- **`embedder`** — engine-owned dispatch pool (sized `num_cpus::get()`);
  invokes caller-supplied embedder synchronously; per-call timeout;
  eager warmup at open; identity-aware, encoding-naive.
- **`scheduler`** — Tokio runtime (default 2 worker threads);
  post-commit dispatch of projection jobs; `spawn_blocking` to embedder
  pool; mpsc back to writer; 4-layer backpressure.
- **`vector`** — `vec0` virtual table inside same SQLite file; LE-f32
  BLOB encoding; rebuild-from-canonical for physical recovery.
- **`projection`** — push-model FTS + vector projections; cursor
  allocation + advancement on writer thread; terminal-state advancement
  only.
- **`retrieval`** — fixed-stage pipeline (parse → match → fetch →
  expand); safe FTS5 grammar parser; soft-fallback signal.
- **`lifecycle`** — phase enum {Started, Slow, Heartbeat, Finished,
  Failed}; host-routed diagnostics; cumulative counters; per-statement
  profiling opt-in.
- **`recovery`** — CLI-only operator surface; `fathomdb recover
--accept-data-loss` (lossy) + `fathomdb doctor <verb>` (read-only).
- **`bindings facade`** — per-binding marshal of typed errors, async
  shape (TS Promise via `ThreadsafeFunction` + Rust pool; Python, Rust,
  and CLI sync), cursor surfaces.
- **`release`** — version-consistency check, atomic multi-registry
  publish, sibling co-tagging, registry-installed-wheel smoke.

On-disk file set during operation: `<db>.sqlite` + `<db>.sqlite-wal` +
`<db>.sqlite.lock`. (`.shm` is absent under WAL + EXCLUSIVE writer.)

Sibling crates installable separately: `fathomdb-embedder-api` (shared
trait surface), `fathomdb-embedder` (default candle + tokenizers + sqlite-
vec bundle).

Full data-flow diagrams for write path and read path live in
`dev/architecture.md` § 3 and § 4 respectively.

## 8. Alternatives Considered

Per axis, the losing options summarised. This section absorbs the per-
ADR "Rejected" notes so a reader can see the trade space at a glance.

**§5.1 Storage substrate.**

- _Custom KV + sidecar FTS + sidecar ANN (Option B)._ Lost on REQ-041
  single-file deploy and on license posture (RocksDB GPLv2). Cross-store
  consistency story too weak for canonical data.
- _sqlx async + sqlite (Option C)._ Lost on sqlite-vec extension parity
  (the dep audit accepted sole-maintainer risk on sqlite-vec only
  because rusqlite's extension-loading story is clean) and on the
  ~5–8k LoC engine rewrite cost. Reopens too many Phase-1 ADRs.

**§5.2 Concurrency model.**

- _Writer pool + `BEGIN IMMEDIATE` retry (Option B)._ Lost on the
  empirical 0.5.x `SQLITE_SCHEMA` flood — each retry re-reads schema,
  amplifying the regression window. The `learnings.md` Stop-doing entry
  is the durable record.
- _MVCC via WAL2/BEDROCK fork (Option C)._ Lost on supply-chain (forks
  SQLite, breaks sqlite-vec parity), license, and rewrite cost. Deferred
  to 0.7+ behind a forcing function.

**§5.3 Database lock mechanism.**

- _SQLite EXCLUSIVE only (Option A)._ Lost on lock-acquisition timing —
  SQLite EXCLUSIVE is acquired on first read, not at open, so a doomed
  open pays the warmup cost first.
- _Sidecar flock only (Option B)._ Lost on same-process exclusion: BSD
  `flock` is per-OFD; two `File::open` + `try_lock` calls in the same
  process can both succeed.
- _Direct file lock (BoltDB-style, Option E in ADR)._ Lost because
  locking the file SQLite is also touching may interact with SQLite's
  internal POSIX locks in undefined ways.
- _`open(O_EXCL | O_CREAT)` marker file (Option F in ADR)._ Lost on
  crash-safety — leftover marker after SIGKILL blocks all future opens
  until manual cleanup. Anti-pattern.
- _Process-global Rust registry (Option H in ADR)._ Lost on cross-process
  coverage — covers same-process only and even fails across same-process
  cdylib instances (Python + TypeScript loaded separately each have
  their own static registry).

**§5.4 Op-store placement and write-tx shape.**

- _Sibling `.opstore.sqlite` file (Option B)._ Lost on REQ-041 and on
  cross-file transactional consistency.
- _Drop op-store entirely (Option C)._ Lost on aggregate user cost — N
  clients re-implementing the same primitives is a worse outcome than
  the engine surface.

**§5.5 Projection model.**

- _Pull / lazy (Option B)._ Lost on first-read tail latency under
  interactive workloads; freshness window becomes per-read-trigger and
  resists a clean SLI.
- _Auto-detect hybrid (Option C)._ Lost on the speculative-knob class
  — "what counts as frequent" has no defensible default.

**§5.6 Operator surface and embedder ownership.**

- _CLI + read-only application query (Option B)._ Lost on C5 — adds a
  second public query surface for little operator value over `trace` /
  `dump-*` / `check-integrity`.
- _Full CLI surface (Option C)._ Lost on C5 hard — typed writes via CLI
  flags get unwieldy and query parity doubles the surface.
- _Engine-owned embedder (Option D)._ Lost on NEED-017 / REQ-033 / C2
  before any other criterion is scored.

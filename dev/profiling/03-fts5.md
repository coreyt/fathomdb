# 03 — FTS5 full-text index (`search_index`)

**Component:** the FTS5 virtual table `search_index(body, kind UNINDEXED,
write_cursor UNINDEXED)` (migration `005_search_index.sql`). The keyword/text
branch of hybrid search; query compiled by `fathomdb-query::CompiledQuery`.

## Why it matters

FTS5 is the text-recall half of hybrid retrieval and the cheapest branch to keep
fresh, but its ingest cost is **bursty, not linear** — the classic FTS5 trap
that makes naive "avg ms/row" ingest numbers lie.

## Ingest path — what to measure (the important part)

- **Segment-merge spikes.** FTS5 keeps level-based B-tree segments and **merges
  roughly every 64 inserts**, promoting merged segments up a level. Cost is a
  sawtooth: most inserts are cheap, periodic inserts trigger an expensive merge.
  **Profile the distribution, not the mean** — capture p99 and the merge spikes
  via `on_slow_statement`, and plot ms-per-insert over insert-index to see the
  sawtooth. A mean hides it.
- **Body size sensitivity** — tokenization + posting-list growth scale with body
  length, not just row count. Correlate with the body-length distribution from
  `02-canonical-store.md`.
- **`automerge` / `crisismerge` behavior** — large bulk loads accumulate many
  level-0 segments; query latency creeps until a merge. If the profiler does a
  bulk load, note whether an explicit `INSERT INTO search_index(search_index)
  VALUES('merge'|'optimize')` was run and at what cost (maintenance-window cost,
  not steady-state).

## Retrieval path — what to measure

- **`MATCH` cost** for the compiled query — separate the FTS5 match (posting-list
  walk) from body materialization. Today results are `ORDER BY write_cursor`
  (insertion order), **not** relevance.
- **`bm25()` cost (G9 input)** — G9 switches the text branch to `ORDER BY
  bm25(search_index)` for a real relevance rank. `bm25()` is intrinsic to FTS5
  (no schema change) but adds ranking computation; baseline `MATCH`-only now so
  the bm25 delta is attributable.
- **Unfiltered text-branch leak (G10 note)** — today the FTS5 branch has no
  metadata filter. When G10 adds filtered KNN, the text branch must be
  constrained too or it leaks unfiltered hits; profile the join cost that the
  constraint will add.

## Sharp edges

- `kind` and `write_cursor` are `UNINDEXED` columns — stored, not searchable;
  filtering by them on the text branch needs a join to `canonical_nodes`, not an
  FTS5 predicate.
- FTS5 tokenizer choice affects index stability and reproducibility
  (`docs/positions/tokenizer-policy.md`) — pin/record it in any profiling run.

## Scaling expectation

Match cost scales with matching-posting-list length (term frequency), not corpus
size directly — generally sub-linear and cheap relative to vec0 phase-1. Ingest
cost is amortized-linear but spiky; the spikes are the story.

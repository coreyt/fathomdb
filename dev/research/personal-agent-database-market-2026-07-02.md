# The Personal-Agent-Database Market — Landscape, Feature Taxonomy, and FathomDB's Next-Best Steps

**Date:** 2026-07-02
**Subject:** Market research on "personal agent databases" (databases / memory layers purpose-built for AI agents) and where FathomDB stands.

> **What this is.** A decision-ready market-research report on the agent-memory / "personal agent database" category, plus a code-verified audit of FathomDB against that market and three ranked next-best steps. It was produced by an 11-agent workflow: a **2-round, loop-until-dry web survey** across **4 angles** (products & vendors; technical substrates; capabilities & framing; builder needs & gaps) run by **8 survey agents**, feeding a **feature taxonomy** agent, a **code-verified FathomDB audit** agent (read against the repo, not the web), and a **synthesis/recommendations** agent. The survey found **~76 distinct players** across the two rounds (round 1: 34; round 2: 76 cumulative); **69 distinct products** survived name-level deduplication into the catalog in Appendix A. Every surveyed item is web-sourced with a URL (Appendix A); the FathomDB audit is verified against the repository's shipped code and roadmap.
>
> **Honest caveats preserved throughout:** MCP is absent from FathomDB's code *and* roadmap; the in-house graph arm adds ≈0 recall lift over BM25; FathomDB's benchmarks are used internally but unpublished (LoCoMo's CC-BY-NC license blocks payload publication); many competitor benchmark numbers (LongMemEval / LoCoMo) are vendor self-reported and not independently reproduced; several names seen in listicles were excluded as academic prototypes or unverifiable.

# FathomDB in the Personal-Agent-Database Market — Position & Next-Best Steps

## 1. Market map

The "personal agent database" / agent-memory category has exploded into ~60 players spanning four postures: **managed cloud** (Supermemory, AWS AgentCore, Cloudflare Agent Memory, Zep, Oracle AI Agent Memory, Microsoft Foundry), **database-vendor plumbing** (Oracle AI Database, MongoDB/LangGraph, TiDB/mem9, Tiger Data Agentic Postgres), **open-source frameworks/servers** (Mem0, Cognee, Letta, MemMachine, Redis Agent Memory), and a large **local-first indie cohort** hand-rolling SQLite + sqlite-vec + FTS5 (AIngram, memoirs, ClawMem, Rememora, TencentDB, engram, agentmemory) plus a few **from-scratch Rust engines** (MenteDB, MinnsDB, Memvid). The field has converged hard: RRF hybrid retrieval and an MCP server are now table-stakes, temporal/bi-temporal knowledge graphs are the dominant differentiator, and the frontier moats are from-scratch engines, cryptographic provenance, parametric memory, and converged single-ACID engines. FathomDB sits squarely in the local-first camp — but built as a *real Rust engine* rather than a thin wrapper — deliberately scoped **below** the agent loop as a substrate its own consumers (Memex/Hermes/OpenClaw) build on.

| Player | Posture | Signature bet |
|---|---|---|
| Supermemory / Zep / AWS AgentCore | Managed cloud | Temporal graph, profiles, sub-300ms, benchmark-led |
| Oracle / MongoDB / TiDB (mem9) / Tiger Data | DB-vendor plumbing | "Converged engine kills the vector+graph+relational stitch" |
| Mem0 / Cognee / Letta / MemMachine | OSS framework | Extraction pipelines, graph-vector, tiered memory |
| MenteDB / MinnsDB | From-scratch Rust engine | Custom WAL/HNSW, bi-temporal, ontology-cascade |
| AIngram / ClawMem / Rememora / TencentDB | Local-first indie (SQLite) | Single file, RRF, MCP + hooks, $0 embeddings |
| **FathomDB** | **Local-first engine (SQLite, Rust)** | **Converged-ACID embedded engine + supersession + real RRF, below the agent** |

## 2. Feature taxonomy (condensed)

**TIER 1 — Table-stakes:** persistent cross-session memory; RRF hybrid retrieval (vector + BM25/FTS); **MCP server (+REST + SDK)**; automatic LLM fact extraction; tiered memory taxonomy; dedup + conflict handling; local/model-agnostic embeddings; user/session/agent scoping; LoCoMo/LongMemEval benchmark reporting.

**TIER 2 — Important (differentiators):** temporal/bi-temporal knowledge graph; graph-native entity/relation traversal; fact invalidation/supersession; forgetting mechanics (decay/TTL); retrieve-then-rerank + advanced retrieval (HyDE/RAPTOR/HippoRAG/MMR); autonomous consolidation loops; local-first/air-gapped; multi-tenant governance; explicit memory typing; user-profile memory; connectors + multimodal ingest; coding-agent hooks; progressive context loading; polystore backends.

**TIER 3 — Exceptional (moats):** self-evolving skill distillation; parametric/activation memory; from-scratch storage engine; cryptographic/tamper-evident provenance; synthesis-over-ranked-list ("Reflect"); ontology-cascade invalidation + temporal query language; 5–7-source retrieval fusion; write-time cognitive intelligence; novel non-DB substrates; CRDT/P2P sync; screen/activity capture; **converged single-ACID engine**; Web3 memory; built-in eval harness.

**Strengths:** (1) best-in-class local-first embedded substrate — the exact stack consumers hand-roll, built as a real engine; (2) genuine RRF + CPU CE-rerank done properly; (3) identity + supersession + edge valid-time — a real bi-temporal foundation; (4) the converged-ACID embedded bet; (5) disciplined, injection-safe governed surface with Py↔TS parity.

**Gaps:** (1) **no MCP/REST** — most damaging, absent from code and roadmap; (2) nothing above the storage line (connectors, hooks, profiles, multimodal); (3) extraction/consolidation are seams, not features (consolidation unmerged, defaulted OFF); (4) no memory-tier taxonomy, forgetting, or multi-tenancy; (5) no advanced retrieval beyond CE-rerank; (6) benchmarks unpublished.

## 4. Three next-best steps (ranked)

### #1 — Ship an MCP server (+ thin REST) over the existing governed surface

**What:** Wrap the already-clean `read.*` / `graph.*` / ingest surface in an MCP server (and a minimal REST facade), exposing search/neighbors/search_expand/ingest as MCP tools. No new engine work — a protocol adapter over the existing SDK boundary.
**Why:** This is the single biggest table-stakes gap and it is *absent from the roadmap entirely*. MCP is now the category default integration surface — near-universal across Redis, Cognee, Mem0, and virtually the entire local-first indie cohort (AIngram, ClawMem, Rememora, TencentDB all ship MCP + hooks). FathomDB's consumers (OpenClaw/Hermes/Claude Code) are MCP-native; without it, FathomDB can only be reached through bespoke SDK integration while every indie competitor is a one-line MCP install. The governed, injection-safe closed-filter surface makes this unusually cheap and low-risk to expose — FathomDB is *more* ready for this than most.
**Effort:** Medium (protocol adapter + tool schema + auth-less local transport; the hard part — a safe, parity-enforced surface — already exists).
**Roadmap relation:** **Net-new.** Should preempt lower-urgency roadmap items; nothing on the 0.8.x line addresses it.

### #2 — Make identity first-class: user / session / agent namespacing

**What:** Promote scoping from caller-encoded `kind`/`source_id` convention to a first-class namespace dimension (schema column + `SearchFilter`/read-path plumbing + conformance parity), so multi-agent consumers get isolated memory without overloading `kind`.
**Why:** Table-stakes (scoping is Tier 1) and currently only PARTIAL — callers must hand-encode identity, which is exactly the friction a shared substrate should remove. FathomDB's stated consumers are multi-agent (Memex/Hermes/OpenClaw share vaults); competitors like agent-memory-store, MongoDB LangGraph namespaces, and Mem0 treat user/session/agent scoping as a primitive. This is the prerequisite that unlocks credible multi-agent use and is a stepping stone toward the multi-tenant governance FathomDB entirely lacks.
**Effort:** Small–medium (schema migration + filter plumbing + Py↔TS conformance; contained, no retrieval-algorithm risk).
**Roadmap relation:** **Reprioritize** — pull forward ahead of speculative retrieval experiments; it compounds with #1 (MCP tools need clean identity scoping to be safe for shared agents).

### #3 — Turn the bi-temporal foundation into an AS-OF point-in-time query (leapfrog)

**What:** Build the point-in-time query surface on top of the existing `t_valid`/`t_invalid` edge validity — an "AS OF" read that returns the graph/facts as they were valid at a given time — and land the projection-durability fix already identified as the 0.8.12 blocker.
**Why:** This is FathomDB's clearest **leapfrog**. Temporal/bi-temporal graphs are the market's *dominant differentiator* (Zep/Graphiti, MinnsDB, MenteDB, ByteRover, Memobase), and a temporal query language with AS-OF clauses is Tier-3 EXCEPTIONAL (MinnsDB's MinnsQL). FathomDB already has the hard part — valid-time on edges plus invalidate-not-delete supersession — a *foundation most indie competitors lack entirely*. Exposing the query surface converts a latent strength into a headline differentiator, and it composes with the converged-ACID pitch (temporal correctness under one transaction). Be honest: demand is less proven than #1/#2, and nodes remain untimestamped, so scope it to edge/fact validity first.
**Effort:** Medium (query surface + the known projection-durability fix; foundation already schema-v15).
**Roadmap relation:** **Accelerate** — unpauses and refocuses the stalled 0.8.12 temporal/consolidation work around the query surface (the durable-fact payoff) rather than the STAY-OFF consolidation provider.

**Bottom line:** FathomDB is engineered above the indie floor on substrate, RRF, and supersession, but it is *behind on the table-stakes integration surface* (MCP) and identity scoping — close those first (#1, #2). Its one credible path to *leapfrog* the field is the bi-temporal query surface (#3), where it already holds a foundation most competitors don't. Do not chase connectors, profiles, hooks, or in-library extraction — those are deliberately ceded to consumers and are not where this engine wins.

**Agent notes:**

BUILDER NEEDS & GAPS (the requested angle), synthesized from vendor engineering posts, GitHub READMEs, dev.to/Medium builder writeups, and OSS-Insight/DevelopersDigest commentary:

1. TRUSTWORTHINESS BEYOND RETRIEVAL is the loudest gap. developersdigest.tech argues 'retrieval benchmarks reward finding stored facts' but builders actually need contradiction handling, provenance tracking, and authority validation — none of which LongMemEval measures. Concrete failure: an agent confidently applies stale memory ('deploys use Vercel') after a project switched platforms.

2. THE 'WHAT TO STORE' PROBLEM. Manual tagging => nobody does it consistently; save-everything => noise. No consensus on verbatim vs extracted vs graphed storage. MemPalace stores verbatim (96.6% LongMemEval) but with unbounded growth; others extract but pay LLM cost/error tax.

3. DELETION / ROLLBACK / STALENESS. Stale memories persist with no clear removal path and get retrieved because they're semantically close. Builders want deterministic deletion and supersession — this is why Cloudflare (supersession chains, Forget), Mem0 (conflict resolution) and Memori (SQL versioning) all advertise it, and why critics still call it unsolved.

4. CROSS-AGENT COORDINATION & GOVERNANCE. Multiple agents keep separate memory silos => duplicated context + conflicting facts + no unified deletion. Shared memory 'multiplies value and blast radius at the same time' — leakage of sensitive/project-scoped rules across scopes is a named fear. Drives the shared-profile + tenant-isolation features.

5. CONTEXT ROT / DURABILITY OVER TIME. Even million-token windows suffer context rot; the tension is retain-everything (quality degrades) vs prune-aggressively (lose needed info). Builders want memory that 'stays useful as it grows, not just memory that performs on a clean benchmark' (Cloudflare). Benchmarks don't prove 'longitudinal value' over months (OSS Insight).

6. RAG-vs-MEMORY CONFUSION is repeatedly cited as a top source of developer error, producing agents that are over-engineered or 'systematically blind.'

7. EVALUATION IS HARD because memory failures are invisible — a plausible answer grounded in stale/missing episodic context. Builders lack good eval harnesses.

8. ARCHITECTURAL UNCERTAINTY: no one knows whether the memory layer belongs in the agent, the backend, the context loader, or the file system. Installation complexity varies wildly (single Go binary vs multi-language build pipelines) and is itself a complaint.

9. COST & TRANSPARENCY BACKLASH against vector DBs: Memori/GibsonAI explicitly pitch SQL-native memory as 80-90% cheaper and inspectable ('why was this memory retrieved?'), and a wave of local-first SQLite-FTS5 MCP servers (engram, ai-memory-mcp, shodh-memory) reject cloud dependency entirely — directly relevant to FathomDB's SQLite+FTS5 positioning.

CAVEATS ON SOURCES: The OSS-Insight 'Agent Memory Race 2026' blog lists repos with implausibly high star counts and possibly anonymized/renamed names (MemPalace 43k, OpenViking 21k 'by ByteDance/volcengine', code-review-graph, SimpleMem) — I could NOT independently verify those as real shipping products, so I excluded them from the products list and cite only their claims as market signal. # Agent-Memory Database — Feature Taxonomy

Tier placement is by prevalence and defensibility across the ~60 surveyed players: TABLE-STAKES = present in a large majority (its absence reads as incomplete); IMPORTANT = present in a meaningful subset and used to differentiate; EXCEPTIONAL = one-off or handful, cutting-edge, hard to copy.

## TIER 1 — TABLE-STAKES (a credible product must have it)

- **Persistent cross-session memory** — Durable recall surviving session/restart/model changes; the category's baseline. *Effectively all players; explicit in MemMachine, mem9, OpenBrain, Rememora.*
- **Hybrid retrieval (dense vector + BM25/FTS keyword, fused via RRF)** — Two-signal recall merged by Reciprocal Rank Fusion rather than vector-only. *Near-universal: Cloudflare, AIngram, memoirs, ClawMem, agent-memory-store, TencentDB, memweave, personal-ai-memory, Supermemory, Tiger Data.*
- **MCP server interface (plus REST + Python/TS SDK)** — Model Context Protocol server as the default integration surface, with REST/SDK alongside. *Near-universal: Redis, Cognee, Mem0/OpenMemory, MemMachine, shodh-memory, engram, ai-memory-mcp, doobidoo, and most indie servers.*
- **Automatic LLM-based fact extraction from conversation** — Pipeline that pulls facts/entities out of raw turns instead of storing verbatim only. *Redis, Mem0, Cognee ECL, Hindsight, Supermemory, Oracle, Microsoft Foundry, MenteDB. (Counter-trend: MemMachine/Memori/MemPalace deliberately avoid per-turn LLM extraction on cost grounds — an intentional design axis, which confirms it's the default.)*
- **Tiered memory taxonomy (working/short-term vs. long-term)** — At minimum a session buffer plus durable store, usually split further into semantic/episodic/profile. *MemMachine, Redis, AWS AgentCore, Letta, ai-memory-mcp, CrewAI, LlamaIndex.*
- **Deduplication + basic conflict/contradiction handling** — Content-hash dedup and conflict resolution so re-ingest and restated facts don't pollute. *Redis (content hashing), Supermemory (SM tokens), Mem0, AWS AgentCore, ai-memory-mcp, VEKTOR.*
- **Local / model-agnostic embedding + LLM support** — Works across providers and offers local embeddings (all-MiniLM, BGE, nomic, GGUF, ONNX). *Redis (100+ via LiteLLM), MemMachine, agent-memory-mcp, sqlite-memory, agentmemory, MenteDB.*
- **Scoping / namespacing (user / session / agent)** — Records keyed by identity so memories don't cross-contaminate. *Mem0, Oracle, MongoDB LangGraph namespaces, agent-memory-store, LangMem.*
- **Benchmark reporting (LoCoMo / LongMemEval)** — Published accuracy/recall on the two de-facto suites is expected table-stakes marketing. *Supermemory, Hindsight, EverMind, Zep, ByteRover, Memobase, agentmemory, MenteDB, MIRIX.*

---

## TIER 2 — IMPORTANT (differentiates serious players)

- **Temporal / bi-temporal knowledge graph (valid_at/invalid_at, valid_from/valid_until)** — Facts carry validity windows so the agent knows *when* something was true; enables point-in-time queries. *Zep/Graphiti, MinnsDB, MenteDB, Memobase, ByteRover, memoirs, Minigraf.*
- **Graph-native memory (entity/relationship extraction + traversal)** — Entities and typed relations stored as a graph and traversed for multi-hop reasoning. *Cognee, Neo4j/Lenny's, Memary, Zep, Papr, Memory-Graph, Membase.*
- **Fact invalidation / supersession (retire, don't delete; keep provenance)** — Contradicted facts marked superseded with history preserved rather than overwritten. *Zep (most explicit), Cognee memify, MenteDB "Supersedes" edges, Mem0.*
- **Memory lifecycle / forgetting mechanics** — Decay curves, recency bias, TTL/auto-expiry, promotion of high-signal facts. *Supermemory (decay/expiry), agentmemory (Ebbinghaus), ByteRover AKL, shodh-memory, Microsoft Foundry TTL, agent-memory-store TTL.*
- **Two-stage retrieve-then-rerank + advanced retrieval techniques** — Bi-encoder shortlist then cross-encoder rerank, plus HippoRAG/PPR, RAPTOR, HyDE, MMR, intent routing. *ClawMem, memoirs (HippoRAG/RAPTOR), Cloudflare (HyDE), memory-lancedb-pro, VEKTOR, memweave (MMR).*
- **Autonomous consolidation / self-improvement loops** — Background stages that prune, reweight, merge, or synthesize derived facts. *Cognee Memify, VEKTOR REM/AUDN, agentmemory (sleep-consolidation), OpenViking self-iteration, Rememora AUDN, AgentMemory hourly consolidation.*
- **Local-first / single-file / air-gapped deployment** — Runs fully on-device with zero egress; single SQLite file or single binary. *AIngram, memoirs, ClawMem, Memvid, MenteDB, VEKTOR, Rememora, Pieces, Supermemory (self-host).*
- **Multi-tenant governance (isolation, permissions, audit, retention)** — Per-user/tenant isolation with access control, retention policies, provenance, compliance delete. *Zep, Cognee (notesets), Oracle, Microsoft Foundry, MenteDB (JWT/ACLs), Papr RBAC.*
- **Explicit memory typing / classification** — Facts vs. Events vs. Instructions vs. Tasks, or Working/Episodic/Semantic/Procedural tiers as first-class types. *Cloudflare (4 types), MIRIX (6 types), agentmemory (4-tier), Memory-Graph typed memories, AWS AgentCore strategies.*
- **User-profile memory built from behavior** — Structured, evolving profile of intent/preferences/habits, separate from raw episodic recall. *Supermemory, MemMachine, Memobase, Agno, AWS AgentCore user-preferences.*
- **Data/source connectors + multimodal ingest** — Notion/Drive/Gmail/S3/Slack connectors and PDF/image/audio/video extraction. *Supermemory, Cognee (38-80+ sources), MinnsDB (49 connectors), EverMind mRAG, MemMachine.*
- **Coding-agent lifecycle hooks (auto-capture + context injection)** — SessionStart/Stop/pre-compaction hooks that capture and re-inject context with zero manual effort. *agentmemory (12 hooks), ClawMem, shodh-memory, Rememora, Supermemory, AIngram.*
- **Tiered / progressive context loading for token reduction** — L0 abstract → L1 overview → L2 full, resolving common queries without LLM calls. *OpenViking (L0/L1/L2), ByteRover 5-tier, Rememora, MenteDB (~80% token cut), codebase-memory-mcp.*
- **Polystore / pluggable backends** — One API over swappable vector/graph/relational stores. *Cognee (Kuzu/Neo4j/NetworkX × LanceDB/Qdrant/Weaviate × PGVector/SQLite), Redis (Redis/Pinecone/Chroma/PG), Papr (Mongo+Qdrant+Neo4j), Memory-Graph (8 backends).*

## TIER 3 — EXCEPTIONAL (rare / cutting-edge / moat)

- **Self-evolving skill distillation (activity → reusable skills/policies)** — Records agent runs as cases and distills durable, versioned skills so the agent compounds over time. *EverMind (Cases→Skills), MemOS (L1/L2/L3 + Crystallized Skills), MentisDB (git-like skills registry), OpenViking (accumulated experience).*
- **Parametric / activation memory as a memory type** — KV-cache reuse and LoRA/adaptation weights treated as first-class memory beyond text. *MemOS (Textual/Activation/Parametric MemCubes) — essentially unique in the set.*
- **Purpose-built storage engine written from scratch** — Real DB internals (crash-safe WAL+LZ4, buffer pool, HNSW, roaring bitmaps, CSR/CSC graph) rather than a wrapper over SQLite/a vector DB. *MenteDB, MinnsDB (single Rust binary: TKG + vector + BM25 + claim store + temporal tables).*
- **Cryptographic provenance / tamper-evident memory** — Every entry signed into a verifiable hash chain. *AIngram (Ed25519 hash chain) — unique; decentralized zk-verified variant in Membase/Unibase DA.*
- **Synthesis-over-ranked-list retrieval ("Reflect")** — Returns an LLM-synthesized insight across the whole memory bank instead of a ranked hit list. *Hindsight Reflect; related to MemOS/EverMind self-evolution.*
- **Ontology-driven cascade invalidation + custom temporal query language** — OWL/RDFS ontology drives structural supersession of dependent facts; MinnsQL with WHEN/AS OF point-in-time clauses and NL→query synthesis. *MinnsDB (and live/subscription queries that push deltas when facts change — unique).*
- **Seven-source retrieval fusion with exotic channels** — Beyond vector+BM25+graph: edge/triplet vectors, verified-claim confidence, DRIFT community search, episodic recall, all RRF-fused. *MinnsDB; Cloudflare's 5-channel (incl. HyDE + exact fact-key) is the next closest.*
- **Write-time cognitive intelligence** — Belief propagation on fact changes, quality filtering, "pain signals"/"phantom memories," attention-aware U-curve context assembly. *MenteDB — distinctive branded internals.*
- **Novel non-DB substrates** — Single portable video-inspired `.mv2` file with rewindable Smart Frames (Memvid); virtual filesystem addressed by `viking://` URIs with recursive directory retrieval (OpenViking); copy-on-write instant DB forks/snapshots per agent branch (Tiger Agentic Postgres); browser IndexedDB + in-WASM ONNX embeddings (personal-ai-memory).
- **CRDT offline-first / P2P federated sync** — Central-server-free multi-agent memory convergence. *sqlite-memory (block-level LWW CRDT via sqlite-sync), AgentMemory (authenticated HTTPS mesh federation), Membase/Unibase (decentralized DA).*
- **Passive screen/activity capture as a memory source** — Continuously ingests on-screen/app/audio activity into structured memory. *MIRIX (ScreenshotVQA), Pieces LTM-2.7 (~9 months, all-apps on-device).*
- **Converged single-ACID-engine pitch** — One transactional engine spanning vector + relational + graph + JSON so there are no stitched stores or consistency gaps. *Oracle AI Database, Tacnode Context Lake; TiDB (mem9) and MongoDB adjacent.*
- **Verifiable/decentralized Web3-native memory** — zk-verified data-availability storage with agent-to-agent identity protocol. *Membase/Unibase (Membase-MCP + AIP + Unibase DA) — the only Web3 entrant.*
- **Symbolic / human-readable state encodings** — Short-term memory as Mermaid state-transition diagrams; personas/scenarios as Markdown. *TencentDB Agent Memory — unusual symbolic representation.*
- **Built-in evaluation harness shipped with the product** — Recall-quality eval infra and memory visualization bundled in, not just a benchmark claim. *mem9/PingCAP (eval infra + "Memory Farm"); rare as a shipped feature vs. a marketing number.*

### Notes on grounding and judgment calls

- **RRF hybrid retrieval and MCP delivery moved from "differentiator" to table-stakes** within this cohort — the digest's own `featuresObserved` repeatedly calls both "now the default"/"table stakes," and they appear in the overwhelming majority of players.
- **Temporal/bi-temporal graphs sit at the top of IMPORTANT, not EXCEPTIONAL**: the digest flags them as a "dominant differentiator" and lists ~8+ implementers, so they're widespread among serious players but still absent from the many SQLite-FTS5 indie servers — differentiating, not universal.
- **Local-first is IMPORTANT, not EXCEPTIONAL**, because it's a whole camp (the SQLite+sqlite-vec+FTS5 cohort); only its *exotic substrates* (video-file, viking://, CoW forks, IndexedDB/WASM, decentralized DA) rise to EXCEPTIONAL.
- **EXCEPTIONAL items are each carried by one or a couple of players** and represent genuine engineering moats (from-scratch engines, cryptographic chains, parametric memory, ontology-cascade + query language, CRDT/P2P, screen capture) rather than configuration of common parts.

## FathomDB's clear STRENGTHS vs the market

1. **Best-in-class local-first embedded substrate.** Single SQLite file, CPU-only deterministic query path, zero egress, no LLM in the read path — it *is* the exact SQLite+sqlite-vec+FTS5 stack that shipping agents (OpenClaw, Hermes) hand-roll, but built as a real Rust engine with a writer thread, reader pool, and projection system.
2. **Genuine RRF hybrid retrieval + CPU cross-encoder rerank** — table-stakes done properly (real rank fusion, not union-dedup), with an opt-in TinyBERT reranker and tunable α-blend. Above the indie-server floor.
3. **Identity + supersession substrate (invalidate-not-delete) with provenance** — the Zep-pattern fact-retirement done at the engine level, plus valid-time (`t_valid`/`t_invalid`) on edges: a real bi-temporal *foundation*.
4. **Converged engine bet** — vector + relational/KV + typed graph + JSON under one ACID SQLite transaction, with bounded graph traversal (`neighbors`, `search_expand`) — no stitched stores.
5. **Disciplined governed surface** — `read.*`/`graph.*` namespaces, closed typed filter (injection-safe, not a DSL), Python↔TypeScript parity enforced by conformance.

## FathomDB's clear GAPS vs the market

1. **No MCP server and no REST server** — the single most damaging table-stakes gap. Integration is Python/TS SDK only; the category default (MCP) is absent from code *and* roadmap.
2. **No connectors, no multimodal ingest, no coding-agent hooks, no user-profile building** — everything above the storage/retrieval line is pushed to the consumer. FathomDB is deliberately the engine *below* the agent, not a batteries-included memory product.
3. **Extraction and consolidation are seams, not features** — BYO-LLM extractor protocol ships, but no extractor; consolidation/recency provider is built-but-unmerged and defaulted OFF (0.8.12 paused). Coverage-lift de-prioritized (`CEILING-ABSORBED`).
4. **No first-class memory-tier taxonomy, no forgetting mechanics** (decay/TTL off/unshipped), **no multi-tenant governance** (single-user embedded, no RBAC/isolation/retention).
5. **No advanced retrieval beyond CE-rerank** (no HyDE/RAPTOR/HippoRAG/MMR); the planner-router/dispatcher that would route per-intent is PLANNED (0.8.15), not shipped. Graph arm shows ~0 recall lift over BM25 in-house.
6. **Benchmarks used internally but not published**; LoCoMo licensing (CC-BY-NC) blocks payload publication.

**Net positioning:** FathomDB is a rigorously-engineered **local-first embedded retrieval+graph engine** — strong on the substrate, hybrid retrieval, identity/supersession, and the converged-ACID bet — deliberately scoped *below* the agent loop. It cedes the entire product-surface layer (MCP/REST, connectors, hooks, profiles, extraction/consolidation content, multi-tenancy) to its consumers, and its most conspicuous competitive omission is the now-table-stakes **MCP interface**.

# Appendix E — Methodology & provenance

**Workflow shape.** An 11-agent orchestrated workflow across four phases:

| Phase | Agents | Role |
|---|---|---|
| 1 · Survey | 8 (`survey:r0:a0..a3`, `survey:r1:a0..a3`) | Web survey — 4 angles × 2 rounds, general-purpose agents with web search + fetch |
| 2 · Taxonomy | 1 (`taxonomy`) | Fold the survey digest into a 3-tier feature taxonomy |
| 3 · Audit | 1 (`audit`) | Read the FathomDB repo and grade it against the taxonomy (code-verified) |
| 4 · Recommend | 1 (`synthesize`) | Produce the market map, condensed taxonomy/audit, and 3 ranked next-best steps |

**2-round, loop-until-dry survey design.** Round 1 ran the four angles cold (no seed list). Its finds were pooled and fed back into round 2 as an "already-found players (find NEW ones, do not just repeat)" exclusion list, so round 2 pushed into the long tail rather than re-reporting incumbents. The workflow's own progress log recorded **34 distinct players after round 1** and **76 after round 2**. The four angles:

- **A — Products & vendors:** "agent memory", "personal agent database", "memory layer for AI agents", "long-term memory for LLM agents".
- **B — Technical substrates:** how products store memory — vector DBs, embedded SQLite + sqlite-vec, FTS/BM25, knowledge-graph memory, local-first / on-device.
- **C — Capabilities & framing:** temporal / knowledge-graph memory, consolidation/supersession, retrieval + reranking, extraction/ingestion, multi-tenant, privacy/local-first, benchmarks (LoCoMo etc.).
- **D — Builder needs & gaps:** what agent developers say they need or complain is missing (blogs, HN, Reddit, product docs).

**Web-sourced vs code-verified.** Every player in Appendix A is **web-grounded** — each survey agent fetched or cited a live first-party page (GitHub / PyPI / vendor site / vendor blog); URLs are retained in Appendix A. Survey agents flagged their own verification gaps in the notes (Appendix B): several names appearing only in listicles or arXiv papers were **excluded** as academic prototypes (e.g. MemoriesDB, SuperLocalMemory, A-MEM) or unverifiable community claims (e.g. the MemPalace benchmark numbers, publicly debunked; MinnsDB's `minnsdb.com` failed to resolve, corroborated only via secondary sources). Competitor benchmark figures (LongMemEval / LoCoMo R@5) are **vendor self-reported marketing**, not independently reproduced — and several vendors' claims are mutually inconsistent. By contrast, **Appendix D (the FathomDB audit) is code-verified** against the repository: shipped code in `src/rust/crates/` and `src/python/fathomdb/`, `docs/reference/python-api.md`, `CHANGELOG.md`, schema `fathomdb-schema/src/lib.rs`, and the roadmap (`dev/plans/0.8.6-0.8.16-PROGRAM-SEQUENCING.md`, `plan-0.8.12.md`), distinguishing merged-on-`main` from branch-built-but-unmerged from roadmap-planned.

**Honest caveats carried into the recommendations (do not lose these).**

# 0.7.0 test-corpus research

## Why this exists

`tests/perf_gates.rs` AC-013 seeds a 1 M-row synthetic single-kind
corpus with `VaryingEmbedder` — uniform random "doc" rows whose
embeddings concentrate mass on six coordinates out of 768. That
fixture is fine for **latency gating** and even survives binary
quantization (per the Pack 1/Pack 2 plan in
`dev/notes/0.7.0-vector-cost-research.md`), but it is **not
representative**: zero metadata variety, zero threading, zero
cross-document semantics, and atypically friendly to bit-sign
quantization.

Pack 2's recall-floor RED test (≥ 0.90 recall@10 vs f32 ground
truth) currently runs against the same synthetic fixture — which
risks the gate passing for the wrong reason. A real-data composite
corpus is required (a) to validate that recall floors hold on
production-shaped data, (b) to exercise metadata + partition_key
filtering once Pack 1 lands, and (c) to lay groundwork for
cross-document reasoning evaluation.

### 1.1 E-mail — Enron + EnronQA

**Primary substrate: CMU Enron Email Dataset**

- URL: <https://www.cs.cmu.edu/~enron/>
- Size: ~500 K messages, ~150 users (senior management). The "May 7,
  2015 Version" tarball (~1.7 GB tarred+gzipped) is the
  community-standard distribution.
- **2026 update flag.** The CMU page now carries a footer dated
  April 2026 noting that "digital forensics experts" have raised
  authentication concerns about possible message impersonation
  within the corpus. This does not affect the dataset's usefulness
  as a synthetic-realistic substrate but should be cited in any
  paper-grade work and noted on FathomDB's corpus card.
- License: distributed by CMU "as a resource for researchers"; FERC
  released the underlying mail in the Enron investigation. There is
  **no explicit OSI-style license**; CMU asks users to "be sensitive
  to the privacy of the people involved." Treat as **research-use,
  redistribution unclear**.
- Friction: one-click HTTP download from CMU.
- Format: maildir directory tree (raw RFC-822 messages, no
  attachments).
- Metadata richness: full headers (sender, recipients, cc, bcc,
  date, subject, in-reply-to, references), folder structure → gives
  us threading + person + folder organisation for free.
- **FathomDB fit: yes-with-caveats.** Use as e-mail substrate.
  Caveats: redistribution posture is ambiguous (see §5).

**Evaluation overlay: EnronQA**

- Paper: <https://arxiv.org/abs/2505.00263> (May 2025, Ryan et al.,
  Stanford + Snowflake).
- HuggingFace mirror: <https://huggingface.co/datasets/MichaelR207/enron_qa_0922>
- Confirmed size from HF dataset card: **221 K rows across train /
  dev / test (73.8 K each)**, parquet format. (The paper's headline
  "~100 K emails, ~500 K QA pairs" figure refers to the underlying
  email + QA-pair counts; each HF row is one email with 2–11 QA
  pairs.)
- Fields (verified from HF card): `email`, `questions`,
  `rephrased_questions`, `gold_answers`, `alternate_answers`,
  `incorrect_answers`, `gold_rationales`, `alternate_rationales`,
  `include_email` (binary flags), `path`, `user`, `questions_count`.
- License: **not explicitly stated** on the HF card; the paper does
  not surface a license either. Treat as derived from the Enron
  base — same redistribution ambiguity (see §5).
- Friction: HuggingFace one-line `load_dataset`.
- **FathomDB fit: yes** — gives us a ready-made gold QA layer for
  per-user personalized retrieval, which maps to FathomDB's
  user-isolation story. The `user` and `path` fields supply
  per-user partitioning.

Ratio recommendation: 15–25 % of total corpus (per user research,
confirmed sound — Enron's corporate slant means it should not
dominate).

### 1.2 Meeting minutes — QMSum + ELITR

**Query-focused retrieval: QMSum**

- Repo: <https://github.com/Yale-LILY/QMSum>
- Paper: <https://aclanthology.org/2021.naacl-main.472/> (NAACL 2021).
- Size: 1,808 query-summary pairs over 232 meetings across three
  domains — `Academic`, `Product`, `Committee` (verified from the
  README). The user's research described "AMI / ICSI / Committee";
  the current README names the domains as Academic / Product /
  Committee. The Academic and Product splits are derived from ICSI
  and AMI respectively, but the published QMSum distribution
  exposes them under the rebranded names.
- License: **not surfaced in the GitHub README.** The QMSum paper
  notes that AMI and ICSI carry CC-BY-4.0 / CC-BY-NC variants.
  Treat as **research-use; explicit license check required before
  any redistribution.**
- Friction: clone repo, JSON/JSONL files in `data/ALL`.
- Format: JSON. Each meeting has `meeting_transcripts` (speaker +
  utterance), `general_query_list`, `specific_query_list`. No
  timestamps in the public release.
- Metadata: speaker names, domain tag, query↔meeting alignment.
- **FathomDB fit: yes** — small (232 meetings) but each meeting is
  long; ideal for chunk-level retrieval evaluation and for the
  "Summarize the part of the meeting about Y" workload.

**"Minutes as documents": ELITR Minuting Corpus**

- Landing page: <https://ufal.mff.cuni.cz/elitr-minuting-corpus>
- LINDAT/CLARIAH-CZ repository: <https://lindat.mff.cuni.cz/repository/xmlui/handle/11234/1-4692>
- Paper: <https://aclanthology.org/2022.lrec-1.340/> (LREC 2022).
- **Size correction.** The user's research cited "113 English + 53
  Czech, 160 hours." The current LINDAT v1.0 record shows **120
  English + 59 Czech meetings, ~180 hours**. Numbers updated
  post-publication.
- License: **CC BY-NC-SA 4.0** (verified from LINDAT). NonCommercial
  - ShareAlike — incompatible with commercial CI artifacts in some
  legal readings (see §5).
- Friction: LINDAT click-through (accept license, download tarball).
- Format: transcripts + multiple manually-written minutes per
  meeting + alignments. Available in raw text and in JSON via
  <https://github.com/guokan-shang/elitr-minuting-corpus>.
- Metadata: meeting ID, language, multiple-minutes-per-meeting
  (each created independently — useful for evaluating
  agreement-style retrieval).
- **FathomDB fit: yes-with-caveats.** Excellent for "real meeting
  minutes as personal knowledge docs." NC-SA license blocks
  inclusion in any commercial-leaning artifact.

### 1.3 Journal articles — PMC OA + S2ORC + OpenAlex

**Full-text scholarly: PMC Open Access Subset**

- Landing: <https://pmc.ncbi.nlm.nih.gov/tools/openftlist/>
- NIH organises the subset into three buckets: **Commercial Use
  Allowed** (CC0, CC-BY, CC-BY-SA, CC-BY-ND), **Non-Commercial Only**
  (CC-BY-NC, CC-BY-NC-SA, CC-BY-NC-ND), and **Other** (custom /
  no machine-readable license).
- **Important access constraint (verified 2026-05-27).** NIH
  documentation states: *"The PMC Cloud Service, PMC OAI-PMH
  Service, PMC FTP Service, E-Utilities and BioC API are the only
  services that may be used for automated retrieval of PMC
  content. Systematic bulk retrieval through other automated
  processes is prohibited."* For FathomDB, this means: pull
  through the allowed channels, then redistribute only the
  Commercial-Use-Allowed subset.
- Format: JATS XML; PDFs where available; plain-text via BioC API.
  Bulk packages are tarred.
- Metadata: full bibliographic record (authors, journal, DOI,
  PMID, dates, references, sections, figures, tables).
- **FathomDB fit: yes-with-caveats.** Use **only the
  Commercial-Use-Allowed bucket** if anything ships in a
  publicly committed artifact.

**Broad academic-text: S2ORC**

- Repo: <https://github.com/allenai/s2orc>
- Current size (per repo): metadata for **~136 M papers, full
  text for ~12 M** (figures larger than the user's quoted
  81.1 M / 8.1 M from the 2020 paper — corpus has grown).
- License: **ODC-By 1.0** (attribution required).
- Access: Semantic Scholar Public API (Bulk Dataset endpoint;
  requires free API key). Direct S3 download was deprecated.
- Format: JSON per paper with structured `body_text`, `bib_entries`,
  inline-citation spans.
- Metadata: rich — title, authors, year, venue, citations as
  graph edges between papers.
- **FathomDB fit: yes** — and the citation graph is the killer
  feature for cross-document evaluation (see §2).

**Metadata index: OpenAlex**

- AWS Registry: <https://registry.opendata.aws/openalex/>
- License: **CC0** (verified — "complete dataset is free under
  CC0").
- Size: **477 M+ works** indexed; AWS S3 bucket `s3://openalex`
  with no-auth access; quarterly snapshots.
- Format: JSON-Lines, partitioned by entity (`works`, `authors`,
  `institutions`, `concepts`, `sources`).
- **FathomDB fit: yes** as metadata-enrichment layer (CC0 is
  perfect for redistribution); **no** as a full-text substrate
  (no full text, only metadata + abstracts where licensed).

## 2. Cross-document fact reconstruction

This is the highest-value evaluation property: testing whether
retrieval reconstructs the user's working context rather than just
returning semantically-similar chunks. The bar is *not* "did we
recall the right document?" — it's "given a query, did we surface
the EMAIL + the MEETING + the TODO + the NOTE that together carry
the decision?"

### 2.1 What ships with native cross-doc links

| Source | Native cross-doc structure | Coverage |
|---|---|---|
| Enron | `In-Reply-To`, `References`, `Message-ID` → **reply threads** + folder organisation. Sender/recipient → person graph. | Strong intra-email. No cross-modal. |
| EnronQA | Per-user QA pairs anchored to single emails. `user` field groups emails by inbox. | **QA pairs are single-doc** — no multi-doc reasoning chains in the published distribution. |
| QMSum | Query → meeting alignment; query → utterance-span alignment. | Strong intra-meeting; **no cross-meeting** links. |
| ELITR | Multiple minutes per meeting (multi-annotator) + minute↔transcript alignment. | Strong intra-meeting + multi-minute disagreement. No cross-meeting links. |
| S2ORC | **Citation graph** between papers (bibliographic references resolved to `paper_id`). | Strongest native cross-doc graph in the catalogue. |
| OpenAlex | Citation + author + institution + venue + concept graph. | Strongest metadata-only graph; CC0. |
| PMC OA | Section-level references resolved within PMC. | Strong within PMC; weak outside. |
| CNN/DailyMail | None. | None. |
| Landes to-dos | None. | None. |
| Daily-logs | Hyperlinks within Markdown; @-tags. | Per-author only. |

### 2.2 What's missing — the EMAIL → MEETING → TODO → NOTE chain

No public dataset carries the cross-modal chain the user described.
This chain must be **synthesized by the corpus-build pipeline** by
weaving together pieces from the catalogue. Concrete generator
policy (downstream work, sketched here):

1. **Anchor on Enron threads.** Pick an Enron thread (via
   `Message-ID` / `References` headers) that mentions a specific
   topic. The user-mailbox isolation from EnronQA already gives us
   the "persona" partition_key.
2. **Generate a meeting that follows the email.** Either pick a
   QMSum meeting whose query topic matches the email's subject (LLM
   classifier in the generator), or synthesize a meeting transcript
   from QMSum-style templates seeded with entities pulled out of
   the Enron thread.
3. **Derive to-dos from the meeting minutes.** ELITR-style minutes
   already separate "decisions" and "action items." For each
   action, emit a Landes-format to-do row tagged with the meeting
   ID and the assignee (pulled from Enron `From:`/`To:`).
4. **Generate a follow-up note** in bahmutov-daily-log style: a
   Markdown daily log entry dated after the meeting that
   references the to-do and (in a known fraction of cases)
   **contradicts** the original decision — this is the ground
   truth for "later note changes the decision."
5. **Optionally attach a news / journal article** from CNN/DM or
   S2ORC as background-knowledge context, linked via a synthetic
   `mentions` relation.

Ground-truth labels emitted by the generator: a relation table
(`source_doc_id`, `target_doc_id`, `relation_type ∈ {replies_to,
follows_up_on, summarizes, action_from, contradicts, mentions,
cites}`, `evidence_span`). The downstream RED test asserts that
the top-K retrieval surfaces the full chain for a defined
fraction of multi-hop queries.

Generator effort estimate: **multi-slice** work, almost certainly
out-of-0.7.0 (see §6). 0.7.0's contribution is the corpus catalogue

- schema, not the synthesis tooling.

## 3. Three corpus versions

Refined blend, validated against the catalogue. **Bytes-on-disk**
estimates assume raw text (no embeddings); **dim-768 f32 vector
volume** assumes one embedding per ~500-token chunk at ~4 chunks
per average document (calibrate on real ingestion); **bit-quant
volume** is 96 bytes/row vs 3072 bytes/row f32. Seed cost
extrapolates from AC-013's measured 562 s for 1 M rows at
dim 768 f32 = **~0.56 ms/row** writer-side; real-doc chunking
adds tokenization + embedder cost, dominant over the writer
insert.

### Version A — local dev corpus (~2,200 docs)

| Category | Source | Docs |
|---|---|---|
| News | CNN/DailyMail (Apache-2.0) | 500 |
| Academic | PMC OA — Commercial-Use bucket | 500 |
| Emails | Enron / EnronQA subset | 500 |
| Meetings | QMSum | 200 (utterance windows from 232 meetings) |
| Notes | Synthetic (daily-log style) | 300 |
| To-dos | Landes seed + synthetic | 200 |

- Bytes on disk: ~50–150 MB raw text.
- Chunks ≈ 9,000; f32 vector volume ≈ 27 MB; bit-quant ≈ 0.9 MB.
- Seed cost: < 30 s ingestion (chunking + embedding dominates).
- **Use:** quick iteration, smoke tests, recall-floor sanity
  checks, developer laptops.

### Version B — realistic personal archive (~10,000 docs)

| Category | Source | Docs |
|---|---|---|
| News | CNN/DailyMail | 2,500 |
| Academic | PMC OA Commercial + S2ORC sample | 2,500 |
| Emails | Enron + EnronQA | 2,000 |
| Meetings | QMSum + ELITR (dev-only if NC-SA blocks repo) | 1,000 |
| Notes | Synthetic + daily-log seeds | 1,500 |
| To-dos | Synthetic from Landes templates | 500 |

- Bytes on disk: ~250 MB – 1 GB raw text.
- Chunks ≈ 40,000; f32 vector volume ≈ 120 MB;
  bit-quant ≈ 3.8 MB.
- Seed cost: ~few minutes on canonical CI; dominated by
  embedder I/O.
- **Use:** Pack 2 recall-floor RED tests, metadata-prefilter
  correctness tests, partition_key exercise.

### Version C — stress corpus (~120,000 docs)

| Category | Source | Docs |
|---|---|---|
| News | CNN/DailyMail (most of the train split) | 25,000 |
| Academic | PMC OA + S2ORC | 25,000 |
| Emails | Enron full + EnronQA | 50,000 |
| Meetings | QMSum (full) + ELITR + synthetic | 5,000 |
| Notes | Synthetic | 10,000 |
| To-dos | Synthetic | 5,000 |

- Bytes on disk: ~3–10 GB raw text.
- Chunks ≈ 500,000; **f32 vector volume ≈ 1.5 GB**;
  **bit-quant ≈ 48 MB**.
- Seed cost: extrapolate from AC-013 — at the 500 K-row scale, the
  writer-side seed is ~280 s + embedder cost. With realistic
  embedder I/O (let's call it 5 ms/chunk amortised) the
  end-to-end ingest is ~40 minutes on the canonical 4-core CI
  runner — too long for in-CI seeding. **Pre-build offline,
  cache as an artifact, restore from cache.**
- **Use:** 100k+ stress / partition-fan-out tests / Pack 2
  large-scale latency sanity. Not Version A's daily driver.

## 4. Schema fit with the Pack 1 vec0 migration

Pack 1 (`dev/notes/0.7.0-vector-cost-research.md` §"Pack 1") adds
metadata columns + partition_key to the vec0 schema. sqlite-vec's
hard limits: **16 metadata columns, 4 partition keys**.

The corpus-research schema proposed by the user maps as follows.
**Application-level table (`_fathomdb_documents` etc.):** carries
the full document-level metadata (no limit). **vec0 schema
(`vector_default`):** carries the *filter-relevant* subset — what
queries need to push down into KNN prefiltering.

Proposed vec0 metadata column allocation (well under the 16-col
cap):

| vec0 column | Source | Reason in vec0 (vs app-level) |
|---|---|---|
| `kind` | doc kind | Cited in `vector-cost-research.md` Pack 1 |
| `source_type` | catalogue category | Push-down filter for per-modality search |
| `doc_id` (rowid linkage) | document key | Join key back to app table |
| `created_at` (epoch int) | document timestamp | Push-down range filter |
| `partition_key` (chunk-organised) | see below | Storage layout |

**partition_key recommendation: `source_type`** (one of
`email`, `meeting`, `news`, `academic`, `note`, `todo`). This is
the highest-cardinality / highest-filter-selectivity dimension that
queries actually use (per the FathomDB UX: "search my notes" /
"search my emails" / "search everything"). It maps directly to
catalogue category, has cardinality 6 — small enough that
chunk-organised storage doesn't fragment, large enough that
partition pruning cuts ~6× off the scan for kind-restricted
queries. The current Pack 1 plan tentatively proposed `kind`; the
recommendation here is to **promote `source_type` to partition_key
and demote `kind` to a metadata column**, on the basis that
`source_type` is the dimension the human-facing search verbs
actually filter on.

Of the four partition keys available, only one is consumed; the
remaining three are reserved for future use (likely `user_id` for
the multi-tenant story and `created_year` for time-bucketing).

Chunk-level columns (`chunk_id`, `chunk_index`, `start_char`,
`end_char`, `section_heading`, `embedding_model`,
`embedding_dim`, `created_at`) live in `_fathomdb_vector_rows` —
the existing FathomDB sidecar — not in vec0. The vec0 table only
carries the embedding + the pushdown-filter metadata + the
rowid join. This keeps the 16-column budget healthy.

Relationship table (`source_doc_id`, `target_doc_id`,
`relation_type`, `confidence`, `evidence_span`) is purely
application-side; no vec0 interaction. This is what the §2
cross-document evaluation hits.

**Schema is a clean fit. No vec0 limits exceeded; partition_key
choice (`source_type`) aligns with both the catalogue and the
query model.**

## 6. Drafter's recommendation

### Default version for Pack 2 RED tests

**Version B (10,000 docs).** Version A is too small to discriminate
recall@10 differences below the 5-percentage-point band (the gap
between 90 % and 95 % recall on 200 queries is one query — noise
dominates). Version C is too expensive to seed in canonical CI
(40+ minutes). Version B sits in the sweet spot: ~40 K chunks gives
≥ 100-query held-out sets per partition with statistical power,
seeding completes in single-digit minutes, and the
realistic-personal-archive size is the right shape for the
"recall floor that translates to production workloads" guarantee
that Pack 2 needs to make.

### v1 source inclusion

**Include unconditionally (license-clean):** CNN/DailyMail (news),
S2ORC sampled subset (academic), OpenAlex (metadata enrichment),
Landes to-dos (to-do seeds), bahmutov daily-logs (note style
seeds), synthetic notes / to-dos / cross-doc chains.

**Include behind a download-at-test-time gate (license-blocked
from commit):** Enron + EnronQA (emails), QMSum (meetings).

**Defer to v2:** ELITR (NC-SA license + lower marginal value vs
QMSum), PMC-NC bucket (license-blocked), NOW Corpus (access-blocked),
CC-News (worth tracking).

**Primary fallback if Enron becomes unavailable:** AESLC
(Annotated Enron Subject Line Corpus) — smaller, derived from
Enron, often distributed with a clearer license.
**Primary fallback if S2ORC API access becomes restricted:** swap
to OpenAlex-works full-text where licensed, or arXiv bulk dump
(<https://info.arxiv.org/help/bulk_data/index.html>) which is
CC-BY / CC0 per submitter declaration.

### Effort to build the corpus-ingestion harness

**Multi-slice, almost certainly out-of-0.7.0.** The harness needs
to: (a) pull each source through its allowed channel, (b) normalize
into the FathomDB document schema, (c) chunk + embed, (d) write the
relationship-graph ground truth, (e) cache the corpus as a CI
artifact. Rough sizing: single slice for license-clean static sources
(CNN/DM + S2ORC + Landes + bahmutov) and the schema-normalization
layer; a second slice for Enron/EnronQA download + de-personalize;
a third for QMSum integration; a fourth (the big one) for the
**synthetic cross-document chain generator** described in §2. The
first slice is plausibly co-shippable with 0.7.0 Pack 2 if scope
permits; the rest belongs to 0.7.1 or 0.8.0.

The minimum viable Pack 2 RED test does not strictly need the full
harness — it needs **one** real-data slice (recommend
CNN/DailyMail) plus the existing synthetic AC-013 fixture. The
contention then is: does that satisfy "recall floor on
production-shaped data"? Drafter's read: yes for the gate
(latency + same-modal recall); no for the long-term evaluation
story.

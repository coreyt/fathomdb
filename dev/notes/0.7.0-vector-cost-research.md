# Vector-retrieval cost research (AC-013 / AC-019)

## Problem restated

Canonical CI under the locked W4.1-stacked-O1 (PCACHE2) stack measures
AC-013 at **p50 = 2048 ms, p99 = 2327 ms** for 1 M rows × dim 768 f32,
seeded by `VaryingEmbedder` (`tests/perf_gates.rs:296`). ADR-draft
budgets are 80 / 300 ms — measurement is **25–40× over**.

Root cause is a per-query O(N) full scan through `vec0`. The hot SQL
is in `read_search_in_tx` at
`src/rust/crates/fathomdb-engine/src/lib.rs:2317-2323`:

```
SELECT rowid FROM vector_default
WHERE embedding MATCH vec_f32(?1)
ORDER BY distance
LIMIT 10
```

`vector_default` is declared at
`src/rust/crates/fathomdb-engine/src/lib.rs:3278-3283` as

```
CREATE VIRTUAL TABLE IF NOT EXISTS vector_default
USING vec0(embedding float[<dim>])
```

i.e. a single global vec0 table, no metadata columns, no partition
keys, full-precision f32. `register_sqlite_vec_extension()` at
`src/rust/crates/fathomdb-engine/src/lib.rs:3107-3117` registers the
extension via `sqlite3_auto_extension`. The pinned upstream is
**`sqlite-vec = "=0.1.7"`** (`crates/fathomdb-engine/Cargo.toml:18`)
with `libsqlite3-sys` shipped by `rusqlite 0.31 + bundled`
(`Cargo.toml:14`).

The single-writer + projection-cursor freshness contract is owned by
the writer loop (`src/rust/crates/fathomdb-engine/src/lib.rs:2174,
2242, 2846`). Any vector-encoding change must round-trip through
`PreparedWrite::Node` and `_fathomdb_vector_rows`, not bypass it.

## TIER 1 — scan optimizations on existing vec0

Per-query work at 1 M × 768 × 4 B = 3 GB scanned; 2048 ms ⇒ ~1.5 GB/s
effective bandwidth (close to single-thread memory-read ceiling). Any
technique that **shrinks the bytes-per-row** or **shrinks the row
count** translates ~linearly to latency.

### 1. Metadata prefiltering

- **Mechanism.** `vec0` (sqlite-vec ≥ 0.1.6) supports auxiliary
  metadata columns and partition-key columns; KNN can be pre-filtered
  with `WHERE kind = ?` so distance compute runs only over the
  matching chunk(s). Cited:
  [alexgarcia.xyz blog — metadata release](https://alexgarcia.xyz/blog/2024/sqlite-vec-metadata-release/index.html);
  vec0 features page lists up to 16 metadata columns and 4 partition
  keys.
- **Expected gain.** Scales with selectivity. If FathomDB carries 8
  vector kinds with rough uniformity, prefiltering by `kind` cuts
  scanned rows ~8× ⇒ ~256 ms. AC-013's fixture seeds a *single*
  `"doc"` kind (`tests/perf_gates.rs:350`), so the benchmark itself
  gets **no benefit** from `kind` prefiltering. Real workloads do.
- **Implementation cost.** Schema migration (add metadata + partition
  cols to `vector_default` or split into per-kind tables);
  `engine.search()` API extension to accept filter predicates;
  ingest path change in `_fathomdb_vector_rows` insert
  (`lib.rs:2846`). Moderate.
- **Recall / correctness.** Exact within the filter set — semantics
  change (results restricted), not approximated.
- **Compatibility.** Plays cleanly with writer lock + projection
  cursor; vec0 partition keys are chunk-organised so writes scale
  per partition.

**Verdict on its own: insufficient for AC-013's synthetic
single-kind fixture; valuable for real workloads but not load-bearing
for the gate.**

### 2. Partition keys (physical / logical)

- **Mechanism.** Partition-key columns on `vec0` (sqlite-vec
  feature; max 4) physically chunk storage. Logical alternative:
  N per-kind `vec0` tables.
- **Gain.** Same scaling as metadata prefilter; for AC-013's
  single-kind fixture, zero. For domain workloads with k=8 partitions
  → ~8× cheaper scans (~250 ms).
- **Cost.** Either a multi-table search planner (logical) or schema
  migration (partition_key). Logical option additionally creates a
  fan-out merge problem at the search layer.
- **Recall.** Exact within partition set.
- **Compatibility.** Per-table flavour is incompatible with the
  current single `vector_default` table assumption baked into
  `read_search_in_tx`; partition_key flavour is a simpler change.

**Verdict: same as #1 — orthogonal to the gate, but worth shipping
together if we do a vec0 migration anyway.**

### 3. Smaller embeddings (lower dim)

- **Mechanism.** dim 768 → 384 (MiniLM-L6) halves distance compute;
  → 256 cuts 3×; → 128 cuts 6×.
- **Gain (alone).** 384 ⇒ ~1024 ms; 128 ⇒ ~340 ms. Neither closes
  the 25–40× gap by itself.
- **Cost.** This is **the embedder identity**, not a vec0 setting.
  `EmbedderIdentity` is profile-pinned via
  `_fathomdb_embedder_profiles` (`lib.rs:3248-3260`); switching
  embedders requires a corpus rebuild and a profile rotation. Not a
  knob we can flip without writer cooperation and a re-embedding
  pass.
- **Recall.** Domain-dependent; MTEB shows ~3–6 pp recall drop from
  768→384 for general-purpose embedders.
- **Compatibility.** Touches the embedder contract and the schema
  pinning logic. **HITL Q1: do we have the appetite to switch the
  reference embedder in 0.7.0?** Default assumption: no.

**Verdict: out of scope for 0.7.0 unless HITL signals otherwise.**

### 4. Binary / int8 quantization (THE LEVER)

- **Mechanism.** Replace the f32 column with `bit[768]` or
  `int8[768]`. sqlite-vec supports both natively in `vec0`
  (`bit[N]`, `int8[N]` declarations; cited:
  [sqlite-vec API reference / DeepWiki](https://deepwiki.com/asg017/sqlite-vec/3-api-reference);
  [binary quantization guide](https://alexgarcia.xyz/sqlite-vec/guides/binary-quant.html)).
  Binary distance becomes Hamming (popcount xor) on 96 bytes/row
  vs 3072 bytes/row of f32 — **32× narrower**, plus 4× lighter
  arithmetic. Int8 is 4× narrower with SIMD-able dot products.
- **Gain.** Binary: 32× storage × ~1× arithmetic-density advantage
  ⇒ 30–50× in practice on memory-bound scans. Published benches
  (Kyle Howells, Cohere) show 15–40× faster than f32 brute force at
  ~85–95 % recall@10 with **rerank-on-f32**. Expected p50 if we
  binarize: **~50–100 ms** for 1 M rows. Int8 alone: 4–8× ⇒
  ~250–500 ms (borderline on the p99 = 300 ms budget).
- **Cost.** Schema migration (add `embedding_bin bit[768]` column
  or a sibling `vector_default_bin` table); ingest path adds a
  `vec_quantize_binary()` call alongside the f32 insert; query path
  becomes a two-phase: **bit-distance prefilter → f32 rerank** of
  the top ~256 candidates. The f32 column is retained for rerank
  (so storage grows 1.03×, not shrinks). All changes are SQL-level
  and application-side; no SQLite build flags. sqlite-vec ≥ 0.1.7
  (our pinned version) already ships the relevant types per
  [release notes](https://github.com/asg017/sqlite-vec/releases) —
  no upgrade required to land bit/int8, though upgrading to 0.1.9
  is low-risk and unlocks the constraint-column KNN refinements.
- **Recall.** Binary alone: ~70–80 % @ k=10. Binary + f32 rerank
  on top-K (K = 4–8 × k): ~95–99 % @ k=10, equivalent to brute
  force for most practical purposes. **HITL Q2: what's the recall
  floor we'll enforce in the AC-013 RED test?** Suggested: ≥ 90 %
  recall@10 vs f32 ground truth on the AC-013 fixture.
- **Compatibility.** Single-writer lock and projection cursor are
  not affected — both columns commit in the same writer
  transaction. `VaryingEmbedder` produces normalised f32; binary
  quantization preserves the sign-mass pattern it encodes, so the
  fixture remains discriminative.

**Verdict: binary + f32 rerank is the single highest-leverage Tier-1
technique. Plausibly closes the gate alone.**

### 5. Matryoshka / truncated dimensions

- **Mechanism.** Run KNN against a 128-d prefix, rerank on full 768.
- **Gain.** ~6× on the prefilter pass ⇒ ~340 ms. Less leverage than
  binary; binary already wins on bandwidth.
- **Cost.** Requires a Matryoshka-trained embedder (`nomic-embed
  v1.5`, `text-embedding-3-*`). VaryingEmbedder is not Matryoshka;
  the AC-013 fixture wouldn't exhibit the recall property.
  sqlite-vec exposes `vec_slice()` but no documented native first-K
  dim KNN.
- **Recall.** Embedder-dependent.
- **Compatibility.** Same embedder-identity coupling as #3.

**Verdict: dominated by binary quantization for our setting. Skip.**

### 6. Caching / candidate limiting

- **Mechanism.** Result cache (no help on AC-013's uniform random
  queries — held-out distribution per `ac013_query_bodies`,
  `tests/perf_gates.rs:373`). `LIMIT 10` is already applied
  (`lib.rs:2322`) but `vec0`'s brute-force scan still visits all
  rows to compute the top-10 ordering. No early-exit on a
  brute-force scan.
- **Gain.** Zero on AC-013.
- **Cost.** N/A.

**Verdict: not load-bearing.**

### Tier 1 combined estimate

Stacking the realistic non-embedder-switching levers — **binary
quantization + f32 rerank** (#4) as the primary, with **metadata
prefilter / partition_key** (#1 + #2) as an architectural neighbour
that ships for free in the same migration — gives:

- **p50 expectation: 50–100 ms** (within the 80 ms p50 budget on
  the optimistic side, ~10 ms over on the pessimistic).
- **p99 expectation: 100–200 ms** (well under the 300 ms p99
  budget).
- **Recall: ≥ 95 % @ k=10** with K-rerank = 64, depending on
  workload.

**This plausibly closes the 25–40× gap.** Tier 2 (vectorlite) and
Tier 3 (broader ANN) become unnecessary for 0.7.0.

The one residual concern: the AC-013 fixture's `VaryingEmbedder`
concentrates mass on 6 coordinates out of 768
(`tests/perf_gates.rs:317`), which is *atypically* friendly to
binary quantization (sign pattern dominates). Real embedders will
show somewhat less favourable ratios. The fixture itself should
still GREEN; HITL should note that AC-013's headroom may be
overstated relative to production workloads.

## TIER 2 — vectorlite (HNSW SQLite extension)

Investigated for completeness. **Skip in 0.7.0** if Tier 1 closes
the gap.

- **Maintenance.** `1yefuwang1/vectorlite`, Apache-2.0, latest
  release **v0.2.0 (2024-08-19)** per the GitHub repo
  ([vectorlite](https://github.com/1yefuwang1/vectorlite)). No
  releases in the ~21 months since. 361 stars / 10 forks. Maturity:
  research-grade.
- **API.** Loadable extension, virtual table (`USING vectorlite(...)`).
  Would slot into `register_sqlite_vec_extension()` as a sibling
  registration (different entry point).
- **Coexistence with sqlite-vec.** No conflict — different virtual
  table names. But a vectorlite table is a separate corpus; we'd
  either dual-write (storage 2×) or migrate.
- **HNSW params.** `M` default 16, `ef_construction` 200, `ef`
  query-time default 10. Sane defaults; would need tuning for dim 768.
- **Benchmarks.** Author's published benches stop at 3000 vectors
  (recall 36–99 % depending on `ef`; latency 13 µs – 1.4 ms). **No
  1 M × 768 benchmark.** Extrapolation: HNSW is log-time, so the
  step from 3 K → 1 M is ~2× per `log N` factor; plausibly 50–500 µs
  per query, well under budget. But unbenchmarked at our scale.
- **Storage.** HNSW typical 1.5–2× overhead; at 3 GB raw, +1.5–3 GB.
  Acceptable but non-trivial for embedded-DB posture.
- **Single-writer / lifecycle.** Vectorlite stores its HNSW graph
  inside the SQLite file via the virtual table's shadow tables —
  no external files. Compatible with `database_lock.rs`. Build is
  synchronous on insert (no async rebuild) — fits the writer
  transactional model.
- **Snapshot / projection cursor.** Inserts commit synchronously
  with the writer txn; AC-017 cursor freshness unaffected.
- **Fallback.** No documented opt-out; would need to retain
  `vec0` brute-force in parallel for recall validation.
- **libsqlite3-sys 0.28 compatibility.** Not specifically tested
  upstream; vectorlite requires SQLite ≥ 3.38 for metadata
  filtering. rusqlite 0.31 / bundled ships SQLite ≥ 3.44 → OK.

**Verdict: shippable with caveats (stale maintenance + unbenchmarked
at our scale + architectural-lever cost), but unnecessary if Tier 1
closes the gap. If Tier 1 falls short, vectorlite is a credible
Tier 2 lever.**

Slice scope if pursued: 1 slice (extension registration + schema
migration + dual-corpus rebuild). Adds an architectural lever on
top of PCACHE2 in 0.7.0 — **HITL "one architectural lever per
release" rule applies; this would require an exception.**

## TIER 3 — broader ANN architecture

Pursued only as a sanity check; Tier 1 wins.

- **sqlite-vec upstream ANN.** v0.1.10-alpha.1 introduced
  experimental ANN indexes (`rescore`, `ivf` not-enabled,
  `DiskANN`) per
  [GitHub releases](https://github.com/asg017/sqlite-vec/releases).
  Status: alpha, not production. **Not 0.7.0-eligible.** Worth
  tracking for 0.8.0 / 0.9.0.
- **Algorithm choice.** For our dim 768 / 1 M / read-heavy /
  embedded-single-process workload, HNSW is the obvious default;
  IVF wins at much larger N; PQ wins under tight memory budgets.
  Binary quantization (Tier 1 #4) *is* a scalar-quantization ANN
  technique applied at brute-force scan time, which is why it
  works without a graph index.
- **Integration shapes** (A. inside vec0 future, B. vectorlite-
  style external vtab, C. Rust-side `usearch` / `instant-distance`,
  D. hybrid). C is the most interesting alternative if Tiers 1+2
  both fail: `usearch` is actively maintained, has Rust bindings,
  and supports binary + matryoshka natively. Would require
  rebuilding the FathomDB search verb to leave SQLite for the KNN
  step — large blast radius.
- **Recall validation.** RED-test would express a recall floor
  against f32 brute-force ground truth on a held-out query set;
  bound: `recall@10 ≥ 0.90` (HITL to confirm).
- **Slice scope.** Multi-phase, almost certainly slips to 0.7.1.

**Verdict: do not pursue in 0.7.0.**

## Drafter's recommendation — locked plan

**Tier 1 wins. Tiers 2 and 3 are not pursued in 0.7.0.**

Ship in **two packs** within 0.7.0. Both packs live under the
existing 0.7.0 release line; the split exists for review
granularity, not for release-boundary reasons.

### Pack 1 — schema migration + ingest double-write

Lands the new schema and the new ingest path. No reader behavior
change yet; queries still go through the existing f32 path.

- **Add `bit[768]` column** (or sibling `vector_default_bin`
  vec0 table) alongside the current f32 `embedding` column.
  sqlite-vec 0.1.7 already supports `bit[N]` natively
  ([release notes](https://github.com/asg017/sqlite-vec/releases))
  — no upstream upgrade required.
- **Add metadata columns + partition_key** to the vec0 schema
  (Tier 1 #1 + #2). HITL-confirmed to ship in 0.7.0. Doesn't
  help the AC-013 gate (single-kind fixture) but is the correct
  shape for real workloads and avoids a second migration in
  0.7.1. Partition_key candidate: vector kind from
  `_fathomdb_vector_rows` (chunk-organised storage). Metadata
  columns: at minimum a `kind` column; broader metadata schema
  per the corpus / API work tracked separately.
- **Ingest path change** in the writer (`lib.rs:2846` insert
  site for `_fathomdb_vector_rows`): on every write, also
  compute `vec_quantize_binary(embedding)` and insert into the
  bit column. This is the "double-write" — both columns commit
  in the same writer transaction, so the writer-lock and
  projection-cursor freshness contract are unchanged.
- **Schema migration** for existing data: rebuild the bit
  column from the existing f32 corpus at upgrade time (one-pass
  `INSERT ... SELECT vec_quantize_binary(embedding) FROM
  vector_default`). Migration cost: ~3 GB read + ~96 MB write
  for a 1 M row corpus; well under a minute on disk.
- **Tests:** Pack 1 adds no new RED gate; existing AC-013
  remains pinned at the unindexed 2048/2327 numbers until
  Pack 2 lands. Pack 1's correctness gate is the migration
  round-trip test (insert f32, read back bit column,
  `vec_distance_hamming` matches the expected popcount).

### Pack 2 — query-path swap + RED tests

Switches the reader to the two-phase shape and pins the new
budgets.

- **`read_search_in_tx` query rewrite** at `lib.rs:2317-2323`:
  two-phase query, bit-distance KNN top-K (K=64) → f32 rerank
  for final top-10. K = 64 picked from binary-quant literature
  (Kyle Howells, Cohere) to land recall@10 ≥ 95 % at ~50–100 ms;
  may be tunable.
- **AC-013 latency budget re-pin** in `tests/perf_gates.rs:149-150`:
  p50 ≤ 80 ms, p99 ≤ 300 ms (per ADR-0.7.0 draft). RED test
  must fail at canonical-CI scale before Pack 2 code change.
- **New AC recall-floor RED test** asserting recall@10 ≥ 0.90
  vs f32 brute-force ground truth on the AC-013 fixture
  (HITL-locked). Ground truth computed at test-setup time
  via a brute-force pass; cached as expected top-10 IDs per
  query. The brute-force pass uses the existing f32 column
  retained for rerank — this is one of the column's two
  jobs.
- **AC-019 verdict** under the new query path: re-measure;
  bound expression (`10× baseline`) stays.

### Architectural-lever accounting

The binary-quant + rerank query shape is a **data-encoding
change** (schema + query rewrite), not an architectural lever
in the sense of the `feedback_reliability_principles` rule
(PCACHE2 was the architectural lever — replaces SQLite's page
cache). The HITL doc records this disposition; if a future
reviewer disagrees and counts it as a second lever, the fallback
is to ship Pack 1 in 0.7.0 and Pack 2 in 0.7.1. Drafter's
recommendation: ship both in 0.7.0.

### Out of scope for both packs

- Embedder change (HITL confirmed: keep existing embedder).
- ANN graph index (vectorlite, sqlite-vec ANN alpha, usearch /
  instant-distance / Rust-side index). Tracked in
  `dev/notes/pcache2-followups.md` for post-0.7.0 if real-workload
  telemetry shows the binary-quant scan is insufficient.

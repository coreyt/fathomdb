## Items

### 1. Edge property filter in traversal

**Status**: Not started  
**Breaking**: No

The `edges` table has a `properties BLOB NOT NULL` column (written via `EdgeInsert.properties: String`). The traversal CTE joins on `edges` but discards edge properties — they are never filtered on or returned.

**Work:**

- Add edge property filter predicates to the traversal `JOIN edges` condition. Two new `Predicate` AST variants:
  - `EdgePropertyEq { path: String, value: ScalarValue }` — SQL: `AND json_extract(e.properties, ?{p}) = ?{v}`
  - `EdgePropertyCompare { path: String, op: ComparisonOp, value: ScalarValue }` — SQL: `AND json_extract(e.properties, ?{p}) {op} ?{v}`
  - Both equality and comparison are required; equality alone is insufficient for Cypher `WHERE r.prop > $v`.
  - Naming follows the existing `JsonPathEq` / `JsonPathCompare` pattern on nodes.
- Expose edge `properties` in expansion results. Add an `EdgeRow` result type (or `edge_properties: String` field on expansion hits) so callers can read edge properties, not just filter on them.
- Extend the public `QueryBuilder` / `SearchBuilder` API to accept edge property predicates on `.expand()` calls.
- Parity across Python and TypeScript bindings.

**Cypher enablement note:** These variants are required for `WHERE r.prop = $v` and `WHERE r.prop > $v` in the 0.6.0 Cypher layer. The `EdgeRow` result type is required for `RETURN r` / `RETURN r.prop`. Without both, the Cypher translator cannot emit edge-scoped WHERE predicates or return edge data.

**Deferred (not in 0.5.1):**

- Expression indexes on `json_extract(properties, '$.key')` — defer until a specific high-cardinality query pattern is observed in practice. The existing `(source_logical_id, kind, superseded_at)` index narrows traversal candidates before any JSON extraction runs.

### 3. `filter_json_fused_text_in` — set membership fused filter

**Status**: Not started  
**Breaking**: No

`filter_json_fused_text_eq` accepts a single value. Filtering on a set (e.g. 10 relationship strings in Memex's `_DEPENDENCY_RELATIONSHIPS`) requires either N queries merged client-side or a fused IN variant. Neither is acceptable; N queries is strictly worse than the current over-fetch workaround.

**Work:**

- New AST variant (fused): `JsonPathFusedIn { path: String, values: Vec<String> }`.
  - SQL: `AND json_extract(src.properties, ?{path_idx}) IN (?, ?, ...)` — bind path once, then bind each value. Placeholder list is dynamic; generated at compile time from `values.len()`.
  - Builder method `filter_json_fused_text_in_unchecked(path, values)` following the same tethered/checked pattern as the existing fused eq methods.
  - Compile site: `coordinator.rs` fused clause builder (search CTE path) and `compile_expansion_filter` (expand path), mirroring the existing `JsonPathFusedEq` handling at both sites.
  - Same `BuilderValidationError::MissingPropertyFtsSchema` guard as existing fused methods.
  - Parity: Python `filter_json_fused_text_in(path, values)`, TypeScript `filterJsonFusedTextIn(path, values)`.
- New AST variant (unfused): `JsonPathIn { path: String, values: Vec<ScalarValue> }`.
  - Required for the Nodes driving table where no FTS schema exists (e.g. Cypher `WHERE n.prop IN [...]`). The fused variant requires an FTS registration; the unfused variant works against bare `nodes` scans.
  - SQL: same `IN (?, ?, ...)` pattern, applied in `base_candidates` WHERE for the Nodes driver and in the outer WHERE for FTS/Vec drivers (residual, same classification as `JsonPathEq`).
  - Values accept `ScalarValue::Text`, `ScalarValue::Integer`, `ScalarValue::Bool` — mixed-type IN lists are a caller error and rejected at compile time.
  - Builder method `filter_json_text_in(path, values: Vec<&str>)` on `QueryBuilder` (no FTS-schema gate, no `_unchecked` suffix).
  - Parity: Python `filter_json_text_in(path, values)`, TypeScript `filterJsonTextIn(path, values)`.

**Empty `values` behavior**: raise `BuilderValidationError` / `CompileError` at filter-add time for both variants — an empty IN list is always a bug and SQLite's `IN ()` is a syntax error.

**Cypher enablement note:** `JsonPathFusedIn` alone is insufficient for Cypher `WHERE n.prop IN [...]`; the Cypher translator uses the Nodes driver for lookup-by-property queries where no FTS schema exists. `JsonPathIn` (unfused) is the required variant for that path.

### 4. `filter_json_fused_bool_eq` — boolean fused filter

**Status**: Not started  
**Breaking**: No

`filter_json_bool_eq` exists as a post-filter but has no fused variant. The fused surface is missing the boolean case entirely. Concretely needed for answer feedback patterns (`resolved=false`) as usage grows.

**Work:**

- New AST variant: `JsonPathFusedBoolEq { path: String, value: bool }`.
- SQL: `AND json_extract(src.properties, ?{path_idx}) = ?{value_idx}` where value binds as integer `1` or `0` — matching what the post-filter `filter_json_bool_eq` already does and how SQLite JSON stores booleans.
- Builder method `filter_json_fused_bool_eq_unchecked(path, value: bool)`.
- Same compile sites and validation guards as item 3.
- Parity: Python `filter_json_fused_bool_eq(path, value)`, TypeScript `filterJsonFusedBoolEq(path, value)`.

### 6. `StellaEmbedder` TypeScript: remove unverified default baseUrl

**Status**: Live bug

`StellaEmbedder` in the TypeScript package hardcodes `baseUrl` to `https://api.stella.ai/v1` but `stella_en_400M_v5` has no known public hosted API at that address. Any caller constructing `StellaEmbedder` without an explicit `baseUrl` will get silent network failures.

**Work:**

- Update documentation and constructor JSDoc.

## Architecture hygiene work (added 2026-04-18)

### ARCH-005: `configureFts` workflow divergence

**Status**: Pending
**Breaking**: Yes (minor — removes no-op `mode` parameter from Python)

TypeScript `configureFts` auto-re-registers the FTS property schema when the tokenizer
changes: it fetches existing schema via `describeFtsPropertySchema`, then calls
`registerFtsPropertySchemaWithEntries` with the new resolved tokenizer. Python
`configure_fts` does not — callers must re-register manually or the old schema stays
active under the new tokenizer, causing silent indexing errors.

**Work:**

- Python `_admin.py::configure_fts`:
  - After `set_fts_profile` succeeds, call `describe_fts_property_schema(kind)`.
  - If schema exists for that kind, extract its paths and call
    `register_fts_property_schema_with_entries(kind, paths, tokenizer=resolved)`.
  - If no schema exists, raise `ValueError` — explicit over silent.
  - Remove the `mode` parameter (currently accepted but silently ignored). 0.x semver permits.
- TypeScript `admin.ts::configureFts`: docstring note that schema re-registration is automatic.
- Test (red → green): register FTS schema → call `configure_fts` with different tokenizer →
  assert the existing schema paths remain indexed under the new tokenizer.

**Files**: `python/fathomdb/_admin.py`, `python/tests/` (new test), `typescript/packages/fathomdb/src/admin.ts` (doc only).

## Cypher enablement gate

Items 1–4 collectively unblock the Cypher compatibility layer planned for 0.6.0.
The table below tracks which Cypher constructs each item unlocks:

| Cypher construct | Requires |
|---|---|
| `WHERE r.prop = $v` | Item 1 (`EdgePropertyEq` AST variant) |
| `WHERE r.prop > $v` (and `>=`, `<`, `<=`) | Item 1 (`EdgePropertyCompare` AST variant) |
| `WHERE r.active = true` | Item 1 (`EdgePropertyEq` with `Bool` value) |
| `RETURN r` / `RETURN r.prop` | Item 1 (`EdgeRow` result type) |
| `WHERE n.prop IN ['a', 'b']` | Item 3 (`JsonPathIn` unfused + `JsonPathFusedIn`) |
| `WHERE n.active = false` | Item 4 (`JsonPathFusedBoolEq`) |

All six Cypher constructs listed here are blocked until their corresponding 0.5.1 item ships. The
Cypher parser and translator (0.6.0) can be developed in parallel with 0.5.1, but edge predicate
and IN support cannot be wired end-to-end until 0.5.1 is released.

See `dev/pathway-to-basic-cypher-2026-04-17.md` for the full Cypher implementation plan.

## Motivation

**Edge properties (items 1–2):** Memex writes `WMProvenanceLink` as both a node and an edge, querying the node at read time because no edge property filter exists. With edge property filtering, the relationship type can be stored on the edge and queried directly, eliminating the redundant node.

**Fused filter completeness (items 3–4):** Memex filters `WMProvenanceLink` on a 10-element relationship set; `filter_json_fused_text_eq` only handles single values, forcing a 10x over-fetch workaround. `filter_json_fused_bool_eq` is a gap that grows as answer feedback usage grows. Both are additive changes with no breaking surface.

**Chunk matched_paths (item 5):** Explicit codebase placeholder; the correct behavior is documented and the data is available.

**StellaEmbedder baseUrl (item 6):** Silent network failure for any TypeScript caller using the default.

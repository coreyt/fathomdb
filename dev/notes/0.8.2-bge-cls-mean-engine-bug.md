# Bug: CandleBgeEmbedder defaults to Mean pooling for a CLS-pooled model

**Flagged:** 0.8.2 Slice 5 fix-2 (codex §9 finding [P2])
**Severity:** Product defect — shipped dense retrieval is degraded vs the model's intended mode
**Status:** Flagged; NOT fixed here (fix touches stored-vector compatibility — separate slice)
**File:** `src/rust/crates/fathomdb-embedder/src/candle_bge.rs`

---

## The bug

`CandleBgeEmbedder::new()` initialises with `pooling: Pooling::Mean`:

```rust
// candle_bge.rs line ~229
Ok(Self { identity, tokenizer, model, device, pooling: Pooling::Mean })
```

`bge-small-en-v1.5` was trained with CLS pooling (`1_Pooling/config.json` →
`pooling_mode_cls_token: true`).  BGE's docs explicitly warn that mean-pooling
causes **"a significant decrease in performance"** for this model family.

The `Pooling::Cls` variant already exists in the codebase and the Rust
`with_pooling(Pooling::Cls)` builder is available, but `new()` does not use it.

## Why the harness is CORRECT

The M1 strong-baseline harness (`eval/m1_baseline.py`) uses **CLS pooling**
for all dense-derived arms — it extracts `x[0]` (the `[CLS]` token at sequence
position 0) and L2-normalises.  The ground-truth for bge-small-en-v1.5 pooling
is therefore the **harness**, not the engine default.

Verification: `src/rust/crates/fathomdb-embedder/src/candle_bge.rs` lines 59–71
document the `Pooling` enum and confirm `bge-small-en-v1.5` ships with
`pooling_mode_cls_token: true`.

## Why the fix is NOT done here

The `Pooling::Mean` default was established in design §0.4 ("mean-pool over
attention mask, then L2-normalize") and the pinned vector corpus was built with
it.  Flipping the default from `Mean` → `Cls` would change the embedding space
geometry of every stored vector, breaking the ANN recall guarantee (AC-075)
without a re-embed pass.

A correct fix requires:

1. A re-embed pass over all stored vectors in the wild.
2. An embedder-identity-change migration (the ADR workflow in
   `dev/adr/ADR-0.8.0-embedder-identity-change-workflow.md`).
3. A recall-floor gate re-measurement (eu7 re-run).

This is non-trivial production work.  It is **not** in scope for 0.8.2 Slice 5.

## Impact on THE BAR (Slice 5)

The absolute F1/recall numbers in the M1 pilot are in the harness's **CLS
space** (which is the correct space for this model).  They are internally
consistent (all four arms use the same CLS embedder, so the comparisons are
valid) but they are **not directly comparable** to the engine's production
retrieval (which is in Mean space).

The relative arm ordering (fused ≈ fused_rerank > passage_dense > bm25) is
valid regardless of which pooling space is used, because the comparison is
within-space.

## Slice-15 prerequisite

Slice 15's PPR seeding must embed using the same CLS space as the harness
(see docstring in `eval/m1_baseline.py`).  If Slice 15 calls the engine's live
embedder without setting `with_pooling(Pooling::Cls)`, the PPR arm will be in
Mean space and the recall comparison will be invalid.

## Recommended fix (separate slice)

- Change `new()` to default `pooling: Pooling::Cls`.
- Bump the embedder identity (so stored Mean-vectors are not re-used).
- Run the re-embed workflow and re-measure eu7.
- Gate behind a recall-floor re-confirmation.

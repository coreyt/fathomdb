# Context Research — Technical Documentation

## Scope
"Technical documentation context" here means the curated, structured artifacts a project hands to an AI coding agent (Claude Code, Codex, Cursor, Aider, Devin, Cline, etc.) to ground its reasoning: requirements documents, architecture overviews, design docs, ADRs, interface specifications (OpenAPI / type stubs / protocol schemas), and the agent-specific instruction files (`CLAUDE.md`, `AGENTS.md`, `.cursor/rules/`, `CONVENTIONS.md`) that point at or summarize them. The question is *how* to provision these so the agent reasons more reliably without paying excessive token cost or inheriting hallucinations from stale or contradictory text.

## Findings

### F1 — Long architecture/design dumps degrade reasoning even when retrieval is perfect
**Evidence:** "Even when models can perfectly retrieve all the evidence—in the strictest possible sense, reciting all tokens with 100% exact match—its performance still degrades substantially as input length increases." — [arXiv 2510.05381](https://arxiv.org/html/2510.05381v1). Confirmed by the original "Lost in the Middle" finding: "performance is often highest when relevant information occurs at the beginning or end of the input context, and significantly degrades when models must access relevant information in the middle of long contexts, even for explicitly long-context models." — [Liu et al., arXiv 2307.03172](https://arxiv.org/abs/2307.03172). Anthropic frames this as "context rot: as the number of tokens in the context window increases, the model's ability to accurately recall information from that context decreases." — [Anthropic Engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents).
**Observations:** Pasting a full architecture document or requirements doc into the prompt is *empirically* worse than a tightly distilled summary, even ignoring token cost. Reported degradation reaches 24–59% on reasoning benchmarks at 7.5k–30k tokens, and 50% on HumanEval at 30k masked tokens — i.e. the loss is intrinsic to length, not noise.
**Recommendations:** Cap upfront tech-doc context per session (the Anthropic-community heuristic of <300 lines for CLAUDE.md is consistent with this). Keep canonical long docs on disk; let the agent retrieve subsections via grep/read.
**Impact on agent LLM:** HIGH — large doc dumps are a measurable accuracy regressor, not just a cost concern.

### F2 — Just-in-time retrieval beats upfront dumping for tech docs
**Evidence:** "Rather than pre-processing all relevant data up front, agents built with the 'just in time' approach maintain lightweight identifiers (file paths, stored queries, web links, etc.) and use these references to dynamically load data into context at runtime using tools." — [Anthropic Engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents). Cursor echoes the same: "You don't need to manually tag every file in your prompt… Including irrelevant files can confuse the agent about what's important." — [Cursor blog](https://cursor.com/blog/agent-best-practices). And HumanLayer: "Prefer pointers to copies." — [HumanLayer](https://www.humanlayer.dev/blog/writing-a-good-claude-md).
**Observations:** The frontier consensus across Anthropic, Cursor, and the Claude Code community is to treat tech docs as a *file system* the agent navigates, not as prompt payload. This converges with Aider's repo-map approach, which sends only "the most relevant portions of the repo map" using a graph-ranking algorithm and a 1k-token default budget — [Aider](https://aider.chat/docs/repomap.html).
**Recommendations:** Keep design docs / ADRs / requirements as committed files. From CLAUDE.md / AGENTS.md, list *titles + one-line descriptions + paths*. Let the agent read on demand.
**Impact on agent LLM:** HIGH — the same tokens cost ~0 when unused and full price when read on demand only for the right sub-task; this is a strict Pareto improvement.

### F3 — Spec-driven flows materially reduce hallucination, but only with disciplined specs
**Evidence:** "Specifications don't serve code—code serves specifications… Specifications must be precise, complete, and unambiguous enough to generate working systems." — [github/spec-kit](https://github.com/github/spec-kit/blob/main/spec-driven.md). And: "When creating this spec from a user prompt: Mark all ambiguities: Use [NEEDS CLARIFICATION: specific question]." Practitioners corroborate: "Aim for a clear spec covering just enough nuance… without overwhelming it… As you pile on more instructions or data into the prompt, the model's performance in adhering to each one drops significantly." — [Addy Osmani](https://addyosmani.com/blog/good-spec/).
**Observations:** Spec-driven approaches help, but two failure modes recur: (a) implementation detail leaking into spec ("Focus on WHAT users need and WHY" while avoiding "HOW to implement" — spec-kit), and (b) over-stuffed specs that exceed the model's "attention budget." The Stage-1 registered report on spec-driven generation (arXiv 2601.03878) is *not yet* empirical, so the strongest empirical signal in this area is currently from spec-quality interventions in adjacent domains (see F5 on OpenAPI).
**Recommendations:** Treat the spec/requirements doc as the agent's primary upfront context, but explicitly partition: high-level intent + acceptance criteria upfront; detailed mechanism in linked `implementation-details/` files. Mark uncertainty; do not let the agent silently resolve it.
**Impact on agent LLM:** HIGH — spec-first flows align directly with how planner/executor architectures decompose work; they also create a stable artifact for re-planning across sessions.

### F4 — ADRs are unusually high-density, low-cost context for agents
**Evidence:** "ADRs provide structured and rich context that allows AI agents to: Understand Past Decisions, Suggest Consistent Solutions, Identify Conflicts, and Generate Documentation." — [Lutz, ADR practical guide](https://micheltlutz.me/en/articles/architecture/adr/adr-guia-iniciantes-ia-generativa). Practitioner guidance: "an agent that knows 'we chose event sourcing over CRUD for the orders domain because of audit requirements' generates code consistent with that decision" — surfaced in the [packmind context-engineering best-practices roundup](https://packmind.com/context-engineering-ai-coding/context-engineering-best-practices/).
**Observations:** ADRs are short (typically <1 page), self-contained, and explicitly encode the *reason* for a constraint — exactly the information that would otherwise be hallucinated as a plausible-but-wrong default (e.g. defaulting to CRUD when event sourcing is mandated). They compress well: an agent only needs the relevant ADR for the file/module it's editing, not the whole set.
**Recommendations:** Keep ADRs as individual files (`docs/adr/ADR-NNNN-*.md`). Reference them by ID in code comments and in `CLAUDE.md` index. Prefer many small ADRs over a single architecture.md.
**Impact on agent LLM:** HIGH — per-token, ADRs encode constraint+rationale better than any other doc form, and they are naturally retrievable by topic.

### F5 — Interface contracts (OpenAPI, type stubs) are the strongest single anti-hallucination lever
**Evidence:** "OpenAPI specifications serve as the single source of truth for function definitions, ensuring AI agents construct valid API calls without hallucinating parameters or endpoints." — [OpenAPI agents guide (Xano)](https://www.xano.com/blog/openapi-specification-the-definitive-guide/). Empirically: "Manual failure analysis revealed five recurring issues, all attributable to inconsistencies or omissions in the OpenAPI contracts. After minor fixes, averaging just 19 lines of spec changes per API, AutoMCP achieved 99.9% success." — [AutoMCP, arXiv 2507.16044](https://arxiv.org/html/2507.16044v1). The general claim is corroborated by smart-contract work showing "LLMs' propensity for hallucination can be tamed by setting a hard frame around it through mechanical model synthesis" — [arXiv 2501.12972](https://arxiv.org/html/2501.12972).
**Observations:** Of all tech-doc formats, machine-checkable interface specs (OpenAPI, JSON Schema, Protobuf, type stubs) provide the highest reasoning-gain-per-token. They (a) constrain the output surface, (b) are mechanically validatable, and (c) are short. AutoMCP's result — 99.9% with 19-line fixes — is the cleanest empirical demonstration that *spec quality* dominates the rest of the pipeline.
**Recommendations:** Keep API/protocol specs canonical and current; surface them to the agent as upfront context (or as MCP tool schemas). Invest in spec hygiene over prose docs when forced to choose.
**Impact on agent LLM:** HIGH — type/schema constraints both reduce hallucination and shorten tool-use round-trips, dominating prose docs on a tokens-vs-correctness curve.

### F6 — Stale or conflicting docs are net-negative; staleness, not fabrication, is the dominant failure mode
**Evidence:** "The most common form of documentation-driven AI errors is not fabrication—it is staleness." — [BSWEN](https://docs.bswen.com/blog/2026-03-17-prevent-ai-hallucinations-outdated-docs/). And: "An AI agent confidently installed a deprecated authentication method… The agent was coding against documentation from a version that no longer existed." (same source). The Anthropic ecosystem warns: "Documentation can contain malicious instructions for agents." — [The Register](https://www.theregister.com/2026/03/25/ai_agents_supply_chain_attack_context_hub).
**Observations:** Outdated docs are worse than no docs because the agent treats them as authoritative and produces high-confidence wrong code. Conflicting docs (e.g. two ADRs whose later one supersedes the earlier without marking it) cause silent inconsistency. The HumanLayer rule "Don't include code snippets in these files if possible — they will become out-of-date quickly" (— [HumanLayer](https://www.humanlayer.dev/blog/writing-a-good-claude-md)) is the practical corollary.
**Recommendations:** Date-stamp ADRs and design docs; mark superseded ADRs explicitly in their header; avoid embedding code snippets in long-lived prose docs; prefer references to the actual source files; gate `CLAUDE.md`/`AGENTS.md` changes through PR review.
**Impact on agent LLM:** HIGH — stale prose creates confident wrong code; the failure is hard to detect because the output looks idiomatic.

### F7 — Agentic search (grep + read) generally outperforms vector RAG for code/docs in a known repo
**Evidence:** Boris Cherny (Claude Code) reports agentic search "outperformed everything. By a lot," and Anthropic replaced their traditional RAG pipeline with agentic search for Claude Code — summarized in [Heubanks](https://robertheubanks.substack.com/p/anthropic-replaced-their-rag-pipeline) and [Morph: Agentic Search](https://www.morphllm.com/agentic-search): "Agentic search is an AI search paradigm where a model autonomously plans and executes multi-step searches using tools — grep, file read, directory listing." For external/multi-repo retrieval, hybrid retrieval still wins: "Combining Contextual Embeddings and Contextual BM25 reduced the top-20-chunk retrieval failure rate by 49%… Reranked Contextual Embedding and Contextual BM25 reduced the top-20-chunk retrieval failure rate by 67%" — [Anthropic Contextual Retrieval](https://www.anthropic.com/news/contextual-retrieval).
**Observations:** Inside a single repo with a coherent file tree, grep/glob/read is competitive with or better than vector search and avoids index-staleness. Across many repos or large external doc corpora, hybrid (BM25 + contextual embeddings + reranker) is the empirically strongest setup. The repo-survey [arXiv 2510.04905](https://arxiv.org/html/2510.04905v1) notes "as repository size grows or structural complexity increases, RAG demonstrates clear advantages," and graph retrieval "is particularly suited for tasks involving global consistency or cross-file reasoning."
**Recommendations:** For project-internal tech docs: rely on agentic grep/read; do not invest in a vector store. For external doc bases (vendor SDKs, multi-repo orgs): use hybrid BM25+embeddings+rerank.
**Impact on agent LLM:** MEDIUM — the choice changes infrastructure cost and latency more than per-call reasoning quality, but a wrong choice (e.g. vector RAG over a 50-file repo) wastes effort and adds staleness risk.

### F8 — Instruction files (CLAUDE.md / AGENTS.md / .cursor/rules) should be small, pointer-heavy, and gradually grown
**Evidence:** "< 300 lines is best, and shorter is even better… our root CLAUDE.md file is less than sixty lines." — [HumanLayer](https://www.humanlayer.dev/blog/writing-a-good-claude-md). "Keep rules focused on the essentials: the commands to run, the patterns to follow, and pointers to canonical examples in your codebase. Reference files instead of copying their contents; this keeps rules short and prevents them from becoming stale as code changes." — [Cursor Rules docs / blog](https://cursor.com/blog/agent-best-practices). Martin Fowler / Subramaniam: "build context like rules files up gradually, and not pump too much stuff in there right from the start… There is a tendency to overengineer the context with unnecessary, copied & pasted instructions up front." — [Martin Fowler](https://martinfowler.com/articles/exploring-gen-ai/context-engineering-coding-agents.html).
**Observations:** All three frontier-tool ecosystems (Anthropic, Cursor, Aider) converge on: small upfront file, pointers to deeper docs, version-controlled, evolved when the agent makes a repeatable mistake. Cursor explicitly excludes "Copying entire style guides (use a linter instead)," "Documenting every possible command," and "instructions for edge cases that rarely apply" — [Cursor blog](https://cursor.com/blog/agent-best-practices).
**Recommendations:** Treat the instruction file as an *index plus invariants*, not a manual. Each entry should be either (a) a non-inferable invariant ("never write to `migrations/` directly"), or (b) a one-line pointer to a canonical file. Add an entry only after observing a failure that the entry would have prevented.
**Impact on agent LLM:** MEDIUM — the file is read on every call, so each line has high cumulative cost; the *delta* effect on any single call is moderate but compounds.

### F9 — Planner/executor flows (Devin, Plan-and-Act, Claude Code todo lists) externalize tech-doc state
**Evidence:** "Devin operates on a compound AI system architecture… including a Planner (a high-reasoning model that outlines the strategy), a Coder… a Critic… and a Browser." — Devin 2.0 / [Endo summary](https://medium.com/@takafumi.endo/agent-native-development-a-deep-dive-into-devin-2-0s-technical-design-3451587d23c0). Cognition: "A good Playbook includes: The outcome we want Devin to achieve; The steps required to get there; Specifications describing postconditions; Advice to correct Devin's priors." — [Cognition](https://cognition.ai/blog/how-cognition-uses-devin-to-build-devin). Claude Code mirrors this with todo lists and per-context-window summaries: "use the first context window to set up a framework… then use future context windows to iterate on a todo-list… Prefer starting afresh over compacting." — [Clune.org](https://clune.org/posts/anthropic-context-engineering/).
**Observations:** Effective long-horizon agents externalize tech-doc-derived plans into *durable artifacts* (playbooks, todo files, plan markdown) that survive context-window resets. This means the design-doc and the working plan are distinct: the doc describes the intended end state; the plan is regenerated each session from the doc plus current code state.
**Recommendations:** When kicking off agent work, generate a session-scoped plan file from the design doc rather than re-feeding the entire design doc. Persist that plan; let the agent edit it across context windows. Re-derive from the design doc when the plan goes stale, not when the agent is stuck.
**Impact on agent LLM:** MEDIUM — this is more about session architecture than per-prompt reasoning, but it largely determines whether design docs scale beyond a single window.

### F10 — "Oracle" (curated) summaries beat both raw docs and autonomous summarization
**Evidence:** From SWE-bench-style context-architecture work: "Only 'oracle' summary provision reliably boosts both accuracy and efficiency. Autonomous or mismatched retrieval (free search or summary) is often detrimental—misleading or irrelevant memories dilute the agent's reasoning. Summarized trajectories are more effective than full ones, confirming that information bottlenecking is essential for reliable knowledge reuse." — surfaced in the [SWE-bench summary roundup](https://www.vals.ai/benchmarks/swebench) and consistent with the agent-context findings catalogued in [arXiv 2509.16941 (SWE-Bench Pro)](https://arxiv.org/pdf/2509.16941).
**Observations:** Human-curated, task-relevant summaries dominate both raw docs and agent-generated auto-summaries. Auto-summarization tends to drop the load-bearing constraints (the same constraints ADRs are good at preserving). This is the empirical backing for the spec-kit / Addy Osmani prescription.
**Recommendations:** Invest one-time human effort in writing tight summaries of large architecture docs. When forced to compress automatically, validate against acceptance tests, not against a "looks good" rubric.
**Impact on agent LLM:** MEDIUM-HIGH — the oracle-vs-auto gap shows up most on long-horizon multi-file tasks; on single-file edits, raw docs are often acceptable.

### F11 — Doc-format and tool-config conventions are converging on AGENTS.md + skill-style manifests
**Evidence:** "Skills written in [SKILL.md] format work across Claude Code, Codex, Gemini CLI, and other tools adopting the standard." — [agensi.io comparison](https://www.agensi.io/learn/claude-code-skills-vs-cursor-rules-vs-codex-skills). Multi-tool teams are advised: "Your team uses both Cursor and Claude Code. Maintain .cursorrules for Cursor users and CLAUDE.md for Claude Code users… You are an OSS project. Ship AGENTS.md for broad compatibility and CLAUDE.md for Claude-specific optimization." — [thepromptshelf](https://thepromptshelf.dev/blog/cursorrules-vs-claude-md/).
**Observations:** Cross-tool portability matters because teams increasingly use multiple agents (Claude Code for terminal, Cursor for IDE, Codex for headless CI tasks). Vendor-opinion right now favors AGENTS.md as the lowest-common-denominator file with tool-specific overrides.
**Recommendations:** For projects targeting multiple agents, primary file = `AGENTS.md`, with a thin `CLAUDE.md` / `.cursor/rules/` that delegates ("see AGENTS.md") plus tool-specific tweaks. Keep all of them well under the size thresholds in F8.
**Impact on agent LLM:** LOW-MEDIUM — convention choice has minimal direct accuracy impact but materially affects whether the same tech-doc context reaches the agent the team is actually using.

### F12 — Provide verifiable goals (tests, types, lints) alongside docs
**Evidence:** "Provide verifiable goals—agents can't fix what they don't know about. Use typed languages, configure linters, and write tests to give the agent clear signals for whether changes are correct." — [Cursor blog](https://cursor.com/blog/agent-best-practices). Spec-kit reinforces: "Production metrics and incidents don't just trigger hotfixes—they update specifications for the next regeneration." — [github/spec-kit](https://github.com/github/spec-kit/blob/main/spec-driven.md).
**Observations:** Tech-doc context becomes far more useful when paired with mechanical verification. Tests turn prose requirements into a feedback loop the agent can drive itself with; type stubs turn architecture rules into compile-time guarantees. Devin's "tight feedback loop where as soon as tests fail or lints complain, Devin can iterate autonomously" — [Cognition](https://cognition.ai/blog/how-cognition-uses-devin-to-build-devin) — is the canonical example.
**Recommendations:** For every requirement that an agent will work against, ensure there is at least one mechanical check (test, type, lint, schema validation). Reference those checks from the design doc. The doc says *what*; the check enforces it.
**Impact on agent LLM:** HIGH — the empirical evidence (Devin, Cursor, AutoMCP) is consistent: agents with verifiable feedback close the gap between doc-implied behavior and actual behavior far better than agents working from prose alone.

## Synthesis
The cross-cutting takeaway for tech-docs context is that **the goal is the smallest set of high-signal, mechanically-grounded artifacts the agent can navigate on demand**, not a comprehensive doc dump. The frontier-lab and practitioner consensus is remarkably tight: keep the upfront agent-instruction file small (<300 lines, ideally <60), index design docs and ADRs by path, prefer interface contracts (OpenAPI, types, schemas) over prose for anything machine-checkable, retrieve via grep/agentic search inside the repo and via hybrid BM25+embeddings+rerank across external corpora, and pair every requirement with a verifiable check. The dominant *risks* are not under-provisioning context but (a) length-induced reasoning degradation that hits even with perfect retrieval, (b) staleness that produces confident wrong code, and (c) auto-summaries that drop load-bearing constraints. Spec-driven and planner/executor flows mitigate these by externalizing intent into stable, human-curated artifacts and re-deriving session-scoped plans from them. Empirical anchors — AutoMCP's 99.9% success after 19-line spec fixes, Anthropic's 49–67% retrieval-failure-rate reductions, the 24–59% accuracy drops at 30k tokens — all point in the same direction: invest in spec quality and retrieval discipline before investing in more context.

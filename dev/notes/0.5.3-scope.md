## Items

### 1. scale.rs race: surface actionable assertions instead of thread panics

**Status**: Confirmed (deferred from 0.5.1 → previously 0.5.2-Item-1)
**Breaking**: No
**Design**: `dev/notes/design-0.5.3-scale-race.md`

`crates/fathomdb/tests/scale.rs:1494` calls
`handle.join().expect("heavy writer joins")` on spawned heavy-writer
threads. Under WAL lock contention during parallel test runs, a writer
thread can panic; `.expect()` propagates that as the test's failure
without preserving the panic payload or the thread identity. The test
currently flakes under sustained `cargo test --workspace` load and
passes reliably in isolation.

**Work:**

- Replace the raw `.join().expect(...)` with a helper that captures
  `thread::Result<()>`. On `Err(payload)`, downcast `payload` to `&str`
  and `String`, format with the writer index, and emit a `panic!` or
  `assert!` with useful context.
- Add a minimal reproduction test that verifies the helper surfaces the
  panic message (not the flake itself — just the error-surface
  contract).
- If the root-cause WAL contention is tractable, also bound the join
  with a timeout and emit a diagnostic before the main thread panics.

**Reference**: 0.5.1 CHANGELOG "Known issues", investigation transcript
available via the 2026-04-18 session's `scale.rs race investigation`
agent output.

### 2. TypeScript `configureFts` parity: raise on missing FTS schema

**Status**: Confirmed (post-0.5.1 follow-up → previously 0.5.2-Item-2)
**Breaking**: Yes (minor — TypeScript callers)
**Design**: `dev/notes/design-0.5.3-configure-fts-ts-parity.md`

Python `configure_fts` (as of 0.5.1) raises `ValueError` when no FTS
property schema is registered for the target kind. TypeScript
`configureFts` still silently skips re-registration in that case. The
scope decision for 0.5.1 was to accept the divergence and ship; 0.5.3
unifies the contract.

**Work:**

- Update `typescript/packages/fathomdb/src/admin.ts::configureFts`:
  after `setFtsProfile` succeeds, fetch the schema via
  `describeFtsPropertySchema`; if null, throw a typed error matching
  the Python `ValueError` contract (message naming the kind).
- Add a TypeScript test parallel to
  `python/tests/test_profile_management.py::test_configure_fts_raises_when_no_schema_registered`.
- Document the breaking change in the 0.5.3 CHANGELOG Breaking section.

**Reference**: 2026-04-18 ARCH-005 review notes ("Parity Gap" section).

### 3. FTS5 query crashes on unescaped metacharacters (GH #31)

**Status**: Candidate (user-reported bug → previously 0.5.2-Item-3)
**Breaking**: No
**Design**: `dev/notes/design-0.5.3-fts5-metachar-escape.md`

`text_search()` passes user input raw into the FTS5 MATCH expression,
so queries containing `'`, `"`, or other FTS5 metacharacters raise
`SqliteError: fts5: syntax error near "'"`. Hit in production against
Memex's retrieval layer on `User's name`.

**Work:**

- Escape FTS5 metacharacters in the query renderer
  (`render_text_query_fts5` or the SDK-facing wrapper). At minimum:
  double embedded `"` and wrap each whitespace-separated token in
  double quotes.
- Tests: Rust + Python + TypeScript round-trip tests for queries
  containing `'`, `"`, `(`, `)`, `*`, `:`, `-` at start of token.

**Reference**: `gh issue view 31`.

### 9. Per-session `/tmp/fathomdb-*` root for all test harnesses (GH #40)

**Status**: Confirmed (originally deferred from 0.5.2 → previously
0.5.3-Item-3)
**Breaking**: No

Multiple fathomdb test paths currently leak `/tmp/fathomdb-*` files
across runs, eventually producing disk pressure that flakes unrelated
tests. The 0.3.1 short-term fix was a wholesale `rm -rf` in
`scripts/preflight-CI.sh`. The long-term fix is a per-session root
minted fresh at harness start and removed wholesale on exit.

**Work:**

- Rust: `test_session_root() -> PathBuf` helper using
  `tempfile::TempDir::new_in("/tmp")` held for the binary lifetime.
- Python: `session_tmp_root()` pytest fixture with
  `tempfile.TemporaryDirectory(prefix="fathomdb-")`.
- TypeScript: replace
  `typescript/apps/sdk-harness/src/skip.ts::tempDbPath()` with a
  session-rooted helper; mint root once per `npm test` invocation;
  clean up via `process.on('exit', ...)`.
- Go: migrate `go/fathom-integrity/test/e2e/` to `t.TempDir()` or a
  shared package-level helper.
- Migrate all call sites off bare `/tmp/fathomdb-harness-…` paths.
- Remove the `rm -rf /tmp/fathomdb-harness-*` lines in
  `scripts/preflight-CI.sh` once every test path goes through the
  session root.

**Acceptance:**

- After a full test-matrix run, `ls /tmp/fathomdb*` returns nothing
  owned by the completed session.
- `scripts/preflight-CI.sh` no longer needs the wholesale `rm -rf`.

**Reference**: `gh issue view 40`.

### 10. Missing-docs sweep: resolve ~420 public-API doc warnings (GH #34)

**Status**: Confirmed (originally deferred from 0.5.2 → previously
0.5.3-Item-4)
**Breaking**: No

`cargo clippy --workspace --all-targets` currently emits ~420
`missing_docs` warnings across the public API surface. CI suppresses
them via `-A missing-docs`. Over time this mutes signal on genuine
undocumented new APIs; the sweep removes the warning base so future
additions get caught at PR time.

**Work:**

- Add doc comments to every flagged public item in `crates/fathomdb`,
  `crates/fathomdb-engine`, `crates/fathomdb-query`,
  `crates/fathomdb-schema`.
- Remove `-A missing-docs` from CI clippy invocations and from
  `scripts/hooks/pre-push` + `scripts/preflight.sh`.
- Gate: pre-push runs without `-A missing-docs` and still passes.

**Reference**: `gh issue view 34`.

### 11. Property FTS: nested-object search + per-field weighting (GH #36) — maybe

**Status**: Maybe (confirm scope at 0.5.3 release planning →
previously 0.5.3-Item-5)
**Breaking**: Depends on implementation

Two adjacent gaps in the property FTS surface:

- **Nested objects**: JSON paths pointing at nested objects are
  currently treated as a single blob; queries can't address nested
  scalar values independently.
- **Per-field weighting**: BM25 weights on the per-column FTS5 virtual
  table are wired but callers have no way to declare different weights
  per schema path.

Both items could grow into their own sub-release if the schema surface
needs to change. Confirm scope at 0.5.3 planning. If the fix requires a
new `FtsPropertyPathSpec` variant or a schema migration, slip to
0.6.0. If it's a pure query-time renderer fix with no schema impact,
ship in 0.5.3.

**Reference**: `gh issue view 36`.

## Release mechanics

Standard 0.5.x release flow (same as 0.5.2):

- CHANGELOG.md entry with Breaking / Added / Fixed sections (#2 in
  Breaking; rest in Fixed). Call out #10's clippy lint-budget removal
  explicitly.
- Version bump (root `Cargo.toml`, `crates/fathomdb/Cargo.toml`,
  `python/pyproject.toml`,
  `typescript/packages/fathomdb/package.json`).
- Regenerate `Cargo.lock`.
- Run `scripts/preflight.sh --release` — all three release gates must
  pass (clippy tracing/python, nextest tracing).
- Run `scripts/preflight-CI.sh` for the full CI-equivalent set before
  tagging.
- Tag `v0.5.3` on HEAD. Release workflow handles publishing (gated on
  the `all-builds-passed` cross-ecosystem aggregator added in 0.5.1).

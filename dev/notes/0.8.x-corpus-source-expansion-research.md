# 0.8.x Corpus Source Expansion Research

## Executive Recommendation

Add data in three lanes:

1. **QAConv first**: add 1,000-1,500 conversational documents and
   export 3,000-5,000 eval-only QA rows. It is the cleanest
   personal-agent fit because it covers business emails, panel
   discussions, and work channels, with 34,204 QA pairs over 10,259
   conversations under BSD-3-Clause.
2. **Export latent QA from existing corpus sources**: add no new
   documents, but export 3,000-10,000 eval-only QA rows from the
   already-pinned EnronQA and QMSum acquisitions. This is the
   lowest-risk accuracy lift.
3. **Populate `paper` with QASPER plus a small PMC OA subset**:
   add 1,000-1,500 paper docs from QASPER and 500-1,000 PMC OA
   commercial-use full-text docs. QASPER contributes 5,049 paper
   QA/evidence questions over 1,585 NLP papers under CC-BY-4.0;
   PMC OA supplies license-filterable full text and metadata for
   realistic paper retrieval.
4. **Add LongMemEval-S as an eval-only agent-memory pack**: do not
   ingest all sessions into the main corpus yet. Export the 500
   long-term-memory questions and selected conversation sessions as
   a separate accuracy fixture. License is MIT.
5. **Add MeetingBank only if more meeting volume is needed**:
   1,000-1,500 meeting-segment docs with reference summaries are
   useful, but they are civic-public-meeting shaped rather than
   personal-workplace shaped.
6. **Track LongMemEval-V2 and Asta Interaction Dataset for later**:
   both are very relevant to agent memory/research-agent workflows,
   but neither should be first for the current six-source-type
   corpus. LongMemEval-V2 is small but useful as eval-only;
   Asta is large, query/clickstream-only, and best used as a query
   distribution source rather than ingested documents.

Recommended first-pass document growth:

| Slice | Source | Docs | Eval QA / labels | Resulting corpus size |
|---|---:|---:|---:|---:|
| Baseline | Current full corpus | 7,667 | 301 chain queries | 7,667 |
| 1 | QAConv | +1,250 | +3,000-5,000 QA | 8,917 |
| 2 | QASPER | +1,250 | +5,049 QA/evidence | 10,167 |
| 3 | PMC OA commercial-use | +750 | none unless paired with PubMedQA | 10,917 |
| 4 | Existing EnronQA/QMSum QA export | +0 | +3,000-10,000 QA | 10,917 |

This reaches the lower target bound without overfitting to generic
news, without adding a seventh `source_type`, and without increasing
synthetic-doc share.

### 1. QAConv

Source:
[salesforce/QAConv](https://github.com/salesforce/QAConv),
[paper](https://arxiv.org/abs/2105.06912).

Why it fits:

- The dataset is explicitly QA over informative conversations,
  including business emails, panel discussions, and work channels.
- The GitHub README reports 34,204 QA pairs from 10,259 selected
  conversations, including span-based, free-form, and unanswerable
  questions.
- The repo is BSD-3-Clause.

Mapping:

- Business email conversations -> `email`.
- Panel/work-channel conversations -> `meeting` or `note` depending
  on source metadata.
- QA rows -> eval-only artifact referencing conversation doc IDs.

Subsample:

- Sort by source split, conversation ID, then question ID.
- Ingest 1,250 conversations/doc chunks.
- Export all QA rows for selected conversations, capped at 5,000
  eval rows for first pass.

### 2. Existing EnronQA QA export

Source:
[EnronQA paper page](https://huggingface.co/papers/2505.00263),
existing script `tests/corpus/scripts/acquire_enronqa.py`.

Why it fits:

- The upstream benchmark reports 103,638 emails with 528,304 QA
  pairs across 150 user inboxes.
- This is directly personal/private RAG over email, matching the
  Hermes/OpenClaw/Memex workload.
- The current script intentionally defers QA extraction.

Mapping:

- Keep current 200 ingested email docs for now.
- Add `data/corpus-data/eval/enronqa_qa.jsonl` with question,
  gold answers, alternate answers, rationales, user, path, and
  source doc ID when the selected email is in the corpus.

Posture:

- Cache-only until the undeclared HF license is clarified.

### 3. QASPER

Source:
[allenai/qasper](https://huggingface.co/datasets/allenai/qasper),
[paper](https://arxiv.org/abs/2105.03011).

Why it fits:

- It fills the missing `paper` source type.
- Hugging Face metadata lists CC-BY-4.0.
- The paper reports 5,049 information-seeking questions over 1,585
  NLP papers, with supporting evidence from answers.

Mapping:

- One paper document per paper, `source_type = "paper"`.
- Paper sections can become chunks inside `body`; figures/tables
  stay as text metadata for now.
- QA rows become eval-only, with evidence section/sentence
  references mapped to doc IDs.

Subsample:

- Include all train/validation/test papers if the final count stays
  near 1,585; otherwise cap to 1,250 papers sorted by paper ID and
  keep all QA rows for selected papers.

### 4. LongMemEval-S

Source:
[xiaowu0162/longmemeval](https://huggingface.co/datasets/xiaowu0162/longmemeval),
[paper](https://arxiv.org/abs/2410.10813).

Why it fits:

- It is a direct long-term-memory benchmark for chat assistants.
- The HF dataset page lists MIT license and includes
  `longmemeval_s`, `longmemeval_m`, and `longmemeval_oracle`.
- Published descriptions report 500 questions in the small public
  benchmark.

Mapping:

- First pass should be eval-only, not mixed into the main corpus.
- Convert sessions into memory docs only if a later slice wants a
  conversation-shaped `note` fixture.

Risk:

- Recent community audits argue some memory benchmarks can become
  context-window tests. Use it as a comparative eval, not the sole
  accuracy gate.

### 5. PMC OA Commercial-Use Subset

Source:
[PMC Open Access Subset](https://pmc.ncbi.nlm.nih.gov/tools/openftlist/).

Why it fits:

- It supplies real full-text papers and article metadata.
- PMC explicitly separates Commercial Use Allowed licenses
  (`CC0`, `CC BY`, `CC BY-SA`, `CC BY-ND`) from non-commercial and
  other buckets.
- It gives `paper` volume and supports future `cites` chains.

Mapping:

- `source_type = "paper"`.
- Use title, abstract, selected section text, authors, publication
  date, journal, DOI/PMCID, and references.

Constraints:

- Automated retrieval must use PMC Cloud, OAI-PMH, FTP, E-Utilities,
  or BioC API only.
- License is per article, not per dataset; acquisition must filter
  machine-readable license before writing JSONL.

### 6. MeetingBank

Source:
[MeetingBank dataset](https://meetingbank.github.io/dataset/),
[paper](https://arxiv.org/abs/2305.17529).

Why it fits:

- The release includes 1,366 meetings from six U.S. cities.
- It provides transcript segments and reference summaries with
  train/validation/test split sizes 5,169 / 861 / 862.
- Metadata includes agenda item IDs, start/end times, item type,
  URLs, and summaries.

Mapping:

- One agenda segment per `meeting` document, not one full meeting.
- Reference summaries become eval-only summary rows or child docs
  with `relation:summarizes`.

Risk:

- It is public-meeting/civic shaped, less like private personal
  agent memory.
- Review `LICENSE.txt` before marking commit-eligible.

### 7. LongMemEval-V2

Source:
[xiaowu0162/longmemeval-v2](https://huggingface.co/datasets/xiaowu0162/longmemeval-v2),
[paper](https://arxiv.org/abs/2605.12493).

Why it fits:

- Hugging Face metadata lists Apache-2.0.
- The dataset card says it contains 451 manually curated questions
  and 1,870 task trajectories from customized WebArena-style and
  ServiceNow-style environments.
- It evaluates static recall, dynamic state tracking, workflow
  knowledge, environment gotchas, and premise awareness.

Mapping:

- Eval-only first; trajectories can become `note` documents later.
- Screenshots make it partly multimodal, which is out of scope for
  the current text corpus.

### 8. PubMedQA

Source:
[pubmedqa/pubmedqa](https://github.com/pubmedqa/pubmedqa),
[paper](https://arxiv.org/abs/1909.06146).

Why it fits:

- Repo is MIT licensed.
- The task pairs biomedical research questions with PubMed abstract
  context, long answers, and yes/no/maybe labels.
- It can pair naturally with PMC OA or PubMed abstract-style paper
  docs.

Mapping:

- Prefer eval-only over paper abstracts, not main corpus docs,
  unless a matching `paper` document is generated per abstract.

Risk:

- Biomedical QA is less personal-agent-like than QASPER, but useful
  for evidence-hit accuracy.

### 9. S2ORC

Source:
[allenai/s2orc](https://github.com/allenai/s2orc),
[Semantic Scholar Datasets API](https://api.semanticscholar.org/api-docs/datasets).

Why it fits:

- S2ORC is a large scientific-paper corpus with parsed full text and
  citation structures.
- The current README says S2ORC is available through the Semantic
  Scholar Public API bulk datasets and is released under ODC-By 1.0.

Mapping:

- `source_type = "paper"`.
- Best for citation graph and `cites` relation coverage, not QA.

Risk:

- API key and bulk-release machinery make it higher friction than
  QASPER or PMC OA.
- Underlying contents still need care despite ODC-By metadata.

### 10. Asta Interaction Dataset

Source:
[allenai/asta-user-interactions](https://huggingface.co/datasets/allenai/asta-user-interactions),
[Ai2 blog](https://allenai.org/blog/asta-interaction-dataset).

Why it fits:

- It is real usage from AI-powered research tools.
- The dataset card reports anonymized opt-in user queries, thread
  IDs, timestamps, tool names, clicks, shown results, report section
  titles, and cited paper IDs.
- HF metadata lists ODC-By.

Mapping:

- Do not ingest as ordinary docs.
- Use as a query distribution and retrieval-workload generator for
  paper/search-agent experiments.

Risk:

- No gold answer field. Accuracy labels would need click/citation
  proxies.

## QA / Accuracy-Test Plan

Add a new eval-only JSONL shape:

```jsonc
{
  "qa_id": "stable deterministic id",
  "source": "qaconv|enronqa|qmsum|qasper|longmemeval|...",
  "source_type": "email|meeting|paper|article|note|todo",
  "question": "string",
  "answers": ["gold answer strings"],
  "answer_type": "span|free_form|yes_no_maybe|summary|abstain",
  "evidence_doc_ids": ["doc ids in corpus"],
  "evidence_spans": [
    {"doc_id": "id", "start": 0, "end": 120, "text": "optional"}
  ],
  "negative_doc_ids": ["optional hard negatives"],
  "relation_type": "optional relation vocabulary value",
  "metadata": {
    "split": "train|validation|test",
    "user_id": "optional",
    "thread_id": "optional",
    "upstream_id": "optional"
  },
  "license": "SPDX or LicenseRef",
  "provenance": "pinned upstream tag"
}
```

Metrics:

- `recall@k`: does retrieval include at least one evidence doc?
- `MRR`: rank of the first evidence doc.
- `NDCG`: useful when multiple evidence docs are acceptable.
- `exact evidence-hit`: retrieved chunk overlaps evidence span.
- `answer-support hit rate`: generated answer is supported by
  retrieved evidence.
- `abstention accuracy`: for unanswerable QAConv / LongMemEval
  cases, retrieval should not invent evidence.

Source-specific eval handling:

- **QAConv**: export QA rows for selected conversations; evidence
  maps to conversation/chunk IDs.
- **EnronQA**: export QA rows only when email doc ID exists in the
  corpus; keep full upstream QA in cache-only data.
- **QMSum**: export original query-answer pairs even though summary
  docs are already ingested; evidence doc is the meeting transcript.
- **QASPER**: export questions with paper evidence references.
- **LongMemEval**: keep a separate memory-benchmark eval harness
  with session ordering and temporal metadata.
- **PMC OA/S2ORC**: do not create synthetic QA first; use them as
  paper/citation chain anchors unless paired with PubMedQA/QASPER.

## Reject / Defer List

- **SciFact**: scientifically useful, but CC-BY-NC 2.0 makes it
  cache-only and the claim-verification shape is less personal-agent
  realistic than QASPER/PubMedQA.
- **S2ORC**: strong for citations but high-friction; defer until
  QASPER or PMC OA proves the `paper` ingest path.
- **Asta Interaction Dataset**: excellent for real research-agent
  query distribution, but no gold answers and too large to ingest.
- **LoCoMo**: relevant memory benchmark, but hold until license and
  recent audit concerns are resolved; LongMemEval is cleaner first.
- **ELITR**: valuable meeting-minutes data, but NC-SA license keeps
  it cache-only and higher-friction than MeetingBank.
- **Generic SQuAD/Natural Questions/QuAC-only expansion**: good QA
  labels, but too generic and not aligned with personal-agent
  memory unless used as a tiny negative-control baseline.

# Design: Edge Property Filter in Traversal (0.5.1 Items 1–2)

**Release:** 0.5.1  
**Scope item:** Items 1 and 2 from `dev/notes/0.5.1-scope.md`  
**Breaking:** No (additive — new optional params, new fields, new result struct)

---

## Problem

`edges` table has `properties BLOB NOT NULL` (stored as SQLite TEXT). The traversal
CTE joins `edges` but neither filters on edge properties nor returns them.
Memex provenance queries require filtering on the relationship type stored in
`edge.properties`. The Cypher layer (`WHERE r.prop = $v`, `RETURN r`) requires
both filter predicates and edge property access in results.

## Design

### 1. New AST Predicate variants (`ast.rs`)

```rust
/// Equality check on an edge property at the given JSON path.
/// Used in the traversal JOIN condition on `e.properties`.
EdgePropertyEq {
    path: String,
    value: ScalarValue,
}
/// Ordered comparison on an edge property at the given JSON path.
EdgePropertyCompare {
    path: String,
    op: ComparisonOp,
    value: ScalarValue,
}
```

Structurally identical to `JsonPathEq`/`JsonPathCompare`. Distinct variants so
the compiler can statically validate which predicate goes to `compile_expansion_filter`
(node) vs `compile_edge_filter` (edge). Cross-variant usage is an unhandled match
arm — panic at compile time.

### 2. New field on `ExpansionSlot` (`ast.rs`)

```rust
pub struct ExpansionSlot {
    pub slot: String,
    pub direction: TraverseDirection,
    pub label: String,
    pub max_depth: usize,
    pub filter: Option<Predicate>,       // target node filter (existing)
    pub edge_filter: Option<Predicate>,  // edge property filter (new)
}
```

`None` for both fields preserves exact pre-0.5.1 behavior.

### 3. Edge property in result types (`coordinator.rs`)

Add `edge_properties: Option<String>` to `NodeRow`:

```rust
pub struct NodeRow {
    pub row_id: String,
    pub logical_id: String,
    pub kind: String,
    pub properties: String,
    pub content_ref: Option<String>,
    pub last_accessed_at: Option<i64>,
    pub edge_properties: Option<String>,  // new: last-hop edge props, None when not from traversal
}
```

All existing `NodeRow` construction sites in `coordinator.rs` set `edge_properties: None`.
Traversal results set it to `Some(e.properties)` from the JOIN. This is additive — callers
already reading `NodeRow` fields are unaffected; only direct struct construction breaks, and
`NodeRow` is only constructed internally.

### 4. New `compile_edge_filter` function (`coordinator.rs`)

Mirror of `compile_expansion_filter`, but uses `e.properties` alias and handles
`EdgePropertyEq` / `EdgePropertyCompare` variants only:

```rust
fn compile_edge_filter(
    filter: Option<&Predicate>,
    first_param: usize,
) -> (String, Vec<Value>) {
    let Some(predicate) = filter else {
        return (String::new(), vec![]);
    };
    match predicate {
        Predicate::EdgePropertyEq { path, value } => { /* AND json_extract(e.properties, ?p) = ?v */ }
        Predicate::EdgePropertyCompare { path, op, value } => { /* AND json_extract(e.properties, ?p) {op} ?v */ }
        _ => panic!("compile_edge_filter: non-edge predicate passed; use compile_expansion_filter"),
    }
}
```

### 5. Traversal CTE changes (`coordinator.rs` ~line 2040–2122)

**5a. Add `edge_properties` column to the recursive CTE:**

Base case (seed row): `NULL AS edge_properties` (no traversal edge).
Recursive step: `e.properties AS edge_properties` (last-hop edge).

Updated CTE column list: `traversed(root_id, logical_id, depth, visited, emitted, edge_properties)`.

**5b. Inject `{edge_filter_sql}` into the JOIN condition:**

```sql
JOIN edges e ON {join_condition}
    AND e.kind = ?{edge_kind_param}
    AND e.superseded_at IS NULL
    {edge_filter_sql}
```

`edge_filter_sql` is generated by `compile_edge_filter(expansion.edge_filter.as_ref(), ...)`.

**5c. Pass `edge_properties` through `numbered` CTE:**

```sql
numbered AS (
    SELECT t.root_id, n.row_id, n.logical_id, n.kind, n.properties
         , n.content_ref, am.last_accessed_at, t.edge_properties
         , ROW_NUMBER() OVER (PARTITION BY t.root_id ORDER BY n.logical_id) AS rn
    FROM traversed t
    ...
)
```

Final SELECT also includes `edge_properties`.

**5d. Bind order:** root IDs (1..=N), edge kind (N+1), edge filter params (N+2..M),
node filter params (M+1...).

**5e. Row mapper** reads column 7 as `edge_properties: Option<String>` on `NodeRow`.

### 6. Builder API change (`builder.rs`)

Extend `expand()` with `edge_filter: Option<Predicate>` parameter:

```rust
pub fn expand(
    mut self,
    slot: impl Into<String>,
    direction: TraverseDirection,
    label: impl Into<String>,
    max_depth: usize,
    filter: Option<Predicate>,
    edge_filter: Option<Predicate>,
) -> Self {
    self.ast.expansions.push(ExpansionSlot {
        slot: slot.into(),
        direction,
        label: label.into(),
        max_depth,
        filter,
        edge_filter,
    });
    self
}
```

All existing call sites in tests add `None` for `edge_filter`.

Python binding: `expand(..., edge_filter=None)` — optional keyword, non-breaking.  
TypeScript binding: `expand(..., edgeFilter?: Predicate)` — optional, non-breaking.

### 7. Write-side round-trip test (Item 2)

New integration test in `crates/fathomdb/tests/grouped_query_reads.rs` (or a
dedicated `edge_property_surface.rs`):

1. Write an edge with non-empty `properties` (e.g. `{"relationship":"cites"}`).
2. Execute an expansion traversal.
3. Assert `NodeRow.edge_properties == Some("{\"relationship\":\"cites\"}")` for the hit.
4. Execute the same expansion with `edge_filter = Some(EdgePropertyEq { path: "$.relationship", value: ScalarValue::Text("cites".into()) })`.
5. Assert 1 result returned.
6. Execute with a non-matching edge filter. Assert 0 results.

---

## Compile sites for `EdgePropertyEq` / `EdgePropertyCompare`

These variants appear only in `compile_edge_filter`. They must NOT appear in:
- `compile_expansion_filter` (node filter — panics on unhandled arm)
- The fused clause builders in `execute_compiled_vector_search` / `run_search_branch`
  (not fusable — no FTS schema for edge properties)

`fusion::is_fusable` must return `false` for these variants (or they must be absent
from the fused predicate classification logic entirely).

---

## Acceptance criteria

1. `ExpansionSlot.edge_filter` compiles; existing tests pass with `edge_filter: None`.
2. Expansion traversal with `EdgePropertyEq` returns only hits where the edge matches.
3. `NodeRow.edge_properties` carries `Some(json_str)` for traversal hits, `None` for flat query results.
4. Multi-hop traversal (`max_depth > 1`) returns the last-hop edge's properties.
5. Traversal with no edge filter and `max_depth=1` returns `Some("{}")` for edges written with empty properties (not `None`).
6. Python: `expand(slot, dir, label, max_depth, filter=None, edge_filter=None)` works; hit `.edge_properties` is accessible.
7. TypeScript: same surface.
8. Round-trip test passes (Item 2 acceptance above).

---

## Out of scope

- Expression indexes on `json_extract(e.properties, '$.key')`
- FTS or vector search over edge properties
- Multiple edge predicates per expansion slot (single `Option<Predicate>` only)
- Edge property filtering in non-expansion traversal steps (`QueryStep::Traverse`)

---

## Risks

**Recursive CTE column count change.** Adding `edge_properties` to the `traversed`
CTE base case and recursive step requires the column list to be consistent. If the
base case `SELECT` column count doesn't match the recursive `SELECT` count, SQLite
returns an error at prepare time. Test must cover the multi-hop case explicitly.

**Bind parameter offset shift.** Adding `edge_filter` params before `node_filter`
params changes the `?N` indexes for existing node filter params. `first_param` must
be calculated after edge filter params are known. Keep a single running `param_idx`
counter rather than computing offsets separately.

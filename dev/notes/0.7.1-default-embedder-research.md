## TL;DR

- Three candidate embedders evaluated end-to-end on the 7,667-doc fathomdb
  corpus through the production pipeline mock (mean-pool ⇒ L2-norm ⇒ sign-bit
  binary quantize ⇒ top-K Hamming ⇒ float32 cosine re-sort ⇒ top-10).
- At the production canonical **K = 64**:
  - `BAAI/bge-small-en-v1.5` (384d): **recall@10 = 0.793** (95% CI 0.751–0.832)
  - `intfloat/e5-small-v2` (384d): **recall@10 = 0.448** (95% CI 0.391–0.506)
  - `BAAI/bge-base-en-v1.5` (768d): **recall@10 = 0.885** (95% CI 0.852–0.913)
- **No 384d model clears the 0.85 threshold at K = 64.** Per the EU-0 handoff,
  this triggers escalation. The escalation experiments give us three viable
  paths; the recommendation block picks one with explicit caveats and a HITL
  question.
- Mean-centering the binary code (subtract corpus-mean before sign) lifts
  bge-small at K=64 by **+5.0 pts** (paired-bootstrap 95% CI on the diff
  +0.024…+0.078), making it the cheapest fix on the table.

## 1 — Methodology

### 1.1 Corpus

- 7,667 documents loaded from `data/corpus-data/raw/*.jsonl` (8 files):
  cnn_dailymail (2500), enron (2000), synthetic_notes (1200), qmsum (600),
  landes_todos (500), chain_connectives (367), bahmutov_dailylogs (300),
  enronqa (200).
- Doc text fed to embedder = `title + "\n" + body` (or body alone if no title),
  truncated by the tokenizer to `max_length=512`.

### 1.2 Query set construction

The 0.7.0 corpus contains no real queries — `enronqa.jsonl` (despite the name)
is 200 doc-only rows tagged `enronqa-user:*`, no question field. We therefore
**deviate from the HITL plan's "mix enronqa + synthetic" guidance** and
synthesize queries across all sub-corpora.

- N = 100 queries, deterministic seed `0xE0` (224) for the shuffle.
- Per-source quotas: cnn_dailymail 30, enron 20, enronqa 20, synthetic_notes
  15, balance 15 distributed across bahmutov_dailylogs / chain_connectives /
  landes_todos / qmsum proportionally to size (3/3/4/5).
- Each query is a doc's title if usable (≥6 chars and not "untitled"),
  otherwise the lead sentence/first ~140 chars of body at a sentence boundary.
  The synthesized text must not equal the body verbatim.
- Every query carries its synthesis-source `target_doc_idx`. During retrieval
  **the target doc is excluded** from both the ground-truth pool and the
  reranked candidates for that query. This is what makes the task non-trivial:
  the query is often a literal substring of its source doc, so without
  exclusion every model would score ≈1.0 by self-retrieval. With exclusion the
  task becomes "find the nearest *other* documents", which is the right proxy
  for production retrieval.
- Limitation: queries are still synthesised from doc text rather than from
  real user information needs. This biases the eval toward lexical-leaning
  models; an MTEB-style relevance-judged set would be a better signal but
  none exists for this corpus. Documented as such.

### 1.3 Pipeline mock

Mirrors the binary-quant + f32-rerank path that is RED in the engine (Pack 2
`ac-013b`). For each model M:

1. Embed all 7,667 docs with M. Mean-pool over the attention mask
   (`(h*mask).sum / mask.sum`, **not** unconditional mean across pad tokens).
   L2-normalize. Result: `f32 (N, D)` corpus matrix.
2. Sign-bit quantize: `bits[i] = (v[i] >= 0)` → `bool (N, D)` corpus bitmap.
   Packed into uint8 for Hamming.
3. Embed queries with M. **E5 gets the `"query: "` prefix on queries; bge
   models do not.** (Spec did not mandate the `"passage: "` E5 corpus prefix
   and we did not apply it — this is a known mild disadvantage to E5 and is
   noted as a caveat in §3.)
4. Per query: XOR/popcount against the corpus bitmap → top-K by Hamming → fetch
   those K corpus f32 vectors → cosine-rerank → top-10 (excluding
   `target_doc_idx`).
5. **Ground truth**: brute-force f32 cosine over the *same* model's full
   corpus, also excluding `target_doc_idx`, take top-10. This is intentionally
   the same model — we are measuring **quantization-induced recall loss**
   relative to that model's exact retrieval, not absolute retrieval quality.
   A cross-model ground truth would be self-fulfilling for whichever model
   produced it.
6. **Recall@10** per query = `|reranked_top10 ∩ gt_top10| / 10`. Mean and
   bootstrap CI (1000 resamples, percentile method) over 100 queries.

### 1.4 Mean-centering ablation

After the main sweep we identified the strongest 384d model (bge-small at
K=64) and re-ran *that one model* at K=64 with mean-centering: compute the
corpus-mean f32 vector, subtract from each doc and query vector **before**
sign-quantization (re-rank step still uses original L2-normed f32 vectors).
Single-axis ablation only.

### 1.5 Stats

- N = 100 queries throughout. Reported per (model, K): mean recall@10, 95% CI
  (percentile bootstrap, 1000 resamples).
- Pairwise comparisons in §2.3 use **paired** bootstrap on per-query diffs
  (same queries, different model), since the noise is correlated across
  models.

## 3 — Literature cross-check

Anchoring against published numbers (the hugging face `embedding-quantization`
blog, vespa.ai's quantization tradeoffs piece, MTEB leaderboard, and model
cards).

**Published MTEB Retrieval scores** (point estimate, NDCG@10 averaged over 15
retrieval datasets; from the bge-base model card and MTEB leaderboard):

| Model              | MTEB Retrieval (15 datasets) |
|--------------------|------------------------------|
| bge-small-en-v1.5  | 51.68                        |
| bge-base-en-v1.5   | 53.25                        |
| e5-small-v2        | ~49 (paper / leaderboard)    |

So *on absolute retrieval quality* the three are quite close (4-point spread
on MTEB). Our empirical numbers are not directly comparable to this — we are
measuring **recall@10 against same-model f32 ground truth**, i.e. the *binary
quantization retention*, not absolute quality.

**Published binary-quant retention** (HF `embedding-quantization` blog,
`rescore_multiplier=4` ≈ "fetch 4× as many candidates as you return"):

| Model                | retention with binary + f32 rerank |
|----------------------|-------------------------------------|
| mxbai-embed-large-v1 (1024d) | 96.45% |
| nomic-embed-text-v1.5 (768d) | 87.7%  |
| all-MiniLM-L6-v2 (384d)      | 93.79% |
| e5-base-v2 (768d)            | **74.77%** |
| e5-small-v2 (384d, vespa)    | ~87% (0.602 binary vs 0.651 float) |

**Where empirical agrees with literature.** The published retention numbers
say E5 family handles binary quantization noticeably worse than mxbai /
all-MiniLM. Our empirical result — e5-small-v2 collapsing to 0.448 at K=64
while bge-small holds 0.793 on the *same queries and corpus* — is consistent
with that direction. The magnitude of the e5 gap in our run is larger than
the vespa "87%" retention figure suggests; some of that may be that we did
not apply the `"passage: "` E5 corpus prefix (per spec). E5 is known to be
sensitive to that asymmetry and our number is best read as "E5 with
spec-mandated prefix handling".

**Where empirical diverges from literature.** The HF blog reports ~93–96%
retention for small-dim models with `rescore_multiplier=4`. Our K=64 corresponds
to roughly `multiplier=6` (K/top10), yet bge-small retains only ~79% relative
recall. Two plausible drivers: (a) our query set is synthetic and noisier
than MTEB's relevance-judged queries — top-10 sets are less stable so the
intersection metric is harsher; (b) our corpus (7.7k docs) is smaller than the
BEIR corpora the literature reports on, which can make Hamming candidate
ordering less robust. Both increase observed quant loss without changing
the *ordering* of models.

The HF blog also explicitly observed `rescore_multiplier > 4` gave "no
further boost"; we see substantial further gain through K=256. That's an
artifact of (a) and (b) above and the fact that we are measuring recall, not
NDCG. The qualitative ordering (bge-small > e5-small-v2 for binary
quantization, bge-base substantially ahead of either) is robust.

## 4 — Recommendation

**Pick bge-small-en-v1.5 with mean-centering, K=64.** (≤200 words.)

The 0.85 escalation threshold is missed by plain bge-small at K=64 (0.793) by
~6 pts. Mean-centering closes most of the gap (0.843, within 1 pt of the
threshold). Mean-centering is essentially free at runtime — one f32
subtraction per query, one constant corpus-mean vector to ship — and the
ablation gives a tight, statistically-significant +5 pt lift (paired 95% CI
+0.024…+0.078). The 0.90 recall floor at K=64 is not met by any 384d option;
clearing it requires either raising K (bge-small needs K≈140 plain or
K=128 mean-centered — measured 0.907 in §2.2.1) or jumping to bge-base
768d (K=96 → 0.914). 768d
doubles vector storage and roughly doubles embed wall-time (797s vs 288s for
corpus embed in our run), which is a meaningful prod cost given fathomdb's
on-device aspirations. The pragmatic recommendation is therefore:
**default = bge-small + mean-centering at K=64**, and **document a
recall caveat**: the 0.90 floor in `tests/perf_gates.rs` is calibrated for
the isotropic AC-013b fixture, not for real BGE embeddings; until that
fixture is replaced with real embeddings the floor needs a separate
calibration anyway (this is consistent with the existing 0.7.0 STATUS note
on isotropic-vs-real direction). If the team wants the floor strictly
respected today, **fall back to bge-base at K=96** (0.914) and accept the
2× embed cost.

### 5.1 — Empirical picture (consolidated)

The measured data tightens the table — bge-small+mc still clears 0.90 at
K=128, just by less margin than the linear extrapolation suggested:

| K   | bge-small plain | bge-small + mc | mc lift       | bge-base plain |
|-----|-----------------|----------------|---------------|----------------|
|  64 | 0.793           | 0.843          | +5.0 pp       | 0.885          |
|  96 | 0.849           | 0.880          | +3.1 pp       | 0.914          |
| 128 | 0.882           | **0.907**      | +2.5 pp       | 0.928          |
| 256 | 0.933           | 0.945          | +1.2 pp (ns)  | 0.964          |

What changed from the §2.4 extrapolation:

1. **bge-small+mc clears the 0.90 floor at K=128 (0.907), not K≈100.** The
   Pareto-B option still exists, just at a slightly higher K than predicted.
2. **The MC effect is geometric, not additive.** Lift roughly halves every
   2× in K and becomes statistically insignificant by K=256. There is no
   path to ≥0.95 recall@10 on bge-small + mc + binary-quant on this
   query set — it tops out near 0.945.
3. **The "~0.93 at K=128" extrapolation was ~2 pp optimistic.** Methodology
   note for future ablation reads: do not trust linear extrapolation of
   bias-correction techniques across K — they saturate.

### 5.2 — Updated Pareto frontier

| Option                 | Recall@10 | Relative latency | Storage | Headroom above 0.90 |
|------------------------|-----------|------------------|---------|---------------------|
| A: bge-small+mc, K=64  | 0.843     | 1.0×             | 1.0×    | below floor         |
| **B: bge-small+mc, K=128** | **0.907** | ~1.1–1.2×    | 1.0×    | +0.007 (thin)       |
| C: bge-base, K=64      | 0.885     | ~2.5–3×          | 2.0×    | below floor         |
| D: bge-base, K=96      | 0.914     | ~3×              | 2.0×    | +0.014              |
| E: bge-base, K=128     | 0.928     | ~3–4×            | 2.0×    | +0.028              |
| F: bge-base, K=256     | 0.964     | ~4–5×            | 2.0×    | +0.064              |

**B vs D head-to-head** (the two minimum-cost options that clear the floor):
- Recall: 0.907 vs 0.914 — a 0.7 pp gap. CIs overlap (B ≈ ±0.034; D's CI
  0.885–0.938 from §2.1). The recall difference sits below the noise floor
  of this 100-query sample.
- Cost: B is roughly half the latency and half the storage of D.
- Headroom: D gives 2× the cushion above 0.90. If real-corpus retrieval
  shifts results down 1–2 pp from these synthetic-query numbers, B sits
  below the floor while D survives.

### 5.3 — Decision recorded (2026-05-28, revised again 2026-05-29 for K=192)

HITL initially picked **(c) bge-base @ K=96** on the strength of the
floor-headroom argument. After the §5.1 / §5.2 update, HITL revised the
choice to bge-small + mean-centering at K=128 (Option B). After §5.4's
fine-grained K-sweep measurement, **HITL revised K from 128 to 192** —
the smallest K where the lower CI bound clears the 0.90 floor.

**Chosen embedder configuration for 0.7.1:**
- Model: `BAAI/bge-small-en-v1.5` (384d)
- Rerank K: **192**
- Mean-centering: **ON** — one f32 corpus-mean vector stored per workspace
  alongside the index; subtracted at both ingest-quantize and query-quantize
  before the sign step.
- Expected recall@10 on synthetic-query corpus: **0.933** (95% CI 0.912–0.953;
  lower CI bound 0.912 statistically clears the 0.90 floor — see §5.4).

**Accepted risks / follow-ups for EU-1+:**
1. Headroom above the 0.90 floor is **0.033** (point estimate cushion, with
   lower CI 0.912 also above floor). Re-derivation of the
   `AC013B_RECALL_FLOOR` against real bge-small embeddings (instead of the
   isotropic AC-013b fixture it was calibrated on) remains the right thing
   to do before EU-5b's lock-flip; if real-corpus recall lands materially
   below 0.933, the documented fallback is K=256 (mc, recall ≈ 0.945) or
   escalation to bge-base.
2. Mean-centering adds workspace state: a 384-float (1536 byte) corpus-mean
   blob in `_fathomdb_embedder_profiles`. Recomputation policy locked in
   EU-2 design as compute-once-on-first-ingest at `MEAN_VEC_PIN_THRESHOLD = 256`.
3. The MC technique caps at ~0.945 recall@10 on this corpus — K=192 sits at
   0.933, leaving ~0.012 of headroom on the technique itself before
   diminishing returns make further K bumps not worth their rerank cost.

### 5.4 — Fine-grained K-sweep (measured 2026-05-29)

After the initial K∈{32, 64, 96, 128, 256} sweep, HITL asked for direct
measurement between K=128 and K=256 to validate the linear-interpolation
intuition. Script: `dev/research/eu-0/run_k192_check.py` (git-excluded;
reuses the saved bge-small doc/query .npy vectors — no re-embed).

| K   | recall@10 | 95% CI            | cushion above 0.90 | lower-CI clears 0.90? |
|-----|-----------|-------------------|--------------------|------------------------|
| 128 | 0.907     | 0.877 – 0.933     | +0.007             | no (lower 0.877)       |
| 160 | 0.919     | 0.892 – 0.943     | +0.019             | no (lower 0.892)       |
| **192** | **0.933** | **0.912 – 0.953** | **+0.033**     | **yes (lower 0.912)**  |
| 224 | 0.941     | 0.921 – 0.959     | +0.041             | yes                    |
| 256 | 0.945     | 0.926 – 0.961     | +0.045             | yes                    |

Key findings:

1. **K=192 is the smallest K where the lower CI bound clears 0.90**, not
   just the point estimate. K=128 and K=160 fall short on this stronger
   floor-clearance criterion.
2. **Measured K=192 (0.933) is +0.007 pp above the linear interpolation**
   of K=128 (0.907) and K=256 (0.945) — the recall curve has a beneficial
   bend in that region rather than being strictly linear.
3. **K=192 → K=256 buys +0.012 pp** for 33% more rerank work — diminishing
   returns are visible in the data. K=192 is the cost/recall knee.
4. **K=128 → K=192 buys +0.026 pp** for 50% more rerank work — real return
   on the additional candidate budget.

Sync/async note that informed the cost calculus: doc-embed at write time
is **async** (background projection workers, `fathomdb-engine/src/lib.rs:2507`),
so embed wall-time is hidden from user-visible write latency. Query-embed
is **synchronous** on the caller thread (`lib.rs:1469`). With bge-small at
384d the per-query embed is the cheapest of the three candidates — that
alignment with the synchronous query path was a non-trivial input to the
B-vs-D choice.

`SEED = 0xE0` (224 decimal). Used for: Python `random` (query shuffling),
NumPy default RNG (bootstrap resampling), `torch.manual_seed`. Note:
embedding inference is deterministic on CPU with `model.eval()` and no
dropout, so seed only affects query selection and bootstrap CIs.

### 6.7 Self-review checklist

  measuring quantization loss, not cross-model quality.

  trivial substring-match self-retrieval).

- [x] L2-norm applied **after** pooling, **before** sign quantize.
- [x] E5 gets `"query: "` prefix on queries; bge does not. (E5
  `"passage: "` corpus prefix **not** applied — followed spec literally;
  noted as a caveat.)

- [x] Pairwise comparisons use paired bootstrap on per-query diffs (not
  unpaired, since the noise is correlated across models on the same query

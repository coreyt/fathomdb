# Finding ‚Äî CPU embedder `.so` defect blocks all dense/CE eval in this environment

> **Discovered:** 2026-07-02, during the 0.8.12 EXP-COV-1 coverage‚Üíoutcome sufficiency sweep.
> **Severity:** blocks any dense-retrieval / CE-rerank evaluation in the current environment.
> **Ties to:** the 0.8.14 GPU-rerank / `embed_batch_cls` work (branch `0.8.14-gpu-rerank`).

## Symptom

The installed CPU embedder native library is pathologically slow and intermittently stalls:

- A ~700-token document body takes **~13 s** to embed on CPU.
- It intermittently **stalls at ~1500% CPU with zero forward progress** (no completion).
- CE-rerank over long LOCOMO bodies is **~8 s/query**.
- No GPU embed is available: the installed `.so` was **not built with the `embed-cuda` feature**, so
  `FATHOMDB_EMBED_DEVICE=cuda*` cannot accelerate it.

## Impact on the EXP-COV-1 sufficiency read

The sufficiency question EXP-COV-1 must answer is whether closing the edge/relation coverage gap moves a
downstream metric **above the ~0.571 embedder ceiling** ‚Äî which *requires* the dense + CE-rerank stack to
be runnable at corpus scale. The defect made that stack intractable, forcing a documented cascade of
deviations to keep the `$0` downstream pass moving:

1. drop doc-dense vectors ‚Üí 2. drop CE-rerank ‚Üí 3. drop fact-dense ‚Üí
   final held-fixed stack = **FTS + structural graph-arm BFS only**.

Even the degraded stack is slow: **graph-arm BFS ‚âà 2.5‚Äì3 s/query** on the dense relation fact-graph (this
per-query graph latency is itself a real OPP-6 finding worth carrying forward).

**Consequence:** a coverage‚Üíoutcome verdict computed on the degraded FTS+graph config would NOT test the
"above the embedder ceiling" question and must not be anchored on. Per the HITL decision (Option A,
2026-07-02) the sufficiency **verdict is DEFERRED to a GPU-embedder re-run**; only the `$0` downstream pass
re-runs (it reuses the already-paid `$4.79` extraction ‚Äî see the manifest on branch
`0.8.12-expcov1-sweep`).

## What is NOT affected

- The **priced extraction** was unaffected ‚Äî it is a caller-side BYO-LLM pass over the airlock proxy, not
  the CPU embedder; it completed cleanly (272 LOCOMO sessions, `$4.79`, 0 failures).
- The `$0` **coverage census** (Slice 5 / EXP-COV-0) was unaffected ‚Äî it scores extraction recall against
  gold, no embedder needed.
- The consolidation slices (15/20) ‚Äî their tests run on `open_without_embedder_for_test` (FTS path), also
  unaffected.

## Recommended fix / next step

A **GPU-embedder build** (`embed-cuda` feature, on one of the idle RTX 3090s ‚Äî see memory
`embed-completeness-and-gpu-readiness`) makes the dense + CE stack tractable and unblocks:

- the EXP-COV-1 coverage‚Üíoutcome sufficiency verdict (reusing the preserved `$4.79` extraction);
- any future dense/CE eval in this environment.

This pairs naturally with the **0.8.14 GPU-rerank / `embed_batch_cls`** work (branch `0.8.14-gpu-rerank`),
which is the home for the GPU embed/rerank path. Until a GPU embedder is available, treat dense/CE eval in
this environment as blocked and prefer FTS-only or gold-scored (`$0`) reads.

# engine.write batch collapse into a single vec0 row

## Symptom

When `Engine::write(batch)` is called with `batch.len() > 1` and every
node in the batch is vector-indexed (e.g. `kind = "doc"` after
`configure_vector_kind_for_test("doc")`), **only one row lands in
`vector_default`** for the entire batch — regardless of how many
nodes the batch contained.

Empirical reproduction (from Corpus-Pack 4 development):

- Ingest 21 short-body real-corpus docs (bodies ≤600 chars) in a single
  `engine.write(&batch_of_21)` call.
- `engine.drain(30_000)` → returns Ok.
- Direct read: `SELECT count(*) FROM vector_default` → **1**.
- `SELECT count(*) FROM canonical_nodes` → 21 (as expected).
- `engine.search(body_of_doc_K)` returns at most 1 vector result, and
  the result body is not necessarily doc K's body.

## Root cause

`Engine::write_inner` advances the write cursor by the *batch size*
and then passes **the same final cursor value** into every per-row
INSERT inside `commit_batch`. Excerpts from `lib.rs`:

```rust
// lib.rs:~1342 — cursor is the *final* slot after reserving batch.len() positions
let increment = u64::try_from(batch.len()).unwrap_or(u64::MAX);
let cursor = self.next_cursor.load(Ordering::SeqCst).saturating_add(increment);

// lib.rs:~3556 — every row in the batch is inserted with `cursor` (singular)
for (write, plan) in batch.iter().zip(plans) {
    match (write, plan) {
        (PreparedWrite::Node { kind, body, source_id }, WritePlan::Node) => {
            tx.execute(
                "INSERT INTO canonical_nodes(write_cursor, kind, body, source_id)
                 VALUES(?1, ?2, ?3, ?4)",
                params![cursor, kind, body, source_id],
            )?;
            // ...
        }
```

`canonical_nodes` has no PK / UNIQUE on `write_cursor`
(`fathomdb-schema/src/lib.rs:111` — `CREATE TABLE IF NOT EXISTS
canonical_nodes(write_cursor INTEGER NOT NULL, kind TEXT NOT NULL,
body TEXT NOT NULL)`), so all `batch.len()` rows land — they just
share a cursor value.

The projection runtime then picks up that cursor once. In
`commit_projection_outcomes` (`lib.rs:~2840`):

```rust
ProjectionOutcome::Success { cursor, kind, blob } => {
    if terminal_state_for_cursor(&tx, *cursor)?.is_some() {
        continue;
    }
    tx.execute(
        "INSERT OR IGNORE INTO _fathomdb_vector_rows(rowid, kind, write_cursor) VALUES(?1, ?2, ?3)",
        params![cursor, kind, cursor],
    )?;
    tx.execute(
        "INSERT OR IGNORE INTO vector_default(rowid, embedding) VALUES(?1, ?2)",
        params![cursor, blob],
    )?;
    record_projection_terminal(&tx, *cursor, "up_to_date")?;
}
```

`vector_default.rowid` is the SQLite PK. The first `Success` for that
cursor populates vec0; every subsequent success in the same batch
either (a) sees `terminal_state_for_cursor.is_some()` and `continue`s,
or (b) hits the `INSERT OR IGNORE` PK conflict and is silently
dropped. Either way, only **one** body per batch reaches vec0.

## Why this matters

1. **Pack 2 recall@10 RED test (AC-013b)**: `seed_ac013_corpus`
   (`perf_gates.rs:~345`) batches 1024 nodes/call. If the same
   collapse happens, the canonical corpus has ~977 vec0 rows for a 1M
   ingest (1024 calls × 1 vec0 row), not 1M. The recall@10 floor
   becomes a measurement of an undersized index; the gate's pass/fail
   may not represent real recall behavior. **Measurement needed.**
2. **Pack 3 corpus ingest** (`examples/ingest_corpus.rs`): batches at
   `NODE_BATCH = 200`. For the 7,667-doc Pack-1+Pack-2 corpus, vec0
   ends up with ~38 rows instead of ~7,667. The harness reported
   "7,667 nodes written" — accurate at the canonical_nodes layer —
   but the vector index is ~200× sparser than intended.
3. **Pack 4 vector validation (this work)**: the symptom that surfaced
   the bug. Worked around in `tests/corpus_vector.rs` by calling
   `engine.write` with a singleton batch per node; documented at the
   top of that file.

## Suggested resolution

Two options:

- **A: allocate one cursor per row inside the batch.** Change
  `write_inner` so each row gets its own monotonically-increasing
  cursor; `commit_batch` advances cursor per-iteration. This is the
  most intuitive fix and aligns the canonical row count with the
  vec0 row count. Possible cost: cursor-density assumptions
  elsewhere (e.g. `dev/design/engine.md` § Cursor contract,
  `_fathomdb_projection_state.last_enqueued_cursor`) may need a
  pass.

- **B: keep one cursor per batch, but project ALL bodies for that
  cursor**. Drop the `terminal_state_for_cursor` short-circuit (or
  scope it per-row). Requires vec0 to accept multiple rows per
  rowid (needs an alternate keying scheme — currently
  `vector_default.rowid` is the PK) or to extend `_fathomdb_vector_rows`
  with a sub-id and join via that.

Option **A** is the smaller surface change. The cursor contract docs
(`dev/design/engine.md`) define `write_cursor` as a monotonic
identifier; the current "shared cursor per batch" is an
implementation detail not surfaced in the public contract.

## Reproducer

```rust
use std::sync::Arc;
use fathomdb_engine::{Engine, PreparedWrite};
use fathomdb_embedder_api::{Embedder, EmbedderError, EmbedderIdentity, Vector};
use tempfile::TempDir;

// ... VaryingEmbedder (FNV-1a + 6-coord), see perf_gates.rs ...

fn main() {
    let dir = TempDir::new().unwrap();
    let path = dir.path().join("repro.sqlite");
    let opened = Engine::open_with_embedder_for_test(
        &path, Arc::new(VaryingEmbedder::new(768))
    ).unwrap();
    let engine = opened.engine;
    engine.configure_vector_kind_for_test("doc").unwrap();

    let batch: Vec<_> = (0..21)
        .map(|i| PreparedWrite::Node {
            kind: "doc".into(),
            body: format!("body-{i}"),
            source_id: Some(format!("id-{i}")),
        })
        .collect();
    engine.write(&batch).unwrap();
    engine.drain(15_000).unwrap();

    let count = engine.vector_row_count_for_test().unwrap();
    assert_eq!(count, 21, "expected 21 vec0 rows, got {count}"); // FAILS — count == 1
}
```

## What changes pending the fix

- `tests/corpus_vector.rs` does single-node writes so the wiring gate
  passes; comments + this note explain why.
- `examples/ingest_corpus.rs` and `perf_gates.rs::seed_ac013_corpus`
  are NOT changed in this drop — the PVQ Pack 2 agent owns the
  perf-gate behavior and the engine fix.
- Once a fix lands, the per-node workaround in
  `tests/corpus_vector.rs` should be reverted to a batched write so
  the test exercises the same write path Pack 3 + perf_gates use.

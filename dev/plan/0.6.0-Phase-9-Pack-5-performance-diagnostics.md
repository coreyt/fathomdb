### Reviewer: `codex` CLI

- After each landed experiment (B.1, B.2, B.3, C.1, D.1) and before
  the §8 verification gate, run a `codex exec` reviewer pass.
- Reviewer mandate:
  1. Check the diff against this packet's acceptance criteria (§1)
     and the per-experiment decision rule.
  2. Robustness check against **Phase 7 and Phase 8** invariants:
     lifecycle event taxonomy, cursor-from-snapshot contract
     (REQ-013 / AC-059b / REQ-055), profile-callback SAFETY
     (commit `ba900a1`), reader-pool ADR (commits `09415b4`,
     `123a0d6`, `7852269`), database-lock-mechanism revision.
     The reviewer flags any change that weakens those invariants
     even if perf gates pass.
  3. Cross-provider diversity (OpenAI vs Anthropic) lowers the chance
     of correlated-blind-spot mistakes on FFI / mutex / SQL changes.
- Reviewer is read-only; it produces a structured verdict
  (PASS / CONCERN / BLOCK) with file:line citations into a

### Workflow per phase

1. **Pre-write** the prompt file in full at plan time (acceptance,
   touch lists, verify, output contract).
2. **Update-before-spawn:** append a dated "Update log" entry with
   latest baseline numbers, latest related findings, any plan
   amendments. The freshly appended notes are what makes the prompt
   current. Without this step the prompt is stale.
3. **Spawn** `claude` (or `codex` for reviewer) with the prompt file
   as stdin / `--prompt-file`.
4. **Capture** stdout + stderr to `dev/plan/runs/`. Structured
   `Required output` fields written to `<phase>-output.json` (subagent
   responsibility per the prompt's Log destination section).
5. **Orchestrator decision point (main thread):**
   - Read the run log + `<phase>-output.json`.
   - Read the codex reviewer verdict (when applicable).
   - Decide: proceed to next phase / re-run with amended prompt /
     escalate / abort packet.
   - **Update `dev/plan/runs/STATUS.md`** — single up-to-date
     progress board. Edit the matching row in "Phase results",
     append measurements, update active phase / worktrees /
     concerns / next action. This file is the durable source of
     truth on packet progress; keep it current at every step-5.
   - Record the decision in §12 experiment log with one-line entry.
   - Append the decision + carried-forward numbers to the next
     phase's prompt-file Update log before spawning it.
   - Append a §11 narrative paragraph to
     `dev/notes/performance-whitepaper-notes.md` only on KEEP
     (and to §5 on REVERT).

This step 5 is the only place the orchestrator (this Claude session)
makes a routing call. Subagents do not chain to each other directly.

## 2. Background snapshot

Best retained AC-020: `sequential=456ms / concurrent=127ms / bound=85ms`.
Speedup 3.59x on 8 threads, 12-core ARMv8 box. Bound requires 5.33x.

Already exhausted: reader-side hold-time, `prepare_cached` on read path,
reader open flags, tx teardown style, pool size, vec0 single-stmt
materialize, subscriber empty fast-path.

Working hypothesis ladder (from notes §6):

1. SQLite global allocator mutex (THREADSAFE=1 default). Primary.
2. Page-cache mutex / lookaside not per-conn. Secondary.
3. Per-search prepare cost amplifies allocator traffic. Tertiary.

Pattern (1.5x–3.6x cap on 8 threads, 12 cores) is consistent with a
single shared mutex.

## 3. Design-of-experiments principles

- **Diagnose before tuning.** No more speculative reverts. First
  capture is profile-driven.
- **One variable per experiment.** Compose only after each is measured
  in isolation.
- **Reversible first.** Runtime config before rebuild before structural
  change.
- **N=5 minimum** per measurement. Report median + min/max. Variance
  on this hardware has been observed at ~10%.
- **Same hardware, same fixture, same `--release` build.** Run with
  CPU governor pinned to `performance` and no other heavy load.
- **Fail fast.** Each experiment has a written decision rule; if it
  does not hit, revert immediately and move to the next.
- **Always re-run AC-017 + AC-018** after any change. AC-018 must stay
  green; AC-017 must not regress.

## 4. Phase A — Diagnostic capture (no production-code changes)

### A.0 Harness split

Add a feature-gated way to run the AC-020 harness in **sequential-only**
and **concurrent-only** modes. Today the test mixes both phases; clean
profiles need them as separate processes.

- Implementation sketch: add two `#[ignore]` or env-gated tests
  (`ac_020_sequential_only`, `ac_020_concurrent_only`) that share the
  fixture seed but run only one phase. Or split via env var
  (`AC020_PHASE=sequential|concurrent`). Keep the existing combined
  test as the gate.
- Land this change before profile capture. It is a test-only edit and
  has no production-code impact.

### A.1 perf record + flamegraph (primary diagnostic)

- Hardware: same 12-core ARMv8 box. CPU governor pinned to
  `performance`.
- Command shape (executed by hand per session, not in CI):

  ```bash
  perf record -F 999 -g --call-graph dwarf -- \
      cargo test -p fathomdb-engine --test perf_gates --release \
      AC020_PHASE=sequential ac_020_sequential_only -- --nocapture
  ```

  Repeat with `AC020_PHASE=concurrent`. Collect both `perf.data` files
  separately.

- Render with `flamegraph.pl` (or `inferno-flamegraph`). Save SVGs in
  `dev/notes/perf/ac020-{sequential,concurrent}-<short-sha>.svg`
  (gitignored or LFS — confirm policy before commit).

- Diff the two using `inferno-diff-folded` or manual stack inspection.

### A.2 Targeted symbol focus

When reading the diff, classify time spent under:

- `pthread_mutex_lock`, `pthreadMutexEnter`, `sqlite3_mutex_*`
- Allocator: `mem1Malloc`, `mem1Free`, `malloc`, `free`, `je_*` if jemalloc
- Page cache: `pcache1Fetch`, `pcache1Truncate`
- vec0 / FTS: any `sqlite3VtabCall*`, `vec0_*`, `fts5_*`
- Our own code: `read_search_in_tx`, `ReaderPool::*`, `dispatch*`

The **diagnostic question**: which frame's _fraction of total time_
grows super-linearly going from sequential to concurrent? That frame
is the bottleneck.

### A.3 Secondary diagnostics (cheap, run alongside)

- `strace -c -f` on the concurrent run. Time share in `futex` vs file
  I/O (`pread64`, `pwrite64`, `fdatasync`) differentiates SQLite
  internal mutexes from VFS contention.
- Lightweight in-process counters (test-only, behind `#[cfg(test)]` or
  a `perf_diag` feature) around:
  - reader checkout wait (time blocked on `ReaderPool::borrow`)
  - embedder time
  - `read_search_in_tx` total
  - `tx.prepare` count and aggregate time
- Capture `EXPLAIN QUERY PLAN` once for each of the four SQL
  statements in `read_search_in_tx`. Verify no planner regression.
- Capture `sqlite3_threadsafe()` return value and `PRAGMA compile_options`
  output once. Confirms the bundled threading flag.

### A.4 Phase A exit criteria

- Sequential and concurrent flamegraphs captured and saved.
- One named symbol (or named symbol family) identified as the
  contention source, supported by a >2x growth in time-share between
  sequential and concurrent runs.
- Decision recorded: which Phase B experiment to attempt first.

If Phase A is inconclusive (no clear contention symbol), do **not**
proceed to B. Instead extend instrumentation and recapture.

## 5. Phase B — Cheap reversible runtime experiments (gated on Phase A)

Each experiment below is independent. Run only the one(s) Phase A
points at first. Each requires a baseline-run, an intervention-run,
and a rollback if the decision rule does not fire.

### B.1 Runtime SQLite multi-thread mode

- **Hypothesis:** allocator + pcache mutexes from `THREADSAFE=1`
  serialize 8 readers.
- **Intervention:** at engine first-init, run
  `sqlite3_shutdown` → `sqlite3_config(SQLITE_CONFIG_MULTITHREAD)` →
  `sqlite3_initialize`, **before any** `Connection::open` call. Place
  this in a new `init_sqlite_runtime()` `Once`-guarded function called
  at the head of `Engine::open_locked` (lib.rs:740) ahead of
  `register_sqlite_vec_extension` (lib.rs:1824).
- **Verification:** capture and assert `sqlite3_config` return code
  (must be `SQLITE_OK`); log `sqlite3_threadsafe()` return value
  (expect `2`).
- **Decision rule:** AC-020 concurrent drops by >= 30% vs Phase A
  baseline AND speedup >= 5.0x → keep. Otherwise revert.
- **Risk:** misconfiguration silently failing. Mitigation: assert on
  the return code at runtime; surface as an `EngineOpenError` if not
  `SQLITE_OK`.

### B.2 `SQLITE_CONFIG_MEMSTATUS=0`

- **Hypothesis:** allocator stats add a small but real lock-step cost.
- **Intervention:** add to the same init function as B.1.
- **Decision rule:** B.1 + B.2 combined yields >= 5% additional
  improvement vs B.1 alone → keep. Otherwise revert.

### B.3 Per-connection lookaside / page cache

- **Hypothesis:** even in MULTITHREAD mode, frequent small allocations
  cross a global pool; per-connection lookaside reduces traffic.
- **Intervention:** per-connection `PRAGMA cache_size` already tried
  and reverted. Do **not** retry that. Instead try
  `SQLITE_CONFIG_LOOKASIDE` once at init (e.g. `slot_size=128, cnt=500`).
- **Decision rule:** AC-020 concurrent drops by >= 10% vs prior keep.
- **Skip if:** B.1 already passes the 5.33x bound.

### B.4 Phase B exit criteria

- Either AC-020 passes (5x margin gate from §1) → go to §8 verification.
- Or no Phase B intervention closes the gap → go to Phase C.

## 7. Phase D — Structural search-shape change (parallel track)

Track this in parallel from the start of Phase A; do **not** land
without Phase A signal supporting it.

### D.1 Single combined SQL statement for vec0 + FTS5

- **Hypothesis:** per-search prepare cost (4 statements) amplifies
  whatever lower-level contention exists; collapsing reduces
  malloc / mutex traffic per query and improves both sequential and
  concurrent latency.
- **Intervention:** in `read_search_in_tx` (lib.rs:1283), one prepared
  statement combining vec0 retrieval and FTS5 retrieval via UNION,
  inside the same read transaction. Optionally `prepare_cached`.
  Snapshot derivation unchanged.
- **Required tests to keep green:**
  - `tests/projection_runtime.rs` (vector-only and hybrid dedupe
    behavior).
  - `tests/cursors.rs` (cursor consistency).
  - `tests/error_taxonomy.rs`.
  - AC-017, AC-018, AC-020.
- **Decision rule:** AC-020 sequential improves by >= 15% AND
  concurrent improves by >= 15% AND speedup ratio does not regress
  → keep. Otherwise revert.
- **Risk:** correctness regressions in dedupe / soft-fallback /
  cursor. Run the full engine test suite, not just perf gates,
  before keeping.

## 9. Risks and pitfalls

- **Goodhart's law on the ratio.** A change that makes sequential
  _slower_ can pass the bound without improving real concurrency.
  Always report absolute sequential and concurrent times; do not
  optimize the ratio in isolation.
- **`sqlite3_config` ordering.** Earlier session's silent no-op was
  caused by `sqlite3_initialize` running before the config call.
  B.1 explicitly captures the return code to avoid repeating that.
- **Variance.** ~10% run-to-run variance observed. N=5 minimum;
  reject any conclusion drawn from a single run.
- **Auto-extension timing.** `register_sqlite_vec_extension` calls
  `sqlite3_auto_extension`, which itself triggers `sqlite3_initialize`.
  Any `sqlite3_config` must run **before** this. The new
  `init_sqlite_runtime()` must be sequenced first.
- **Test fixture size.** Fixture is intentionally tiny. If a Phase D
  change happens to win on this size but lose on real workloads,
  consider adding a larger-fixture variant before keeping.

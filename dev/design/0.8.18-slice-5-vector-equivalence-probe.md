## What it is

An open-time self-check that re-embeds the 45 committed equivalence probes
(`fathomdb-embedder/tests/fixtures/candle_onnx_equivalence_probes.txt`, copied
crate-local as `fathomdb-engine/src/vector_equivalence_probes.txt` with a
byte-identity drift-guard test) with the LIVE backend and refuses dense retrieval
if the backend has diverged beyond the frozen D4 floor. It is **ADDITIVE** to the
existing cross-vendor `EmbedderIdentity` refusal (`check_embedder_profile`) — the
identity gate proves the backend *claims* the same embedder; the probe proves it
*produces equivalent 1-bit codes* (R-VEQ-5, additive-only).

## The FROZEN D4 tolerance floor

- **P1 (Phase-1 binary):** mean-centered `embedding_bin` sign-flip count floor =
  **0 (exact)** — any single flip across the 45 probes ⇒ divergence
  (`VECTOR_EQUIVALENCE_P1_FLIP_FLOOR = 0`).
- **P2 (Phase-2 float):** un-centered L2 ε = **1e-5** — `‖reembed − reference‖₂`
  (`vec_distance_l2` semantics) `> 1e-5` ⇒ divergence
  (`VECTOR_EQUIVALENCE_L2_EPSILON = 1e-5`). Single named const; final ε gets a
  HITL look at landing.

BOTH are asserted (correctness trap: asserting only P1 lets P2 float drift
silently within recalled candidates; asserting only P2 lets a centered-code flip
silently drop candidates from Phase-1 recall).

## Storage (R-VEQ-1) — schema step 19 (`SCHEMA_VERSION` 18→19)

New internal table `_fathomdb_embed_probe(probe_ordinal, probe_text,
reference_vec BLOB, embedder_name, embedder_revision, dim)`. Stores **UN-centered
f32 references ONLY** — the P1 bits are **NEVER persisted** (recomputed at check
time from the un-centered reference + the live pinned `mean_vec`, U1-d). The 45
rows are written by the engine at first vector-kind registration (open path,
after mean-recovery + `ensure_vector_partition`); the migration creates the empty
table only. Pure additive `CREATE TABLE` (eu7 no-op basis — no vec0 rewrite).

## Check (R-VEQ-2/3) — where + how

`run_vector_equivalence_probe` runs in `Engine::open_with_migrations` on the
FINAL post-recovery writer connection (**after** open-time mean-recovery/requantize
and `ensure_vector_partition`, so it reads the FINAL live `mean_vec` — U1-b).
First registration persists references (dense stays enabled). A subsequent open
re-embeds each probe and asserts:

- **P1:** `vec_quantize_binary(sign(x − mean_vec))` bits for reference vs reembed,
  via the SAME SQL `vec_quantize_binary` path as `build_vector_phase1_sql`; the
  Hamming (flip) count must be `≤ 0`.
- **P2:** un-centered L2 must be `≤ 1e-5`.

Mean-centering is gated by `identity_requires_mean_centering(identity)` ∧
`mean_vec` pinned (true only for the default `fathomdb-bge-small-en-v1.5`;
NoopEmbedder no-op), applied symmetrically to both sides with an un-centered
fallback (R-VEQ-3c). The check never fails open: a probe-infrastructure error
(embed failure, malformed row) leaves dense ENABLED (the identity gate stays
primary); divergence is only ever asserted on a successfully re-embedded probe.

## Refusal (R-VEQ-4) — degraded-open + single choke point

On divergence the open **SUCCEEDS** with `dense_disabled = true` (NOT an open
error). The query-time refusal is a new **`EngineError::VectorEquivalenceMismatch
{ reason }`** raised at the SINGLE choke point `Engine::search_inner_with_stats`,
BEFORE any embedding / vector SQL / graph seeding / CE rerank — covering EVERY
vector-dependent arm (`search`, `search_filtered`, `search_reranked`,
`search_explained`, `search_expand`, and the `use_graph_arm` graph arm, which all
funnel through it). Bindings map it to leaf classes/codes at parity with
`EmbedderIdentityMismatch`:

- Rust: `EngineError::VectorEquivalenceMismatch { reason }` (`stable_code =
  "VectorEquivalenceMismatchError"`).
- Python: `fathomdb.errors.VectorEquivalenceMismatchError` (leaf of `EngineError`;
  carries `reason`).
- TypeScript: `VectorEquivalenceMismatchError` (leaf of `FathomDbError`; carries
  `reason`); envelope code `FDB_VECTOR_EQUIVALENCE_MISMATCH`.

### FTS-only escape hatch

`Engine::search_text_only(query)` (+ Py `search_text_only`, TS `searchTextOnly`)
is the explicit text-only / FTS-only path. It does NOT embed the query and does
NOT route through the choke point, so it NEVER raises the refusal — FTS stays
serviceable in degraded mode (D2 / codex R2 U1-2). It is a governed application
command (added to `src/conformance/governed-surface-allowlist.json`).

## Observability (R-VEQ-6)

`OpenReport.dense_disabled` (+ `dense_disabled_reason`) surfaced on Rust / Python
/ TypeScript. Engine accessors `dense_disabled()`, `dense_disabled_reason()`, and
the telemetry counter `vector_equivalence_refusal_count()` (increments per
refused query) — all observability, excluded from the governed-command allowlist.
The state is RE-DERIVED at every open (the probe re-runs), so a reopen with a
still-divergent backend stays degraded (never silently re-enables dense) and a
reopen with a matching backend clears it.

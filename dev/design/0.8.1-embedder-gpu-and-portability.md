## 2. Cross-vendor portability â€” the `Embedder` trait is the boundary (B.2.a)
**candle 0.10 supports ONLY CPU / CUDA / Metal â€” no ROCm, no Vulkan.** So AMD and Intel
GPUs are NOT reachable through candle's `Device`. The durable abstraction is the
`Embedder` trait itself: a new backend is a new `impl Embedder`, plugged via the
existing `EmbedderChoice::Caller(Arc<dyn Embedder>)` â€” **zero engine changes**.

Future backends (no engine surgery; each is a new crate-local impl behind the trait):
- **ONNX Runtime (`ort` crate)** â€” one API covering CUDA, **ROCm (AMD)**, **DirectML
  (AMD/Intel)**, **OpenVINO (Intel)**, CoreML, CPU. The recommended cross-vendor path;
  BGE-small has standard ONNX exports. â†’ `OrtBgeEmbedder`.
- **burn + wgpu** â€” Vulkan/Metal/DX12 (any GPU, pure Rust). Bigger lift; the true-Vulkan
  option. â†’ `WgpuBgeEmbedder`.

Design rules locked now so the swap stays a localized change: (a) never hardcode the
device â€” `resolve_device()` behind env/config; (b) all device/backend logic stays inside
the embedder crate behind the trait (the engine only ever sees `Arc<dyn Embedder>`);
(c) backend selected at `Engine::open` via config/env, not compile-time-only.

## 3. Vector equivalence â€” required guard before any backend/machine move (B.2.b)
**Problem.** A DB's stored vectors are valid only for the *exact* embedding function
(model + revision + **backend numerics**). Moving a DB to another machine, or swapping
backend (CPUâ†”CUDAâ†”ROCmâ†”ONNX), can produce subtly different vectors for the same text â†’
the dense index silently degrades (queries embedded by backend B against an index built
by backend A). `EmbedderIdentity` (name, revision, dim) + the open-time identity/dimension
check catches a *model/revision/dim* mismatch but **NOT numeric divergence between two
backends sharing the same identity** (e.g. CUDA vs CPU of the same model can differ at the
float level; kernel/rounding differences).

**Requirement (0.8.x TODO â€” not 0.8.1).** Add a **vector-equivalence self-check** so
"a database cannot move to a different machine / backend unless vector-equivalent" is
*enforced*, not assumed:
1. Store a small canonical **probe set** in the DB at first vector-kind registration:
   a `_fathomdb_embed_probe(probe_text, reference_vector)` table â€” N (~16â€“32) fixed,
   diverse strings + their reference vectors, computed by the embedder that built the index.
2. On `Engine::open` with an embedder present (and on any backend/machine change), re-embed
   the probe set with the **live** embedder and assert each probe within tolerance
   (cosine â‰¥ 1âˆ’Îµ, or L2 â‰¤ Î´) of its stored reference. Diverge beyond tolerance â‡’ refuse to
   serve the dense/fused arm (or require a rebuild) â€” a loud, typed error, never silent.
3. **Tolerance is subtle:** vectors are 1-bit quantized at store time, so the check should
   compare at the representation actually used for retrieval (post-quantization Hamming, or
   pre-quant cosine with a margin chosen so a true backend change trips it but float-noise
   on the SAME backend does not). Calibrate against the binary-quantization floor.
4. This subsumes/strengthens the `EmbedderIdentity` guarantee: identity = "claims to be the
   same embedder"; the probe set = "*proves* it produces equivalent vectors here." Both
   ship; the probe set is the enforcement, identity is the cheap pre-filter.

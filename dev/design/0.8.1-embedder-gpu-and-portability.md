# Embedder GPU acceleration + cross-vendor portability + vector equivalence

> Decision record, 2026-06-15 (HITL-approved). Covers: (1) GPU-now via a device
> seam; (2) the abstraction that lets AMD/Intel GPUs swap in later (B.2.a); (3) the
> vector-equivalence requirement so a DB cannot silently move across machines /
> backends and corrupt its index (B.2.b). (1) ships in 0.8.1; (3) is a **0.8.x
> TODO** (tracked in `dev/roadmap/0.8.2.md`).
>
> Grounding: the embedder is serialized (PR-9 guard, one `embed()` at a time),
> `Device::Cpu` is hardcoded (`fathomdb-embedder/src/candle_bge.rs:168`), the trait
> is `Embedder::embed(&str) -> Vector` (no batch), and the engine already accepts a
> caller-supplied embedder via `EmbedderChoice::Caller(Arc<dyn Embedder>)`. Hardware:
> 2× RTX 3090 (compute-idle), CUDA 12.6 toolkit present.

## 1. GPU now — candle `Device` seam (NVIDIA CUDA; Apple Metal a one-liner later)

> **Status (2026-06-26, 0.8.7):** §1 LANDED and VALIDATED. The seam + the
> `embed-cuda`/`embed-metal` feature chain shipped with `02297cb3` (in the 0.8.5
> baseline); 0.8.7 confirmed the design is still current, added the pure
> `parse_device_request` + env-grammar unit tests, and measured the speedup on an
> RTX 3090: per-doc **15.6×**, batched **93×**, best-path bulk re-embed **~46×**;
> CPU↔CUDA vectors agree to cosine 0.99999983 and produce **identical** 1-bit
> codes (0/6144 sign-bit diff) on the probe set. §2/§3 remain backlog. See
> `dev/plans/runs/STATUS-0.8.7.md`.

Replace the hardcoded `Device::Cpu` with `resolve_device()`:
- env `FATHOMDB_EMBED_DEVICE` = `cpu` (default) | `cuda` | `metal`; `cuda[:N]` selects GPU N.
- cargo features (mirror the existing `default-embedder` chain through embedder→engine→py):
  - embedder: `embed-cuda = ["default-embedder", "candle-core/cuda", "candle-nn/cuda", "candle-transformers/cuda"]` (and `embed-metal` analogously).
  - engine: `embed-cuda = ["default-embedder", "fathomdb-embedder/embed-cuda"]`; py likewise.
- **Default build stays CPU** → all existing tests + byte-stability gates unchanged; GPU is opt-in via feature + env. Build: `maturin develop --features pyo3/extension-module,embed-cuda` (in the MAIN tree — never a worktree, per [[agent-worktree-stale-base-trap]]).
- **Why serialization is fine on GPU:** the PR-9 guard (one concurrent `embed()`) prevents concurrent CUDA calls on the shared model — exactly what you want. Model loads at `Engine::open`, so the 30 s per-embed watchdog never sees cold-start cost. Single-stream GPU still ≈100–300× CPU → 27 h embed → minutes.

## 2. Cross-vendor portability — the `Embedder` trait is the boundary (B.2.a)
**candle 0.10 supports ONLY CPU / CUDA / Metal — no ROCm, no Vulkan.** So AMD and Intel
GPUs are NOT reachable through candle's `Device`. The durable abstraction is the
`Embedder` trait itself: a new backend is a new `impl Embedder`, plugged via the
existing `EmbedderChoice::Caller(Arc<dyn Embedder>)` — **zero engine changes**.

Future backends (no engine surgery; each is a new crate-local impl behind the trait):
- **ONNX Runtime (`ort` crate)** — one API covering CUDA, **ROCm (AMD)**, **DirectML
  (AMD/Intel)**, **OpenVINO (Intel)**, CoreML, CPU. The recommended cross-vendor path;
  BGE-small has standard ONNX exports. → `OrtBgeEmbedder`.
- **burn + wgpu** — Vulkan/Metal/DX12 (any GPU, pure Rust). Bigger lift; the true-Vulkan
  option. → `WgpuBgeEmbedder`.

Design rules locked now so the swap stays a localized change: (a) never hardcode the
device — `resolve_device()` behind env/config; (b) all device/backend logic stays inside
the embedder crate behind the trait (the engine only ever sees `Arc<dyn Embedder>`);
(c) backend selected at `Engine::open` via config/env, not compile-time-only.

## 3. Vector equivalence — required guard before any backend/machine move (B.2.b)
**Problem.** A DB's stored vectors are valid only for the *exact* embedding function
(model + revision + **backend numerics**). Moving a DB to another machine, or swapping
backend (CPU↔CUDA↔ROCm↔ONNX), can produce subtly different vectors for the same text →
the dense index silently degrades (queries embedded by backend B against an index built
by backend A). `EmbedderIdentity` (name, revision, dim) + the open-time identity/dimension
check catches a *model/revision/dim* mismatch but **NOT numeric divergence between two
backends sharing the same identity** (e.g. CUDA vs CPU of the same model can differ at the
float level; kernel/rounding differences).

**Requirement (0.8.x TODO — not 0.8.1).** Add a **vector-equivalence self-check** so
"a database cannot move to a different machine / backend unless vector-equivalent" is
*enforced*, not assumed:
1. Store a small canonical **probe set** in the DB at first vector-kind registration:
   a `_fathomdb_embed_probe(probe_text, reference_vector)` table — N (~16–32) fixed,
   diverse strings + their reference vectors, computed by the embedder that built the index.
2. On `Engine::open` with an embedder present (and on any backend/machine change), re-embed
   the probe set with the **live** embedder and assert each probe within tolerance
   (cosine ≥ 1−ε, or L2 ≤ δ) of its stored reference. Diverge beyond tolerance ⇒ refuse to
   serve the dense/fused arm (or require a rebuild) — a loud, typed error, never silent.
3. **Tolerance is subtle:** vectors are 1-bit quantized at store time, so the check should
   compare at the representation actually used for retrieval (post-quantization Hamming, or
   pre-quant cosine with a margin chosen so a true backend change trips it but float-noise
   on the SAME backend does not). Calibrate against the binary-quantization floor.
4. This subsumes/strengthens the `EmbedderIdentity` guarantee: identity = "claims to be the
   same embedder"; the probe set = "*proves* it produces equivalent vectors here." Both
   ship; the probe set is the enforcement, identity is the cheap pre-filter.

**Status:** 0.8.x backlog (see `dev/roadmap/0.8.2.md`). The GPU work in §1 lands first;
because it is opt-in and rebuilds a fresh index for eval, it does not *require* §3 — but
§3 MUST land before FathomDB advertises portable DBs or runtime backend-swap, and before
any production CUDA-built index is moved to a CPU (or other-vendor) reader.

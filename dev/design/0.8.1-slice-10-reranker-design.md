## Decision 1: Implementation crate structure

The cross-encoder (CE) lives **inside `fathomdb-engine`**, gated behind a new
`default-reranker` Cargo feature (parallel to `default-embedder` on
`fathomdb-engine`). No new crate is created.

**Rationale:**

- The 0.8.1 plan calls for K=200 candidate reranking that happens in `rerank_fused`
  inside `read_search_in_tx`, which is entirely within `fathomdb-engine`. Putting
  the CE in a sibling crate would require either a trait seam (like
  `fathomdb-embedder-api`) or a direct dependency. A trait seam costs another crate
  and an API surface; a direct dep costs ML dependency bleed.
- The `fathomdb-engine` `default-embedder` feature already re-exports
  `fathomdb-embedder/default-embedder` (Candle stack), so the `default-reranker`
  feature can follow the same convention: when on, it pulls in Candle + tokenizers
  (via workspace deps already gated in `fathomdb-embedder`).
- The engine crate's `rerank_fused` is the only call site; there is no other
  consumer of CE inference in 0.8.1. Introducing a crate boundary adds churn with
  no payoff until a second consumer exists (reservable-gap candidate).
- **Default build invariant:** `cargo test -p fathomdb-engine` (no features) must
  not compile any ML code. The `#[cfg(feature = "default-reranker")]` blocks
  satisfy this.

**7 current workspace crates:** `fathomdb`, `fathomdb-cli`, `fathomdb-embedder`,
`fathomdb-embedder-api`, `fathomdb-engine`, `fathomdb-napi`, `fathomdb-py`,
`fathomdb-query`, `fathomdb-schema`. No new crate for this slice.

## Decision 2: Model and inference backend

**Model:** TinyBERT-L-2 (2-layer, ~4 MB weights), per the Slice 5 recommendation.

**Evidence from IR-C-r0-findings.md § 4.1:**

- TinyBERT-L-2: p50 = 1.536 ms/pair, p95 = 2.85 ms/pair.
- At K=200: p50 total = 200 × 1.536 ms = **307 ms** (within AC-013 0.x budget).
- MiniLM-L12 (22 MB): 200 × 16.82 ms = 3364 ms — exceeds 0.x budget.

**Backend:** Candle (`candle-core`, `candle-nn`, `candle-transformers` +
`tokenizers`) — already in the workspace under `fathomdb-embedder`'s
`default-embedder` feature. Reusing the existing stack avoids an ONNX Runtime
(`ort`) dependency, which would bring a distinct C++ artifact into the build.

TinyBERT-L-2 uses the standard BERT architecture (2 transformer layers, 312 hidden
dim). `candle-transformers::models::bert` provides `BertForSequenceClassification`
with configurable hidden dimensions — TinyBERT-L-2 is a direct instantiation with a
[1]-dim classifier head (relevance score, raw logit → sigmoid for probability,
then optionally used in the blend).

**Gating:** The CE inference struct and all Candle imports live behind
`#[cfg(feature = "default-reranker")]` in `fathomdb-engine/src/lib.rs` (or a
sub-module). The default build (no feature) never compiles or links any ML code.

## Decision 3: Model weight loading

The `default-reranker` feature follows the `default-embedder` pattern exactly:

- Weights are fetched once at first use via `ureq` + `sha2` checksum verification,
  cached under `~/.cache/fathomdb/reranker/` (or a configurable override).
- `CandleTinyBertReranker::try_load() -> Result<Self, RerankerLoadError>` attempts
  to load from cache; on a cache miss, downloads from a pinned HuggingFace URL
  (SHA-256 verified before use). Returns `Err` (never panics) on network failure.
- **No network call in the default (feature-off) build.** No network call in any
  production code path when `rerank_depth == 0`. Network call is only triggered
  when `default-reranker` feature is on AND `rerank_depth > 0` AND weights absent
  from cache.
- The `rerank_fused` soft-fallback contract (Decision 7 below) means production
  callers who never set `rerank_depth > 0` never trigger a download.

## Decision 4: Signature change for `rerank_fused`

**New signature:**

```rust
pub fn rerank_fused(query: &str, hits: Vec<SearchHit>, rerank_depth: usize) -> Vec<SearchHit>
```

**Change from current:** `rerank_fused(hits: Vec<SearchHit>) -> Vec<SearchHit>`

The `query` string is needed by the CE to score `(query, passage)` pairs. The
`rerank_depth` is the per-request opt-in (`0` = soft-fallback, `N` = rerank top-N).

**Threading to the call site at `lib.rs:5051`:**

The raw query string is not currently present in `read_search_in_tx`. I add:

1. A `raw_query: &str` parameter to `read_search_in_tx`.
2. A `raw_query: String` field in `ReaderRequest::Search` (sent from `search_inner`
   alongside `compiled`).
3. A `rerank_depth: usize` field in `ReaderRequest::Search` (read from
   `ProjectionRuntimeShared::rerank_depth`, an `AtomicUsize`, off by default = 0).
4. A `rerank_depth: usize` parameter to `read_search_in_tx`.

The call site becomes:

```rust
rerank_fused(&raw_query, apply_recency_reweight(
    fuse_rrf(vector_results, text_results),
    recency_enabled,
), rerank_depth)
```

**`rerank_fused_is_identity_stub` disposition (`pr_g9_rrf_fusion.rs:114`):**

The existing test calls `rerank_fused(hits.clone())` — it must be adapted to the
new 3-argument signature. RED-3 updates it to:

```rust
assert_eq!(rerank_fused("", hits.clone(), 0), hits);
```

This preserves the test's semantic contract (soft-fallback at depth=0 is identity)
under the new signature. The test is NOT deleted; it is the primary pin for the
identity-stub behavior, now expressed as `depth=0`.

## Decision 5: Score-blend formula

When `rerank_depth > 0` and the CE model is loaded:

**Formula:**

```text
ce_normalized  = sigmoid(raw_logit) ∈ [0, 1]
final_score    = α × ce_normalized + (1 - α) × rrf_score_normalized
```

where:

- `rrf_score_normalized` = min-max normalization of `hit.score` over the top-K
  pool being reranked (i.e., `(s - s_min) / (s_max - s_min)`, clamped to [0,1];
  if all scores are equal, normalized = 1.0 for all).
- `α = 0.3` (constant, not configurable per-request in 0.8.1 — reserved-gap 11).

**Why α = 0.3 (CE weight is the minority):**
The C6 guard requires that a high-CE-but-wrong document cannot displace a
BM25-ranked correct factoid. Under α = 0.3, even a maximally CE-scored document
(CE = 1.0) against a low-CE correct factoid (CE = 0.0) with a strong RRF gap:

- CE-wrong: 0.3 × 1.0 + 0.7 × 0.0 = 0.30
- RRF-right (rrf_norm = 1.0, ce_norm = 0.0): 0.3 × 0.0 + 0.7 × 1.0 = 0.70

The correct document wins by 0.40. Factoid regression requires a CE score high
enough to close a large RRF gap — plausible only if the CE is genuinely
mis-assigning relevance. In practice the BM25 arm heavily dominates the factoid
class (exact_fact R@10 0.905, rrf_fused at K=200 0.970), so the CE's 0.3 weight
is a quality nudge, not an override.

**Hits beyond `rerank_depth`:** not re-scored by CE. Their RRF scores are
preserved. They are appended to the reranked top-N in their original RRF order.

**Determinism of CE normalization:** min-max is a pure function of the input
vector; same (query, corpus snapshot) → same scores → same order. No stochastic
element.

## Decision 6: `rerank_depth` exposure on the public API

**Pattern:** per-request opt-in, default 0.

**Rust Engine:**

- `search(query)` → unchanged (internally passes `rerank_depth = 0`).
- `search_filtered(query, filter)` → unchanged (internally passes `rerank_depth = 0`).
- New: `search_reranked(query, filter, rerank_depth: usize) -> Result<SearchResult, EngineError>` — exposes the opt-in explicitly. Validated: `rerank_depth` ≥ 0 (usize, always true).
- `search_inner` gains a `rerank_depth: usize` parameter (private).

**Governed surface:** `search_reranked` is a new public method on `Engine` (in
`fathomdb-engine`) and re-exported from the `fathomdb` facade crate. No new
*type* is introduced by this change (rerank_depth is `usize`).

**Python** (`fathomdb.Engine.search`):

```python
def search(self, query: str, filter: SearchFilter | None = None,
           *, rerank_depth: int = 0) -> SearchResult
```

Validation: `rerank_depth < 0` raises `ValueError("rerank_depth must be >= 0")`.
Passes to native `search_reranked` when `rerank_depth > 0`; falls back to
`search_filtered` when 0 (no change to existing path).

**TypeScript** (`engine.search`):

```typescript
async search(query: string, filter?: SearchFilter, rerankDepth?: number): Promise<SearchResult>
```

Validation: `rerankDepth !== undefined && !Number.isInteger(rerankDepth)` →
`TypeError`; `rerankDepth < 0` → `RangeError`. Calls `searchReranked` on the
native when `rerankDepth > 0`.

Both SDKs expose `search_reranked` / `searchReranked` on the native layer;
the pure-Python/TS wrappers route through the right native call.

**No new type in the governed surface allowlist** (no new struct/enum). The
existing `SearchResult`, `SearchHit`, etc. are unchanged.

## Decision 7: Soft-fallback contract

When `rerank_depth == 0` OR the CE model is not loaded (feature off or weights
absent), `rerank_fused` returns `hits` **unchanged** — byte-identical to the
current identity stub.

Implementation:

```rust
pub fn rerank_fused(query: &str, hits: Vec<SearchHit>, rerank_depth: usize) -> Vec<SearchHit> {
    if rerank_depth == 0 {
        return hits; // identity path — byte-identical to old stub
    }
    #[cfg(feature = "default-reranker")]
    {
        if let Some(model) = try_get_loaded_reranker() {
            return blend_rerank(model, query, hits, rerank_depth);
        }
    }
    hits // model absent → soft-fallback
}
```

Testing the soft-fallback:

- `rerank_fused_soft_fallback_preserves_fused_order`: call with `depth=0`;
  assert hit order + scores are byte-identical to input.
- `rerank_depth_0_is_byte_identical_to_identity_stub`: the adapted
  `rerank_fused_is_identity_stub` test (RED-3) pins this at `depth=0`.

In the default build (no `default-reranker` feature), the `#[cfg(...)]` block
compiles away and `rerank_fused` is always identity regardless of `rerank_depth`.

## Decision 8: Determinism

For the soft-fallback path (default build): `rerank_fused` is a pure identity
function → trivially deterministic.

For the CE path (when model loaded):

- Tokenizer: deterministic (same text → same token IDs).
- Candle inference: single-threaded CPU; no stochastic sampling (cross-encoder is
  inference-only, no dropout at eval time). Same model + same tokenizer + same
  input → same raw logit → same normalized score.
- `min-max` normalization and score blending are deterministic.
- Final sort uses `partial_cmp` with a stable fallback → deterministic given same
  float scores.

**Conclusion:** the same (query, corpus snapshot) produces byte-identical
reranked output across calls.

## Decision 9: `RECENCY_WEIGHT` fix

**Current value:** `0.5 / RRF_K = 0.5 / 30.0 ≈ 0.01667`.

**Red test scenario:**

- `hit(9, "recent", 0.01)`, `hit(1, "strong", 0.02)`, `enabled = true`.
- `min_id = 1`, `max_id = 9`, `span = 8`.
- For "recent" (id=9): `norm = (9-1)/8 = 1.0`.
- For "strong" (id=1): `norm = (1-1)/8 = 0.0`.
- After reweight: recent = 0.01 + 0.01667 × 1.0 = 0.02667; strong = 0.02 + 0 = 0.02.
- "recent" wins, but the test asserts "strong" wins.

**Condition for "strong" to win:** `0.02 > 0.01 + RECENCY_WEIGHT × 1.0`
  → `RECENCY_WEIGHT < 0.01`.

**Condition for tie-breaking still working** (`recency_enabled_breaks_equal_rrf_tie`):

- Equal scores (both = 0.01), "recent" id=2, "older" id=1.
- `norm("recent") = (2-1)/(2-1) = 1.0`, `norm("older") = 0.0`.
- For "recent" to win: `0.01 + RECENCY_WEIGHT × 1.0 > 0.01` → `RECENCY_WEIGHT > 0`.

**Chosen value:** `RECENCY_WEIGHT = 0.002`.

Verification:

- `0.02 > 0.01 + 0.002 × 1.0 = 0.012` → 0.02 > 0.012 ✓ "strong" wins.
- `0.01 + 0.002 × 1.0 = 0.012 > 0.01` → "recent" tie-breaks ✓.
- Latency gate: the weight change does not affect O(N) loop speed.

`RECENCY_WEIGHT = 0.002` is in the range (0, 0.01) and is "safe": it takes a
gap of > 0.002 to override any recency nudge, which means a single RRF rank-step
difference (≈ 0.0164 for rank 2 vs rank 1 at K=30) still dominates recency.

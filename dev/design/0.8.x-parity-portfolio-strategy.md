# FathomDB — Parity-or-better as a measure-routed PORTFOLIO (not one retrieval path)

**Status:** recommendation (pending codex §9 validation) · **Date:** 2026-06-25 · **Owner:** program steward
**Scope:** synthesizes 0.8.0→0.8.4 measured results into a strategy to reach parity-or-better vs **Mem0**
(agentic memory) and **GraphRAG** (sensemaking), across fidelity / IR-recall / IR-precision / relevance /
comprehensiveness / multi-hop, under the latency + footprint gates.

> **The reframe (the unlock).** Parity-or-better does **not** require one common FathomDB ingest or one
> common retrieval function. Different measures want *different mechanisms* — and 0.8.0–0.8.4 proved no
> single path wins all of them (needle-precision ≠ global-coverage ≠ multi-hop ≠ fidelity). So the strategy
> is a **portfolio**: plural ingest indexes **designed to coexist** in one store. (Note: vector rows are
> already `kind`-stamped, but today that `kind` is *document-type only* — `email/article/paper/meeting/note/
> todo/doc`, locked by the quant migration in `fathomdb-schema` — so a leaf-vs-coverage-vs-graph row-kind is a
> **future engine step**, and the D2 coverage index lives **eval-side** today; see §3.) Plural
> retrieval mechanisms, and a **query-intent router** (the agent, or an in-library dispatcher) that **picks
> the (ingest-index, retrieval, stack) tuple per query/measure.** This dissolves the tension that ran
> through the whole program: stop forcing one function to win every axis; route to the one that does.

## 1. The measures we compete on (the scorecard)

| # | Measure | Metric / gate | Competitor | Current FathomDB number | State |
|---|---|---|---|---|---|
| M1 | **Fidelity** (ANN/quant health) | eu7 recall@10 **≥ 0.90** (hard gate) | — (self) | **0.896** (case-A/SUT, not CLS) | BELOW floor |
| M2 | **IR recall** (gold in candidate pool) | recall@K | BM25 / Mem0 | **< BM25** on ku/multi_session; −0.10…−0.17 vs Mem0 | the surpass-blocker |
| M3 | **IR precision / ranking** (gold at top) | MRR, r@1 | — | CE-rerank α=1.0: **MRR 0.347→0.587, r@1 ×3.9** | the parity lever (works) |
| M4 | **Relevance** (embedder ceiling) | eu8 recall@10 | — | **≈0.571** (embedder-bound; report-only) | ceiling, no swap helps (15a) |
| M5 | **Answer accuracy** (agentic memory) | `decide_083`, identical-answerer | **Mem0** | fused 0.137 → CE α0.3 0.186 → **CE α1.0 ~0.33 vs Mem0 0.323** | provisional PARITY |
| M6 | **Sensemaking** (global QFS) | `decide_084` win-rate (comp/div/emp) | **GraphRAG** | **C surpass-direction (prov., `NOT_REACHED` power)** 0.72/0.61/0.72 vs L1; **D2 loses comp** | C provisional SURPASS; D2 no |
| M7 | **Multi-hop** | F1 / all-bridges (MuSiQue) | HippoRAG-2 | graph arm **refuted** (M1 ΔF1 −0.0405, ≯ BM25); HippoRAG **unmeasured** | graph dead; HippoRAG unknown |
| M8 | **Latency** | AC012/013/020 (hard gate) | — | dev-box green; CI-conditioned | gate, watch on engine changes |
| M9 | **Cost / footprint** | per-query compute tier | — | CPU-default; LLM/GPU opt-in | tiered: CPU→GPU→local-LLM→net-LLM |

**The two load-bearing facts the scorecard encodes:**
- **Precision (M3), not recall (M2), is what reached Mem0 parity.** The oracle proof: gold-only context
  lifts FathomDB **+0.39 above Mem0** — the gold is usually *in-window but buried*. Ranking fixes that;
  recall is the *separate*, surpass-blocking residual on multi_session/temporal.
- **No single mechanism wins M5 and M6.** Needle memory (M5) wants chunk-precision + CE-rerank; global
  sensemaking (M6) wants whole-corpus coverage (C) — and the cheap coverage index (D2) that wins M6-vs-weak
  loses M6-vs-strong and would *hurt* M5 (summarization discards the exact fact needle recall needs).

## 2. How each measure informs INGEST · RETRIEVAL · STACKING (the portfolio matrix)

Each row is an independent ingest/retrieval/stack the agent can select. Indexes are **designed** to coexist
(leaf/coverage/graph row-kinds are a future engine step — today the schema `kind` is document-type only and
D2 is eval-side, §3); nothing forces a common path.

| Measure | INGEST (build-time) | RETRIEVAL (query-time) | STACKING | Why this shape |
|---|---|---|---|---|
| **M5 needle / factoid memory** | leaf-chunk embed (bge-small) + FTS | fused RRF (vector-ANN + FTS) | **→ CE-rerank α=1.0, narrow pool (k≈10) → answerer** | gold in-window, buried → reorder, don't widen k |
| **M5 multi_session (recall-bound)** | leaf chunks **+ depth-1 coverage nodes (D2-as-router)** | **wider** candidate-gen, or coverage-node→member-chunk expand | recall-gen ↑ → **CE-rerank narrows back** → answerer | ranking can't add gold it never retrieved → lift recall first, then concentrate |
| **M5 knowledge_update / temporal** | bi-temporal edges + **supersession (G0 logical_id)** | leaf + valid-time filter | **consolidation/recency judgment** (OPP-2 provider) then rerank | needs latest-fact disambiguation, not coverage |
| **M6 global sensemaking (quality)** | none (reads raw) | **map-reduce QFS over all chunks (C)** | map (per-chunk extract, drop none) → hierarchical reduce | coverage-complete; **clean surpass *direction* vs full-strength GraphRAG (provisional — `NOT_REACHED` on power)** |
| **M6 sensemaking (cheap, amortized)** | **depth-1 coverage index (D2)** → coverage rows (intended kind-tag; eval-side today) | CPU-ANN over coverage summaries (±drill to members) | retrieve coverage nodes → reduce | cheap/CPU query, **but below full-strength GraphRAG comp** → needs richer structure or re-scope |
| **M7 multi-hop / entity-relationship** | entity/relationship + (Fork E) Leiden | iterative retrieval (caller-side) **OR** `search_expand` (G6) | **graph traversal refuted for recall** → use as evidence-linking only | PPR-over-graph refuted twice; Fork E re-opened only for entity-rich corpora |
| **M1 fidelity** | 1-bit sign-quant vectors | ANN fan-out K, f32 rerank | **quant path**: whitening → K>192 → 2-bit | eu7 recovery is a quant problem, NOT pooling (bisect = case-A) |
| **M4 relevance** | embedder choice (CLS-corrected bge-small) | dense arm | (no swap helps — 15a) | embedder-bound ceiling; a stronger ONNX/GPU embedder is the only lever (now permitted) |

**The router contract (the new in-library or agent-side dispatcher):** classify query intent →
{needle | multi_session | update/temporal | global-sensemaking | multi-hop} → select the row above →
emit the (index, retrieval, stack). The agent **chooses**; FathomDB **exposes** each mechanism as a
selectable function with its cost/latency tier surfaced.

## 5. Next steps to parity-or-better (sequenced, per measure)

**Tier A — ship/bank the provisional wins (cheap, highest-value):**
1. **M5:** **ship the parity win from the provisional evidence** — **expose tuned α (≈1.0) + pool_n in engine
   `ce_rerank`** (it is hardcoded 0.3) so the Mem0-parity result is *shippable*, not eval-only; re-clear eu7 if
   a re-embed is triggered. **Do NOT** spend to finish/register the incomplete α=1.0 run — the 0.8.3 HITL
   ruling was **take-as-is, no further priced spend** (`runs/0.8.3-resolution-verdict.md`); a fresh
   registration run would require a **new** HITL spend decision that supersedes it.
2. **M6:** **C's `decide_084` registration is corpus-blocked, not a spend question** — the gating re-run
   already used **N_Q=200 (the AP-News corpus maximum)** and still returned `NOT_REACHED` (comprehensiveness
   MDE 0.058 > ε); because the bootstrap clusters *by question*, more runs cannot tighten it — **only a
   larger/different question corpus, or a `decide_084` rule change, would** (`runs/0.8.4-gating-rerun-RESULT.md`).
   So registering C = acquire/build a larger entity-rich question set (ties into the gating-rerun corpus work),
   not another N=200 run. Independently, **productize C** as a `query --global` mode with the per-query cost
   surfaced (cost-tier opt-in).

**Tier B — close the open gaps:**
3. **M1:** the **quant path** (whitening → K>192 → 2-bit) to re-clear eu7 ≥ 0.90.
4. **M5 surpass / M2:** the **recall lever** — wider candidate-gen → CE-rerank (recall + ranking compose),
   D2-as-router for multi_session, PRF, fielded FTS/BM25F (F5). Test as accuracy arms on the 0.8.3 harness.
5. **M6 cheap product:** D2 needs a **richer coverage structure** (multi-level, toward but not necessarily
   reaching Fork E) **or** re-scope the product claim to "C-quality at a cost vs D2-cheap below-parity."

**Tier C — settle the unknowns:**
6. **M7:** measure **HippoRAG-2** (the only honest way to answer the multi-hop competitor); FathomDB's own
   graph multi-hop stays refuted. **Fork E** (entity/Leiden) only if an entity-rich corpus shows a gap C/D2 can't close.
7. **M5 update/temporal:** the **consolidation/recency provider** (OPP-2), gated on the lossiness-vs-latency value test.

**Tier D — the meta-unlock:**
8. **Build the portfolio dispatcher** (query-intent → index/retrieval/stack). This is what makes "the agent
   chooses" real: one store, kind-tagged indexes, a router that picks the winning mechanism per measure and
   surfaces its cost/latency tier. Everything above becomes a selectable arm under it.

**Sequencing rationale:** Tier A is mostly **engineering, not priced spend** — it *ships* the Mem0 parity win
(the engine α retune, currently eval-only) and productizes C. Note the registration gaps are **not** closable
by more spend on the current corpora: M5's α=1.0 run is closed under the take-as-is ruling, and M6's
`decide_084` is corpus-capped at N=200 — both need a *new decision* (HITL spend / a larger question corpus /
a rule change), not a routine re-run. Tier B/C are the surpass + open-gap work. Tier D is the architecture
that lets all of it coexist without a common-path compromise.

## 6. Risks / honesty guards

- **Provisional ≠ registered.** M5 α=1.0 and M6 C are both `NOT_REACHED` on power (underpowered/incomplete).
  Direction is robust; do **not** state either as a fully-powered citable claim until registered.
- **D2 surpass was a level-0 artifact.** Do not re-assert "D2 = product, Leiden unneeded" — the gating re-run
  refuted it; Fork E is re-opened.
- **Recall ≠ a free win.** More candidates without the reranker = more distractors → *hurts* M5. Recall only
  pays when stacked under CE-rerank.
- **Graph stays refuted for retrieval.** Any graph work (G6/Fork E) must clear a corpus-specific bar; default is off.
- **Lossiness penalty is measured.** C's map step, D2's summaries, and any consolidation LLM all distill —
  the 0.8.3 blind-distiller HURT (−0.362). Isolate the mechanism benefit from content lossiness in every value test.

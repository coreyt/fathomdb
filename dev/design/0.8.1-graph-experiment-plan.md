## 1. Principles (the methodology that keeps us honest)

1. **Two metrics, two loops.**
   - **Retrieval Recall@K vs `answer_session_id`** — LLM-free, deterministic, fast.
     This is the **design-iteration** instrument; base-retrieval, the graph-arm
     fix, coverage, fact-surfacing, and rerankers are all measured here with **no
     answerer**.
   - **End-to-end answer accuracy** (airlock reader + LLM judge) — slower, the
     **gate/validation** metric, run only at decision points.
   Most work runs in the cheap loop; airlock tokens are spent only at gates.
   **Recall@K validity caveats (must be built into the scorer):** (a)
   **multi_session** questions need *all* gold sessions → score "hit" only when the
   **full gold set** is in top-K, not "≥1" (the naive any-hit definition
   mis-measures exactly the class graph is meant to help); (b) **abstention**
   questions have no gold session → exclude from Recall@K and score separately;
   (c) recall@K is an **upper bound** on retrieval's end-to-end contribution (right
   session retrieved, reader may still fail). Because the §6 headline is
   reader-dependent, **validate the recall@K ↔ end-to-end-accuracy correlation early
   (a G1 AC)**, not only at G2 — otherwise we may optimize a dimension that doesn't
   move accuracy.
2. **Match the mechanism to the benchmark.** Evaluate each design bet on the
   benchmark its mechanism actually targets. Fact-edges / temporal / base-retrieval
   → LME. Community summaries → a sensemaking eval (**not** LME) → parked. Judging a
   community tier on LME would falsely condemn it.
3. **Fix only what blinds the instrument, then experiment, then fix what
   experiments prove.** Three fix categories: (a) instrument-blinding (mandatory,
   first); (b) experiments; (c) feature fixes — done *only* when an experiment shows
   the payoff. No speculative graph features before the arm earns its keep.
4. **Always paired, always fixed sample.** Compare configs on the **same** fixed,
   stratified question set (paired McNemar / per-question deltas), never on
   independently drawn sets — paired tests are far more powerful here (§3).
5. **Never shrink the per-question haystack.** Task difficulty is set by the
   per-question distractor count (~48 sessions/Q — measured, §3.2), not by the
   19,195 union. Iterate by subsampling **questions** (with their full haystacks),
   not by truncating the corpus.
6. **Tracer-first (cheapest-validation-first / fail-fast).** Before *any* sized or
   expensive run (embedding, extraction, multi-question smoke), a **capability
   tracer** must prove — at minimal cost (seconds, pennies) — that the instrument
   can actually emit **each** signal the run depends on. One tiny call per
   capability. **No embedding/extraction/sized run until the tracer is green.**
   *(This principle exists because the first P0-A attempt embedded ~760 sessions
   (~25 min) before discovering the dense arm was unreachable from the SDK and the
   gpt-5 reader was rate-limited — both detectable in one call each. See §8.)*

## 3. Smallest-but-valid iteration set (the math)

**Goal:** the smallest question set that still yields *valid* design signal, so the
iterate loop is fast and cheap.

### 3.1 Two independent knobs — only one is safe to shrink

- **Per-question haystack size** (difficulty). LME `s_cleaned` gives each question
  ~38 sessions (gold evidence + ShareGPT/UltraChat distractors). **Shrinking this
  changes the task** (recall over 38 sessions ≠ recall over 5) → **hold it full.**
- **Number of questions** (statistical precision). Subsampling questions widens
  confidence intervals but **does not change difficulty** → **this is the safe
  knob.**

### 3.2 Sessions to extract — scales with questions, but the multiplier MUST be measured

**MEASURED (P0-A tracer, 2026-06-14):** per-question haystack `H` = **48.1 mean**
(min 43, max 57) — *higher* than the naïve 19,195÷500 = 38.4 (that ratio was just
union÷count, which carries no overlap information). The smoke's 16-question union
was 760 (union÷count = 47.5 ≈ `H`) → **haystacks are near-disjoint** (distractor
pools are not shared). So a K-question subset needs ≈ **48 × K** sessions:
**K=150 → ≈ 7,200 sessions** (≈100% coverage vs Option 3's ~1%), **not** the 5,760
first estimated — a ~25% undercount, now corrected throughout (§4). The direction
(coverage scales with questions, not the 19k union) holds; the multiplier is now
measured, not assumed.

### 3.3 Estimation precision — per-class recall CI

Per-class Recall@K is a binomial proportion `p` over `n` questions. Wald
half-width `h = 1.96·√(p(1−p)/n)` (Wilson is tighter at small n — use it in
practice). At the hardest-to-estimate `p≈0.4`:

| n (questions in class) | 95% CI half-width `h` |
|---:|---:|
| 23 | ±0.200 |
| 40 | ±0.152 |
| 50 | ±0.136 |
| 78 (KU @ full LME) | ±0.109 |
| 133 (temporal / multi_session @ full) | ±0.083 |
| 156 (factoid @ full) | ±0.077 |

### 3.4 Decision power — paired comparison (McNemar)

Design decisions compare two configs on the **same** questions → **paired** data;
use McNemar on the discordant pairs (far more powerful than two independent
samples). Sample size to detect a marginal-recall difference `δ` at 80% power,
α=0.05, with discordant-pair rate `p_d`
(`n = (z_{α/2}√p_d + z_β√(p_d−δ²))² / δ²`):

| effect `δ` (per class) | assumed `p_d` | n/class needed |
|---:|---:|---:|
| 10 pp | 0.25 | ≈ 194 |
| 15 pp | 0.25 | ≈ 85 |
| 20 pp | 0.30 | ≈ 56 |

For contrast, the **independent** two-proportion test needs ≈384/class to detect
10 pp (recomputed: ≈385). Paired efficiency vs that baseline depends entirely on
the discordance rate `p_d`: `p_d`=0.10→**5.1×**, 0.15→3.3×, 0.20→2.5×, 0.25→2.0×,
0.30→1.6×, 0.40→1.2×. For near-identical retrieval configs (most of our A/Bs)
correlation is high and `p_d` is low, so the gain is likely **3–5×**, not 2× — but
**`p_d` is unmeasured and is the dominant uncertainty in this whole power analysis**
(n ∝ ~`p_d`/δ²; a true `p_d`=0.4 nearly doubles every requirement). **Measure `p_d`
from the first paired run** and recompute MDEs rather than trusting the 0.25/0.30
assumptions.

### 3.5 Conclusions

> **Estimation precision (CI, §3.3) ≠ detection power (paired MDE, §3.4).** The MDE
> to *detect a difference between two configs* is strictly **larger** than the CI
> half-width for estimating one config's recall. Decisions are detections → use the
> **MDE** column, not the CI column. (This distinction was the main error the
> adversarial review caught; the table below is the corrected basis.)

**Paired per-class MDE** (80% power, α=0.05, `p_d`=0.25 unless noted) vs the
single-proportion CI half-width, at the class sizes we actually have:

| n (class size) | CI half-width (estimate one config) | **paired MDE (detect a difference)** |
|---:|---:|---:|
| 40 (iteration, balanced) | ±15.2 pp | **≈ 21.5 pp** |
| 78 (KU @ full LME) | ±10.9 pp | **≈ 15.6 pp** (17.1 at `p_d`=.30) |
| 133 (temporal / multi_session @ full) | ±8.3 pp | **≈ 12.0 pp** |
| 156 (factoid @ full) | ±7.7 pp | **≈ 11.1 pp** |
| pooled 160 (iteration) | — | **≈ 11.0 pp** |
| pooled 500 (gate) | — | **≈ 6.2 pp** |

Conclusions (corrected):

- **`knowledge_update` (n=78) is the binding constraint.** Its paired per-class MDE
  is **≈16–17 pp even on the full benchmark** — LME can essentially never adjudicate
  a sub-15 pp KU graph lift. Any KU-driven go/no-go must respect this.
- **Iteration set: class-balanced ~40/class (≈160 Q), fixed, paired.** Per-class
  detection only for **≥~21 pp** moves; the useful iteration signal is the
  **pooled** metric (MDE ≈11 pp) + graded metrics. Class-balanced (not
  proportion-matched) for even per-class power. *Held-out:* keep KU's ~38 unused
  questions (and a held-out slice of each class) out of the iteration set for the
  gate (see §6 / finding below).
- **Gate set: held-out portion (see §6), reported at full-500 scale.** Per-class
  detection ≈**11 pp** (factoid) to ≈**16–17 pp** (KU); pooled ≈**6 pp**.
- **Honest floor:** **no per-class comparison on LME resolves below ~11 pp**
  (≥~16 pp for KU). The ~6–8 pp regime is reachable **only on the pooled metric or
  with graded scoring** (MRR/nDCG/Recall@K-curve, continuous → lower variance).
  Reranker micro-tuning moving recall 2–3 pp is **not resolvable** — judge it pooled
  or treat as noise. **Target big moves.**
- **Multiple comparisons:** P1 runs E1–E5 × ~4 classes × several configs ≈ 30–60
  tests at α=0.05. Apply **Benjamini–Hochberg / Bonferroni** (or confirm on
  held-out data) before adopting any per-class "signal" — given the high MDEs, most
  uncorrected per-class blips are noise.
- **Variance-reduction levers** (use all): paired McNemar; graded metrics; bootstrap
  CIs over questions; fixed sample across all runs.
- **Selection-bias guard:** draw the iteration set **once**, fixed seed, stratified;
  never re-draw per experiment. In P0, validate that iteration-set pooled recall
  tracks the full-500 baseline within CI before trusting it.

> multiple-comparison control (above).

## 4. Cost forecast

**Token assumptions:** session ≈ 12k chars ≈ **3,000 input tok**; extraction prompt
overhead ≈ 600 → **~3,600 in / ~800 out per session**. Answerer per question:
**~8,500 in / 200 out** (retrieved context). Judge: **~1,000 in / 100 out**.
Prices: CloudZero May-2026 list, $/Mtok in/out. **Airlock actuals may differ;
prompt-caching the static extraction prompt trims input further.**

### 4.1 What costs money, and when (the lifecycle)

| Activity | LLM? | One-time or per-run? | Reused by? |
|---|---|---|---|
| **P0-A / E2 base-retrieval ablation** | **No** ($0) | per-run, but free | — run as often as wanted |
| **Recall@K eval** (E0/E1/E3/E5 retrieval loop) | **No** ($0) | per-run, but free | — unlimited re-runs |
| **Extraction** (ELPS graph build) | Yes | **ONE-TIME per (coverage × extractor model)** — persisted in the DB, *reused* by every later recall@K run | all retrieval/graph runs on that DB |
| **End-to-end gate** (answerer + judge) | Yes | **per gate run** (~5–10 over the program) | — |

**Key:** extraction is the cost driver but is **amortized** — you pay once per
(coverage, extractor-model) and reuse it across unlimited LLM-free recall@K runs.
You only re-extract when you **change the extractor model or coverage level**.

iteration set**. The "oracle" extraction (E4) is a *separate* DB at Sonnet/Opus
quality (its own cost).

### 4.3 Embedding wall-clock (LLM-free, but the real bottleneck for fused runs)

Dense/fused needs local BGE embedding — **$0 in tokens but hours in wall-clock**.
The P0-A smoke embedded ~760 sessions and was the dominant cost (~25 min in the slow
attempts). **T4 (tracer) measures sessions/min precisely**; pending that, the
order-of-magnitude forecast is:

| Coverage | est. embedding wall-clock (range pending T4) |
|---|---|
| ~7,200 (iteration) | **~1–4 hours** |
| 19,195 (full corpus) | **~3–11 hours** |

**Implication:** embed **once per coverage level and reuse the DB** across all
recall@K runs; treat any full-corpus fused embed as a **scheduled batch**, never an
interactive step; and seriously weigh whether the gate even needs full-19k fused (vs
iteration set + held-out) given the §3.5 power limits. This is now a first-class
planning constraint, surfaced by the tracer rather than discovered mid-run.

### 4.5 Data lifecycle — runs do NOT overwrite each other

- **Extraction → one DB file per (extractor-model × coverage)**, named by config
  (e.g. `lme_haiku_5760.sqlite`). Re-extraction with a different model writes a
  *separate* DB; nothing is clobbered, and configs stay comparable side-by-side.
- **Eval outputs → versioned JSON per run** (config + timestamp in the filename);
  never overwrite, so every ablation is reproducible and diff-able.

### 4.6 Program total (order of magnitude)

P0.0 tracer = **pennies**. P0-A + all recall@K iteration = **$0** in tokens
(LLM-free; but **hours of local embedding wall-clock**, §4.3). One iteration-set
extraction (~$15 Flash-Lite / ~$55 Haiku, at H≈48). ~5–10 end-to-end gate runs
(~$7–16 each ≈ $50–160). Optional: full-corpus extraction ($40–146), oracle E4
extraction (Sonnet ~$164 / Opus ~$274 at the iteration size). **→ low hundreds of
dollars total**, extraction-dominated; the iteration loop is token-free but
embedding-wall-clock-bound.

     unverifiable at this n; do not state it). KU is reported but, per §3.5, cannot
     gate sub-15 pp moves. Use **graded metrics** (MRR/nDCG) for the finer reads.
  2. **End-to-end headline (memory + named reader):** LME accuracy with a **named
     strong airlock reader**; **target 90%** (per HITL), explicitly attributing the
     reader's contribution (the 90%+ leaderboard systems all use Gemini-3-Pro /
     GPT-5-mini-class readers — `longmemeval-leaderboard-reference.md`).

    `p_d` is measured** (§3.4). Apply multiple-comparison correction (§3.5).
  - If R3 regresses any class beyond its MDE → capture + HALT to HITL; do **not**
    flip `use_graph_arm` default to `true`.
  - **Held-out graph-arm extraction (DECIDED 2026-06-14 → option b).** Adjudicate
    **graph-vs-base on the *iteration* set** (already extracted); use the held-out
    500 only for the **base/headline** number. This **avoids a ~$120 second
    near-full-corpus extraction** (held-out haystacks are near-disjoint from the
    iteration set, H≈48). The graph delta is therefore measured on the
    tuned-on iteration set — acceptable because the graph-vs-base comparison is
    paired/relative; the unbiased *headline* still comes from held-out base.

   default *local-class* extractor, with **Opus/Sonnet as the E4 oracle**? (Flash-

   90% target? **Cost per 500-Q gate (§4.4):** Haiku ≈$7, Gemini 3.1 Pro ≈$12,
   GPT-5.4 ≈$14, Sonnet ≈$16, Opus ≈$26. *Effect on the score:* the reader is a
   large confound — the leaderboard's 90%+ systems all use Gemini-3-Pro/GPT-5-mini-
   class readers (Mastra moved 84→95 on identical memory by swapping readers,
   `longmemeval-leaderboard-reference.md`). **Recommendation:** pin one **strong
   reader** (Gemini 3 Pro or GPT-5.4) as the headline anchor *and* run **one cheap
   reader** (Haiku) as a control, so we can separate FathomDB's retrieval delta
   (reader-held-constant) from the reader contribution.
3. **Retrieval-gate thresholds** — confirm "set after P0-A re-baseline **and `p_d`
   measured**," then lock per-class pp targets bounded by the §3.5 MDEs (~11 pp good
   classes, ~16–17 pp KU); rely on pooled/graded for finer reads.
4. **Iteration-set composition** — class-balanced ~40/class (≈160 Q) vs
   proportion-matched (factoid 47 / temporal 40 / KU 23 / multi_session 40 for 150).
   *How it affects scores & understanding:* composition does **not** change a
   class's recall *value* (recall is a within-class proportion); it changes **per-
   class precision** and the **pooled aggregate**. Class-balanced → **equal power
   per class**, so the hard classes graph targets (temporal/KU/multi_session) get
   tighter CIs (KU 40 vs 23 → MDE ~21 vs ~28 pp); the pooled number is *not*
   distribution-representative. Proportion-matched → the pooled number tracks the
   headline LME distribution but **starves KU** (23) of power. Because design
   decisions are per-class and the **headline always comes from the full-500 gate**
   (proportion-matched by construction), class-balanced iteration gives cleaner
   design signal **without** biasing the reported number. **Recommendation:
   class-balanced for iteration, full-500 (held-out) for the gate headline.**

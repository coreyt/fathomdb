## R-REL-4b ‚Äî exercised verification (REAL local-registry round-trips)

A bare `cargo-publish-if-new.sh --dry-run` SKIPS the dependent Rust crates, PyPI,
smoke, and co-tagging, so it does not exercise #11-full. This slice exercises the
publish state machine against **REAL local registries** (a genuine
publish ‚Üí index-sees-it ‚Üí re-run-no-ops ‚Üí install round-trip), not argv-recording
shims. Real vs simulated is stated per registry:

- **npm ‚Äî REAL** (`scripts/tests/test_idempotent_republish.sh`). A minimal but
  faithful in-process npm registry implements the publish/packument/tarball HTTP
  protocol (publish `PUT`, packument `GET`, tarball `GET`). `npm publish` really
  uploads the thin main
  `fathomdb` package AND `@fathomdb/fathomdb-linux-x64-gnu` through
  `npm-publish-if-new.sh`; the tarballs are stored and served back with the
  integrity npm computed. A re-run **no-ops** via the helper's registry-query
  path (npm is NOT re-invoked ‚Äî a server-side PUT counter proves this is not
  vacuous). A real `npm install` then pulls the os/cpu-gated platform package,
  and the REAL loader (`src/ts/src/platform.ts` `loadPlatformBinding`) resolves
  it on this linux-x64 host and throws `UnsupportedPlatformError` for a foreign
  host (real `require`, not a mocked seam).
- **PyPI ‚Äî REAL** (`scripts/tests/test_pypi_publish_roundtrip.sh`). A minimal
  real PyPI index (stdlib `http.server`: legacy upload API + JSON API) accepts a
  genuine `twine` upload of a real sdist through `pypi-publish-if-new.sh`
  (exercising Fix-1's `--repository-url` routing); a re-run **no-ops** via the
  helper's query-sees-it path (twine NOT re-invoked ‚Äî upload counter proves it).
  twine 6.x refuses `--skip-existing` against a non-prod repository, so the test
  uses/self-provisions twine<6; if no twine<6 and no pip/venv/network are
  available it **SKIPS LOUDLY** (documented residual) rather than passing a shim
  as real.
- **crates.io ‚Äî SIMULATED** (documented residual; `test_idempotent_republish.sh`
  crates leg + `test_cargo_publish_if_new.sh`). A real local crates registry
  (sparse index + `cargo publish` `/api/v1/crates/new` endpoint + auth/on-disk
  index) is infeasible in this harness. The crates leg keeps an offline
  http-fixture for the registry QUERY + a publish-tool shim (branch logic +
  query-registry routing only). The REAL crates publish path is covered at
  real-tag time by `cargo publish`'s own verify/idempotency.
- **Fix-1 publish-registry safety** (`scripts/tests/test_publish_registry_safety.sh`).
  A run that overrides the query registry must never publish to prod:
  `npm publish` is pinned to `--registry "$BASE"` and `twine upload` to
  `--repository-url` derived from the query host (`upload.pypi.org` is used ONLY
  when no override is set). RED-first: these assertions fail against the pre-Fix-1
  helpers (which published with no routing flag).
- `test_wait_for_crate_version.sh` ‚Äî poll/timeout/manifest-resolution path;
  `test_release_workflow_scope.sh` ‚Äî static `release.yml` structure (matrix
  scope, ordering, dist-tag, split jobs); `verify-release-gates.sh` exits 0;
  `set-version.sh --check-files` keeps BOTH version axes (W + E) consistent.

**Live CI PyPI path vs local script path (intentionally distinct, both
idempotent).** In `release.yml` the PyPI leg publishes via
`pypa/gh-action-pypi-publish` with `skip-existing: true` (its OIDC
trusted-publishing path cannot be replaced by a shell `twine` call). The
`pypi-publish-if-new.sh` script is the locally-exercisable equivalent + the
manual/emergency-republish fallback; it is what the idempotency test drives.
Both paths are idempotent: the gh-action no-ops an already-present file via
`skip-existing`; the script no-ops via its registry query (primary) with
`twine --skip-existing` as same-second-race defence-in-depth. The tested
script-path idempotency therefore reflects the live gh-action behaviour. (npm
and crates use the SAME `*-publish-if-new.sh` scripts in CI and in the test, so
there the tested path IS the live path.)

**What a real `v*` tag (Slice 40) would additionally exercise** against live
registries: real `cargo publish` uploads T1..T7 (immutable), real PyPI trusted-
publish, real npm OIDC publish of the platform package then the main package,
the post-publish install smokes, co-tagging, and the GitHub Release. Those
require live credentials/registries and are HITL-gated at Slice 40.

## R-REL-4c ‚Äî coordinated-publish resilience (NOT atomicity)

True cross-registry atomicity is impossible (crates.io is immutable; npm/PyPI
reject re-uploads). The contract for the 0.8.20 OPP-12 breaking-pair publish is
resilience:

- **Preflight** ‚Äî `verify-release-gates.sh` + `verify-embedder-api-no-drift.sh`
  - `all-builds-passed` gate every publish; nothing uploads until all builds pass.
- **Ordered commit points** ‚Äî T1..T7 run strictly in dependency order
  (`t(N)` needs `t(N-1)`; `t1` needs `all-builds-passed`); tier-8 npm/PyPI run
  after T5 (engine). Asserted by `test_release_workflow_scope.sh`.
- **Per-registry retry/no-op idempotency** ‚Äî a re-run no-ops on already-published
  versions: crates.io via `cargo-publish-if-new.sh`; npm via
  `npm-publish-if-new.sh`; PyPI via `skip-existing: true` +
  `pypi-publish-if-new.sh`. Proven by REAL local-registry round-trips (npm in
  `test_idempotent_republish.sh`; PyPI in `test_pypi_publish_roundtrip.sh`;
  crates simulated ‚Äî see R-REL-4b).
- **Poll-for-resolvability** ‚Äî `wait-for-crate-version.sh` replaces the fixed
  `sleep 60`: it polls the index until the just-published version resolves
  (bounded timeout ‚Üí fails loudly on a genuine stall).
- **Rollback-forward** (not rollback-back) ‚Äî crates.io versions are immutable and
  cannot be un-published, so recovery is always *forward*: fix the failed leg and
  re-run; idempotent helpers no-op the already-landed legs and complete the
  missing ones. A partial landing is recovered by re-running the same tag, never
  by trying to delete/rewind a published version.

## R-REL-4f ‚Äî npm per-platform split topology

- **Thin main `fathomdb` package** ‚Äî ships `dist/` only (NO `.node`). Its loader
  (`src/ts/src/platform.ts`, used by `src/ts/src/binding.ts`) resolves the host
  triple, tries a local dev `.node` then `require("@fathomdb/fathomdb-<triple>")`,
  and throws `UnsupportedPlatformError` ("unsupported platform") when neither is
  present ‚Äî never a silent runtime segfault. Contract test:
  `src/ts/tests/npm-platform-split.test.ts` (`npm_platform_split_contract`).
- **Per-platform binary packages** ‚Äî `src/ts/npm/<triple>/package.json` named
  `@fathomdb/fathomdb-<triple>` with `os`/`cpu`(/`libc`) + the one `.node`. npm
  installs only the host-matching one.
- **optionalDependencies** ‚Äî injected into the main package.json **at publish
  time** by `scripts/release/npm-inject-optional-deps.sh` (napi-rs idiom), one
  entry per published `src/ts/npm/<triple>/` directory. Committing them would
  desync `package-lock.json` (the platform packages are unpublished at dev time)
  and break every `npm ci`; injection keeps the committed tree `npm ci`-clean
  while the PUBLISHED package carries the contract.
- **0.8.18 publishes ONLY** `@fathomdb/fathomdb-linux-x64-gnu` + the main
  package, under a **non-`latest` dist-tag** (`NPM_DIST_TAG`, default `next`;
  HITL confirms the exact label). Serving `latest` with linux-only coverage
  would install-then-crash on mac/win. The follow-on publishes the remaining
  platform packages and promotes `latest`.

## R-REL-4d ‚Äî deferred to the follow-on orchestrator (noted, not dropped)

The cross-platform matrix (aarch64 / darwin / windows / musl, py + napi) stays
SUPPORTED but OFF the critical path. The deferred matrix legs live as inline
comments in `release.yml`; the deferred platform packages are added as sibling
`src/ts/npm/<triple>/` directories by the follow-on (auto-picked-up by the
injection script). No CI matrix expansion in 0.8.18.

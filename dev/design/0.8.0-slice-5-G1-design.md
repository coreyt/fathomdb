# 0.8.0 Slice 5 — G1 Structured Search Hits + Global FTS5 Tokenizer Upgrade (design memo)

**Status:** design-first (pre-RED). Authoritative contract:
`dev/plans/0.8.0-implementation.md` § "Slice 5". ADR:
`dev/adr/ADR-0.8.0-canonical-identity-substrate.md` (interim `id = write_cursor`).

This slice reshapes both retrieval branches to emit a structured `SearchHit`,
changes `SearchResult.results` from `Vec<String>` to `Vec<SearchHit>`, drops the
illegal `Eq` derive (now that a hit carries `score: f64`), and folds in the
zero-surface global FTS5 tokenizer-default upgrade via a new forward migration.
Py + TS parity in lockstep. AC-057a-clean: enriches `search()`, adds no verb.

## 1. `SearchHit` shape

```rust
#[derive(Clone, Debug, PartialEq)]   // NOT Eq — score is f64
pub struct SearchHit {
    pub id: u64,                  // interim = write_cursor (swaps to logical_id at Slice 15, no reshape)
    pub kind: String,
    pub body: String,
    pub score: f64,
    pub branch: SoftFallbackBranch,
}
```

`SearchResult` becomes `#[derive(Clone, Debug, PartialEq)]` (Eq dropped — a real
compile failure that gates everything downstream) and `results: Vec<SearchHit>`.

### Interim-`write_cursor`-id decision

`id` carries the canonical row's `write_cursor`. Per the canonical-identity ADR,
`write_cursor` is the **interim** identity carrier; Slice 15 swaps it to
`logical_id` with **no carrier reshape** (the `u64` field stays; only its source
changes). We do **not** introduce `logical_id` here.

## 2. Per-branch score semantics

- **Vector phase-1** (`read_search_in_tx`): score = `vec_distance_l2(embedding,
  query)` — the same f32 rerank distance already computed in the phase-2 ORDER BY.
  Lower is closer. We surface it directly. `branch = Vector`.
- **FTS phase-2**: score = `bm25(search_index)`. SQLite's `bm25()` returns a
  *negative* relevance where more-negative = more-relevant. We surface it raw.
  `branch = Text`.

**The two scores are NOT comparable raw.** Normalizing distance-vs-`bm25()` into a
single comparable scale is a **Slice-10 RRF concern**, explicitly out of scope
here. This slice only *carries* `score: f64` so Slice 10's RRF has a field to land
on. Ordering within this slice is unchanged from 0.7.2 (see §3).

## 3. Dedup-on-body + vector-first ordering (PRESERVED, byte-for-byte)

The 0.7.2 merge is: vector hits first (in vec-distance order), then FTS hits, with
a `seen`-set dedup keyed on **body** so a row surfaced by both branches appears
once (vector-first wins). We preserve this exactly:

- vector hits pushed first, deduped on `body`;
- FTS hits appended, skipped if their `body` was already `seen`.

The dedup key stays `body` (not `id`) so behavior is identical to 0.7.2 and the
recall harnesses measure the same ordering. `branch` is tagged per surviving hit.

## 4. Tokenizer-migration mechanism (NEW forward step — do NOT edit step 5)

Migrations are forward-only and immutable (`apply` runs only `step_id > before`;
FTS5 has no `ALTER … tokenize`). Editing the existing step-5 DDL would change the
tokenizer for **new DBs only**, leaving migrated DBs on `unicode61` — defeating the
global upgrade and making the recall delta meaningless on a real corpus.

So we **append a new `step_id: 11`** to `MIGRATIONS` and **bump `SCHEMA_VERSION`
10 → 11**. (Plan-adjustment, recorded: the contract says "next step_id"; witnessed
max is 10; the open-time guard `before > SCHEMA_VERSION → IncompatibleSchemaVersion`
plus the contiguity check `step_id == current+1` force version 11. Slice 15's ADR
text still says "step 11 / 10→11" — that is now stale; the orchestrator renumbers
Slice 15 to step 12 / 11→12. We do not touch Slice 15.)

```sql
-- MIGRATION-ACCRETION-EXEMPTION: tokenizer-default upgrade (drop+recreate FTS5 projection; no source-record migration)
DROP TABLE IF EXISTS search_index;
CREATE VIRTUAL TABLE search_index USING fts5(
    body,
    kind UNINDEXED,
    write_cursor UNINDEXED,
    tokenize = 'porter unicode61 remove_diacritics 2'
);
```

`DROP TABLE` already satisfies `check_migration_accretion`'s `names_removal`
branch; the exemption marker is carried anyway to document intent and match the
established pattern.

### Re-tokenization wiring (the load-bearing part)

Dropping + recreating `search_index` leaves it **empty** on a migrated DB. The
projection scheduler will **not** auto-repopulate it: `database_has_pending_
projection_work` keys "pending" off `_fathomdb_projection_terminal` (by
`write_cursor`), which the migration does **not** clear — so already-projected
rows look "done" and the FTS index stays empty. This is precisely the
no-op-on-existing-DB failure mode the contract pins RED.

**Fix:** in `open_locked`, immediately after `migrate_with_event_sink` returns,
if the migration crossed the step-11 boundary (`schema_version_before < 11 &&
schema_version_after >= 11`), synchronously re-tokenize `search_index` from
`canonical_nodes` (the FTS-only half of `rebuild_shadow_state`) on the writer
connection, single-threaded, before readers spawn. This is projection-only
(it reads canonical source rows and rewrites the FTS shadow); **no source-record
migration** occurs (canonical rows are untouched). It is idempotent: re-open is a
no-op because `schema_version_before` is then already 11.

## 5. Engine fan-out

- `SearchHit` struct + `SearchResult` reshape (`lib.rs` ~819–823).
- `ReaderResponse` alias (`Vec<String>` → `Vec<SearchHit>`).
- `read_search_in_tx` return + body: vector branch now SELECTs `write_cursor,
  kind, body` (+ carries the rerank distance as score); FTS branch SELECTs `body,
  kind, write_cursor` + `bm25(search_index)`.
- `search_inner` destructure of the reader tuple (no logic change beyond the type).
- Re-tokenization helper invoked from `open_locked` (recorded as in-scope fan-out).

## 6. SDK parity (Py + TS, same slice)

- **Py:** `PySearchHit` pyclass (`id, kind, body, score, branch`);
  `PySearchResult.results: Vec<PySearchHit>`; `.pyi` stubs.
- **TS:** `SearchHit` napi object; `SearchResult.results: SearchHit[]`;
  `binding.ts` `NativeSearchHit` + `index.ts` public `SearchHit` interface + mapper.

## 7. Consumer rework

Every `.results` consumer reads `hit.body` (mechanical API-tracking migration, not
a spec-test edit). Includes the recall harnesses `eu7_real_corpus_ac.rs` and
`eu8_ir_validation.rs` so the floor is measured identically before/after, plus the
`engine_open_emits_migration_step_events` test (now expects step 11), and the
Py/TS binding `from_rust` mappers.

## 8. Test plan

- **`pr_g1_search_hits.rs`** (RED first): `results` is `Vec<SearchHit>`; each hit
  has `id == write_cursor`, populated `kind`/`body`, finite `score`, correct
  `branch`; `SearchResult` no longer derives `Eq` (compile-level — covered by the
  type assertions); dedup-on-body + vector-first order preserved. Uses a
  deterministic in-process `Embedder` (no network) so both branches exercise.
- **`pr_g1_tokenizer_recall.rs`** (RED first): builds a corpus on a DB **migrated
  from SCHEMA_VERSION 10** (open with steps 1–10, ingest+drain, close; re-open with
  the full set so step 11 runs), then asserts FTS recall ≥ 0.90 on known
  query→relevant-doc pairs **after** the migration. If the tokenizer change only
  affected fresh DBs (i.e. re-tokenization not wired), the migrated DB's FTS index
  is empty and recall is 0 → RED. Also captures the pre-migration (v10/unicode61)
  recall on the same corpus for the before/after delta.

The real-corpus recall harnesses (eu7/eu8) are gated behind `AGENT_LONG` + a real
network embedder and are not part of `agent-verify`; `pr_g1_tokenizer_recall.rs`
is the deterministic, in-suite floor gate that runs every `cargo test`.

## 10. Guardrails

Recall floor ≥ 0.90 must hold **before and after** the tokenizer change, measured
across the migration. A sub-0.90 breach is a substantive blocker → record + STOP
(orchestrator HALTs to HITL). `id = write_cursor` interim only; no `logical_id`.

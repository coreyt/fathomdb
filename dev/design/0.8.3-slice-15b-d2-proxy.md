# 0.8.3 Slice-15b — D2 content-at-scale proxy (design + TDD contract)

> **Orchestrator draft for implementer (TDD) + codex review.** Grounds the §5-frozen 15b probe into an
> executable, deterministic, $0 measurement. Gate semantics are frozen in
> `decision_rule_083.probe_15b_pass` — this slice CONSUMES it and must not reimplement or relax it.

## 1. Purpose, gate, and the D0b reprioritization

**Purpose.** Decide — cheaply, before any schema migration — whether **fielded enrichment content**
(entity/alias/fact/temporal keys appended to a doc's FTS content) carries *retrieval* value at scale,
separated from the length artifact. Gate output: **pass ⇒ Slice 25 (D2 engine) is eligible; FAIL ⇒
Slice 25 defers/drops** (no migration on an unvalidated lever).

**Frozen pass criterion (`probe_15b_pass`, design §5).** Enrichment *passes* iff BOTH:
1. `margin_ci_lo > 0` — paired (enriched − length-matched placebo) recall margin, 95% bootstrap CI
   **lower bound > 0** (content beats placebo *at power*, not just on the point); AND
2. `removes_length_norm_penalty == True` — fielding/`b`-tuning removes the FTS length-norm penalty the
   placebo exposes.
The harness builds `enriched = {recall, margin_ci_lo, removes_length_norm_penalty}` and
`placebo = {recall}` and calls `probe_15b_pass(enriched, placebo)`. **No raw-point gate** (mirrors the
Slice-0 fix-1 discipline — CI lower bound only).

**D0b reprioritization (load-bearing context, surface honestly).** D0b measured FathomDB **−20…−27 pp
accuracy vs Mem0** on 3 classes, with the **accuracy gap ≫ the recall gap** (e.g. factoid: recall
−0.10 but accuracy −0.24) — which points at **answer-formation from raw bodies**, not only retrieval
recall. 15b measures a *retrieval-recall* lever, so a **15b pass is necessary-but-not-sufficient**: it
tells us D2 is worth building as a recall lever, but D2 alone will not close a 20-pt accuracy gap. The
orchestrator runs the **gap-decomposition probe** (see the parity-path note) alongside 15b so we do not
over-invest in a recall lever if the gap is answer-formation. This split is recorded in the output.

## 2. Method (reuse `r6_index_key_enrichment.py`, extend to the frozen probe)

- **Corpus/arms:** the LME retrieval corpus + the three FTS arms r6 already builds — `fathomdb_fts`
  (plain), `fathomdb_fts_enriched` (entity/fact keys appended to each doc's own FTS content — one row
  per doc, NOT a graph arm), `fathomdb_fts_placebo` (length-matched, **foreign-only** vocab, excludes
  the doc's own entity tokens; r6 `placebo_doc`). All LLM-free at measure time, deterministic.
- **Enrichment keys (OFFLINE-BUILD, $0):** reuse cached 0.8.2 extractions where applicable
  (`data/corpus-data/graph-cache/0.8.2-m1-v1/extractions.json`); extract LME (and, if cheaply derivable,
  the IR corpus) session keys with the **local Qwen** airlock (OFFLINE-BUILD, $0 — NOT a priced model).
  GPU is currently idle (D0b done); **coordinate before using it** (the user's rule: only use GPU at
  <5% util; check no sibling needs it). If extraction is the bottleneck, the LME-only proxy is the
  primary; IR-corpus enrichment is a reserved follow-on.
- **"At scale":** run on the **power-sized** LME gold (per-class N from Slice 5; reuse the same
  re-pinned gold so the proxy is comparable), not the n=10 smoke.
- **Metrics:** strict Recall@K per class + pooled; the **paired (enriched − placebo)** per-query margin
  with a fixed-seed bootstrap CI (same helper/shape as Slice 15a's `paired_bootstrap_ci`).
- **`removes_length_norm_penalty`:** the placebo (pure length, no content) exposes the FTS length-norm
  penalty (longer docs rank differently for no content reason). The flag is **True** iff, after
  enrichment (and/or the r6 `--tune-b` lower-`b` length-norm path), the enriched arm does **not** inherit
  the placebo's length-driven distortion — operationalized as: `enriched_recall − placebo_recall > 0` at
  power (criterion 1) **and** the placebo does not itself beat plain FTS (i.e. length alone is not what
  helps) / the tuned-`b` setting neutralizes the placebo's length advantage. The implementer pins the
  exact deterministic test from r6's placebo+tune-b outputs; the design requires it be a **mechanical,
  reproducible** boolean, not a judgment call.

## 3. TDD plan (RED → GREEN, backend-light)
**RED (commit first):**
- (a) probe wiring: build `enriched`/`placebo` dicts and call the REAL `probe_15b_pass`; truth-table
  (pass / fail-on-margin-CI / fail-on-length-norm-flag) over synthetic known values. Must call the real
  frozen function; no raw-point gate.
- (b) length-matched placebo invariant: a fixture asserting the placebo is token-length-matched +
  foreign-only + excludes the doc's own entity tokens (reuse/extend `test_r6_enrichment.py`).
- (c) paired bootstrap CI: fixed seed, deterministic; +delta ⇒ ci_lo>0, zero/neg ⇒ ci_lo≤0; known value.
- (d) `removes_length_norm_penalty` computation: deterministic boolean on fixed recall fixtures.
**GREEN:** the runner (extend r6) producing the outputs; `--smoke` green end-to-end (small per-class).
Tests run with `FATHOMDB_TESTS_NO_REBUILD=1 PYTHONPATH=src/python` under the canonical `.venv`
(pure-logic) — the heavy FTS/extraction path is the smoke/full run, not the unit gate.

## 6. Risks / open items
1. **The gap may be answer-formation, not recall** (D0b accuracy≫recall) → a 15b pass still may not
   move parity; that's why the gap-decomposition probe runs alongside (§1).
2. **Extraction footprint:** LME/IR enrichment may need a Qwen pass (GPU/OFFLINE-BUILD); reuse cached
   first; coordinate GPU; LME-only is the primary, IR is a reserved follow-on.
3. **`removes_length_norm_penalty` must stay mechanical** — pin it to r6 placebo/tune-b outputs, not a
   judgment call (codex to verify).
4. Graph-arm framing is OUT (M1 NO-GO) — enrichment is **per-doc FTS content append**, never a graph arm.

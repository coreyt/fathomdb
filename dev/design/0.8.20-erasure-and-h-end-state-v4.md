## 1. Root cause

Four hand-maintained projection lists, all different:

| | `search_index` | `search_index_edges` | **`search_index_v2`** | `vector_default` | `_fathomdb_vector_rows` | `_fathomdb_projection_terminal` |
|---|---|---|---|---|---|---|
| `purge_inner` `:6180` | ✅ | ✅ | **✅** | ✅ | ✅ | ✅ |
| `excise_source_inner` `:6398` | ✅ | ✅ | **❌ MISSING** | ✅ | ✅ | ✅ |
| `rebuild_projections` `:6520` | ✅ | ✅ | **❌ NEVER TOUCHED** | ✅ | ✅ | ✅ |
| write path `project_canonical_node_row` `:11779` | ✅ | — | **✅** `:11806` | ✅ | ✅ | ✅ |

**`search_index_v2` is a content-storing FTS5 table** (`fathomdb-schema:427`, no `content=''`) — it holds the
**body verbatim**, and is prunable by **no path except `purge`**. It **monotonically accumulates every body
ever written**, forever. Invisible, because both FTS read paths gate on `canonical_nodes` (`:11948`, `:12002`).
`secure_delete=ON` cannot help — it only zeroes pages freed by a real `DELETE`.

**Second root cause, same shape:** `rebuild_projections` **reimplements** projection with its own inline
`INSERT` (`:6548`) instead of calling the projector — which is *why* it diverged. And **there is no edge
projector at all** (only `project_canonical_node_row`, `:11779`); edge projection is inlined in `commit_batch`
(`:12166+`). *(This is codex P0 — v3's "replay through `project_canonical_node_row`" was incomplete: it would
not rebuild edge FTS/vectors.)*

## 2. Fix: two projectors, one registry, one erasure path

### 2.1 Extract the edge projector (new)

```rust
fn project_canonical_node_row(tx, cursor, kind, body, row_kind) -> Result<bool>;  // EXISTS :11779
fn project_canonical_edge_row(tx, cursor, kind, from, to, body, ..) -> Result<bool>;  // NEW — extract from
                                                                    // commit_batch's Edge arm (:12166+)
```

**`commit_batch` and `rebuild_projections` both call these two, and nothing else projects.** One projector per
row type. Rebuild can never again diverge from write — and now populates `search_index_v2` (which it has never
done) *and* edge projections (which v3 would have dropped).

### 2.2 One row-owned projection registry

```rust
/// THE row-owned projection tables (rows keyed 1:1 to a canonical `write_cursor`; they die with that row).
/// EVERY verb that erases or rebuilds projections MUST go through this module.
const ROW_OWNED_PROJECTIONS: &[RowOwned] = &[
    RowOwned { table: "search_index",                  key: Key::WriteCursor },
    RowOwned { table: "search_index_edges",            key: Key::WriteCursor },
    RowOwned { table: "search_index_v2",               key: Key::WriteCursor },  // <-- the missing one
    RowOwned { table: "vector_default",                key: Key::RowId       },  // vec0: rowid == write_cursor
    RowOwned { table: "_fathomdb_vector_rows",         key: Key::WriteCursor },
    RowOwned { table: "_fathomdb_projection_terminal", key: Key::WriteCursor },
];
fn erase_row_projections(tx, cursor) -> Result<u64>;        // purge + excise, per cursor
fn truncate_all_row_projections(tx) -> Result<u64>;         // rebuild
```

`purge_inner`, `excise_source_inner`, `rebuild_projections` → consume the registry. **Delete all three
hand-rolled lists.**

**Rebuild semantics preserved:** truncate via registry → reset projection cursor → replay **node** rows through
`project_canonical_node_row` and **edge** rows through `project_canonical_edge_row`. Vector work is *enqueued*
exactly as on the write path (the projector already owns that), so no double-scheduling and no worker deadlock —
the async embed path is unchanged.

### 2.3 The guard that makes this class unrecurrable

```rust
#[test] guard_row_owned_registry()
  // Introspect sqlite_master on a migrated DB. Every table carrying a `write_cursor` column
  // (or a rowid aliased to one — incl. fts5/vec0) MUST be in ROW_OWNED_PROJECTIONS.
  // A new projection table cannot be added without this test failing.
```

*Per `guardrail-failures-fix-tooling-not-people`.*

### 2.4 WAL — and erasure must FAIL LOUDLY, never silently succeed

`secure_delete=ON` zeroes pages freed in the main DB, but deleted content **persists in `-wal` frames until
checkpoint** (`structural-lifecycle-contract.md:52` requires file-level erasure; no verb names this today).

- `purge` and `excise_source` end with **`PRAGMA wal_checkpoint(TRUNCATE)` inside the verb.**
- **`wal_checkpoint(TRUNCATE)` can return `SQLITE_BUSY`.** Erasure completeness depends on it, so:
  **bounded retry with backoff; on persistent failure return a typed `EngineError::ErasureIncomplete` and the
  CLI exits non-zero.** A best-effort checkpoint is not acceptable — **an erasure verb must never report
  success on an incomplete erasure.**
- `--vacuum` remains an opt-in CLI flag for freelist defence-in-depth.

## 3. Making deletion reachable

### 3.1 The axis is provenance — and lawful erasure MUST be an application verb (D6)

**This section reverses an earlier draft that said "erasure stays CLI-only." That was wrong.** Two verified
facts settle it:

**(a) The SDK ships no CLI.** `src/python/pyproject.toml` declares **no `[project.scripts]`** (the wheel is
`_fathomdb.abi3.so` + Python modules only); the npm package declares **no `"bin"`**. A `pip install fathomdb` /
`npm i fathomdb` consumer — **i.e. Memex, the primary consumer** — has **no `fathomdb` binary to shell out to.**
"CLI-only" therefore does not *route* erasure elsewhere; it **removes** it. An application cannot discharge a
legal erasure obligation by asking its user to open a terminal.

**(b) The project already made this carve-out and shipped it.** REQ-037 (`dev/requirements.md:332`) forbids
application callers from invoking *"`excise_source`, **`purge_logical_id`**, or `safe_export`"* — yet
**`purge(logical_id)` IS an SDK verb today** (`fathomdb-py:1297`, `fathomdb-napi:1102`, shipped 0.8.19):
destructive, irreversible, GDPR-grade hard erasure, exposed to applications. That is legal because **AC-041
(`dev/acceptance.md:688`) tests only the REQ-054 five-name recovery denylist** {`recover`, `restore`, `repair`,
`fix`, `rebuild`} — and `purge` is not one of those names.

**So the project's actual policy is NOT "destructive ⇒ CLI-only." It is "RECOVERY-NAMED verbs are CLI-only."**
Deliberate data-lifecycle management is an **application** concern; disaster recovery is an **operator** concern.
`excise_source` is the *recovery* seam (REQ-026, "Phase 9 Pack B recovery seam" — built to excise a **corrupt/bad
ingest**). Conscripting it for lawful erasure was always a category error.

**The real defect is an ASYMMETRY.** 0.8.19 gave the **`l:` governed axis** a first-class application erasure
verb (`purge`). The **`h:` anonymous axis — the dominant corpus class — has no counterpart.** Closing it is a
**consistency fix**, not a new policy.

**Fix — add a first-class lifecycle verb on the provenance axis:**

```text
erase_source(source_id)        # SDK: Python + TS + Rust facade, X1 parity — alongside transition/purge
```

- **NOT a recovery verb** — the name is not in the REQ-054 five-name denylist ⇒ **AC-041 stays green, the
  denylist stays five, the byte-frozen guardrail is untouched.**
- **NOT `excise_source`** — REQ-037's literal name and the CLI recovery seam
  (`recover --excise-source --accept-data-loss`) both stay **exactly as they are**, for their recovery use.
- **Sits alongside `purge`** as a lifecycle verb — precisely the precedent 0.8.19 set.
- **Shares ONE engine code path** with `excise_source` (the single delete-set, §2.2). No second implementation
  to drift — the same root-cause discipline as everything else in this document.

**HITL-APPROVED (2026-07-12).** REQ-037's prose gains an explicit **lawful-erasure carve-out**, naming what
0.8.19 already did *de facto* for `purge`:

> **REQ-037 (amended).** Recovery tooling is unreachable from the runtime SDK: application callers cannot
> accidentally invoke the recovery surface (`recover`, `restore`, `repair`, `fix`, `rebuild` — REQ-054), nor
> `excise_source` or `safe_export`, which are **operator disaster-response** verbs and remain CLI-only.
> **CARVE-OUT — lawful erasure is an application capability.** Deliberate, caller-initiated data-lifecycle
> erasure is **not** recovery and IS exposed on the SDK: `purge(logical_id)` for governed records (shipped
> 0.8.19) and `erase_source(source_id)` for anonymous content (0.8.20). These are lifecycle verbs, gated by
> explicit invocation, never reachable by accident. AC-041 (the REQ-054 five-name denylist) remains the
> enforcing test and **remains GREEN** — neither verb is a recovery name.

`purge_logical_id` is struck from REQ-037's forbidden list — it was already contradicted by shipped code
(`fathomdb-py:1297`, `fathomdb-napi:1102`). The amendment records reality rather than changing it.

### 3.2 Provenance: structurally mandatory (D4 — v3's enum was BROKEN)

**v3 said** `Provenance::{Source, EngineDerived}` with `EngineDerived` "unrepresentable from the SDKs."
**That was wrong (codex P0):** `fathomdb/src/lib.rs:23-27` re-exports `PreparedWrite`, and `Engine::write` is
public (`:3364`) — so **any Rust caller could construct `EngineDerived`.** Python/TS rejection does not close a
Rust hole.

**Correct fix — make "no provenance" inexpressible on the public type:**

```rust
/// A caller-supplied stable, opaque, NON-PII source-document id. The erasure handle.
/// Validated: non-empty; reserved prefixes (`_engine:`, `_legacy:`) rejected.
pub struct SourceId(String);

pub enum PreparedWrite {
    Node { kind: String, body: String, source_id: SourceId, logical_id: Option<String>, .. },
    Edge { ..,                          source_id: SourceId, .. },
}
```

- **No `Option`. No `EngineDerived` variant on the public type.** There is nothing to bypass — the public write
  surface *cannot express* an un-provenanced row. This closes the Rust facade, Python, and TS in one move.
- **Engine-derived rows do not use `PreparedWrite` at all.** They already bypass it — the internal row writer
  does its own `INSERT` (`:5447`). They get a **reserved engine provenance** (`_engine:coverage`,
  `_engine:graph`, `_engine:importance`), which additionally makes them **enumerable and erasable** — a strict
  improvement over today's NULL.
- Only two internal callers currently route through `Engine::write` with `source_id: None` (`:5186`, `:5209` —
  the `#[doc(hidden)]` poison fixture). They supply a real `SourceId`. Trivial.
- **Breaking** — uses the HITL-authorized lever.

### 3.3 Legacy NULL provenance: back-fill ONLY the anonymous class (v3 was WRONG)

**v3 said** back-fill *every* NULL `source_id` → `_legacy:pre-0.8.20`. **That was wrong (codex P0):** governed
rows can also carry NULL `source_id`, so `excise_source _legacy:pre-0.8.20` would have **deleted governed
records**, contradicting v3's own "governed rows unaffected" claim (`excise_source_inner:6454-6457` deletes by
`source_id`, ignoring `logical_id`).

**Migration back-fills `source_id = '_legacy:pre-0.8.20'` ONLY where `logical_id IS NULL`.**

- Governed rows (`logical_id NOT NULL`) keep NULL `source_id` and are **already `purge`-addressable by their
  `logical_id`** — they neither need nor may join the bloc.
- The anonymous legacy corpus — the genuinely un-erasable class — becomes erasable **as one bloc**:
  `fathomdb recover --excise-source _legacy:pre-0.8.20 --accept-data-loss`. Coarse, but **total**. A permanently
  unreachable class is unacceptable at any granularity.
- `doctor orphan-provenance` (read-only) lists per-`source_id` counts, incl. the bloc, before an operator acts.

### 3.4 Telemetry: selective redaction, not truncation (v3 was UNSAFE)

The sink emits `result_stable_ids = h.id.to_prefixed()` to JSONL (`:4463-4467`). For governed hits that is
`l:SHA256("{kind}:{name}")` — a hash of a **low-entropy natural key**, i.e. **dictionary-attackable**. After
purging a person, the sink still confirms they existed and which queries retrieved them. Neither `purge` nor
`excise` touches it.

**v3 said "truncate the sink." That was unsafe (codex P1):** the sink path is **caller-supplied** (`:4383`), so
truncation could destroy unrelated operator-owned eval history.

**Correct fix — selective redaction.** `purge`/`excise` already know the ids they are erasing (they hold the
`logical_id` / cursors and bodies *before* deletion). They **atomically rewrite the sink, dropping every record
whose `result_stable_ids` reference an erased id.** The operator's legitimate eval history survives; the erased
identity does not. Telemetry stays **off by default**. Failure to redact ⇒ `EngineError::ErasureIncomplete`
(§2.4) — never a silent partial erasure.

### 3.5 Op-store records are INSIDE the boundary (codex NEW-1)

`operational_mutations.payload_json` accepts **arbitrary application JSON** (`schema lib.rs:127-135`;
`:11633`) and is **untouched by `purge`/`excise`**. There is **no record-level delete verb** — only a cap-based
retention sweep (`enforce_provenance_retention:10070`). So app-authored personal data can sit in the op-store,
un-erasable. That falsifies any "everything the engine wrote" claim.

**Fix: add `excise_collection_record(collection, record_key)`** — CLI-only, same recovery surface
(`recover --excise-collection-record`). The engine **cannot** know which op-store records relate to a purged
subject — the payloads are app-authored — so **the app must call it**. But **the primitive must exist**, or the
boundary is a lie. Documented as: *op-store erasure is app-driven; the engine provides the verb.*

### 3.6 Outside the boundary — enumerated and disclosed

`safe_export` archives, operator backups, and curated gold files retain pre-erasure bytes. The engine cannot
reach copies the operator chose to make. **Obligation discharged by disclosure:**

> Erasure covers the database, its WAL, the telemetry sink, and op-store records you excise. It does **not**
> cover exports (`safe_export`), backups, or gold files you created. **Re-generate or destroy those after any
> erasure.**

`excise_source_audit` retains `source_id` **permanently** (`:6479`) — by design; that is the audit trail.
Therefore **`source_id` MUST NOT contain PII** (enforced by `SourceId` docs; the audit of an erasure must not
re-identify the subject).

### 3.7 Provenance from the caller, not the model

`ExtractDocument.source_doc_id` is non-optional (`:1934`) and is serialized **into the prompt** (`:3598`) — yet
both ingest arms read it back from the **model's JSON response** (`:3643`, `:3770`) with **no fallback**. A
non-echoing extractor lands NULL provenance ⇒ un-erasable row. **Erasure completeness must not depend on an LLM
echoing a string.** Fix: use `d.source_doc_id`, already in scope.

> **Pin A — terminal-forever, by explicit OVERRULE of `structural-lifecycle-contract.md §2(ii)`.**
> §2(ii) mandates an opaque, auto-assigned surrogate `logical_id` for every anonymous node — doc-chunks
> **named** (`:97-107`; restated `README.md:108`). **OVERRULED for the anonymous class.** An opaque surrogate is
> not re-derivable from content, so re-ingest would mint a *different* id — destroying the re-ingest-stable
> content-addressed identity `h:` exists to provide (gold keying, telemetry, explain-correlation all rest on
> *same bytes ⇒ same handle*). §2(ii) would imply B-like forward minting and carries B's hazard.
> **Anonymous nodes stay `h:<content-hash>` permanently.** Any Phase-2 surrogate serves **only
> registry-admitted governed entities**; eligibility is decided **at write time**, and a stored row's id-space is
> **never re-derived.**

- `derive_logical_id` is **not** the surrogate mechanism: `:54` requires "never content-derived/**hashed**", and
  it *is* hashed (`:11152`); §2:98 names it as a **separate** mechanism.
- §2(ii) has **no consumer**: `:106` — *"Memex's world-model records already carry `logical_id`s, so its
  lifecycle problem is closed by (i) alone."*
- Goal already met: the C-2 `IdSpace` is **total** — `h:` **is** an address (`:1207-1290`).
- **No new column needed.** The record is `canonical_nodes.logical_id`'s null-ness; the invariant is a
  **prohibition**: *no migration, backfill, or verb shall ever populate `logical_id` on an existing canonical
  row.* Guard test: rows transitioning `logical_id NULL → NOT NULL` = **0**.
- Never key the pin on "declares a natural key" — that's a property of `kind`, and a future registry would sweep
  stored anonymous-but-typed rows into a backfill (= option C). Anonymous-but-typed rows are real
  (`Engine::write` public `:3364`; `kind` unvalidated; `logical_id` a thin pass-through).

## 7. Note to dependent applications (Memex, Hermes, OpenClaw)

**`SearchHit.id` does not change.** Every hit stays `l:`/`h:`/`p:`, non-null; `IdSpace::to_prefixed()` remains
byte-identical to the pre-0.8.19 `stable_id` (`:1286-1290`). **No gold-remap, no id churn, no re-baseline.**

1. **An `h:` hit will never become `l:`.** Permanent guarantee. Want an object governed and lifecycle-
   addressable? **Supply a `logical_id` at write time** — the engine cannot infer it, and registering a kind
   later will **not** retroactively govern rows you already wrote.
2. **`h:` is addressable but not lifecycle-addressable.** `transition`/`purge` are `l:`-only. **If you write the
   same thing anonymously *and* governed, you get two rows and both will surface in search** — the engine will
   not dedupe them. Remove the anonymous one by excising its source.
3. **Erasure IS an SDK call — a new lifecycle verb, `erase_source(source_id)`**, alongside `transition`/`purge`.
   You do **not** need a terminal, and you do **not** need the CLI (which the wheel/npm package does not ship).
   The *recovery* seam (`recover --excise-source --accept-data-loss`) stays CLI-only for operator disaster-
   response — that is a different job with a different caller. **Governed records:** `purge(logical_id)`.
   **Anonymous content:** `erase_source(source_id)`. Both are first-class application verbs.
4. **BREAKING in 0.8.20: `source_id` becomes mandatory on every public node/edge write.** Supply a stable,
   opaque, **non-PII** id now. It is the **only** handle by which unversioned content is erasable. Rows written
   today without one are swept into `_legacy:pre-0.8.20` — erasable only *en masse*.
5. **Never put PII in `source_id`** — retained **permanently** in the erasure audit row.
6. **If you write personal data into operational collections, YOU must erase it** via
   `recover --excise-collection-record`. The engine cannot know which op-store payloads relate to a subject.
7. **Erasure does not cover your exports, backups, or gold files.** Re-generate or destroy them after erasure.
8. **Until 0.8.20 ships, erasure is INCOMPLETE.** `excise_source` leaves the body in `search_index_v2`; the
   telemetry sink retains ids; op-store payloads are un-erasable. **Do not represent FathomDB as
   GDPR-erasure-capable yet.**

# Slice 0.8.5 design memo — expose tuned CE-rerank α / pool_n / ce_score in the engine

**Type:** engineering · $0 · no new HITL gate · **EXP-0** in the portfolio experiment tree.
**Provenance:** measured-provisional 0.8.3 result (α=1.0 reblend = Mem0 parity-or-better via the eval
`ReblendRerankAdapter` `score=0` trick). This slice **promotes** that known-good computation into the
native engine surface as an **opt-in knob** — default behavior **byte-identical**.

Spec: `dev/plans/0.8.5-ce-rerank-alpha-expose-slice.md`. Anchors below re-verified at `main` (paths carry
the real `src/rust/crates/...` prefix; the spec wrote them without it).

---

## 1. Goal / non-goals (unchanged from spec §2)

**Goal.** Make `(alpha, pool_n)` caller-settable and expose the per-candidate **CE score**
(`ce_norm = sigmoid(ce_logit)`) on the shipped `rerank()` and `Engine.search` surfaces, so the
agentic-answer path can opt into the measured-parity α≈1.0 / pool_n=10 config (or re-blend client-side).

**Non-goals.** No default-α change (stays 0.3 — the C6 factoid guard, Decision 5). No finishing/registering
the M5 α=1.0 run. No recall/embedder/quant work. No router. No raw-logit exposure (decision below).

---

## 2. Locked decisions (resolves spec §7 open questions)

| # | Question | Decision | Rationale |
|---|----------|----------|-----------|
| D1 | `ce_score` populate scope | **`Some` only for in-pool reranked hits; `None` for the unreranked remainder, depth=0, empty list, and the no-CE-model soft-fallback** | cleaner identity; matches "ce only computed in the pool"; lets a caller tell "reranked" from "passed through" |
| D2 | `pool_n` default when unset | **`= rerank_depth`** | preserves today's pool == depth semantics exactly → byte-identity for all existing callers |
| D3 | α out of `[0,1]` | **clamp silently** (`alpha.clamp(0.0, 1.0)`) | matches the existing `rerank_depth.min(hits.len())` clamp idiom; no new error surface |
| D4 | `pool_n` resolution + clamp | **At the binding: `pool_n = pool_n.unwrap_or(rerank_depth)`. In `ce_rerank`: `n = pool_n.min(hits.len())`; `n == 0` is the identity/no-model-load path.** Document that `rerank_depth == 0` is still the early identity gate in `rerank_fused` — so `rerank_depth=0, pool_n=10` does NOT rerank. | codex design-review: cleaner than special-casing explicit-0; removes binding-boundary ambiguity |
| D5 | expose raw CE logit | **No** — expose only `ce_norm` (sigmoid) | no caller needs the logit; `ce_norm ∈ [0,1]` is what the reblend math uses |
| D6 | fold `recover_ce_norm` into a shared engine helper | **No (keep slice small)** | the eval adapter stays as the equivalence *oracle*; native path is tested *against* it. Folding is a separate cleanup. |

**Default-preservation invariant (the load-bearing one):** every existing call site that does **not** pass
`alpha`/`pool_n` must produce identical **ordering AND scores AND `soft_fallback`** to pre-slice (the
*ranking* fields). The defaults `alpha=0.3, pool_n=rerank_depth` make the new blend arithmetic reduce to the
old `const ALPHA=0.3` + `n=rerank_depth.min(len)` path exactly. **Caveat (codex P1):** `ce_score`/`ceScore`
is a NEW additive field on the result shape, so the *whole returned object* is not literally byte-identical —
the invariant is over the ranking fields only, and `ce_score` is `None`/null on every hit of any default-path
(α=0.3) result. The regression-pin tests assert ranking-field identity, not object byte-identity.

## 4. TDD test plan (RED first, commit RED sha)

**Rust `fathomdb-engine/tests/pr_g10_reranker_ce.rs`** (CE tests run under `--features default-reranker`;
gate as the existing CE tests do):
- `α=0.3, pool_n=rerank_depth` (defaults) → ordering + scores **byte-identical** to a pre-slice fixture
  (regression pin; C6 guard holds).
- `α=1.0` → pool ordering == sort-by-`ce_norm` desc (pure-CE concentration).
- `α=0.0` → pool ordering == RRF order (no CE influence).
- `pool_n < rerank_depth` and `pool_n > hits.len()` → clamp correctly (`min(len)`).
- `α = -0.5` and `α = 2.0` → clamp to `[0,1]` (assert behaves as 0.0 / 1.0).

  `ce_score == None` for the remainder.
- `rerank_depth=0` (and no-CE-model path) → identity ordering+scores AND `ce_score == None` on every hit.
**Rust `fathomdb-engine/tests/pr_e2_rerank_passages.rs`** — `alpha`/`pool_n` plumbed through

## 5. eu7 guard (spec §5)
Reranking is **post-retrieval** — reorders already-retrieved hits, touches **no** embedding/quant/vector-SUT
path. Expectation: **no eu7 re-run required**. The reviewer confirms no embed/quant/SUT path is touched; if
(unexpectedly) it is, re-run `eu7_real_corpus_ac.rs` (`AGENT_LONG=1`) and confirm one-sided gate
`ci_hi ≥ 0.90` still holds. Record the confirmation in the closure.

## 6. Risks / review focus
- **Byte-identity** is the #1 risk: any existing caller (incl. the 2 `rerank_fused` call sites in
  `read_search_in_tx`, the napi/pyo3 search defaults) must thread `alpha=0.3, pool_n=rerank_depth`. The
  default-α regression-pin test is the guard. Codex review must confirm no default drift.
- **`SearchHit` field add** ripples to all 8 construction sites + any external match/serde. Confirm no
  exhaustive struct pattern elsewhere breaks; confirm `SearchResult` serialization (if any) intends to
  carry/omit `ce_score`.
- **`rerank_passages` return-type change** ripples to its single pyo3 caller + tests. Confirm no other
  consumer.
- **ReaderRequest::Search size** — adding `f64 + usize` (16B) is fine; the variant is already boxed-heavy
  fields. No channel-size regression of concern.

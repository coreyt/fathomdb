## HITL rulings applied (2026-07-08)

- **D1 ✅ (superseded/expanded by the D4 trace):** the dense/fused arm is a **TWO-STAGE** pipeline, so the probe
  asserts **BOTH** retrieval representations (asserting one lets the other silently drift):
  **(P1)** Phase-1 candidate recall — Hamming over the **mean-centered** 1-bit `embedding_bin` =
  `vec_quantize_binary(sign(x − mean_vec))` (`build_vector_phase1_sql`, engine `lib.rs` ~6525-6542; query centered
  symmetrically ~4064-4074); and **(P2)** Phase-2 final ordering — `vec_distance_l2(embedding, vec_f32(query))`
  over the **UN-centered** float `embedding` (**L2, not cosine**; both doc+query un-centered).
- **D2 ✅** on divergence: refuse the vector-dependent (dense/fused) arm, keep **FTS** servable, loud typed error — **refined by round-2 resolutions to a query-time `EngineError::VectorEquivalenceMismatch` at `search_inner_with_stats` (NOT an open-time `EngineOpenError`)**; see R-VEQ-4.
- **D4 ⏳ PENDING — floor NOT frozen; now TWO components.** The Steward's ingest→query→compare trace is COMPLETE
  (engine-source-verified). The tolerance floor is a design-review/HITL parameter with **two parts**: a **Phase-1
  binary-code flip count** (mean-centered) AND a **Phase-2 L2 epsilon** on the un-centered `embedding`. Lean:
  0 flips / small L2 ε on the 45-probe set — **not ruled**.
- **D5 ✅ platform scope.** ONLY **`x86_64-unknown-linux-gnu`** (this host) is on the 0.8.18 critical path. The
  repo's other declared targets (py `aarch64-linux-gnu`, `x86_64/aarch64-apple-darwin`, `x86_64-pc-windows-msvc`;
  napi `darwin-x64/arm64`, `win32-*`, musl) **stay supported but are OFF the critical path** — delivered by a
  **separate follow-on orchestrator** after x86_64-linux GA works. Slice 20's hardening + dry-run is scoped to
  x86_64-linux; the cross-platform matrix is **deferred-to-follow-on (noted, not dropped)**.
- **D6 ✅** #11-full hardening targets the **0.8.20 OPP-12 breaking-pair-publish contract** (F-19/F-21).
- **D3 / L3 ONNX-GPU-EP — OFF THIS TRACK.** A separate Steward-commissioned OOB subagent procures a CUDA
  `libonnxruntime.so`, measures the ONNX-GPU-EP Δ, and writes durable results **regardless of outcome**. This

## Design-review resolutions — codex round 1 (verdict: U1 BLOCK · U2 BLOCK · U3 BLOCK · U4 CONCERN)

All findings ACCEPTED (source-grounded); the requirements/ACs below are revised accordingly. Terminal verdict
after these revisions is returned to the Steward for re-verification → HITL sign-off (BLOCKs not overridden).

- **[U1-a BLOCK — refusal seam]** `EngineOpenError` at open contradicts "FTS still servable" (no engine to serve).
  **FIX:** `Engine::open` **succeeds into a degraded state** carrying `dense_disabled=true` (new engine flag);
  the typed error (`VectorEquivalenceMismatch`) is raised at **first vector-dependent (dense/fused) query**, not
  at open. FTS/text queries serve normally. (Matches the plan's "open/**serve** error" wording.) → R-VEQ-4.
- **[U1-b BLOCK — check ordering]** the probe must run **after** any open-time mean-recovery/requantize (which can
  pin/re-quantize `embedding_bin` on crash-recovered DBs), NOT immediately after `check_embedder_profile`.
  **FIX:** seam = after mean-recovery + `ensure_vector_partition`, reading the FINAL live `mean_vec`. → R-VEQ-3c/U1.B.
- **[U1-c — scope language]** "dense/fused arm" was too loose; production ranking also RRF-fuses text + applies
  recency/importance/CE (all backend-independent transforms over the verified vector arm). **FIX:** #5 guards the
  **vector-representation** arm (P1 recall + P2 vector ordering); the guard refuses queries that DEPEND on the
  vector arm; pure-text/FTS is unaffected. Language tightened throughout.
- **[U1-d — mean lifecycle]** store **UNCENTERED f32 reference vectors ONLY**; **never persist P1 bits**; recompute
  expected P1 bits at check time from the uncentered reference + the live `mean_vec`. → R-VEQ-1/3a.
- **[U1-e / R-VEQ-5]** codex: relaxing the cross-vendor identity refusal on a 45-probe PASS is **NOT sound** (does
  not prove tokenizer/model/pooling/EP equivalence for arbitrary/truncated/future-drift text). **RESOLVED: #5 is
  ADDITIVE-ONLY** — the distinct-identity cross-vendor refusal STAYS; portability-relaxation is deferred to a
  future dedicated ADR with a stronger contract. (HITL confirms; codex-supported.)
- **[U2-a BLOCK — dry-run is not a real publish]** the default `cargo-publish-if-new.sh --dry-run` SKIPS dependent
  Rust crates, PyPI publish, post-publish smoke, and co-tagging → a green dry-run does NOT exercise #11-full.
  **FIX:** R-REL-4b redefined to require an **exercised** verification (staging/test-registry or a dry-run that
  actually runs the tiered dependent-crate + PyPI + smoke + co-tagging paths); a bare `--dry-run` is insufficient.
- **[U2-b BLOCK — GA-tag matrix gate hole]** `release.yml` still builds macOS/Windows/linux-aarch64/napi on ANY
  real tag; hardening only x86_64-linux while the tag fires the full matrix could **block or ship broken legs**.
  **FIX:** new **R-REL-4e** — Slice 20 must **gate the tag's matrix to x86_64-linux ONLY** for the 0.8.18 GA tag
  (deferred platforms excluded from this tag; re-enabled by the follow-on orchestrator), so no unreviewed leg
  ships or blocks the capstone.
- **[U2-c BLOCK — false atomicity]** true cross-registry atomicity is impossible; npm/PyPI lack idempotency; the

  preflight + ordered commit points + **per-registry retry/no-op idempotency (incl. npm/PyPI)** +
  **poll-for-resolvability** (replace the fixed sleep) + explicit **rollback-forward** recovery for the 0.8.20
  breaking-pair.
- **[U3-a BLOCK — mean-centered P1 needs the REAL pinned mean]** mean=0 is raw-sign, not a proxy for the engine's

  `_fathomdb_embedder_profiles.mean_vec` / engine DB) for the P1 mean-centered measurement; raw-sign reported
  separately and labeled as such.
- **[U3-b — pooling default mismatch]** `CandleBgeEmbedder::new()` defaults to `Mean`; the 0.8.16 baseline pins
  `Cls`. **FIX:** R-CAL-2 explicitly pins **`Cls` on BOTH** candle+ONNX and **hard-asserts 0 flips / 45 probes**
  (not inferred from printed output).
- **[U3-c — process-global device env]** candle device is selected process-globally via `FATHOMDB_EMBED_DEVICE`.
  **FIX:** U3.B uses **explicit per-leg device constructor seams** (or strict serialization), not ambient env.
- **[U4 CONCERN — non-blocking is sound IFF U1 re-probes the live backend]** confirmed: U1's open-time re-embed
  runs against WHATEVER backend is live (incl. ONNX-GPU), so the guard is **backend-agnostic**; L3 is

  **EFFECTIVE ORT provider** (guard against silent CPU fallback mislabeled as GPU).

## Design-review resolutions — codex round 2 (verdict: U1 BLOCK · U2 BLOCK · U3 CONCERN · U4 BLOCK)

Round-1 fixes CONFIRMED CLEARED by codex: U1-b (ordering), U1-d (mean lifecycle), U1-e (additive-only), U2-a
(exercised verification), U2-c (resilience-not-atomicity), U3-a (real pinned mean), U3-b (Cls + hard-assert).
Round 2 surfaced deeper INTEGRATION issues (SDK error mapping, the choke point, npm platform contract, the
provider seam). ALL accepted; resolved by making the design concrete (name the seam), NOT by writing code.

- **[U1-1 BLOCK — wrong enum]** the query-time refusal was specified as `EngineOpenError`, but queries surface
  `EngineError` (Py `fathomdb-py/src/lib.rs`, TS `fathomdb-napi/src/lib.rs`). **FIX:** the refusal is a new
  **`EngineError::VectorEquivalenceMismatch`** variant raised at query time (open-time detection only SETS the
  degraded flag); Py (pyo3) + TS (napi) map it to **leaf error classes/codes** (parity with
  `EmbedderIdentityMismatch`). → R-VEQ-4.
- **[U1-2 BLOCK — "FTS served" not an exposed contract]** public `search` is HYBRID (embeds, then vector+text
  together); there is no text-only search route, so a degraded engine could refuse ALL user-visible search. **FIX:**
  R-VEQ-4 defines the degraded contract precisely: a **hybrid/vector-dependent** call raises the typed error
  (loud), AND an **explicit text-only/FTS-only search path** is exposed (engine + Py + TS) so FTS remains
  serviceable in degraded mode. (D2 "keep FTS servable" now has a real surface.)
- **[U1-3 BLOCK — no named choke point]** **FIX:** the refusal is enforced at the single engine choke point
  **`search_inner_with_stats`**, BEFORE embedding / vector SQL / graph seeding / CE rerank; the AC enumerates
  EVERY public vector-dependent arm (`search`, `searchExpand`, explain/rerank, graph-arm) and asserts each
  refuses. → R-VEQ-4.
- **[U1-4 BLOCK — degraded-state observability]** `dense_disabled` was an in-memory flag only; `OpenReport`
  surfaces only schema/embedder/mean today, so a caller cannot know it is degraded before querying (and a reopen
  could silently serve dense). **FIX:** new **R-VEQ-6** — surface `dense_disabled` (+ reason) on the **`OpenReport`
  (Rust/Py/TS)** + a **telemetry counter**; the state is re-derived at every open (probe re-run), so it is not a
  persisted-stale risk but IS observable pre-query.
- **[U2-1 BLOCK — npm platform contract]** matrix-gating CI is necessary but not sufficient: `src/ts/package.json`
  ships ONE `fathomdb` package (`files: [dist, fathomdb.*.node]`, no `os`/`cpu`), so a linux-x64-only artifact
  published under `latest` still installs on mac/win and fails at RUNTIME. **FIX:** new **R-REL-4f** — gate the
  **artifact/platform contract**, not just CI: set napi per-platform `optionalDependencies` + `os`/`cpu` so a
  mismatched install fails at INSTALL time (or fs the platform), AND **publish 0.8.18 under a non-`latest` npm
  dist-tag** until the follow-on completes the matrix. **The `latest`-vs-prerelease dist-tag choice is HITL's**
  (publish-matrix decision).
- **[U3-1 CONCERN — device seam]** candle construction is process-env driven (`FATHOMDB_EMBED_DEVICE`); a per-leg
  seam in one process is fragile. **FIX:** the calibration harness uses **strict per-leg PROCESS isolation** (one
  subprocess per (vendor,device) leg, env set once at start) — no new engine API needed — OR, if preferred, an
  explicit `new_on_device(dev)` constructor seam. → R-CAL-1/U3.B.
- **[U4-1 BLOCK — effective-provider seam]** `OrtBgeEmbedder` stores identity/tokenizer/session/pooling, NOT the
  effective provider; stderr warnings are not durable calibration data. **FIX:** R-D3-2 requires a small additive
  **`effective_provider()` accessor** on `OrtBgeEmbedder` (queries the live ORT session's active providers), whose
  value the harness records — so a silent CPU fallback is captured as data, not just a warning.

**Note for sign-off (unchanged):** the only HITL-reserved items once the design clears are **R-VEQ-5 = additive-only**
(codex-supported; pending HITL confirm) and the **D4 floor** (set after U3 measures). The npm dist-tag (R-REL-4f)
is a third HITL publish-matrix call surfaced by round 2.

### U1.B Design

- **Store:** new internal table `_fathomdb_embed_probe`; write hook adjacent to `ensure_vector_partition`
  (engine `lib.rs:2807`) at first vector-kind registration; migration `Migration{step_id:19}` in `fathomdb-schema`.
  Probe fixture = the committed 45-probe `candle_onnx_equivalence_probes.txt` (diverse coverage).
- **Check (two representations):** on `Engine::open` in `open_locked`, **after** `check_embedder_profile`
  (engine `lib.rs:2806`), re-embed each probe with the live embedder and assert **both** dense-arm stages:
  **(P1)** produce the mean-centered `embedding_bin` via the same `vec_quantize_binary(sign(x − mean_vec))` path
  as `build_vector_phase1_sql` and compare the binary-code flip count to the stored reference; **(P2)** compare
  the **un-centered** float `embedding` under `vec_distance_l2` within an L2 ε. Centering is gated by
  `identity_requires_mean_centering(identity)` ∧ mean-pinned (verified: true only for the default
  `fathomdb-bge-small-en-v1.5`; NoopEmbedder no-op), applied symmetrically ingest+query with un-centered
  fallback. Divergence beyond EITHER (review-set) floor ⇒ set `dense_disabled=true` at open; the query-time
  choke point `search_inner_with_stats` then raises the typed **`EngineError::VectorEquivalenceMismatch`**
  (sibling of `EmbedderIdentityMismatch` `lib.rs:2052`; NOT an open-time error); the text-only/FTS path is untouched.
- **Why both (correctness trap flagged by the D4 trace):** asserting only P1 lets P2 float drift silently
  (re-ranking within the recalled candidates), and asserting only P2 lets a centered-code flip silently drop
  candidates from Phase-1 recall. Both must hold. `vec_quantize_binary` = the sqlite-vec threshold-at-0 sign
  function (registered `lib.rs` ~9408).
- **Landing:** Slice-5 is a **schema bump (18→19) ⇒ HITL-gated landing** regardless of the standing mandate.

### U3.B Design

Extend the device seam so the existing `candle_onnx_equivalence.rs`-style harness parameterizes each embedder's
device (candle via `FATHOMDB_EMBED_DEVICE`/`resolve_device`; ONNX via the `ort_bge.rs` runtime EP), replacing the
hard-coded `FATHOMDB_EMBED_DEVICE=cpu` (`:149`) with **one subprocess per (vendor,device) leg** (each sets `FATHOMDB_EMBED_DEVICE` once at start — NO in-process per-instance switching, since candle device is process-env-global; an explicit `new_on_device(dev)` seam is the alternative). **Build/run discipline:** the
candle-CUDA build+run happens on the **MAIN tree only** (pure `cargo test --features embed-cuda`, `nvcc` on PATH
via `/usr/local/cuda-12.6`, `CUDA_HOME` set), **never** from a worktree; **never** `maturin`/`pip -e`. An
**ONNX-GPU-EP leg plugs in later** by pointing `ORT_DYLIB_PATH` at a CUDA-enabled `libonnxruntime.so` (the OOB
L3 subagent's deliverable) — no harness change needed. The harness feeds U1/D4's floor decision; it does **not**
enforce anything.

## U4 — D3 ONNX-GPU-EP calibration leg (OOB, non-blocking) — governed by the process gate

> The ONNX-GPU-EP Δ measurement (D3) is the **cuda-EP leg of the U3 harness, executed OOB** by a separate
> Steward-commissioned subagent. Per HITL (2026-07-08) the process gate governs it: A+B (requirements+ACs+design)
> reviewed BEFORE it runs. It stays **non-blocking** for the capstone; the U1 #5 floor references its durable
> result when it lands (until then the ONNX GPU EP ships **un-probe-verified** — TC-track).

### U4.B Design

The cuda-EP leg reuses U3 unchanged: point `ORT_DYLIB_PATH` at a CUDA-enabled `libonnxruntime.so` and select the
CUDA EP via the `ort_bge.rs` runtime provider path (`FATHOMDB_EMBED_DEVICE=cuda:0`) — **no code change, no
`.venv` mutation** (dlopen). Runs on the MAIN tree (GPU discipline). Footprint: the ONNX GPU EP is
**OFFLINE-BUILD/EVAL-only**; nothing about the shipped CPU library query path changes. Because it is OOB and
non-blocking, the U1 #5 mechanism must be **complete and correct without it** — L3 only *narrows the calibration*
of the (already two-sided) floor; it is not a prerequisite for the refuse-to-serve guard to function.

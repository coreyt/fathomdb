### 1.1 Refinement — v4's registry model is too coarse (row-owned vs kind-owned)

v4 §2.2 lists `vector_default` and `_fathomdb_vector_rows` in `ROW_OWNED_PROJECTIONS` and asserts the write-path
projector owns vector work. **Re-verification shows the write path does not synchronously write either table.**
`project_canonical_node_row` (:11817-11830) instead **enqueues** by upserting `_fathomdb_projection_state`, which
is keyed **by `kind`**, not by `write_cursor`; the async embed worker later materializes the vector rows. Rows
that will never be vector-projected short-circuit via `record_projection_terminal`.

**Consequence for the design (adopt):** the registry must model **two distinct classes**, not one.

- **Row-owned** (1:1 with a canonical `write_cursor`; dies with the row): `search_index`, `search_index_edges`,
  `search_index_v2`, `vector_default` (vec0 rowid == write_cursor), `_fathomdb_vector_rows`,
  `_fathomdb_projection_terminal`. These are what `erase_row_projections(tx, cursor)` covers.
- **Kind-owned watermark state** (NOT row-keyed; must **not** be deleted per-cursor): `_fathomdb_projection_state`.
  Erasing a row must **not** rewind a kind's watermark. Rebuild resets it deliberately; erasure must leave it alone.

v4's single-list model would have invited a per-cursor delete against a kind-keyed table. The
`guard_row_owned_registry` test must therefore assert on **`write_cursor`-keyed** tables specifically and
**explicitly allowlist** `_fathomdb_projection_state` as kind-owned, or the guard will either miss it or demand
the wrong thing.

## 2. Defects found in the v4 design of record

### D-A · **`excise_source_audit` is NOT retained permanently — v4 §3.6 is REFUTED** *(new, material)*

v4 §3.6 states the audit row "retains `source_id` **permanently** (`:6479`) — by design; that is the audit trail",
and the plan's §8 repeats it as the justification for the non-PII `source_id` rule.

**Verified false.** The audit row is written into `operational_mutations` (**:6479**), and
`enforce_provenance_retention` (**:10070**) sweeps that table **cap-first, oldest-`id`-first, with NO collection
filter** (**:10083-10089**):

```sql
DELETE FROM operational_mutations
 WHERE id IN (SELECT id FROM operational_mutations ORDER BY id LIMIT ?1)
```

Erasure audit rows are swept like any other op-store row. **Two consequences, both in scope for Slice 5:**

1. **The erasure audit trail is not durable.** Accountability (demonstrating *that* an erasure occurred) is a
   distinct obligation from erasure itself. A retention sweep can silently destroy the proof.
2. **It collides with work item 9.** `excise_collection_record` operates on the op-store — the *same table* the
   audit rows live in. The erasure record and the app payloads being erased share one retention pool.

**The non-PII `source_id` rule still stands** (a swept audit row may not have been swept *yet*), but its stated
justification must be corrected — it is currently argued from a false premise.

**Recommended resolution (Slice 5):** exempt the audit collection from `enforce_provenance_retention` (filter the
sweep by collection), and state the audit-durability guarantee explicitly. **Flagged for HITL** — it is a
retention-policy change, not a pure defect repair.

### D-B · Naming drift — `rebuild_projections` is not the function to change

v4 §1/§2.2/§6 name **`rebuild_projections` :6520** as the site consuming the registry. Verified: `:6520` is inside
**`rebuild_shadow_state` (:6515)**, a private helper. `rebuild_projections` (**:5949**) is the *public entry
point*; `run_rebuild` (**:6496**) sits between them. Slice 5 must modify **`rebuild_shadow_state`**. Taking v4
literally would patch the wrong function.

### D-C · `derive_logical_id` **lowercases** its inputs

v4 §3.4 / plan §0 give `logical_id = SHA256("{kind}:{name}")`. Verified (**:11156**): the derivation is
`SHA256(lowercase(kind) + ":" + lowercase(name))`, and it additionally rejects a `kind` containing `:` and an
empty `name` (:11153-11155). Case-insensitivity is load-bearing for identity collapse.

**Why it matters here:** it *strengthens* v4 §3.4's dictionary-attack argument — case-folding shrinks the
preimage space further. The telemetry-redaction requirement is unchanged; the rationale should cite the real
derivation.

### D-D · Edge-projection anchor drifted

v4 §1 cites the inlined edge projection at `commit_batch` `:12166`. Verified: edge FTS projection is at
**:12186-12194**, edge vector enqueue at **:12195+**. The *finding* (no edge projector fn exists) is
**CONFIRMED** — the nearest thing, `prune_edge_projection_shadows` (**:4127**), only deletes.

## 3. The RED-test contract (binding on Slice 5)

**Rule 1 — assert on RAW TABLE CONTENTS.** Erasure/rebuild coverage for `search_index_v2` MUST be asserted with
direct SQL against the table, e.g. `SELECT count(*) FROM search_index_v2 WHERE write_cursor = ?` and
`SELECT body FROM search_index_v2`. A test that issues a **search** is **invalid as an erasure witness** — both
read paths gate on `canonical_nodes` (§1), so it passes on the broken code. Any Slice-5 test that proves erasure
via search results must be rejected at review.

**Rule 2 — the RED test must fail on today's code for the right reason.** Each item below must be demonstrated
failing at `4ca70ba6` before the fix, and the failure output recorded in the slice `output.json`.

**Rule 3 — byte-level assertions for WAL.** The WAL requirement (E5) is a claim about bytes at rest; assert by
scanning the raw `.db` and `-wal` files for the erased body, not by querying.

- **AC-077 is a RESERVED PLACEHOLDER**, not a defined gate — `dev/acceptance.md:1286` reserves it for the
  **IR-eval IR-1/IR-2** initiative ("no fabricated numbers"; thresholds all TBD).
- **AC-078 is *conditionally* reserved to the same initiative** — `:1297`: *"+ AC-078… only if the consensus

the §4 test names above. Only genuinely gated surfaces mint an AC:

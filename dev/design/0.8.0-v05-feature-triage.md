# FathomDB 0.8.0 — v0.5.x Feature Triage (Consolidated)

> Status: synthesis for HITL review. Decides which v0.5.0 features enter 0.8.0
> scope vs defer, and **how** to get each there (v0.5.x-ready / web-research /
> experiment / design-ADR / scaffolding). Produced by a ground → assess(+web
> research) → adversarial-verify → synthesize workflow (12 features, each
> verified). Source-of-truth: `ADR-0.8.0-supersede-five-verb-surface-cap`,
> `ADR-0.8.0-agent-memory-retrieval-and-identity`, `0.8.0-agent-memory-fit.md`,

## Lead thesis

The five-verb scope **cap** (AC-057a) is being superseded by a **governed**
surface. The supersession does NOT open a free-for-all; it swaps **count
enforcement** ("exactly five") for **rule enforcement**: (a) SDK parity (Py+TS
allowlist lockstep), (b) a permanent recovery-name denylist
`{recover,restore,repair,fix,rebuild}` keeping recovery/mutation CLI-only
(`doctor` is SDK-absent via the allowlist, not this denylist),
(c) a typed-write boundary + a small fixed filter grammar (equality+range over
body-JSON), never raw SQL/DSL. Single-writer, no-data-migration (additive ALTER
only; derived-projection rebuild is exempt), vec0-brute-force, and
migration-accretion-guard invariants all still hold.

The critical inversion: under AC-057a reads were *forbidden*; under the governed
surface reads are *additive under explicit rules*. This unblocks the table-stakes
read floor (G1/G2/G3) and the integrity floor (G8) in 0.8.0 without a fresh
"should we open to reads?" decision — that decision is "yes, under governance."
What remains is sequencing and namespace (`read.*` B1 vs `admin.*` B2).

**The keystone:** everything readable rides on **G0 canonical-identity
substrate** (`logical_id` + `superseded_at` + partial-unique-active index on
`canonical_nodes`/`canonical_edges`). The current schema
(`canonical_nodes(write_cursor,kind,body)`,
`canonical_edges(write_cursor,kind,from_id,to_id,source_id)` — `schema/src/lib.rs:111,199`,
migrations 2+8) has **no `logical_id`/`superseded_at`**, so G0 gates
F1/F2/F4(SDK)/F9/F10/F11.

### Add/defer split at a glance

- **Add in 0.8.0 (5):** F12 structured hits (G1), F2 by-id read (G2), F4-READ
  operational read-back (G3), F10 dangling-edge flag (G8), F11 identity substrate
  (G0; valid-time *designed* now, *implemented* later).
- **Defer to 0.8.x (5):** F1 graph traversal (G5/G6), F3 filtered-list small
  grammar (G4 core), F8 retrieve+expand fan-out (folds into G6), F9 confidence
  (resolve vs G12 first), F4-GOVERNANCE (retention/secondary-index/compact).
- **Drop (2):** F6 per-kind tokenizer presets (dropped 2026-06-01 — HITL); F7
  in-process admin/maintenance SDK API. (F5 FTS property schemas → defer-0.8.x but
  **reshaped** away from the v0.5.x table-per-kind machinery.)

> **Note (global tokenizer default — KEEP):** dropping F6 drops the *per-kind*
> tokenizer surface entirely. The unrelated *global*-default FTS5 tweak
> (`unicode61` → `porter unicode61 remove_diacritics 2`) is **not** part of F6
> and is **KEPT in 0.8.0** (HITL 2026-06-01) as a zero-surface recall win — see
> F6 detail below.

## Per-feature detail (adversarial corrections folded in)

### F12 — Structured search hits + HitAttribution (G1) — ADD 0.8.0
Highest-leverage table-stakes item; critical for all three consumers. World-class
core (Elastic / Azure AI Search / OpenSearch 2025): return human-readable
non-vector fields `{id, kind, body, score, branch}` with **per-branch
(non-cross-comparable) scores** tagged by modality; fuse separately via RRF (=G9).
Match-field highlighting (`HitAttribution.matched_paths`) is a **separable
optional highlighter layer** — correctly a reserved Phase-1 stub in v0.5.x; defer.

No new `read.*` verb — enriches the *existing* `search()` payload, so it is
AC-057a-clean and ships even without the supersession ADR. It IS a breaking
data-class change that MUST land in Python AND TypeScript simultaneously
(`PySearchHit`/`PySearchResult` + napi `SearchHit.id:i64` since napi has no u64;
`index.d.ts` regen).

**Corrected firstStep (folds in the FTS-branch omission):** reshape **BOTH**
branches — vector (`lib.rs:3247-3250`, `SELECT body FROM canonical_nodes WHERE
write_cursor=?1`) AND FTS5 text (`lib.rs:3296-3305`, `SELECT body FROM
search_index ... ORDER BY write_cursor`) — to carry `id/kind/score/branch`.
Introduce `SearchHit{id:i64, kind, body, score:f64, branch}`, change
`SearchResult.results` from `Vec<String>` to `Vec<SearchHit>`, and **drop the `Eq`
derive on `SearchResult`** (f64 forbids Eq — `impl-strategy.md:79-81`). Add
`vec_distance_l2(...) AS dist` to the phase-2 CTE. Preserve vector-first ordering +
dedup keyed on `body` (BTreeSet `lib.rs:3275`). Interim id = `write_cursor`, swaps
to `logical_id` when G0 lands with no carrier reshape; the FTS branch must *add*
`write_cursor` to its SELECT to emit an id. Recall floor watch:
`eu8_ir_validation.rs:325-326` and `eu7_real_corpus_ac.rs:477` assert on bodies —
extract `hit.body` so the 0.90 floor measurement is identical.

### F2 — By-id read `read.get` / `filter_logical_id_eq` (G2) — ADD 0.8.0
Table-stakes point-lookup; standard solved pattern, fully encoded in v0.5.x
(`builder.rs:120` → `Predicate::LogicalIdEq`, SDK wrappers + napi roundtrip). No
research needed.

Lands as `read.get` / `read.get_many` under the new `read.*` namespace
(ADR-supersede B1). Non-destructive, NOT on the recovery denylist (logical-id
*mutations* stay CLI-only; reads do not). Typed args only — no filter grammar
needed for pure by-id, trivially inside the typed boundary. Must ship with SDK
parity (allowlist+parity conformance test rewrite). Add `ReaderRequest::GetById`
alongside the existing `Search` variant (`lib.rs:350`, reader pool exists), backed
by a point lookup on `canonical_nodes.logical_id` — **only after G0** adds the
column + partial-unique index. Open: interim `get(write_cursor)` before G0 vs
strict block on G0; `get_many` partial-result vs all-or-nothing; active-only by
default (`superseded_at IS NULL`).

### F4 — Operational-collection lifecycle (G3 read only) — ADD 0.8.0 (READ); GOVERNANCE deferred
READ subset is table-stakes: Mem0/Zep/OpenClaw all expose by-key + history reads
of stored agent state. World-class read shape = parameterized, allowlisted-field
equality+range filter with mandatory limit + cursor over an append-only log —
exactly v0.5.x `read_operational_collection`/`trace_operational_collection`
(`admin.rs:1651-1754`).

**Web finding (GOVERNANCE subset):** server-side configurable retention/TTL is NOT
a product feature in Mem0 or Zep (2025-26); both leave TTL/eviction to the app.
The world-class staleness pattern is **supersession-on-write (G0) +
retrieval-score decay (G9/G12)**, not op-collection retention. So defer
retention/secondary-index/validate/compact.

**Correction 1 (hard gate):** the engine seam (`ReaderRequest::ReadCollection` +
worker-loop arm + `OpStoreRow` + `Engine::read_collection` on the
ReaderWorkerPool/DEFERRED-tx path) is dormant-shippable *unconditionally* in
0.8.0, but the **SDK `read.collection`/`read.mutations` verb cannot land until
ADR-supersede is HITL-signed** (`impl-strategy.md:47-48,253-255`).

**Correction 2 (firstStep precision):** treat the `_for_test` SELECTs at
`lib.rs:2245-2297` as a **read-shape oracle** (columns, append-only `ORDER BY id`,
cursor semantics) — they are `COUNT(*)`/single-row helpers, NOT verbatim multi-row
reads. Author NEW multi-row SELECTs with mandatory limit + after-id cursor for
`operational_mutations` (~1M cap). Destructive governance ops
(compact/purge/retention-run/disable) stay CLI-only if ever revived.

### F10 — Dangling-edge detection + `restore_validated_edges` (G8) — ADD 0.8.0
World-class already encoded; no external source. Pattern: edge dangling iff
`(edge active) AND (NOT EXISTS active source node OR NOT EXISTS active target
node)`, `active = superseded_at IS NULL` (`admin.rs:864-875`). Critical
correctness nuance: bulk loaders legitimately insert edges before targets, so the
write-path check **defaults to FLAG-AND-COUNT, not reject**, and must be a
cross-row check inside the open transaction, never single-row `validate_write`.

**Correction (consumer):** OpenClaw downgraded `important → nice` — OpenClaw is
hybrid-search + dedup, not a graph consumer (`agent-memory-fit.md:340` "none
(markdown)"). Top-line **differentiating** holds (only Mem0/Zep-graph genuinely
need it). The bi-temporal framing is world-class *precisely because* it pairs with
G0/G11 — v0.5.x predicate is sufficient only because G0 lands first (hard dep).

No new `read.*` verb: write-path half is an additive
`WriteReceipt.dangling_edge_endpoints:u64` field (Py+napi parity).
`restore_validated_edges` stays CLI-only under `recover --restore-logical-id`.
Reconcile the EXISTS predicate against G0's `(logical_id,kind)` composite so the
probe is index-backed. Ship FLAG-only first; strict rollback mode optional later.

### F11 — Bitemporal `superseded_at` + identity substrate (G0/G11) — ADD 0.8.0 (G0 subset; design valid-time)
The keystone. v0.5.x encodes **only transaction-time** (single `superseded_at` +
partial-unique-active index + per-fact confidence) — **no valid-time /
four-timestamp model**, so v0.5.x is NOT a sufficient reference for the full
end-state.

**Web finding:** world-class end-state = Zep/Graphiti **bi-temporal**: four
timestamps `t'created/t'expired` (transaction/ingestion) + `t_valid/t_invalid`
(event/valid), with **invalidate-not-delete** edge supersession (new edge
invalidates but does not discard prior, preserving history without recompute).
v0.5.x `superseded_at` maps cleanly to transaction-time expiry; the valid-time
pair is net-new and the substrate ADR must **reserve column shape now** (Option
2A), even though 0.8.0 implements only the single-supersession subset.

**Framing correction:** the transaction-time identity half (G0) is
**table-stakes** for all three (by-id read, dedup-on-re-ingest, stable addressing —
the keystone constraint); the valid-time edge half (G11 full) is the
differentiating/world-class graph-flavored part. Labeling the whole
"differentiating" *under*-states G0.

firstStep: draft `ADR-0.8.0-canonical-identity-substrate` settling the
four-question checklist (ADR-retrieval §2A): (1) column shape admitting
transaction-time now + additive valid-time later without reshape, (2)
invalidate-not-delete explicit, (3) whether edges carry temporal cols, (4)
op-store cascade under supersession. Implement minimal subset
(`row_id/logical_id/superseded_at` + partial-unique-active index), porting v0.5.0
`bootstrap.rs` index shapes (`idx_*_active_logical_id WHERE superseded_at IS
NULL`). Open: forward-migration shape (in-place ALTER vs re-open) under
no-data-migration; shadow-table/projection reconciliation is a required follow-on
slice (G0 alone leaves stale projections competing for KNN slots).

### F1 — Graph traversal traverse/expand (G5/G6) — DEFER 0.8.x
v0.5.x `compile.rs:~523-560` ships a production-grade `WITH RECURSIVE`
depth-bounded BFS: directional joins (Out: `e.source_logical_id=t.logical_id`; In:
`e.target_logical_id=t.logical_id`), `instr()`-over-comma-joined-path visited-set
cycle guard, `superseded_at IS NULL` filtering, hard depth cap
`MAX_TRAVERSAL_DEPTH=50` (`compile.rs:197`; `TraversalTooDeep` `compile.rs:71-72`).
`traverse()` (`builder.rs:103`) + `expand()` (`builder.rs:374`) confirmed. This
**is** the world-class bounded-BFS-over-KG pattern (Zep/Graphiti) — **no web
research needed, reference directly portable.**

**Genuine porting work:** v0.5.x joins on `source_logical_id/target_logical_id`,
but current `canonical_edges` is `(write_cursor,kind,from_id,to_id,source_id)` with
NO logical_id/superseded_at — the CTE must be re-targeted to `from_id/to_id` once
G0 adds `logical_id+superseded_at`.

**Correction (consumer):** OpenClaw `important → nice` (no graph). Mem0 has a
graph variant + entity linking, so Mem0/WC `important` is defensible, but Mem0 ≠
OpenClaw. Top-line **differentiating** + defer-0.8.x unaffected
(`agent-memory-fit.md:308-309`).

Adds `read.neighbors(id, edge_type?, depth=1)` (G5) + `expand=` param on
`search()` (G6) — non-destructive reads, governance-eligible, Py+TS lockstep,
typed args + small fixed grammar (server-side compiled CTE, no raw SQL).
**Recommend SDK depth ceiling ≪ 50 (e.g. ≤3).** G6 retrieve+expand is the
higher-leverage primitive (search to seed roots → 1-hop expansion in one call) and
composes G1+G4+G5+G9. Open: namespace B1 vs B2; G5 alongside G0 in 0.8.0 vs hold
to 0.8.x; sequence G6 ahead of standalone G5; does traversal filter on edge
valid-time (G11) or only `superseded_at` at 0.8.x.

### F3 — Rich typed JSON-path filter — revive CORE as small grammar (G4); drop DSL+fused — DEFER 0.8.x
**Web finding:** industry standard (Pinecone/Qdrant/Weaviate 2025) is a FIXED
structured filter grammar (equality + range + list/in + geo over JSON payload),
pre-filtered before/alongside vector KNN (Qdrant pre-filter best-in-class). **None
expose a JSON-path query DSL to apps.** This validates the 0.8.0 "small fixed
grammar, not a DSL" stance and argues AGAINST reviving the full v0.5.x DSL surface.
(Verified against Qdrant primary filtering docs Match/Range gt/gte/lt/lte; two
aggregator URLs could not be independently confirmed — substance sound, secondary
URLs unverified.)

**Tier correction (split):** the **filtered-list capability (equality+range core)
is table-stakes** (all three: "all open to-dos / products rated >4★"). **F3-as
scoped** (full DSL framing + fused predicates) is broader than table-stakes; fused
predicates are query-planner internals (below differentiating). So: **core =
table-stakes (revive); fused/DSL-framing = drop.**

Revive the v0.5.x `Predicate` core (`ast.rs:83` — `JsonPathEq`,
`JsonPathCompare{Gt/Gte/Lt/Lte}`, `ScalarValue{Text/Integer/Bool}`;
`builder.rs:167-266`, with `filter_json_timestamp_*` delegating to
`filter_json_integer_*` at `247-266`) as a **closed typed enum**, surfaced under
`read.list(kind, filter?, limit)`. **Explicitly EXCLUDE**
`JsonPathFusedEq`/`JsonPathFusedTimestampCmp` and the `..._unchecked` builder
methods (`builder.rs:279-354`, which bypass the schema gate). Decide in the G4
grammar HITL (`fit-doc §7 Q3`).

**Dependency correction:** G10 SearchFilter sharing is **coordination, not a hard
blocker** — G4 list-filter and G10 search-filter can ship independently; strongly
prefer one shared closed enum. Open: shared enum vs two grammars; OR/nested boolean
vs implicit-AND of stacked typed predicates; JSON-path strings vs whitelisted
indexed-field set.

### F8 — Grouped/"aggregation" query — FOLD into G6 — DEFER 0.8.x
**Web finding:** shipping systems (Mem0, Zep, Letta) do NOT expose storage-layer
statistical aggregation (COUNT/SUM/AVG/GROUP BY); "aggregation" is done via
community-summary clustering (GraphRAG) + graph traversal, with true counting
app-side over retrieved sets. **No world-class storage-layer aggregation verb to
revive.**

Disambiguate two capabilities hiding under one v0.5.x name: (1) the retrieve+expand
fan-out (root nodes + named expansion slots) — **functionally G6, track it there**;
v0.5.x `compile_grouped_query` + `execute_compiled_grouped_read` +
`read_expansion_nodes_chunked` is a sufficient batched, depth/limit-bounded
reference. (2) true statistical aggregation — v0.5.x never implemented; **defer to
0.8.x+ as app-side.** No new verb; fan-out lands as the expand half of G5/G6;
expansion-slot filter reuses the small fixed grammar (do NOT revive the full DSL).

### F9 — Per-fact confidence REAL — DEFER 0.8.x (needs design-ADR + experiment)
**Web finding:** world-class confidence is NOT a bare nullable column. In
Zep/Graphiti confidence is a first-class attribute of a bi-temporal fact edge —
each edge carries a validity window + a confidence-of-assessment that
**strengthens with corroboration/source-count, decays with age when unconfirmed,
drops on contradiction (invalidate-not-delete)**. "Belief Memory" (arXiv
2605.05583) and SSGM (arXiv 2603.11768) formalize this as probabilistic belief
P(fact) updated by evidence, time-decayed, used directly in retrieval ranking. The
world-class artifact is a **population+update+decay POLICY feeding rerank**, not a
storage column — so v0.5.x is NOT sufficient.

Implementation pattern that fits FathomDB: store confidence as a vec0/projection
REAL (3-way sentinel reshape, same as planned G12 importance — keeps
migration-accretion-guard happy), populate from `job.body` at projection time,
reweight in Rust **AFTER** the bit-KNN prefilter (never as a vec0 predicate — would
starve recall), gated behind G9's `fusion_mode` flag. firstStep: design-ADR
disentangling confidence (epistemic: believability, evidence-driven, decays) from
G12 importance (salience: how-much-it-matters); decide whether they are even
distinct for the named consumers or whether one signal suffices for 0.8.x. Anchor
confidence-update to the G0 supersession (invalidate-not-delete) contract. No new
`read.*` verb; optionally appears in the G1 hit payload (Py+TS lockstep). Open:
distinct-from-importance?; population policy (writer-supplied vs corroboration
derived vs LLM-scored app-side); decay model (query-time exponential from
write_cursor vs materialized); storage locus (vec0 reshape preferred); edges-only
vs nodes too.

### F4-GOVERNANCE — retention/TTL/secondary-index/compact — DEFER 0.8.x
Per the web finding above, server-side retention/TTL is not table-stakes;
staleness is owned by G0 (supersession-on-write) + G9/G12 (retrieval-score decay).
Matches `op-store.md`'s documented 0.6.0 narrowing. Defer; if revived, destructive
ops stay CLI-only.

### F5 — FTS property schemas / fielded text — DEFER 0.8.x, RESHAPED (v0.5.x NOT sufficient)
**Web finding:** world-class fielded full-text = per-field/multi-column FTS with
**BM25F-style per-column weighting** (weight name/title/tags above body), NOT
concatenation into one opaque blob and NOT per-kind table proliferation. FTS5
supports this natively: a multi-column virtual table + column weights to `bm25()`
(e.g. `bm25(t, 10.0, 3.0, 1.0)`). The v0.5.x design (per-kind separate
`fts_props_<kind>` tables + recursive JSON-leaf extraction + position sidecar +
async rebuild staging) is **heavier and a different shape** than the world-class
minimum — reuse its JSON-path projection-spec *concept* but NOT its table-per-kind
/ position-sidecar machinery.

firstStep: design-ADR comparing (a) keep single `body` but add named FTS columns
`{name/title, tags, body}` with bm25 column weights (smallest blast radius,
world-class-aligned), (b) v0.5.x-style projection spec mapping kind→JSON-paths into
those columns at write time, (c) v0.5.x per-kind-table model (reject as
over-engineered) — **recommend (a)+(b)-lite.** Baseline current single-body FTS5
cost (`dev/profiling/03-fts5.md`) before adding columns. Projection spec is a typed
declaration via `admin.configure` (already a core verb), not a new read verb; any
property-FTS REBUILD stays CLI-only.

### F6 — Per-kind FTS/vec tokenizer presets — DROP (HITL 2026-06-01)
**Decision:** the per-kind tokenizer/profile surface (`set_fts_profile`,
`resolve_tokenizer_preset`, per-kind FTS/vec config) is **dropped** — not deferred.
Rationale stands and is *strengthened* by the consumer evidence: per-collection
tokenizer config is a niche language/domain-adaptation knob (Milvus 2.6 per-field
BM25; LanceDB; TencentDB Agent Memory), secondary to the primary lever (hybrid RRF
fusion + rerank, ~70/30 vector/BM25), and **no named consumer (Memex/Hermes/
OpenClaw — English-first) requires it.** Dropping it also removes the
per-kind-profile surface-governance burden under the superseded-cap regime. If a
genuinely non-English / source-code consumer ever appears, this re-enters via a
fresh ADR — it is not on the 0.8.x/0.9 roadmap.

**KEPT in 0.8.0 — NOT F6 — global FTS5 tokenizer default (HITL 2026-06-01):**
upgrade the hardcoded FTS5 tokenizer (`schema/src/lib.rs:158`, currently bare
`unicode61`) once-globally to **`porter unicode61 remove_diacritics 2`** (the
recall-optimized-English default) via the existing projection-rebuild path
(`search_index` is a DERIVED projection — recreating it + `Engine::rebuild_projections`
`lib.rs:2635` is the sanctioned pattern and does NOT violate no-data-migration,
which governs SOURCE records per `ADR-0.6.0-json-schema-policy.md:51`). This is a
**single global config choice with zero new surface** (no per-kind machinery, no
new SDK verb, no governance/ADR cost), independent of the dropped F6 per-kind
feature. It is an **in-scope 0.8.0 recall win** — fold it into the first
FTS-touching slice (e.g. the G1 structured-hits or a G9 fusion slice). Baseline
recall before/after via `eu8_ir_validation.rs` to confirm the gain and protect the
0.90 floor.

### F7 — In-process admin/maintenance SDK API — DROP
No algorithmic world-class approach — this is operational-surface PLACEMENT. Best
practice for embedded-DB maintenance (SQLite VACUUM/integrity_check, RocksDB
compaction) is out-of-band operator tooling, which FathomDB's CLI-only
doctor/recover already matches. The v0.5.x in-process AdminClient is the LESS-good
reference (it conflated operator recovery with the app runtime); the 0.6.0 reset
that moved it to CLI was the correction, and ADR-supersede codifies it permanently.

No code change. Engine seams already exist (`lib.rs` `check_integrity:2578`,
`safe_export:2596`, `rebuild_projections:2635`, `trace_source_ref:2652`,
`dump_schema:2765`) wired through `fathomdb-cli`. **Mechanism note:** the

## Research findings (world-class approaches surfaced via web — preserved with sources)

1. **Structured hybrid-search hits (F12/G1):** return human-readable non-vector
   fields `{id,kind,body,score}`, per-branch non-cross-comparable scores tagged by
   modality; fuse separately via RRF; match-field highlighting is a separable
   optional layer. Sources: elastic.co/what-is/hybrid-search;
   learn.microsoft.com/azure/search/hybrid-search-how-to-query;
   opensearch.org/blog/building-effective-hybrid-search-in-opensearch.
2. **Fixed structured metadata-filter grammar, not a DSL (F3/G4):** equality +
   range + list/in + geo over JSON payload, pre-filtered before/alongside vector
   KNN (Qdrant pre-filter best-in-class); no major vector DB exposes a JSON-path
   query DSL to apps. Primary: Qdrant filtering docs (Match/Range). Secondary
   (unverified URLs): aimultiple.com/vector-database-for-rag;
   zenml.io/blog/vector-databases-for-rag.
3. **Retention/forgetting (F4-governance):** Mem0/Zep leave TTL/eviction to the
   app; world-class staleness = supersession-on-write + retrieval-score decay, not
   server-side TTL. Sources: mem0.ai/blog/memory-eviction-and-forgetting-in-ai-agents;
   dev.to/…/trace-continuity-vs-mem0-vs-zep-ai-memory-governance-compared.
4. **Bi-temporal knowledge graph (F11/G0-G11):** four timestamps + invalidate-not
   delete edge supersession. Sources: Zep arXiv 2501.13956 (arxiv.org/html/2501.13956v1);
   Neo4j Graphiti blog (neo4j.com/blog/developer/graphiti-knowledge-graph-memory/).
5. **Probabilistic confidence/belief (F9):** confidence as a first-class
   bi-temporal fact-edge attribute; strengthens with corroboration, time-decays
   when unconfirmed, drops on contradiction; consumed as a rerank weight. Sources:
   Belief Memory arXiv 2605.05583; SSGM arXiv 2603.11768; Hindsight arXiv 2512.12818;
   atlan.com/know/zep-vs-mem0; mem0.ai/blog/graph-memory-solutions-ai-agents.
6. **Bounded BFS over a (bi-temporal) KG (F1/G5-G6):** cosine+BM25+BFS with
   RRF/MMR/cross-encoder rerank; highest-leverage form is retrieve+expand.
   Reference: Zep/Graphiti. v0.5.x recursive-CTE BFS is exactly this — no greenfield.
7. **Aggregation in agent memory (F8):** no storage-layer GROUP BY in shipping
   systems; "aggregation" via community-summary clustering (GraphRAG) + traversal,
   counting app-side. Sources: mem0.ai/blog/state-of-ai-agent-memory-2026; Mem0
   paper arXiv 2504.19413.
8. **Fielded full-text / BM25F (F5):** per-field multi-column FTS with per-column
   weights via FTS5 + `bm25()` column weights; strictly better than single-body and
   lighter than v0.5.x per-kind tables. Sources: spice.ai/learn/bm25-full-text-search;
   sqlite.org/fts5.html.
9. **Per-collection tokenizer config (F6 — DROPPED):** real but niche
   (CJK/source-code); secondary to hybrid fusion+rerank (~70/30); no named
   consumer needs it → dropped (HITL 2026-06-01). Evidence retained for the record:
   milvus.io/blog/…hybrid-search; github.com/CortexReach/memory-lancedb-pro;
   marktechpost.com 2026/05/23 TencentDB Agent Memory.
10. **Maintenance-op placement (F7):** out-of-band operator tooling, not in-process
    app-runtime API — corroborates CLI-only doctor/recover.

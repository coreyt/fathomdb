## 3. F9 design decision (detail in ADR-0.8.16-f9-...)

- **Importance column:** `canonical_nodes.importance REAL` (step-18). **3-way sentinel** (recommended,
  pending HITL ratification):
  1. **`NULL` = never assigned (graceful-absent)** → ranking treats the node as neutral; contribution ==
     the unweighted baseline. *This is the OPP-12 graceful-absent state — load-bearing for R-F9-4.*
  2. **`0.0` = explicitly de-weighted** → ranking applies the floor.
  3. **`(0.0, 1.0]` = explicit importance** → ranking scales the fused contribution.
  Symmetric with the existing genuine-NULL `canonical_edges.confidence` (NULL = neutral, defaults 1.0).
- **Ranking integration:** a new **opt-in, OFF-by-default** reweight (mirroring `recency_reweight_enabled`)
  — importance scales the node's fused RRF contribution; edge confidence scales the graph-arm contribution
  per the deferred ADR §4.1 sketch `graph_rrf_score(edge) = confidence × 1/(K + bfs_rank)`. **The exact
  weighting formula is an HITL sign-off item** (deferred-ADR §5 gate 3).
- **Observability (R-F9-2):** extend `PerHitExplain` (`#[non_exhaustive]`, so additive) with the
  importance/confidence contribution; `explain=True` surfaces it. This observability is *why 0.8.16 is the
  right slot* — the signal is only useful if a caller can see it act.
- **Ships OFF-by-default as a correct, observable, forward-compatible MECHANISM** — 0.8.16 does NOT claim
  an eval-quality win (that tuning stays gated on the deferred-ADR eval evidence; see ADR §5).

## 4. F9 → OPP-12 `rankable` mapping (the F-18/F-20 load-bearing constraint)

Per `OPP-12-C1-converged-contract.md` Q6a, F9's importance/confidence signal algebra **is** the ranking
mechanism the 0.8.20 projection registry's `rankable` role will graft. F9 must be shaped so 0.8.20's
idempotent `configure_projections` adopts it **without reshape**. Each contract clause → F9 mechanism:

| OPP-12 Q6a `rankable` clause | F9 (0.8.16) mechanism that makes it forward-compatible |
|---|---|
| "rankable = graceful-absent, **never blocking**" | `NULL` importance = neutral; never errors, never blocks a query (3-way sentinel state 1) |
| "engine **defers** a rankable projection it cannot yet honor — no error, no build" | F9 ranking is OFF-by-default and opt-in; declaring a `rankable` role stays inert until the weight is invoked |
| "grafts on the next idempotent `configure_projections` **once F9 exists**" | importance column + reweight are **add/drop idempotent**; 0.8.20 points a `rankable` role at them with no schema reshape / no data migration |
| "same idempotent build/drop mechanism as Q4" | the signal is a **projection-addressable per-record scalar** (`importance` on nodes, `confidence` on edges), NOT a query-only hardwired flag → a role can address it |
| C-1 "break-if-later fields frozen while the seam is fresh" | F9 schema shape carries **no field on the OPP-12 break-if-later list**; the column is named/typed so per-`IdSpace`/per-projection scoping is possible later (do NOT hardwire a single global ranking) |

**R-F9-4 (candidate AC):** *F9 is `rankable`-forward-compatible per OPP-12 Q6a — (a) importance/confidence
is a projection-addressable per-record scalar; (b) absent = graceful-neutral, never blocking; (c) ranking
integration is opt-in / idempotent add-drop so 0.8.20's `configure_projections` grafts a `rankable` role
without reshape or migration.* Acceptance: this mapping holds under review + a test asserting
absent-importance ranks identically to unweighted + no F9 field on the OPP-12 break-if-later list.

## 5. candle↔ONNX equivalence-measurement plan (R-ONNX-3 → feeds 0.8.18 #5)

**Goal:** measure and record the numeric Δ between the candle CPU reference (`CandleBgeEmbedder`,
BAAI/bge-small-en-v1.5, dim 384, CLS-corrected pooling) and `OrtBgeEmbedder` (ONNX Runtime), so 0.8.18 #5
can calibrate its vector-equivalence tolerance against a real number.

- **Probe set:** a fixed, deterministic, committed text probe set (order-stable), sized to exercise short/
  long/multilingual/edge-token inputs; re-embed on the 3090s (cuda:0/1, exclude K620) when there is GPU room.
- **Metrics (per vector, then distribution):**
  1. cosine similarity + L2 distance + max-abs element Δ (raw float divergence);
  2. **the load-bearing metric — 1-bit sign-quantization FLIP RATE.** The in-library query path is 1-bit
     Hamming; eu7 fidelity is quant-fidelity, so what matters downstream is whether ONNX vs candle produce
     the same sign bits. Record the per-vector and corpus flip rate — *this* is the number 0.8.18 #5 gates on.
- **Backend matrix:** candle-CPU (reference) × {ONNX-CPU (same-backend baseline), ONNX-CUDA, ONNX-ROCm/
  DirectML/OpenVINO as reach-hardware is available}. Report each Δ separately (cross-backend divergence is
  the hazard 0.8.18 exists to catch).
- **Interim guard (documented, not enforced at 0.8.16):** **same-backend build-and-read** — vectors WRITTEN
  by a given embedder backend must be READ/queried with the SAME backend until 0.8.18 #5 enforces a
  tolerance. Never mix candle-written vectors with ONNX-read queries. This mirrors the 0.8.14 eu7 policy
  (`649a8d45`: fidelity gates run CPU same-backend; a cross-backend run is an artifact, not a regression).
- **Output:** a recorded Δ table in the Slice-15 run log + a one-line hand-off to the 0.8.18 #5 tolerance.

## 6. ONNX design decision (detail in ADR-0.8.16-onnx-...)

- New `OrtBgeEmbedder` in `fathomdb-embedder` (`mod ort_bge`, sibling of `candle_bge`/`nomic`), `impl
  Embedder` from `fathomdb-embedder-api`. Injected via `EmbedderChoice::Caller(Arc::new(OrtBgeEmbedder::…))`
  → **zero engine diff**.
- New `onnx-embedder` Cargo feature gating optional `ort` (+ tokenizer/loader deps), mirroring the
  `default-embedder` gating so the thin `default = []` build stays ML-free (EMB-3 wheel-size gate).
- Device selection: reuse `parse_device_request` for `FATHOMDB_EMBED_DEVICE` grammar parity; add an
  ONNX-specific `resolve_device()` sibling mapping `DeviceRequest → ORT execution provider` (CUDA / ROCm /
  DirectML / OpenVINO / CPU). Config/env selects at `Engine::open` (R-ONNX-2), not compile-only.
- The `Default` variant stays candle-only (no change to `open_default_embedder`) unless separately wired —
  ONNX is a caller-supplied backend, keeping the footprint invariant intact.

## 2. Existence-axis state machine (freezes R-EX-1/2, R-TR-1/2)

- **`LifecycleState` = `{ pending, active, deleted, purged }`** — one mutually-exclusive typed enum, stored in
  **one indexed `state` column** + **one `reason` column** on `canonical_nodes` (C9; structural §1).
- **`InitialState` = create-time subset `{ pending, active }`** (default `active`). Cannot *create*
  `deleted`/`purged`.
- **Legal-transition table (engine-enforced):**

  | From → To | Name | Verb |
  |---|---|---|
  | pending → active | promote | `transition` |
  | pending → deleted | reject | `transition` |
  | active → deleted | soft-delete | `transition` |
  | deleted → active | **undelete** (never `restore`) | `transition` |
  | deleted → purged | purge | **`purge`** (separate verb) |

  Effective `transition` `to_state` domain = `{ active, deleted }`. All other moves illegal.
- **`IllegalTransitionError { from_state, to_state, legal: [LifecycleState] }`** — real typed pyo3 exception +
  napi error-with-fields; fields **`from_state`/`to_state`** (S7). Enumerates legal targets.
- **`state`-index + default-read exclusion — CONCRETE (was MISSING):** add a composite index
  `canonical_nodes_state_active_idx` chosen to serve the hot path, and add the predicate **`AND state = 'active'`
  co-located with the existing `superseded_at IS NULL` clause at EVERY retrieval site** (61 `superseded_at IS
  NULL` occurrences in `fathomdb-engine/src/lib.rs` — search + `read.*` + graph traversal). The design
  mandate: the exclusion is enforced *inside the index/query at each site*, never derived post-hoc (structural
  §0 — an un-honored exclusion surfaces excluded content in FTS/vector results). RED test: enumerate the
  retrieval entry points and assert a `deleted` node is absent from each. No-op on the shipped corpus (no
  non-active row exists) → eu7 no-op basis (§9).

## 3. `transition` / `purge` verb design (freezes R-TR-1/2, R-PG-1/2)

- **`transition(logical_id, to_state: LifecycleState, reason?)`** — keys on the **bare `logical_id` (`l:`
  only)**; enforces the table; illegal → `IllegalTransitionError`.
- **Non-`l:` id refusal (was MISSING):** a lifecycle verb called with a `Content` (`h:`) or `Passage` (`p:`)
  id returns a typed **`NotLifecycleAddressableError { id_space }`** (pyo3 exception + napi error-with-fields,
  X1-parity) — not a panic, not a silent no-op. RED test: `transition`/`purge` on an `h:` id raises it.
- **`reason` mutation semantics (was MISSING):** `reason` is a per-row column reflecting the CURRENT state's
  cause. On each `transition`: **promote/undelete → `reason` CLEARED to NULL** (the row is admitted; no
  standing cause); **reject/soft-delete → `reason` SET to the supplied value** (expected on the delete-family;
  NULL allowed but discouraged). `reason` is advisory (Memex's `retired`-vs-user-`deleted` discriminator), never
  engine-interpreted. RED test asserts the clear-on-admit / set-on-exclude behavior.
- **`purge(logical_id)`** — separate irreversible verb (C2), NOT on `recovery_denylist`. **Precondition:
  deleted-first** (legal only from `deleted`; else `IllegalTransitionError`); **idempotent** (purging an
  already-absent/purged id is no-op success). In **one transaction**, physically erase the node across the FULL
  target set (was MISSING — enumerated): all versions in `canonical_nodes`; its rows in `search_index`,
  `search_index_edges`, `search_index_v2`; its vectors in `vector_default` (vec0) + `_fathomdb_vector_rows`;
  its terminal-projection bookkeeping in `_fathomdb_projection_terminal`; and **edges** touching it in
  `canonical_edges` per gap-3 (cascade-remove). **These are the ROW-OWNED targets only.** The global/kind-level
  metadata registries `_fathomdb_projection_state` (kind high-water state) and `_fathomdb_vector_kinds` (kind
  registry) are **NOT purge targets** — they are not keyed to a node id and must not be touched by `purge`.
  The implementer must assert (a test-time sweep) that NO row keyed to the purged `logical_id`/content remains
  in ANY row-owned target. Honor the erasure preconditions (§8 gap-4).

## 4. C-2 typed `SearchHit.id` swap — `IdSpace` newtype (freezes R-ID-1/2)

- **`SearchHit.id` moves `write_cursor: u64` → `IdSpace { space: IdSpaceKind, value: String }`** (C-2 binding).
  `IdSpaceKind = { Logical /*l:*/, Content /*h:*/, Passage /*p:*/ }`.
- **`id` is non-null + id-space-total:** `l:<logical_id>` (governed) · `h:<content-hash>` (doc-seeded — the
  dominant class) · `p:<value>` (synthetic rerank). **`p:` value format (was MISSING):** synthetic passages
  carry `stable_id = None` today (`lib.rs:1192`); at the swap the `Passage` id `value` = the **caller-supplied
  passage ordinal** from `rerank_passages` input `(u64, …)` → `p:<u64>`. Non-addressable, but non-null + total.
- **`write_cursor` AND `stable_id` subsumed *into* `id`, not dropped.** **`id` VALUE = today's `derive_stable_id`
  output** (`l:`/`h:` unchanged for governed/doc-seeded) → real-gold keying continues on `id`; the positional
  `write_cursor` u64 is retired from the caller surface (kept engine-internal where a positional cursor is still
  needed for book-keeping).
- **Lifecycle-addressability is a type check:** verbs accept only `Logical` (`l:`); `Content`/`Passage` →
  `NotLifecycleAddressableError` (§3).

## 8. The six underspecified-for-Phase-1 rulings (plan §9) — for HITL sign-off

### Gap 1 (LOAD-BEARING) — surrogate `logical_id` ↔ `h:`: does the dominant hit class migrate `h:→l:`?

**RECOMMENDED RULING (1a): NO. Doc-seeded/anonymous hits STAY `h:<content-hash>` in Phase-1, AND the
anonymous-node surrogate-minting mechanism is DEFERRED IN FULL to Phase-2** (it lands-together with the
natural-key / projection registry that gives a surrogate a re-ingest-survivable *matching* key). Phase-1
lifecycle-addressability = governed nodes only (they already carry a `logical_id`).

Why defer the *mechanism*, not just the backfill (this is the rev-2 sharpening codex's BLOCK forced):
under a "mechanism lands, corpus not backfilled" halfway house, the surrogate minter would have **no Phase-1
consumer and no RED test** — dead, untested code (codex item 8). The two honest options are (1a) defer it
entirely, or (1b) land it lazily + RED-testable. **1a is recommended.**

Rationale (any one sufficient):

1. **Irreconcilable requirement conflict.** The opaque-surrogate purge precondition (structural §1.2 —
   surrogate MUST be opaque, never content-derived, else a retained stub leaks erased content) **directly
   conflicts** with real-gold keying stability (the key must be re-ingest-**survivable** = content-derived —
   the reason Cause-A `stable_id` exists). One id for the doc-seeded class cannot satisfy both. Phase-1
   priorities (eu7 no-op + real-gold) ⇒ keep `h:`.
2. **No natural key in Phase-1.** An opaque surrogate is minted fresh per ingest; without the Phase-2
   natural-key registry the engine cannot match a re-ingested chunk to its prior surrogate → re-ingest mints a
   *new* id → not meaningfully version-eligible AND real-gold breaks. The surrogate's value is realized only
   WITH Phase-2 machinery → premature in Phase-1.
3. **The Phase-1 consumer does not need it.** Structural §2: *"Memex's world-model records already carry
   logical_ids, so its lifecycle problem is closed by (i) [the C-2 swap] alone; (ii) [surrogate minting] is
   the general 'a database provides an object id' fix (net-new)."* No Phase-1 consumer needs doc-seeded
   addressability.
4. **Consistent with shipped invariants** (`lib.rs:11610` "never superseded-in-batch"; plan §1 "`h:` not
   lifecycle-addressable by design").

**RED-testable under 1a (resolves codex BLOCK):** (a) a governed-node hit carries `l:<logical_id>` and is
`transition`/`purge`-addressable; (b) a doc-seeded hit carries `h:<content-hash>` and a lifecycle verb on it
raises `NotLifecycleAddressableError`; (c) `SearchHit.id` VALUES are byte-identical to pre-swap `stable_id`.
No surrogate minter exists to leave untested.

**Alternative 1b (if HITL wants Phase-1 anonymous addressability):** mint an opaque surrogate **lazily,
on first lifecycle touch** of an anonymous node (never at rest / never a corpus backfill) — doc-seeded hits
still read `h:` until explicitly addressed. More surface, marginal value; still no corpus migration.
**Rejected option (1c — full `h:→l:` backfill):** breaks real-gold + eu7 no-op, needs the gap-2 remap; not
recommended.

**Contract amendments 1a requires (HITL sign-off, §11):** R-ID-3 re-scoped from "surrogate minted for
anonymous/doc-seeded nodes (lands-together)" → "governed nodes addressable via `l:`; anonymous stay `h:`;
anonymous surrogate minting → Phase-2." R-MIG-1 re-scoped: migration carries **existence columns only** (no
surrogate backfill). Plan R-ID-3 already reserves the "documented `h:` fallback per the Slice-0 ruling" option.

### Gap 2 — migration backfill vs real-gold keying (remap recipe)

**Under 1a: NO remap needed** — `SearchHit.id` VALUE = today's `stable_id` value → real-gold keyed on
`stable_id` continues to work; eu7 gold-keying is a true no-op. (Only 1c would need a
`write_cursor|old-h: → new-l:` remap artifact + re-key of stored gold; precedent F-8a @0.8.14 — not
recommended.)

### Gap 3 — purge edge-cascade scope

**RULING: edge handling IS Phase-1 `purge` scope (intrinsic to correct erasure); Phase-1 uses
edge-CASCADE-REMOVE, not the content-free stub.** A purged node with retained content-carrying edges leaks —
`purge` is incorrect without it (structural §1.2). Cascade-remove over stub because the stub form needs a new
representation *and* a Phase-2 `include_existence_stubs` reader to ever surface — no Phase-1 reader → a stub is
write-only dead state. Cascade-remove is complete erasure with no new read surface. (Edges key from/to by
`logical_id`; only `l:` nodes are purgeable → only governed edges affected.) HITL choice: cascade-remove
(recommended) vs stub (defers a Phase-2 reader).

### Gap 4 — `secure_delete` fresh-DB vs migration

**RULING: `PRAGMA secure_delete=ON` in the connection-open PRAGMA block (`fathomdb-engine/src/lib.rs:2846-2854`,
every open — fresh + migrated); `purge` relies on it (no per-purge VACUUM); pre-20 freelist content is a
DOCUMENTED RESIDUAL with an optional one-time operator VACUUM.** `secure_delete=ON` zeroes every future freed
page → `purge`'s own erasure is complete without VACUUM; it is a connection PRAGMA, not a schema column, so it
belongs at open, not in migration DDL. It **cannot** retroactively scrub freelist written before it was on
(structural honesty posture); a migration-time full VACUUM would but is O(db-size) expensive → document the
residual, don't force it. HITL choice: documented-residual (recommended) vs mandatory migration-VACUUM.

### Gap 5 — default-read exclusion without `ReadView`

**RULING: acceptable — a complete surface, not half-built.** No Phase-1 way to *search* a `deleted`/`pending`
row (that is Phase-2 `include_deleted`/`include_superseded`). **Undelete still works** via
`transition(deleted→active)` addressing the node directly by its known `logical_id` (not via search) — the
only Phase-1 mint of non-active states is the new verbs, so the consumer knows the id. **`deleted` ≠ `purged`:**
a `deleted` node RETAINS its FTS/vector index rows ("stays indexed behind a flag", structural §1) — excluded
only by the `state='active'` filter — so Phase-2 `include_deleted` surfaces it without re-indexing; `purged`
physically removes them.

### Gap 6 — `PreparedWrite::Node` state/reason extension

**RULING: in scope — required to create `pending` nodes.** `PreparedWrite::Node` (`lib.rs:1859`,
`#[non_exhaustive]`) gains `state: InitialState` (default `active`; `{pending, active}` — cannot create
`deleted`/`purged`) + `reason: Option<String>` (meaningful for `pending` = quarantine cause; **at create,
`reason` is stored verbatim; thereafter it follows the §3 clear-on-admit / set-on-exclude rule**). Additive;
`logical_id: None` legacy default preserved. In X1 + AC-074 delta (§10).

## 9. eu7 basis + gold-keying (R-GATE)

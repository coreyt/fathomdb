## 1. The two surfaces serve distinct User Needs

| | **G10 `search_filtered`** | **G4 `read.list`** |
| --- | --- | --- |
| User need | "Find relevant memories about **X**, **but only from source Y / kind K / after date Z / status S**." | "**List all** my open to-dos." "Give me **all** products rated >4‚òÖ." |
| Shape | A *constraint on a ranked hybrid/semantic search* (vector-KNN + RRF, optionally CE-rerank). | A *typed enumeration* over active canonical nodes of a `kind`. No ranking, no vectors. |
| Mechanism | Indexed **pre-KNN** filter over vec0 metadata columns ‚Äî narrows the candidate set before/alongside the vector scan. | `json_extract(body,'$.path') <op> ?` over the `canonical_nodes(kind)` partition; active rows only. |
| Kind of need | Retrieval-precision / scoping. | Table-stakes structured browse / query. |

These are **not** the same need. One narrows *relevance ranking*; the other *enumerates by structured
predicate*. A unified type must express both without collapsing either into the other's execution model.

## 3. Requirements the unification MUST consider

1. **R-FIL-1 / R-FIL-2** ‚Äî one typed `Filter` contract serving both paths; the shipped G10
   re-expressed **byte-identically** (RED‚ÜíGREEN G10 parity; `filter=None ‚Üí 0.7.2 SQL` pin holds).
2. **R-X-1 (X1 SDK parity)** ‚Äî Py + TS expose the *identical* unified contract (`test_surface.py`,
   `surface.test.ts`).
3. **0.8.15 router `constraints` block (the downstream consumer)** ‚Äî the reason #17 exists: give
   the dispatcher **one** typed-constraint surface to reason about, not two. This is the
   forward-looking requirement that motivates "unified type" over "thin adapter."
4. **Inherited ADR-0.8.0 safety constraints** ‚Äî the unified type stays a **closed grammar**: no
   DSL, no fused predicates, no `*_unchecked` builders, parameterized `json_extract` only,
   server-side path **allowlist**, injection-safety, implicit-AND composition.
5. **G0 substrate** ‚Äî `canonical_nodes(kind)` index; active rows = `superseded_at IS NULL` on
   logical_id-alone identity (the partition G4 enumerates over).
6. **G10 pre-KNN performance guarantee** ‚Äî the indexed vec0-metadata pre-filter must **not** be
   demoted to a slower post-KNN `json_extract`. This is a hard dispatch requirement: a
   metadata-eligible predicate in a search context must compile to the indexed pre-KNN path.
7. **acceptance.md locked (max AC-074)** ‚Äî no new AC ids; track by G-gap / feature tags only.

---

## 4. Consequence for the unification design (Option A)

Because the needs and stores differ, **Option A (one unified `Filter` type, two compilation
backends)** is correct *only if* it preserves **both** User Needs and **both** execution models:

- the **enumerate** need keeps its full json-path predicate set (G4 / `read.list`);
- the **scoped-search** need keeps its indexed pre-KNN guarantee (G10 / `search_filtered`).

This forces the ADR's **total backend-dispatch with typed rejection** (no silent fallback, no
demotion) and the **field-availability matrix** ‚Äî the mechanisms that keep "one type" from
degrading either need. A reserved-gap fallback is legitimate **only** if a field genuinely cannot
resolve in a required store (named), never to skip dispatch work.

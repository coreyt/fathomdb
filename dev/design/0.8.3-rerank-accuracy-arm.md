# CE-rerank ACCURACY arm — does reranking convert the +0.39 oracle headroom into real accuracy?

## 1. Why this axis (not recall)
The gap-decomp says the lever is retrieval PRECISION — distractor-suppression / ranking — which **strict
recall@K cannot see** (gold is in-window but buried; the answerer fails). Recall gap −0.10 ≪ accuracy gap
−0.24. So the Slice-20 go/no-go must be measured on **answer accuracy**, not recall@10 (the ce_rerank_probe
recall arm is side-evidence only).

## 2. The arm
New answer arm **`fathomdb_reranked`**: for each gold query —
1. FathomDB retrieves its fused top-N pool (N=50, pinned — same as the recall probe).
2. `fathomdb.rerank(query, pool, rerank_depth=50)` → take reranked top-K (K=10).
3. Feed those K bodies (reranked order) to the SAME answerer (gpt-5.4) as gap-decomp/d0b.
4. Score accuracy with the SAME judge/exact-match the d0b/gap-decomp cells used.
Compare against the **already-paid** cells (reused, NOT recomputed): `fathomdb` (no rerank), `mem0`,
`oracle_raw`. Per-class + pooled, paired bootstrap CI (fixed seed), MDE — mirror gap_decomposition_run.

## 3. Frozen rule (pure module `rerank_accuracy_rule.py`; do NOT edit decision_rule_083 / gap_decomposition_rule)
- `lever_realized`: PASS iff paired **(fathomdb_reranked − fathomdb) accuracy** CI lower bound > 0 AND
  powered (per-class mde ≤ ε=0.05); underpowered ⇒ INCONCLUSIVE (mirror decide_083).
- Diagnostics (non-gated): `gap_to_mem0_closed = (acc_reranked − acc_fathomdb)/(acc_mem0 − acc_fathomdb)`;
  `oracle_headroom_captured = (acc_reranked − acc_fathomdb)/(acc_oracle_raw − acc_fathomdb)`.
- **GO** (→ Slice-20 engine wire) iff PASS AND `gap_to_mem0_closed ≥ 0.5` (closes ≥½ the Mem0 accuracy gap).

# Research note: enabling FathomDB for agent-memory applications

> Scope: how to evolve FathomDB so applications like **Memex**, **Hermes Agent**,
> and **OpenClaw Agent** can use it to (a) store incoming data, (b) retrieve
> data/knowledge, (c) work with "world-model" objects, and (d) support reasoning.
> Companion to [`dev/memex-note-on-0.6.0.md`](../memex-note-on-0.6.0.md).
> Grounded in the 0.7.2 tree (verified against source, not docs — see citations).

---

## 1. Central finding

**FathomDB's storage substrate is already most of a document + typed-graph + KV
store, and its hybrid retriever is genuinely strong. Almost all of what these
applications need is a *read surface* over data the engine already holds — with
one real data-model addition, record identity, which the project has already
scheduled for 0.8.0.**

The canonical schema physically persists nodes (`kind`, `body`, `source_id`),
typed edges (`kind`, `from_id`, `to_id`, `source_id`), an operational KV store
(`operational_state`) with an append-only mutation log (`operational_mutations`),
an FTS5 index, and a 768-dim vector index — see
`src/rust/crates/fathomdb-schema/src/lib.rs:105-202` (migrations 2–10). The
public SDK deliberately exposes only five verbs (`open`, `write`, `search`,
`close`, `admin.configure`) and `search()` returns `list[str]` of bodies only
(`docs/reference/python-api.md:58-64`, `src/rust/crates/fathomdb-engine/src/lib.rs:1766`).

**One sharp caveat that reshapes everything below:** the current substrate has
**no record identity**. `canonical_nodes` is `(write_cursor, kind, body,
source_id)` — there is no `logical_id`, no `properties` column, no timestamps,
no soft-delete. Structured fields live as JSON *inside* `body`; node addressing
is by internal `write_cursor` only. Edges store **opaque caller-provided
strings** in `from_id`/`to_id` — the engine never joins them to nodes, never
validates them, and there is no `from_id`/`to_id` index (only `source_id` is
indexed). The `logical_id` / `superseded_at` / soft-delete substrate is
**deferred to 0.8.0** (`dev/roadmap/0.8.0.md:84-92`) and is the keystone these
applications actually need — see §3c.

So the question "is FathomDB too limited for these apps?" splits cleanly:

- **As a write target:** rich in *shape*, thin in *identity*. One `write()`
  batch can mix four `PreparedWrite` variants — `Node {kind, body, source_id}`,
  `Edge {kind, from, to, source_id}`, `OpStore {collection, record_key, body}`,
  and `AdminSchema` (`src/rust/crates/fathomdb-py/src/lib.rs:642-706`;
  `fathomdb-engine/src/lib.rs:5000-5066`). But there is no per-record id to
  address what you wrote (§3c).
- **As a read/knowledge surface:** intentionally narrow. There is **no**
  by-id read, filtered list, graph walk, or operational-collection read anywhere
  in the codebase (`grep` for `fn get`/`fn neighbors`/`fn list_`/`fn read_collection`/
  `fn traverse` returns nothing). `search()` runs a vector branch and a text
  branch, **dedupes by `body` string**, and returns `Vec<String>` of bodies
  (`read_search_in_tx`, `fathomdb-engine/src/lib.rs:3130-3245`) — each hit's
  `rowid`/`write_cursor` and `kind` are in hand during the read and dropped
  before the result is built.

The implication: enabling these applications is **one keystone data-model
addition (record identity, already scoped for 0.8.0) plus a surface-exposure
project over data the engine already holds** — not a storage-model rebuild.

## 2. Ground truth: what the engine stores vs what it exposes

| Capability | Physically stored? | Writable via SDK? | Readable via SDK? |
|---|---|---|---|

| Structured properties | ⚠️ only as JSON inside `body` (no column) | ✅ (in body) | ❌ not as fields |
| Stable record identity (`logical_id`) + idempotent upsert | ❌ **not yet** — deferred 0.8.0 `roadmap/0.8.0.md:84` | ❌ | ❌ |
| Typed edges (`kind`, `from_id`, `to_id`, `source_id`) | ✅ `schema:111,202` | ✅ `py:675-683` | ❌ no traversal |
| — edge endpoints joined/validated against nodes | ❌ opaque caller strings; no `from_id`/`to_id` index | — | ❌ |
| Operational KV (`operational_state`) | ✅ `schema:136-143` | ✅ `engine:5056` | ❌ no read |
| Operational append-only log (`operational_mutations`) | ✅ `schema:127-135` | ✅ `engine:5044` | ❌ no read |
| Provenance (node/edge → `source_id`), indexed | ✅ `schema:200-203` | ✅ | ❌ SDK; ✅ CLI `doctor trace` / `recover --excise-source` |
| FTS5 keyword index | ✅ `schema:158` | ✅ (projection) | ✅ (text branch of `search()`) |
| Vector index (`FLOAT[768]` + binary quant, sqlite-vec) | ✅ `engine:4524`,`schema:171` | ✅ (projection) | ✅ (vector branch of `search()`) |
| Soft delete / supersession (`superseded_at`) | ❌ **not yet** — deferred 0.8.0 `roadmap/0.8.0.md:86` | ❌ | ❌ |
| Hybrid retrieval (vector branch + text branch, dedup by body) | ✅ `engine:3130-3245` | — | ✅ `search()` → `list[str]` |

**Read it as:** the rows on the left are the union of what Memex/Hermes/OpenClaw
need. *Storage shape* and *write* are mostly green; *read* is mostly red; and
two foundational rows — **record identity** and **supersession** — are not built
yet and are exactly what 0.8.0 is scoped to add. That identity gap is upstream of
almost every red cell: you cannot offer `get`, a stable search hit, an edge
join, or dedupe-on-re-ingest until records have ids.

## 3. The four application needs, mapped to FathomDB

### 3a. Store incoming data
>
> websites, journal articles, how-tos, journals, to-do lists, videos, meeting
> transcripts, products + evaluations, podcasts, named entities + attributes,
> graph relationships

**Verdict: well-supported today, with one shaping decision.**

Each of these is a `canonical_node` with a `kind` discriminator
(`"website"`, `"article"`, `"transcript"`, `"product"`, `"entity"`, …), the
indexable text in `body`, and everything structured packed **as JSON inside
`body`** (there is no separate `properties` column today): URLs, authors,
durations, ratings, timestamps, entity attributes. Relationships
(article→author, transcript→project, product→evaluation, entity→entity) are
`canonical_edges` written **in the same `write()` call** as the node
(`py/lib.rs:675-683`).

Shaping decisions the app owns:

- **Chunking.** `body` is what gets embedded (768-dim) and FTS-indexed (the
  async projection path off the write stream). Long sources (articles,
  transcripts, podcasts) should be chunked into child nodes (`kind:"chunk"`)
  edged back to a parent (`kind:"document"`) — FathomDB does not chunk for you.
- **Binary blobs.** Videos/audio themselves are not stored; store the
  transcript/description as `body` and a pointer (URI) in `properties`/`source_ref`.
- **Entity attributes.** Named entities fit as `kind:"entity"` nodes with
  attributes in `properties`; "information about the entities" that is itself
  searchable (a bio, a description) goes in `body`.

⚠️ The real constraint is **identity, not ordering**. The engine never joins or
validates `from_id`/`to_id` against any node — they are opaque strings it stores
verbatim (`py:675-683`; no `from_id`/`to_id` index exists). Worse, a node's only
handle is its internal `write_cursor`, and `write()` returns a single batch
`WriteReceipt{cursor}`, not a per-row id. So today an application **must mint and
carry its own id scheme** (e.g. a UUID embedded in `body` and reused as
`from_id`/`to_id`) to wire a graph at all; FathomDB will not detect dangling
edges. This is precisely the hole the 0.8.0 `logical_id` substrate fills, and the
strongest argument for sequencing identity first (§5).

### 3b. Retrieve data and knowledge
>
> canonical data, FTS5 index, vector search, graph relationships

**Verdict: hybrid text/vector retrieval is a strength; the other three retrieval
modes are blocked on the read surface.**

| Retrieval mode | Status |
|---|---|
| Semantic (vector) | ✅ vector branch: binary-quant ANN prefilter + exact-L2 rerank `engine:3141-3167` |
| Keyword (FTS5) | ✅ text branch: FTS5 `MATCH` over `search_index` `engine:3207-3237` |
| Hybrid (both) | ✅ run both branches, concat vector-first, **dedup by `body`** (BTreeSet) — *not* a fused/RRF score |
| Canonical by-id ("give me record X") | ❌ no `get()` |
| Filtered list ("all products rated > 4") | ❌ no `list(kind, filter)` |
| Graph ("everything linked to this project") | ❌ no traversal |

The hybrid retriever is genuinely good and is the part the apps should lean on
hardest for *candidate generation*. The three missing modes are exactly the
"structured-read" workstream the project has already drafted for 0.8.0
(`dev/roadmap/0.8.0.md:18-52`).

A second, lower-effort gap: `search()` returns bare strings, so an app cannot map
a hit back to its record without embedding an id inside `body` and parsing it
out — which is what the Memex note describes doing (`memex-note:37-58`). Returning
a structured hit (`kind`, branch/rank, `body`, and a record id) is cheap once G0
lands: the read already has each hit's `rowid`/`write_cursor` and `kind` in hand
and drops them when it builds the `Vec<String>` (`engine:3130-3245`). Before G0
it could emit the interim `write_cursor`; the durable answer is the `logical_id`.

### 3c. Work with "world-model" objects
>
> projects, colleagues, to-do items, named entities, meeting transcriptions,
> action items, project plans, schedules, emails, longitudinal + graph understanding

**Verdict: this is the hardest fit, and the crux of the Memex note.**

These are *mutable, structured, point-addressable* objects with lifecycles
("to-do: open → done", "action item assigned to colleague", "project plan
revised"). The substrate can hold them — node + JSON `body` + edges + the
operational KV for fast-changing state — but the app needs to **read them back by
identity, by filter, and by relationship**, all of which the surface forbids
today, and **address them by a stable id**, which the schema can't do until G0.

Mapping:

- **Projects, colleagues, to-do items, action items, entities** → canonical
  nodes by `kind`, related via edges (`todo --belongs_to--> project`,
  `action_item --assigned_to--> colleague`, `transcript --about--> project`).
- **Meeting transcripts about projects** → `kind:"transcript"` node, body =
  text (chunked), `--about--> project` edge; action items extracted as their own
  nodes edged to the transcript.
- **Schedules, project plans** → JSON in `body`, and/or mirrored into
  `operational_state` if they change frequently (the op-store's `latest_state`
  kind exists precisely for "latest-value app operational data",
  `schema:136-143`).
- **Emails** → nodes (`kind:"email"`), `--from/--to--> colleague` edges.
- **Longitudinal understanding** → there are **no per-node timestamps** today;
  ordering is the monotonic `write_cursor`, and mutable-state history lives in
  the append-only `operational_mutations` log (`schema:127-135`). Full record
  history (supersession of a node over re-ingestion) needs G0 + `superseded_at`
  (0.8.0). So the substrate for history is partly present, partly pending — and
  unreadable either way via the SDK.
- **Graph understanding** → edge rows exist but there is **no `from_id`/`to_id`
  index** (`schema:200-203` index only `source_id`), so both the index and the
  traversal verb are missing.

So world-model support is **blocked on record identity (G0) plus three reads**:
by-id, filtered list, and bounded graph walk — identity is the 0.8.0 anchor and
the reads sit directly on top of it.

### 3d. Reasoning support (provide relevant data)

**Verdict: best-served today, and the natural anchor for an agent-facing API.**

Reasoning support = "given a question/goal, return the most relevant context."
That is candidate generation, which the hybrid retriever already does well. The
upgrades that most help a reasoning agent:

1. **Structured hits** so the agent gets `kind` + id and can decide what to
   hydrate (cheap, §3b).
2. **Graph expansion of results** ("retrieve, then pull the 1-hop neighborhood")
   — the single highest-leverage feature for agent reasoning, because it turns
   flat retrieval into subgraph retrieval. This is my proposed G5+G6; note the
   0.8.0 retrieval list as written is "semantic + structured filter + rank
   fusion" (`roadmap/0.8.0.md:48-49`) and does **not** yet name graph expansion —
   worth adding to that scope explicitly.
3. **Filtered retrieval** ("search but only `kind:"action_item"` where
   `status="open"`") — combine vector recall with a structured predicate.

## 4. Gap analysis → what to build

Two of these need a small storage change (G0 + indexes); the rest are **additive
read surface over data already on disk**. Ordered so prerequisites come first.

| # | Capability | New verb / change | Backed by today | Needs | Effort | Unblocks |
|---|---|---|---|---|---|---|
| **G0** | **Record identity** (`logical_id` col + idempotent upsert + per-row id in receipt) | schema bump on `canonical_nodes`/`canonical_edges` | nothing — must be built | — | M | **prerequisite for G1/G2/G5/G7**; already the 0.8.0 anchor (`roadmap/0.8.0.md:84-92`) |
| G1 | Structured search hits | `search()` → `[{id, kind, score, body}]` | ids/kind in hand at `engine:3130-3245`, discarded | G0 for a *stable* id (interim: emit `write_cursor`) | XS | hit→record mapping (§3b/c/d) |
| G2 | By-id read | `get(id)` / `get_many([...])` | full table; no read verb | G0 + index on `logical_id` | S | world-model hydration |
| G3 | Operational read | `admin.read_collection(name, key?, filter?)` | `operational_state`/`_mutations` `schema:127-143` | nothing (data on disk) | S | schedules/plans/fast-changing state, history |
| G4 | Filtered list | `list(kind, *, filter?, limit)` | no `kind` index today (only `source_id`) | new index on `kind`; small filter grammar over body-JSON | M | "all open to-dos", "products > 4★" |
| G5 | Bounded graph walk | `neighbors(id, *, edge_type?, depth=1)` | edges stored but no `from_id`/`to_id` index | G0 + `from_id`/`to_id` indexes | M | graph understanding, subgraph retrieval |
| G6 | Retrieve + expand | `search(query, *, expand=1, filter?)` | — | G1+G4+G5 composed | M | the killer reasoning primitive |
| G7 | History read | `history(id)` / `mutations(collection)` | mutation log `schema:127-135`; `superseded_at` (0.8.0) | G0 + G3 | M | longitudinal understanding |
| G8 | Optional edge referential check | validate/flag dangling `from_id`/`to_id` at write | none — edges unvalidated (`py:675-683`) | G0 (endpoints become ids) | S | graph integrity for bulk ingest |

G0 is the keystone the project has already correctly identified and scheduled.
G1 + G3 are nearly free *without* G0 (G1 can emit the interim `write_cursor`; G3
touches no canonical schema) and remove much of the friction the Memex note
describes. G2/G5/G7 ride on G0. G4 needs only an index plus a deliberately small
filter grammar. G6 is pure composition — the highest-value reasoning primitive.

## 5. Recommended path — and the philosophy tension it must respect

The project's stated stance is explicit and load-bearing: FathomDB is
**"deliberately a retrieval/index engine, not a general-purpose document or graph
database"** (`docs/concepts/index.md`), the five-verb surface is locked under
invariant **AC-057a**, and SDK parity + recovery-as-CLI are positions, not
accidents (`docs/positions/*`). 0.8.0 is already drafted as the *identity +
structured-read* cycle and asks the right HITL question: *"is the five-verb
invariant negotiable to admit read verbs, and should they live under `admin.*`
(no invariant change) or as top-level verbs?"* (`0.8.0.md:55-64`).

Two coherent strategies, not mutually exclusive:

**Strategy A — Two-layer (ship now, no FathomDB changes).**
Exactly the Memex note's plan: app keeps an authoritative SQLite store for
structured/world-model state; FathomDB is the retrieval subsystem. Works against
0.7.2 *today*. Cost: the app reimplements identity, graph, and operational reads
it would otherwise get for free, and every app (Memex, Hermes, OpenClaw)
reimplements them separately. This is the right answer *if* FathomDB stays
retrieval-only by conviction.

**Strategy B — Open a curated structured-read surface (the 0.8.0 cycle).**
Ship G1–G6 as **additive read verbs**, which lets these apps collapse to a
single store without FathomDB becoming a SQL database. The discipline that keeps
this from sliding into "general document DB":

- Reads are **id/kind/edge-scoped**, never a SQL-like DSL. `list(kind, filter)`
  takes a *small, fixed* filter grammar over `properties` (equality + range), not
  arbitrary SQL. (`0.8.0.md:67-70` already draws this line.)
- Put the structured reads under `admin.*` *or* a new `read.*` namespace so the
  "five-verb write/retrieve core" stays conceptually intact and SDK-parity is
  enforced by the conformance suite (`docs/positions/sdk-parity.md`).
- Graph walks are **bounded** (`depth`, `edge_type`, `limit`) — a retrieval
  primitive, not Cypher.

**Recommendation:** the project's own 0.8.0 anchor already *is* this plan —
G0 (identity substrate) + knowledge-store + retrieval verbs. Make that explicit:
land **G0 first** (it's the keystone), pair it with **G1/G2** (cheap once ids
exist) and **G3** (needs no schema change at all), then gate **G4–G7 on the
explicit HITL invariant decision** the roadmap already names. **G8** rides on G0.
Tell the three apps to build against Strategy A *now* but design their storage
ports so the structured reads can migrate into FathomDB as B lands — i.e. a thin
repository interface that is satisfied by local SQLite today and by FathomDB read
verbs later. That makes FathomDB a *reference consumer-driven* roadmap rather
than guessing.

## 6. Suggested phasing (aligned to existing roadmap)

- **0.7.x (hardening / perf-only, in progress):** no new surface — preserve the
  five-verb invariant. Only doc work: write `docs/concepts/when-to-use.md`
  (Pattern A vs B) so the next app doesn't repeat the Memex investigation.
- **0.8.0 (knowledge-store + identity anchor, already scoped
  `roadmap/0.8.0.md:42-92`):** **G0 (identity substrate)** — the gating item —
  plus **G1 (structured hits)**, **G2 (`get`)**, and **G3
  (`admin.read_collection`, no schema change)**. These are the cheap,
  high-value wins once G0 lands.
- **0.8.x / 0.9.0 (gated by the HITL invariant decision):** **G4 (`list`+filter
  - `kind` index)**, **G5 (`neighbors` + edge indexes)**, **G6
  (retrieve+expand)**, **G7 (history)**, **G8 (edge integrity)** — the
  world-model and reasoning enablers, shipped only after the invariant call.

## 7. Open decisions for HITL (carry-overs, now with consumer context)

1. Is AC-057a negotiable to admit **read** verbs? (Reads don't touch the
   write/projection invariants; the risk is scope-creep, not correctness.)
2. `admin.*` / new `read.*` namespace vs top-level verbs for structured reads?
3. Filter grammar for `list()`: fixed equality+range over `properties`, or
   nothing (id/kind/edge only)? Draw the "not a query DSL" line explicitly.
   - *Triage resolution (F3):* pin the grammar to the v0.5.x `Predicate` CORE
     (`JsonPathEq`, `JsonPathCompare{Gt/Gte/Lt/Lte}`,
     `ScalarValue{Text/Integer/Bool}`) as a **closed typed enum** under
     `read.list` — never a parsed DSL string, never raw SQL. EXCLUDE
     `JsonPathFusedEq`/`JsonPathFusedTimestampCmp` and the `..._unchecked` builder
     methods (planner internals / bypass the schema gate). Prefer ONE shared
     closed enum with G10's `SearchFilter`. Industry basis: Pinecone/Qdrant/
     Weaviate fixed structured filter, no app-facing JSON-path DSL. See
     `0.8.0-v05-feature-triage.md` F3.
4. Do Memex/Hermes/OpenClaw commit to the **shared repository-port** design so
   FathomDB can treat them as reference consumers driving G4–G6?
5. Graph: ship bounded `neighbors()` in 0.8.x, or hold for 0.9.x as the roadmap
   currently hedges (`0.8.0.md:50`)?
   - *Triage resolution (F1):* v0.5.x `compile.rs:~523-560` `WITH RECURSIVE`
     bounded BFS is directly portable (visited-set `instr()` cycle guard,
     directional joins, `superseded_at IS NULL`, `MAX_TRAVERSAL_DEPTH=50`). Port
     AFTER G0; re-target joins from `source_logical_id`/`target_logical_id` to
     `canonical_edges.from_id`/`to_id` once G0 adds `logical_id`+`superseded_at`.
     SDK: `read.neighbors(id, edge_type?, depth=1)` + `expand=` on `search()`;
     clamp SDK depth ceiling to **≤3** (not 50). Sequence G6 (retrieve+expand)
     as the higher-leverage primitive. Defer-0.8.x. See triage F1.
6. Does "hybrid search" mean real **RRF fusion + rerank** (§8) or stay
   union-dedup? Today it's neither-scored union; shipping peers all fuse+rerank.
7. Do vectors carry **metadata columns** for filtered KNN (§8 table-stakes), or
   is filtering left to a post-fetch pass in the app?

## 8. External validation: shipping peers + the research literature

> Added after (a) confirming Memex / Hermes / OpenClaw are **real, public,
> local-first agent-memory products** (not internal), and (b) a verified
> deep-research pass over the agent-memory literature (Zep/Graphiti, Mem0,
> Microsoft GraphRAG, the Zhang et al. 2024 survey, Generative Agents,
> sqlite-vec). 25/25 researched claims confirmed under 3-vote adversarial
> verification. Citations are external URLs, listed at the end of this section.

### 8a. The decisive fact: the peers run on FathomDB's own substrate

Two of the three named consumers are built on **SQLite + sqlite-vec / FTS5** —
the exact stack FathomDB *is*:

| System | Substrate | Retrieval | World-model / graph | Read verbs it relies on |
|---|---|---|---|---|
| **OpenClaw** | markdown + per-agent **SQLite + sqlite-vec** | **hybrid BM25+vector**, weighted fusion, **MMR rerank**, **exp. temporal decay** (30-day half-life) | none (markdown) | `memory_search` (hybrid) + **`memory_get` (by-id/range)** |
| **Hermes** (Nous Research, OSS, Feb 2026) | **SQLite FTS5** | FTS5 + **LLM summarization** for cross-session recall; **Honcho** longitudinal user model | dialectic user model (not a graph) | session search; `/insights`, `/compress` |
| **Mem0** (memory layer; used by OpenClaw etc.) | vector store (+ graph variant) | **3-signal fusion**: dense + BM25 + **entity match**; consolidation | **entity extraction + linking** (native, no external graph DB) | search / store / **get** / **list** / forget |
| **Zep / Graphiti** (the reference design) | bi-temporal KG (Neo4j-class backend) | **cosine + BM25 + BFS graph**, RRF/MMR/cross-encoder rerank | full **bi-temporal entity graph**; edge invalidation | by-id, graph traversal, point-in-time |

The single most important conclusion: **FathomDB is the engine layer underneath
the precise stack these shipping agents already hand-roll on raw SQLite.** The
narrow five-verb scope was correct *sequencing* (get the substrate world-class
first); it is not a correct *ceiling* for this consumer class.

### 8b. What the literature says each pillar requires

- **Pillar 1 — store.** Provenance is first-class: keep raw **episodes**
  (non-lossy ground truth) that every derived fact traces back to (Graphiti's
  episode tier). Consolidation — **summarize / merge-dupes / forget** — is a
  named first-class operation (Zhang survey; Mem0). FathomDB already has the
  provenance column (`source_id`) and the op-store append log; episodes map onto
  canonical nodes; consolidation is gated on identity (G0).
- **Pillar 2 — retrieve.** The dominant pattern is **vector + BM25 + graph**,
  **fused via Reciprocal Rank Fusion** (`score = Σ 1/(rank+k)`, k≈60) because the
  methods score on incomparable scales, then **re-ranked** (MMR for diversity;
  cross-encoder for precision). **Filtered/metadata-constrained vector search** is
  table-stakes and **sqlite-vec already does it in one statement** via
  metadata/partition columns (brute-force, no ANN). FathomDB's current
  "union-dedup-by-body" is *below* this floor: no fusion score, no rerank, no
  metadata filter on the vector branch.
- **Pillar 3 — world model.** The world-class design is a **bi-temporal knowledge
  graph**: entity nodes with evolving summaries, fact edges as
  Entity→Relationship→Entity triplets carrying **four timestamps** (system
  created/expired + real-world valid/invalid). Longitudinal change is handled by
  **invalidating, not deleting** superseded facts (set old edge `t_invalid` =
  new edge `t_valid`) — which *is* entity-resolution-across-re-ingestion plus
  conflict resolution. This is the shape FathomDB's deferred 0.8.0 `logical_id` +
  `superseded_at` substrate (G0) begins; design it with the bi-temporal
  end-state in mind, not just single supersession.
  *(Slice 32 resolves the intended graph model around this fact-on-edge shape:
  [`../adr/ADR-0.8.0-graph-model-and-edge-addressing.md`](../adr/ADR-0.8.0-graph-model-and-edge-addressing.md)
  — one ontology-neutral substrate serving both corpus (GraphRAG) and memory
  (Graphiti) ontologies; edge-enrichment reserved-additive; no 0.8.0 substrate
  change.)*
- **Pillar 4 — reasoning.** Retrieval ranking should combine **recency +
  relevance + importance** (Generative Agents: weighted sum, exp recency decay,
  LLM-scored importance). **Subgraph / neighborhood retrieval** (retrieve, then
  expand 1-hop) and, for whole-corpus sensemaking, **community summaries**
  (GraphRAG map-reduce over clustered entities) are the multi-hop patterns —
  FathomDB's G5/G6 plus a rerank layer.

### 8c. New gaps this surfaced (beyond §4's G0–G8)

| # | Capability | Why it's now in scope | Maps to |
|---|---|---|---|
| **G9** | **Real fusion + rerank** — RRF over the vector & text branches, then optional MMR / recency-importance reweight | Every shipping peer fuses+reranks; FathomDB does scoreless union-dedup | upgrades `search()` internals; no new verb |
| **G10** | **Metadata columns on vectors** — filterable KNN (`kind`, time, status) in one statement | sqlite-vec supports it today; table-stakes for filtered retrieval | engine: add metadata cols to `vector_default`; pairs with G4 |
| **G11** | **Bi-temporal edges** — valid/invalid + created/expired on edges; invalidate-not-delete | the world-class longitudinal-understanding mechanism | extends G0/G5 schema; design now, ship later |
| **G12** | **Recency/importance signals** — per-record timestamps + optional importance score for rerank | no per-node timestamp today (only `write_cursor`) | small schema add; feeds G9 |

### 8d. Capability ranking for this datastore (research-backed)

**Table-stakes** (shipping peers have these *today*; FathomDB lacks most):
by-id read (G2), list/filtered list (G4), **FTS5 + vector hybrid with RRF fusion**
(G9), **metadata-filtered vector search** (G10), structured search hits (G1),
record identity (G0, prerequisite for all the above).

**Differentiating:** entity/fact **graph with bounded traversal** (G5) and
retrieve-then-expand (G6); basic **temporal validity** on facts (G11 minimal);
**consolidation** (merge/summarize/forget) built on identity; **recency +
importance re-ranking** (G9+G12).

**World-class:** full **bi-temporal knowledge graph** with **entity resolution
across re-ingestion** and **edge invalidation** (G11 full); **community
summaries** for global sensemaking (GraphRAG-style); importance/recency-aware
fusion as a first-class, tunable retrieval contract.

### 8e. Net read for the strategy

The research **does not contradict** the retrieval/index framing — it says a
world-class store for this class *is fundamentally a retrieval engine*, but one
whose retrieval includes graph traversal and whose records have temporal
identity. The honest conclusion: FathomDB's substrate and the perf-first
sequencing are right, but **table-stakes for the named consumers already exceed
the current surface** (by-id, list, RRF fusion, metadata-filtered KNN ship in
peers now). The 0.8.0 identity + knowledge-store anchor is the correct next step;
this section adds (1) treat **fusion+rerank (G9)** and **vector metadata (G10)**
as table-stakes, not differentiators, and (2) design the G0 identity substrate
with the **bi-temporal end-state (G11)** in view so supersession isn't built
twice.

### 8f. Open questions the research could not close

- Does sqlite-vec brute-force KNN scale to a single-user *lifetime* corpus, or is
  an ANN index / partition strategy required for world-class latency? (FathomDB's
  binary-quant prefilter + L2 rerank is already a partial answer worth measuring.)
- Native-SQLite graph traversal (recursive CTEs over edge tables) vs an external
  graph engine — latency ceiling for bounded `neighbors()`?
- Cost/latency of LLM-heavy ingestion (entity extraction, edge invalidation,
  community summaries) for an offline single-user agent — which consolidation
  steps run incrementally vs as background batch?
- Entity resolution / dedup across **multi-modal** sources (video, podcast,
  transcript) — the literature is thin; likely app-side for now.

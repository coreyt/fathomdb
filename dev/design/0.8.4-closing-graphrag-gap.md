# 0.8.4 — Closing the GraphRAG sensemaking gap: Tier 1 (fairness) → Tier 2 (function, almost graph-free) → Tier 3 (RAPTOR, then graph)

**Status:** decision-ready (Tier 1 build-and-run authorized; Tier 2 design for HITL; Tier 3 staged)
**Date:** 2026-06-23 · **Owner:** program steward · **Supersedes the "fund a graph-heavy S1" lean** in `0.8.4-HANDOFF.md` §3 fork A.

> **Premise.** Measured against a running Microsoft GraphRAG 3.1.0, FathomDB loses global
> sensemaking decisively (win-rate 0.00–0.33 vs the 0.45 near-parity band;
> `0.8.4-vs-microsoft-graphrag-RESULT.md`). A mechanism read (below) shows the loss is **part
> real** (flat top-K cannot cover a global question) and **part self-inflicted artifact** (the
> FathomDB reader was capped at `max_tokens=600` vs GraphRAG's ~1500; top-K was hardcoded at 8).
> This doc lays out a **three-tier** path to close the gap that is **almost graph-free** — it
> wins sensemaking on FathomDB's *embedding* mechanism (its measured strength), not on the
> *graph* mechanism (its measured weakness: M1 multi-hop NO-GO n=300; graph-arm ≯ BM25).

## 0. Design stance (HITL 2026-06-23) — function over footprint, user-controlled spend

FathomDB competes in a crowded space where **function can outweigh the original "small / local /
CPU-only" goal.** Local machines routinely have LLMs (cloud, frontier, local). The design
principle is to **expose the spend knobs and let the USER choose the operating point** for their
machine. The query-time CPU/1-bit path remains the *default* and stays intact; LLM-/GPU-using
work is an **opt-in build-time spend tier**, not a footprint violation. Every tier below names its
spend tier and keeps a cheap/local default. (Memory: `fathomdb-function-over-footprint-user-spend`,
`characterize-underperformance-then-hitl`.)

## 1. Mechanism — why vector loses comprehensiveness (the thing we are fixing)

A global question ("what are the themes/tensions across the corpus?") needs **coverage**; vector
top-K is a **similarity filter**. Four stacked failure modes:

1. **Similarity ≠ coverage.** An abstract global query embeds to a vague centroid near no concrete
   passage; top-K returns a biased narrow sample, not a representative cover.
2. **Redundancy collapse.** Similarity ranking has no diversity term → the K slots fill with
   near-duplicates of the single most-similar theme. (This is why *comprehensiveness* craters to
   0.00 while *diversity* holds at 0.32.)
3. **Abstraction gap.** "themes/patterns/tensions" don't match concrete evidence → weak ranking signal.
4. **Flat-synthesis ceiling.** Even at k=all, a flat reader does flat synthesis; GraphRAG
   synthesizes over **pre-summarized hierarchical community reports**, not raw text.

**How structural:** #1–#3 are intrinsic to flat top-K (the documented motivation for GraphRAG /
RAPTOR). The capability actually required is **hierarchical, whole-corpus coverage** — a graph is
*one* way to manufacture it, not the only way. Note real GraphRAG's *global* search map-reduces
over community **reports**, barely touching the entity graph at query time: the operative
ingredient is "a hierarchy of summaries," obtainable additively (RAPTOR / clustering) without any
graph.

## 2. TIER 1 — Fairness (build it now): match budget, raise k, add MMR, re-run

**Goal:** convert the *decisive-but-unfair* 0.00 into an **honest** number, before sizing any
build. Cheap (~$5–15 on the **preserved** Microsoft GraphRAG workspace), $0 footprint, eval-only.

**frozen-field: tier1-levers (2026-06-23)** — exactly three, no scope creep:
1. **Match generation budget.** FathomDB reader `max_tokens` 600 → **~1500** (≈6000 chars, matching
   GraphRAG's `[:6000]` global-search output). The single biggest comprehensiveness artifact.
2. **Configurable / raised top-K.** `k=8` was hardcoded; for a global question raise to **k=N_docs**
   (full coverage) — and expose `k` as a first-class retrieval parameter (product follow-on).
3. **MMR diversification (lever A).** Add Maximal Marginal Relevance to retrieval so the K slots
   cover *distinct* themes (fixes redundancy collapse #2). λ=0.5 default. **Moot at k=N (15-doc
   re-run) but essential for the powered full-corpus run (k≪N)** — built once, used by Tier 2/3 too.

**Build spec (eval harness, $0):**
- `eval/baselines_084.py::VectorRagAdapter` — add `retrieve_mmr(question, k, *, lambda_=0.5)`
  reusing `_embed`/`_cosine` (relevance = query·doc cosine; diversity = max doc·selected cosine).
  Keep `retrieve()` byte-unchanged (backward-compatible; MMR is opt-in). Deterministic tie-breaks
  (score desc, doc_id asc) preserved. New tests: determinism + a diversity property (near-duplicate
  corpus → MMR returns a more diverse set than plain top-K).
- `dev/plans/runs/0.8.4-tier1-fair-rerun.py` — re-run on `/tmp/grag` (preserved Microsoft GraphRAG
  index via `/tmp/gtest-venv`); **cache GraphRAG's 8 global answers to disk** (its side is fixed →
  future re-runs $0 on the competitor leg). FathomDB arms with matched `max_tokens≈1500`, `k=N_docs`,
  MMR-on. Re-judge cross-family (claude-haiku), ≥5 runs, order-swap, through `decide_084`. Resilient:
  per-key answer checkpoint + idempotent resume + `--max-usd` ledger guard (memory
  `priced-runs-need-resilience-before-spend`).

**What Tier 1 will and won't show (pre-registered honest read):** matched budget + k=all most
plausibly lifts the `fathomdb_mapreduce` arm on diversity/empowerment and *narrows*
comprehensiveness; it will **not** lift `fathomdb_vector` comprehensiveness much (structural — flat
retrieval, even uncapped, is not hierarchical synthesis). If fair-config FathomDB lands ≥0.45 on a
metric → that metric is *not* a real gap (it was the cap). Residual gaps after Tier 1 are the
**genuine** capability deficit that Tier 2 targets.

**Exit:** a fair head-to-head number + a decomposition (artifact-share vs structural-share per
metric) → decides Tier 2 scope. Re-uses `decide_084` unchanged (no endpoint switching).

## 3. TIER 2 — Function, almost graph-free (design for HITL): C (map-reduce QFS) + D2 (depth-1 coverage nodes)

Two additive capabilities that deliver whole-corpus coverage **without** entity extraction,
relationship modeling, or Leiden communities. Both reuse the embedder; the query path stays CPU-only
(LLM cost is build-time, user-controlled spend).

### 3a. C — Map-reduce QFS reader mode (the cheap additive win)

**What:** a first-class **global query mode** that, instead of top-K retrieval, **maps over all
chunks** (extract per-chunk relevant points, dropping NONE), then **reduces** into a synthesized
answer. This is coverage-complete by construction (it reads everything). It is the
`fathomdb_mapreduce` arm, promoted from an eval script to a real query mode and de-throttled.

**Design:**
- Query API gains a `mode="global"` (vs default `mode="local"`). Global routes to map-reduce; local
  keeps today's top-K retrieval. (Product: a `query --global` verb; eval: the existing arm.)
- **Map:** chunk the corpus (chunk = article at AP-News scale; configurable chunker for larger docs),
  batch chunks (≈5/call), extract relevant points or "NONE". Embarrassingly parallel, resumable.
- **Reduce:** hierarchical reduce for large corpora (reduce in batches → reduce-of-reduces) so the
  final synthesis is not a single overloaded pass — this closes the flat-synthesis ceiling (#4)
  **without** community structure. Matched generation budget (Tier 1).
- **Spend tier:** map+reduce LLM calls are **per-query** and scale with corpus size → the expensive
  mode. Default reader = **local Qwen3.6 ($0)**; opt-in frontier (gpt-5.4) for quality. The user
  picks per query. Cost is surfaced (projected $ before run).

**Footprint:** build-time = none (no index). Query-time = LLM-heavy *if* the user opts into global
mode — this is the explicit user-spend knob; local mode stays CPU-only/1-bit. **Weakness:** per-query
cost grows with corpus → at large scale, D2/D1's *pre-built* summaries amortize this. C is the
**right answer at small-to-mid corpus** and the always-available fallback.

### 3b. D2 — Depth-1 cluster-summary coverage nodes (the additive index, built via the ingest queue #6)

**What:** the minimal hierarchical index — **cluster all chunk embeddings once**, **LLM-summarize
each cluster** into a "coverage node," **store the summaries as new rows in the existing vector
store** (flagged `kind=coverage`). A global query then retrieves over the **coverage nodes** (whole-
corpus themes, pre-summarized) ± leaf chunks. This is "embedding-based communities" — GraphRAG's
hierarchy benefit, manufactured by clustering, **no graph**. At 15-doc / ~100-cluster scale this
likely captures most of GraphRAG's 115-community benefit for a fraction of the build.

**Design:**
- **Cluster:** k-means / GMM over chunk embeddings (reuse the engine embedder; CPU). Cluster count
  ≈ √N_chunks or elbow; deterministic seed. (No UMAP needed at this scale; add later for D1.)
- **Summarize:** one LLM summary per cluster ("synthesize the theme/claims of these chunks"). This is
  the only LLM cost, and it's **build-time, once.** Default **local Qwen3.6 ($0)**; opt-in frontier.
- **Store side-by-side:** coverage-node = `{embedding, summary_text, member_chunk_ids, kind:coverage}`
  written as **additional vector rows** alongside leaf chunks (`_fathomdb_vector_rows.kind` already
  distinguishes row kinds — see memory `embed-completeness-and-gpu-readiness`). **Purely additive:**
  if coverage nodes absent, global query falls back to C (map-reduce) or local top-K — zero
  regression risk.
- **Query (global):** retrieve top-K **coverage nodes** by query·summary cosine (+ MMR), feed their
  summaries to the reader; optionally drill into member chunks for evidence. CPU-only at query.
- **The ingest queue (#6):** D2's cluster+summarize build runs as a **rate-adaptive background
  queue** — ingest at a rate appropriate to the host (local-LLM throughput / cloud budget / idle
  GPU). Resumable, throttled, user-set spend ceiling. New docs → re-cluster affected region
  incrementally (full re-cluster only on a threshold of drift). This is where "user decides when to
  spend to unlock function" becomes concrete: the queue exposes rate + provider + budget.

**Footprint:** build-time LLM = opt-in spend tier (local $0 default; **idle 2×RTX 3090 + local
Qwen** is the natural $0 GPU play — memory `embed-completeness-and-gpu-readiness`). **Query-time =
CPU-only ANN over the same index** (coverage nodes are just vectors). Footprint invariant preserved
at query; build is the user-controlled spend.

**frozen-field: tier2-scope (2026-06-23)** — Tier 2 = {C reader mode, D2 depth-1 coverage nodes}.
**Explicitly excluded:** recursion (D1), entity extraction, relationship graph, Leiden communities.

**Tier 2 measurement:** re-run the head-to-head with FathomDB-global = D2 (and/or C) vs Microsoft
GraphRAG, same `decide_084` gate, powered + length-controlled → registered parity verdict. Kill-
early: if D2 ≯ C (depth-1 index buys nothing over map-reduce), don't build D1 — ship C.

## 4. TIER 3 — RAPTOR (D1), then graph (E) only if justified (staged design)

**frozen-field: tier3-sequence (2026-06-23)** — strictly ordered; each stage gated on the prior's
measured return.

### 4a. D1 — RAPTOR-style recursive summary tree (the side-by-side index that matches GraphRAG hierarchy)

**What:** generalize D2 to **recursion** — cluster chunks → summarize → **re-embed summaries →
cluster again → summarize again →** … building a *tree* of increasingly abstract summary nodes.
Query retrieves across **all tree levels** (collapsed-tree retrieval), so an answer mixes fine detail
and high-level themes. This is the documented **non-graph** competitor to GraphRAG (RAPTOR, Sarthi et
al. 2024) and the full version of D2's hierarchy.

**Design:** same store/query/queue machinery as D2 (all nodes are vector rows, `kind=coverage`,
`level=L`); adds UMAP-before-cluster at depth and a recursion-stop rule (≤K nodes at top level).
Build cost = D2 × tree depth (still build-time, opt-in spend, local-Qwen default). **Gated on D2's
return:** only build D1 if D2 reaches near-parity but a *deeper* hierarchy is needed for larger
corpora (D2's single level saturates).

### 4b. E — Entity / relationship / Leiden graph build (last resort, corpus-gated)

**What:** the full GraphRAG-grade graph index (entity/rel extraction → graph → Leiden communities →
community reports). Maximum fidelity **for entity-centric corpora** (people/orgs/events with dense
relationships). Heaviest build; this is the path that overloads FathomDB's graph mechanism.

**Gate (frozen):** E is funded **only if** (a) D1 has been built and measured, AND (b) a *later,
entity-centric corpus* (not AP-News sensemaking) shows D1/D2 leave a relationship-reasoning gap that
clustering cannot close. Absent that evidence, **do not build E** — the program's graph mechanism is
measured-weak (M1 NO-GO n=300; graph-arm ≯ BM25), so graph work must clear a high, corpus-specific
bar. E is documented here for completeness and to fix its precedence *after* D1, not before.

## 5. Sequencing, spend, and footprint summary

| Tier | What | Build spend | Query footprint | Gate to next |
|---|---|---|---|---|
| **1** | match budget + raise k + MMR + fair re-run | ~$5–15 eval (one-time) | CPU-only | honest decomposition → Tier 2 scope |
| **2C** | map-reduce QFS global mode | $0 (no index) | LLM per global query (opt-in; local $0 default) | C ≥0.45 on a metric → that gap was the cap |
| **2D2** | depth-1 cluster-summary coverage nodes | LLM once at ingest (local Qwen $0 / GPU) | **CPU-only ANN** (coverage nodes = vectors) | D2 ≯ C → ship C, skip D1 |
| **3D1** | RAPTOR recursive tree | LLM × depth at ingest (opt-in) | CPU-only ANN | D2 near-parity but deeper hierarchy needed |
| **3E** | entity/Leiden graph | heaviest (extraction + reports) | CPU-only ANN over reports | D1 built **and** entity-centric corpus shows a relationship gap |

**Invariant:** every tier keeps a cheap/local **default**; LLM/GPU is an **opt-in, user-controlled
build-time spend**; the **query path stays CPU-only/1-bit** except C's explicit global-mode opt-in.
Drop the graph (E) unless a future entity-centric corpus earns it.

## 6b. Batch-execution plan (HITL: "use batch mode via airlock") — resolved

Airlock batch (`docs/guide/batch.md`): OpenAI `/v1/files`+`/v1/batches` (~50% off, ~24h
turnaround) AND an **Airlock Batch Gateway "executor mode"** for **local vLLM**
(`qwen36-27b-vllm-batch` alias) that runs a submitted JSONL against the live vLLM endpoint
with bounded concurrency — **batch ergonomics, $0, no 24h wait.** Reused harness:
`eval/p0a_batch_e2e.py` (`build_batch_jsonl`/`run_batch`) + `autoe_judge.build_autoe_batch_jsonl`.

**The constraint that resolves the GraphRAG fork:** GraphRAG's **indexing AND global-query are
synchronous, dependent pipelines** calling `/v1/chat/completions` directly — they neither submit
to `/v1/batches` nor tolerate a submit-once/poll-24h model (stage N+1 needs stage N's output). So
**GraphRAG cannot use the batch API at all.** It doesn't need to: point GraphRAG at **local vLLM
`qwen3.6-27b` (sync, $0)** — cheaper than batch's 50%-off and with no 24h latency. (This is the
honest reading of the directive: batch can't apply to GraphRAG's internal calls; local Qwen
achieves the cost goal more completely.)

**Where batch DOES apply — our own independent call sets:**
| call set | who drives it | execution |
|---|---|---|
| GraphRAG index + global-query answers | GraphRAG (sync pipeline) | **local Qwen sync, $0** (batch N/A) |
| FathomDB C/D2 answers + D2 cluster summaries | us (independent per item) | **`qwen36-27b-vllm-batch`** ($0/fast) — or OpenAI `gpt-5.4-nano` batch for a stronger registered answerer |
| cross-family judge | us | sync **claude-haiku** (cheap; anthropic batch not configured here) — or `gemini-*-aistudio` batch |

**Two run tiers:** ($0 iteration) Qwen on both sides (matched synthesis model — good fairness,
like Tier-1 nano-both) + claude-haiku judge → cost ≈ the judge only (~$0.5–2 at powered scale).
(Registered verdict) matched `gpt-5.4-nano` both sides (FathomDB via OpenAI batch, GraphRAG sync-
nano) + claude-haiku judge. Self-preference control holds: judge family (anthropic) ≠ any
answerer family.

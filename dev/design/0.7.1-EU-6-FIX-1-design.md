## 1. Context

Codex reviewed EU-6 (commits `aeb9caf` RED + `c27712f` GREEN, not pushed) in
two passes and surfaced four packaging-axis findings that all reduce to one
root cause:

- **Pass-1 P1 — blocking.** `src/ts/package.json` `build:native` (the
  release-path script consumed by `release.yml`'s build-napi job) does not
  pass `--features default-embedder`, so the shipped `.node` never compiles
  the BGE loader — `useDefaultEmbedder: true` on a published install always
  raises `EmbedderNotConfigured`. EU-6's headline binding feature is broken
  on the actual artifact.
- **Pass-2 #5 — blocking.** `src/python/pyproject.toml [tool.maturin]
  features` lists `test-hooks` unconditionally, so every wheel
  `release.yml`'s build-python job emits via maturin-action ships the
  `_write_vector_for_test` / `_configure_vector_kind_for_test` hooks on the
  `Engine` PyO3 class (and the AC-067 force-panic probe) — leaking dev-only
  surface to PyPI consumers.
- **Pass-2 #6 — blocking.** Same `pyproject.toml` enables
  `default-embedder` for editable/dev wheels, which means the release
  workflow's reliance on pyproject feature discovery is dev/prod skew: tests
  pass against a binary nobody installs.
- **Pass-2 #4 — concern.** The EU-6 wheel-size gate
  (`.github/workflows/ci.yml` ≈line 152) only covers Linux x86_64 +
  macOS x86_64. Linux aarch64 and Windows x86_64 are first-class release
  matrix entries; the gate is partial coverage.

Net effect: pytest and vitest both validate a feature surface
(`test-hooks + default-embedder`) that diverges in both directions from
what the release workflow actually ships. P1, #5, #6 are the same
architectural mistake (dev-time-only toggles baked into the release-build
feature axis); #4 is a smaller variant (gate covers only 2 of 4 shipped
platforms). One slice fixes all four because they touch the same five
files and would re-collide if split.

## 2. Decisions register

| # | Decision | Rationale / legal basis |
|---|---|---|
| D1 | **Shipped feature axis: `default-embedder` ON, `test-hooks` OFF, on both bindings, for all 4 release platforms (Linux x86_64, Linux aarch64, macOS, Windows).** | EU-5b → EU-6 exists to give end users a `use_default_embedder=True` opt-in surface. A no-feature wheel makes that flag always raise `EmbedderNotConfigured`, defeating the entire campaign. The Cargo feature is compile-time (see ADR-0.7.1-default-embedder-weight-fetch.md §"Reversibility" + §"Scope guardrails": "the `default-embedder` Cargo feature gates the loader's network surface"), so the runtime opt-in only functions when the feature is compiled in. `test-hooks` is dev-only by design (see `src/rust/crates/fathomdb-py/src/lib.rs:543, 551, 707, 727` and the symmetric `fathomdb-napi/src/lib.rs:868, 890, 900, 908` — all are `#[cfg(any(test, feature = "test-hooks"))]` and have no production semantics). |
| D2 | **Wheel-size gate gates the with-feature wheel per platform** (the actual shipped artifact). No-feature wheel size is recorded **informationally** to detect dependency creep on the minimal-deps build path. | ADR-0.6.0-default-embedder.md EMB-3 says "Per-platform CI gate flags regressions > 20 MB". The ADR was written before EU-0 picked bge-small and before the EU-6 shipped-artifact decision (D1); applying EMB-3 to the no-feature wheel today would gate a build path no end user installs. The 20 MB budget moves to the with-feature artifact (the canary that matters); the no-feature line stays in the job summary so a future regression on the minimal-deps tree is still visible. |
| D3 | **Test-hooks reinstated for tests via an explicit `maturin develop --features ...` pytest fixture, NOT via pyproject discovery.** | `pyproject.toml [tool.maturin] features` is read by every consumer of `pip install -e .` AND by maturin-action in `release.yml` (it inherits unless overridden by `args:`). Leaving `test-hooks` there couples dev-build feature surface to release-build feature surface. The fix is mechanical separation: pyproject lists only what the release ships; pytest's session-scoped fixture rebuilds the binding with the dev-only features before any test imports `fathomdb._fathomdb`. TS side already has this separation (`build:native` vs `build:native:debug` in `src/ts/package.json:18-19`); only the production script needs `--features default-embedder` added. |
| D4 | **Single-wheel ship: no extras/split-package mechanism in 0.7.1.** One wheel per (platform, python). `pip install fathomdb` gets the with-feature build; there is no `pip install fathomdb[default-embedder]` extra-name in 0.7.1. | A split-package story (`fathomdb` + `fathomdb-default-embedder` PyPI packages, with the latter shipping the candle binary and the former pulling it transitively) is its own multi-slice campaign (PyPI does not let optional extras change native-binary contents; it would require a split-publish workflow + a separate napi sibling-package scheme). EU-6 / FIX-1 is not the place to invent that. Captured under "Alternatives considered" (§9). |

### 3.1 AC concerns

None blocking. Two notes:

- **AC-FIX1-5 ambiguity on "macOS (universal or x86_64)".** `release.yml`'s
  build-python matrix today emits two macOS wheels (x86_64 + arm64 — see
  `release.yml:75-78`). The wheel-size gate should follow the same shape,
  so this design treats AC-FIX1-5 as "macOS x86_64 + macOS arm64" (4
  platforms = 5 matrix entries) rather than a single universal/x86_64
  entry. If the orchestrator reads AC-FIX1-5 as "exactly 4 entries, pick
  one macOS target," collapse to macOS x86_64 only — the gate's purpose is
  per-platform-family regression detection and one macOS entry is the
  bare minimum the AC requires. Flagging this so A3 doesn't have to guess.
- **AC-FIX1-9 "`CARGO_BUILD_FLAGS` OR feature in `build:native`".** Both
  approaches work, but only the first survives a developer running
  `npm run build:native` locally without `CARGO_BUILD_FLAGS` set. Strong
  recommendation: put the `--features default-embedder` in `build:native`
  directly (AC-FIX1-7), and rely on that (i.e. AC-FIX1-9 is satisfied
  transitively by AC-FIX1-7, with no extra `CARGO_BUILD_FLAGS` env wiring
  in `release.yml`). The introspection test (D3 below) verifies it
  shipped, regardless of how it got there.

## 4. Per-platform wheel-size baseline derivation method

A3 derives the baselines as part of the GREEN commit by running, for each
matrix entry, the same `maturin build --release` invocation CI will run,
recording the resulting `.whl` size, and committing the number inline in
`.github/workflows/ci.yml`'s wheel-size matrix (the current matrix at
`ci.yml:170-185` is the template; A3 replaces the two `baseline_bytes`
literals with four/five and adds the missing matrix rows). The number
must be the **with-feature** wheel size (per D2). The no-feature size is
also captured per-step and emitted to the job summary, but is NOT a gate.

Baselines are recorded as raw bytes (matches the existing matrix style).
A3 should pad the baseline with a small headroom margin (≤5%) to absorb
Cargo.lock churn — the existing `3.06 MB → 3.1 MB` rounding in `ci.yml`
is the precedent. The 20 MB regression budget (AC-FIX1-5) is enforced
on top of the recorded baseline.

### 4.1 Per-platform commands (A3 runs these on a release-equivalent shell)

For each platform, A3 runs the command below **on the matching runner**
(GH-hosted, ideally via an ad-hoc CI dispatch or a local cross-build) and
records both `dist-feature/*.whl` and `dist-nofeature/*.whl` sizes. The
recorded number to commit is the with-feature size; the no-feature size
becomes a comment alongside.

|---|---|---|---|---|---|
| Linux x86_64 | `ubuntu-latest` | `x86_64-unknown-linux-gnu` | `cd src/python && maturin build --release --out dist-feature --target x86_64-unknown-linux-gnu -i python3.11 --features pyo3/extension-module,default-embedder` | TODO (A3 fills) | TODO (A3 fills) |
| Linux aarch64 | `ubuntu-latest` (cross via `--target` + `manylinux 2_28`) | `aarch64-unknown-linux-gnu` | same as above with `--target aarch64-unknown-linux-gnu`; if cross-compile fails locally, A3 may need to dispatch the wheel-size-gate job on CI once to capture the number | TODO (A3 fills) | TODO (A3 fills) |
| macOS x86_64 | `macos-latest` | `x86_64-apple-darwin` | same as Linux x86_64 with `--target x86_64-apple-darwin` | TODO (A3 fills) | TODO (A3 fills) |
| macOS arm64 (optional per §3.1) | `macos-latest` | `aarch64-apple-darwin` | same with `--target aarch64-apple-darwin` | TODO (A3 fills) | TODO (A3 fills) |

Cross-platform note: the gate is per-`runner`. A3 does NOT need to
cross-compile from one host; the wheel-size-gate matrix job runs on each
target's native runner (see `ci.yml:166-186` for the existing two-row
shape — extend it). If A3 cannot measure aarch64 on their local box,
they can dispatch the gate job once on CI and read the baseline from the
job summary's `::notice title=wheel-size::` line (which the existing
gate already emits).

### 4.2 NAPI side

`release.yml`'s build-napi job ships `.node` binaries, not `.whl`s, so
the size gate scope (per AC-FIX1-5) is the Python wheel. The `.node`
size is incidentally captured by build-napi's artifact upload; no
separate gate. (If the orchestrator wants a `.node` size gate too, that
is a follow-up — flag at GREEN review, not now.)

## 5. Test architecture

### 5.1 Release-surface tests (AC-FIX1-1 / -2 / -3 / -4)

These tests verify the **actually shipped** binary surface, not the
dev/editable surface. They are slow (a full release build) and
network-gated (for -3 / -4); the orchestrator runs them on demand.

**Python (`src/python/tests/test_release_surface.py`, new):**

A session-scoped pytest fixture builds the release-equivalent wheel
once, installs it into a throwaway venv (under `tmp_path_factory`),
re-execs python in that venv for each assertion. The fixture command
is exactly what `release.yml`'s build-python job invokes:

```python
# fixture body (pseudocode)
subprocess.check_call([
    "maturin", "build", "--release",
    "--out", str(out_dir),
    "-i", sys.executable,                 # use the running interpreter
    "--features", "pyo3/extension-module,default-embedder",
], cwd=python_src_dir)
# create a throwaway venv, pip install the produced wheel into it,
# yield the venv's python path.
```

Tests (run via `subprocess.check_output([venv_python, "-c", "..."])`
so they introspect the installed wheel, not the dev binding):

- **AC-FIX1-1 (negative)**: `assert not hasattr(Engine, "_write_vector_for_test")`
  AND `assert not hasattr(Engine, "_configure_vector_kind_for_test")` AND
  `assert "_force_panic_for_test" not in dir(<module>)`. Failure proves
  test-hooks leaked.
- **AC-FIX1-3 (positive, network-gated)**: `Engine.open(path, use_default_embedder=True)`
  succeeds and `engine.open_report.default_embedder.name == "fathomdb-bge-small-en-v1.5"`.
  Gated by `RELEASE_SURFACE_TESTS=1` AND not `FATHOMDB_SKIP_NETWORK_TESTS=1`
  (same skip-pattern as EU-5c).

**TypeScript (`src/ts/tests/release-surface.test.ts`, new):**

Symmetric mechanism. The fixture invokes:

```bash
# from src/ts; conceptually
cargo build --release --features default-embedder \
  --manifest-path ../rust/crates/fathomdb-napi/Cargo.toml
# rename target/release/libfathomdb_napi.{so,dylib,dll} to fathomdb.<platform>.node
# (or run `npx napi build --release --features default-embedder --js false`
#  which is what build:native will now do — preferred, since it tests the
#  exact production script)
```

Then `require('../fathomdb.<platform>.node')` (or invoke via a small
sub-`node` process that imports the file) and:

- **AC-FIX1-2 (negative)**: `assert.strictEqual('writeVectorForTest' in inner, false)`
  where `inner` is the loaded native module's `Engine` prototype. Also
  assert the `forcePanicForTest`/`forcePanicForTestPanic` exports are
  absent.
- **AC-FIX1-4 (positive, network-gated)**: instantiate
  `engineOpen(path, { useDefaultEmbedder: true })` and assert
  `openReport.defaultEmbedder.name === "fathomdb-bge-small-en-v1.5"`.

**Gate**: both test files top-line check `process.env.RELEASE_SURFACE_TESTS`
(TS) / `os.environ.get("RELEASE_SURFACE_TESTS")` (Py); if unset, the test
file skips with a clear log message. The default `npm test` / `pytest`
invocations therefore stay fast. CI sets `RELEASE_SURFACE_TESTS=1` on a
new dedicated job (see §6 below) so the surface is validated on every
push to `main`; the network-gated subset (-3 / -4) additionally requires
`FATHOMDB_SKIP_NETWORK_TESTS` to be unset (and benefits from the EU-5c
warm-cache step that already exists in `ci.yml`).

### 5.2 Pytest fixture rebuilds with `test-hooks` (AC-FIX1-6)

The fixture lives in `src/python/tests/conftest.py` (extend the existing
file shown at lines 1-19). Session-scoped, runs once before any test
imports `fathomdb._fathomdb`:

```python
# conceptual body
@pytest.fixture(scope="session", autouse=True)
def _install_test_hooks_binding():
    # Only run when invoked from the source tree (skip in installed-wheel
    # CI smoke runs). Detect via "src/python/pyproject.toml exists".
    if not (Path(__file__).parents[1] / "pyproject.toml").exists():
        return
    subprocess.check_call([
        sys.executable, "-m", "maturin", "develop",
        "--release" if os.environ.get("FATHOMDB_TESTS_RELEASE") else "",
        "--features",
        "pyo3/extension-module,test-hooks,default-embedder",
        "--manifest-path", "../rust/crates/fathomdb-py/Cargo.toml",
    ], cwd=Path(__file__).parents[1])
```

This is the only way `_write_vector_for_test` / `force_panic_for_test`
appear on the installed-into-venv `fathomdb._fathomdb` module that the
existing `test_ffi_safety.py` / `test_no_recovery_surface.py` / etc.
reach for. Without it the existing suite fails because pyproject no
longer carries `test-hooks` (the AC-FIX1-6 mechanical change). The
fixture run is one-shot per pytest session (~30-60s on cold cache;
warm-cache is near-instant since maturin reuses `target/`).

Risks (§8.1 below) discuss caching strategy if this becomes painful.

## 8. Risks & rollback

### 8.1 Pytest fixture build cost

The session-scoped `maturin develop` runs once per `pytest` invocation.
Cold-cache cost is ~30-60s (candle + tokenizers compile); warm-cache
(target/ from a previous `cargo build`) is ~5s. Local DX: developers
running `pytest -k test_errors` repeatedly hit warm cache, ~5s overhead.
CI: each `verify` job currently runs `agent-verify.sh` which does
`cargo build --workspace` first → target/ is warm → maturin develop is
fast.

Caching strategy if it becomes painful:

- Detect a pre-existing `fathomdb._fathomdb` module with the expected
  symbols (`hasattr(mod, "_write_vector_for_test")`) and skip the
  fixture's rebuild. This lets a developer prep the build once with
  `maturin develop --features test-hooks,default-embedder` and then
  iterate with `pytest --no-rebuild` (env-var gate).
- CI: cache `target/` via `Swatinem/rust-cache` (already in use) +
  cache `src/python/.venv` (not yet).

A3 ships the basic fixture; caching is a follow-up if cost shows up
on the CI dashboard.

### 8.2 Wheel-size baselines: bump procedure

When a legitimate dependency upgrade pushes the with-feature wheel past
its baseline budget:

1. Reviewer sees `::error title=wheel-size regression:: ... grew NNN B
   vs baseline MMM B` in the job summary.
2. Verify the growth is intentional (new dep, bump candle, etc.).
3. Edit the `baseline_bytes:` literal in `ci.yml`'s wheel-size matrix
   for the affected platform. Commit alongside the dep bump so the diff
   is co-located.
4. PR review sees both deltas in one place; no separate bookkeeping.

This matches the existing pattern (`baseline_bytes: 3100000` at
`ci.yml:181, 185`) so reviewers already know the shape.

### 8.3 Release-surface tests are slow / network-gated

`RELEASE_SURFACE_TESTS=1` is the on-switch. Fallback if not set:

- Default `pytest`/`npm test` skips the release-surface files with a
  clear `pytest.skip("set RELEASE_SURFACE_TESTS=1 to enable")` log line.
- CI exports the env var in a dedicated `release-surface` job (one
  Linux runner is enough — the surface is platform-agnostic for the
  introspection ACs -1/-2; -3/-4 are network-gated and skip if HF is
  unreachable, same as EU-5c).
- If both `RELEASE_SURFACE_TESTS=0` AND `FATHOMDB_SKIP_NETWORK_TESTS=1`
  in a given environment, the ACs go un-asserted in that run but the
  YAML-content tests (§6) still run, catching the most common
  regressions (someone re-adds `test-hooks` to pyproject).

### 8.4 Rollback

FIX-1 GREEN is one commit. If a follow-up campaign needs to revert:

- `git revert <FIX-1-GREEN-sha>` restores the pre-FIX state.
- No schema or migration implications (no DB changes, no
  `fathomdb-schema` step bump, no on-disk format change).
- The reverted state is "broken on shipped artifact" again; revert is
  only appropriate if a deeper redesign (e.g. split-package extras) is
  about to land in the same release.

## 9. Alternatives considered

- **Split-package PyPI extras (`pip install fathomdb[default-embedder]`).**
  Rejected: PyPI optional extras select Python dependencies, not native
  binary contents. To actually ship a smaller no-feature wheel + a
  separate package whose install adds the BGE bits would require a
  two-package publish (`fathomdb` + `fathomdb-default-embedder`) with a
  release-axis dependency. Symmetric npm story (sibling package +
  conditional dynamic require) is its own design. Multi-slice campaign,
  worth doing in 0.8.x if wheel size becomes a real friction point;
  not in scope for 0.7.1 FIX-1.
- **Ship no-feature wheel; leave `use_default_embedder=True` permanently
  raising `EmbedderNotConfigured`.** Rejected: defeats EU-6's reason for
  existing (per ADR-0.7.1-default-embedder-weight-fetch.md context
  section: "Shipping that model without first-use weight fetch would
  require either (a) bundling ~133 MB of weights into every wheel even
  for users who do not opt in, or (b) forcing the caller to supply the
  weights out-of-band — both of which nullify the 'default embedder = on'
  happy path"). The whole point of EU-1's ADR exception is to make the
  bindings work; reverting that is reverting the campaign.
- **Per-runtime feature detection (e.g. `use_default_embedder=True`
  attempts to dlopen candle at runtime, falls back gracefully).**
  Rejected: cargo features are compile-time. There is no runtime path
  that adds a Rust feature after the binary is built. A "graceful
  fallback" would have to be a different artifact entirely (the
  split-package alternative above).

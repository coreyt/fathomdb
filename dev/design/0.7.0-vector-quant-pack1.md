# 0.7.0 Vector binary-quant — Pack 1 design

This memo resolves the open design questions called out in
`dev/plans/prompts/0.7.0-PERF-VECTOR-QUANT-HANDOFF.md` § "Pack 1"
and § "Open items the implementing agent should escalate to HITL"
so that Pack 1 implementation can land mechanically. Pairs with
ADR-0.7.0-vector-binary-quant.md (which records the *what*); this
memo records the *exact shape*.

Scope: Pack 1 only — schema migration + writer double-write. No
reader changes. The Pack 2 query rewrite is out of scope for this
memo and is owned by a separate design pass when Pack 2 starts.

## D1 — Sibling column vs. sibling table

**Decision: sibling column on `vector_default`.**

The migrated declaration is:

```
CREATE VIRTUAL TABLE vector_default USING vec0(
  embedding float[<dim>],
  embedding_bin bit[<dim>],
  source_type TEXT partition key,
  kind TEXT,
  created_at INTEGER
);
```

(`<dim>` = 768 for the current default profile; the existing
`ensure_vector_partition` builds the dimension from
`load_default_profile` at `lib.rs:3278-3283`.)

- Alternatives weighed: sibling table `vector_default_bin` (one
  extra join on every read; fan-out cost on the Pack 2 reader; two
  inserts per row instead of one; two DELETE sites in every
  excise / rebuild path — `lib.rs:2177`, `:2181`, `:2265`).
- Citation: the sqlite-vec binary-quant guide demonstrates the
  exact mixed-flavour declaration
  `create virtual table vec_movies using vec0(synopsis_embedding
  float[768], synopsis_embedding_coarse bit[768])`
  (https://alexgarcia.xyz/sqlite-vec/guides/binary-quant.html);
  confirmed empirically by WebFetch on 2026-05-27 against the
  same page.

The sibling-column shape is the default in the handoff
(§ 1.1) and in the ADR (§ 2 / Pack 1.1). The sibling-table
fallback remains documented in both as a recovery path **only if**
the 16-metadata-column cap is exceeded (see D2) — that is not the
case for Pack 1.

## D2 — Metadata column inventory (vec0 16-col cap)

vec0 caps metadata columns at 16 and partition keys at 4
(`dev/notes/0.7.0-vector-cost-research.md:70-71`). The Pack 1
schema lands the minimum-viable set with headroom for the
real-workload extension already foreseen by the ADR and the
HITL doc.

Pack 1 column inventory on `vector_default`:

| # | Column            | Type           | Role                  | Source                                                         |
|---|-------------------|----------------|-----------------------|----------------------------------------------------------------|
| 0 | `rowid`           | INTEGER        | (implicit pk)         | `cursor` (existing — `lib.rs:2845-2851`).                      |
| 1 | `embedding`       | `float[<dim>]` | vector                | existing — `vec_quantize_binary` input + Pack 2 rerank.        |
| 2 | `embedding_bin`   | `bit[<dim>]`   | quantized vector      | `vec_quantize_binary(embedding)` at write time.                |
| 3 | `source_type`     | `TEXT`         | **partition key**     | derived from `kind` per D3.                                    |
| 4 | `kind`            | `TEXT`         | metadata col          | existing `ProjectionOutcome::Success.kind` (`lib.rs:2841`).    |
| 5 | `created_at`      | `INTEGER`     | metadata col          | UNIX seconds at write time (mirrors `excised_at` pattern at `lib.rs:2206`). |

Headroom: **11 free metadata column slots + 3 free partition-key
slots** after Pack 1.

Deferred to Pack 2 / 0.7.1 (NOT landed in Pack 1):

- `tags`, `project_mentions` — named in the handoff § 1.1 but not
  populated by today's writer; the chunk record carries no such
  field. Wiring requires upstream code changes that are out of
  scope for the Pack 1 schema-only slice. Headroom preserved.
- Any per-source metadata (sender, doc URL, etc.) — corpus-card
  work; lives outside Pack 1.

Rationale for landing `created_at` now: it is the one non-
controversial metadata field with a deterministic write-time
source (clock at insert), it is cheap to add to the migration,
and the alternative (a second migration in 0.7.1 to add it) costs
another ~64 s rebuild of the 1 M-row corpus. Adds zero
implementation risk because the writer site already calls
`SystemTime::now()` for the excise-audit path (`lib.rs:2206`).

## D3 — `kind` -> `source_type` mapping

The locked `source_type` vocabulary is
{`email`, `article`, `paper`, `meeting`, `note`, `todo`}
(`dev/plans/0.7.0-HITL-recommendations.md:135-137`; cardinality
"~6" in the ADR § 2).

The current writer at `commit_projection_outcomes`
(`lib.rs:2841`) receives `kind: String` from
`ProjectionOutcome::Success`. The only kind exercised by tests
today is `"doc"` (AC-013 fixture seed at
`tests/perf_gates.rs:350`), a synthetic name introduced for the
performance fixture and **not** in the locked source_type
vocabulary.

**Decision: the locked 6-value `source_type` vocabulary holds
unchanged. The synthetic test fixture kind `"doc"` is coerced
at the writer to `source_type = "article"`; the vocabulary is
not extended.** No HITL re-open required — the lock at
`dev/plans/0.7.0-HITL-recommendations.md:135-137` is honored
exactly.

Mapping table (Pack 1, deterministic, total):

| Incoming `kind` value | Resolved `source_type` |
|-----------------------|------------------------|
| `"email"`             | `email`                |
| `"article"`           | `article`              |
| `"paper"`             | `paper`                |
| `"meeting"`           | `meeting`              |
| `"note"`              | `note`                 |
| `"todo"`              | `todo`                 |
| `"doc"` (synthetic test fixture from `tests/perf_gates.rs:350`) | `article` (coerced) |
| anything else         | **writer error** — fail the projection outcome (`EngineError::Storage`); do not silently default. |

Rationale for the `doc` -> `article` coercion: AC-013 seeds
synthetic doc-shaped bodies (long-form text) that are the most
semantically aligned with the `article` source within the locked
6-value set. The coercion (a) preserves the partition_key
utility for the AC-013 fixture rows (they aggregate under a real
source_type bucket the corpus-build work will also populate),
(b) avoids any test churn — the fixture continues to write
`kind="doc"`; only the resolved `source_type` differs, (c)
matches what `CORPUS-1` (CNN/DailyMail acquisition, commit
`27ab8df`) will land as the production article source. The
coercion is writer-internal; the canonical `kind` value is
preserved verbatim in the `kind` metadata column (and in
`_fathomdb_vector_rows`), so any consumer needing to distinguish
fixture rows from corpus rows can still do so by reading `kind`.

Rejected alternatives:

- "Add `doc` as a 7th `source_type` value" — extends the
  HITL-locked vocabulary without authorization. This was the
  original draft and is the blocking finding in the codex
  verdict (`dev/plans/runs/0.7.0-PVQ-P1-DESIGN-review-20260527T132959Z.md`
  finding 1).
- "Rename the fixture kind from `doc` to one of the 6 locked
  values" — touches every test that reads the AC-013 fixture
  (larger blast radius than a writer-side coercion).
- "Leave `source_type` nullable for unmapped kinds" — loses the
  partition_key benefit for the AC-013 fixture and breaks the
  "partition_key is locked" guarantee in HITL Q5.

Citation: HITL lock at
`dev/plans/0.7.0-HITL-recommendations.md:135-137` (vocabulary +
"supersedes the `kind` default in the Pack 1 handoff").

The mapping is implemented in a single Rust helper:

```rust
fn resolve_source_type(kind: &str) -> Result<&'static str, EngineError> {
    Ok(match kind {
        "email" => "email",
        "article" => "article",
        "paper" => "paper",
        "meeting" => "meeting",
        "note" => "note",
        "todo" => "todo",
        // Synthetic AC-013 test fixture; coerced to "article" so
        // the 6-value HITL lock holds. See D3 rationale above.
        "doc" => "article",
        _ => return Err(EngineError::Storage),
    })
}
```

(no allocator pressure; returns `&'static str`; used at both
writer sites in D7.)

## D4 — NOT NULL + migration strategy

vec0 virtual tables in sqlite-vec `=0.1.7` (this repo's pin)
support neither `ALTER TABLE ... ADD COLUMN` (a SQLite-level
rule for virtual tables — https://www.sqlite.org/vtab.html) nor
`ALTER TABLE ... RENAME` (rename support landed in sqlite-vec
`v0.1.10-alpha.2`, April 2026, per the sqlite-vec release
history). The migration is therefore a `DROP TABLE` + recreate
under the same name. Repopulation must be **synchronous and
in-transaction**, copying the existing f32 blobs verbatim — it
must NOT go through the engine's async rebuild driver.

**Decision: land NOT NULL constraints from the start. The
migration is a single SQL migration step that (a) stages the
existing f32 rows + kind into a regular SQL table, (b) drops
and recreates `vector_default` with the new schema under the
same name, (c) repopulates `vector_default` from the staging
table with `vec_quantize_binary` evaluated SQL-side at INSERT
bind, and (d) drops the staging table — all inside the migration
framework's `BEGIN IMMEDIATE` / `COMMIT` envelope. The writer
enforces non-nullability from commit zero.**

Schema versioning is owned by the engine's real migration
framework: `migrate_with_event_sink(...)` at
`src/rust/crates/fathomdb-engine/src/lib.rs:1238`, which is a
thin re-export of `fathomdb_schema::migrate_with_event_sink`
implemented at `src/rust/crates/fathomdb-schema/src/lib.rs:218-271`.
`apply_one` (`src/rust/crates/fathomdb-schema/src/lib.rs:273-288`)
wraps each migration step in `BEGIN IMMEDIATE` ... `COMMIT`,
calling `conn.execute_batch(migration.sql)` followed by
`pragma_update(PRAGMA_USER_VERSION, step_id)`, and rolling back
on any error. `Migration` itself is `{ step_id: u32, sql:
&'static str }` (`fathomdb-schema/src/lib.rs:51-54`) — pure SQL,
no Rust callback hook. The new vec0 schema lands as a new
`fathomdb_schema::Migration` step appended to the existing
migration array passed into `open_with_migrations`
(`lib.rs:1119-1131`); `migrate_with_event_sink` bumps
`PRAGMA user_version` on commit and gates re-entry on the same
sentinel, so re-opens are naturally idempotent. The previously
proposed `operational_mutations` schema stamp is **not** the
schema-version owner — that collection is excise audit rows
(`lib.rs:2217-2221`) — and is dropped from the design.

Migration step (Pack 1) — single SQL blob, executed atomically
by `apply_one` (`fathomdb-schema/src/lib.rs:273-288`):

```sql
-- Step body. Runs inside BEGIN IMMEDIATE ... COMMIT supplied
-- by apply_one. <dim> is rendered at migration-build time
-- (768 for the default profile, see ensure_vector_partition at
-- lib.rs:3278-3283). <now> is the migration timestamp (UNIX
-- seconds), rendered by the engine when the migration array is
-- constructed at open_with_migrations (lib.rs:1119-1131); the
-- whole corpus is backfilled with the same created_at value.

-- 1. Stage existing f32 blobs + kind into a regular SQL table.
--    _fathomdb_vector_rows.kind is the source of truth for the
--    existing rows' kind (writer site at lib.rs:2845-2853).
CREATE TABLE _fathomdb_vector_migration_v0_7_0 (
  rowid     INTEGER PRIMARY KEY,
  embedding BLOB NOT NULL,
  kind      TEXT NOT NULL
);
INSERT INTO _fathomdb_vector_migration_v0_7_0(rowid, embedding, kind)
SELECT v.rowid, v.embedding, r.kind
  FROM vector_default v
  JOIN _fathomdb_vector_rows r ON r.rowid = v.rowid;

-- 2. Drop the old vec0 table and recreate under the same name.
DROP TABLE vector_default;
CREATE VIRTUAL TABLE vector_default USING vec0(
  embedding float[<dim>],
  embedding_bin bit[<dim>],
  source_type TEXT partition key,
  kind TEXT,
  created_at INTEGER
);

-- 3. Repopulate from staging. vec_quantize_binary runs SQL-side
--    once per row against the existing f32 blob; source_type is
--    coerced from kind via the D3 mapping using a CASE
--    expression. Any kind outside the D3 vocabulary raises
--    SQLITE_CONSTRAINT (NOT NULL) on source_type, which aborts
--    the migration and rolls back the whole step — matching the
--    "writer error on unknown kind" rule in D3.
INSERT INTO vector_default(
  rowid, embedding, embedding_bin, source_type, kind, created_at
)
SELECT
  s.rowid,
  s.embedding,
  vec_quantize_binary(s.embedding),
  CASE s.kind
    WHEN 'email'   THEN 'email'
    WHEN 'article' THEN 'article'
    WHEN 'paper'   THEN 'paper'
    WHEN 'meeting' THEN 'meeting'
    WHEN 'note'    THEN 'note'
    WHEN 'todo'    THEN 'todo'
    WHEN 'doc'     THEN 'article'  -- AC-013 fixture coercion; see D3.
    ELSE NULL
  END AS source_type,
  s.kind,
  <now>
FROM _fathomdb_vector_migration_v0_7_0 s;

-- 4. Drop the staging table.
DROP TABLE _fathomdb_vector_migration_v0_7_0;
```

Why the D3 mapping is inlined in SQL (and not a Rust loop): the
`Migration` shape at `fathomdb-schema/src/lib.rs:51-54` is
`{ step_id, sql: &'static str }` — pure SQL, no Rust callback
extension point. The CASE WHEN above is the literal byte-for-byte
transcription of the D3 mapping table; a Rust-side loop would
require extending the schema crate's migration trait (a
larger-blast-radius change that this slice does not authorize).
The CASE shape also avoids registering an ad-hoc Rust scalar
function on the migration connection. The D3 helper
`resolve_source_type` (defined in this memo for the writer
sites D5/D7) is the canonical lookup at write time; the CASE
above is the migration-time mirror of that table and must be
kept in lockstep with it — P1-IMPL adds a co-located assertion
in tests/migration coverage to catch drift.

Properties of this migration step:

- **Synchronous.** All four sub-steps run inside the same
  `apply_one` transaction (`fathomdb-schema/src/lib.rs:273-288`),
  on the migration thread, before `migrate_with_event_sink`
  returns. There is no async hand-off.
- **No embedder rerun.** The f32 blobs are copied verbatim from
  `vector_default.embedding` into the staging table and back
  out. The embedder is never invoked.
- **`vec_quantize_binary` call site.** SQL-side at the
  repopulation `INSERT`'s SELECT projection, once per row,
  evaluated against the f32 blob already materialised in the
  staging row. No Rust-side bit math; no double scan of the
  source blob.
- **Atomicity.** `apply_one` wraps the whole SQL blob in
  `BEGIN IMMEDIATE` ... `COMMIT` and `ROLLBACK`s on any error
  (`fathomdb-schema/src/lib.rs:273-288`). If any sub-step
  fails (e.g. an unmapped `kind` value triggers the NULL ->
  NOT-NULL constraint at sub-step 3), the migration is rolled
  back, `PRAGMA user_version` stays at the previous value, and
  the next open re-attempts the migration cleanly. The staging
  table is part of the same transaction and is dropped on
  rollback alongside the would-be new `vector_default`.
- **NOT NULL enforcement at write time.** vec0 metadata columns
  do not have a declarative `NOT NULL` constraint in the
  sqlite-vec DDL surface
  (https://alexgarcia.xyz/blog/2024/sqlite-vec-metadata-release/index.html);
  non-nullability is contract-enforced by the migration and by
  the D5/D7 writers. Pack 1 writers always supply all five
  metadata columns; the migration always supplies them (or
  fails atomically). No row can land with a NULL.

Migration cost: dominated by two sequential scans of the
existing 3 GB f32 corpus on the 1 M-row fixture — one for the
staging `INSERT ... SELECT` (sub-step 1) and one for the
repopulation `INSERT ... SELECT` (sub-step 3). The
`vec_quantize_binary` work is a sign-bit fold per coordinate,
negligible relative to the I/O. The working budget remains the
60-90 s estimate from ADR § 4. The migration **does not** re-run
the embedder, so the AC-013 562 s seed time
(`dev/plans/runs/0.7.0-PERF-EXP-W4.1-ac013-canonical-output.json`)
is **not** a comparable baseline — that figure includes embedder
work this migration avoids by construction. A concrete
measurement lands in P1-IMPL's GREEN gate.

Idempotence: `PRAGMA user_version` already guards re-entry
(`fathomdb-schema/src/lib.rs:234`, `:277`). On the second open
the migration step is skipped because `migration.step_id <=
before`, and `vector_default` keeps the rows from the first
pass.

### Rejected alternative — repopulate via `Engine::rebuild_vec0` / `RebuildKind::Vec0`

The fix-1 draft of D4 proposed driving repopulation through
`Engine::rebuild_vec0` (`lib.rs:1947`), which calls
`run_rebuild` -> `rebuild_shadow_state` with `RebuildKind::Vec0`
(`lib.rs:2234-2289`). Codex fix-1 review (finding 1,
`dev/plans/runs/0.7.0-PVQ-P1-DESIGN-fix-1-review-20260527T134214Z.md`)
correctly flagged that this path:

- Is asynchronous. `rebuild_shadow_state` only truncates vec0
  state and resets the projection cursor; the actual
  re-derivation runs on the projection scheduler. The test at
  `src/rust/crates/fathomdb-engine/tests/rebuild_vec0.rs:80-98`
  observes `rebuild_vec0` returns immediately and polls
  `projection_status_for_test` to wait for the async drain.
- Re-runs the embedder. The projection scheduler's
  `run_projection_job` calls `embedder.embed(&job.body)` at
  `src/rust/crates/fathomdb-engine/src/lib.rs:2528-2557` for
  every row it re-derives.
- Is therefore not equivalent to copying the existing f32
  blobs: it would multiply the migration cost by the embedder
  work the migration is specifically designed to avoid, and it
  would leak a window in which `vector_default` is empty before
  the async drain catches up.

The staged-copy migration above replaces this path entirely.
`rebuild_vec0` remains as it is in tree for the operator-driven
"rebuild from canonical" tool; Pack 1's migration does not call
it.

## D5 — Writer transaction shape

The writer site is `commit_projection_outcomes` at
`lib.rs:2845-2853`. The double-write lands in the **same**
`tx.execute` sequence between the existing
`_fathomdb_vector_rows` insert and the existing
`record_projection_terminal` call. No extra commit; no extra
lock; no extra connection.

**Decision: compute `vec_quantize_binary` inside the SQL
statement (single `INSERT` per row), not in Rust ahead of bind.**

The rewritten statement:

```rust
tx.execute(
    "INSERT OR IGNORE INTO vector_default(
        rowid, embedding, embedding_bin, source_type, kind, created_at
     ) VALUES(?1, ?2, vec_quantize_binary(?2), ?3, ?4, ?5)",
    params![cursor, blob, resolve_source_type(kind)?, kind, now_unix],
)?;
```

Justification (one paragraph):

The SQL-side shape keeps the f32 blob single-bound (one rusqlite
`Vec<u8>` allocation, one bind), lets sqlite-vec compute the
bit-vector against its own internal blob layout (which is what
the Pack 2 reader will also see), and avoids pulling a Rust-side
`vec_quantize_binary` dependency or re-implementing the sign-bit
fold in the engine crate. The "double-evaluation" concern (vec0
materialising `?2` twice) is a non-issue because the binary
quantization is a single sequential sign-bit pass over the same
in-memory blob — both passes operate on the same bound parameter
and the SQLite parameter store does not copy. The Rust-side
alternative would (a) require either linking against sqlite-vec's
C symbols from Rust or shipping a hand-rolled sign-fold, (b)
double the f32 footprint at bind time, and (c) couple the engine
crate to the on-the-wire bit format which is sqlite-vec's
internal concern. SQL-side is the cleaner contract surface.

## D6 — `vec_quantize_binary()` contract validation plan

The Pack 1 RED test is a round-trip:

1. Build a known f32 vector `v` of length `<dim>` whose sign
   pattern is hand-checkable.
2. Insert via the writer path.
3. Assert `vec_distance_hamming(embedding_bin, vec_quantize_binary(vec_f32(v))) = 0`.

Specified fixture (deterministic, no normalisation involved):

```
dim = 768.
v[i] = +1.0 for i in 0..384, -1.0 for i in 384..768.
```

Expected `embedding_bin`: 384 set bits in the leading half, 384
clear bits in the trailing half. Popcount = 384. The exact bit
ordering (LSB-first within each byte vs MSB-first) is an
sqlite-vec internal contract; the round-trip via
`vec_quantize_binary` guarantees both writer and reader see the
same packing, which is the only invariant the engine relies on.
`vec_distance_hamming` against the same vector binarised the same
way must be 0.

One fixture is sufficient. The previously-drafted secondary
"normalised alternating-sign" fixture is removed (its claim that
`normalise([1,-1,1,-1,...])` and the half/half vector share
"identical sign patterns at every coordinate" is wrong — one is
half-positive/half-negative, the other alternates — see codex
verdict finding 3). If P1-IMPL wants a second fixture for
defence-in-depth, the recommended shape is a vector that is +1
on every fourth coordinate and -1 elsewhere
(`v3[i] = +1.0 if i % 4 == 0 else -1.0` for dim=768): popcount
of the quantization is exactly 192 set bits, trivially
hand-checked, and clearly distinct from the half/half fixture's
popcount of 384.

Validation status: **empirical validation deferred to P1-IMPL's
RED test.** A one-shot `sqlite3` confirmation against a temp DB
with `sqlite-vec` loaded was considered for this memo but
rejected because (a) the Rust crate is the binding the writer
actually uses and the sqlite3 CLI binding is a separate code
path, (b) P1-IMPL's RED test exercises the engine writer
end-to-end which is the only contract that matters, and (c) the
binary-quant guide
(https://alexgarcia.xyz/sqlite-vec/guides/binary-quant.html)
already documents the round-trip shape and `vec_distance_hamming`
behaviour. If the RED test surfaces a contract surprise (signed
zero, NaN, byte-order), that is HITL escalation territory per
the ADR § 6 / "vec_quantize_binary() input expectations" — not
something this design memo can resolve in advance.

Open-item linkage: the ADR § 6 third bullet calls out the
fixture's "atypically friendly" sign-pattern concentration (mass
on 6 of 768 coordinates per `tests/perf_gates.rs:317`). That is a
recall-headroom concern for Pack 2, not a contract concern for
Pack 1. Pack 1's RED test is correctness, not recall.

## D8 — Reader-path NO-OP guarantee

The Pack 2 reader rewrite is out of scope. The Pack 1 question is:
does the **existing** reader at `read_search_in_tx`
(`lib.rs:2317-2323`) keep working over the migrated schema?

The existing query is:

```sql
SELECT rowid FROM vector_default
WHERE embedding MATCH vec_f32(?1)
ORDER BY distance
LIMIT 10
```

**Decision: confirmed no-op. The existing reader continues to
return identical results post-migration.**

Reasoning:

- The `embedding` column is **retained** (ADR § 2.1 "the f32
  `embedding` column is retained"). Its declared type
  `float[<dim>]` is unchanged.
- vec0's MATCH semantics dispatch on the column referenced in the
  predicate; adding sibling vector columns
  (`embedding_bin bit[<dim>]`) and metadata columns
  (`source_type`, `kind`, `created_at`) does not alter the
  semantics of `embedding MATCH vec_f32(?)`.
- The metadata + partition-key release blog
  (https://alexgarcia.xyz/blog/2024/sqlite-vec-metadata-release/index.html)
  demonstrates a vec0 declaration with `partition key` + plain
  metadata columns + the vector column, queried by the vector
  predicate alone; the additional columns are filterable via
  `WHERE` but do not become mandatory in the query shape.
- The Pack 1 migration preserves the `rowid <-> write_cursor`
  identity used throughout the writer loop (`lib.rs:2174-2175`
  comment, `:2177`, `:2181`, `:2265`); the `INSERT INTO
  vector_default_v2 ... SELECT v.rowid` clause is explicit
  about it.

AC-013 latency therefore stays at 2048 / 2327 ms post-Pack 1

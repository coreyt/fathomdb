# Slice 5 (D0a) — power-sized gold re-pin + answerer seam + Mem0-OSS de-risk

This slice establishes the **prerequisites** for the Slice-10 priced parity
measurement (`FathomDB − {Mem0, Graphiti/Zep, naive-RAG}` per memory class). It is
**eval-infra only**: Python, tests, and an offline ($0) gold build. No engine/Rust,
no priced parity pass.

The frozen `MEMORY_CLASSES` (decision_rule_083) the resolution scores are
`factoid`, `knowledge_update`, `multi_session`, `temporal`. The frozen COR-2 IR
corpus (`fe973fcd`) carried **N=0 gold** on the three non-factoid memory classes
(its gold is `exact_fact`/`exploratory`/`negative` only), which made every
Slice-25 memory-class delta null. This slice fixes that.

## (A) Mem0-OSS de-risk spike

### Root cause — why `Mem0OSSAdapter.try_build()` returns None

`try_build()` (r2_parity_eval.py ~L434) does `from mem0 import Memory` then
`Memory()`. Two independent failure modes, both confirmed here:

1. **`mem0ai` is not installed** in the eval environment — `import mem0` raises
   `ModuleNotFoundError: No module named 'mem0'` (confirmed). `try_build` swallows
   this and returns None → the Mem0 arm is silently skipped; every Slice-25
   `FathomDB − Mem0` delta was null for this reason **as well as** the N=0-gold one.
2. **Even when installed, `Memory()` with the default config needs a configured
   backend.** Mem0's `add()` runs an LLM fact-extraction pass and `search()` an
   embedding pass; the default config points the LLM at the OpenAI cloud API (needs
   `OPENAI_API_KEY`) and the default vector store is Qdrant (needs a server). With
   neither present `Memory()` raises and `try_build` returns None. **Mem0 cloud
   (`mem0.MemoryClient`) is forbidden — ADR §3.6.**

### Footprint-safe LOCAL backend (the D0b stand-up config)

All three Mem0 backend roles serve locally / $0, no cloud:

| Mem0 role | Local choice | Footprint |
|---|---|---|
| LLM (add-extraction + search) | airlock OpenAI-compatible endpoint, `qwen3.6-27b` (local vLLM at `192.168.1.45:8000`, proxied at `http://localhost:4000/v1`) | local, $0 |
| Embedder | `huggingface` provider, `BAAI/bge-small-en-v1.5` (CPU) — same family FathomDB embeds with, for a fair axis | local CPU, $0 |
| Vector store | `chroma` (embedded, on-disk) — no server | local, $0 |

The exact config is pinned as code in `eval/mem0_local.py::build_local_mem0_config()`
so the de-risk decision is reviewable and the D0b stand-up cannot drift from it:

```python
{
  "llm": {"provider": "openai", "config": {
      "model": "qwen3.6-27b",
      "openai_base_url": "http://localhost:4000/v1",
      "api_key": "<airlock key>", "temperature": 0}},
  "embedder": {"provider": "huggingface",
      "config": {"model": "BAAI/bge-small-en-v1.5"}},
  "vector_store": {"provider": "chroma",
      "config": {"path": "/tmp/mem0_chroma", "collection_name": "r2_eval"}},
}
```

### Install / runtime requirements (deferred to D0b — do NOT pollute the shared `.venv` in this eval-infra slice)

```bash
pip install "mem0ai" chromadb sentence-transformers
```

- The airlock LLM endpoint must be up (verified live this slice: `qwen3.6-27b` is
  the currently-loaded local vLLM model; `qwen3-32b` is NOT loaded — use
  `qwen3.6-27b`).
- `sentence-transformers` downloads `bge-small` once (~130 MB), then CPU-local.

### Residual risk + go/no-go

- **Risk:** Mem0 `add()` issues one LLM extraction call **per memory** — ingesting a
  large corpus through Mem0 is O(docs) local-LLM calls and is the slowest arm.
  Mitigation: restrict Mem0 ingest to the gold-relevant LME sessions (the same set
  the other arms see) and run on the resilient/checkpointed harness
  ([[priced-runs-need-resilience-before-spend]]).
- **GO/NO-GO: GO for D0b (Slice 10).** Mem0-OSS is tractable within the footprint;
  the blocker was (1) the missing install and (2) an unconfigured backend — both
  resolved by the pinned local config. The conformance test proves the adapter
  honours the shared `Hit` contract with a fake in-memory backend, so the wiring is
  validated **backend-free** now; the one-line install + the live local LLM make
  the real stand-up a GO, with the per-`add()` extraction cost flagged.
- **Graphiti/Zep:** stand-up **DEFERRED to Slice 10** (second comparator; same
  identical-answerer seam + a local Neo4j/FalkorDB config — recorded as a Slice-10
  task). This slice de-risks Mem0 only, per the prompt.

## (B) Power-sized gold re-pin

### Source — LongMemEval-derived (design §3)

Design §3 fixes the corpus as **LongMemEval-derived gold**. LME ships **real,
human-labelled** gold for exactly the memory classes, mapped by `LME_CLASS_MAP`:

| LME `question_type` | reporting class | LME-oracle count |
|---|---|---|
| `single-session-{user,assistant,preference}` | `factoid` | 70+56+30 = **156** |
| `temporal-reasoning` | `temporal` | **133** |
| `multi-session` | `multi_session` | **133** |
| `knowledge-update` | `knowledge_update` | **78** |

Three classes start **below** the provisional build target N≥150; the N=0 defect is
eliminated outright.

### Re-pin mechanism — LME real base + Qwen `gold_gen` top-up

Real LME gold is the authoritative base (highest quality, deterministic). Where a
class is short of N_min it is **topped up** with `gold_gen` generation (the slice's
prescribed tool) driven by the **local Qwen** (`qwen3.6-27b`, $0 OFFLINE-BUILD),
class-targeted over the LME sessions **of that class** (so `knowledge_update` is
generated only from sessions that actually contain an update, etc.). Every query
carries a `_provenance` tag (`lme-real` | `qwen-aug:<model>`); the manifest records
the per-class synthetic fraction, so the eval hygiene is auditable.

**Justified deviation (loud, not silent):** the prompt frames (B) as "rebuild via
`gold_gen.generate_memory_gold`". Pure `gold_gen` generation on the frozen IR corpus
(enron emails) yields temporal/multi_hop but **~0** `knowledge_update`/`multi_session`
(measured: 0/batch) — those classes require multi-session / fact-update source data,
LongMemEval's exact domain (design §3). The deviation is therefore **LME real gold
as the base + `gold_gen` as the top-up**, not `gold_gen` alone. Smallest change that
reaches the target with sound provenance.

### N_min and the power rule (design §5)

Frozen rule: N_min = the smallest N whose **paired MDE ≤ ε = 0.05**; the *number* is
provisional N≥150 (pooled ≥600), explicitly "to be confirmed/escalated at Slice 5".
Slice 5 cannot measure the *paired* (two-arm) variance — that needs the Slice-10
parity outcomes — so we:

- use **n_min = 150** as the build target (manifest `n_min`), and
- **emit a Slice-5 variance proxy**: per class, the single-arm naive-RAG strict
  recall@10 outcome variance `p(1-p)` over the gold, plus the implied two-sample N
  for MDE≤ε (`N ≈ (1.96·√(2p(1-p)) / ε)²`). A coarse unpaired upper-bound proxy
  (pairing will shrink it) for Slice-10's real paired power-check.

**HITL-escalation flagged (reserved follow-on 6–9):** the M1 note "base-retrieval
MDE ≈ 21pp at n=160" implies MDE≤0.05 needs N in the high-hundreds-to-thousands —
**above** any feasible memory-class gold size (LME caps at ~130/class real). So the
provisional N=150 makes the classes non-empty and comparable but is **expected to
remain under-powered for a strict MDE≤ε paired claim**. Slice 10 must run the real
paired power-check and, if `mde > ε`, **escalate to HITL** rather than claim parity
(decision_rule_083 already encodes the `underpowered:<class>` guard). Surfaced now so
it is not a Slice-10 surprise.

## (C) Identical-answerer seam wiring

The seam already exists (`LLMAnswerer`, `R2_RUN` + `R2_ANSWERER_*`). Two small,
reviewable additions make it runnable end-to-end against the **re-pinned LME gold**
(non-`fe973fcd` corpus_hash, so it cannot go through the COR-2-pinned
`R2Harness(gold_path=…)` constructor):

- `eval.r2_parity_eval.load_repin_gold(path) -> (corpus_hash, queries)` — loads a
  re-pinned gold file and parses its queries.
- `R2Harness.from_repin_gold(path, answerer)` — builds a harness over the re-pinned
  gold via `from_queries` (the existing LME-mode constructor that bypasses the COR-2
  pin by design).

**Cheap-validate (this slice):** `gemini-flash-lite` (the available cheap model; the
prompt's `gemini-2.5-flash-lite` is superseded on the airlock), ≤2 questions, to
prove the wiring. The full priced parity pass is **Slice 10** and waits on the HITL
Slice-0 gate.

## (D) fix-1 (codex §9 review — 2×[P1] + 1×[P2])

**[P1#1] class-definition predicate (≥2 sessions).** `augment_class()` previously
accepted any query with ≥1 evidence doc, admitting single-session `multi_session` /
`knowledge_update` gold (codex confirmed `d0a-0441`, `d0a-0235`). Fixed by a single
predicate, `corpus_validity.class_predicate_ok()`: `multi_session` /
`knowledge_update` require evidence resolving (via `session_id_of`) to **≥2 distinct
sessions**; `factoid` / `temporal` keep the ≥1-evidence rule. Enforced in three
places — inside `augment_class()` (rejects non-conforming generations), a build-time
**filter pass** in `build_repin()`, and the **runtime** `validate_repin()` guard. The
`knowledge_update` targeted prompt now requires ≥2 doc_ids (earlier + superseding
session). The gold was **regenerated**: every span-class query now resolves to ≥2
sessions (zero violations); all four classes still reach `n_min=150`
(`factoid=156, knowledge_update=150, multi_session=150, temporal=150`) — **no class
hit the HITL under-fill escalation**.

**Re-pin identity — loud justified deviation.** The review asked for a "new
`corpus_hash`". The LME corpus (`documents`) is deterministic and **did not change**
(only the gold/qrels did), so an honest `corpus_hash = sha256(corpus)` is *invariant*
(`1859817a…`); faking a new value would misrepresent the corpus. The fix-1 gold
change is instead captured by a new **`qrels_hash`** (sha256 over the emitted gold)
and **`repin_hash` = sha256(corpus_hash ‖ qrels_hash)** (now `2916cace…`), with
`qrels_version = repin-{repin_hash[:8]}`. `repin_hash` is the changing identity
downstream keys off.

**[P1#2] per-run Mem0 isolation.** `build_local_mem0_config()` no longer points every
run at the shared `/tmp/mem0_chroma` `r2_eval` collection with a fixed `user_id`.
Both the Chroma **collection name** and on-disk **path** are now keyed by
`corpus_hash` + a per-run `run_id` (minted fresh when omitted), and `user_id` is
derived per-run (`run_user_id(run_id)`). A re-run cannot reopen a prior run's
collection / read stale memories — reproducible by construction, no manual wipe.

**[P2] runtime corpus-validity guard.** `load_repin_gold()` (and therefore
`R2Harness.from_repin_gold()`) now calls `validate_repin()` over the gold (per-class
counts derived from the file, `n_min` from the file) and **raises `ValueError`** on
any problem — an underfilled / N=0 / missing / single-session-mislabeled class can no
longer silently run to emit null (the Slice-25 N=0 mode).

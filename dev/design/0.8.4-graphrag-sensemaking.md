## 0. HITL sign-off (Slice-0 gate, 2026-06-23)

The HITL signed the Slice-0 gate with these terms (`honest_prior_cleared` = True,
`budget_approved` = True):

+ **Honest-prior FUND/NO-FUND — CLEARED, but PILOT-FIRST.** S1 is funded as a
  *staged* bet, not a blank check: the Slice-5 corpus + baselines + AutoQ + a small-N
  **long-context pilot** run first. If `strong_baseline_clears(s1_vs_long_context)` is
  **False** at the pilot (the long-context "stuff-it-all-in" control already ≈ S1),
  the program **settles the graph question as a publishable third null BEFORE funding
  the Leiden/community build (Slice 10+)** — record the decision, do not spend the big
  build budget. This is the §6 kill-early predicate made the funding hinge.
+ **Budget — TOP-UP to a powered run APPROVED, amount set post-pilot.** The HITL
  authorized raising the cap so AutoQ/AutoE can run at the question count needed for
  the win-rate **MDE ≤ 0.05** power bar (§5). The **exact number is fixed after the
  Slice-5 pilot reports the measured per-call cost + observed judge variance** — the
  pilot spend itself stays small (cheap-validate + a bounded pilot), then the powered
  AutoE spend is approved against a real number, not the §7 order-of-magnitude guess.

**Operational consequence:** Slice 5 builds the $0 infra + runs the bounded pilot →
the orchestrator returns to the HITL with (a) the `strong_baseline_clears` verdict
(fund the big build or settle the null) and (b) the measured powered-run cost (the
exact top-up). Slices 10/15/20 stay gated behind that return.

## 1. Purpose — the resolution, not a lever bake-off

0.8.4 succeeds iff **FathomDB reaches near-parity-or-better with Microsoft GraphRAG
on the global-sensemaking axis**, under a BenchmarkQED-style **pairwise LLM-judge
win-rate** with the four bias controls applied, against a strong **vector-RAG +
long-context** baseline. The resolution is delivered by two paired workstreams:

+ **S1** — FathomDB's own GraphRAG-style build: entities/relationships → **Leiden
  community detection** → **LLM-generated community summaries** (hierarchical C0–C3)
  → **map-reduce query-focused summarization (QFS)** reader.
+ **G-HH-2** — a **measured head-to-head** of S1 vs a *running* Microsoft GraphRAG
  on the same corpus + judge (primary), with **HippoRAG-2** on the M1/D1 MuSiQue
  multi-hop harness as a secondary cross-check (it bounds the competitor; FathomDB's
  own graph multi-hop is already refuted — M1 NO-GO, [[m1-graph-arm-nogo-registered-n300]]).

The levers (Leiden, community summaries, QFS) are **means to close a measured
competitor gap**, not ends. A win-rate from a literature row is not a parity claim:
the capability report's competitor rows are not apples-to-apples, so GraphRAG must be
stood up as a *running* system on the *same* harness (one judge / one corpus / one
metric).

## 2. Honest expectation — the design-review FUND/NO-FUND gate (load-bearing)

**This section is the gate that funds the largest build of the program.** It is not
a formality. The cross-graph prior across 0.8.1+0.8.2 is **strongly negative**: the
graph multi-hop track closed **NO-GO** on both recall (0.8.1 n=40) and answer-F1
(0.8.2 M1 n=300), and M2 was dropped. So S1 must clear a high bar *before* its build
is funded.

**Why S1 is nonetheless a genuinely open bet (not refuted by the prior).** S1 is a
*different graph structure* (community summaries, not fact-edge traversal) on a
*different axis* (global sensemaking — "what are the themes?" — not needle/multi-hop
retrieval). The refuted arms (PPR/BFS over fact-edges) tested the left column of the
roadmap §0 table; S1 tests the right column — the structure and task GraphRAG ("From
Local to Global", arXiv:2404.16130) was actually built for, where its reported
sensemaking win-rates over vector RAG are large (~72–83% comprehensiveness). The
multi-hop negatives therefore do **not** refute S1.

**But be honest about the skeptical prior to beat:** the Samsung result ("VectorRAG
is almost enough", arXiv:2603.29875) says a strong vector-RAG + long-context baseline
may already match GraphRAG on a corpus that fits the window. A **third graph null** —
GraphRAG-beats-FathomDB, or FathomDB≈vector-RAG≈GraphRAG — is an acceptable,
**publishable** outcome that settles the graph question for FathomDB.

**The funding checklist (frozen as `honest_prior_cleared` in the rule). S1 funds only
if ALL hold** — codex + HITL judge each at the Slice-0 gate:

1. `distinct_from_multihop_prior` — S1 is community-summary *synthesis*, a different
   mechanism + axis than the refuted PPR/BFS *traversal*. (Argued above.)
2. `strong_baseline_planned` — S1 is judged against **vector-RAG AND a long-context
   "stuff-it-all-in" control**, not a strawman. (Slice 5.)
3. `kill_early_criteria_frozen` — the Slice-5 pilot escalation
   (`strong_baseline_clears`) and the Slice-15 baseline check are frozen *before*
   data. (This doc §6.)
4. `budget_approved` — the aggregate judged-run budget is HITL-approved **before**
   spend, with the top-up reality (§7) acknowledged.

**Kill-early (any of these ⇒ S1 does NOT build, record the decision, settle the graph
question):** (a) codex/HITL judge the S1 hypothesis insufficiently distinguished from
the negative prior at this gate; (b) the Slice-5 pilot shows the **long-context
control already ≈ projected S1** (`strong_baseline_clears` is False — §6); (c) the
budget cannot be funded to a powered N (§7).

## 3. Method, datasets, arms (BenchmarkQED-aligned)

+ **Corpus:** the BenchmarkQED **AP-News set (~1,397 articles)** — *news* is also
  0.8.3's MultiHop-RAG domain, so the regimes share a corpus family. **EVAL-ONLY,
  gitignored, NEVER committed** ([[0.8.3-0.8.4-corpus-adequacy-and-locomo]]:
  LOCOMO CC-BY-NC + AP-News MS-Research-no-redistribute — payloads never land in git).
+ **S1 build (OFFLINE-BUILD seam):** entities/relationships → Leiden communities →
  LLM community summaries (C0–C3; the levels literally measure the hierarchy's value)
  → map-reduce QFS. The index-time extractor is **local Qwen3.6-27B ($0/local)**,
  to be **measured for sufficiency** (§6). This is the one initiative that strains
  the CPU/no-API footprint invariant — it stays the offline build seam, never the
  library query boundary.
+ **Arms (one answerer / one prompt / one context budget — the parity invariant):**
  `s1` (FathomDB community-summary QFS) · `graphrag` (running Microsoft GraphRAG) ·
  `vector_rag` · `long_context` (stuff-it-all-in control). Adapters expose only the
  shared `retrieve`/answer seam reused from M1/0.8.3.
+ **Eval harness — BenchmarkQED:** **AutoQ** (persona→task→question synthesis
  spanning local↔global, incl. multi-hop-local) and **AutoE** (pairwise LLM-judge
  win-rates on comprehensiveness/diversity/empowerment + the directness corroboration).
+ **Judge + readers:** the judge is the priced **EVAL-ONLY** seam, **cross-family**
  (different model family than ANY arm's answerer — §5 bias controls). Cheap-validate
  with `gemini-2.5-flash-lite` before any priced run; the priced judge + answerers are
  named at Slice 5 (HITL-approved at the spend gate), pinned temp 0 / seed 0 where the
  provider allows.

## 4. Endpoints

+ **Primary GATE (resolution):** the **S1-vs-GraphRAG pairwise win-rate** per headline
  metric `{comprehensiveness, diversity, empowerment}`, ties counted as 0.5 (so exact
  parity = 0.50), aggregated across **≥5 order-swapped runs**, bootstrapped over
  questions, with a paired 95% CI. *Every* headline metric must clear the near-parity
  band (the conservative bar; a metric-averaged win-rate would hide a weak dimension).
+ **Strong-baseline context (reported + kill-early):** the S1-vs-`vector_rag` and
  **S1-vs-`long_context`** win-rates. The long-context comparison drives the Slice-5
  `strong_baseline_clears` kill-early predicate (§6).
+ **Length-bias corroboration:** the **directness / claim-count** non-judge control
  (the GraphRAG paper's control) — must be **run**, and must **not contradict** (a win
  fully explained by verbosity is not a sensemaking win).
+ **Secondary cross-check:** HippoRAG-2 vs FathomDB on the M1/D1 MuSiQue multi-hop
  harness (≥3-hop F1 / all-bridges@K) — bounds the competitor only; reported, not part
  of the GraphRAG-gate `decide_084`.

## 5. Pre-registration — frozen fields (dated 2026-06-23)

> Frozen as code in `src/python/eval/decision_rule_084.py`; the Slice-0 test lints
> that this doc carries every field below. **Slices 5/10/15/20 import the rule; they
> may not redefine it.**

+ **frozen-field: near-parity-band** (2026-06-23) — exact parity on the pairwise
  win-rate scale is **0.50** (ties = 0.5). "Near-parity-or-better" on a metric = the
  S1-vs-GraphRAG paired win-rate **CI lower bound ≥ 0.50 − ε_wr** with **ε_wr = 0.05**
  (i.e. ≥ 0.45); "better" (surpass candidate) = lower bound **> 0.50**. Constant
  `EPS_WIN_RATE = 0.05`.
+ **frozen-field: power-sizing rule** (2026-06-23) — the per-comparison win-rate
  **MDE (CI half-width) must be ≤ ε_wr = 0.05** on every headline metric, else that
  metric is `underpowered` and **no parity claim is made on it** (escalate N — the
  reserved follow-on). The bootstrap clusters by **question** (≥5 runs × order-swap
  are within-question replicates, not independent N). The AutoQ question count is sized
  at Slice 5 so the clustered MDE clears 0.05 (target ≳100 questions spanning
  local↔global; the **Slice-5 pilot re-estimates** the required N from the *observed*
  judge variance — a point-proportion N≈384 near 0.5 is the floor *before* clustering
  inflation, so the real target is larger and pilot-calibrated).
+ **frozen-field: decision-rule** (2026-06-23) — `decide_084(primary_per_metric,
  bias_controls, length_corroboration)` returns **REACHED** iff, in order: (1) the four
  bias controls are satisfied (else BLOCK), (2) no headline metric is under-powered,
  (3) the length corroboration does not contradict, and (4) **every** headline metric
  clears the near-parity band (`ci_lo ≥ 0.45`). Otherwise **NOT_REACHED** with the
  `binding_constraint` / `blocked_by` naming the failure. `surpass_candidates` = metrics
  with `ci_lo > 0.50`.
+ **frozen-field: bias-controls** (2026-06-23) — the four LLM-judge failure modes are
  **hard preconditions** (a win-rate without them is not a measurement): **position**
  (answer order swapped + averaged, arXiv:2406.07791) · **stochasticity** (≥5 runs,
  `MIN_RUNS = 5`, report variance not a point) · **self-preference** (judge from a
  **different model family** than ANY system under test, arXiv:2410.21819) · **length**
  (the directness/claim-count corroboration is run and does not contradict). Encoded as
  `_check_bias_controls`; a violation BLOCKs with `blocked_by ∈ {bias:position,
  bias:stochasticity, bias:self_preference, bias:length_missing}`.
+ **frozen-field: surpass-option protocol** (2026-06-23) — **parity is the floor, not
  the ceiling.** If `decide_084` returns REACHED and/or `surpass_candidates` is
  non-empty (or an S1 variant — richer C-hierarchy / hybrid summary+retrieval reader —
  is *projected* to overshoot GraphRAG), the orchestrator presents the **surpass
  experiment as an explicit option BEFORE the HITL signs the gate** — "ship at parity"
  vs "spend to surpass," never a silent stop (mirrors the 0.8.3 principle).
+ **frozen-field: honest-prior gate** (2026-06-23) — the §2 funding checklist
  `HONEST_PRIOR_CRITERIA = {distinct_from_multihop_prior, strong_baseline_planned,
  kill_early_criteria_frozen, budget_approved}`; `honest_prior_cleared` requires ALL.
  The cross-graph prior is strongly negative, so a clean codex + HITL PASS here is the
  load-bearing gate that funds the largest build of the program — not a rubber stamp.
  A **third graph null is a valid, publishable outcome.**

## 6. Kill-early predicates (frozen, the cheap-validate-before-spend discipline)

+ **Slice-5 pilot — `strong_baseline_clears(s1_vs_long_context)`:** after the small-N
  pilot judge run, if S1 does **not** clear near-parity against the long-context
  control (`ci_lo < 0.45`), the community build is likely not worth funding (the corpus
  fits the window — the Samsung prior wins). **ESCALATE to the HITL before the full
  build is funded.** This is a $-small pilot gating the $-large build.
+ **Slice-15 baseline check:** if, with the QFS reader stood up, vector-RAG/long-context
  already matches S1 on a bounded check, escalate before the full ≥5-run AutoE spend.

> UNCHANGED).** S1 is built in Slice 10, *after* the Slice-5 pilot, so the literal
> `strong_baseline_clears(s1_vs_long_context)` cannot be evaluated at Slice 5 (S1 does

>

>   AutoE harness end-to-end on a small N over `vector_rag` vs `long_context` (a real

>   predicate intact while putting the cheap check before the expensive build.

## 7. Budget — honest, and a TOP-UP is almost certainly required (HITL decision)

0.8.4 is the **most expensive version of the program**: it pays for (a) the running
**GraphRAG competitor index** (GraphRAG's own indexing is LLM-heavy), (b) **AutoQ**
question synthesis, and (c) **AutoE** = pairwise judgments × questions × **≥5 runs** ×
**order-swap (×2)** × **3 comparisons** (S1-vs-{GraphRAG, vector-RAG, long-context}) ×
**3 metrics** + the directness corroboration. The S1 community-summary build runs on
**local Qwen ($0)**; the judge + competitor LLMs are the priced seam.

+ **Batch-by-default ([[airlock-batch-and-provider-protection]]):** AutoQ + AutoE are
  large sets of *independent* judgments → submit via the **OpenAI/airlock Batch
  interface** (~50% cheaper; **bypasses the interactive TPM quarantine that blocked
  0.8.3 completion**, per `dev/design/0.8.3-openai-batch-completion-howto.md`). The
  batch discount is assumed in the estimate.
+ **Order-of-magnitude (to be tightened at Slice 5 from the pilot's per-call cost):**
  even batched, AutoE at ≳100 questions × 5 runs × 2 orders × 3 comparisons × 3 metrics
  is **tens of thousands of judge calls**; with the GraphRAG index build on top, the
  realistic 0.8.4 spend is **plausibly $30–60+**, dwarfing the **~$11.84 remaining of
  the $50 program cap** (0.8.3 spent ~$38.16). **A budget top-up is a Slice-0 HITL
  decision** — without it, only a reduced-scope pilot (fewer questions / one comparison)
  is fundable, which would likely be *under-powered* (violating the §5 power rule). The
  Slice-5 pilot produces the exact per-call cost so the HITL approves a real number.
+ **Resilient priced runs ([[priced-runs-need-resilience-before-spend]]):** auto-resume,
  atomic checkpoint, 429/5xx backoff honoring `Retry-After`, empty/None completion →
  ABSENT (never `acc=0.0`), completeness validity guard. Batch's idempotent
  missing-row resume covers the batched judge calls. A running **$ ledger** in
  `dev/plans/runs/STATUS-0.8.4.md`.

## 9. This module's contract (`decision_rule_084.py`)

+ **Pure / deterministic / pure-stdlib** (no `fathomdb`/`scipy`/`numpy`/`networkx`),
  so the rule + its test run independent of the native build / `.venv` binding.
  Consumes **already-computed summary statistics** (the AutoE harness runs the
  bootstrap); no RNG, no I/O, no clock.
+ **Fail-loud:** any non-finite endpoint float raises `ValueError`; a missing
  metric/control/key raises `KeyError` — a malformed endpoint never silently returns a
  verdict. The endpoint echo runs **before** the block check so a bad number raises
  even when a control is also absent.
+ **Public surface:** `decide_084`, `strong_baseline_clears`, `honest_prior_cleared`,
  `lint_preregistration_084`, and the frozen constants
  (`EPS_WIN_RATE`, `PARITY`, `MIN_RUNS`, `HEADLINE_METRICS`, `HONEST_PRIOR_CRITERIA`,
  `REQUIRED_FROZEN_FIELDS_084`).

## 10. What 0.8.4 deliberately does NOT do

+ **No in-library LLM.** Community summarization is an OFFLINE build (local Qwen); the
  judge is EVAL-ONLY. The library query boundary stays CPU-only / no-API / 1-bit-safe.
+ **No multi-hop *graph* revival.** HippoRAG-2 is a competitor *cross-check* bound, not
  a FathomDB graph arm — FathomDB's own graph multi-hop is refuted (M1/M2).
+ **No reranking work here.** Per the 0.8.3 verdict §6, reranking is **recall-ceilinged**
  — the α/pool_n re-tune is a separate, recall-gated carry-forward
  (`dev/roadmap/0.8.4.md` §6), not part of the S1 ladder.
+ **No interactive LLM calls for batchable work** — AutoQ/AutoE go via batch.

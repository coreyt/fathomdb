# 0.8.8 Slice 0 ADR — retrieval legibility schemas

> **Status: RATIFIED 2026-06-27.** The Part A `Explanation` field set was ratified via a 6-owner
> cross-component negotiation — see `dev/plans/runs/0.8.8-explanation-fieldset-ratification.md` (the
> authoritative hand-off spec; `dev/plans/runs/0.8.8-carrier-decision-hitl-brief.md` is the
> pre-negotiation brief). The landed Slice 5 engine keystone diverged from this doc's *original* §A.3
> proposal; the negotiation ruled the **landed shape architecturally correct** and amended this ADR
> to match the code (the only added engine delta was `#[non_exhaustive]`). Part B (telemetry/gold)
> carries into Slice 15/20 with the §B amendments below folded in.

This ADR freezes two schemas for 0.8.8:

1. **`Explanation`** — the opt-in `search(..., explain=True) → Explanation` payload (EXP-OBS, #1).
2. **Telemetry event + real-gold record** — the local capture channel and the labeled gold row
   it produces (OPP-9, #10).

It is grounded in an existing-seam recon (file:line below); the design mandate (R-OBS-3) is to
**generalize existing machinery, not invent parallel systems** — codex §9 will check this.

## Part A — `Explanation` payload (R-OBS-1..4)

### A.1 Existing seams this reuses (recon)

| Seam | Location | Reuse |
|------|----------|-------|
| `SearchHit` (carries `score`, `ce_score`, `branch: SoftFallbackBranch`, `source_id`) | `fathomdb-engine/src/lib.rs:1064` | per-hit identity + arm + blended score already present |
| `fuse_three_arms` (RRF; per-arm `rank0` lives in the loop but is **not** carried out) | `fathomdb-engine/src/lib.rs:4774` | the per-arm rank + raw RRF contribution is computed here — capture it under explain |
| `ce_rerank` (computes `rrf_norm` + `ce_norm`, blends `alpha*ce_norm+(1-alpha)*rrf_norm`) | `fathomdb-engine/src/lib.rs:4983` | both score components exist at line 5030/5032 — retain them under explain instead of re-deriving |
| `search_reranked` (entry; already takes `Instant::now()` at 2906; owns `alpha`,`pool_n`,`rerank_depth`,`use_graph_arm`) | `fathomdb-engine/src/lib.rs:2896` | query-level trace assembled here; timings from the existing `Instant` |
| `CounterSnapshot` / profiling | `lib.rs:1479`, `lifecycle.rs:214` | global counters (not per-query) — query timings come from inline `Instant` deltas, not these |
| `explain_graph_neighbors_for_test` (ReaderRequest dispatch pattern) | `lib.rs:3241` | dispatch pattern reused if graph-arm provenance needs the reader pool; no new dispatch type if explain rides the existing search result |
| `embedder identity` (`EmbedderIdentity{name,revision,dim}`) | surfaced via `OpenReport` | query trace records the active embedder identity |

### A.2 Decision: carrier = **sidecar `Explanation` on `SearchResult`** — ✅ HITL-RATIFIED 2026-06-27

`SearchResult` gains `explanation: Option<Explanation>` — `None` whenever `explain=False`.
`SearchHit` stays **byte-identical** across all three bindings. This is the clean way to satisfy
**R-OBS-2 (zero-cost when off)**: no per-hit struct growth, no allocation on the hot path, the
zero-cost bench just proves the `None` path equals baseline.

**HITL ratification (2026-06-27):** the *sidecar* shape is chosen over inline-on-`SearchHit`
explicitly **because it keeps the non-explanation response shape byte-identical to the current
`SearchResult`**. Confirmed contract: when `explain=False` (default), explanations are **fully
suppressed** — `explanation` is `None`, no allocation, and `results` is unchanged.

*Rejected alternative (inline):* embedding `Option<arm_rank/rrf_norm/...>` fields on every
`SearchHit` changes the result contract on all 3 SDKs even when off and complicates the zero-cost
proof.

### A.3 Ratified shape (was Proposed)

Content identical to the landed `fathomdb-engine/src/lib.rs` Slice-5 keystone. The only delta the
ratification added over the landed code is `#[non_exhaustive]` on the four structs (additive-safety
for the deferred `timings_ms` and any future field). The original proposed pseudo-shape diverged on
several points; §1 of the ratification spec records each ruling (raw `fused_score` not `rrf_norm`;
no `timings_ms`; no `excluded_by_filter`; 3 public sub-types; 3 arms folding `text_edge`;
`ce_active` not `ce_model_id`; `recency` not `mmr`) — all resolved **in favor of the landed code**.

```rust
#[derive(Clone, Debug, PartialEq)]
#[non_exhaustive]
pub struct SearchResult {
    pub projection_cursor: u64,
    pub soft_fallback: Option<SoftFallback>,
    pub results: Vec<SearchHit>,           // byte-identical; SearchHit UNTOUCHED
    pub explanation: Option<Explanation>,  // None on every explain=false path (zero-cost)
}

#[derive(Clone, Debug, PartialEq)]
#[non_exhaustive]
pub struct Explanation {
    pub trace: QueryTrace,
    pub per_hit: Vec<PerHitExplain>,       // parallel to SearchResult.results, same order
}

#[derive(Clone, Debug, PartialEq)]
#[non_exhaustive]                          // leaf — absorbs deferred timings_ms (Q3) without a break
pub struct QueryTrace {
    pub query_chars: u32,                  // length only — NEVER the query text (§C)
    pub k: u32,
    pub rerank_depth: u32,
    pub pool_n: u32,
    pub alpha: f64,
    pub use_graph_arm: bool,
    pub recency: bool,
    pub embedder_id: String,               // "name@rev (dim=N)"; "" sentinel, not Option
    pub ce_active: bool,
    pub vector_hits: u32,
    pub text_hits: u32,                    // folds text_edge
    pub graph_hits: u32,
}

#[derive(Clone, Debug, PartialEq)]
#[non_exhaustive]                          // leaf — absorbs new arms / score components
pub struct PerHitExplain {
    pub id: u64,                           // == SearchHit.id; stable logical_id; NO source_id (leak vector)
    pub arm: SoftFallbackBranch,           // == SearchHit.branch
    pub vector_rank: Option<u32>,
    pub text_rank: Option<u32>,
    pub graph_rank: Option<u32>,
    pub fused_score: f64,                  // RAW post-recency, pre-CE-blend — NOT min-max normalized
    pub ce_score: Option<f64>,             // == SearchHit.ce_score; sigmoid(ce_logit) ∈ [0,1]
    pub blended: f64,                      // == SearchHit.score
}
```

**Field-fidelity acceptance (R-OBS-1):** for a known hybrid query with `rerank_depth>0`,
`per_hit[i].id` == `results[i].id`, `per_hit[i].arm` == `results[i].branch`, `per_hit[i].ce_score` ==
`results[i].ce_score`, `per_hit[i].blended` == `results[i].score`; `ce_score ∈ [0,1]` when present;
`fused_score` is the RAW RRF value (not normalized to [0,1]); the config knobs (`alpha`, `pool_n`,
`rerank_depth`, `ce_active`) match the request. Pinned by `tests/exp_obs_explain.rs`
(`r_obs_1_golden_field_fidelity_at_rerank_depth_gt0`).

**Parity (R-OBS-4):** identical payload (snake_case Py / camelCase TS) exercised by the X1 harness.

### A.5 Implementation approach (engine threading) — informs zero-cost + feasibility

`search` does not run inline; it dispatches `ReaderRequest::Search` to the reader worker pool and
receives a `ReaderResponse` tuple `(cursor, soft_fallback, results, graph_stats)` (see
`search_inner_with_stats` `lib.rs:2995`; `read_search_in_tx` `lib.rs:5348`). So explain rides the
**existing side-channel precedent**: `GraphFrontierStats` is *already* threaded through that tuple as
an opt-in meter that never reaches the governed surface. EXP-OBS mirrors it exactly:

1. Add `explain: bool` to `ReaderRequest::Search` (default-constructed `false` on every current caller).
2. Inside `read_search_in_tx`, when `explain`: capture per-arm `rank0` in `fuse_three_arms`
   (`lib.rs:4774`, already in-loop) and `rrf_norm`/`ce_norm` in `ce_rerank` (`lib.rs:4983`, already
   computed at 5030/5032) into an `Option<Explanation>`; when `!explain`, build `None` — **no
   allocation, no extra work** (the zero-cost proof, R-OBS-2).
3. Extend the `ReaderResponse` tuple with `Option<Explanation>` (the `GraphFrontierStats` slot is the
   template).
4. `SearchResult` gains `explanation: Option<Explanation>` (A.2); `search_inner_with_stats` plumbs it.
5. New governed entry `search_reranked_explain(...)` (or an `explain` param on `search_reranked`) — the
   bare `search()`/`search_filtered()` paths keep passing `explain=false` and stay byte-identical.

**Blast radius:** engine-crate only for Slice 5 (the reader-pool protocol types + fusion/rerank +
`SearchResult`). SDK wiring (pyo3/napi/Py/TS) is **Slice 10**, deliberately separated so a schema
amendment after ratification reshapes one crate, not three bindings.

**Byte-stability guard:** every existing (`explain=false`) path must produce a byte-identical
`SearchResult.results` + cursor; a golden test pins the default path unchanged.

### A.4 Ratified answers (was open questions) — RATIFIED 2026-06-27

- **Q1 (raw vs normalized arm scores): RAW.** `PerHitExplain.fused_score` is the raw post-recency,
  pre-CE RRF score. Experiments (`rerank_tune_probe.py:105`, `:200`) apply their own min-max; a
  pre-normalized value would force inversion. Per-arm raw scores are **not** carried (no reader);
  `*_rank` + the single `fused_score` cover every consumed value.
- **Q2 (filter-exclusion reporting): RETURNED-SET ONLY.** No `excluded_by_filter`. Filtered eval
  runs score recall over the returned set (`r2_parity_eval.py:347`); no harness inspects excluded hits.
- **Q3 (timing granularity): NONE shipped, deferred-safe.** No `timings_ms` in the ratified shape.
  No harness reads engine timings (whole-run wall-clock is measured caller-side). Timings remain a
  **known future addition** — `QueryTrace`'s `#[non_exhaustive]` makes adding `timings_ms` later a
  non-breaking, additive change rather than a contract edit.

## Part B — Telemetry event + real-gold record (R-TEL-1..3)

### B.1 Telemetry event (opt-in, local, deterministic, no egress)

```text
TelemetryEvent {
  schema_version: u32,
  ts_monotonic_ms: u64,        // monotonic since engine open — NOT wall-clock (determinism + privacy)
  query_id: String,            // opaque ULID-like; derived deterministically from (open_nonce, seq)
  query_chars: u32,            // length only, never the text
  result_ids: [u64],           // returned hit ids, in rank order
  arm_of: { <id>: SoftFallbackBranch },
  feedback: Option<{           // exogenous agent-supplied relevance label (the ONLY external signal)
    relevant_ids: [u64],
    irrelevant_ids: [u64],
    label_source: String,      // caller-declared, e.g. "agent:hermes"
  }>
}
```

- **Off by default.** Enabled via an explicit opt-in (engine open flag / env). When off: no record,
  no file handle, no allocation (footprint gate at Slice 15).
- **No network.** Events append to a local sink (caller-provided path); a footprint test asserts no
  socket/egress.
- **Deterministic.** `ts_monotonic_ms` + deterministic `query_id` so fixtures are reproducible.

### B.2 Real-gold record (eval-side; capture in-library, curation EVAL-ONLY)

```text
GoldRecord {
  schema_version: u32,
  query_id: String,                       // ADDED (ratification) — join key to TelemetryEvent.query_id
  id_space: "engine-logical-id",          // ADDED (ratification) — machine-checkable namespace tag
  query_chars: u32,
  candidate_ids: [u64],                   // stable logical_id (== SearchHit.id); NEVER write_cursor
  labels: { <id>: relevance ∈ {0,1} },   // from TelemetryEvent.feedback
  embedder_id: String,
  provenance: "telemetry-capture",
}
```

A fixture telemetry run → one valid `GoldRecord` consumable by the eval harness (R-TEL-2).
This is FathomDB's exit from synthetic-gold purgatory: real query→result→agent-feedback rows.

**Id contract (both schemas) — RATIFIED.** Every `u64` id (`TelemetryEvent.result_ids`/`arm_of`/
`feedback.*_ids`, `GoldRecord.candidate_ids`/`labels`) is the **stable `logical_id`** — identical to
`SearchHit.id` and `PerHitExplain.id`. NEVER the interim `write_cursor`; NEVER an external corpus
string id. **`source_id` (free-form `Option<String>`, may hold a path/URI/session id) is a leak
vector — never captured into telemetry/gold and never added to `PerHitExplain`** (convergent
telemetry + privacy ruling).

### B.3 Ratified answer (was open question) — RATIFIED 2026-06-27

- **Q4 (gold-row shape to mirror): align to the ROW SHAPE of `GoldQuery` (`r2_parity_eval.py:118-124`),
  NOT its id values.** Today's gold is string-keyed over an *external* corpus the engine never sees;
  telemetry gold lives in the *engine's own id namespace* — two corpora in two namespaces, so forcing
  one into the other is a category error. The harness `str(u64)` fallback (`:318`, `:666`) bridges to
  the row shape with **no shim**.
- **Consumer reality (B.3):** a `GoldRecord` cannot drive the live re-run harness (it re-runs from
  query *text*, forbidden by §C; and `GoldQuery` has no slot for 0/1 negatives). Its consumer is a
  **new offline frozen-candidate scorer** (score captured `result_ids` against `labels` over the
  frozen pool) — scope into Slice 20.

## Part C — Privacy / footprint contract (R-TEL-3)

- **No content leaves the box.** Neither `Explanation` nor `TelemetryEvent` carries query text or
  document bodies — only ids, lengths, scores, ranks, and caller-supplied labels.
- **Agent-supplied relevance labels are the only exogenous signal.** Everything else is derived
  from the local search.
- **Telemetry is off by default and never egresses.** Documented payload; footprint test enforces.
- **`Explanation` is opt-in and zero-cost when off** (R-OBS-2 hard gate).

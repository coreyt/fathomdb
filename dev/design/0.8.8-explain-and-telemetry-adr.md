# 0.8.8 Slice 0 ADR — retrieval legibility schemas

> **Status: DRAFT — not ratified.** Plan §8/§9 require the `Explanation` payload shape and the
> gold-record schema to be ratified with the active M-experiment owner *before* Slice 5 code,
> because "the explain fields and gold schema must match what the experiments consume." This doc
> is the concrete proposal to ratify or amend. **No engine/SDK code lands until this is signed.**

This ADR freezes two schemas for 0.8.8:

1. **`Explanation`** — the opt-in `search(..., explain=True) → Explanation` payload (EXP-OBS, #1).
2. **Telemetry event + real-gold record** — the local capture channel and the labeled gold row
   it produces (OPP-9, #10).

It is grounded in an existing-seam recon (file:line below); the design mandate (R-OBS-3) is to
**generalize existing machinery, not invent parallel systems** — codex §9 will check this.

## Part A — `Explanation` payload (R-OBS-1..4)

### A.1 Existing seams this reuses (recon)

| Seam | Location | Reuse |
|------|----------|-------|
| `SearchHit` (carries `score`, `ce_score`, `branch: SoftFallbackBranch`, `source_id`) | `fathomdb-engine/src/lib.rs:1064` | per-hit identity + arm + blended score already present |
| `fuse_three_arms` (RRF; per-arm `rank0` lives in the loop but is **not** carried out) | `fathomdb-engine/src/lib.rs:4774` | the per-arm rank + raw RRF contribution is computed here — capture it under explain |
| `ce_rerank` (computes `rrf_norm` + `ce_norm`, blends `alpha*ce_norm+(1-alpha)*rrf_norm`) | `fathomdb-engine/src/lib.rs:4983` | both score components exist at line 5030/5032 — retain them under explain instead of re-deriving |
| `search_reranked` (entry; already takes `Instant::now()` at 2906; owns `alpha`,`pool_n`,`rerank_depth`,`use_graph_arm`) | `fathomdb-engine/src/lib.rs:2896` | query-level trace assembled here; timings from the existing `Instant` |
| `CounterSnapshot` / profiling | `lib.rs:1479`, `lifecycle.rs:214` | global counters (not per-query) — query timings come from inline `Instant` deltas, not these |
| `explain_graph_neighbors_for_test` (ReaderRequest dispatch pattern) | `lib.rs:3241` | dispatch pattern reused if graph-arm provenance needs the reader pool; no new dispatch type if explain rides the existing search result |
| `embedder identity` (`EmbedderIdentity{name,revision,dim}`) | surfaced via `OpenReport` | query trace records the active embedder identity |

### A.2 Decision: carrier = **sibling `Explanation` on `SearchResult`** (RECOMMENDED — pending ratify)

`SearchResult` gains `explanation: Option<Explanation>` — `None` whenever `explain=False`.
`SearchHit` stays **byte-identical** across all three bindings. This is the clean way to satisfy
**R-OBS-2 (zero-cost when off)**: no per-hit struct growth, no allocation on the hot path, the
zero-cost bench just proves the `None` path equals baseline.

*Rejected alternative:* embedding `Option<arm_rank/rrf_norm/...>` fields on every `SearchHit`
changes the result contract on all 3 SDKs even when off and complicates the zero-cost proof.

### A.3 Proposed shape

```
Explanation {
  // ---- query-level trace (one per search) ----
  trace: {
    query_chars: u32,            // length only — NOT the query text (privacy; see C)
    k: u32,                      // final result limit
    rerank_depth: u32,
    pool_n: u32,                 // CE pool size actually used
    alpha: f64,                  // CE blend weight
    use_graph_arm: bool,
    mmr: bool,                   // recency/MMR reweight applied
    embedder_id: String,         // "name@revision" + dim
    ce_model_id: Option<String>, // None when default-reranker off / rerank_depth==0
    arm_hit_counts: { vector: u32, text: u32, text_edge: u32, graph: u32 },
    timings_ms: {                // inline Instant deltas; coarse, monotonic
      vector_arm, text_arm, graph_arm, fuse, rerank, total
    }
  },
  // ---- per-hit breakdown (parallel to SearchResult.hits, same order) ----
  per_hit: [ {
    id: u64,
    arm: SoftFallbackBranch,     // winning arm after RRF dedup (vector-first)
    arm_ranks: { vector: Option<u32>, text: Option<u32>, text_edge: Option<u32>, graph: Option<u32> },
                                 // 0-based rank within each arm the hit appeared in; None = absent from that arm
    rrf_norm: f64,               // min-max normalized fused RRF score (from ce_rerank line 5030)
    ce_score: Option<f64>,       // in-pool sigmoid CE (== SearchHit.ce_score; None outside pool)
    blended: f64,                // final score == SearchHit.score
    excluded_by_filter: bool,    // true if dropped by a SearchFilter predicate (reported, not returned)
  } ]
}
```

**Field-fidelity acceptance (R-OBS-1):** for a known hybrid query, `per_hit[i].blended` ==
`hits[i].score`, `per_hit[i].ce_score` == `hits[i].ce_score`, and `arm_ranks`/`rrf_norm` match the
values computed inside `fuse_three_arms`/`ce_rerank`. Golden test asserts the arm attribution and
ranks for a fixture corpus.

**Parity (R-OBS-4):** identical payload (snake_case Py / camelCase TS) exercised by the X1 harness.

### A.5 Implementation approach (engine threading) — informs zero-cost + feasibility

`search` does not run inline; it dispatches `ReaderRequest::Search` to the reader worker pool and
receives a `ReaderResponse` tuple `(cursor, soft_fallback, results, graph_stats)` (see
`search_inner_with_stats` `lib.rs:2995`; `read_search_in_tx` `lib.rs:5348`). So explain rides the
**existing side-channel precedent**: `GraphFrontierStats` is *already* threaded through that tuple as
an opt-in meter that never reaches the governed surface. EXP-OBS mirrors it exactly:

1. Add `explain: bool` to `ReaderRequest::Search` (default-constructed `false` on every current caller).
2. Inside `read_search_in_tx`, when `explain`: capture per-arm `rank0` in `fuse_three_arms`
   (`lib.rs:4774`, already in-loop) and `rrf_norm`/`ce_norm` in `ce_rerank` (`lib.rs:4983`, already
   computed at 5030/5032) into an `Option<Explanation>`; when `!explain`, build `None` — **no
   allocation, no extra work** (the zero-cost proof, R-OBS-2).
3. Extend the `ReaderResponse` tuple with `Option<Explanation>` (the `GraphFrontierStats` slot is the
   template).
4. `SearchResult` gains `explanation: Option<Explanation>` (A.2); `search_inner_with_stats` plumbs it.
5. New governed entry `search_reranked_explain(...)` (or an `explain` param on `search_reranked`) — the
   bare `search()`/`search_filtered()` paths keep passing `explain=false` and stay byte-identical.

**Blast radius:** engine-crate only for Slice 5 (the reader-pool protocol types + fusion/rerank +
`SearchResult`). SDK wiring (pyo3/napi/Py/TS) is **Slice 10**, deliberately separated so a schema
amendment after ratification reshapes one crate, not three bindings.

**Byte-stability guard:** every existing (`explain=false`) path must produce a byte-identical
`SearchResult.results` + cursor; a golden test pins the default path unchanged.

### A.4 Open ratification questions for the M-work owner

1. **Do the experiments need raw (un-normalized) per-arm scores**, or are `arm_ranks` + `rrf_norm`
   sufficient? (Raw scores are available but larger; default proposal omits them.)
2. **Filter-exclusion reporting** — do experiments want excluded hits surfaced in `per_hit`
   (`excluded_by_filter`), or only the returned set? (Default: returned set only; flag reserved.)
3. **Timing granularity** — are coarse per-stage `timings_ms` enough, or is per-arm CPU attribution
   needed? (Default: coarse wall-clock; finer needs more instrumentation = more hot-path risk.)

## Part B — Telemetry event + real-gold record (R-TEL-1..3)

### B.1 Telemetry event (opt-in, local, deterministic, no egress)

```
TelemetryEvent {
  schema_version: u32,
  ts_monotonic_ms: u64,        // monotonic since engine open — NOT wall-clock (determinism + privacy)
  query_id: String,            // opaque ULID-like; derived deterministically from (open_nonce, seq)
  query_chars: u32,            // length only, never the text
  result_ids: [u64],           // returned hit ids, in rank order
  arm_of: { <id>: SoftFallbackBranch },
  feedback: Option<{           // exogenous agent-supplied relevance label (the ONLY external signal)
    relevant_ids: [u64],
    irrelevant_ids: [u64],
    label_source: String,      // caller-declared, e.g. "agent:hermes"
  }>
}
```

- **Off by default.** Enabled via an explicit opt-in (engine open flag / env). When off: no record,
  no file handle, no allocation (footprint gate at Slice 15).
- **No network.** Events append to a local sink (caller-provided path); a footprint test asserts no
  socket/egress.
- **Deterministic.** `ts_monotonic_ms` + deterministic `query_id` so fixtures are reproducible.

### B.2 Real-gold record (eval-side; capture in-library, curation EVAL-ONLY)

```
GoldRecord {
  schema_version: u32,
  query_chars: u32,
  candidate_ids: [u64],
  labels: { <id>: relevance ∈ {0,1} },   // from TelemetryEvent.feedback
  embedder_id: String,
  provenance: "telemetry-capture",
}
```

A fixture telemetry run → one valid `GoldRecord` consumable by the eval harness (R-TEL-2).
This is FathomDB's exit from synthetic-gold purgatory: real query→result→agent-feedback rows.

### B.3 Open ratification question

4. **Gold-record key shape** — must `GoldRecord` match an existing eval-harness gold schema (e.g.
   the R2/m1 harness rows) so the experiment track can ingest without a shim? Need the owner to point
   at the canonical gold schema to mirror.

## Part C — Privacy / footprint contract (R-TEL-3)

- **No content leaves the box.** Neither `Explanation` nor `TelemetryEvent` carries query text or
  document bodies — only ids, lengths, scores, ranks, and caller-supplied labels.
- **Agent-supplied relevance labels are the only exogenous signal.** Everything else is derived
  from the local search.
- **Telemetry is off by default and never egresses.** Documented payload; footprint test enforces.
- **`Explanation` is opt-in and zero-cost when off** (R-OBS-2 hard gate).

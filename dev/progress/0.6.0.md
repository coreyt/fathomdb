    runs in the broad CI gate. `agent-verify` continues to run the 5 s

- **In progress:** none.
- **Blocked:**
  - **Fix 1 (restore EXCLUSIVE writer locking_mode) — escalated.**
    Per ADR-0.6.0-database-lock-mechanism § 2 the writer connection
    should set `PRAGMA locking_mode=EXCLUSIVE` while reader pool
    connections stay at default NORMAL. SQLite documents
    (<https://www.sqlite.org/wal.html>) that "if EXCLUSIVE locking
    mode is set prior to the first WAL-mode read transaction, then
    SQLite skips the use of shared memory wal-index ... only a single
    database connection at a time can access the database file."
    With the writer holding EXCLUSIVE WAL+no-shm, reader-pool
    connections cannot acquire SHARED via the wal-index — every
    `search()` reader call hits rusqlite's 5 s `busy_timeout`, e.g.
    AC-004b's 500 search ops × 5 s = ~42 min wall-clock for one
    test. The ADR mandates the pragma; the Phase 8 reader pool
    mandates concurrent multi-reader access; both cannot be true with
    current SQLite semantics. Reverted my Fix-1 attempt; lib.rs writer
    setup remains as the implementer left it (no EXCLUSIVE pragma).
    Resolution requires an architectural decision (drop the reader
    pool, OR amend the ADR to accept `-shm` so multi-reader works,
    OR run readers via the writer's connection through a shared-cache
    handle) — that is orchestrator scope.
- **AC status (final):**
  - Green: AC-002, AC-003a/b/c/d, AC-004a/b/c, AC-005a/b, AC-008,

    (deterministic-slow-cte fixture not yet pinned to CI baseline),
    AC-009 (poison fixture deferred).
  - Blocked: writer EXCLUSIVE locking_mode (see above).
- **Decisions:**
  - AC-022b moved into a child-process sub-test (matching AC-022a's
    pattern). The previous "before-close vs after-close" reframing in
    the same process is retracted; it answered a different question
    than P-FD-TOL.
  - `Engine::execute_for_test` is `#[cfg(debug_assertions)]` +
    `#[doc(hidden)]` so release builds do not expose it; it routes
    through `detect_slow` so AC-007a/b can bind to the slow path
    without inventing a new emission site.
  - `scripts/check.sh` opts into `AGENT_LONG=1`. `agent-verify.sh` does
    not, keeping the agent-loop gate fast.
- **Retracted from prior 2026-05-02 entry:**
  - "AC-007a (slow event at threshold)" and "AC-007b (threshold
    reconfigurable at runtime)" green claims — both reverted to
    `#[ignore]`; the prior tests asserted threshold=0 + sub-ms search,

    `Finished`/`Failed`. `set_profiling`/`set_slow_threshold_ms` mutate
    the live values without restart.
  - Added a bounded `ReaderPool` of 8 read-only connections opened
    `journal_mode=WAL` + `query_only=ON`. `search()` borrows a reader so
    concurrent reads never serialize behind one connection. Dropped
    `locking_mode=EXCLUSIVE` on the writer; the sidecar lock provides
    engine-lifetime exclusivity. `close()` drains readers before
    closing the writer so SQLite's last-handle checkpointer runs on the
    writer.
  - AC flips: AC-002 (no log files without subscriber); AC-003a/b/c/d
    (writer/search/admin/error events route to host subscriber);
    AC-004b (counter delta exact over 1,000 mixed ops); AC-005a
    (profiling toggleable on running engine); AC-007a (slow event at
    threshold); AC-007b (threshold reconfigurable at runtime); AC-008
    (slow signal feeds lifecycle phase); AC-021 (zero `SQLITE_SCHEMA`
    error events under 8 concurrent readers + 1 admin DDL writer).
  - `./scripts/agent-verify.sh` green after each step.

  - Reader pool size = 8 (matches AC-021's reader-thread count). Pool
    is `Mutex<Vec<Connection>>` + `Condvar` rather than a lock-free

    concurrent test interleaving in the previous formulation; the new
    invariant ("close releases at least as many fds as it consumes")
    is equivalent for leak detection and is robust to parallel tests.
  - `AdminSchema`-only batches count `admin_ops`; mixed canonical +
    `AdminSchema` batches still count `writes` because `dev/design/
    engine.md` § Batch submission semantics treats `admin.configure`

    for both.

# 0.6.1 Release Notes

**Released: 2026-05-24** (pending RC1 → V → GA slices on the
`0.6.1-rewrite` release branch).

0.6.1 is a **patch release** on the 0.6.x line per
`ADR-0.6.0-no-shims-policy.md` § 54 (literal contract:
**no API breaks, bugfix-only**). It closes three items deferred
at 0.6.0 GA, resolves three open Dependabot advisories, and carries
the AC-012 canonical-runner re-measurement as documented evidence
(verdict RED — Pack 7 performance work escalates to 0.7.0 per
HITL 2026-05-24).

## What's new in 0.6.1

### `Engine.open` structured open report — bindings closure

Both the Python and TypeScript SDKs now surface the structured
open report that native Rust populates at `Engine.open` time.
0.6.0 GA dropped this on both bindings (the
[`12-TX-OPENREPORT` deferred item](https://github.com/coreyt/fathomdb/blob/main/docs/release-notes/0.6.0.md#engineopen-structured-open-report));
0.6.1 surfaces it via a new engine-attached accessor — Shape D
per HITL 2026-05-24. The `Engine.open(...)` signature is
unchanged in both bindings (additive accessor; zero legacy-
callsite risk).

- **Python** (`AC-068c`):

  ```python
  engine = fathomdb.Engine.open(...)
  report = engine.open_report()
  report.schema_version_before   # int
  report.schema_version_after    # int
  report.migration_steps         # list
  report.embedder_warmup_ms      # int
  report.query_backend           # str
  report.default_embedder        # embedder-identity payload
  ```

  Snake-case identifiers mirror the native `OpenReport` struct
  verbatim. Idempotent — repeat calls return identical data
  (the report is a snapshot, not live state).

- **TypeScript** (`AC-068d`):

  ```ts
  const engine = await fathomdb.Engine.open(...);
  const report = engine.openReport();
  report.schemaVersionBefore;   // number
  report.schemaVersionAfter;    // number
  report.migrationSteps;        // ReadonlyArray<MigrationStepReport>
  report.embedderWarmupMs;      // number
  report.queryBackend;          // string
  report.defaultEmbedder;       // embedder-identity payload
  ```

  CamelCase mirror of the native fields. `openReport()` is
  **sync** — the report lives in the napi engine struct once
  the `Engine.open` Promise has resolved. Idempotent.

This closes the last ❌ row in
`dev/notes/12-TX-parity-matrix.md` and is the TypeScript-parity
anchor for the 0.6.x release-line goal.

### Axis-E independence — first post-GA demonstration

`fathomdb-embedder-api` (axis E) stayed at `0.6.0` across the
axis-W bump from `0.6.0` → `0.6.1`. Per Wake decision `d-001`,
the embedder-API trait surface bumps on its own cadence — only
when the trait surface changes — and 0.6.1 ships no trait-
surface change. This is the first post-GA exercise of the
two-axis discipline; a regression-sentinel test
(`scripts/tests/test_set_version.sh` test #13) codifies the
invariant for future bumps.

### Security advisories resolved

Three of the four open Dependabot advisories at 0.6.0 GA are
closed in this release; the fourth was a false-positive against
fathomdb.

| Advisory                  | Ecosystem | Fix                                                                 |
| ------------------------- | --------- | ------------------------------------------------------------------- |
| **RUSTSEC-2025-0020**     | cargo     | `pyo3` `0.22.6` → `0.24.1`; `*_bound` API rename (24 callsites)     |
| **GHSA-mh29-5h37-fv8m**   | npm       | `js-yaml` `4.1.0` → `4.1.1` via `markdownlint-cli2` `0.18` → `0.22.1` |
| **CVE-2024-3651** (`idna`) | python   | Confirmed false-positive — `idna` is not in the dependency graph; `src/python/uv.lock` checked in to make the audit reproducible |

### B-001 forward-retag

`scripts/security/check_removal_changelog.py` and its bash
wrapper advance their default `--base` ref from `v0.6.0` to
`v0.6.1`, so the "removals since last released API" anchor
tracks the new GA. The AC-050c regression-sentinel test #4
(live-git default-base-ref path) will be **transiently RED in
the BUMP → RC1 → GA window** until the `v0.6.1` tag is pushed;
this is the documented hand-off cost flagged in the B-001
codex CONCERN (2026-05-24).

## What stays deferred

### AC-012 canonical re-measurement — RED, escalated to 0.7.0

`AC-012` (text-query latency on FTS5, p50 ≤ 20 ms / p99 ≤ 150 ms)
was originally a 0.6.1 candidate, intended to close via canonical-
runner re-measurement (the dev runner that produced the 0.6.0 RED
verdict was aarch64 Tegra ~3× slower than canonical x86_64). The
0.6.1-AC012-measure slice ran on 2026-05-23 via
[`.github/workflows/perf-canonical.yml`](https://github.com/coreyt/fathomdb/actions/runs/26346417896)
on canonical x86_64 tier-1 hardware (AMD EPYC 9V74, 4 cores,
Ubuntu 24.04.4, rustc 1.95.0, SQLite 3.45.x via
`libsqlite3-sys` 0.28.0) at N=1,000,000.

Verdict: **RED on both percentiles.**

| Metric     | Observed (ms) | Budget (ms) | Verdict | Multiple over budget |
| ---------- | ------------- | ----------- | ------- | -------------------- |
| AC-012 p50 | 140.95        | ≤ 20        | **RED** | 7.05×                |
| AC-012 p99 | 458           | ≤ 150       | **RED** | 3.05×                |

The Pack 7 un-defer trigger (regression > 20% vs ADR-pinned
budget) fires by ~30× on p50. Per HITL 2026-05-24, Pack 7
escalates to **0.7.0** as a perf-only release (budget revision +
tuning). 0.6.1 carries this measurement as evidence and does
NOT claim AC-012 closure. Full slice output:
`dev/plans/runs/0.6.1-AC012-measure-output.json`.

### Pack 7 sibling perf ACs

`AC-013` (vector retrieval latency), `AC-019` (mixed-retrieval
stress tail), and `AC-020` (N=8 concurrent reader scaling) stay
deferred to 0.7.0 alongside AC-012. AC-020 closure is
architectural (vendor-SQLite work — `SQLITE_CONFIG_PCACHE2` /
WAL2 / reader-writer physical separation); see the 0.6.0 release
notes for the substrate analysis.

### Logical-id verbs — 0.8.0

`purge_logical_id` and `restore_logical_id` stay deferred to
**0.8.0** per HITL 2026-05-24 rescope (the 0.6.0 release notes
said 0.7.x; this slipped one minor as Pack 7 perf work claimed
the 0.7.0 anchor). The canonical-identity substrate stays
design-only in 0.6.1 — same as 0.6.0.

For 0.6.x clients, bulk-delete by source still uses
`fathomdb recover --excise-source <id>` (Phase 10a).
Restore-from-log workflows wait for 0.8.0.

## Upgrade notes

Patch release — **no breaking changes.** Semver patch implies
the public surface from 0.6.0 still works unchanged. The two
new accessors (`engine.open_report()` / `engine.openReport()`)
are **additive**; `Engine.open(...)` signatures are unchanged
from 0.6.0 in both bindings.

```sh
# crates.io (Rust)
cargo update -p fathomdb --precise 0.6.1

# PyPI
pip install --upgrade fathomdb==0.6.1

# npm
npm install fathomdb@0.6.1
```

Axis-E consumers — `fathomdb-embedder-api` is unchanged at
`0.6.0`. No update required for downstream embedder
implementations.

## Compatibility

**No 0.5.x compatibility shims or migrations** — same posture
as 0.6.0 per `ADR-0.6.0-no-shims-policy.md`. Clients on 0.5.x
must follow a separate migration path (not provided in this
release).

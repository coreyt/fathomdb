# fathomdb 0.6.0 Rewrite Proposal

Status: draft 2026-04-24
Supersedes: `0.6.0-roadmap.md` (Cypher-forward direction)
Inputs:
- `0.5.7-corrected-scope.md` (recovery plan for 0.5.x)
- Needs audit 2026-04-24 (agentic-client requirements)
- Role retrospective 2026-04-24 (what fathom should be between client and SQLite)

## Thesis

**fathom has been building thick. Every four-day breakage came from fathom trying to own something the client was better positioned to own.**

The rewrite adopts **thin-plus**: thin about data and schema (client owns meaning), thick about the three things SQLite genuinely doesn't do well and every agent client would otherwise rebuild identically:

1. Durable serialized write with post-commit projection enqueue.
2. Embedder-identity-bound vector search.
3. Composed FTS + vector + graph query fusion with shape-hash prepared-statement caching.

Everything else is client concern or CLI concern, not runtime SDK concern.

**2000-line test.** If I deleted fathom and rebuilt a minimum agent datastore against `rusqlite + sqlite-vec + fts5`, which ~2,000 lines would I re-import verbatim? Those 2,000 lines define the core. Today's engine is ~45,000 lines. The rewrite target is ~8,000 (core + Python binding + tests).

## Agentic-client requirements (pinned)

**Essentials** (must have; any rewrite dropping these cedes position to Chroma/LanceDB/Turso):

1. Canonical typed records keyed by stable `logical_id`.
2. Schemaless JSON properties.
3. FTS over canonical text with safe, forgiving grammar (agent/LLM-generated queries must not inject FTS5 control syntax).
4. Vector search with managed projection.
5. Hybrid retrieval (FTS + vector) with `was_degraded` soft-fallback contract.
6. Durable, crash-safe writes with read-your-writes.
7. `superseded_at` versioning (logical_id stable across correction).
8. Graph edges with typed labels.
9. Chunks as first-class units under a node.
10. Provenance trail (`why do I believe this?` — competitor differentiator).
11. Single-process embedder configured once at open, reused for read and write.
12. Hard-error vs soft-degrade split (EmbedderNotWired / KindNotVectorIndexed / DimensionMismatch = hard; `was_degraded` = soft).
13. Canonical writes never blocked by projection unavailability.
14. `drain(timeout)` hook for tests and batch ingest.
15. Python + Rust at 0.6.0. TS committed to 0.7.0 (consumer confirmed; TS-based app recovery depends on it).
16. Auto-migrating schema (agents ship as single binaries, no DBA).
17. Single-file DB, no server, no network.
18. Structured typed filters alongside text/semantic search.
19. Integrity / semantics / rebuild recovery surface (rare-use, load-bearing).

**Anti-requirements** (must NOT do):

- Embedder model hosting / downloading / lifecycle.
- Chunking strategy (client passes chunks; fathom stores them).
- Summarization / compression of old turns.
- Agent orchestration, tool dispatch, loops.
- Cross-embedding-model vector comparison.
- Pre-embedded raw vector inserts as a primary API (admin/import only).
- A query DSL larger than engine-lowering guarantees.
- Multi-tenant access control, distributed consensus, server protocol.
- Silent degradation on misconfiguration.
- Identity strings on vector configs. Identity belongs to the embedder.

## Public API: five verbs

```python
engine = Engine.open(path, embedder=embedder)
engine.admin.configure({
    "kinds": {
        "KnowledgeItem": {
            "fts":  {"paths": ["$.body", "$.title"], "tokenizer": "unicode61"},
            "vec":  {"source": "chunks"},
            "edges": ["references", "derived_from"],
        },
        "ConversationTurn": {
            "fts":  {"paths": ["$.text"]},
            "vec":  {"source": "chunks"},
        },
    },
})
engine.write(batch)                          # nodes + chunks + edges + provenance in one txn
hits = engine.search(...).execute()          # typed builder; FTS + vector + edge traversal composed
engine.close()
```

That's the whole public admin surface. Primitives (`configure_fts`, `configure_embedding`, `configure_vec_kinds`, `register_fts_property_schema`, `drain_vector_projection`, etc.) are **private implementation**; they exist inside the engine but have no public FFI entry.

Recovery tooling is a CLI, not an SDK:

```
fathomdb doctor check-integrity <path>
fathomdb doctor regen-vectors <path>       # reads embedder from DB, re-embeds
fathomdb doctor excise-source <path> <id>
fathomdb doctor safe-export <path> <dest>
```

## Scope boundary

### Fathom owns

- Opening + closing a durable SQLite file with correct drop order (reader pool → writer → vector actor → rebuild actor → lock).
- Writer actor: single-writer mpsc, fsync-on-commit, post-commit projection enqueue, typed `WriteReceipt`.
- Storage: `nodes`, `edges`, `chunks`, `provenance_events`, `vec_nodes(kind, row_id, embedding)` (single shared table, no per-kind DDL), per-kind FTS5 virtual tables (one-time DDL at `configure`, not runtime).
- One embedder per engine wired at open. Engine enforces identity invariant; regen path takes an embedder, never an identity struct.
- Query surface that composes: logical_id lookup, FTS over configured paths with safe grammar, vector search over the embedder, edge traversal, branch fusion, shape-hash prepared-statement cache, `was_degraded` metadata.
- Durable vector-projection work queue with incremental + backfill priorities; blocking `drain(timeout)` hook for tests.
- Typed errors: `EngineError`, `EmbedderNotWired`, `EmbedderNotConfigured`, `DimensionMismatch`, `KindNotVectorIndexed`, `WriterTimedOut`, `SchemaMismatch`.
- Five-verb public SDK (Python + Rust). TS deferred.

### Client owns

- Kind definitions: what a `KnowledgeItem` is, what properties it has, which paths are FTS'd, which kinds are vector-indexed.
- Conversation state, agent-level concepts (turn, tool call, thought). Fathom stores nodes; client knows what they mean.
- Embedder choice and wiring. `fathomdb-embedder` is a separate package (convenience BGE-small wrapper); client may implement `QueryEmbedder` trait directly.
- Rebuild policy: tokenizer migrations, post-model-swap regen. Run via CLI.
- Chunking strategy. Fathom stores chunks with `{node_logical_id, text, position}`; client decides segmentation.
- Operational collections: **extracted to `fathomdb-ops` sibling package or dropped**. Not core admin.
- Cypher: **extracted to `fathomdb-cypher` translator package or dropped**. Not core engine. Removes ~40% of `fathomdb-query` AST surface.

## What we got right (preserve verbatim)

1. **Identity belongs to the embedder.** One of the strongest single decisions in the codebase. Memorialize as an architectural CI test: grep the `VectorRegenerationConfig` type for identity fields; any addition fails the build.
2. **Writer actor serialization with documented drop order.** Load-bearing for graceful shutdown.
3. **`logical_id` / `superseded_at` append-only model.** Promote to documented core data model, not implementation detail.
4. **Shape-hash prepared-statement cache with clear-all eviction.** Keep.
5. **One embedder per engine at open.** No mid-flight swap API is a feature.
6. **`drain_vector_projection(timeout)` as a blocking synchronization hook.** Every Memex integration test depends on it.
7. **Safe text grammar** (`spec-supported-query-primitives-and-operators-2026-04-11.md`): literal-default, uppercase `OR`/`NOT`, phrases, implicit AND. Agent/LLM input tolerant. Preserve verbatim.
8. **Hard error vs soft degradation contract** (`was_degraded` metadata, not silent zero-results).

That is the entire list. Everything else is either scar tissue or not yet demonstrated reliable.

## What we got wrong (delete or restructure)

Ranked by severity. Not "bugs we shipped" — architectural choices that made bugs inevitable.

1. **Admin DDL at runtime on shared read pool.** Per-kind `vec_<Kind>` partitioning forced `CREATE VIRTUAL TABLE` into the hot path; readers holding `prepare_cached` statements `SQLITE_SCHEMA`-invalidate. Rewrite: **single `vec_nodes(kind, row_id, embedding)` table, one-time DDL at `configure`**. Admin ops route through writer actor's write connection; read pool never sees DDL.
2. **Test suite mocks Rust out.** Zero tests exercise the full Memex flow. Python tests assert JSON shapes against mocks (`test_profile_management.py:112, 419`). Rewrite: **one integration test per release gate**, runs built wheel in fresh venv, full flow + concurrent read during admin DDL, all platforms. Rust-mocking tests are negative-value and deleted.
3. **Surface-by-addition, never deletion.** 25 migrations. Three parallel identity tables simultaneously live. 45 Python admin methods. `configure_vec(kind_or_embedder)` dispatching on `isinstance(str)` across two schema generations in one verb. Rewrite: **single `admin.configure(schema)` declarative call**. Primitives go private. Identity table: one, not three.
4. **Per-pack atomic Rust+Python+TS merges.** Pack F1/F1.5/F2/F3 shipped all three FFI layers in same commit. Rust integration gap became Python gap and TS gap together. Rewrite: **Rust-first with Rust integration test**; Python is a follow-on PR; TS lands in 0.7.0 against a stable Rust+Python API, its own PR cadence.
5. **Silent-degrade branches as a design pattern.** `EmbedderChoice::Builtin → None`. Deprecation shims that lie. Rewrite: **`EmbedderChoice::Builtin` variant deleted**. No embedder wired → `Engine::open` hard-errors. No deprecation shims; breaking changes announced in changelog, period.
6. **God modules.** `admin/mod.rs` 7,150. `writer/mod.rs` 8,213. `coordinator.rs` 6,014. `compile.rs` 2,078. Rewrite: extract by responsibility (`operational_writes.rs`, `projection_enqueue.rs`, `durability.rs` from writer; `branch_execution.rs`, `vector_branch.rs` from coordinator). Code-health norms (§"Code health") surface complexity objectively; human review decides. No hard LoC ceilings — gamable by file split.
7. **Query builder encoded half of Cypher**, shipped to enable a future public API that isn't the agentic-client use case. Rewrite: **kill Cypher from core engine entirely**. Not in 0.6.0, not as a sibling, not on the roadmap. If a future named consumer asks for Cypher, design then.
8. **Rebuild-as-runtime.** `rebuild_actor.rs` 983 lines elevates human-operator concerns to first-class runtime. Rewrite: **rebuild runs in the writer actor, triggered by `admin.configure` diff application**. No separate actor. Background periodic tick is a one-liner on the projection actor, not a new subsystem.
9. **Configuration surface = API surface.** 45 methods = 45 wire formats = 45 changelog entries on every shape change. Rewrite: **declarative `configure(schema)` is one method with one wire format**. Schema shape evolves; method signature does not.

## Architecture

### Storage

```
nodes(logical_id TEXT, kind TEXT, properties JSON, row_id INT PK, created_at, superseded_at)
  idx_nodes_active_logical_id UNIQUE WHERE superseded_at IS NULL
  idx_nodes_kind_active

edges(source_logical_id, target_logical_id, kind, properties JSON, row_id PK, created_at, superseded_at)
  idx_edges_source_kind_active
  idx_edges_target_kind_active

chunks(node_logical_id, position INT, text, row_id PK, created_at, superseded_at)
  idx_chunks_node_active

provenance_events(subject_logical_id, event_type, metadata JSON, created_at)
  idx_provenance_subject

vec_nodes(kind TEXT, row_id INT, embedding BLOB)   -- single table, kind column, no per-kind DDL
  idx_vec_nodes_kind

vec_chunks(kind TEXT, row_id INT, embedding BLOB)  -- same
  idx_vec_chunks_kind

fts_<Kind>_nodes  -- FTS5 virtual table per kind, one-time DDL at first configure
fts_<Kind>_chunks -- same

schema_config(version INT, schema JSON)  -- the declarative schema, persisted
vector_embedding_profile(model_identity, model_version, dimensions, normalization, active_at)  -- singleton
vector_projection_work(id, kind, row_id, priority, created_at, ...)
```

**One identity table (`vector_embedding_profile`, singleton row).** No `vector_profiles`. No `projection_profiles('*','vec')`. No per-kind enable table separate from `schema_config`. Diff is computed by parsing `schema_config.schema` and comparing to the desired payload.

**One shared vec table per facet (`vec_nodes`, `vec_chunks`).** `kind TEXT` column with index. No runtime DDL. `SQLITE_SCHEMA` class eliminated by construction.

### Writer actor (preserved)

```
Client → mpsc<WriteRequest> → writer actor → one write connection
                                 ↓ post-commit
                             projection work enqueue
                                 ↓ wakeup
                             VectorProjectionActor
                                 ↓ on `drain(timeout)` blocks until idle
```

Drop order: reader pool → writer → projection actor → lock → file handle. Enforced by field order on Engine struct.

### Query execution (trimmed from current `coordinator.rs`)

- `logical_id` point lookup: direct SELECT.
- FTS branch: MATCH on `fts_<Kind>_nodes` with safe-grammar tokenization at parse time (NOT passed raw to FTS5).
- Vector branch: `active_embedding_profile.dimensions` gate; `query_embedder` at coordinator; KNN on `vec_<facet>` filtered by `kind`.
- Edge traversal: CTE, typed labels, depth ≤ N.
- Branch fusion: shape-hash caches the assembled SQL; `was_degraded` aggregated.

Kill: `EdgePropertyEq`, `EdgePropertyCompare`, `JsonPathFusedIn`, `EdgeExpansionSlot`, and all the Cypher-adjacent AST nodes that don't serve a five-verb use case. If Memex needs them, it calls through `QueryBuilder` typed filters, which lower to parameterized SQL, not an AST intermediary.

### Embedder packaging

- Core crate `fathomdb-engine`: declares `QueryEmbedder` + `BatchEmbedder` traits only.
- Sibling crate `fathomdb-embedder`: ships the BGE-small Candle implementation. Optional install (`pip install fathomdb-embedder` / `cargo add fathomdb-embedder`).
- `EmbedderChoice::Builtin` deleted. Callers who want builtin import it explicitly:

```python
from fathomdb import Engine
from fathomdb_embedder import BuiltinBgeSmall

engine = Engine.open(path, embedder=BuiltinBgeSmall())
```

- No feature-gated silent-None surface. No wheel-feature divergence. No "did the wheel ship the embedder?" bug class.

### Recovery tooling (CLI, not SDK)

Commands moved from `AdminClient` methods to `fathomdb doctor` subcommands:

```
fathomdb doctor check-integrity <db>
fathomdb doctor check-semantics <db>
fathomdb doctor regen-vectors <db> --embedder-package=fathomdb-embedder
fathomdb doctor rebuild-missing-projections <db>
fathomdb doctor rebuild-fts <db> <kind>
fathomdb doctor excise-source <db> <source-logical-id>
fathomdb doctor purge-logical-id <db> <logical-id>
fathomdb doctor restore-logical-id <db> <logical-id>
fathomdb doctor safe-export <db> <dest>
fathomdb doctor trace-source <db> <subject>
```

Library surface drops ~10 verbs. Human-operator surface gains a proper ergonomic CLI.

### Operational collections — HITL decision

**Status:** needs more context before scope lock.

Operational collections in 0.5.x: 17 admin methods (`register_operational_collection`, `read_operational_collection`, `compact_operational_collection`, `plan_operational_retention`, `purge_operational_collection`, etc.), 1,683 LoC in `admin/operational.rs`, mirrored across Python and TS bindings. A filtered-materialized-view system: client declares a filter predicate, engine maintains a populated view, reads go against the pre-filtered set.

**Trade-offs:**

| Option | Pros | Cons |
|---|---|---|
| **A. Core admin verb in 0.6.0** | Existing callers keep working; no package to install; tight coupling to canonical writes is efficient | Adds 1,683 LoC back to core; extends admin surface beyond 5 verbs; re-inflates `admin/mod.rs` toward god-module territory |
| **B. Sibling `fathomdb-ops` package in 0.6.0** | Core stays lean; users who don't need it pay zero; independent release cadence | New version-skew surface; requires clean internal API for core → ops access; double the CI packaging work |
| **C. Drop entirely from 0.6.0, reintroduce in 0.7.0 if asked** | Simplest 0.6.0; proves demand before investing | Breaks any current caller; may be what pushes a client to a competitor |
| **D. Fold into `configure(schema)` as a first-class kind facet** | Declarative; no new surface; collections become just another kind declaration with a filter predicate | Changes the data model; biggest design work; may not match existing semantics |

**Required before decision:**

1. Grep each current fathomdb consumer (Memex, Armada, Wake, openclaw, hermes) wrapper code for `operational_collection` call sites. Count of distinct usages per client.
2. For each usage, what is the filter predicate and the access pattern? This determines whether Option D is feasible.
3. Does any client use retention/compaction? If nobody uses `plan_operational_retention` or `compact_operational_collection`, those drop immediately regardless of A/B/C/D for the rest.

**HITL request:** user confirms which clients actually use operational collections and what they use them for. Decision deferred to that data.

**Interim:** 0.6.0 scope is committed on everything else; operational-collections resolution is the last gate before implementation starts.

## Designs

- **One declarative admin call.** `admin.configure(schema)` computes diff vs persisted `schema_config` and applies idempotently. Returns per-kind per-facet outcome (`added` / `unchanged` / `replaced` / `rebuild_required` / `rebuild_in_progress`). Primitives are private.
- **`logical_id` / `superseded_at` promoted to documented data model.** Intro doc + architecture diagram + test-contract doc. Not an implementation detail buried in bootstrap.
- **Vector identity invariant as a type-level constraint** (not a grep). Private `EmbedderIdentity` struct, constructible only via `QueryEmbedder::identity()`. FFI callers cannot construct one from JSON; they supply an embedder and the engine derives identity. Replaces the 0.5.x pattern where identity fields appeared in wire payloads.
- **Cypher dropped.** Not in 0.6.0 as sibling, not on roadmap. The value prop is agent-datastore, not graph-DB. If a future named consumer asks for Cypher, design then. No speculative AST surface in 0.6.0's `fathomdb-query`.
- **`fathomdb-embedder` as its own package.** Versioned independently. Wheel size for core drops ~100MB candle/hf-hub weight.
- **Sibling-package version-skew policy.** See next.

### Version-skew policy (sibling packages)

A core engine and a sibling embedder package installed separately can drift. 0.5.x did not have this bug class because everything was monolithic; 0.6.0 introduces it with `fathomdb-embedder`. Policy:

- Tiny third crate `fathomdb-embedder-api` (~100 LoC, no deps besides `std`): declares `QueryEmbedder`, `BatchEmbedder`, `EmbedderIdentity`. SemVer-stable. Changes only on minor version.
- `fathomdb-engine` depends on `fathomdb-embedder-api = "0.6"`.
- `fathomdb-embedder` depends on `fathomdb-embedder-api = "0.6"`.
- Neither engine nor embedder-builtin crate imports each other directly.
- Python: `fathomdb` and `fathomdb-embedder` both declare `fathomdb-embedder-api` as a direct dep with `>=0.6,<0.7`. pip resolves to a single version.
- PyPI + crates.io: three packages to version and publish. Release discipline applies equally — every release tags all three even when content unchanged, to prevent implicit coupling through version drift.

Same pattern applies if `fathomdb-ops` lands as a sibling (Option B above): `fathomdb-ops-api` for the shared trait/types if there are any, or just a direct `=0.6.x` pin on `fathomdb-engine`.

## POST (power-on self-test)

Preference: validate at startup, not at runtime. Every invariant that can be checked once at `Engine::open` must be checked there and only there. Runtime code assumes invariants hold. No defensive re-checking scattered through hot paths.

POST runs on every `Engine::open`, completes in <50ms on an open store, and either succeeds clean or hard-errors with a typed `EngineError`. No soft degradation. No warnings.

### POST checks (in 0.6.0)

Each check justifies its inclusion: (J) = justified, (O) = out-of-scope, HITL = user decides.

1. **Schema version is v1 exactly.** (J) Schema is forward-only with no v2 yet; any other version means the store belongs to a different fathomdb line. Hard error, name the version seen.
2. **`vector_embedding_profile` singleton consistency.** (J) Zero or one active row; `dimensions` is a positive integer; `model_identity` non-empty. Memex Issue 2 root cause class.
3. **Embedder identity match.** (J) If a store has an active embedding profile AND `Engine::open` was passed an embedder, `embedder.identity()` must equal the stored profile. Mismatch is the class that silently serves wrong vectors. Hard error; name both sides.
4. **Embedder dimension match.** (J) Profile `dimensions` must equal `embedder.identity().dimensions`. Caught at open, not mid-drain.
5. **`sqlite-vec` extension available.** (J) If the store has any `vec_nodes` or `vec_chunks` row, the extension must load. No runtime check per query. Hard error at open if extension is missing and vectors exist.
6. **Schema-config parses.** (J) `schema_config.schema` column is valid JSON matching the declarative shape. If corrupted, the whole store is unreachable; better to fail open than first write.
7. **`vec_nodes` / `vec_chunks` `kind` values all appear in `schema_config.kinds`.** (J) Orphaned vec rows = abandoned prior `configure()` paths. Hard error; `fathomdb-doctor` fixes. Replaces the current silent `was_degraded` path for "kind vanished."
8. **Writer connection holds SQLite lock.** (J) If lock contested (another process already holds the DB), fail open, don't queue writes into a dead actor.

### Out of POST

- (O) FTS tokenizer matches prior configure. Cheap at runtime if it ever matters; not a startup correctness concern.
- (O) Edge logical_id referential integrity across all rows. O(n), not O(1); `fathomdb-doctor check-integrity` on demand.
- (O) Vector backfill queue age. Metric, not an invariant.
- (O) Chunk text non-empty. Data quality, client concern.

### Deferred (HITL — decide before scope lock)

- **Cross-check `schema_config.kinds[k].vec.dimensions` matches stored vec row widths.** One-row sample per kind at open. Catches the case where a store was hand-modified or a prior `configure()` stopped mid-apply. Cheap but requires HITL call: always-on vs `--paranoid` flag.
- **Embedder probe (send a short test string, assert dim-sized output).** Catches broken embedder wiring before first real query. Cost: one embed call per open, may be slow for network embedders. HITL: on by default, opt-out, or opt-in.

### Why POST over runtime checks

Runtime checks (the current pattern in `coordinator.rs` with `was_degraded` degradation branches + retry wrappers + silent-None resolution) have failure modes that compound: a check fails mid-query, the client gets a soft-zero result, the bug surfaces hours later in a user-facing symptom. POST moves the same checks to a single, testable, observable moment. One test covers all startup paths. One error message per failure. No partial state.

Every POST check has a red test in `memex_flow.py` that corrupts or omits the thing being checked and asserts the specific typed error.

## Tests

Strategy: one gate, one real flow, against the built artifact. Everything else is signal, not gate.

1. **Integration test is the only release gate.** `tests/memex_flow.py` (Python) + `tests/memex_flow.rs` (Rust). Runs against **built wheel in fresh venv** / compiled crate. Full flow: open → configure → write → drain → search → close. Concurrent reader thread during admin DDL. Zero `SQLITE_SCHEMA` warnings asserted in tracing. Must cover, at minimum:
   - Happy path, multi-kind, real embedder (`FakeEmbedder`).
   - Concurrent read during `configure()` apply.
   - **Each POST check** — one subtest per check that corrupts the invariant and asserts the typed error.
   - **Crash recovery** — kill -9 mid-write, reopen, read-your-writes holds for committed rows.
   - **Dimension mismatch** — second open with a differently-dimensioned embedder on an existing store. Hard error at POST.
   - **Embedder panic** — embedder raises mid-drain; drain surfaces the error without corrupting the queue.
   - **Platform file-lock** — Windows LockFileEx and POSIX fcntl both honored. Two opens → second fails cleanly.
   - All four platforms: Linux x86_64, Linux aarch64, macOS, Windows.
2. **Delete Rust-mocking Python tests.** Negative-value coverage. `test_profile_management.py`-style mocks go.
3. **Per-verb unit tests demoted.** Catch unit regressions, not feature coverage. Not a release gate input.
4. **Architectural health signals (informational, not CI-gate).** See §"Code health" below.
5. **Built wheel, fresh venv, every release.** Not `pip install -e`. Feature-flag stripping is only caught by built-wheel runs.
6. **Optional pre-tag dogfood run** (HITL — pending Memex availability). One engineer, one workload, fresh DB, their real ingestion pipeline, 24-48h before `git tag 0.6.0`. Not soak. Staged rollout. If Memex cannot spare the engineer, skip and rely on the integration test.
7. **No soak.** If the integration test passes on all platforms, ship. If not, fix.

## Code health

Team norms, not CI gates. LoC ceilings are gamable by file split; they do not measure what matters. Each norm below is an objective signal surfaced on every PR as a comment; the gate is human review, not a red X.

### Norms (0.6.0 targets)

| Norm | Target | Measure | Tool | Escalation on exceedance |
|---|---|---|---|---|
| Cyclomatic complexity per function | CCN ≤ 15 (≤ 10 preferred) | McCabe decision-point count | `lizard` (polyglot, single binary) | PR description names function + justifies; one architecture-inspector review |
| Cognitive complexity per function | ≤ 20 | SonarSource algorithm (weights nesting) | `lizard --ECC` / `rust-code-analysis` / `flake8-cognitive-complexity` | HITL review; consider extraction |
| Function fan-out | ≤ 25 unique callees | Static call graph | `rust-code-analysis` / `pyan3` | Critic flags for decomposition; no block |
| Module public surface | ≤ 30 public symbols | Exported name count | `cargo public-api` / `griffe` | Team conversation; intentional facade vs pub leak |
| Inbound coupling (Ca) | review at Ca > 20 | Distinct modules importing this one | `cargo-modules` / `grimp` | API-change stability review; not a block on new code |
| Duplication ratio | < 3% (blocks ≥ 50 tokens) | Token-level near-dup | `jscpd` / `similarity-rs` | PR comment lists blocks; author extracts or justifies |
| Per-function NLOC | ≤ 80 | Non-comment non-blank lines | `lizard` | Inspector review; often legitimate state machine |

### Measured automatically on every PR (informational only)

- `lizard` diff: any function crossing any threshold, named with before/after.
- `cargo public-api` diff: symbols added/removed.
- `jscpd` delta: new duplicate blocks.

No CI red X on any of these. Human reviewer reads the comment.

### On demand (quarterly, or when refactoring a subsystem)

- Full Ca/Ce coupling graph.
- Fan-out heatmap for hotspot modules.
- Full duplication report.

### What NOT to measure (explicit)

- File-level LoC ceilings — gamable by file split.
- File-level cyclomatic sum — meaningless aggregate.
- Test count / test-to-code ratio — rewards low-value tests.
- Comment-to-code ratio — rewards noise comments.
- Trait-impl depth per type — Rust feature, not smell.

### Architectural invariants (CI-enforced, narrow)

These are not code-health metrics; they are structural invariants whose violation is a bug. Narrow, cheap, low false-positive:

- **Identity invariant.** Preferred: type-system invariant — a private `EmbedderIdentity` type constructible only through `QueryEmbedder::identity()` (not by JSON deserialization of a caller-supplied struct). Replaces the earlier "grep `VectorRegenerationConfig` for identity fields" proposal, which only catches the specific name. HITL: confirm the type-level shape works for FFI.
- **Public admin method count.** Python `AdminClient` has exactly 3 public methods: `configure`, `current_config`, `describe_kind`. Test imports the module and asserts.
- **No `#[allow(deprecated)]` in crate root.** Grep check. Deprecation shims are removed, not carried.

Three checks, each one line, each testing a specific invariant. Not proxies for "is the code healthy."

## Greenfield 0.6.0 (no 0.5.7, no data migration, no TS in 0.6.0)

0.5.x abandoned. No recovery release, no deprecation dance, no parallel API. No 0.5.x → 0.6.0 data migrator in 0.6.0. No TypeScript binding in 0.6.0 (it lands in 0.7.0, committed).

**Rationale.** 0.5.x broke because surface grew faster than tests. Transitional releases (0.5.7, 0.5.8, 0.6.0-alpha) repeat the accretion pattern. Greenfield ships the target directly. Data migration and TS both cost dependency-heavy work that would couple 0.6.0 to old-shape concerns; deferring scopes 0.6.0 tight enough to ship correctly.

- **0.7.0 must deliver for TS-based apps to recover.** TS binding is committed to 0.7.0, not speculative.
- Data migration is deferred on the bet that fresh-DB cutover is an acceptable cost in exchange for a correct release. If a specific client cannot accept fresh-DB, see §"Data migration" below for the decision path.

### Future schema-migration policy (brief)

Keeps the v1..v25 chain from growing back. Stored as a contract in `crates/fathomdb-schema/CONTRIBUTING.md`. Enforced by PR review, not CI.

### Dependency sequencing (scope-locked, no calendar)

Schedule is not the concern; clear requirements, correct designs, dependency identification, and sequencing are. Work order:

1. **Data model frozen first.** v1 schema + POST checks + typed errors. Nothing else can be sequenced until this is locked.
2. **`fathomdb-embedder-api` crate published** (even if only to local path during 0.6.0 dev). Engine and embedder both depend on it; defining it early prevents retroactive trait churn.
3. **Writer actor extracted from 0.5.x.** Single-connection mpsc + fsync + post-commit hook. Known-load-bearing; extraction is the hardest piece of preserved code and its shape dictates what `admin.rs` and `projection_actor.rs` can do. Until the extracted writer actor is green against its own tests, nothing downstream is real.
4. **`admin.configure(schema)` implementation.** Depends on writer actor (all DDL goes through it) and on the frozen data model.
5. **Query execution.** Coordinator trim. Depends on data model; does not depend on admin.
6. **Projection actor.** Depends on writer + admin. Folds in the 0.5.x rebuild-actor responsibilities.
7. **Python binding.** Depends on Rust public surface stable. Thin wrapper, no logic.
8. **`fathomdb-doctor` CLI.** Depends on engine. No migration subcommand in 0.6.0.
9. **Integration test `memex_flow.py`** — written in parallel with step 1, red throughout, goes green at step 6. Gate on ship.
10. **Operational-collections resolution** (HITL gate before step 4).
11. **Optional pre-tag dogfood run** (HITL — Memex availability).

Ship when integration test passes on all four platforms on the built wheel. No alphas, no RCs, no soak. Passes → tag. Fails → fix.

## Principles re-applied

From `feedback_reliability_principles`:

1. **Soak is not engineering.** Release gate is the integration test. Passes → ship. Fails → fix.
2. **Delete before add.** Rewrite is net −37,000 LoC. Every addition in 0.6.x must name what it deletes.
3. **No punt for bugs that already burned clients.** `SQLITE_SCHEMA` class is eliminated by schema shape (single `vec_nodes`), not by retry wrapper. Silent-None embedder is eliminated by deleting the variant. Each Memex-reported issue has a structural fix, not a band-aid.

## The one sentence

fathomdb is a five-verb embedded agent datastore: open an SQLite file, declare your kinds and their FTS + vector indexes, write nodes + edges + chunks + provenance in transactions, search with composed FTS + vector + graph retrieval, close cleanly — and nothing else in the SDK.

## Appendix: Vector Shape

It should change by becoming simpler and more derived.

Right now the vector layer is shaped around per-kind `sqlite-vec` tables plus multiple
profile/config tables and admin verbs. That shape is what the rewrite is trying to delete.

The rewrite target is the right one:

- One canonical store: `nodes`, `edges`, `chunks`.
- One vector projection per facet, not per kind.
- One embedder identity record, singleton, owned by the open-time embedder.
- One declarative schema source (`configure(schema)`), not vector-specific admin verbs.
- One durable projection work queue, so vectors are rebuilt from canonical state instead of being
  treated as primary data.

So the concrete vector-shape changes are:

1. Replace per-kind vec tables with shared facet projections.
   Current shape: `vec_<kind>` physical tables and kind-derived table naming.
   Target shape: logical `vec_nodes` and `vec_chunks`, keyed by `row_id` with `kind` as data, not
   schema.

2. Collapse vector metadata to one identity source.
   Current shape: `vector_profiles`, `projection_profiles`, and regen config carrying destination or
   profile metadata.
   Target shape: one singleton `vector_embedding_profile`; kind enablement lives in
   `schema_config`, not parallel tables.

3. Make vectors purely derived from canonical `chunks`.
   The write path should write canonical rows, enqueue projection work, and let `drain(timeout)` be
   the synchronization boundary. Vectors are post-commit derived state, never canonical state.

4. Remove vector-specific public configuration.
   `admin.configure(schema)` declares which kinds and facets are vector-indexed, and the engine
   computes the diff. The current spread of vector-specific verbs should disappear from the public
   SDK.

5. Keep query-time vector execution inside the same fused planner.
   The vector branch stays part of one planner that also owns FTS and graph traversal. It does not
   become a second engine or a sidecar database.

One implementation detail to be careful about: the logical shape above is correct, but the
physical SQLite shape may still need a small split if `sqlite-vec` cannot efficiently do "single
table with kind filter and KNN" in one shot. If so, keep the public and logical shape as one
shared vector projection, but implement it internally as:

- one shared vec virtual table keyed by `row_id`
- one ordinary sidecar table mapping `row_id` to kind and node or chunk metadata

That still preserves the real goal: no per-kind runtime DDL, no schema-per-kind explosion, no
duplicated identity/config surfaces.

The short version: change the vector layer from "many kind-specific vector schemas with public
admin surface" to "one derived projection subsystem with one identity, one schema declaration, and
one queue."
